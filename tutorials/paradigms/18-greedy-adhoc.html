1. Problem Definition FOUNDATIONS

In competitive programming, greedy and adhoc represent two complementary problem-solving paradigms. Understanding where each applies — and where neither does — is the first step toward contest transfer.

What is a greedy algorithm?

A greedy algorithm builds a solution incrementally by making the locally optimal choice at each step, committing to that choice permanently. The critical requirement: this local rule must lead to a globally optimal solution. Without a proof of this property, a greedy guess is just a guess — and contest judges exploit unproven guesses ruthlessly.

What is an adhoc problem?

An adhoc problem has no standard algorithmic classification. It requires you to discover structure from scratch: simulate small cases, find invariants, reframe the problem into a simpler equivalent, and only then choose a technique. Many adhoc problems end up having greedy solutions — but the modeling step comes first.

📌

The adhoc modeling toolkit: Simulation on small cases, searching for invariants, parity and counting arguments, rewriting the problem as a different combinatorial object, and greedy-as-subsequence construction. These are the tools you reach for before any named algorithm.

The three families

⚠

The default is DP, not greedy. If you cannot write a correctness proof with one of the four templates (exchange, stays-ahead, cut, rollback), assume the problem requires DP, flow, or a different model entirely. Greedy is the exception, not the rule.

2. Intuition from Scratch FOUNDATIONS

Why can a local rule ever produce a global optimum? The answer lies in a structural property: the greedy choice does not reduce the set of reachable optimal solutions. Making the greedy choice first still leaves at least one optimal completion available.

The "never falls behind" intuition

Consider filling a knapsack with divisible materials. If you always pour in the densest material first, you never waste capacity on something less valuable. At every prefix of your decisions, you are at least as good as any alternative strategy. This is the stays-ahead principle in action.

The "swap without regret" intuition

Consider scheduling activities. If an optimal solution starts with activity A, but your greedy picks activity G that finishes earlier, you can swap A for G without losing anything — G ends sooner, so everything that fit after A still fits after G. This is the exchange principle: any disagreement between greedy and optimal can be resolved in greedy's favor.

Greedy is safe when: for any optimal solution OPT that disagrees with greedy's first choice, you can "fix" OPT to agree with greedy without worsening the objective.
✓

The contest workflow: (1) Simulate tiny inputs (n=3,4,5) to find structure. (2) Guess a greedy rule. (3) Try to break it with a counterexample. (4) If it survives, choose a proof template and write the proof. (5) Only then implement.

4. Formal Abstraction FOUNDATIONS

Formally, a greedy algorithm works on a problem that exhibits the greedy-choice property and optimal substructure.

Greedy-choice property

A globally optimal solution can be arrived at by making a locally optimal choice. Specifically: there exists an optimal solution that includes the greedy's first choice. This must be proven, not assumed.

Optimal substructure

After making the greedy choice, the remaining subproblem is a smaller instance of the same problem. The optimal solution to the original problem contains optimal solutions to subproblems.

Greedy-choice property + optimal substructure = greedy correctness

Exchange argument (formal)

Define greedy solution G and an arbitrary optimal solution OPT. Identify the first position where OPT differs from G. Swap OPT's choice with G's choice. Prove: (a) feasibility is preserved, and (b) the objective does not worsen. Repeat until OPT becomes G. Therefore G is optimal.

Comparator derivation (the N=2 technique)

For sorting-based greedy problems, analyze two adjacent items and determine when swapping them improves the objective. This two-item reasoning directly yields the comparator for std::sort. This technique is explicitly described in competitive programming references as the standard approach for deriving "strange comparators."

📌

Example of N=2 technique: For "Largest Number" (LeetCode 179), compare strings a+b vs b+a. If a+b > b+a, then a should come before b. This comparator is derived by asking: "for just two elements, which order gives a better result?"

5. Key Observations FRAMEWORK

Before diving into individual problems, internalize these observations that govern when greedy works and when it does not.

✓

The core insight: Greedy works when each local commitment provably cannot block a globally optimal completion. The proof templates in §6 formalize this — the decision table in §8 maps it to specific problem patterns.

✗

When greedy breaks: If a local choice can block future feasibility in ways that no simple swap can fix, the problem needs DP or another model. The counterexamples below illustrate exactly where this boundary lies.

Counterexample: greedy coin change fails

For denominations {1, 4, 5} and amount 8: greedy picks 5+1+1+1 = 4 coins. Optimal is 4+4 = 2 coins. The greedy choice (largest denomination first) blocks a better combination. This is a classic exchange-argument teaching example — the swap does not preserve feasibility because coins are indivisible and non-exchangeable in the required way.

Counterexample: weighted interval scheduling

When activities have weights (profits), earliest-finish greedy fails. A single long high-profit activity may be better than many short low-profit ones. The weighted variant requires DP, not greedy.

6. Proof Templates PROOFS

These four templates are what convert a greedy guess into a contest-correct solution. They are not "extra theory" — they are required for any greedy approach to be trustworthy.

Template 1: Exchange Argument (Swap-to-Greedy)

Setup: Let G be the greedy solution and OPT be any optimal solution.

Step 1: Find the first position where OPT differs from G.

Step 2: Replace OPT's choice at that position with G's choice.

Step 3: Prove feasibility is preserved.

Step 4: Prove the objective does not worsen.

Step 5: Repeat — OPT converges to G. Therefore G is optimal.

Used by: Fractional knapsack (§10), activity selection (§9), job sequencing (§11).

Template 2: Stays-Ahead (Prefix Dominance)

Setup: Define a measurable prefix progress: finish time, covered distance, remaining slack, total cost so far.

Base case: Greedy's first step is best by definition (e.g., earliest finish).

Inductive step: If greedy is ahead at step r, then step r+1 keeps it ahead.

Conclusion: Greedy completes ≥ tasks / achieves ≤ cost at every prefix. Therefore it is optimal overall.

Used by: Activity selection (§9), interval scheduling variants.

Template 3: Cut / Safe-Edge Property

Setup: Define a cut (S, V\S) that respects the current chosen set.

Claim: The lightest edge crossing the cut is safe — it belongs to some MST.

Proof: Assume an MST T that excludes this light edge e. Adding e to T creates a cycle that crosses the cut via another edge f. Since w(e) < w(f), removing f yields a lighter tree — contradiction.

Used by: Kruskal (§14), Prim (§14).

Template 4: Rollback / Replacement Pattern

Setup: Process items in a constraint order (often by deadline).

Step 1: Insert each candidate into a chosen set.

Step 2: If constraints are violated, remove the "worst contributor" (often the item with maximum duration or minimum profit).

Invariant: Among all processed items so far, the chosen set is a best feasible subset.

Used by: Course Schedule III / scheduling with durations (§13).

Proof drill: deriving a comparator via N=2 click to expand

Problem: Given n strings, concatenate them to form the largest possible number.

N=2 analysis: For two strings a, b, compare a+b vs b+a (as strings). If a+b > b+a, then a should precede b.

Extension to N: This pairwise comparison defines a total order. Sorting by this comparator and concatenating gives the globally optimal result. The exchange proof: if any adjacent pair is out of order, swapping them improves the result — so the sorted order is optimal.

7. Invariants PROOFS

Every greedy algorithm maintains invariants — properties that hold before and after each step. Understanding them is essential for correctness proofs and for debugging.

A) State invariant

The property that holds at each moment during execution. For activity selection: "the set of chosen activities is pairwise non-overlapping, and the last chosen activity has the earliest possible finish time among all valid choices." For Dijkstra: "every finalized vertex has its true shortest-path distance."

B) Transition invariant

How each greedy step preserves the state invariant. For activity selection: "adding the next earliest-finish compatible activity preserves non-overlapping and advances the finish time minimally." For rollback scheduling: "adding or removing an element maintains the property that the heap contains a best feasible subset."

C) Termination invariant

Why the algorithm stops and what holds at termination. For Kruskal: "after processing all edges, the DSU contains exactly the MST edges (or a spanning forest if disconnected)." For Bellman-Ford-style rollback: "after processing all items sorted by deadline, the heap contains the maximum feasible subset."

⚠

Failure mode — wrong applicability: Applying greedy to a problem that lacks the greedy-choice property. Classic example: 0/1 knapsack with density sort. The invariant "densest items are always best" fails because an item with lower density but smaller weight may leave room for a more profitable combination.

⚠

Failure mode — hidden assumption: The greedy works only under conditions not stated in the problem. Example: Dijkstra's greedy "finalize closest vertex" requires non-negative edge weights. A negative edge can improve a finalized vertex later, breaking the "final when popped" invariant.

Quick reflex rules

Sort + Scan
intervals / deadlines / densities

If you can sort by one key and scan left-to-right making irrevocable choices, it is likely greedy. Prove via exchange or stays-ahead.

Heap + Maintain
deadline + duration / merge cost

If you process items in order and need to maintain a "best so far" set, use a heap. Drop the worst element when constraints break (rollback) or merge the two smallest (Huffman).

DSU + Edges
connect components / latest free slot

If the problem involves connectivity (MST) or slot assignment (job sequencing), DSU gives near O(1) amortized operations per step.

Cut Property — "The lightest crossing edge is safe"

💡

Super-simple idea: Split graph into two groups (a "cut"). The smallest edge between the groups is always safe to include in the MST. Excluding it leads to a contradiction.

When to use:
Graph problems where you greedily pick the cheapest connecting edge (MST).
Proves:
Kruskal (§14), Prim (§14), any MST variant.

Rollback — "Add everything, then fix mistakes"

💡

Super-simple idea: Keep adding items. If a constraint breaks, remove the worst contributor. You always maintain the best feasible set seen so far.

When to use:
Process items in order, maintain a set, drop the worst element when overloaded.
Proves:
Scheduling with durations (§13), Course Schedule III, priority scheduling.

Interactive Scheduling Rollback Tracer

📌

Problem — Job Sequencing with Deadlines. You have n jobs. Each job j has a duration dj (how long it takes to run) and a deadline Dj (time by which it must be finished). You process jobs one at a time with no preemption. A chosen subset S is feasible iff, when you run its jobs in deadline order, every job finishes by its deadline — equivalently, the sum of durations of all jobs chosen with deadline ≤ t is ≤ t for every t. Goal: maximize |S| — the number of jobs completed on time.

⚙️

Algorithm — Add-then-Rollback. (1) Sort jobs by deadline ascending. (2) Maintain a max-heap H of chosen durations and running total T = Σ dj∈H. (3) For each job j in order: push dj into H and add dj to T. (4) If T > Dj, pop the max of H and subtract it from T. The popped element may be j itself (a rejection) or an earlier commitment (a swap). Either way, H always holds the largest feasible subset of processed jobs seen so far.

Example jobs (already sorted by deadline): A(d=3, D=3), B(d=2, D=4), C(d=1, D=5), D(d=4, D=8). Watch step 4 — that's where a true swap happens.

9. Activity Selection CLASSIC GREEDY

Problem

Select the maximum number of pairwise non-overlapping activities with start time sᵢ and finish time fᵢ on a single resource.

Greedy rule

Sort by finish time ascending. Take the next activity that starts at or after the last chosen finish time.

Intuition

The activity that finishes earliest leaves the maximum remaining room for future activities. By always choosing the earliest-finishing compatible activity, you greedily preserve future flexibility.

Formal proof (exchange)

Proof sketch: Let greedy pick g₁ with the earliest finish. For any optimal solution whose first chosen activity is o₁, we have f(g₁) ≤ f(o₁). Replace o₁ with g₁: feasibility remains because g₁ ends earlier, and the solution size remains unchanged. Repeat on the remaining subproblem — each swap makes OPT agree with greedy at one more position without losing any activity. Therefore greedy is optimal.

Formal proof (stays-ahead)

Stays-ahead lemma: Define greedy choices g₁, g₂, …, gₖ and optimal choices o₁, o₂, …, oₘ (both ordered by finish). Prove by induction that f(gᵣ) ≤ f(oᵣ) for all r ≤ min(k,m).

Base case: f(g₁) ≤ f(o₁) by definition — g₁ has the earliest finish among all activities.

Inductive step: Assume f(gᵣ) ≤ f(oᵣ). Then s(oᵣ₊₁) ≥ f(oᵣ) ≥ f(gᵣ) — so oᵣ₊₁ is compatible with gᵣ. Greedy picks gᵣ₊₁ with the earliest finish among all compatible activities, so f(gᵣ₊₁) ≤ f(oᵣ₊₁).

Conclusion: Greedy matches or exceeds optimal at every prefix. Since greedy never falls behind, k ≥ m. Therefore greedy selects at least as many activities as any optimal solution.

C++ template

#include <bits/stdc++.h>
using namespace std;

struct Act { long long s, f; };

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n; cin >> n;
    vector<Act> a(n);
    for (int i = 0; i < n; i++) cin >> a[i].s >> a[i].f;

    // GREEDY RULE: sort by finish time ascending
    sort(a.begin(), a.end(), [](const Act& x, const Act& y) {
        if (x.f != y.f) return x.f < y.f;
        return x.s < y.s;
    });

    long long last = LLONG_MIN / 4;
    int cnt = 0;
    for (auto& act : a) {
        if (act.s >= last) {   // compatible: starts at or after last finish
            cnt++;
            last = act.f;       // commit — irrevocable
        }
    }
    cout << cnt << "\n";
    return 0;
}
Line-by-line explanation click to expand

sort by finish time: The greedy rule requires processing activities in order of earliest finish. Secondary sort by start is a tiebreaker only.

last = LLONG_MIN/4: Initial "last finish" is set to negative infinity so the first activity is always compatible.

if (act.s >= last): Compatibility check. Change >= to > if the problem requires strict non-overlap.

last = act.f: Commit to this activity. The greedy never revisits this choice.

Complexity

O(n log n) sort + O(n) scan = O(n log n)
⚠

Common mistakes: Sorting by start time (fails — a long early activity blocks many short ones). Sorting by duration (fails — short activities may overlap with many others). Confusing with weighted interval scheduling (requires DP, not greedy).

10. Fractional Knapsack CLASSIC GREEDY

Problem

Maximize total value in a knapsack of capacity W, where each item i has value vᵢ and weight wᵢ, and items are divisible — you may take any fraction xᵢ ∈ [0,1].

Greedy rule

Sort by density rᵢ = vᵢ / wᵢ descending. Take as much as possible from the highest-density item first.

Intuition

Every unit of capacity should be spent on the "most valuable material per unit weight" first. Divisibility is the critical assumption — it allows continuous swapping of material between items, which is what makes the exchange argument work.

Formal proof (exchange)

Proof: Consider any solution that takes some positive weight from item j while leaving room to take more from item i, where rᵢ > rⱼ. Move δ weight from j to i: the value changes by δ(rᵢ − rⱼ) > 0, and feasibility is unchanged (total weight stays the same). Repeating this swap eliminates all inversions — the density-sorted order is the unique optimum.

C++ template

#include <bits/stdc++.h>
using namespace std;

struct Item {
    long long w, v;
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    long long W;
    cin >> n >> W;

    vector<Item> a(n);
    for (int i = 0; i < n; i++) cin >> a[i].w >> a[i].v;

    // GREEDY RULE: sort by density descending
    // Cross-multiply to avoid floating-point error in comparator
    sort(a.begin(), a.end(), [](const Item& A, const Item& B) {
        return (__int128)A.v * B.w > (__int128)B.v * A.w;
    });

    long double ans = 0.0L;
    for (auto& it : a) {
        if (W == 0) break;
        if (it.w <= W) {
            ans += it.v;       // take entire item
            W -= it.w;
        } else {
            ans += (long double)it.v * (long double)W / (long double)it.w;
            W = 0;             // knapsack full — take fraction
        }
    }

    cout << fixed << setprecision(6) << (double)ans << "\n";
    return 0;
}

Complexity

O(n log n) sort + O(n) scan = O(n log n)
⚠

Common mistakes: Sorting by value alone (fails — a heavy high-value item may be less dense than a light item). Sorting by weight alone (fails similarly). Using double in the comparator (may violate strict weak ordering — prefer cross-multiplication with __int128).

⚠

Critical: 0/1 knapsack. If items are indivisible, this greedy fails completely. Consider items: (w=10, v=60), (w=20, v=100), (w=30, v=120) with capacity 50. Density sort picks items 1 and 2 (value 160), but the optimal is items 1 and 3 (value 180). The exchange argument breaks because you cannot continuously swap fractions. Use DP instead.

11. Job Sequencing with Deadlines CLASSIC GREEDY

Problem

Each job takes 1 unit of time, has a deadline dᵢ and profit pᵢ. Schedule at most one job per time slot to maximize total profit. A job earns profit only if it finishes by its deadline.

Greedy rule

Sort by profit descending. For each job, place it in the latest free slot ≤ deadline.

Intuition

High-profit jobs get priority. But each job should be scheduled as late as possible — this preserves early slots for tighter-deadline jobs that come later in the sorted order.

Formal proof (exchange)

Why profit order is safe: If a lower-profit job blocks a higher-profit job in a feasible schedule, swapping them cannot decrease feasibility (both take one slot) and increases profit.

Why latest-slot placement is safe: Placing a job earlier than necessary can only reduce options for other jobs with smaller deadlines. Pushing each job to its latest possible slot preserves feasibility for future choices.

C++ template (DSU slots — near O(n log n))

#include <bits/stdc++.h>
using namespace std;

struct Job {
    int id, d;
    long long p;
};

// DSU for latest-available-slot queries
struct DSU {
    vector<int> parent;
    DSU(int n = 0) { init(n); }
    void init(int n) {
        parent.resize(n + 1);
        iota(parent.begin(), parent.end(), 0);
    }
    int find(int x) {
        if (parent[x] == x) return x;
        return parent[x] = find(parent[x]);  // path compression
    }
    // Occupy slot x: next available becomes find(x-1)
    void occupy(int x) {
        parent[x] = find(x - 1);
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    cin >> n;
    vector<Job> jobs(n);

    int maxD = 0;
    for (int i = 0; i < n; i++) {
        cin >> jobs[i].id >> jobs[i].d >> jobs[i].p;
        maxD = max(maxD, jobs[i].d);
    }

    // GREEDY RULE: highest profit first
    sort(jobs.begin(), jobs.end(), [](const Job& a, const Job& b) {
        return a.p > b.p;
    });

    DSU dsu(maxD);
    long long totalProfit = 0;
    int cnt = 0;

    for (auto& j : jobs) {
        int slot = dsu.find(j.d);   // latest free slot ≤ deadline
        if (slot > 0) {
            dsu.occupy(slot);
            totalProfit += j.p;
            cnt++;
        }
    }

    cout << cnt << " " << totalProfit << "\n";
    return 0;
}

Complexity

O(n log n) sort + O(n · α(D)) DSU ≈ O(n log n)
⚠

Common mistakes: Placing a job at the earliest available slot (suboptimal — blocks tighter-deadline jobs). Missing the unit-time assumption — if durations vary, this algorithm does not apply. Forgetting that slot 0 means "no slot available."

12. Huffman Coding & Optimal Merge CLASSIC GREEDY

Problem

Build an optimal prefix-free binary code for symbols with frequencies fᵢ, minimizing expected code length Σ fᵢ · depthᵢ. Equivalently: given n sorted files of sizes fᵢ, merge them pairwise to minimize total merge cost.

Greedy rule

Repeatedly merge the two smallest frequencies. Push the merged sum back into a min-heap.

Intuition

Smaller frequencies should end up deeper in the tree (longer codewords). Merging the two smallest guarantees they become siblings at maximal depth. Each merge costs the sum of the two merged values — so merging small values first minimizes the accumulated cost at each level.

Formal proof (sibling lemma + exchange)

Sibling lemma: In an optimal prefix-code tree, consider the two least-frequent symbols. There exists an optimal tree where they are deepest siblings. If not, swap them downward with deeper leaves: since they have the smallest frequencies, moving them deeper cannot increase Σ fᵢ · depthᵢ.

Recursive reduction: Contracting these siblings into a single pseudo-symbol of frequency f₁ + f₂ reduces the problem to n−1 symbols — exactly the greedy merge. By induction, the greedy produces an optimal tree.

C++ template (optimal merge cost)

#include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    cin >> n;
    priority_queue<long long, vector<long long>, greater<long long>> pq;
    for (int i = 0; i < n; i++) {
        long long x; cin >> x;
        pq.push(x);
    }

    long long cost = 0;
    while (pq.size() > 1) {
        long long a = pq.top(); pq.pop();   // smallest
        long long b = pq.top(); pq.pop();   // second smallest
        cost += a + b;                       // merge cost
        pq.push(a + b);                      // push merged node
    }

    cout << cost << "\n";
    return 0;
}

Complexity

O(n log n) — each of n−1 merges involves two heap operations
⚠

Common mistakes: Using a max-heap by accident (merging the two largest first gives the worst possible result). Confusing Huffman coding (optimal for prefix codes) with arithmetic coding (different model, not based on this greedy). Forgetting the n=1 edge case (no merges needed).

13. Scheduling with Durations (Rollback Pattern) SCHEDULING

Problem

Each job has duration tᵢ and deadline dᵢ. Execute sequentially on one machine. Maximize the number of jobs completed by their deadlines.

Greedy rule (rollback)

Sort by deadline ascending. Add each job's duration to a max-heap; track total time. If total > current deadline, remove the job with largest duration from the heap.

Intuition

Processing earlier deadlines first handles tight constraints. When overloaded, dropping the longest job frees the most time and maximizes future feasibility — the replacement invariant guarantees this is always optimal.

Formal proof (replacement invariant)

Invariant: After processing the first i jobs (sorted by deadline), the heap contains a feasible set with maximum size. Among all feasible sets of that size, it has minimum total time.

Transition: When adding job i breaks feasibility (sum > dᵢ), any feasible subset must drop some job. Dropping the maximum-duration job yields the smallest remaining sum — the "best chance" to remain feasible for future deadlines.

Termination: After all jobs are processed, the heap contains the maximum number of completable jobs.

C++ template

#include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n; cin >> n;
    vector<pair<int, int>> jobs(n);  // {duration, deadline}
    for (int i = 0; i < n; i++) cin >> jobs[i].first >> jobs[i].second;

    // GREEDY RULE: sort by deadline ascending
    sort(jobs.begin(), jobs.end(), [](auto& a, auto& b) {
        return a.second < b.second;
    });

    priority_queue<int> pq;  // max-heap of durations
    long long sum = 0;

    for (auto& [t, d] : jobs) {
        sum += t;
        pq.push(t);
        if (sum > d) {
            sum -= pq.top();  // ROLLBACK: remove longest job
            pq.pop();
        }
    }

    cout << (int)pq.size() << "\n";
    return 0;
}

Complexity

O(n log n) — sort + at most n heap operations
⚠

Common mistakes: Removing the smallest duration (wrong — you want to free the most time). Sorting by duration instead of deadline. Forgetting that this maximizes count, not profit — for maximum profit with deadlines, a different formulation is needed.

Variant: feasibility-only check

"Can we finish all jobs?" — sort by deadline, keep a running sum of durations, and check that the sum never exceeds the current deadline. No heap needed. This is the core of problems like AtCoder ABC131 D Megalomania.

14. Graph Greedy — MST & Dijkstra GRAPH

14a. Minimum Spanning Tree

Given a weighted undirected graph, find a spanning tree with minimum total edge weight.

Cut Property (the proof that governs both Kruskal and Prim)

For any cut (S, V\S), the minimum-weight edge crossing the cut is safe — it belongs to some MST.

Proof: Assume an MST T that does not contain the light edge e across a cut. Add e to T — a cycle forms and must cross the cut again via another edge f. Since w(e) < w(f), removing f yields a strictly lighter spanning tree — contradiction. Therefore e must belong to some MST.

Kruskal's algorithm

Sort edges by weight ascending. Add each edge if it connects two different components (no cycle). Uses DSU for cycle detection.

C++ template: Kruskal

#include <bits/stdc++.h>
using namespace std;

struct Edge { int u, v; long long w; };
struct DSU {
    vector<int> p, sz;
    DSU(int n = 0) { init(n); }
    void init(int n) {
        p.resize(n); sz.assign(n, 1);
        iota(p.begin(), p.end(), 0);
    }
    int find(int x) { return p[x] == x ? x : p[x] = find(p[x]); }
    bool unite(int a, int b) {
        a = find(a); b = find(b);
        if (a == b) return false;
        if (sz[a] < sz[b]) swap(a, b);
        p[b] = a; sz[a] += sz[b];
        return true;
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, m; cin >> n >> m;
    vector<Edge> edges(m);
    for (int i = 0; i < m; i++) {
        cin >> edges[i].u >> edges[i].v >> edges[i].w;
        --edges[i].u; --edges[i].v;
    }

    // GREEDY RULE: lightest edge first
    sort(edges.begin(), edges.end(),
         [](auto& a, auto& b) { return a.w < b.w; });

    DSU dsu(n);
    long long cost = 0;
    int used = 0;

    for (auto& e : edges) {
        if (dsu.unite(e.u, e.v)) {   // no cycle — safe edge (cut property)
            cost += e.w;
            used++;
        }
    }

    if (used != n - 1) cout << "IMPOSSIBLE\n";
    else cout << cost << "\n";

    return 0;
}

C++ template: Prim (sparse graph, heap)

#include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, m; cin >> n >> m;
    vector<vector<pair<int, long long>>> adj(n);
    for (int i = 0; i < m; i++) {
        int u, v; long long w;
        cin >> u >> v >> w; --u; --v;
        adj[u].push_back({v, w});
        adj[v].push_back({u, w});
    }

    const long long INF = (long long)4e18;
    vector<long long> dist(n, INF);
    vector<char> vis(n, 0);
    priority_queue<pair<long long, int>, vector<pair<long long, int>>,
                   greater<pair<long long, int>>> pq;

    dist[0] = 0;
    pq.push({0, 0});

    long long cost = 0;
    int taken = 0;

    while (!pq.empty()) {
        auto [d, u] = pq.top(); pq.pop();
        if (vis[u]) continue;
        vis[u] = 1;
        cost += d;
        taken++;

        for (auto [v, w] : adj[u]) {
            if (!vis[v] && w < dist[v]) {
                dist[v] = w;
                pq.push({dist[v], v});
            }
        }
    }

    if (taken != n) cout << "IMPOSSIBLE\n";
    else cout << cost << "\n";
    return 0;
}

Complexity

Kruskal: O(m log m) + O(m · α(n))

Dominated by sorting. DSU operations are near-constant.

Prim (heap): O((n + m) log n)

Each vertex processed once, each edge relaxed once, each heap operation O(log n).

14b. Dijkstra — Single-Source Shortest Paths

Given a graph with non-negative edge weights, find shortest paths from a source to all vertices.

Greedy rule

Repeatedly finalize the unfinalized vertex with smallest tentative distance, then relax its outgoing edges.

Formal proof (stays-ahead / finalization invariant)

Claim: When vertex u is popped from the priority queue, dist[u] is its true shortest-path distance.

Proof: Suppose a shorter path to u exists via some unvisited vertex x. That path has length ≥ d[x] because all edge weights are non-negative. But d[x] ≥ d[u] by the extraction rule. Contradiction. Therefore u's distance is final when popped.

Why negative edges break it: A later vertex can have a negative edge back that reduces an already-finalized vertex. The "final when popped" invariant fails.

C++ template

#include <bits/stdc++.h>
using namespace std;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, m; cin >> n >> m;
    vector<vector<pair<int, long long>>> adj(n);
    for (int i = 0; i < m; i++) {
        int u, v; long long w;
        cin >> u >> v >> w; --u; --v;
        adj[u].push_back({v, w});
        adj[v].push_back({u, w});  // remove if directed
    }

    const long long INF = (long long)4e18;
    vector<long long> dist(n, INF);
    vector<int> parent(n, -1);

    priority_queue<pair<long long, int>, vector<pair<long long, int>>,
                   greater<pair<long long, int>>> pq;

    int s = 0;
    dist[s] = 0;
    pq.push({0, s});

    while (!pq.empty()) {
        auto [d, u] = pq.top(); pq.pop();
        if (d != dist[u]) continue;   // discard stale PQ entry

        for (auto [v, w] : adj[u]) {
            if (dist[u] + w < dist[v]) {
                dist[v] = dist[u] + w;
                parent[v] = u;
                pq.push({dist[v], v});
            }
        }
    }

    for (int i = 0; i < n; i++) {
        cout << (dist[i] == INF ? -1 : dist[i]) << (i + 1 == n ? '\n' : ' ');
    }
    return 0;
}

Complexity

O((n + m) log n) with binary heap
⚠

Common mistakes: Running Dijkstra on graphs with negative edges (wrong answers). Not discarding stale PQ entries (if (d != dist[u]) continue;) — causes TLE. Overflow: always use long long and a sufficiently large INF.

16. Edge Cases & Common Pitfalls FRAMEWORK

✓ Centralized Edge-Case Audit

  • n = 0 or n = 1: empty input or trivially solved
  • All items identical: greedy must still terminate correctly
  • All deadlines = 1: only one job can be scheduled
  • Capacity W = 0: no items taken (fractional knapsack)
  • Single-node graph: MST cost = 0, Dijkstra dist = all 0 or INF
  • Disconnected graph: MST does not exist; Dijkstra reports INF for unreachable
  • Negative edge weights: Dijkstra fails; MST still works (weights are just edge attributes)
  • Ties in comparator: must handle stably or break consistently
  • Overflow: density cross-multiplication may overflow long long — use __int128
  • Strict vs non-strict overlap: check problem statement carefully (≥ vs >)

⚠ Common Pitfalls (by topic)

  • Activity Selection: Sorting by start time or duration
  • Fractional Knapsack: Sorting by value or weight alone
  • Fractional Knapsack: Using double in comparator (strict weak ordering violation)
  • Job Sequencing: Earliest slot instead of latest slot
  • Job Sequencing: Assuming non-unit durations work with this algorithm
  • Huffman: Using max-heap instead of min-heap
  • Rollback: Removing smallest duration instead of largest
  • Rollback: Sorting by duration instead of deadline
  • Kruskal: Running on directed graph without undirected conversion
  • Dijkstra: Running on negative-weight edges
  • Dijkstra: Not discarding stale PQ entries (causes TLE)
  • All greedy: Applying greedy without a proof — WA on hidden test cases

17. Variants & Extensions VARIANTS

Variant 1: 0/1 Knapsack (Greedy Fails → DP)

Problem: Same as fractional knapsack, but items are indivisible. Take each item entirely or not at all.

Why greedy fails: Density sort can miss profitable combinations. The exchange argument requires continuous fraction swapping, which is impossible with whole items.

Algorithm: DP with state dp[i][w] = max value using first i items with capacity w. Time O(nW), space O(W) with rolling array.

When to recognize: If items are indivisible and there is a capacity budget, always use DP.

Variant 2: Weighted Interval Scheduling (Greedy Fails → DP)

Problem: Same as activity selection, but each activity has a profit. Maximize total profit.

Why greedy fails: Earliest-finish greedy may select many low-profit activities instead of one high-profit activity that covers the same time span.

Algorithm: Sort by finish time. Binary search for the latest compatible job. DP: dp[i] = max(dp[i-1], profit[i] + dp[p(i)]) where p(i) is the latest compatible job.

Practice: CSES Projects

Variant 3: Minimum Deletions for Non-Overlapping Intervals

Problem: Minimum number of intervals to remove so that the rest are pairwise non-overlapping.

Why it fits: This is exactly n − (max compatible set). Solve activity selection, then subtract from n.

Practice: LeetCode 435 — Non-overlapping Intervals

Variant 4: k-ary Huffman

Problem: Same as Huffman, but each internal node has k children instead of 2.

What changes: Merge the k smallest values at each step. Pad with zeros if (n−1) is not divisible by (k−1) to ensure the final merge has exactly k children.

Why it fits: Same sibling lemma, generalized to k-ary trees.

Variant 5: Maximum Spanning Tree

Problem: Find a spanning tree with maximum total weight.

What changes: Negate all weights and run Kruskal, or equivalently sort edges in descending order.

Why it fits: The cut property applies symmetrically — the heaviest edge across a cut is safe for a max-weight tree.

19. Final Mental Model FRAMEWORK

greedy = correct modeling + provably-safe local rule + optimal data structure

Six statements. If you can recall and explain all six, you have mastered greedy and adhoc.

The Six Things You Must Know

①
Greedy is the exception, not the rule
Most optimization problems require DP. Greedy is only safe when you can prove it with one of four templates: exchange, stays-ahead, cut, or rollback. If you cannot write the proof, do not trust the greedy.
②
The proof IS the algorithm
The exchange argument tells you what to sort by. The stays-ahead lemma tells you which prefix measure to track. The cut property tells you which edge is safe. The rollback pattern tells you what to remove. The proof directly generates the code.
③
Derive comparators from the N=2 case
For any sorting-based greedy, analyze two adjacent items: when does swapping them improve the objective? The answer is your comparator. This is the standard "strange comparator" technique in competitive programming.
④
Counterexamples before code
Before implementing any greedy, spend 2 minutes trying to break it with small inputs (n=3,4,5). Coin change {1,4,5} amount 8. Density sort for 0/1 knapsack. Earliest start for activity selection. If it breaks, pivot to DP.
⑤
Data structure = implementation of the proof
Sort for exchange-based proofs. Min-heap for Huffman and Dijkstra. Max-heap for rollback. DSU for connectivity and slot assignment. The data structure is chosen to make the proven greedy rule efficient — never the other way around.
⑥
Adhoc = find the structure first
Adhoc problems require structural discovery before any algorithm applies. Simulate small cases, identify invariants, reframe the problem. Many adhoc problems have greedy solutions — but the modeling step is creative, not mechanical.

✓ Core Concepts Checklist

  • Exchange argument: swap OPT's choice with greedy's, show no loss
  • Stays-ahead: greedy's prefix measure ≤ optimal's at every step
  • Cut property: lightest edge across any cut belongs to some MST
  • Rollback: maintain best feasible set; remove worst on overflow
  • N=2 comparator derivation for sorting-based problems
  • Activity selection: earliest finish time → maximum compatible set
  • Fractional knapsack: density sort → exchange on continuous fractions
  • Job sequencing: profit descending + DSU for latest free slot
  • Huffman: merge two smallest → sibling lemma + induction
  • Rollback scheduling: deadline sort + max-heap drop on overflow
  • Dijkstra: non-negative weights → finalize closest vertex
  • 0/1 knapsack and weighted intervals require DP, not greedy
