Binary & Ternary Search

Boundary Finding · Answer Space · Transformed Search · Unimodal Optimization
Prepared by The Coach (Dr. Wahab) & Ashraf

🧩 Problem Definition

Binary search and ternary search are algorithms that exploit structure in the search space to discard large portions of candidates at each step. Used naively, they find elements in sorted arrays. Used powerfully, they solve optimization problems that seem impossible to brute-force.

Binary Search — finds a boundary in a monotone predicate Ternary Search — finds the optimum of a unimodal function Both reduce O(N) or worse to O(log N) per decision

What binary search is really doing

It is not primarily about "taking the middle of an array." That is only one application. The deep idea is:

Find the boundary between a bad region and a good region.

The bad/good structure comes from a monotone predicate — a yes/no question where the answer flips at most once.

What ternary search is really doing

It is not a "binary search on yes/no." The structure is completely different:

Find the minimum (or maximum) of a unimodal function.

There is no flip from false to true — instead there is a valley (or hill) with one best point.

⚠️ The most important design principle

Binary search is for monotone predicates. Ternary search is for unimodal functions. Do not confuse them. Using binary search on a unimodal function or ternary search on a monotone predicate will give silently wrong answers.

The four flavors of search — at a glance

Before going further, orient yourself. Every search problem you will encounter in competitive programming fits one of these four patterns.

💡 Intuition from Scratch

The dictionary principle

Imagine you are looking for a word in a dictionary. You do not start from page 1 and flip through one page at a time. Instead:

  1. Open somewhere near the middle
  2. Compare — are you too far left or too far right?
  3. Discard the half that cannot contain your target
  4. Repeat on the smaller half

This works because the dictionary is ordered. Order is what allows you to safely throw away half the possibilities at each step.

The deeper principle

Search becomes powerful when structure lets you safely discard a large region. For binary search, the structure is a monotone predicate — once the answer becomes "good," it stays "good." For ternary search, the structure is unimodality — there is one valley (or peak) with no fake valleys elsewhere.

Plain-language definitions (read these carefully)

What "monotone predicate" means in plain language

Imagine a long row of boxes, each labeled YES or NO. A monotone predicate means all the NO boxes come first, then all the YES boxes come after — with no mixing. Once you see YES, every box to the right is also YES. Once you see NO, every box to the left is also NO.

Binary search exploits this: check the middle box. If it is YES, you can throw away everything to the right of it (all YES, no point searching there). If it is NO, throw away everything to the left.

If the boxes mix (NO, YES, NO, YES…), binary search is useless. That is when you need a different algorithm entirely.

What "unimodal function" means in plain language

Imagine a mountain or a valley. The height (or depth) changes in one direction until it reaches a single peak (or bottom), then changes back. There are no smaller hills or extra dips along the way.

Ternary search exploits this shape: probe two points inside the interval. Whichever probe is worse tells you which outer third cannot contain the best point. Discard that third and repeat.

If the function has multiple peaks or valleys (wiggly), ternary search will converge to the wrong one. Only use it when you are sure the function is genuinely unimodal.

Why exact array search is a special case

Most students learn binary search as "find a target in a sorted array." But that is only one consequence of the deeper boundary view. A sorted array gives us the predicate a[i] >= target, which is monotone over indices. We are really doing boundary search, just on array indices.

This unification means the exact same code pattern — with the same invariant, the same loop condition, the same return structure — handles:

array lower bound array upper bound first/last occurrence maximize minimum distance minimize maximum load minimum sufficient time/capacity fractional / ratio optimization

🖼 Visual Mental Model

The most important picture to have in your head for binary search:

Monotone predicate — the boundary

Binary search keeps one known bad point (s) and one known good point (e), and shrinks the gap between them until they are adjacent. At that point, s is the last bad and e is the first good.

Unimodal function — the valley

At each step, comparing f(a) and f(b) tells us which third cannot contain the minimum. One third is safely discarded, reducing the interval to 2/3 its previous size.

📐 Formal Abstraction

Definition 1: Ordered search space

A set of candidates S where "left" and "right" have meaning, and we can evaluate any candidate in O(T) time. Examples: array indices, integer answer values, real intervals.

Definition 2: Monotone boolean predicate

A function ok(x) → {false, true} that changes in only one direction across the ordered search space. Canonical forms:

Form A — false then true

ok(0) = false
ok(1) = false
ok(2) = false
ok(3) = true
ok(4) = true
ok(5) = true
Binary search → first true (e)

Form B — true then false

ok(0) = true
ok(1) = true
ok(2) = true
ok(3) = false
ok(4) = false
ok(5) = false
Binary search → last true (s) via flip

Definition 3: Unimodal function

A function f: S → ℝ with exactly one best region. For minimization: strictly decreasing to one minimum, then strictly increasing. For maximization: the mirror image. No fake valleys or plateaus that mislead the search.

How to recognize monotonicity — checklist

Before writing any binary search code, ask these questions about your check function ok(x):

  • If ok(x) is true, is ok(x−1) guaranteed to also be true (or ok(x+1) always true)? → one-directional propagation
  • As x increases, does feasibility only get easier, or only get harder — never both? → no oscillation
  • Can you prove a statement like "if distance d works, then any smaller d also works"? → formal monotonicity argument
  • If you cannot prove any of the above, binary search is the wrong tool.

Theorem: Binary search finds boundaries in O(log R) evaluations

If ok(x) is monotone over a search space of size R, binary search locates the boundary in at most ⌈log₂ R⌉ evaluations of ok(x).

Proof idea: each evaluation halves the search range. Starting from R, after k evaluations the range is at most R/2^k. When this drops to 1, the boundary is pinned.

🔍 Key Observations

Observation 1

The whole point of binary search is not to find a target value. It is to find where a predicate changes from false to true (or vice versa). Target-in-array is just one predicate.

Observation 2

The hardest part of a binary search problem is rarely the loop. It is designing the correct check function ok(x) and proving it is monotone.

Observation 3

The answer-space pattern: when a problem asks for an optimal value, binary search over candidate answers and convert to a feasibility check. This transforms optimization → decision.

Observation 4

Sorted-array queries (lower bound, upper bound, first/last occurrence) are identical in structure to answer-space search. Same template, same invariant, just different predicates.

Observation 5

Transformed binary search: some optimization problems become monotone only after an algebraic transformation. Guessing a ratio x and checking if a positive cycle exists is the key example.

Observation 6

Ternary search ≠ binary search variant. They live in different worlds. Binary search needs monotone predicates. Ternary search needs unimodal numeric functions. Confusing them is a class-defining bug.

⚙️ Algorithm Explanation

The master integer binary search algorithm operates as follows: maintain s (known bad) and e (known good). Repeatedly evaluate the midpoint m. If ok(m) is true, move e left to m. If ok(m) is false, move s right to m. Repeat until e − s = 1.

🔒 Invariants

A. State invariants

Invariant I — s is always bad

Throughout the entire execution: ok(s) = false. The initial s must be a known bad point. Every update sets s = m only when ok(m) is false, so this is preserved.

Invariant II — e is always good

Throughout the entire execution: ok(e) = true. The initial e must be a known good point. Every update sets e = m only when ok(m) is true, so this is preserved.

B. Transition invariants

The midpoint is strictly inside

As long as e − s > 1, the midpoint m = s + ((e−s) >> 1) satisfies s < m < e. This ensures every step genuinely shrinks the interval and termination is guaranteed.

C. Termination invariant

The boundary is fully pinned when e − s = 1

When the loop exits, e − s = 1, meaning no integer lies strictly between them. Combined with Invariants I and II: s is the last bad integer, e is the first good integer. This is exactly what we wanted to find.

D. Failure modes (invariant violations)

Failure Mode 1: Initial s is not bad

If ok(s_initial) = true, Invariant I is violated immediately. The final s may not be the last bad point. This causes wrong answers silently.

Failure Mode 2: Initial e is not good

If ok(e_initial) = false, Invariant II is violated. The true boundary may lie outside [s, e] entirely, and the algorithm will return incorrect results.

Failure Mode 3: ok(x) is not monotone

If the predicate has multiple false→true or true→false transitions, binary search may discard the very interval containing the answer. This is the deepest silent bug — the code looks correct but produces wrong output.

✅ Proof of Correctness

We prove the master integer template once. Every other use of it then inherits this correctness automatically.

Full Correctness Proof — Integer Binary Search click to expand/collapse

Lemma 1: Midpoint is strictly inside

If e − s > 1, then s < m < e where m = s + ((e − s) >> 1).

Proof: Since e − s ≥ 2, the shift (e − s) >> 1 = ⌊(e−s)/2⌋ ≥ 1, so m > s. Also ⌊(e−s)/2⌋ ≤ e−s−1 (since e−s ≥ 2), so m ≤ s + e − s − 1 = e − 1 < e. □

Lemma 2: Updates preserve invariants

Case A — ok(m) = true: m is a good candidate. The first good value lies in (s, m]. Set e = m. Now ok(s) = false (unchanged) and ok(e) = ok(m) = true. Invariants preserved. The search range shrinks to [s, m].

Case B — ok(m) = false: m is a bad candidate. The first good value lies in (m, e]. Set s = m. Now ok(s) = ok(m) = false and ok(e) = true (unchanged). Invariants preserved. The search range shrinks to [m, e].

In both cases the range strictly shrinks (by Lemma 1). □

Theorem: Returned pair {s, e} is correct

Termination: Each iteration reduces e − s by at least 1. The loop runs while e − s > 1, so it terminates in at most ⌈log₂(initial range)⌉ steps.

Correctness: When the loop exits, e − s ≤ 1. From the invariants: ok(s) = false and ok(e) = true. Since e − s = 1 (no integer strictly between them), s is the last bad integer and e is the first good integer. □

Correctness Proof — Double Binary Search Template click to expand/collapse

Invariant: s is always feasible

The template only executes s += sz when ok(s + sz) = true. So s always remains a valid feasible candidate. Initial s must be a valid feasible point.

Why step-halving is better than while(e-s>eps)

The common alternative while(e-s > eps) can stall due to floating-point rounding: the computed midpoint may equal s or e, creating an infinite loop. The step-halving loop shrinks sz geometrically regardless of floating-point behavior, guaranteeing termination in exactly ⌈log₂((e−s)/eps)⌉ iterations. □

⏱ Complexity Analysis

Integer Binary Search

If the search range has size R and the check ok(x) costs T:

O(T · log R)

The log R comes from halving: R → R/2 → R/4 → ... → 1 takes ⌈log₂ R⌉ steps. Each step calls ok(x) once.

Double Binary Search

Starting step size s₀ = (e₀ − s₀)/2, ending at eps:

O(T · log₂((e₀−s₀) / eps))

With eps = 1e-15 and range ≤ 1e6: about 70 iterations. Each halves the step size.

Ternary Search

Interval shrinks by factor 2/3 each iteration:

O(T · log_{3/2}(range / eps))

= O(T · log(range/eps) / log(3/2)) ≈ O(2.4 T · log₂(range/eps)). Slightly more than binary search.

🎯 Core Template Philosophy

The three master templates form a unified family. They share the same mental model and the same invariant structure. You should memorize the philosophy, not just the code.

Integer master

When to use: integer answer, discrete search space, predicate has a clear boolean boundary.

Invariant: s = last bad, e = first good.

Returns: pair {last bad, first good}.

AGGRCOW, Koko, Allocate Books

Double master

When to use: real-valued answer, continuous search space, same predicate logic but need precision.

Invariant: s = current feasible answer.

Returns: maximum feasible real value.

PIE, Ropes, Sightseeing Cows

Ternary master

When to use: unimodal numeric objective, want the minimum (or maximum) of a continuous function.

Invariant: interval [s, s+sz] contains the minimizer.

Returns: pair {x at minimum, f(x)}.

Weakness & Poorness, Meeting Place

Naming convention

All templates use the same names. s = start / bad side, e = end / good side, m = midpoint, ok(x) = check. Ternary uses f(x) = objective. Never rename these in your code — it destroys the mental model.

🚦 When NOT to Use Binary Search / Ternary Search

Choosing the wrong tool wastes time and gives wrong answers with no obvious error message. Learn to recognize when not to reach for binary search or ternary search.

Do NOT use Binary Search when…

  • No monotone predicate exists. If ok(x) can flip true → false → true, binary search will give wrong answers silently.
  • The answer space has no natural order. You cannot binary-search over a set of unordered objects.
  • A direct algorithm already exists. If greedy, dynamic programming, or a graph algorithm solves the problem directly, do not force a binary search wrapper around it.
  • The check function is too expensive. If T is large and log R does not compensate, binary search may be slower than brute force or other methods.
  • You cannot build valid initial boundaries. If you cannot find a guaranteed bad initial s and a guaranteed good initial e, the template invariant is broken from the start.

Do NOT use Ternary Search when…

  • The function is not unimodal. Multiple local minima/maxima cause convergence to the wrong point, with no warning.
  • Just because the answer is real-valued. A real-valued answer alone is not a reason for ternary search. If you have a monotone feasibility check, use double binary search instead.
  • The domain is discrete. On integers, ternary search is trickier and often less reliable than binary search with a carefully designed predicate.
  • The objective has flat plateaus. f(a) == f(b) breaks the standard discard rule. Either verify unimodality more carefully or use a perturbed comparison.
  • A closed-form optimum exists. If you can find the minimum analytically (e.g., derivative = 0 for a polynomial), ternary search is overkill.

⚡ The key question before you start

Ask yourself: "What structure does this problem have?"

→ One-way transition (false then true)? Use binary search.

→ One valley or one peak in a continuous function? Use ternary search.

→ Neither? Do not use either. The problem needs a different algorithmic idea.

📄 Master Integer Binary Search Template

s = last bad, e = first good Requires: ok(s_initial)=false, ok(e_initial)=true O(T · log R)
Full C++ Template — Integer Binary Search click to expand/collapse 
#include <cstdio>
#include <utility>

using namespace std;

bool ok(int x) {
    return x >= 10;   // ← replace with your predicate
}

pair<int, int> bs(int s, int e) {
    while (e - s > 1) {
        int m = s + ((e - s) >> 1);
        if (ok(m))
            e = m;
        else
            s = m;
    }
    return {s, e};
}

int main() {
    pair<int, int> ans = bs(0, 20);
    // ans.first  = last bad  (9)
    // ans.second = first good (10)
    printf("%d %d\n", ans.first, ans.second);
    return 0;
}

📄 Master Double Binary Search Template

s = current feasible answer Termination: step size < eps Step-halving avoids floating-point stall

How to think about step-halving (before reading the code)

The intuition is simple. Imagine you are standing at point s on a number line — a known feasible value — and you want to move as far right as possible while staying feasible.

  1. Try a jump of size sz. If ok(s + sz) is true, take the jump: s += sz.
  2. Whether you jumped or not, halve the jump size: sz *= 0.5.
  3. Repeat until sz is negligibly small (below eps).

After enough halvings, the remaining unexplored gap is smaller than eps, so s is the maximum feasible value up to the desired precision. This is exactly the same idea as the integer template — it is just expressed differently because floating-point has no concept of "adjacent integers."

This is not a different theorem. It is the same boundary-finding logic, adapted for real-valued domains by controlling progress through explicit step sizes rather than interval endpoints.

Full C++ Template — Double Binary Search click to expand/collapse 
#include <cstdio>

using namespace std;

bool ok(double x) {
    return x < 10;    // ← replace with your check
}

const double eps = 1e-15;

double bs(double s, double e) {
    for (double sz = (e - s) * 0.5; sz > eps; sz *= 0.5)
        if (ok(s + sz))
            s += sz;
    return s;
}

int main() {
    printf("%.17f\n", bs(0, 15));
    // Returns maximum feasible double in [0,15]
    return 0;
}

Why this design is correct

s = current feasible answer throughout. We only update s += sz when ok(s + sz) = true, so s is always valid. The step size sz shrinks by half each iteration regardless of floating-point precision. After about 50 iterations, sz drops below 1e-15 and we stop with s approximating the maximum feasible value.

⚠️ The danger of while(e-s > eps)

The common pattern while(e-s > eps){ m=(s+e)/2; ... } can stall: floating-point arithmetic can make m == s or m == e, triggering an infinite loop. The step-halving loop avoids this entirely because sz shrinks unconditionally.

📄 Master Ternary Search Template

For unimodal minimization f(x) must have exactly one valley Returns {x_min_approx, f_min_approx}
Full C++ Template — Ternary Search click to expand/collapse 
#include "bits/stdc++.h"
using namespace std;

double f(double x) {
    return (x - 5.0) * (x - 5.0);   // ← replace with your function
}

pair<double, double> ternarySearch(double s, double e) {
    double fs = f(s);
    for (double sz = e - s; sz > 1e-15; sz *= 2.0 / 3.0) {
        double a = s + sz / 3.0, b = a + sz / 3.0;
        double fa = f(a), fb = f(b);
        if (fa > fb) {    // ← for minimization: left third is higher, discard it
            s = a;
            fs = fa;      // ← cache to avoid recomputation on next step
        }
        // else: right third is higher or equal, sz shrinks and right side is trimmed
    }
    return {s, fs};
}

int main() {
    pair<double, double> ans = ternarySearch(0, 100);
    printf("%.10lf %.10lf\n", ans.first, ans.second);
    return 0;
}

How the interval shrinks

At each step, the effective search window is [s, s+sz]. Test points are at a = s+sz/3 and b = s+2sz/3.

• If f(a) > f(b): move s to a. New window: [a, s+sz]. Size = 2sz/3.

• If f(a) ≤ f(b): keep s. New window: [s, s+2sz/3]. Size = 2sz/3.

Either way, size shrinks by factor 2/3 per step. □

Why fs is cached

When s moves to a, we have already computed f(a). We store it as fs so that after convergence, fs ≈ f(minimum). This avoids one extra call to f, which matters when f is expensive (e.g., runs a graph algorithm).

After convergence, s is near the minimizer, so fs ≈ f(s) ≈ minimum value. This is the value returned as ans.second.

🔢 Sorted Array Queries

All sorted-array queries use the same master template. The only thing that changes is the predicate. The search space is array indices. The array being sorted makes each predicate monotone over indices.

Lower Bound — first index where a[i] ≥ target

Predicate: a[m] >= target. Pattern: F F F T T T. Answer is e (first good).

int s = -1, e = (int)a.size();

    int m = s + ((e - s) >> 1);

    else
        s = m;

// e = first index where a[i] >= target
// (or a.size() if no such index exists)

Upper Bound — first index where a[i] > target

Predicate: a[m] > target. Pattern: F F F F T T. Answer is e.

int s = -1, e = (int)a.size();
while (e - s > 1) {
    int m = s + ((e - s) >> 1);
    if (a[m] > target)   // ← upper bound predicate
        e = m;
    else
        s = m;
}
// e = first index where a[i] > target

First Occurrence

= Lower bound of target, then verify equality:

// run lower bound → get index e
if (e < (int)a.size() && a[e] == target)
    cout << e;    // first occurrence
else
    cout << -1;   // target not present

Last Occurrence

= Upper bound of target, then step back and verify:

// run upper bound → get index e
int ans = e - 1;
if (ans >= 0 && a[ans] == target)
    cout << ans;  // last occurrence
else
    cout << -1;

Exact Search

Exact search is not a separate algorithm. It is lower bound + equality check. There is no special "exact binary search" in this framework. This is the unification:

Exact search = lower bound + verify

If the lower bound index e is valid and a[e] == target, the target is present at index e. Otherwise it is absent. This handles all cases correctly, including duplicates.

🎯 Binary Search on Answer

This is the most important pattern in CP binary search. Instead of searching an array, we search over the answer values themselves. The predicate asks whether a candidate answer is feasible.

How to derive an answer-space binary search — step by step

  1. Identify the answer you want to optimize. Examples: minimum time, maximum spacing, minimum sufficient capacity.
  2. Guess a candidate value X. Instead of finding X directly, ask: "if the answer were X, would that be achievable?"
  3. Write the feasibility check. Define can(X) = true if the guessed answer X satisfies all constraints.
  4. Prove monotonicity. Show that if X works, then all values on one side of X also work. This is the critical step — without it, binary search is invalid.
  5. Set initial boundaries. Find one value definitely on the bad side and one value definitely on the good side.
  6. Run the master template. Read the answer from ans.first or ans.second depending on which direction feasibility propagates.

The flip pattern — last-true vs first-true

The master template always has s = last bad, e = first good. For "last true" problems (maximize minimum), define ok(x) = !can(x). Then bad = feasible and good = infeasible, so s = ans.first = last feasible = answer.

🐄 Full Worked Example — AGGRCOW

spoj.com/problems/AGGRCOW  ·  vjudge.net/problem/SPOJ-AGGRCOW

Maximize minimum distance Greedy check · O(N) Last-true via predicate flip

Problem story

You have N stalls spread along a line, each at a different position. You must place C cows in some of these stalls. The cows dislike each other — you want to place them so the closest pair is as far apart as possible. What is the maximum possible minimum distance?

Abstraction — why this is a binary search problem

Direct optimization is hard because there are C(N, C) ways to choose stalls. But if we guess a minimum distance d, we can answer in O(N): just greedily place cows and count how many fit. This feasibility check is monotone: if spacing d works, any smaller spacing d' also works.

Why the check is monotone

If you can place C cows with minimum spacing d, you can certainly place them with minimum spacing d' < d (just use the same placement — it still satisfies the weaker requirement). So as d increases, feasibility can only stay or get harder. The truth pattern is: T T T T F F F F — a classic last-true binary search.

Core idea — greedy check

Sort the stall positions. Greedily place cows: start with the first stall, then always choose the next stall that is at least d away from the last placed cow. Count how many cows fit. If the count is ≥ C, the spacing d is feasible.

Proof: greedy check is optimal click to expand

Claim: if any valid placement exists, greedy succeeds

Let greedy place cows at g₁ ≤ g₂ ≤ ... and some valid placement use p₁ ≤ p₂ ≤ ...

Base case: Greedy picks the earliest stall so g₁ ≤ p₁.

Inductive step: Assume gₖ ≤ pₖ. The next valid position after pₖ is ≥ pₖ + d ≥ gₖ + d. The earliest valid stall after gₖ is ≤ (earliest valid after pₖ), so g(k+1) ≤ p(k+1).

By induction, greedy never places a cow later than any other valid solution. If greedy cannot place all C cows, no solution can. □

Full C++ Code click to expand/collapse 
#include <bits/stdc++.h>
using namespace std;

int n, c;
vector<int> a;

bool can(int d) {
    int cnt = 1;
    int last = a[0];
    for (int i = 1; i < n; i++) {
        if (a[i] - last >= d) {
            cnt++;
            last = a[i];
        }
    }
    return cnt >= c;
}

bool ok(int d) {
    return !can(d);    // flip: bad = feasible, good = infeasible
}

pair<int, int> bs(int s, int e) {
    while (e - s > 1) {
        int m = s + ((e - s) >> 1);
        if (ok(m))
            e = m;
        else
            s = m;
    }
    return {s, e};
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);
    int tc;
    cin >> tc;
    while (tc--) {
        cin >> n >> c;
        a.resize(n);
        for (int i = 0; i < n; i++) cin >> a[i];
        sort(a.begin(), a.end());
        int s = 0;
        int e = a.back() - a.front() + 1;  // infeasible: spacing > total range
        pair<int, int> ans = bs(s, e);
        cout << ans.first << '\n';          // last bad = last feasible = answer
    }
    return 0;
}

Complexity

Sorting: O(n log n). One check: O(n). Binary search: O(log R) where R = max position − min position. Total: O(n log n + n log R).

🥧 Full Worked Example — PIE

spoj.com/problems/PIE  ·  vjudge.net/problem/SPOJ-PIE

Maximize slice area Double binary search floor(area/x) per pie

Problem story

You have N circular pies with known radii. You want to cut them all into pieces of the same area x so that there are enough pieces for F+1 people. Each person gets exactly one piece; you cannot combine partial pieces from different pies. Maximize the area x of each piece.

Abstraction

For a candidate slice area x, pie i contributes floor(area_i / x) full pieces. If the total number of pieces across all pies is ≥ F+1, the candidate x is feasible.

Why this fits double binary search

Feasibility is monotone: if slice area x is feasible, any smaller x gives at least as many pieces (so it is also feasible). As x increases, feasibility can only decrease. The answer is the maximum feasible x — exactly what the double master template finds, with s = current maximum feasible area.

Full C++ Code click to expand/collapse 
#include <bits/stdc++.h>
using namespace std;

const double PI = acos(-1.0);
const double eps = 1e-15;

int n, f;
vector<double> area;

bool ok(double x) {
    int pieces = 0;
    for (double a : area)
        pieces += (int)(a / x);
    return pieces >= f + 1;
}

double bs(double s, double e) {
    for (double sz = (e - s) * 0.5; sz > eps; sz *= 0.5)
        if (ok(s + sz))
            s += sz;
    return s;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);
    int tc;
    cin >> tc;
    while (tc--) {
        cin >> n >> f;
        area.resize(n);
        double mx = 0;
        for (int i = 0; i < n; i++) {
            double r; cin >> r;
            area[i] = PI * r * r;
            mx = max(mx, area[i]);
        }
        double ans = bs(0.0, mx);
        cout << fixed << setprecision(4) << ans << '\n';
    }
    return 0;
}

Why floor(area/x) is the right count

Each piece must have area exactly x. A pie with area A can yield exactly ⌊A/x⌋ such pieces (we cannot combine partial pieces across pies). The floor is correct because we only count full pieces. Summing across all pies gives the total number of pieces available.

Complexity

One check: O(N). Double binary search: ~50 iterations. Total: O(N · log((max_area) / eps)) per test case.

🔄 Transformed Binary Search

Some optimization problems do not look like binary search at all. The predicate is not obviously monotone. The key insight is that an algebraic transformation can reveal the hidden monotone structure.

In plain terms: instead of maximizing the ratio directly, we guess a candidate value X and rearrange the inequality algebraically into a yes/no feasibility check (“does a solution beating X exist?”). That check happens to be monotone — as X rises, the answer can only flip from yes to no, never back — which is exactly why binary search can home in on the optimum.

The core idea — in plain language

Sometimes you want to optimize a ratio: "find the cycle with the maximum average fun per unit cost." You cannot binary-search the ratio directly because checking "does there exist a cycle with exactly ratio R?" is hard.

But you can binary-search a threshold: "does there exist a cycle with average greater than x?" That is a yes/no feasibility question. If the optimal ratio is R*, then:

  • For any x < R*, the answer is yes (the best cycle beats x).
  • For any x > R*, the answer is no (no cycle can beat x).

This means the feasibility check is monotone in x — binary search applies. The trick is turning "does a cycle with average > x exist?" into an algebraic check that standard algorithms can solve.

The ratio / average family

Problems asking for "maximize average," "maximize ratio," "maximize fun per cost" all follow this pattern:

  1. Guess the optimal value x
  2. Ask: "does there exist a solution with average > x?"
  3. Multiply through to eliminate the division: total_fun − x · total_cost > 0
  4. Now the question is purely additive — amenable to standard graph/DP algorithms
  5. This check is monotone in x: if it works for x, it fails for larger x. Why? Because if a solution beats threshold x, that same solution also beats any smaller y < x — so as x increases, feasibility can only decrease from true to false, never flip back.
  6. Binary search over x

The core algebraic identity

For any substructure C with total value F(C) and total cost W(C):

F(C)/W(C) > x  ⟺  F(C) − x·W(C) > 0

This is the transformation that makes everything work. Once we multiply, the check becomes "does a substructure with positive transformed total exist?" — a standard feasibility question.

Common confusion — not every optimization can be binary-searched

The transformation only works if the resulting feasibility check is monotone in x. Always verify: as x increases, does the answer to "is there a solution beating x?" strictly decrease from true to false? If the answer can flip back and forth, the transformation has not created a valid monotone predicate, and binary search is not applicable.

🗺 Full Worked Example — Sightseeing Cows

vjudge.net/problem/POJ-3621

Maximize cycle average (fun/cost) Transformed binary search SPFA positive cycle detection

Problem story

A directed graph where each node has a "fun" value and each edge has a "cost." Find the cycle that maximizes the ratio: (total fun on the cycle) / (total cost on the cycle).

Why this fits transformed binary search

We cannot optimize the ratio directly. Instead we binary-search the answer x and ask: "does there exist a cycle with average fun/cost > x?" By the algebraic identity above, this is equivalent to asking whether the graph contains a positive-weight cycle after transforming each edge: w' = fun[v] − x · cost(edge). SPFA-style Bellman-Ford can detect positive cycles in O(V·E).

The full reduction

Step 1: Binary search on the answer x = optimal ratio.

Step 2: For a candidate x, ask: "is there a cycle with average > x?"

Step 3: Transform edge weight to w' = fun[v] − x · cost(edge).

Step 4: The question becomes: "does there exist a positive-weight cycle in the transformed graph?" — a standard SPFA-detectable property.

Proof: SPFA detects positive cycles correctly click to expand

Lemma: Repeated relaxation beyond n steps implies a cycle

A simple path in an n-node graph has at most n−1 edges. If Bellman-Ford-style relaxation improves a node's value after a chain of ≥ n edges, some node must repeat — forming a cycle.

If the improvement persists: the cycle on the repeated path contributes positive transformed weight, because we could keep traversing it to improve indefinitely.

Conversely: if a positive-weight cycle exists, relaxing along it repeatedly gives unbounded improvement, so SPFA will detect a chain of length ≥ n before converging. □

Full C++ Code click to expand/collapse 
#include <bits/stdc++.h>
using namespace std;

struct Edge { int to, w; };

const double eps = 1e-15;

int n, m;
vector<int> funv;
vector<vector<Edge>> adj;

bool ok(double x) {
    vector<double> dist(n + 1, 0.0);
    vector<int> cnt(n + 1, 0);
    vector<int> inq(n + 1, 1);
    queue<int> q;
    for (int i = 1; i <= n; i++) q.push(i);

    while (!q.empty()) {
        int u = q.front(); q.pop();
        inq[u] = 0;
        for (auto e : adj[u]) {
            int v = e.to;
            double w = funv[v] - x * e.w;  // ← transformed edge weight
            if (dist[v] < dist[u] + w) {
                dist[v] = dist[u] + w;
                cnt[v] = cnt[u] + 1;
                if (cnt[v] >= n) return true;  // ← positive cycle detected
                if (!inq[v]) { inq[v] = 1; q.push(v); }
            }
        }
    }
    return false;
}

double bs(double s, double e) {
    for (double sz = (e - s) * 0.5; sz > eps; sz *= 0.5)
        if (ok(s + sz))
            s += sz;
    return s;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);
    cin >> n >> m;
    funv.assign(n + 1, 0);
    for (int i = 1; i <= n; i++) cin >> funv[i];
    adj.assign(n + 1, {});
    for (int i = 0; i < m; i++) {
        int u, v, w; cin >> u >> v >> w;
        adj[u].push_back({v, w});
    }
    double ans = bs(0.0, 1000.0);
    cout << fixed << setprecision(2) << ans << '\n';
    return 0;
}

Why all nodes are pushed initially

We want to find any positive cycle, not just cycles reachable from a specific start. Pushing all nodes initially allows SPFA to explore from everywhere simultaneously, finding any positive cycle in the transformed graph.

Complexity

One check: O(V·E). Double binary search: ~50 iterations. Total: O(V·E · log(max_ratio / eps)).

📈 Ternary Search Deep Dive

Ternary search is for finding the minimum (or maximum) of a unimodal function over a continuous domain. There is no monotone yes/no predicate — instead there is a numeric objective with one valley.

Binary search uses shape: monotone

The function is a step function: false until some point, then true forever. There is a boundary — one place where behavior changes. Binary search homes in on that boundary by asking yes/no questions.

F F F F | T T T T
         ↑
    this boundary

Ternary search uses shape: unimodal

The function goes down to one minimum, then up. There is no yes/no question — there is a numeric objective and we want its lowest point. Two interior probes tell us which outer third to discard.

high         high
     \       /
      \     /
       \   /
        min
   ← valley →

Why comparing two interior points works — plain language

Suppose the interval is [s, e] and we probe at a (left-third point) and b (right-third point), with a < b.

If f(a) > f(b): The function is higher at a than at b. If the minimum were somewhere to the left of a, the function would have to go from the minimum up to a's height, and then down again to b's height (since b is lower). But a unimodal function cannot go down twice. So the minimum cannot be to the left of a — we safely discard [s, a].

If f(a) ≤ f(b): By the same logic but mirrored, the minimum cannot be to the right of b — discard [b, e].

Each case removes one third of the interval while keeping the minimum inside the remaining two thirds.

⚠️ Common confusion: "The answer is real-valued, so use ternary search"

This is wrong. A real-valued answer is not a reason to use ternary search. The reason is always the shape of the objective:

  • Real-valued answer + monotone feasibility check → use double binary search (PIE, Ropes, Sightseeing Cows).
  • Real-valued answer + unimodal objective function → use ternary search (Weakness & Poorness, Meeting Place).

If you are unsure which, ask: "Can I write a monotone yes/no check for this?" If yes, binary search. If not, and the objective is genuinely unimodal, ternary search.

Core idea

At each step, choose two interior points a and b (at 1/3 and 2/3 of the current interval). Compare f(a) and f(b). Discard the outer third that cannot contain the minimum. Repeat.

Formal correctness

Proof: ternary search converges to the minimum click to expand

Lemma: Each step discards a third while keeping the minimizer

Let x* be the unique minimizer. Consider interval [s, s+sz] known to contain x*.

Set a = s + sz/3 and b = s + 2sz/3 with a < b.

Case f(a) > f(b): Suppose x* ≤ a. Then f is non-increasing on [s, a] (since x* is the minimum and f is unimodal). Also f(a) > f(b) means f increased from a to b, but b > a ≥ x* so f should be non-decreasing. Contradiction. Therefore x* > a, so [s, a] does not contain x*. We safely discard it and set s = a.

Case f(a) ≤ f(b): By symmetric argument, x* < b. We discard [b, s+sz] by shrinking sz to 2sz/3.

In both cases the new interval has size 2sz/3 and still contains x*. □

Convergence

Each step multiplies the interval size by 2/3. After k steps, size = (e−s)·(2/3)^k. This goes to 0, so the interval converges to {x*}. □

💪 Full Worked Example — Weakness and Poorness (CF 578C)

codeforces.com/contest/578/problem/C

Ternary search on shift parameter Minimize worst transformed subarray imbalance f(x) is unimodal

Problem story

The problem involves finding a real parameter x that minimizes a measure of "imbalance" of the input array. The measure is computed by subtracting x from each element, then finding the worst absolute deviation of prefix sums from their running extremes. The key insight is that this measure, as a function of x, has exactly one valley — making ternary search the right tool.

Why binary search does NOT work here

There is no monotone yes/no question. We are not asking "is a threshold achievable?" We are finding where a continuous function is minimized. The function f(x) decreases then increases — not monotone in either direction. Ternary search is the right choice because the objective is unimodal.

Role of the parameter x

We choose a real shift x and transform each array element to b[i] = a[i] − x. The function f(x) measures the worst absolute deviation of transformed subarray sums from zero — a measure of how "unbalanced" the array is under shift x.

Why prefix sums

Any subarray sum of b[] is a difference of its prefix sums. So to find the most extreme transformed subarray sum, we track: current prefix sum, maximum prefix sum seen so far, and minimum prefix sum seen so far. Their differences give the worst positive and negative transformed sums.

Full C++ Code click to expand/collapse 
#include <bits/stdc++.h>
using namespace std;

int n;
vector<double> a;

double f(double x) {
    double mx = 0, mn = 0, sum = 0, ans = 0;
    for (int i = 0; i < n; i++) {
        sum += a[i] - x;
        // sum - mx = current prefix minus max previous prefix = most negative subarray
        // sum - mn = current prefix minus min previous prefix = most positive subarray
        ans = max(ans, fabs(sum - mx));
        ans = max(ans, fabs(sum - mn));
        mx = max(mx, sum);
        mn = min(mn, sum);
    }
    return ans;
}

pair<double, double> ternarySearch(double s, double e) {
    double fs = f(s);
    for (double sz = e - s; sz > 1e-15; sz *= 2.0 / 3.0) {
        double aa = s + sz / 3.0, b = aa + sz / 3.0;
        double fa = f(aa), fb = f(b);
        if (fa > fb) {   // minimize: left point is higher, discard left third
            s = aa;
            fs = fa;
        }
    }
    return {s, fs};
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);
    cin >> n;
    a.resize(n);
    for (int i = 0; i < n; i++) cin >> a[i];
    pair<double, double> ans = ternarySearch(-20000.0, 20000.0);
    cout << fixed << setprecision(10) << ans.second << '\n';
    return 0;
}

Why f(x) is unimodal

When x is too small, transformed elements lean positive → worst subarray sums are large positive values → f(x) is large. When x is too large, they lean negative → large negative sums → f(x) is large. Between these extremes lies the balancing shift x* where the worst imbalance is minimized. This creates exactly one valley — the structure ternary search needs.

🎮 Decision / Recognition Guide

Use this guide in contest. Read the problem, identify the signal, pick the tool.

⚡ 5-Second Recognition Rules

  • Sorted array, find first position satisfying condition → Lower / upper bound
  • "Smallest x such that…" or "largest x such that…" → Integer binary search on answer
  • "Maximize the minimum" or "minimize the maximum" → Integer binary search on answer
  • "Minimum time / capacity / speed to achieve…" → Integer binary search on answer
  • Maximize average / ratio / score-per-cost → Double binary search + transformed check
  • One peak or one valley in a continuous function → Ternary search
  • No monotonicity, no unimodal shape → Not a search problem — think DP / greedy / graph

Quick reflex rules

🔴 Stop — these signal binary search

  • "smallest x such that…"
  • "largest x such that…"
  • "minimum time / capacity / cost to achieve…"
  • "maximize the minimum…"
  • "minimize the maximum…"
  • Sorted array queries (lower/upper bound)
  • Maximize ratio / average (after transformation)

  • "choose a parameter x to minimize/maximize a continuous function"
  • Geometric optimization (optimal position on a line/plane)

  • f(x) = convex or concave-shaped continuous expression

🔀 Variants & Extensions

Variant 1 — Minimize maximum (Allocate Books / Painter's Partition)

Story: Divide an array into k contiguous subarrays. Minimize the maximum subarray sum.

Abstraction: Binary search on maximum sum x. Check: can we partition into ≤ k parts each with sum ≤ x?

Why it fits: If maximum x is feasible, any larger x is also feasible → monotone. Answer = first feasible x = ans.second.

Predicate: can(x) = greedily assign elements to current partition, start new part when sum exceeds x

Complexity: O(N log(sum)).

Links: spoj.com/problems/BOOKS · LeetCode 1011

Full Code click to expand 
#include <bits/stdc++.h>
using namespace std;
int n, k;
vector<int> a;

bool can(long long x) {
    int parts = 1;
    long long cur = 0;
    for (int v : a) {
        if (v > x) return false;  // single element exceeds limit
        if (cur + v > x) { parts++; cur = v; }
        else cur += v;
    }
    return parts <= k;
}

bool ok(long long x) { return can(x); }

pair<long long,long long> bs(long long s, long long e) {
    while (e - s > 1) {
        long long m = s + ((e - s) >> 1);
        if (ok(m)) e = m;
        else s = m;
    }
    return {s, e};
}

int main() {
    cin >> n >> k;
    a.resize(n);
    long long sum = 0, mx = 0;
    for (int i = 0; i < n; i++) { cin >> a[i]; sum += a[i]; mx = max(mx, (long long)a[i]); }
    auto ans = bs(mx - 1, sum + 1);  // s = bad (too small), e = good (whole sum works)
    cout << ans.second << '\n';  // first good = minimum feasible max
}

Story: Drop k tests from a series. Maximize the average of the remaining tests.

Why it fits: Transformed binary search — guess ratio x, check if sum(score_i − x) over the best (n−k) tests is positive.

📋 Final Summary — 7 Core Truths

Truth 1

Binary search is boundary finding, not middle-taking. The middle is just the mechanism. The goal is to locate where a monotone predicate transitions from false to true.

Truth 2

The invariant is sacred. s = last bad, e = first good. Everything else follows from maintaining this. Violate it and the proof collapses.

Truth 3

The hard part is always the predicate. Once ok(x) is designed correctly and proven monotone, the loop is trivial. Invest your thinking in the check function, not the loop.

Truth 4

Sorted-array queries are the same as answer-space search. Different domain (indices vs. answer values), identical structure. One framework handles both.

Truth 5

Transformed binary search reveals hidden structure. When the predicate isn't obvious, multiply through: F(C)/W(C) > x ↔ F(C) − x·W(C) > 0. This unlocks the class of ratio/average optimization problems.

Truth 6

Step-halving is safer than while(e−s>eps). Floating-point arithmetic can stall midpoint computation. The step-halving loop avoids this entirely with guaranteed convergence.

Truth 7

Ternary search and binary search live in different worlds. Binary: monotone yes/no. Ternary: unimodal objective. Confusing them causes incorrect discards and wrong answers. Always identify the structure first, then pick the tool.
