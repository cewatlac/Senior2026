0. What Changes in “Advanced” DP? ORIENTATION

The core DP law is unchanged. Only the state became richer.

If you finished the first DP tutorial, you already know the four pillars: define a state, derive a transition, choose a fill order, and prove that the state is sufficient for the future. You also know the canonical families: 1-D linear DP, LIS, LCS, edit distance, 2-D grid DP, and the 0/1 knapsack.

A learner hitting this sequel often expects a different theorem. There is none. What changes in advanced DP is almost always one of four things. The state became richer, meaning it now encodes a subset, a tree structure, a connectivity pattern on a boundary, or a digit prefix rather than just a single index or pair of indices. The transition became more structured, meaning instead of “look at the previous cell,” you now merge children's tables, unify two bitmask sub-states, or relax along graph edges. The answer type changed, from just “best value” to also counting, constructing, or optimizing under exact constraints. Or the problem object is no longer a line or table but a tree, a graph, a set, or a sweeping frontier.

① Define state

Capture everything the future needs. If two histories with the same state value can have different futures, the state is too small.

② Derive transitions

Show how each state assembles from smaller ones. Story: “the last move was…” or “the first cut was…”

③ Fill in dependency order

Every state is computed only after all the states it needs. Ascending index, by interval length, by mask weight, by topological order.

④ Prove future-sufficiency

If two worlds have the same state value, their best achievable futures are identical. That one sentence is the whole proof foundation.

The ten families in this tutorial are just the recurring answers people discovered across decades for the state-design question. Knapsack answers it with resource dimensions. Interval DP answers it with a range. Subset DP answers it with a used-set. Frontier DP answers it with the boundary between past and future. Every section below is a different answer to the same question.

💡

Study contract: Do not try to memorise all variants. Instead, after each section, ask yourself: what was the decisive new thing the future needed to know that was not captured by the simpler state? That delta is the lesson.

1. Advanced Knapsack Family KNAPSACK

Secret: almost every knapsack variant is “choose a subset or multiset under resource constraints.”
The variant is in the reuse rule and the legality check.

The first DP tutorial established the foundational 0/1 knapsack: you have n items, each with weight and value, you have a capacity budget, and you want to maximize total value subject to total weight ≤ capacity. Every item is either taken once or skipped.

Advanced knapsack changes exactly one question: how many times can I use each item? Or it adds a new constraint: what additional rules govern which items I can select together? The underlying state shape — which prefix of items have I considered, and how much resource remains — stays the same. Only the inner update rule and the number of state dimensions change.

The common state shapes are dp[cap], dp[i][cap], dp[i][cap1][cap2], or dp[u][cap] on a tree. The master question is always: when considering this option, what are the legal ways to use it, and what must remain true after the update?

1.2 Complete / Unbounded Knapsack LOOP DIRECTION

Problem shape

You are given a set of item types, each with a weight and value. You have a capacity budget. You may use each item type as many times as you want. Maximize total value within budget. This is called the complete knapsack or unbounded knapsack.

The canonical example is the coin-change minimization problem: given coin denominations, find the fewest coins whose sum equals a target. Each denomination can be used repeatedly.

Why the naive approach fails

A beginner might think: “it is just 0/1 knapsack but we can take more copies.” If you try to enumerate the number of copies of each item (0, 1, 2, …), you get a triple loop and O(n × C × max_copies) runtime — too slow for large C. The key insight is that the same recurrence as 0/1 knapsack works, but with a tiny change to the loop direction.

State definition

dp[c] = the minimum number of coins (or maximum value) achievable using exactly (or at most) capacity c. The state remembers only the current resource usage. Because items are reusable, we do not need to track which item we are currently considering — any item can always be added at any point.

Why the loop direction is everything

In 0/1 knapsack you scan capacity descending so that when you update dp[c] from dp[c-w], that source value still belongs to the world before the current item was used. That prevents using the item twice. Complete knapsack wants the opposite: when you compute dp[c] from dp[c-w], you want that source to already reflect using the current item, because the item is reusable. Scanning ascending achieves exactly that.

0/1 Knapsack — descending (prevents reuse)
for (auto& [w, v] : items) {
    // descend: dp[c-w] is BEFORE item i
    for (int c = C; c >= w; c--)
        dp[c] = max(dp[c], dp[c-w] + v);
}
Complete Knapsack — ascending (allows reuse)
for (auto& [w, v] : items) {
    // ascend: dp[c-w] ALREADY includes item i
    for (int c = w; c <= C; c++)
        dp[c] = min(dp[c], dp[c-w] + 1);
}

Base case and fill order

dp[0] = 0 (zero coins needed to make sum zero). All other entries start at ∞ (unreachable). Fill order: outer loop iterates over item types; inner loop iterates over capacity ascending from w to C. Because we ascend, when we update dp[c], the value at dp[c-w] is the already-updated value that reflects using this coin type — so it can represent using this coin zero, one, two, or more times.

Micro-example — coins [1, 3, 4], target 7

After processing coin 3 with ascending scan: dp[3]=1 (one coin of 3), dp[6]=2 (two coins of 3), and since dp[3]=1 was already updated in this same pass, dp[6] correctly found two coins of denomination 3. With a descending scan, dp[3] would still be its old value when computing dp[6], preventing the reuse.

✅

Invariant: After processing item i, dp[c] equals the best answer using items 1..i where item i may be used any number of times. Induct on capacity from small to large: at capacity c, the transition either (a) does not use the current item, or (b) uses one more copy at the end, reading from dp[c-w] which was already updated in this same pass — meaning it already reflects unlimited reuse.

⚠

Classic pitfall: Using descending loop by habit silently turns complete knapsack back into 0/1 knapsack. The bug produces wrong answers for cases where a denomination should be used more than once, and it is extremely hard to catch by inspection alone.

💡

Recognition rule: When you see “unlimited uses” or “each item may be taken any number of times” — that is complete knapsack. Change the inner loop to ascending.

1.3 Bounded & 1.4 Mixed Knapsack BINARY SPLIT

Problem shape

Bounded knapsack gives each item type a count limit cnt[i]. You may take 0, 1, 2, …, or exactly cnt[i] copies of item i. The classic example: you have specific coin inventories (12 coins of denomination 5, 7 coins of denomination 10, etc.) and want to make a target sum using at most the given quantities.

Why naive enumeration is too slow

The straightforward approach enumerates the number of copies of each item: dp[c] = min over k in [0..cnt[i]] of dp[c - k*w[i]] + k. This is O(n × C × max_count), which is too slow when counts are large (say, each count up to 105).

The binary-splitting trick

Observe that any integer from 0 to 13 can be written as a sum of distinct values chosen from the set {1, 2, 4, 6}: for example, 0–1–2–3 = 2+1–4–5 = 4+1–6–7 = 4+2+1–8 = 4+2+1? No — let us be careful. The splitting rule is: take bundle sizes 1, 2, 4, … doubling until the remaining count is exhausted. For count 13: bundles 1, 2, 4, 6 (since 1+2+4+6 = 13). Every integer 0..13 is a subset-sum of these four values because they cover all remainders by the greedy binary representation.

The key idea: instead of “take 0 to 13 copies of item i,” we create bundle items of sizes 1×, 2×, 4×, 6× of item i, each a normal 0/1 item. Then run standard 0/1 knapsack on all bundles (with descending loop). The total number of bundles is O(∑ log cnti), giving runtime O(C × ∑ log cnti).

// items stored as vector of {weight, value, count}
int bounded_knapsack(vector<tuple<int,int,int>>& items, int C) {
    vector<pair<int,int>> bundles;
    for (auto& [w, v, cnt] : items) {
        int k = 1, rem = cnt;
        while (rem > 0) {
            int take = min(k, rem);
            bundles.push_back({w * take, v * take});
            rem -= take;
            k *= 2;   // double the bundle size each iteration
        }
    }
    // now standard 0/1 knapsack on bundles
    vector<int> dp(C + 1, 0);
    for (auto& [bw, bv] : bundles)
        for (int c = C; c >= bw; c--)   // DESCENDING — bundles are 0/1
            dp[c] = max(dp[c], dp[c - bw] + bv);
    return dp[C];
}

Recognition and pitfall

💡

Recognition rule: When you see “each item has a count limit” and the counts are large — think binary splitting. The bundles are O(log cnt) per item, keeping the overall runtime manageable.

🔥

Pitfall: After binary splitting, you must use the descending loop because bundles are now 0/1 items. Switching to ascending would allow taking a bundle more than once — which defeats the entire purpose of bounding. References: POJ 1742 Coins, HDU 2191.

1.4 Mixed Knapsack

When some items are 0/1, some are complete (unlimited), and some are bounded, there is no new DP family. Just process each item using its correct update rule. The global dp state stays the same; only the local inner loop style changes per item type. This is really just an implementation detail, not a new algorithm.

1.5 Group & 1.6 Multi-Constraint Knapsack STATE ENRICHMENT

Group / Multiple-Choice Knapsack: problem shape

Items are partitioned into groups. From each group you may choose at most one item (or exactly one, depending on the variant). Examples: choose one course from each mandatory course category; choose one weapon per equipment slot; choose one person from each department for a team.

Why in-place updates fail

If you update the dp table in-place while iterating items within a group, a later item in the same group can chain off an update made by an earlier item in the same group. That effectively takes two items from one group — violating the “at most one” rule. The fix is to snapshot the table before processing each group, and during the group, always read from the snapshot.

for (auto& g : groups) {
    vector<int> old = dp;    // snapshot BEFORE this group
    for (int c = C; c >= 0; c--)
        for (auto& [w, v] : g)
            if (c >= w)
                dp[c] = max(dp[c], old[c - w] + v);
    // dp[c] is now the best using groups so far, with at most one
    // item chosen from this group, reading only the pre-group state
}
✅

Invariant: After finishing group g, dp[c] reflects legal selections from groups 1..g with the per-group constraint respected. The snapshot ensures no two updates within the same group chain off each other.

⚠

Common pitfall: Updating in-place without the snapshot. Two items from the same group can silently be combined. This is one of the most frequent knapsack bugs in contests because it produces plausible-looking but slightly wrong answers.

Multi-Constraint Knapsack: problem shape

Now one capacity is not enough. You may have weight, volume, time, and budget — all simultaneously. Each item consumes multiple resource types. The state gains one dimension per constraint: dp[i][cap1][cap2]. The transition is the same take/skip logic, but all resources decrease together when an item is taken. Every extra constraint multiplies the state space by the corresponding capacity bound.

dp[i][c1][c2] = max(dp[i-1][c1][c2], dp[i-1][c1-w1[i]][c2-w2[i]] + v[i])
💡

Recognition rule: When you see “weight AND volume constraint,” or “time AND cost constraint,” think multi-constraint knapsack. Add one state dimension per resource, but verify that all capacity bounds are small enough that the product of dimensions is manageable.

1.7 Exact-Fill & 1.8 Value-Based Knapsack INITIALIZATION

Exact-Fill Knapsack: why initialization matters

Standard knapsack initializes dp[0] = 0 and all other entries to 0 as well (for maximization) or ∞ (for minimization). When the problem says “reach exactly sum S” or “spend the entire budget exactly,” many capacity values are genuinely impossible. That is different from “achievable with zero benefit.” If you initialize to 0, you cannot tell which states are truly reachable.

❌ Wrong (all zeros)
vector<int> dp(S + 1, 0);
// dp[c]=0 could mean
// "can reach c with value 0"
// OR "impossible to reach c"
// — indistinguishable!
✅ Correct
// For maximization:
const int NEG_INF = INT_MIN / 2;
vector<int> dp(S + 1, NEG_INF);
dp[0] = 0;  // only base is valid

// For minimization:
const int INF = INT_MAX / 2;
vector<int> dp(S + 1, INF);
dp[0] = 0;

After the DP, check whether dp[S] is still NEG_INF / INF. If it is, the target is unreachable.

🔥

Common pitfall: Initializing all entries to 0 and then reading dp[S] without checking whether it was ever updated by a real path. This silently treats impossible targets as achievable with value 0.

Value-Based (Axis-Swapped) Knapsack

Sometimes capacity is huge (say 109) but item values are small (say each value ≤ 103 and there are at most 103 items, so total value ≤ 106). A standard capacity-indexed table would have 109 cells — completely infeasible.

The fix is to swap the axis: instead of asking “what is the best value within capacity C?” ask “what is the minimum weight needed to achieve exactly value v?” The answer to the original problem is then the largest v such that dp[v] ≤ W.

dp[v] = min weight to achieve exactly value v
↓
Answer = max v such that dp[v] ≤ W
💡

Recognition rule: When capacity is around 109 but values are around 103, think value-DP immediately. The canonical example is AtCoder DP E (Knapsack 2). The clue is: “the capacity is huge but the answer value is small.”

1.10 Tree Knapsack & Recognition Rules TREE MERGE

Problem shape: what is a “tree” in this context?

A tree is a connected graph with no cycles. It has exactly n-1 edges for n nodes, which means there is exactly one path between any two nodes. When we “root” a tree at some node r, every other node has a unique parent, and the subtree rooted at node u means “u and all of its descendants.”

In tree knapsack, you have a resource budget M and each node has a weight and value. The selection constraint is: if you select a node, you must also select its parent (dependency constraint). This makes simple 1-D knapsack invalid because selections are no longer independent.

Why the selection dependency matters

In ordinary knapsack, taking item 5 is independent of taking item 2. In tree knapsack, if node 5 is a child of node 2, you cannot take 5 without 2. This hierarchical dependency requires the DP to be performed on the tree structure itself — specifically, bottom-up (children before parents).

State definition

dp[u][k] = the maximum value achievable by selecting some nodes from the subtree rooted at u, using exactly k units of capacity from that subtree.

Transition: the merge convolution

Process children one at a time. When merging the partial table for node u with the DP table of a new child v, you perform a knapsack-style convolution: for each way to use ku capacity in what we have so far at u, and each way to use kv capacity in v’s subtree, we combine them by spending ku + kv total.

// dp[u][k] = best value using exactly k capacity in subtree of u
void dfs(int u, int par) {
    dp[u].assign(M + 1, NEG_INF);
    dp[u][0] = 0;   // base: no capacity used in subtree of u
    for (int v : adj[u]) {
        if (v == par) continue;
        dfs(v, u);
        // Merge dp[u] with dp[v] — knapsack convolution
        vector<int> new_dp(M + 1, NEG_INF);
        for (int ku = 0; ku <= M; ku++) {
            if (dp[u][ku] == NEG_INF) continue;
            for (int kv = 0; ku + kv <= M; kv++) {
                if (dp[v][kv] == NEG_INF) continue;
                new_dp[ku+kv] = max(new_dp[ku+kv],
                                    dp[u][ku] + dp[v][kv]);
            }
        }
        dp[u] = new_dp;
    }
    // Optionally: add value/weight of selecting node u itself
    // (only valid if the selection dependency rule allows it)
}

Fill order and complexity

Fill order is post-order DFS: children are processed before parents. The naive merge is O(n M2) but with careful tracking of subtree sizes it becomes O(n M) in total.

💡

Recognition rule: When you see “select nodes from a tree with a budget, subject to parent-child dependency” — think tree knapsack. References: HDU 1561, classic “course selection on prerequisite tree” problems.

2. Advanced Sequence / String DP STRING

The previous lecture established three roots: LIS (Longest Increasing Subsequence), LCS (Longest Common Subsequence), and edit distance. Advanced sequence DP grows from those roots into four subworlds: contiguous-match world, alignment/counting world, palindrome world, and LIS-family world. The skills transfer directly; only the state and transition change.

2.1 Longest Common Substring (not Subsequence)

This confusion must die before you go further. A subsequence can skip characters; a substring must be contiguous. That single word changes the recurrence completely. For LCS (subsequence), a mismatch allows you to take the best of skipping one character from either string. For LCStr (substring), a mismatch forces the value to zero because contiguity is broken — you must start over.

if a[i] == b[j]: dp[i][j] = dp[i-1][j-1] + 1
else: dp[i][j] = 0    (contiguity broken — must restart)

Answer = max over all (i,j) of dp[i][j]. Rolling O(m) memory is easy since each row depends only on the previous one.

🔥

Common pitfall: Applying the LCS recurrence (which allows skip) to a problem that requires a substring (which requires contiguity). The answers can differ drastically. Always ask: “must the matched characters form a contiguous block in both strings, or may they skip?”

2.2 Shortest Common Supersequence & 2.3 LCIS DUAL CONSTRAINT

Shortest Common Supersequence (SCS)

Find the shortest string containing both A and B as subsequences. LCS tells you what is shared. Every character in the LCS is counted once (shared). Every character not in the LCS must appear uniquely in the supersequence. So the SCS length is |A| + |B| − LCS(A,B).

|SCS| = |A| + |B| − LCS(A, B)

To reconstruct the actual SCS: walk the LCS table and output each character once when matched diagonally, or output the skipped character from the larger predecessor direction.

LCIS — Longest Common Increasing Subsequence

① Problem shape

Given two integer arrays A and B, find the longest sequence that is simultaneously (1) a subsequence of A, (2) a subsequence of B, and (3) strictly increasing. This is the Longest Common Increasing Subsequence (LCIS).

② Why you cannot just glue LCS and LIS together

A natural first idea is: “find the LCS of A and B, then find the LIS of that LCS.” This fails because the longest common subsequence may not contain the longest increasing common subsequence. Example: A = [2, 3, 1, 5], B = [2, 1, 3, 5]. The LCS is [2, 3, 5] (length 3) or [2, 1, 5] (length 3). [2, 3, 5] is itself increasing, so the LCIS is length 3. But if A = [3, 2, 5], B = [3, 5, 2], the LCS is [3, 5] or [3, 2], and only [3, 5] is increasing. The LCS algorithm does not favour increasing subsequences — it may return a non-increasing one, and taking the LIS of a specific LCS misses other common subsequences that are longer and increasing. You need to enforce all three constraints simultaneously during the DP, not in two separate passes.

③ State definition

dp[j] = the length of the longest common increasing subsequence that ends at B[j], using elements from A[0..i−1] seen so far. The outer loop scans A from left to right. For each element A[i], the inner loop scans B from left to right and maintains a running variable best.

Why track “ends at B[j]”? Because the increasing constraint requires knowing the last value of the subsequence built so far. By fixing the ending element, the future knows exactly which values can extend the subsequence (any value strictly greater).

④ The role of the running “best” variable

As the inner loop scans B left to right for a fixed A[i], the variable best accumulates the maximum dp[j] among all positions j where B[j] < A[i]. This means: “best LCIS length ending at some element that is strictly less than A[i].” When we encounter B[j] == A[i], we can extend any of those subsequences by appending A[i] (which equals B[j]), giving length best + 1. This avoids an O(n) inner scan at every match, keeping the whole algorithm O(nm).

⑤ Transition logic

For each pair (i, j) in the double loop: if B[j] < A[i], then B[j] could be the element just before A[i] in an increasing subsequence, so we update best = max(best, dp[j]). If B[j] == A[i], then element A[i] appears in both arrays, and we can extend the best increasing subsequence ending before this value: dp[j] = max(dp[j], best + 1). If B[j] > A[i], we skip — extending would violate the increasing constraint.

│

Micro-example: A = [2, 3, 1], B = [1, 2, 3]. Processing A[0]=2: scan B. B[0]=1 < 2, so best = max(0, dp[0]) = 0. B[1]=2 == 2 (match!), dp[1] = max(dp[1], 0+1) = 1 (LCIS [2]). B[2]=3 > 2, skip. Processing A[1]=3: reset best=0. B[0]=1 < 3, best = max(0, dp[0]) = 0. B[1]=2 < 3, best = max(0, dp[1]) = 1. B[2]=3 == 3, dp[2] = max(dp[2], 1+1) = 2 (LCIS [2, 3]). Final answer = max(dp[0..2]) = 2.

Fill order and base case

Base case: all dp[j] start at 0 (no common increasing subsequence found yet). Fill order: outer loop over A from left to right (i = 0, 1, …); inner loop over B from left to right (j = 0, 1, …). The variable best resets to 0 at the start of each outer iteration (each new A[i]).

// dp[j] = LCIS length ending at B[j]
vector<int> dp(B.size(), 0);
for (int i = 0; i < (int)A.size(); i++) {
    int best = 0;  // best LCIS ending at some B[j] < A[i]
    for (int j = 0; j < (int)B.size(); j++) {
        if (B[j] < A[i])
            best = max(best, dp[j]);    // A[i] can extend this
        if (B[j] == A[i])
            dp[j] = max(dp[j], best + 1);  // match: attach A[i]
    }
}
int ans = *max_element(dp.begin(), dp.end());
💡

Recognition rule: When you see “common subsequence of two arrays that must also be increasing (or decreasing)” — think LCIS. The signature is two simultaneous constraints: commonality across two sequences AND monotonicity within the result.

⚠

Common pitfall: Forgetting to reset best to 0 at the start of each outer-loop iteration. If you carry best across different A[i] values, you may extend with an element that was relevant to a previous A[i] but is invalid for the current one.

2.4 Distinct Subsequences & 2.5–2.6 Matching COUNTING

Distinct Subsequences

Count how many ways T appears as a subsequence of S. At each position i in S, you make a binary decision for each character: either ignore it (it contributes nothing to matching T) or use it to match the current character of T (only possible if they are equal). These two roles are mutually exclusive — each character of S either participates in a particular match or it does not — making the partition disjoint and the recurrence valid.

dp[i][j] = dp[i-1][j] + (S[i]==T[j] ? dp[i-1][j-1] : 0)
🔥

Pitfall: These counts grow enormously. Always apply modulo as you compute, never after the fact, or you will silently overflow.

Wildcard Matching

① Problem shape

Given a text string s and a pattern string p, determine whether p matches s entirely. The pattern may contain normal characters, ? (matches any single character), and * (matches any sequence of zero or more characters, including the empty string). This is the “wildcard” or “glob” matching used in file systems.

② State definition — what does dp[i][j] mean?

dp[i][j] = true if and only if the first i characters of s (i.e. s[0..i−1]) can be matched by the first j characters of p (i.e. p[0..j−1]). In plain language: “can this prefix of the text be fully consumed by this prefix of the pattern?”

Why we need both i and j: The answer depends on how far we have progressed into the text AND how far we have progressed into the pattern. Two different prefixes of the pattern can behave differently even on the same text prefix.

③ Base cases

dp[0][0] = true — empty text matches empty pattern. dp[i][0] = false for i > 0 — non-empty text cannot match an empty pattern. dp[0][j] = true only if p[0..j−1] consists entirely of * characters (each star matches zero characters, consuming nothing from the text). As soon as a non-star character appears in the pattern, dp[0][j] is false for that j and all subsequent j (the pattern demands characters but the text is empty).

④ Why * is different from ?

? is simple: it matches exactly one character, always. It acts like a single-character wildcard. * is far more powerful: it can match zero, one, two, … or any number of characters. This variable-length matching is what makes the problem interesting. The DP must explore both possibilities for *: match nothing further, or absorb one more character from the text.

⑤ Transition logic

Case 1: p[j-1] is a normal character. Then dp[i][j] = dp[i-1][j-1] if s[i-1] == p[j-1], else false. The character must match exactly.

Case 2: p[j-1] == '?'. Then dp[i][j] = dp[i-1][j-1]. The ? matches whatever s[i-1] is. One text character consumed, one pattern character consumed.

Case 3: p[j-1] == '*'. The star can match empty (use zero more characters): that means the star is done, so dp[i][j] inherits from dp[i][j-1] (advance the pattern past the star, but do not consume any text). Or the star absorbs one more text character: dp[i][j] inherits from dp[i-1][j] (consume one text character, but keep the star active in the pattern so it can absorb more).

if p[j-1]=='*': dp[i][j] = dp[i][j-1] || dp[i-1][j]
if p[j-1]=='?': dp[i][j] = dp[i-1][j-1]
if p[j-1]==c:   dp[i][j] = dp[i-1][j-1] && s[i-1]==c
│

Micro-example: s = "ab", p = "a*". dp[0][0]=T. dp[0][1]=F (pattern "a" vs empty text). dp[0][2]=F? No — p[1]='*', so dp[0][2] = dp[0][1] (star matches empty) = F. dp[1][1]: p[0]='a', s[0]='a', match → dp[1][1] = dp[0][0] = T. dp[1][2]: p[1]='*', dp[1][2] = dp[1][1] (star matches empty) || dp[0][2] (star absorbs 'a') = T||F = T. dp[2][2]: p[1]='*', dp[2][2] = dp[2][1] (F) || dp[1][2] (T) = T. So "ab" matches "a*". Correct!

Fill order

Row by row (i from 0 to |s|), column by column (j from 0 to |p|). Every cell only reads from dp[i-1][j-1], dp[i-1][j], or dp[i][j-1] — all of which are already computed in this order.

💡

Recognition rule: When you see “pattern matching with ? for single character and * for any sequence” — think wildcard matching DP. The key is that * generates two choices: match-empty and absorb-one-more.

⚠

Common pitfall: Initialising dp[0][j] incorrectly. Many students set dp[0][j]=false for all j>0, but a pattern like "***" does match the empty string. You must propagate: dp[0][j] = dp[0][j−1] when p[j−1]=='*'.

Regex Matching (. and *)

① How regex * differs from wildcard *

In wildcard matching, * is a standalone token that matches any sequence of characters by itself. In regex matching, * is a modifier that always refers to the character immediately before it. The combination x* means “zero or more copies of x.” The . character matches any single character (like ? in wildcard matching). So .* means “zero or more copies of any character” — equivalent to wildcard *. But a* means “zero or more copies of the letter a only.” Confusing the two * semantics is one of the most common interview mistakes.

② State definition

dp[i][j] = true if and only if s[0..i−1] matches p[0..j−1]. Same shape as wildcard matching, but the transitions for * are fundamentally different because * modifies the previous token, not itself.

③ Transition logic

Case 1: p[j-1] is a normal character or '.'. Then dp[i][j] = dp[i-1][j-1] when the characters match (or p[j-1] is '.'). One text character consumed, one pattern character consumed. Identical to wildcard’s non-star case.

Case 2: p[j-1] == '*'. The star modifies p[j-2] (the atom). The pair p[j-2..j-1] means “zero or more of the atom.” Two sub-cases:

• Use zero copies of the atom: the entire two-character pair x* is skipped. This gives dp[i][j] |= dp[i][j-2] (advance the pattern by 2, consuming nothing from the text).

• Use one or more copies: the current text character must match the atom (s[i-1] == p[j-2] or p[j-2] == '.'). If it does, we consume one text character and keep the star active: dp[i][j] |= dp[i-1][j].

These two cases together cover all possibilities: use zero copies, or use k copies (by repeating the “one more” case k times).

if p[j-1]=='*':
  dp[i][j] |= dp[i][j-2]   (zero copies of atom p[j-2])
  if match(s[i-1], p[j-2]): dp[i][j] |= dp[i-1][j]   (one more copy)
│

Micro-example: s = "aab", p = "c*a*b". Read the pattern: c* = zero or more c’s; a* = zero or more a’s; b = literal b. So the pattern says “some c’s, then some a’s, then b.” This should match "aab" (zero c’s, two a’s, one b). Key DP steps: dp[0][0]=T. dp[0][2]=dp[0][0]=T (c* uses zero copies). dp[0][4]=dp[0][2]=T (a* uses zero copies). dp[1][4]: p[3]='*', atom='a', s[0]='a' matches atom, so dp[1][4] |= dp[0][4] = T (one copy of a). dp[2][4]: atom='a', s[1]='a' matches, dp[2][4] |= dp[1][4] = T (second copy of a). dp[3][5]: p[4]='b', s[2]='b' matches, dp[3][5] = dp[2][4] = T. Match!

💡

Recognition rule: When you see “regex matching with . and *” — remember that * modifies the previous character. The key transition is: dp[i][j-2] (zero copies) or dp[i-1][j] (one more copy, if the atom matches). This is structurally different from wildcard matching.

⚠

Common pitfall: Treating regex * as a standalone token that matches any sequence (that is wildcard semantics). In regex, * always looks back at p[j-2]. If you index into the pattern at j-1 instead of j-2 when checking the atom, the entire recurrence is wrong.

2.7–2.10 Palindrome World INTERVAL ON STRING

Longest Palindromic Subsequence (LPS)

Pick characters in order from a string to form the longest palindromic subsequence. Why is an interval state [l,r] needed? Because whether the endpoints s[l] and s[r] can contribute to a palindrome depends on the interior characters. The optimal answer for the full string depends on answers for all substrings, not just prefixes — so this is naturally an interval DP on a single string.

State, transition, base case

dp[l][r] = the length of the longest palindromic subsequence of s[l..r]. Base case: dp[i][i] = 1 (every single character is a palindrome of length 1). Transition: if s[l] == s[r], both endpoints contribute, so dp[l][r] = dp[l+1][r-1] + 2. If they differ, at least one endpoint is useless, so dp[l][r] = max(dp[l+1][r], dp[l][r-1]). Fill order: by increasing interval length (length-2 before length-3, etc.).

if s[l]==s[r]: dp[l][r] = dp[l+1][r-1] + 2
else: dp[l][r] = max(dp[l+1][r], dp[l][r-1])

Palindrome Partitioning

Split a string into the minimum number of palindromic pieces. Precompute pal[l][r] = is s[l..r] a palindrome? (O(n2) using the same interval DP). Then use a prefix DP: dp[i] = minimum cuts to split s[0..i], trying every possible last palindromic piece.

dp[i] = min over j≤i where pal[j][i]: dp[j-1] + 1

Min Insertions to Make a Palindrome

Characters already in the LPS need no insertion (they are already balanced). Every character outside the LPS must be mirrored on the opposite side, costing one insertion each. So minimum insertions = n − LPS(s). Similarly, minimum deletions to make a palindrome = n − LPS(s). Knowing the LPS reduces both problems to one DP.

2.11–2.14 Subsequence Variants & Recognition VARIANTS

Bitonic Subsequence

A bitonic sequence first increases strictly, then decreases strictly (like a mountain). The strategy is to compute LIS ending at each index (going left to right) and LDS (Longest Decreasing Subsequence) starting at each index (going right to left), then combine: the peak at index i contributes lis_end[i] + lds_start[i] - 1 (subtract 1 because the peak is counted in both halves).

answer = max over all i: lis_end[i] + lds_start[i] − 1

Zigzag / Alternating Subsequence

Signs of consecutive differences must alternate (up, down, up, down… or down, up, down, up…). Maintain two states per index: best length of alternating subsequence ending here with the last step being “up” vs. “down.” A new element can extend an “up” run only if it is strictly larger than its predecessor, and vice versa.

Constrained LIS Variants

Examples include LIS where consecutive elements must differ by at most k, LIS with at most k direction changes, or LIS on pairs (the “Russian Dolls / Envelopes” problem). These remain sequence DP. The key lesson: do not memorise named variants. Instead ask — what defines a legal predecessor? What extra property must be tracked? Does dp[i] still suffice, or do I need dp[i][extra]?

3. Segmentation / Partition DP LAST CUT

Solve prefix 1..i. Let the last segment be the final choice.
That one sentence generates half the family.

Problem shape

A segmentation problem says: divide an array, string, or timeline into contiguous parts; each part has a cost or validity condition; optimize over all legal ways to split. This is different from interval DP: here we only care about prefixes and where the last segment begins, not about arbitrary ranges.

State, transition, fill order

The master state is dp[i] = best answer for the prefix ending at position i. The master transition looks at every possible last-segment starting point j: the segment j+1..i has some cost or validity, and the remaining prefix 1..j is solved recursively.

dp[i] = min/max over j<i: dp[j] + cost(j+1, i)

Fill order: ascending i from left to right. Every state dp[i] only depends on dp[j] for j < i, which have already been computed.

3.2 Weighted Interval Scheduling

① Problem shape

You have n jobs. Each job has a start time, an end time, and a profit (weight). Two jobs overlap if one starts before the other ends. You must choose a subset of non-overlapping jobs that maximizes total profit. This is the weighted interval scheduling problem.

② Why greedy fails

Without weights, the classic greedy “always pick the job that finishes earliest” is optimal. But with weights, that greedy can fail badly. Example: Job A runs 1–10 with profit 100, Job B runs 1–5 with profit 10, Job C runs 6–10 with profit 10. Greedy earliest-finish picks B (ends at 5) then C (ends at 10), total = 20. But taking A alone gives 100. Similarly, “always pick the highest-profit job” is also wrong because a high-profit job may block many smaller jobs whose combined profit exceeds it. We need DP to consider all possibilities.

③ Why sorting by finish time is the right order

Sort all jobs by their finish time. Now, when considering job i, all jobs that finish before job i starts are located earlier in the sorted order. This gives a clean prefix structure: the decision at job i only depends on jobs 1..i−1.

④ What is p(i)?

p(i) is the index of the last job that is compatible with job i — meaning the last job (in sorted order) whose finish time is ≤ the start time of job i. If job i starts at time 5, then p(i) is the largest index j < i such that finish[j] ≤ 5. We find p(i) using binary search on the sorted finish times, giving O(log n) per job.

If no job finishes before job i starts, then p(i) = 0 (a sentinel meaning “no compatible predecessor”).

⑤ State and transition

dp[i] = the maximum profit achievable using only jobs 1..i (after sorting by finish time). At job i, exactly two options exist: skip job i (profit = dp[i−1]), or take job i (profit = profit[i] + dp[p(i)], meaning we earn job i’s profit plus the best we can do among all jobs compatible with i).

dp[i] = max(dp[i-1],   profit[i] + dp[p(i)])
│

Micro-example: Three jobs sorted by finish time: Job 1 (1–3, profit 5), Job 2 (2–5, profit 6), Job 3 (4–7, profit 8). Compute p(i): p(1)=0 (no predecessor), p(2)=0 (no job finishes ≤ time 2 — Job 1 finishes at 3 > 2), p(3)=1 (Job 1 finishes at 3 ≤ 4). dp[0]=0. dp[1]=max(dp[0], 5+dp[0])=5. dp[2]=max(dp[1], 6+dp[0])=max(5,6)=6. dp[3]=max(dp[2], 8+dp[1])=max(6, 8+5)=13. Answer: 13 (take Jobs 1 and 3).

Fill order: ascending job index from 1 to n. Each state only looks at smaller indices. Total complexity: O(n log n) for sorting and binary searches, O(n) for the DP itself.

💡

Recognition rule: When you see “choose non-overlapping intervals/jobs with weights to maximize total value” — think weighted interval scheduling. The clues are: intervals with start/end times, a profit per interval, and a non-overlap constraint. Sort by finish time, precompute p(i) with binary search, and run the take-or-skip recurrence.

⚠

Common pitfall: Confusing the definition of p(i). It must be the last job whose finish time is ≤ the start time of job i, not the last job whose start time is ≤ the start of job i. The finish-to-start comparison is what guarantees non-overlap.

3.3 Partition into Exactly K Groups

Add a second state dimension: dp[i][k] = best way to split the first i elements into exactly k contiguous groups. The transition chooses where the k-th group starts.

dp[i][k] = min over j: dp[j][k-1] + cost(j+1, i)
🔥

Pitfall: “Exactly K” changes initialization. Impossible states must remain impossible (initialized to +INF for min, −INF for max). Using zero everywhere hides infeasibility and produces wrong answers silently.

3.5 Word Break / Segmentation DP

Can the string be split into dictionary words? The state is boolean: dp[i] = can the prefix of length i be segmented? Transition: try all j < i where dp[j] is true and the substring j+1..i is in the dictionary. This is the partition recurrence with a validity check instead of a numerical cost.

Cut-optimality argument (applies to almost every partition DP): Consider an optimal solution for prefix i. Look at its last segment. Suppose it starts after position j. Then everything before position j must itself be optimally solved — otherwise replacing it with a better prefix solution would improve the whole answer. That is the subproblem claim, and it justifies the recurrence.

Difference from Interval DP

Partition DP solves prefixes and chooses the last segment. Interval DP solves arbitrary ranges [l,r] and chooses a split, first operation, or last operation anywhere inside the range. They look similar but they are answering different structural questions. The fill order for partition DP is always left to right; for interval DP it is by increasing interval length.

4. Advanced Interval DP INTERVAL

Problem shape and why interval length fills work

Interval DP appears when the problem lives on a contiguous range [l,r] and the answer for [l,r] depends on how you split, merge, shrink, or remove things inside that range — and that dependency involves sub-intervals, not just prefixes. Every transition shrinks the interval in some way, so shorter intervals are computed before longer ones.

dp[l][r] ← depends on dp[l'][r'] where r'−l' < r−l

Fill order: outer loop over increasing interval length (length-1, length-2, …, length-n), inner loops over the left endpoint l. This guarantees every sub-interval is ready when needed.

4.2 Cutting Sticks

You have a stick and required cut positions. The cost of making a cut equals the current stick length. For a sub-stick between two already-fixed boundaries, choosing which cut to perform first splits the problem into independent left and right sub-sticks. The first cut pays the full current length once; after that, left and right halves are independent sub-problems.

dp[l][r] = min over k in (l,r): dp[l][k] + dp[k][r] + (r − l)

4.3 Stone Merging

Merge adjacent piles one by one. Each merge costs the sum of the merged range (precomputed with a prefix sum). The final merge of [l,r] is between two sub-intervals [l,k] and [k+1,r] for some split point k.

vector<int> prefix(n + 1, 0);
for (int i = 0; i < n; i++) prefix[i+1] = prefix[i] + piles[i];

for (int length = 2; length <= n; length++) {
    for (int l = 0; l + length - 1 < n; l++) {
        int r = l + length - 1;
        dp[l][r] = INF;
        int cost = prefix[r+1] - prefix[l];
        for (int k = l; k < r; k++)
            dp[l][r] = min(dp[l][r], dp[l][k] + dp[k+1][r] + cost);
    }
}

4.4 Burst Balloons — The Inversion Trick

You have a row of balloons with values. Bursting balloon i earns nums[left] × nums[i] × nums[right] where left and right are the current neighbors. The forward direction is messy because bursting changes the neighbors of every remaining balloon. The key inversion: instead of asking “which do I burst first?” ask “which balloon is burst last inside interval [l,r]?” When balloon k is burst last, its neighbors are the outer boundaries l and r, which are fixed throughout the entire computation of dp[l][r].

dp[l][r] = max over k in (l,r): dp[l][k] + nums[l]*nums[k]*nums[r] + dp[k][r]

Pad the array with 1s at both ends so boundary balloons always exist. This is one of the most elegant inversion tricks in all of DP. O(n3). LeetCode 312.

💡

Big lesson: A lot of advanced DP is won by asking the question in reverse order. When the forward direction produces messy, changing context, try asking: “what was the last operation inside this interval?” That question often fixes the context and unlocks the clean recurrence.

4.6 Remove Boxes & State Augmentation AUGMENT STATE

Some interval DP problems expose the most important advanced instinct of all: when dp[l][r] feels almost right but something from outside the interval also matters, add a third parameter to encode that external context.

Why dp[l][r] alone fails for Remove Boxes

In Remove Boxes (LeetCode 546), removing a group of k consecutive equal-colored boxes scores k2. The value of interval [l,r] depends on whether equal-colored boxes from outside [l,r] can later be attached to boxes at the left boundary, growing the group size. So the outside context matters, and dp[l][r] cannot capture it. Two different external situations for the same [l,r] interval can produce different optimal scores.

dp[l][r][k] = best score for interval [l,r] with k extra boxes equal to a[l] attached to the left

Now the missing context is encoded as the third dimension. This is called state augmentation: when an interval state is insufficient, identify precisely what external information is missing and add it as a new dimension.

💡

The augmentation instinct: If you write the recurrence for dp[l][r] and find yourself saying “but this depends on what happens outside…” — that sentence is your signal. The thing that depends on outside context must become a new state dimension.

5. Advanced Subset / Bitmask DP BITMASK

What is a bitmask? Building the idea from scratch

A bitmask is an integer whose binary representation encodes a set. If you have n elements numbered 0 to n−1, then the integer whose bit j is 1 means “element j is in the set.” For example, with n=4 and elements {0,1,2,3}: the integer 0b1010 = 10 represents the set {1, 3}; the integer 0b0101 = 5 represents {0, 2}; the integer 0b1111 = 15 represents all four elements.

Key bit operations: mask | (1<<j) adds element j to the set. mask & ~(1<<j) removes element j. (mask >> j) & 1 tests whether element j is in the set. __builtin_popcount(mask) counts how many elements are in the set.

Why bitmask DP? The core idea

When n is small (≤ 20 or so), the set of “elements already used” can be encoded as a bitmask integer. The DP state dp[mask] remembers the exact set of used elements. This turns the exponential brute-force search (there are 2n subsets) into a DP with 2n states, where each state is computed from smaller ones. That is still exponential in n, but with DP we avoid re-computing the same sub-state multiple times.

The fill order is by increasing number of bits set (popcount), because a mask with k bits set depends only on masks with k−1 bits set (one fewer element used).

5.2 Assignment DP: problem shape

Assign n jobs to n workers minimizing total cost. cost[i][j] = cost of worker i doing job j. Each job must be assigned to exactly one worker, and each worker does exactly one job.

Why the bitmask encodes the right information

Let dp[mask] = minimum cost to assign the jobs in mask to the first popcount(mask) workers. The key insight: because we always assign to the “next” worker (worker 0, then 1, then 2, etc.), we do not need to track which worker is current — that is always implied by popcount(mask). The state only needs to know which jobs have been assigned so far.

vector<int> dp(1 << n, INF);
dp[0] = 0;
for (int mask = 0; mask < (1 << n); mask++) {
    if (dp[mask] == INF) continue;
    int worker = __builtin_popcount(mask);  // next worker = #jobs already taken
    for (int job = 0; job < n; job++) {
        if (mask >> job & 1) continue;     // job already assigned
        int new_mask = mask | (1 << job);
        dp[new_mask] = min(dp[new_mask], dp[mask] + cost[worker][job]);
    }
}

Fill order and base case

Base case: dp[0] = 0 (no jobs assigned, zero cost). Fill order: ascending mask value. Since adding a bit always increases the mask value, we naturally process smaller masks before larger ones — guaranteeing all dependencies are ready.

✅

Why no worker dimension is needed: The current worker index is implied by popcount(mask). The future only needs to know which jobs are already taken; the worker ordering is enforced structurally. Adding a second dimension for the worker index would be redundant.

5.3 Hamiltonian Path DP (Travelling Salesman Skeleton)

Find the shortest path visiting every node exactly once. Let dp[mask][v] = minimum cost to visit exactly the nodes in mask, ending at node v. Transition: extend the path by visiting an unvisited node u adjacent to v.

dp[mask | (1<<u)][u] = min(dp[mask | (1<<u)][u], dp[mask][v] + dist[v][u])

Fill order: ascending popcount(mask). This is O(2n × n2), which is feasible for n up to about 20.

5.5 Steiner Tree DP

① Problem shape

You have a weighted graph with n total nodes, and a small subset of k special nodes called terminals (k ≤ 10 typically). Find the minimum total edge-weight subgraph that connects all terminals. This subgraph will be a tree (since adding cycles only wastes cost). It may route through non-terminal “Steiner points.”

② Why naive approaches fail

You might try: “run Dijkstra or MST.” MST connects all n nodes — that is too expensive and does not target the k terminals specifically. Dijkstra finds shortest paths between pairs but does not handle the multi-terminal constraint. Greedy approaches that grow a spanning tree from one terminal to the next do not find the globally optimal routing through Steiner points.

③ State definition

dp[v][S] = the minimum cost of any connected subgraph that (1) includes vertex v, and (2) has every terminal in set S connected within it. Here v is the “anchor” vertex of the structure, and S is the set of terminals already connected.

④ Why we need both v and S

If you only tracked S, you would not know where the current connected piece sits in the graph. Two states with the same S but anchored at different vertices have completely different extension costs through graph edges. The vertex anchor is what makes Dijkstra-style relaxation possible.

│

Micro-example: Terminals A and C in a graph, non-terminal B. Suppose A–B costs 1, B–C costs 1, A–C costs 5. Then dp[B][{A,C}] = 2 (route A–B–C through Steiner point B), while dp[A][{A,C}] = 5 (direct edge). Both cover terminal set {A,C} but at different costs and anchored at different vertices.

💡

Two distinct transition types: (1) Merge: at the same vertex v, combine two structures covering disjoint terminal subsets: dp[v][S] = min over T⊂S: dp[v][T] + dp[v][S\T]. This grows coverage without moving. (2) Relax: walk a graph edge from v to neighbor u, paying the edge cost: dp[u][S] = min(dp[u][S], dp[v][S] + edge(v,u)). This moves the anchor. Missing either transition type produces an incomplete algorithm.

Complexity: O(3k · V) for the merge step (enumerate all subset-pairs), plus O(2k · (V+E) log V) for the Dijkstra relaxation step.

🔥

Common pitfall: Forgetting the two-transition structure. The merge step and the relaxation step are both required. Omitting the merge step means you can never combine two separately-grown terminal subsets at the same vertex. Omitting the relaxation step means the anchor can never move through non-terminal nodes to reach distant terminals.

💡

Recognition rule: When you see “connect a small set of k special nodes with minimum total edge cost in a general graph” — think Steiner Tree DP. The keyword is small terminal set (k ≤ 10) in a large graph.

5.6 Subset Partition DP

Partition a set into valid groups minimizing cost or number of groups. The canonical trick for enumerating all submasks of a given mask is: for (int sub = mask; sub > 0; sub = (sub-1) & mask). This visits all non-empty submasks of mask in O(3n) total across all masks, because each element is in one of three states: not in mask, in mask but not sub, in both mask and sub.

6. Advanced Tree / Graph DP TREE

What is a rooted tree and why does it enable DP?

A tree is a connected undirected graph with no cycles. When we pick one node as the root, every other node has a unique parent (the node above it on the path to the root) and zero or more children (the nodes directly below it). The subtree of node u means the node u itself and all its descendants.

Why does rooting enable DP? Because once the tree is rooted, every node u has a natural “smaller” part (its subtree) that contributes to a “larger” whole (its parent's subtree). We can compute the answer for each subtree bottom-up, using children’s answers to build the parent’s answer. This is the post-order DFS fill order: process children before the parent.

💡

Central fact: A rooted tree lets you decompose the problem into child subproblems. Everything else in this section is about what information each subtree must report upward to its parent.

6.1 Rerooting DP: computing the answer for every possible root

① Problem shape

Given a tree with n nodes, compute some aggregate value for every node as if that node were the root. The classic example: for each node u, find the sum of distances from u to all other nodes. Other examples include: find the farthest node from u, or find the total subtree-size sum when rooted at u.

② The one-root version first

Pick any node (say node 0) as the root and run a single DFS. During this DFS, for each node u, compute down[u] = the answer using only u’s subtree. For the sum-of-distances problem: down[u] is the sum of distances from u to every node in u’s subtree. Also compute sz[u] = the number of nodes in u’s subtree. The transition is: down[u] = ∑ (down[v] + sz[v]) over all children v of u. The “+ sz[v]” accounts for the fact that every node in v’s subtree is one edge farther from u than from v.

③ The rerooting question: what if every node were the root?

After DFS1, down[0] gives the correct answer for the original root. But down[u] for non-root u only sees u’s subtree. It misses all the nodes above u (the rest of the tree). To get the full answer for every u, we need the contribution from the “parent side” — i.e., the entire tree outside u’s subtree.

④ Why recomputing DFS from every root is too slow

Running a full DFS from each of the n nodes gives O(n2). Rerooting DP achieves O(n) by noticing that when we move the root from a parent u to a child v, we can incrementally update the answer. The key insight: when v becomes the root, all of v’s subtree stays the same, but u’s subtree loses v’s subtree and gains the rest of the tree from above u.

⑤ The arrays, defined in plain language

down[u] = the answer computed using only u’s subtree (with the original root). Computed in DFS1 (bottom-up).

sz[u] = the number of nodes in u’s subtree. Computed in DFS1.

full[u] = the answer when the tree is re-rooted at u — i.e., u sees the entire tree. This is what we want for every node. full[u] = down[u] + from_above, where from_above is the contribution from all nodes outside u’s subtree.

⑥ The reroot transfer formula

In DFS2, we propagate information top-down. When processing child v of node u, we compute the “parent-side contribution” for v:

Step 1: full[u] already encodes the answer for the whole tree as seen from u. Step 2: Remove v’s contribution from full[u]. The amount v contributed is down[v] + sz[v]. Step 3: The remaining nodes (n − sz[v] of them) are now on the parent side of v, and they are each one edge farther from v than from u. So add (n − sz[v]). The parent-side contribution for v is: parent_side = full[u] − (down[v] + sz[v]) + (n − sz[v]).

Then: full[v] = down[v] + parent_side.

Sometimes you need the answer when the tree is rooted at every possible node (e.g., “find the node that minimizes the sum of distances to all other nodes”). Naively re-running the tree DP for each possible root is O(n2). The rerooting technique reduces this to O(n) using two DFS passes.

Why one DFS is not enough

DFS1 computes each node’s answer using only its own subtree. But when the tree is re-rooted at a child v, node v needs to also incorporate the value coming down from the old root side — which is the entire rest of the tree above the old root. DFS1 cannot see that “above” part.

DFS2 fixes this: starting from the root, it propagates the “parent-side contribution” to each child. A child v receives: (what the original root knew about the whole tree) minus (v’s own contribution) plus (the complement, which is the entire non-v side of the tree).

│

Tiny tree walkthrough: Tree with 3 nodes: 0–1–2 (path). Root at 0. DFS1: sz[2]=1, down[2]=0; sz[1]=2, down[1]=down[2]+sz[2]=0+1=1; sz[0]=3, down[0]=down[1]+sz[1]=1+2=3. So from node 0, total distance = 3 (dist to 1 is 1, dist to 2 is 2). DFS2: full[0]=down[0]+0=3. For child 1: parent_side = full[0]−(down[1]+sz[1])+(n−sz[1]) = 3−(1+2)+(3−2) = 3−3+1 = 1. full[1]=down[1]+1=1+1=2. For child 2 of 1: parent_side = full[1]−(down[2]+sz[2])+(n−sz[2]) = 2−(0+1)+(3−1) = 2−1+2 = 3. full[2]=down[2]+3=0+3=3. Final: full = [3, 2, 3]. Node 1 (the center) has the smallest sum of distances. Correct!

void dfs1(int u, int par) {
    down[u] = 0;      // contribution from u's subtree only
    sz[u] = 1;
    for (int v : adj[u]) {
        if (v == par) continue;
        dfs1(v, u);
        sz[u] += sz[v];
        down[u] += down[v] + sz[v]; // example: sum-of-distances
    }
}

void dfs2(int u, int par, long long from_above) {
    full[u] = down[u] + from_above; // total answer when rooted at u
    for (int v : adj[u]) {
        if (v == par) continue;
        // remove v's subtree from u's total, replace with rest-of-tree
        long long child_contr = down[v] + sz[v];
        long long parent_side = full[u] - child_contr + (n - sz[v]);
        dfs2(v, u, parent_side);
    }
}
💡

Mental model: After DFS1, each node knows everything below it. DFS2 tells it everything outside its subtree. Together, the node gets a complete picture of the whole tree. That is why rerooting always works in two passes.

⚠

Common pitfall: Getting the “subtract v's contribution and add back” arithmetic wrong. Write out the invariant carefully: full[u] is the answer if the tree were rooted at u. When passing the parent-side contribution to child v, the parent side is everything outside v’s subtree — which is full[u] - (contribution from v's subtree as seen from u) plus an edge traversal cost.

6.2 Tree Matching

① Problem shape

Given a tree with n nodes, find a maximum matching: the largest set of edges such that no two edges share a node. In other words, every node can be an endpoint of at most one selected edge.

② Why children are not independent

A first instinct is: “for each child, independently decide whether to take the child’s edge.” But if node u matches with child v (taking edge u–v), then u is “used up” — u cannot also match with any other child. So the decision to match u with one particular child blocks all other children from matching with u. This coupling between children is exactly why we need a DP state that tracks whether u is already matched or still free.

③ State definition

dp[u][0] = maximum matching in the subtree of u, assuming u is free (not matched to any of its children). u may still be matched later to its parent, but within its subtree, u is unmatched.

dp[u][1] = maximum matching in the subtree of u, assuming u is matched to exactly one of its children. Once matched, u cannot be matched to its parent.

④ Transition logic

Case dp[u][0] (u is free): Since u is not matched to any child, every child v independently contributes its own best matching. For each child v, take max(dp[v][0], dp[v][1]) (the child can be matched or free within its own subtree). Sum over all children:

dp[u][0] = ∑v child of u max(dp[v][0], dp[v][1])

Case dp[u][1] (u is matched to some child): Choose exactly one child v to match with u (taking edge u–v). That child v must be in state 0 (free) so that the u–v edge is valid. The gain from this match is: instead of taking max(dp[v][0], dp[v][1]) from child v, we take dp[v][0] + 1 (the +1 is the edge u–v). All other children contribute their independent best. So:

dp[u][1] = max over children v of: (dp[u][0] − max(dp[v][0], dp[v][1]) + dp[v][0] + 1)

This formula says: start with dp[u][0] (all children at their independent best), then for one chosen child v, swap its contribution from max(dp[v][0], dp[v][1]) to dp[v][0] + 1. Choose the v that maximizes this swap.

│

Micro-example: Star tree: center u with children v1, v2, v3 (all leaves). Each leaf is a base case: dp[leaf][0]=0, dp[leaf][1]=0 (a leaf has no children to match with, so state 1 is impossible — treat as 0 or −∞). dp[u][0] = 0+0+0 = 0 (u free, no edges taken). dp[u][1]: try matching u with v1: gain = 0 − 0 + 0 + 1 = 1. Same for v2, v3. So dp[u][1] = max(1,1,1) = 1. Answer = max(dp[u][0], dp[u][1]) = 1. Correct — a star with 3 leaves has maximum matching of 1 edge.

void dfs(int u, int par) {
    dp[u][0] = dp[u][1] = 0;
    for (int v : adj[u]) {
        if (v == par) continue;
        dfs(v, u);
        dp[u][0] += max(dp[v][0], dp[v][1]);
    }
    // Try matching u with each child v
    for (int v : adj[u]) {
        if (v == par) continue;
        int gain = dp[v][0] + 1 - max(dp[v][0], dp[v][1]);
        dp[u][1] = max(dp[u][1], dp[u][0] + gain);
    }
}
💡

Recognition rule: When you see “maximum matching on a tree” or “pair up adjacent nodes with no node used twice on a tree” — think two-state tree DP: free vs. matched.

⚠

Common pitfall: Allowing u to match with more than one child. The “choose the best single child to match” logic must be explicit. If you sum gains from multiple children, you violate the matching constraint.

6.3 Tree Vertex Cover

① Problem shape

A vertex cover of a graph is a set of vertices such that every edge has at least one endpoint in the set. On a tree, find the minimum vertex cover — the smallest set of nodes that “covers” every edge.

② Why we select vertices, not edges

Vertex cover asks which nodes to include. Each included node covers all edges incident to it. The goal is to cover every edge with the fewest nodes. This is different from matching (which selects edges). The constraint is: for every edge u–v in the tree, at least one of u or v must be in the cover.

③ State definition

dp[u][0] = minimum vertex cover of u’s subtree when u is NOT in the cover.

dp[u][1] = minimum vertex cover of u’s subtree when u IS in the cover.

④ Transition logic

dp[u][1] (u is in the cover): Since u covers all edges from u to its children, each child v is free to be in the cover or not. Take the cheaper option for each child independently: dp[u][1] = 1 + ∑v min(dp[v][0], dp[v][1]).

dp[u][0] (u is NOT in the cover): The edge u–v must still be covered. Since u is absent, v must be in the cover to cover the edge u–v. This applies to every child: dp[u][0] = ∑v dp[v][1].

dp[u][1] = 1 + ∑ min(dp[v][0], dp[v][1]) over children v
dp[u][0] = ∑ dp[v][1] over children v
│

Micro-example: Path A–B–C. Leaves first: dp[A][0]=0, dp[A][1]=1, dp[C][0]=0, dp[C][1]=1. Node B: dp[B][1] = 1 + min(dp[A][0],dp[A][1]) + min(dp[C][0],dp[C][1]) = 1+0+0 = 1. dp[B][0] = dp[A][1] + dp[C][1] = 1+1 = 2. Answer = min(dp[B][0], dp[B][1]) = min(2,1) = 1 (just select B). Correct — selecting B covers both edges A–B and B–C.

void dfs(int u, int par) {
    dp[u][0] = 0;  // u not in cover
    dp[u][1] = 1;  // u in cover (cost 1 for including u)
    for (int v : adj[u]) {
        if (v == par) continue;
        dfs(v, u);
        dp[u][0] += dp[v][1];               // child MUST be in cover
        dp[u][1] += min(dp[v][0], dp[v][1]); // child is free
    }
}
💡

Recognition rule: When you see “every edge must have at least one endpoint selected” on a tree — think vertex cover. Two states: u taken (children free) vs. u not taken (every child forced to be taken).

⚠

Common pitfall: Forgetting that when u is NOT in the cover, every child must be in the cover — not just one. Each u–v edge needs individual coverage, and only v can provide it when u is absent.

6.4 Dominating Set on Trees

① Problem shape

A dominating set D of a graph is a subset of nodes such that every node is either in D or adjacent to a node in D. On a tree, find the minimum-size dominating set. Unlike vertex cover (which demands every edge be incident to a chosen node), dominating set demands every node be chosen or have a chosen neighbor.

② Why two states are insufficient

With only “u is selected” vs. “u is not selected,” you cannot distinguish two critical sub-cases of “not selected.” Case A: u is not selected, but one of u’s children is selected, so u is already dominated from below. Case B: u is not selected and no child is selected, so u is not yet dominated — u’s parent must be selected to dominate u. The parent needs to know which case applies, because in case B the parent is forced into the dominating set, while in case A the parent is free. A single “not selected” state cannot convey this information upward.

③ The three-state model (consistent definitions)

State 0 — SELECTED: Node u is in the dominating set. It dominates itself and all its neighbors. Cost includes 1 for selecting u.

State 1 — NOT selected, DOMINATED by at least one child: Node u is not in the dominating set, but at least one of u’s children is selected, which dominates u. u does not need its parent to dominate it.

State 2 — NOT selected, NOT yet dominated: Node u is not in the dominating set and no child is selected. u is not dominated within its subtree. This is only legal if u’s parent will be selected (state 0) to dominate u. At the root, state 2 is invalid (the root has no parent).

⑤ Transitions for each state

dp[u][0] (u is SELECTED): Since u is selected, every child v is automatically dominated by u. Each child v can be in any state — whichever is cheapest. But we do not actually need state 2 for a child when its parent (u) is selected, because u already dominates v. So: dp[u][0] = 1 + ∑v min(dp[v][0], dp[v][1], dp[v][2]).

dp[u][1] (u is NOT selected, dominated by a child): At least one child must be in state 0 (selected) to dominate u. All other children can be in state 0 or state 1 (they cannot be in state 2, because u is not selected so u cannot dominate them — they must be dominated within their own subtrees or be selected themselves). To enforce “at least one child selected,” compute the baseline sum assuming every child takes min(dp[v][0], dp[v][1]), then check: if at least one child naturally chose state 0, we are done. If not, force the child where switching from state 1 to state 0 costs the least.

Formally: let base = ∑v min(dp[v][0], dp[v][1]). If any child v already has dp[v][0] ≤ dp[v][1], then that child is selected and u is dominated: dp[u][1] = base. Otherwise, find the child v that minimizes dp[v][0] − dp[v][1] (the cheapest upgrade from not-selected to selected), and add that cost: dp[u][1] = base + minv(dp[v][0] − dp[v][1]).

dp[u][2] (u is NOT selected, NOT dominated): u hopes its parent will dominate it. Within u’s subtree, no child needs to be selected for u’s sake. But each child v must still be dominated within its own subtree (since u is not selected, u cannot dominate v). So each child v must be in state 0 or state 1: dp[u][2] = ∑v min(dp[v][0], dp[v][1]).

dp[u][0] = 1 + ∑ min(dp[v][0], dp[v][1], dp[v][2])
dp[u][2] = ∑ min(dp[v][0], dp[v][1])
dp[u][1] = dp[u][2] + (correction to ensure ≥1 child is selected)
│

Micro-example: Path A–B–C, root at B. Leaves: dp[A][0]=1, dp[A][1]=∞ (no children to dominate A), dp[A][2]=0. dp[C][0]=1, dp[C][1]=∞, dp[C][2]=0. Node B: dp[B][0] = 1 + min(1,∞,0) + min(1,∞,0) = 1+0+0 = 1. dp[B][2] = min(1,∞) + min(1,∞) = 1+1 = 2. dp[B][1]: base = min(1,∞)+min(1,∞) = 1+1 = 2. Both children contribute dp[v][0]=1 (selected), so at least one child is selected. dp[B][1] = 2. Answer (root B, state 2 invalid) = min(dp[B][0], dp[B][1]) = min(1, 2) = 1. Select only B. Correct!

// dp[u][0] = selected, dp[u][1] = not selected but dominated by child,
// dp[u][2] = not selected and not dominated (needs parent)
const int INF = 1e9;
void dfs(int u, int par) {
    dp[u][0] = 1;
    dp[u][1] = 0;
    dp[u][2] = 0;
    bool has_child = false;
    bool any_child_selected = false;
    int min_upgrade = INF; // cheapest cost to force one child into state 0
    for (int v : adj[u]) {
        if (v == par) continue;
        has_child = true;
        dfs(v, u);
        dp[u][0] += min({dp[v][0], dp[v][1], dp[v][2]});
        int best01 = min(dp[v][0], dp[v][1]);
        dp[u][2] += best01;
        if (dp[v][0] <= dp[v][1]) {
            any_child_selected = true;
        }
        min_upgrade = min(min_upgrade, dp[v][0] - dp[v][1]);
    }
    dp[u][1] = dp[u][2]; // start from same base as state 2
    if (!has_child) {
        dp[u][1] = INF; // leaf: no child can dominate u
    } else if (!any_child_selected) {
        dp[u][1] += min_upgrade; // force cheapest child into state 0
    }
    // else: at least one child is already selected, dp[u][1] is correct
}
💡

Recognition rule: When you see “every node must be selected or adjacent to a selected node on a tree” — think dominating set. Three states: selected, not-selected-but-dominated-by-child, not-selected-and-not-yet-dominated. The answer at the root is min(dp[root][0], dp[root][1]) (state 2 is invalid at the root since there is no parent).

🔥

Critical pitfall: In state 1, you must enforce that at least one child is selected (state 0). If all children prefer state 1 (dominated by their own children but not selected themselves), then no child is selected, and u remains undominated. The “min_upgrade” correction handles this — do not forget it.

6.5 Subtree Merge DP

When each child contributes a whole DP table (not just a single value) and the parent must combine them, process children one at a time and merge like a knapsack convolution. State dp[u][k] = best answer using exactly k from the subtree of u. Whenever you see “exactly k nodes / edges / resources inside a subtree,” reach for merge DP.

6.6 Functional Graph DP

① What is a functional graph?

A directed graph where every node has exactly one outgoing edge is called a functional graph. For example: f(x) is some function on integers, and there is an edge from x to f(x) for every x. If you follow edges starting from any node, you will eventually enter a cycle and loop forever. The structure therefore consists of directed cycles (the “rho” shape ρ), with trees draining into the cycle nodes.

①b The mental picture: trees feeding into cycles

Imagine drawing the graph. You will see one or more directed rings (cycles). Growing out of each ring node, like branches from a necklace, are directed trees whose edges all point toward the ring. Every non-cycle node belongs to exactly one such tree, and following its single outgoing edge repeatedly will eventually reach the ring. This “necklace with dangling branches” is the shape of every functional graph. The trees are easy (standard tree DP). The rings are the hard part.

② Why naive tree DP fails

If you try to root the functional graph and run a standard post-order DFS, you will follow back edges into the cycle and either loop infinitely or produce wrong answers. Cycles have no valid base case for a standard DFS. You must detect cycles first, then handle them separately.

③ The two-phase approach

Phase 1 — Solve the trees: Find all cycles (using a coloring DFS or SCC algorithm). Every node not on a cycle belongs to a tree that “drains into” a cycle node. Apply standard tree DP bottom-up on each of these trees. After this phase, every cycle node carries a “hanging tree value” summarising its dangling branch.

Phase 2 — Solve each cycle: For each cycle of length L, break it at one edge to get a chain of length L. Run a linear chain DP on this chain twice — once where the cut-edge is “not used” and once where it is “used” — to cover both boundary conditions. The answer for the cycle is the better of the two runs.

│

Micro-example: Cycle A → B → C → A, with trees hanging off A (chain D → A) and C (chain E → C). Apply tree DP on chains D→A and E→C first. Each cycle node now has a “hanging tree value”. To find the maximum independent set on the cycle {A,B,C}: cut edge C→A. Run a chain DP on A–B–C twice: (case 1) A is forced to state 0; (case 2) A is forced to state 1. Answer = max of both cases.

💡

Recognition rule: When you see “every node has exactly one successor” or “follow function f(x) repeatedly,” think functional graph. The structure is rho-shaped: trees drain into cycles.

🔥

Common pitfall: Trying to DFS the whole functional graph without first identifying cycles. Always detect cycles first (one coloring DFS is enough), then apply tree DP on trees and cycle-breaking DP on cycles.

6.7 DP on SCC-Condensed DAGs

① The problem: cycles break DP

DP requires a strict dependency order: every state must be computed only from already-finished states. But if a directed graph has a cycle (say, node 3 → node 5 → node 3), then dp[3] depends on dp[5] which depends on dp[3] — a circular dependency. No fill order can resolve this. Before running DP on a directed graph with cycles, we must eliminate the cycles.

② The fix: compress each cycle into one super-node

A Strongly Connected Component (SCC) is a maximal set of nodes where every node can reach every other node within the set. Tarjan’s and Kosaraju’s algorithms find all SCCs in O(V+E) time. The key insight: if you replace each SCC with a single super-node and keep only the cross-SCC edges, the result is guaranteed to be a DAG (no cycles remain). This DAG is called the condensation. Now DP in topological order is safe.

③ What the state remembers

dp[scc_id] = the optimal value for reaching or using SCC scc_id, considering all SCCs that can be reached before it in topological order. Inside one SCC, every node can reach every other, so the internal structure often collapses to a single aggregate value (sum of node values, max value, etc.).

│

Micro-example: A graph with 6 nodes: nodes {1,2,3} form one SCC (each can reach the others), node 4 is alone, nodes {5,6} form another SCC. Edge from SCC_A to SCC_B, edge from SCC_B to SCC_C. The condensation is a 3-node DAG: A → B → C. Topological order: A, then B, then C. dp[C] = value_C + max(dp[B]), dp[B] = value_B + max(dp[A]), dp[A] = value_A. Here value_X = sum of node values inside SCC X.

💡

Recognition rule: When you see “directed graph with cycles, find longest path / maximum collected value / count distinct paths” — first check for cycles. If cycles exist, condense SCCs, then DP on the resulting DAG in topological order.

🔥

Common pitfall: Incorrectly aggregating the internal value of each SCC. If the problem asks for the maximum value collectable along a path, all nodes inside an entered SCC are visitable (since the SCC is fully connected), so the SCC’s contribution is the sum of all its node values — not just the entry node. Always think carefully about what “being inside an SCC” allows you to collect.

Fill order: topological order of the condensation DAG. Use the condensation produced by Tarjan’s algorithm directly — Tarjan’s naturally produces SCCs in reverse topological order.

7. Counting / Combinatorial DP COUNTING

Optimization DP asks “what is the best?” Counting DP asks “how many valid structures exist?” The transition operator changes from max/min to addition. Overcounting becomes the central danger instead of suboptimality.

Counting DP core rule:
Does each valid object get counted exactly once?
If you cannot answer that, the recurrence is not yet safe.

The disjoint partition proof technique

To verify that a counting recurrence is correct, partition the set of all valid objects by the “last choice” or “last move.” If those classes are provably disjoint (no valid object belongs to two classes) and their union covers all valid objects (every valid object belongs to exactly one class), then summing the class counts is correct. This is the counting analogue of the optimal-substructure proof.

Why counting DP sums where optimization DP takes max/min

In optimization DP, when you have two ways to reach a state, you keep only the best. In counting DP, both ways represent valid distinct objects, so you add their counts. This is the fundamental difference. The recurrence structure is often identical; only the aggregation operator changes.

Modulo arithmetic: why and how

Counts grow astronomically fast. A 100-element binary string has 2100 possibilities — far beyond any integer type. Contest problems therefore ask for the answer modulo a prime, typically 109+7. Apply modulo at every addition inside the DP transition, not just at the end. If you compute large intermediate values and mod only at the end, you will overflow long before reaching the answer.

Concrete Example 1: Binary strings with no two consecutive 1’s

① Problem

Count the number of binary strings of length n such that no two consecutive characters are both ‘1’.

② Why brute force fails

There are 2n binary strings. For n=50, that is over 1015 strings to check. We need a DP that avoids enumerating each one.

③ State definition

The future cares about the last character placed (because the constraint is about consecutive characters). So: dp[i][c] = number of valid binary strings of length i whose last character is c (c = 0 or 1).

④ Transition

If the last character is 0, the previous character can be anything (0 or 1): dp[i][0] = dp[i-1][0] + dp[i-1][1].

If the last character is 1, the previous character must be 0 (to avoid consecutive 1’s): dp[i][1] = dp[i-1][0].

Total valid strings of length i: dp[i][0] + dp[i][1].

⑤ Base case and fill order

dp[1][0] = 1 (the string “0”), dp[1][1] = 1 (the string “1”). Fill left to right: i = 2, 3, …, n.

│

Trace for n=4: dp[1]=[1,1]. dp[2][0]=1+1=2, dp[2][1]=1. dp[3][0]=2+1=3, dp[3][1]=2. dp[4][0]=3+2=5, dp[4][1]=3. Total = 5+3 = 8. The 8 valid strings: 0000, 0001, 0010, 0100, 0101, 1000, 1001, 1010.

const int MOD = 1e9 + 7;
long long dp[2] = {1, 1}; // dp[0]=ends-in-0, dp[1]=ends-in-1
for (int i = 2; i <= n; i++) {
    long long new0 = (dp[0] + dp[1]) % MOD;
    long long new1 = dp[0]; // only 0 can precede 1
    dp[0] = new0;
    dp[1] = new1;
}
long long ans = (dp[0] + dp[1]) % MOD;

This is the Fibonacci recurrence! The total for length n equals Fibonacci(n+2). The connection is not a coincidence — it arises from the same branching structure.

Concrete Example 2: Count sequences of length n from {1,2,3} with no two adjacent elements equal

① Problem

Using the alphabet {1, 2, 3}, count sequences of length n where no two consecutive elements are the same.

② State definition

dp[i][c] = number of valid sequences of length i ending with character c. Since c ranges over {1,2,3}, we have 3 states per position.

③ Transition

Character c can follow any character that is not c. So: dp[i][c] = ∑c' ≠ c dp[i-1][c']. By symmetry, every dp[i][c] has the same value (call it f). Then f(i) = 2 · f(i-1) because there are exactly 2 predecessors for each character. Base: f(1) = 1 for each character, total = 3. For n=2: f(2) = 2, total = 6. For n=3: f(3) = 4, total = 12.

│

Verification for n=2: Valid pairs: 12, 13, 21, 23, 31, 32 = 6. Matches 3 × 2 = 6. ✔

Overcounting pitfall: why disjoint partitions matter

Suppose you try to count paths in a grid by “going right, then going down” and separately “going down, then going right.” A path that alternates right-down-right-down would be counted in both classes. The classes overlap, so summing their sizes overcounts. The safe approach is to partition by the last move (or some other unique feature), guaranteeing each path appears in exactly one class.

For example, in counting the number of distinct binary strings of length n with no two consecutive 1s: partition by whether the last character is 0 or 1. The class “ends in 0” contains all valid strings ending in 0, and the class “ends in 1” contains all valid strings ending in 1 (the second-to-last must be 0). These two classes are disjoint and cover all valid strings, so dp[n] = dp[n-1] + dp[n-2] is correct (the Fibonacci recurrence).

7.3 Permutations with Restrictions

Counting permutations where adjacent differences must satisfy something, or where some positions are forbidden, or where some relative-order constraints exist. No single approach handles all variants. Small n (≤ 20) suggests bitmask DP. Position-based local restrictions suggest prefix DP. Global forbidden structures suggest inclusion–exclusion. The phrase “count permutations satisfying conditions” does not determine one family — you must ask: what is the natural state that captures what the future needs to know?

7.5 Inclusion–Exclusion + DP Hybrids

Sometimes a direct DP counts too many objects, including invalid ones. You then count the full family by DP and subtract families that violate each constraint, alternating sign in the inclusion–exclusion principle. This principle says: |A or B| = |A| + |B| − |A and B|. Generalizing to k constraints: sum over all non-empty subsets S of constraints of (−1)|S|+1 × (count of objects violating all constraints in S). Inclusion–exclusion is not separate from DP — it is often the outer shell around an inner DP count. Examples: count strings avoiding multiple bad patterns, count colorings with coverage constraints.

8. Advanced Grid DP GRID

Base grid DP tracked a position. Advanced grid DP adds more state when position alone is not sufficient. The key diagnostic question is: is there something about the past path that the future cares about, beyond just where we currently stand? If yes, that extra thing becomes an additional state dimension.

8.1 Two-Person Grid DP (Cherry Pickup)

Two agents both move through the grid (say, only right or down) and together collect the values of cells they visit. A cell visited by both agents contributes its value only once (no double-counting).

Why four coordinates reduce to three

Naively, the state would be dp[r1][c1][r2][c2] for the two agents’ positions — four dimensions. But there is a crucial invariant: if both agents take exactly one step per time unit (always moving right or down), then after t steps, each agent has taken exactly t total steps. That means r1 + c1 = t and r2 + c2 = t for both agents simultaneously. Given r1, r2, and t, we can recover c1 = t−r1 and c2 = t−r2. So three values (t, r1, r2) are sufficient to determine all four coordinates.

dp[t][r1][r2] = best total collection after t steps when agent1 is at row r1, agent2 at row r2

Both column positions are recovered as c1 = t−r1, c2 = t−r2. Collecting from same cell: if r1 == r2 (and therefore c1 == c2), count value once. O(n3) on an n×n grid.

💡

Big lesson: The right state often emerges from an invariant linking multiple variables. Here, synchronized stepping reduces four dimensions to three. Always look for invariants when multiple moving objects share a time axis or a step count.

8.2 Grid DP with Exact K Steps

① Problem shape

A walker on a grid must reach a target cell in exactly K steps (not “at most K”). Maximize the value collected along the way. Each step moves to an adjacent cell (up, down, left, right).

② Why position alone is not enough

In basic grid DP, dp[i][j] = best value to reach cell (i,j). But that state ignores how many steps were used to get there. A walker might reach (i,j) in 3 steps or in 7 steps — those two situations have very different remaining budgets. Since the problem demands exactly K steps total, the number of steps already taken determines how many remain. Two histories with the same position but different step counts have different futures, so step count must be part of the state.

③ State definition

dp[k][i][j] = the best value achievable by standing at cell (i,j) after exactly k steps from the starting cell. The three dimensions: step count k, row i, column j.

④ Transition and fill order

dp[k][i][j] = max over (i',j') adjacent to (i,j): dp[k-1][i'][j'] + value[i][j]. Fill order: k from 1 to K. At each step layer k, we look at layer k−1 only, so all dependencies are satisfied.

Base case: dp[0][r0][c0] = value[r0][c0] (at step 0, we are at the start). All other dp[0][i][j] = −∞ (unreachable at step 0).

Answer: dp[K][target_row][target_col]. If still −∞, the target is unreachable in exactly K steps.

│

Micro-example: 2×2 grid, start (0,0), target (1,1), exactly K=2 steps. Moves: right or down. dp[0][0][0]=0. dp[1][0][1]=0 (right), dp[1][1][0]=0 (down). dp[2][1][1] = max(dp[1][0][1], dp[1][1][0]) + value[1][1] = max(0,0) + value = value[1][1]. If K=3, we cannot reach (1,1) in exactly 3 right/down steps on a 2×2 grid (minimum distance is 2, and K=3 requires a detour, which may be impossible without left/up moves).

⚠

Common pitfall: Using “at most K” initialisation (all zeros) when the problem asks “exactly K.” With exact-K, unreachable states must be −∞ (not 0), or you cannot distinguish “reachable with value 0” from “unreachable.”

8.3 Grid DP with Turn-Limited Movement

① Problem shape

A walker on a grid moves step-by-step, but may change direction at most T times total. For example: “a robot moves on a grid, starting facing right; each time it turns (changes from right to down, or from down to right), that uses one of T available turns.” Maximize or minimize something subject to this turn budget.

② Why direction must become part of the state

If the walker is at cell (i,j) facing right with 2 turns remaining, vs. at (i,j) facing down with 2 turns remaining, the two situations have different future possibilities. A continuation that goes right costs 0 turns in the first case but 1 turn in the second. So the current direction affects future costs. Direction must be in the state.

③ Why turns-used must be tracked

Two walkers at (i,j) facing the same direction, but one has used 1 turn and the other has used 3 turns, have different remaining budgets. The walker with fewer turns used has more flexibility in the future. So turns-used (or equivalently, turns-remaining) must also be in the state.

④ State definition

dp[i][j][dir][t] = the best value at cell (i,j), currently facing direction dir, having used t turns so far. The four dimensions: row, column, current direction, turn count.

⑤ Transition

From state (i,j,dir,t), the walker can: (a) continue in direction dir (no turn consumed) → move to the adjacent cell in direction dir, state becomes (i',j',dir,t); or (b) turn to a new direction dir' (one turn consumed, requires t < T) → stay at (i,j) or move to an adjacent cell in the new direction, state becomes (i',j',dir',t+1).

│

Micro-example: 3×3 grid, start (0,0) facing right, target (2,2), max turns T=1. The walker can go right-right, then turn down (1 turn), then go down-down. With only 1 turn allowed, the walker must choose a single L-shaped path. If T=0, the walker can only go straight — it can reach (0,2) or (2,0) but not (2,2).

💡

Recognition rule: When you see “grid movement with a limit on direction changes (turns)” — add direction and turn count to the state. The clue is any constraint on the shape or curvature of the path, not just its endpoints.

⚠

Common pitfall: Forgetting that a turn can occur even while staying in the same cell in some problem formulations. Read the problem carefully to determine whether a turn only happens when a step is taken in a new direction, or whether a turn is a separate action.

8.5 Recognition Rules for Advanced Grid DP

board or matrix movement constraints one or more walkers exact step / turn count extra local memory while walking (keys, flags) arbitrary edge weights with cycles (use shortest-path on state graph instead)

9. Advanced Digit DP DIGIT

Problem shape: counting integers with properties in a range

Digit DP solves problems of the form: “count all integers in the range [L, R] that have some property P.” The property typically involves the individual decimal digits: “digit sum equals 20,” “all digits are distinct,” “value modulo 7 equals 0,” etc. Direct enumeration works for tiny ranges but fails immediately for large ones like [1, 1018].

The standard reduction

Every range query [L, R] reduces to two single-bound queries: f(R) − f(L−1), where f(N) counts valid numbers from 0 to N. So the core task is always: count valid numbers from 0 to N. You build N digit by digit from the most significant to least significant, and at each position you must stay ≤ the corresponding digit of N.

Why naive counting fails

You might think: just loop from 0 to N and test each number. For N = 1018, that is 1018 iterations — completely infeasible. Digit DP avoids testing each number individually by noticing that many numbers share the same “prefix state,” and all numbers with the same prefix state contribute the same count to the final answer. By caching these prefix-state counts, the DP reduces the work to O(digits × number_of_states).

State definition: built from scratch

The state captures three things at each digit position. First, which position you are currently filling (leftmost = 0, rightmost = last digit). Second, the tight flag: is the prefix you have placed so far an exact digit-for-digit match to N’s prefix? If yes, the current digit is constrained to be ≤ the corresponding digit of N. If no, the current digit is free (0 to 9). Third, whatever accumulated property the problem cares about: running digit sum, running remainder mod m, bitmask of digits seen so far, etc.

dp[pos][tight][extra_state] = count of valid completions from this position

Why the tight flag is not optional

Without the tight flag, you cannot tell whether the next digit can be freely chosen from 0 to 9, or whether it must be ≤ the corresponding digit of N. If you set all digits freely from 0 to 9, you count numbers that exceed N, inflating the answer. If you restrict all digits to be ≤ N’s digit, you miss numbers where an earlier digit was smaller than N’s digit, making subsequent digits free. The tight flag is the precise mechanism that distinguishes these two situations.

🔥

Clarifying “tight”: Tight does NOT mean “we have equalled the bound.” It means “every digit placed so far has exactly matched N’s corresponding digit.” As soon as you place a digit strictly less than N’s digit, tight becomes false for all subsequent positions in this branch — those positions are completely free. Once false, tight can never become true again within the same number construction.

Micro-example — count multiples of 3 from 0 to 19

N = 19, target: value mod 3 == 0. Digits of N: [1, 9]. Fill from left:

Position 0, tight=true: digit choices 0 and 1 (must be ≤ 1). Digit 0 → tight becomes false (0 < 1), remainder = 0. Digit 1 → tight stays true (1 == 1), remainder = 1.

From (pos=1, tight=false, rem=0): digit can be 0..9 freely. Need (0 + d) mod 3 == 0: d in {0,3,6,9}. Count = 4. These represent 00, 03, 06, 09.

From (pos=1, tight=true, rem=1): digit can be 0..9 (must be ≤ 9, so still 0..9). Need (1 + d) mod 3 == 0: d in {2,5,8}. But tight=true means d ≤ 9 — still 2,5,8. Count = 3. These represent 12, 15, 18. Also 10..19 with digit checking: 12,15,18.

Total: 4 + 3 = 7 numbers (0,3,6,9,12,15,18). Correct!

💡

Recognition rule: When you see “count integers in [L, R] with property involving digits or digit-related quantities (sum, product, mod, distinctness)” — think digit DP. The bounding digit N drives the tight flag. The property drives the extra state dimension.

9.1 Digit DP with Modulo State

Count numbers whose numeric value mod m equals a target. Add the remainder as a state dimension. When appending digit d, update the remainder as rem = (rem × 10 + d) % m. The future only needs the current remainder, not the full numeric prefix.

dp[pos][tight][rem] where rem = current prefix value mod m

9.1b Leading Zeros and the “Started” Flag

① What are leading zeros?

When building a number digit by digit from left to right, the number has not “truly begun” until a non-zero digit is placed. Before that, all placed digits are leading zeros. The number 007 is really 7; the number 0042 is really 42. Leading zeros are not real digits of the number — they are artifacts of the fixed-width representation.

② When leading zeros are harmless

If the problem only cares about the numeric value of the number (e.g., “count multiples of 3 from 0 to N”), leading zeros do not affect the answer. The number 007 has value 7, and 7 mod 3 = 1 regardless of how many leading zeros preceded it. In such cases, you do not need any special handling.

③ When leading zeros are dangerous

If the problem cares about which digits appear in the number, leading zeros can cause wrong answers. Example: “count numbers from 1 to N where all digits are distinct.” Consider the number 42. If you build it as 042 in a 3-digit framework, the digit 0 appears in position 0. But 42 does not actually contain the digit 0 — that was a leading zero, not a real digit. If your DP records 0 in the “used digits” bitmask, you will wrongly think digit 0 has been used, potentially rejecting valid numbers (like 402) or miscounting.

④ The “started” flag

Add a boolean started to the state: dp[pos][tight][started][extra]. The flag started = false means “all digits placed so far are zeros — the real number has not begun yet.” As soon as a non-zero digit is placed, started becomes true and stays true.

While started = false: do NOT update the “used digits” mask (leading zeros are not real digits). Do NOT count digit 0 as a “distinct digit.” Do NOT add the digit value to running sums that track digit-level properties. Once started = true: process digits normally.

│

Micro-example showing wrong answer without started flag: Count numbers from 1 to 20 with all digits distinct. Consider the number 5 (a single-digit number). If built as “05” in a 2-digit framework without a started flag, the mask records digits {0, 5}. Now when counting 2-digit numbers, 50 would be rejected because “digit 0 is already used.” But the first 0 was a leading zero! The correct count should allow 50 (digits 5 and 0, both distinct). Without the started flag, your count will be too low.

dp[pos][tight][started][mask] — track whether the number has truly begun
💡

Recognition rule: Whenever the digit DP problem involves “which digits appear,” “all digits distinct,” “digit frequency,” or any property that treats 0 differently when it is leading vs. interior — add a started flag. Without it, leading zeros will contaminate the digit-tracking state.

🔥

Common pitfall: Assuming that the started flag is needed for every digit DP. It is not. For problems like “count multiples of m” or “digit sum equals S,” leading zeros are harmless (they contribute 0 to sums and 0 to products). Only add the started flag when the problem’s property is sensitive to which digits are “real.”

9.2 Digit DP with Bitmask

Track which decimal digits (0–9) have appeared so far using a 10-bit mask (one bit per digit). This enables problems like “all digits distinct,” “contains at least one 7,” or “exactly k distinct digits.” The state is dp[pos][tight][used_digits_mask].

9.4 Constructive Digit DP

The problem asks you to construct the smallest or largest valid number in [L, R], not just count. Strategy: run the DP to precompute feasibility. Then greedily build digit by digit: at each position, try the smallest digit that keeps the remaining suffix feasible. “Feasible” means: with this prefix, there exists at least one valid completion.

✅

Pattern: DP answers “is it feasible from here?” then greedy construction asks “which smallest digit keeps it feasible?” This DP + greedy reconstruction pattern appears across many domains, not just digit DP.

9.5 Optimization Digit DP

Among all numbers in [L,R], maximize or minimize some objective. The stored value becomes the best objective instead of a count, and infeasible states need −INF or +INF instead of zero.

9.6 Recognition Rules

numbers in a range [L,R] per-digit constraint or per-digit property count integers with decimal property prefix of number bounded by target distinct digits / modulo / digit sum / digit product n is tiny enough for direct enumeration (skip digit DP) property is not digit-local but depends on global structure of the integer

10. Advanced Profile / Frontier DP FRONTIER

The core idea: what is a “frontier”?

① The simplest picture

Imagine you have a grid, and you are processing it cell by cell, left to right and then top to bottom (like reading English text). At any moment, some cells have been decided (placed, filled, colored — whatever the problem requires), and the rest have not been touched yet. The frontier is the thin boundary between these two regions.

Think of it as a vertical line sweeping across the grid. Everything to the left of the line is done. Everything to the right is untouched. The cells right on the boundary carry unresolved information — maybe a domino extends across the boundary, or a path has an open end crossing the boundary, or a cell was left empty but an adjacent unprocessed cell might fill it later.

② Why local cell choices alone are insufficient

A first instinct might be: “process each cell independently, choosing the locally best option.” But many grid problems have cross-cell dependencies. If you place a horizontal domino, it occupies two cells in the same row. If you are building a Hamiltonian path, placing a path segment in one cell constrains which adjacent cells can connect to it. These dependencies mean that the decision at cell (i,j) cannot be evaluated without knowing what happened at the boundary cells. That boundary information is what the frontier state encodes.

③ What does a frontier state (profile) encode?

The frontier state is a compact description of the boundary between processed and unprocessed cells. At minimum, it records the occupancy pattern of the frontier slots: which slots are already occupied (by a piece extending from the processed side) and which are empty (available for the unprocessed side to fill). For a grid of width W, the frontier has at most W+1 slots, and the simplest encoding is a W-bit bitmask: bit 1 = occupied, bit 0 = empty.

For harder problems (like counting Hamiltonian paths), occupancy alone is not enough. You also need connectivity information: which frontier slots are connected to each other through paths already built in the processed region. This is a richer encoding called a connectivity profile.

④ Why this compression is sufficient

The key insight: two partial solutions that produce the same frontier state have identical future possibilities. The processed region behind the frontier is done forever — no future cell will look back into it (except through the frontier). The unprocessed region ahead of the frontier has not been touched — it is identical regardless of how the processed region was built. So the frontier state is exactly the “state” in the DP sense: it captures everything the future needs to know about the past. This is the future-sufficiency condition.

If two partial processed worlds induce the same frontier state,
then their future completion possibilities are identical.
That sentence is the proof foundation.
💡

Quick self-check before continuing. Can you answer these? (1) What is the frontier? — The boundary between processed and unprocessed cells. (2) Why can’t I just make local cell-by-cell choices? — Because a piece may extend across the boundary, coupling past choices to future options. (3) Why is the profile sufficient? — Because two partial solutions with the same frontier pattern have identical future possibilities. If you can answer all three, you have the core idea.

Why simpler state is insufficient

If you only tracked the count of cells processed (a single integer), you cannot tell how they are arranged on the frontier. Two arrangements that cover the same number of cells but differ in which frontier slots are occupied have completely different future options. You need the pattern on the frontier, not just a count.

Starting simple: occupancy-mask frontier DP

The simplest frontier problem: domino tiling

Problem: tile an H×W grid with 1×2 dominoes (no gaps, no overlaps). Count the number of valid tilings. Process cells left to right, top to bottom. For each column, the frontier is a W-bit mask recording which cells in the current column are already occupied by a horizontal domino extending from the previous column.

When you process a cell: if its frontier bit is 1, it is already occupied — skip it. If its frontier bit is 0, you must place a domino here. Options: (a) place a vertical domino covering this cell and the cell below it (requires the cell below to also be unoccupied — check the next frontier bit). (b) place a horizontal domino covering this cell and the cell to the right (mark the right cell’s frontier bit as occupied in the new frontier).

Micro-example: 2×4 grid domino tiling from scratch

Problem: count the number of ways to tile a 2×4 grid with 1×2 dominoes.

Process cells left to right, top to bottom. At any moment, the “frontier” is the boundary between already-placed and not-yet-placed cells. For a 2-wide grid, the frontier has 2 slots. Each slot is either “filled” (a horizontal domino extends into it from the left) or “empty” (needs to be covered). We encode the frontier as a 2-bit mask: bit 0 = top row slot, bit 1 = bottom row slot. Bit 1 means filled, bit 0 means empty.

When we process a cell, we decide: (a) place a vertical domino here (both rows in this column), requiring both frontier bits for this column to be empty; (b) extend a horizontal domino rightward, marking this cell’s bit as filled and processing its right neighbor’s frontier bit; or (c) this cell was already filled by a domino from the left (bit=1), so we skip to the next cell. The DP state tracks the frontier as we sweep column by column.

│

Result: For a 2×4 grid, the number of tilings is 5. You can verify: (all-vertical), (vertical 2×1 left, two horizontal right), (two horizontal left, vertical right), (horizontal across each row, no verticals), and (V+2H+V in the middle).

Row-scan walkthrough

Here is how the sweep works for a 2×3 grid. Frontier width = 2 (the grid has 2 rows). Start: frontier mask = 00 (both slots empty).

Column 0: Both slots empty. Option A: place vertical domino covering both rows → new frontier = 00 (domino is self-contained, nothing extends right). Option B: place horizontal domino in top row → top bit = 1 in new frontier. Then bottom row must also get a horizontal domino or pair with top → both bits = 1 in new frontier (mask 11). Continue processing each column using the previous column’s exit frontier as the current column’s entry frontier.

At the end (after column 2), the exit frontier must be 00 (nothing extends beyond the grid). Count all paths from entry frontier 00 (column 0) to exit frontier 00 (column 3). For a 2×3 grid, the answer is 3 tilings.

10.1 Why a frontier is enough

Cells already processed are done forever. Cells not yet touched are irrelevant until reached. The only place where unfinished information can live is the boundary between processed and unprocessed parts. If you encode the frontier correctly, the past becomes compressible into that boundary state, and the DP operates only on the frontier.

10.3 Plug DP

When building paths or loops through a grid (e.g., a Hamiltonian cycle), a frontier position may carry an “open connection” (a plug) that must eventually be matched to another open end. A plain occupancy bitmask says which positions are “active,” but not which ones are connected to which. If two open ends exist on the frontier, the future must know whether they belong to the same component (connecting them later would create a forbidden loop) or different components (safe to connect).

10.4 Connectivity Profile DP

Instead of each frontier slot storing one bit (occupied / not), it stores a component label. Slots with the same label are currently connected to each other. Example frontier state: 0 1 1 2 0 2 means slots 1 and 2 (0-indexed) share component 1, and slots 3 and 5 share component 2, and slots 0 and 4 are not connected to any open path. Labels themselves are arbitrary — only the equality pattern matters (who is connected to whom).

🔥

Critical implementation detail: After each transition, the component labels in the new frontier may have gaps or inconsistencies. For example, if two components merge, the label numbers may become non-contiguous. Without renaming labels to the smallest available sequence (canonical relabelling), the same logical frontier state appears under many numeric encodings — exploding the state space and causing incorrect memoisation. Always canonicalise labels (renumber smallest-first) after every transition step.

10.5 Connected-Component DP

When processing the next cell, you can: create a new component (start a new path), extend an existing component (continue a path), merge two components (connect two path ends), or close a component (complete a loop). The frontier state must tell you exactly which of those actions is legal — specifically, whether a merge would close a partial loop prematurely (potentially illegal if the problem requires exactly one Hamiltonian cycle over the full grid).

10.9 Complexity and Recognition

If frontier width is W, the number of occupancy states is 2W and the number of connectivity states is the Bell number B(W). Bell numbers grow as roughly (W/ln W)W — much slower than 2W in absolute terms, but still exponential. For W=8, B(8)=4140; for W=12, B(12)=4213597. Feasible for small W, quickly infeasible for large W. The grid’s long dimension (say height) contributes only O(height) factor, so large-height small-width grids are exactly the sweet spot for this technique.

one dimension is small (W ≤ 10–15) sweeping row by row makes sense local placement interacts with the boundary occupancy or connectivity of frontier determines legality “tile a grid” or “count Hamiltonian paths in a narrow grid” both dimensions are large (frontier DP will be too slow) problem has no local frontier-boundary structure

11. Cross-Category Proof Habits PROOF

At this level, the hardest part is often not deriving the recurrence — it is proving that the state is sufficient. These four questions form the standard proof checklist for any DP, at any level.

① State Sufficiency

If two histories lead to the same state value, is the best achievable future identical? If no, the state is missing something. Add the missing information as a new dimension.

② Transition Completeness

Did we enumerate every legal final choice, last segment, split point, or predecessor? If any case is missing, the recurrence is incomplete and will produce wrong answers.

③ Transition Safety

Did any transition allow an illegal object? If yes, the recurrence overcounts (for counting DP) or optimizes over illegal states (for optimization DP).

④ Fill-Order Correctness

Are all smaller dependencies computed before a state is read? If no, the fill order is wrong and some states will be read before they have their final value.

14. Common Misclassification Traps PITFALLS

💡

Trap 1: “String problem” does not automatically mean LCS or edit distance. It might be palindrome interval DP, segmentation DP, automaton DP, or constructive digit-like DP on strings. Ask: does it involve two strings? Does it involve contiguity? Does it involve valid splits?

💡

Trap 2: “Array partition” does not mean interval DP. If you only care about prefixes and the last cut, it is partition DP. Interval DP solves arbitrary [l,r] subranges, not just prefixes. The fill order is the giveaway: ascending index = partition DP, ascending interval length = interval DP.

💡

Trap 3: “Small n” does not automatically mean bitmask DP. Ask whether the subset itself is the natural state. If the object is a tree or interval, the right structure might be tree DP or interval DP even with small n. Small n is a necessary but not sufficient condition for bitmask DP.

💡

Trap 4: “Graph problem” does not automatically mean shortest path or DFS. If every subtree or component returns a summary to combine with its parent, it is DP. The test: does solving a subpart help solve the larger part without re-examining old data? If yes, it is DP.

💡

Trap 5: “Need exactly K operations, turns, or segments” almost always means one extra state dimension. Never try to handle “exactly K” inside the transition logic alone; give it its own axis in the state.

15. Final Mental Model SYNTHESIS

All advanced DP reduces to one question:
“What is the smallest state that preserves every piece of information the future still needs?”

Once you answer that well, the rest is routine: write the recurrence, choose the fill order, prove cut / last-choice / merge correctness, analyze dimensions, and optimize later if needed.

The ten families in this tutorial are just the recurring answers people discovered for that question across decades of competitive programming:

Knapsack

Answers it with resource dimensions (capacity, constraints)

Sequence DP

Answers it with position and local relational property

Partition DP

Answers it with last-cut choice over prefixes

Interval DP

Answers it with a contiguous subrange [l,r]

Subset DP

Answers it with a bitmask of used elements

Tree DP

Answers it with subtree summaries merged bottom-up

Counting DP

Answers it with disjoint combinatorial branches (sum instead of max)

Grid DP

Answers it with position plus local motion / history state

Digit DP

Answers it with a prefix-bounded digit state and tight flag

Frontier DP

Answers it with the connectivity/occupancy pattern at the boundary

And that is why, even at the hardest level, this is still the same subject from the first tutorial: define state, justify state, transition safely, fill in dependency order, prove future-sufficiency. The object changed. The law did not.

🏆

Recommended study order for this sequel: Advanced Knapsack → Advanced Sequence/String → Segmentation/Partition → Advanced Interval → Advanced Subset/Bitmask → Advanced Tree/Graph DP → Counting DP → Advanced Grid → Advanced Digit → Frontier/Profile DP. That path moves from “same old state with one twist” toward “frontier topology compression,” which is the hardest.
