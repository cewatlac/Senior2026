0. What Is Dynamic Programming? FROM ZERO

DP = solve smaller subproblems once, store their answers, reuse them forever

Dynamic programming is not a single algorithm. It is a problem-solving strategy that turns exponential brute-force search into polynomial computation by recognising a single fact: you are solving the same small problem over and over again. Once you see that, you simply store each answer the first time and return it instantly every subsequent time.

The best way to understand this is to watch brute-force recursion fail on a small example, see exactly which work is repeated, and then see how storing answers eliminates that waste. We will do this completely from scratch using Fibonacci as our first lens — not because Fibonacci is important, but because it is so small that nothing can hide.

Step 1 — The brute-force recursion and why it explodes

The definition of Fibonacci is F(0)=0, F(1)=1, F(n)=F(n−1)+F(n−2). If you turn this definition directly into a recursive function, you get the following call tree for computing F(5). Every box below is a function call that your computer must execute.

Count the boxes carefully. F(3) appears 2 times. F(2) appears 3 times. F(1) appears 5 times. The number of total calls for F(n) follows the Fibonacci sequence itself — it grows as O(2ⁿ). For n=50 that exceeds one quadrillion calls. The recursion is logically correct; it is just absurdly inefficient because it solves the same tiny problems again and again from scratch.

Step 2 — Memoization: store what you compute

The fix is beautifully simple. The first time you compute F(k), write the answer down somewhere (a cache, array, or hash-map). The next time F(k) is requested — from anywhere in the tree — return the cached answer immediately without entering the recursion at all. The entire recursive tree collapses to a straight chain of n calls, one per unique subproblem. This is called top-down DP or memoization.

Step 3 — Tabulation: build from the bottom up

There is a second equivalent approach that avoids recursion entirely. Instead of starting at F(n) and going down, you start at F(0) and F(1) (which you know immediately) and build upward: F(2), F(3), F(4), …, F(n). Each new value uses only already-computed values. This is called bottom-up DP or tabulation. Both produce identical results; they differ only in the order in which states are evaluated.

💡

Mental model: DP = solve smaller subproblems, store each answer once, reuse stored answers instead of recomputing. That is all it ever is — the complexity of a specific problem comes from choosing the right subproblem definition, not from the idea itself.

When does DP apply?

A problem is a candidate for DP when two structural properties hold simultaneously. First, optimal substructure: the best solution to the full problem can be assembled from best solutions to its subproblems (if you replace any subpart with something worse, the whole becomes worse). Second, overlapping subproblems: a naive recursion would visit the same subproblem multiple times from different branches. If both properties hold, DP will reduce exponential time to polynomial.

1. The DP Design Framework CHECKLIST

Before writing a single line of code for any DP problem, force yourself to answer six questions in order. Skipping even one of them is where most DP bugs begin. This checklist is not a formality — it is the actual thinking process that produces correct DP solutions.

1 State

What exactly is the subproblem?
What does dp[…] represent?
Write it as one precise sentence.

2 Transition

How is the answer for this state assembled from smaller states? Tell it as a story: "the last step was…" or "the first cut was…"

3 Base Case

What are the smallest states whose answer is known immediately without any recursion? (Empty set, single item, zero sum, etc.)

4 Fill Order

In what order must states be computed so that every dependency is ready before it is needed? (Iterative: increasing index, by length, row-major, etc.)

5 Final Answer

Which specific state (or range of states) gives the answer to the original question? It is not always the last cell!

6 Complexity

Time ≈ (number of states) × (work per state). Space = size of DP table (or less if rolling arrays are applicable).

⭐ The 5-Step Beginner Method — used for every problem in this guide

The 6-question checklist above tells you what to figure out. The 5-step method below tells you what to write, in what order, when you sit down to solve a brand-new DP problem. Apply both: think with the checklist, code with the 5 steps.

StepWhat you writeWhy this exact step
1 Recursion A plain recursive function that defines the answer in terms of smaller versions of the same problem. Identify the function arguments (the state), write the base cases, write the choices/transitions. Do not think about ordering, tables, or memory. Then explain in one sentence why this recursion repeats work. Forces you to find the recurrence — the heart of every DP. Until this is right, nothing else can be right.
2 Memoize Add a cache (an array or map) indexed by the function arguments. At the top of the function: if the cache holds an answer for this state, return it. At the bottom: store the result before returning. This is top-down DP. One tiny edit turns exponential time into polynomial. If you only need the optimal value, you can stop here.
3 Build (reconstruct the answer) Most problems do not just want the optimal value — they want the actual choices (which items, which path, which characters). Write a small recursive build(state) that re-walks the memoized table, and at every state asks: "Which of my options produced the value I stored?", then prints/collects the choice and recurses into the chosen sub-state. Without this step you only have a score, not a solution. Doing it now (still in recursive mode) is much easier than from an iterative table.
4 Iterate (full table) Now ask the ordering question: "to compute dp[state], which other states do I need already filled?" Pick a loop order that fills dependencies first. Use the full DP table (no shortcuts yet). Removes recursion overhead and stack-depth risk. Makes dependencies visible at a glance, which is essential for the next step.
5 Reduce memory Look at the iteration. At the moment you compute dp[i], which earlier cells will you ever read again? Only those need to live. Overwrite the rest by collapsing the table: 2D → 1D, or n cells → 2 variables, or whatever the dependency reach allows. Often shrinks memory from O(n²) → O(n) or O(n) → O(1). Easy WIN once iteration is right — and impossible to do correctly if iteration is wrong.
📘

About Step 3 — "the build function": A DP function like best(state) returns the optimal value. But the user often wants the solution itself: which items did you pick, which path did you take, what's the actual longest subsequence string? The build function answers that. It re-runs the same recursive logic, but at each state, instead of returning a number, it asks "when I computed this state, which option gave me the value I stored? That's the choice I took." Then it recurses into that chosen branch. We do this in Step 3 (still in recursive mindset) because once we move to iterative tables it becomes much harder to reason about.

⚠

Why this exact order matters. Most beginners try to write the optimized iterative version directly — and get stuck for hours because they conflated "what to compute" (Step 1), "how to recover the actual answer" (Step 3), and "how to lay it out efficiently" (Step 5) into one giant unsolvable thinking task. Trust the 5 steps: each one is small and easy on its own. Skipping steps is the #1 reason beginners give up on DP.

The four answer types — choose before anything else

Decision
Is it possible?

Stores true/false. Operator: OR. Seed: true at identity state. Example: can I form sum X?

Optimization
Min or Max value?

Stores min/max. Initialize impossible states with ∞ or −∞. Operator: min/max. Example: fewest coins, max knapsack.

Counting
How many ways?

Stores count. Operator: sum. Identity seed = 1 (one way to do nothing). Example: grid paths, coin combinations.

Construction
Print the answer

Any of the above, plus a witness (predecessor, split, take-flag). The value alone is never enough to reconstruct.

⚠ Critical warning
Same state — different problem

dp[x] for coin problems looks identical whether you want fewest coins (min), ordered sequences (sum + sum-outer loop), or unordered combinations (sum + coin-outer loop). The state is the same; the operator and loop structure determine which problem you actually solve.

2. Fibonacci 1D RECURRENCE

Fibonacci is the minimal DP problem. Its purpose here is not to teach you about rabbit populations — it is to make every DP mechanic (brute-force failure, memoization, tabulation, and space compression) visible in a setting where the logic is so transparent that nothing can hide behind complexity.

Problem

Compute F(n) where F(0)=0, F(1)=1, F(n)=F(n−1)+F(n−2) for n≥2.

Brute Force

The recursive function fib(n) = fib(n-1) + fib(n-2) directly encodes the definition. It is correct. It is also catastrophically slow. The recursion tree has O(2ⁿ) nodes because every call spawns two sub-calls, many of which are identical. For n=50 that is over a quadrillion calls.

Why It Fails

The brute-force tree is not a tree of distinct problems. It is a tree of repeated evaluations. F(48) gets computed once from the left branch of F(50) and independently again from the left branch of F(49), and so on exponentially. There is no bug in the logic — only in the refusal to remember what was already computed.

Overlap

Every call F(k) for k ≥ 2 gets re-evaluated an exponential number of times by different branches. The number of distinct subproblems is only n+1 (one for each value from 0 to n). The gap between "number of calls" (O(2ⁿ)) and "number of distinct subproblems" (O(n)) is the exact waste that memoization eliminates.

State

dp[i] = F(i), the i-th Fibonacci number

Transition

dp[i] = dp[i-1] + dp[i-2]

Story: "F(i) is the sum of the previous two Fibonacci numbers — there is no choice here, just a pure addition of two already-known values."

Base Cases

dp[0] = 0 and dp[1] = 1. These come directly from the definition and cannot be expressed in terms of smaller Fibonacci numbers.

Step 1 — Recursion (the brute force, restated)

The plain recursive function fib(n) directly mirrors the mathematical definition. The function arguments are just one integer n — that is the state. The two base cases are n == 0 and n == 1 because we know those answers without any computation. For any larger n the function makes two recursive choices and adds them.

long long fib(int n) {
    if (n == 0) return 0;            // base case
    if (n == 1) return 1;            // base case
    return fib(n-1) + fib(n-2);      // sum of the two smaller answers
}

This is correct but exponential — see the call tree above for why.

Step 2 — Memoize (top-down DP)

Add a cache (an array) indexed by the state n. We use the versioned-memoization template (a saved[] array tagged with the current test-case id sid) so that we never have to clear the cache between test cases — see §16.5 for the full explanation. Top of the function: if the cache already holds an answer for the current test case, return it immediately. Bottom: tag the slot and store the result before returning.

const int MAXN = 100001;
long long memo[MAXN];
int saved[MAXN], sid;        // sid is global; per test case do ++sid;

long long fib(int n) {
    if (n <= 1) return n;                    // base cases — constants, no caching needed
    auto& ret = memo[n];
    if (saved[n] == sid) return ret;         // cache hit for THIS test case
    saved[n] = sid;                           // mark this state as computed in this test
    return ret = fib(n-1) + fib(n-2);         // compute & store in ret (= memo[n])
}

// per test case:
//   ++sid;                                  // O(1) "logical reset" of the whole memo
//   cout << fib(n) << '\n';

Each value F(0), F(1), …, F(n) is now computed at most once. Total work drops from O(2ⁿ) to O(n).

Step 3 — Build (reconstruct the answer)

Fibonacci is a special case: there is no choice to recover. The recurrence has only one option (sum the two smaller values), so there is nothing to "pick". The "answer" is just the single number F(n). For most other DP problems, however, the answer is a sequence of choices, and Step 3 is essential. To illustrate the shape a build function takes, here is one that prints the chain of recursive calls that produced F(n):

// Demonstration only — Fibonacci has no real choices to reconstruct.
void buildFib(int n, int depth = 0) {
    for (int i = 0; i < depth; i++) cout << "  ";
    long long val = (n <= 1) ? n : memo[n];        // base cases short-circuited fib(); read directly
    cout << "F(" << n << ") = " << val << '\n';
    if (n <= 1) return;
    buildFib(n-1, depth+1);   // there is no max() here — both branches always taken
    buildFib(n-2, depth+1);
}

Starting from Problem 3 (0/1 Knapsack) you will see real build functions that choose at every state by comparing values — that's where the pattern earns its keep.

Step 4 — Iterate (full table, no memory tricks)

Ordering question: to compute dp[i] we need dp[i-1] and dp[i-2] — both have smaller index than i. So filling the table left-to-right (i = 0, 1, 2, …, n) guarantees every dependency is ready before we touch it.

vector<long long> dp(n+1);
dp[0] = 0;                          // base case
dp[1] = 1;                          // base case
for (int i = 2; i <= n; i++)
    dp[i] = dp[i-1] + dp[i-2];      // dependencies dp[i-1], dp[i-2] are filled
// Answer is dp[n]

O(n) time, O(n) space — no recursion overhead, no stack-depth risk.

Step 5 — Reduce memory (overwrite stale cells)

Look at the iteration. At the moment we compute dp[i] we use only dp[i-1] and dp[i-2]. We never read dp[i-3] or earlier ever again — those cells are stale. Two variables ("previous" and "previous-previous") are enough.

long long a = 0, b = 1;             // a = F(0), b = F(1)
for (int i = 2; i <= n; i++) {
    long long c = a + b;            // c = F(i)
    a = b;                          // shift the window forward
    b = c;
}
// Answer is b (= F(n)). Special case: if n == 0 the answer is a.

O(n) time, O(1) space — same answer, ~1000× less memory.

Complexity

⏱

Time: O(n) — exactly n−1 additions, one per cell. Compare with O(2ⁿ) brute force.

🗂

Space: O(n) array, or O(1) with the two-variable rolling trick since dp[i] depends only on dp[i-1] and dp[i-2].

Pitfalls

  • Using plain recursion without memoization — will TLE for n ≥ 40.
  • Integer overflow: F(93) exceeds int64_max. Use Python or modular arithmetic for large n.
  • Off-by-one: if the problem defines F(1)=1, F(2)=1 (1-indexed), shift your array by one.

3. 0/1 Knapsack KNAPSACK FAMILY

Knapsack is the prototype for an entire family of choose-under-capacity problems. The "0/1" means each item is either taken once (1) or skipped (0). Understanding this problem deeply — especially the backward loop proof — unlocks every knapsack variant.

Problem

You have n items. Item i has weight w[i] and value v[i]. Your knapsack holds capacity W. Each item can be taken at most once. Maximise total value subject to total weight ≤ W.

Brute Force

Try every subset of n items: 2ⁿ subsets total. For each, check whether it fits and compute its value. Time: O(2ⁿ). For n=40 that is over a trillion subsets — completely infeasible.

Why It Fails & Overlap

Consider the subset decision tree: at each level you decide "take item i or skip it". Two branches that make different choices for item 1 but the same choices for items 2…n arrive at the same remaining problem. That remaining problem — "best value using items 2…n with capacity cap" — is solved from scratch in both branches. This overlap grows exponentially with n.

State

dp[i][cap] = maximum value using items {1…i} with remaining capacity exactly cap

Once you know how many items have been considered (i) and how much capacity remains (cap), the history of which specific items were taken does not affect future decisions. These two numbers capture everything the future needs.

Transition

Story: "For item i, I have exactly two choices: skip it (capacity unchanged, value unchanged) or take it if it fits (capacity decreases by w[i], value increases by v[i]). Take the better option."

dp[i][cap] = dp[i-1][cap] (skip item i)
dp[i][cap] = max(dp[i-1][cap], v[i] + dp[i-1][cap − w[i]]) if w[i] ≤ cap

Base Cases

dp[0][cap] = 0 for all cap: with zero items available, the maximum value is 0 regardless of capacity. Also dp[i][0] = 0 for all i: with zero capacity, nothing can be taken.

Step 1 — Recursion (the brute force, restated)

Define best(i, cap) = "the maximum value I can achieve considering the first i items with cap capacity remaining". The recursion just spells out the two choices for the i-th item.

int n, W;
vector<int> w, v;       // 1-indexed: w[1..n], v[1..n]

long long best(int i, int cap) {
    if (i == 0) return 0;                              // no items left → value 0
    long long skip = best(i-1, cap);                   // option A: don't take item i
    long long take = 0;
    if (w[i] <= cap)                                   // option B: take item i if it fits
        take = v[i] + best(i-1, cap - w[i]);
    return max(skip, take);
}
// answer = best(n, W);

Each call branches into 2, giving O(2ⁿ) time. The same (i, cap) pair is asked again and again from many different paths — that's the overlap memoization will eliminate.

Step 2 — Memoize (top-down DP)

The state is two integers (i, cap), so the cache is a 2D array of size (n+1) × (W+1). We use the versioned-memoization template (see §16.5): a parallel saved[][] table tags every cell with the test-case id sid; bumping sid at the start of a test case logically clears the whole cache in O(1).

// Pick MAXN, MAXW to match the problem's constraints. Example values:
const int MAXN = 105, MAXW = 1005;
long long memo[MAXN][MAXW];
int saved[MAXN][MAXW], sid;

long long best(int i, int cap) {
    if (i == 0) return 0;                              // base — constant, no caching
    auto& ret = memo[i][cap];
    if (saved[i][cap] == sid) return ret;              // cache hit for THIS test case
    saved[i][cap] = sid;                               // tag this state as visited now
    long long skip = best(i-1, cap);
    long long take = (w[i] <= cap) ? v[i] + best(i-1, cap - w[i]) : 0;
    return ret = max(skip, take);                      // store in ret (= memo[i][cap])
}

// per test case:
//   ++sid;
//   cout << best(n, W) << '\n';

Now each (i, cap) pair is computed at most once per test case. O(n × W) time, O(n × W) memo space — and no memset between test cases.

Step 3 — Build (reconstruct which items were taken)

The memo only tells us the best value. The user almost always wants the list of items. The build function re-walks the table, and at every state asks: "At (i, cap), did I skip item i or did I take it?" The answer is whichever option produced memo[i][cap]. We start at (n, W) and recurse to smaller states until i == 0.

vector<int> chosen;       // will hold item indices that were taken

void buildKnapsack(int i, int cap) {
    if (i == 0) return;
    // Did I take item i? Compare against the value if I had skipped it.
    long long skipVal = best(i-1, cap);
    if (memo[i][cap] != skipVal) {       // value changed → I MUST have taken item i
        chosen.push_back(i);
        buildKnapsack(i-1, cap - w[i]);  // recurse into the "took it" sub-state
    } else {
        buildKnapsack(i-1, cap);         // I skipped — recurse with same capacity
    }
}
// usage:
//   long long answer = best(n, W);
//   buildKnapsack(n, W);
//   for (int x : chosen) cout << x << ' ';
📘

How the build function "decides": for each state we know the stored optimum. We re-evaluate just one of our original options — say, the "skip" option — and compare. If the stored value matches the skip value, we skipped. Otherwise we took. We then recurse into the sub-state that matches the choice we just identified. We never recompute anything new — every best(...) call hits the cache.

Step 4 — Iterate (full table, no memory tricks)

Ordering question: dp[i][cap] reads dp[i-1][cap] and dp[i-1][cap - w[i]] — both at row i-1. So filling the table row by row, top to bottom (i from 1 to n), guarantees the row above is always ready. Within a row, the order of cap doesn't matter (we only read row i-1).

This is the version with reconstruction support — the auxiliary take[][] table records the choice at each cell so we can rebuild the answer later.

#include <bits/stdc++.h>
using namespace std;
int main(){
    int n, W; cin >> n >> W;
    vector<int> w(n+1), v(n+1);
    for(int i=1;i<=n;i++) cin>>w[i]>>v[i];

    // dp[i][cap]: max value using first i items, capacity cap
    vector<vector<long long>> dp(n+1, vector<long long>(W+1,0));
    vector<vector<int>> take(n+1, vector<int>(W+1,0));   // 1 if item i was taken at this cell

    for(int i=1;i<=n;i++){
        for(int cap=0;cap<=W;cap++){
            dp[i][cap] = dp[i-1][cap];   // default: skip
            if(w[i]<=cap){
                long long cand = v[i] + dp[i-1][cap-w[i]];
                if(cand > dp[i][cap]){ dp[i][cap]=cand; take[i][cap]=1; }
            }
        }
    }
    cout << dp[n][W] << '\n';

    // Reconstruct which items were taken (same idea as the build function above,
    // but now driven by the iterative table directly)
    vector<int> chosen; int cap=W;
    for(int i=n;i>=1;i--) if(take[i][cap]){ chosen.push_back(i); cap-=w[i]; }
    reverse(chosen.begin(),chosen.end());
    for(int x:chosen) cout<<x<<' ';
}

Step 5 — Reduce memory: 2D → 1D, the backward loop proof

The 2D recurrence reads from row i−1 only. So we can collapse to a single array and update it in-place. The critical question is: when we compute dp[cap] for item i, does dp[cap−w[i]] still represent "row i−1" or has it already been updated to "row i"? If we loop forward (cap from 0 to W), dp[cap−w[i]] would already be updated in this same pass — meaning item i could be "taken" multiple times, silently converting 0/1 into unbounded knapsack. Looping backward (cap from W down to w[i]) ensures dp[cap−w[i]] has not yet been touched in this pass (since cap−w[i] < cap), so it still holds the previous row's value.

// O(W) space — value only (reconstruction requires the full table)
vector<long long> dp(W+1, 0);
for(int i=1;i<=n;i++)
    for(int cap=W; cap>=w[i]; cap--)   // BACKWARD = 0/1 (each item once)
        dp[cap] = max(dp[cap], v[i]+dp[cap-w[i]]);

Complexity

⏱

Time: O(nW) — n items × W+1 capacities, O(1) work per cell.

🗂

Space: O(nW) full table (for reconstruction); O(W) compressed (value only).

Pitfalls

  • Using greedy by value/weight ratio — works for fractional knapsack, fails for 0/1.
  • Forward loop in 1D version — silently becomes unbounded knapsack (items reused).
  • Compressing to 1D before confirming reconstruction is not needed.
  • Initialising impossible minimum states to 0 instead of ∞ in decision variants.

Each book has a price and a page count; you have budget X and want to maximise pages. This is exactly 0/1 knapsack with price=weight, pages=value, X=W. The value of this problem is in the habit it builds: converting story language into the abstract DP template you already know.

4. Coin Change Family KNAPSACK FAMILY

This family is one of the most important teaching tools in competitive DP because three superficially identical problems require different code. The failure to distinguish them is among the most common contest bugs. We will build each variant from brute force and show precisely where the divergence happens.

Variant A — Minimum Coins (Optimization)

Problem: Using unlimited copies of given coin denominations, find the fewest coins to make sum X. Brute force: try every combination — exponential. Overlap: "minimum coins for sum 7" is needed by "minimum for sum 8 using coin 1", "minimum for sum 5 using coin 3-wait, actually for sum 10 using coin 3", and so on — the same target sum reappears from many branches.

dp[x] = 1 + min { dp[x−c] } over all coins c ≤ x

Seed: dp[0]=0. All other values initialised to ∞ (unreachable). Story: "the last coin placed was c, so before it we needed to make sum x−c optimally."

Step 3 — Build (which coins did we actually use?)

The DP value alone tells us "the best is 3 coins" — but the user usually wants to know which coins. The recursive build function asks at every state: "To make sum x optimally, which last coin produced the value dp[x]?" That is, find a coin c such that dp[x − c] + 1 == dp[x]; record it and recurse on x − c.

vector<int> usedCoins;
void buildCoins(int x, vector<int>& coins, vector<int>& dp) {
    if (x == 0) return;                            // base: nothing left to make
    for (int c : coins)
        if (c <= x && dp[x-c] + 1 == dp[x]) {     // this coin matches the optimum
            usedCoins.push_back(c);
            buildCoins(x - c, coins, dp);
            return;                                // pick exactly one coin per call
        }
}

Variant B — Ordered Sequences (Counting, order matters)

Problem: Count the number of distinct ordered sequences of coins summing to X. [1,2] and [2,1] are different. Story: "what was the last coin placed?" For each possible last coin c, the preceding sequence summed to x−c. Summing over all choices of last coin gives the recurrence.

dp[x] = sum { dp[x−c] } over all coins c ≤ x

Seed: dp[0]=1 ("one way to make sum 0: place nothing"). Loop: sum on outside, coin on inside — this naturally captures all orderings because for each target x we ask "what was last?"

Variant C — Unordered Combinations (Counting, order doesn't matter)

For each coin c: dp[x] += dp[x−c], for x from c to X

5. LIS — Longest Increasing Subsequence SEQUENCE DP

LIS is the prototype for sequence DP. The key insight is that the state needs to describe only where the subsequence ends — not what the entire subsequence looks like. This compression from exponential state space to linear is the pattern that recurs throughout sequence DP.

Problem

Given array a[0..n−1], find the length of the longest strictly increasing subsequence (elements can be non-contiguous but must maintain their relative order).

Brute Force

Try all 2ⁿ subsequences. For each, check if it is strictly increasing and track the longest. Time: O(2ⁿ · n). For n=50 this is completely infeasible.

Why It Fails & Overlap

Consider any two valid increasing subsequences that end at the same position i with the same length. All future extensions of both are identical — they can append the same future elements. So "longest increasing subsequence ending at i" is a well-defined subproblem that many branches of the brute-force search implicitly solve multiple times. Memoising it eliminates the redundancy.

State

dp[i] = length of the longest strictly increasing subsequence ending exactly at position i

The word "exactly" is important. dp[i] is not "the best LIS in a[0..i]" — it is the best LIS that must include a[i] as its last element. This makes the transition clean because we know the last element and can check the ≤ condition directly.

Transition

Story: "To build the LIS ending at i, I look backward at every j < i where a[j] < a[i]. I can extend the LIS ending at j by appending a[i]. The best I can do is 1 + the maximum dp[j] over all valid predecessors j."

dp[i] = 1 + max { dp[j] : j < i and a[j] < a[i] }

If no valid predecessor exists (no earlier element is smaller), dp[i] = 1 (a[i] alone is a valid LIS of length 1).

Base Case & Final Answer

Every dp[i] starts at 1 (the element by itself). The final answer is max(dp[i]) over all i — not dp[n−1]! The LIS can end anywhere in the array.

Step 1 — Recursion

Define lisEnd(i) = "the length of the longest strictly increasing subsequence that ends exactly at position i". The recursion looks at every earlier index j with a[j] < a[i] and asks: "can I extend the LIS that ends at j by appending a[i]?" Take the best such extension; if none exists, the answer is just 1 (a[i] alone).

int lisEnd(int i, vector<int>& a) {
    int best = 1;                       // the element itself
    for (int j = 0; j < i; j++)
        if (a[j] < a[i])
            best = max(best, 1 + lisEnd(j, a));
    return best;
}
// answer = max over i of lisEnd(i)

Step 2 — Memoize

The state is one integer i, so the cache is a 1D array. We use the versioned-memoization template (see §16.5) — no memset is needed between test cases; one ++sid at the start of each test case is enough.

const int MAXN = 100005;
int memo[MAXN];
int saved[MAXN], sid;

int lisEnd(int i, vector<int>& a) {
    auto& ret = memo[i];
    if (saved[i] == sid) return ret;          // cache hit for THIS test case
    saved[i] = sid;
    int best = 1;
    for (int j = 0; j < i; j++)
        if (a[j] < a[i]) best = max(best, 1 + lisEnd(j, a));
    return ret = best;
}

Step 3 — Build (recover the actual subsequence)

The memo only tells us lengths. The user almost always wants the actual subsequence. The build function asks: "I'm at position i with stored length memo[i]. Which earlier index j was the best predecessor?" That is, find j < i with a[j] < a[i] and memo[j] + 1 == memo[i]. Record a[i] and recurse on j. Stop when the length becomes 1 (no predecessor — a[i] is the start).

vector<int> lisAns;
void buildLIS(int i, vector<int>& a) {
    lisAns.push_back(a[i]);              // include a[i] in the LIS
    if (memo[i] == 1) return;            // a[i] starts the LIS — done
    for (int j = 0; j < i; j++)
        if (a[j] < a[i] && memo[j] + 1 == memo[i]) {
            buildLIS(j, a);              // recurse into the chosen predecessor
            return;
        }
}

//        then reverse(lisAns) to get it in left-to-right order.

Step 4 — Iterate (full table) with predecessor pointers

Ordering question: dp[i] reads dp[j] for j < i, so filling left-to-right (i = 0, 1, …, n−1) ensures every dependency is already filled. As a bonus, store the chosen predecessor in par[i] so reconstruction is just a backward walk.

#include <bits/stdc++.h>
using namespace std;
int main(){
    int n; cin>>n;
    vector<int> a(n); for(auto&x:a)cin>>x;

    vector<int> dp(n,1), par(n,-1);
    int bestLen=0, bestEnd=-1;

    for(int i=0;i<n;i++){
        for(int j=0;j<i;j++)
            if(a[j]<a[i] && dp[j]+1>dp[i]){
                dp[i]=dp[j]+1;
                par[i]=j;         // record best predecessor
            }
        if(dp[i]>bestLen){ bestLen=dp[i]; bestEnd=i; }
    }
    cout << bestLen << '\n';

    // Reconstruct LIS by following par[] backward (iterative version of buildLIS)
    vector<int> lis;
    for(int cur=bestEnd; cur!=-1; cur=par[cur]) lis.push_back(a[cur]);
    reverse(lis.begin(),lis.end());
    for(int x:lis) cout<<x<<' ';
}

Step 5 — Reduce memory? (and the speed-trade alternative)

The dp[] array cannot be shrunk below O(n) — when we compute dp[i] we may look at any earlier dp[j], so every cell is alive throughout the loop. Memory reduction is impossible. But there is a different trade: we can give up the simple recurrence in exchange for a faster algorithm.

O(n log n) optimisation via patience sorting

The O(n²) DP is the correct conceptual foundation. For competitive programming with n up to 10⁵, there is an O(n log n) algorithm that maintains an array tails where tails[k] = the smallest possible last element of any increasing subsequence of length k+1 seen so far. For each new element a[i], binary search tails to find its position, update it, and the length of tails at the end equals the LIS length. This does not change the DP idea — it is a clever data structure optimisation of the same recurrence.

⏱

O(n²) DP — n positions × n predecessors = O(n²). Clean, correct, sufficient for n ≤ 5000.

⏱

O(n log n) patience sort — binary search tails array. Required for n ≤ 2×10⁵.

Pitfalls

  • Reading dp[n−1] as the answer — the LIS can end anywhere; take max over all dp[i].
  • Confusing subsequence (skip allowed) with subarray (must be contiguous).
  • Using ≤ instead of < for strictly increasing — allows equal elements to extend the LIS incorrectly.

6. LCS — Longest Common Subsequence 2D PREFIX DP

LCS is the prototype for 2D prefix DP — problems where the state is a pair of prefix lengths. The insight is that looking at the last characters of two prefixes gives a clean binary case analysis, which becomes the recurrence naturally.

Problem

Given strings A (length n) and B (length m), find the length of their longest common subsequence — the longest sequence of characters that appears in both strings in the same relative order.

Brute Force & Overlap

Try every subsequence of A (there are 2ⁿ of them) and check if it is a subsequence of B. Time: O(2ⁿ · m). The overlap: computing "LCS of A[1..5] and B[1..6]" is needed by multiple parent subproblems — this pair of prefix lengths appears in many branches of the recursion tree.

State, Transition, Base

dp[i][j] = LCS length of A[0..i-1] and B[0..j-1]
if A[i-1] == B[j-1]: dp[i][j] = dp[i-1][j-1] + 1
otherwise: dp[i][j] = max(dp[i-1][j], dp[i][j-1])

Base: dp[i][0] = dp[0][j] = 0 for all i, j. Final answer: dp[n][m].

Step 1 — Recursion

Define lcs(i, j) = "length of the LCS of the first i characters of A and the first j characters of B". Base case: if either prefix is empty, the LCS is 0. Otherwise look at the last characters and apply the case analysis above.

int lcs(int i, int j, string& A, string& B) {
    if (i == 0 || j == 0) return 0;
    if (A[i-1] == B[j-1]) return 1 + lcs(i-1, j-1, A, B);
    return max(lcs(i-1, j, A, B), lcs(i, j-1, A, B));
}
// answer = lcs(n, m, A, B)

The memo stores only LENGTHS. The user almost always wants the actual common-subsequence STRING. The build function asks at each state: "At (i, j), which option was taken — match, drop A's last, or drop B's last?"

string built = "";
void buildLCS(int i, int j, string& A, string& B) {

    if (A[i-1] == B[j-1]) {                   // characters match → include in LCS
        buildLCS(i-1, j-1, A, B);
        built += A[i-1];

    else if (memo[i-1][j] >= memo[i][j-1])    // dropping A's last gave the better answer
        buildLCS(i-1, j, A, B);
    else                                       // dropping B's last gave the better answer
        buildLCS(i, j-1, A, B);
}

7. Edit Distance 2D PREFIX DP

Edit distance (also called Levenshtein distance) uses the same two-prefix table structure as LCS but handles three operations instead of one comparison. It is one of the most practically important DP problems — it powers spell checkers, DNA alignment tools, diff utilities, and fuzzy search engines. Understanding it deeply is essential because the same three-operation structure reappears in dozens of harder problems.

Problem

Given two strings A (length n) and B (length m), find the minimum number of single-character operations needed to convert A into B. The allowed operations are: insert a character into A, delete a character from A, or replace a character in A with a different one. Each operation costs exactly 1.

Example: A = "kitten", B = "sitting". One optimal sequence: replace 'k'→'s', replace 'e'→'i', insert 'g' at end → 3 operations.

Brute Force

Consider building the answer recursively. At each step, look at the last characters of the remaining parts of A and B. If they match, no operation is needed — just move both pointers backward. If they don't match, try all three choices: (1) replace A's character to match B's, (2) delete A's last character, (3) insert B's last character into A. Take the minimum cost of all three paths.

This recursion branches into up to 3 sub-calls per step. In the worst case (no characters match), the recursion tree has O(3ⁿ⁺ᵐ) leaves — completely infeasible for strings longer than about 15 characters.

Where Overlapping Subproblems Appear

Consider converting "ABC" into "AEC". The recursive call for "convert A[1..2] to B[1..2]" (i.e., convert "BC" to "EC") appears from multiple parent branches — from the "match on A" branch and from various insert/delete branches at earlier positions. The key observation is that the answer for "convert the first i characters of A into the first j characters of B" depends only on i and j — not on how we arrived at that pair. This means dp[i][j] is a well-defined subproblem, and there are only (n+1)×(m+1) distinct subproblems. The brute-force tree visits these same subproblems exponentially many times.

State

dp[i][j] = minimum number of operations to convert A[0..i-1] into B[0..j-1]

In words: dp[i][j] answers "what is the cheapest way to turn the first i characters of A into the first j characters of B?" We use 1-indexed prefixes so that dp[0][…] and dp[…][0] handle the natural base cases of empty strings.

Transition

Story: "Look at A[i-1] and B[j-1] — the last characters of the current prefixes. Either they match, or they don't."

Case 1 — Match: If A[i-1] == B[j-1], no operation is needed for these characters. The cost is the same as converting the previous prefixes: dp[i][j] = dp[i-1][j-1].

Case 2 — Mismatch: If they differ, we must use one operation. Three options, each costing 1 plus the cost of the remaining subproblem:

Replace A[i-1] → B[j-1]

After replacing, A[i-1] matches B[j-1]. The remaining problem is converting A[0..i-2] into B[0..j-2]. Cost: 1 + dp[i-1][j-1]

Delete A[i-1]

Remove A's last character. Now convert A[0..i-2] into B[0..j-1] (B unchanged). Cost: 1 + dp[i-1][j]

Insert B[j-1] into A

After inserting, A's new last character matches B[j-1]. The remaining problem is converting A[0..i-1] into B[0..j-2]. Cost: 1 + dp[i][j-1]

if A[i-1] == B[j-1]: dp[i][j] = dp[i-1][j-1]
else: dp[i][j] = 1 + min(dp[i-1][j-1], dp[i-1][j], dp[i][j-1])

Base Cases

dp[i][0] = i for all i. Converting the first i characters of A into the empty string requires deleting all i characters.

dp[0][j] = j for all j. Converting the empty string into the first j characters of B requires inserting all j characters.

These base cases are not arbitrary — they follow directly from the problem definition applied to the boundary where one string is empty.

Final Answer

dp[n][m] — the cost of converting the entire string A into the entire string B.

Fill Order

dp[i][j] depends on dp[i-1][j-1], dp[i-1][j], and dp[i][j-1] — all of which have smaller i or smaller j. Row-major order (i from 0 to n, j from 0 to m) ensures every dependency is computed before it is needed.

Step 1 — Recursion

Define edit(i, j) = "minimum operations to convert A[0..i-1] into B[0..j-1]". Two base cases (one prefix is empty), then the case analysis above.

int edit(int i, int j, string& A, string& B) {
    if (i == 0) return j;       // insert all of B
    if (j == 0) return i;       // delete all of A
    if (A[i-1] == B[j-1]) return edit(i-1, j-1, A, B);     // match — free
    return 1 + min({edit(i-1, j-1, A, B),   // replace
                    edit(i-1, j,   A, B),   // delete A's last
                    edit(i,   j-1, A, B)}); // insert B's last
}

Step 2 — Memoize

The state is two integers (i, j). Versioned-memoization template (see §16.5) — no memset between test cases.

    saved[i][j] = sid;

Step 3 — Build (recover the actual operation list)

The DP value tells us "the answer is 3 operations" — the user wants to know which 3 operations. The build function asks at each (i, j): "Which option produced memo[i][j] — match, replace, delete, or insert?" Compare against each candidate and take whichever option matches the stored value.

vector<string> ops;

        buildEdit(i-1, j-1, A, B);

    else if (i > 0 && j > 0 && memo[i][j] == memo[i-1][j-1] + 1) {
        buildEdit(i-1, j-1, A, B);
        ops.push_back("Replace A[" + to_string(i-1) + "]='" + A[i-1] + "' with '" + B[j-1] + "'");
    }
    else if (i > 0 && memo[i][j] == memo[i-1][j] + 1) {
        buildEdit(i-1, j, A, B);
        ops.push_back("Delete A[" + to_string(i-1) + "]='" + A[i-1] + "'");
    }
    else {                                                 // insert
        buildEdit(i, j-1, A, B);
        ops.push_back("Insert '" + string(1, B[j-1]) + "'");

    int i = n, j = m;
    vector<string> ops;
    while(i > 0 || j > 0){
        if(i > 0 && j > 0 && A[i-1] == B[j-1]){
            i--; j--;  // match, no operation
        } else if(i > 0 && j > 0 && dp[i][j] == dp[i-1][j-1]+1){
            ops.push_back("Replace A["+to_string(i-1)+"]='"+A[i-1]+"' with '"+B[j-1]+"'");
            i--; j--;
        } else if(i > 0 && dp[i][j] == dp[i-1][j]+1){
            ops.push_back("Delete A["+to_string(i-1)+"]='"+A[i-1]+"'");
            i--;
        } else {
            ops.push_back("Insert '"+string(1,B[j-1])+"' into A");
            j--;
        }
    }
    for(auto& op : ops) cout << op << '\n';
}

Step 5 — Reduce memory: 2D → two rows

When we compute row i, we only ever read row i-1 and the same row i at earlier j. So we can keep just two rows. Caveat: reconstruction needs the full 2D table — only collapse to rolling rows if the operation list is not required.

vector<int> prev(m+1), cur(m+1);
for(int j=0;j<=m;j++) prev[j] = j;          // base row (i = 0)
for(int i=1;i<=n;i++){
    cur[0] = i;                              // base column (j = 0)
    for(int j=1;j<=m;j++)
        cur[j] = (A[i-1]==B[j-1])
                 ? prev[j-1]
                 : 1 + min({prev[j-1], prev[j], cur[j-1]});
    swap(prev, cur);
}
cout << prev[m];   // O(m) space, distance only

Complexity

⏱

Time: O(nm) — fill an (n+1)×(m+1) table with O(1) work per cell. Compare with O(3ⁿ⁺ᵐ) brute force.

🗂

Space: O(nm) for reconstruction; O(m) with rolling rows if only the distance is needed (not the operation sequence).

Pitfalls

  • Forgetting the base cases dp[i][0]=i and dp[0][j]=j — without them, the recurrence has nothing to build on and all values remain 0.
  • Confusing which operation corresponds to which cell movement: replace=diagonal, delete=up, insert=left. Drawing the table movement arrows prevents this.
  • Compressing to rolling rows then trying to reconstruct — keep the full table if you need the operation trace.
  • Off-by-one: dp is 1-indexed on prefixes but the string is 0-indexed. Compare A[i-1] with B[j-1], not A[i] with B[j].
💡

Connection to LCS: For strings with only insert and delete allowed (no replace), edit distance = n + m − 2·LCS(A,B). When replace is also allowed, edit distance and LCS diverge and require separate DPs.

8. To Europe! — Prefix Partition DP PREFIX PARTITION

This classic problem teaches a fundamentally different DP structure from anything we have seen so far. The state is a prefix length and the transition asks "what was the last contiguous group?" This is not a knapsack — the order of elements is fixed and groups must be contiguous segments of the original sequence. The same prefix-partition pattern appears in rod cutting, egg dropping, text justification, and many resource allocation problems.

Problem

Vehicles queue at a bridge in order (no overtaking). Vehicle i has weight w[i] and speed s[i]. The bridge has capacity b and length L. A group of consecutive vehicles may cross together if their total weight does not exceed b. The crossing time for a group is 60·L / (minimum speed in the group), because the group can only move as fast as its slowest member. Minimise the total crossing time for all vehicles.

Concretely: partition vehicles 1, 2, …, n into contiguous groups, each with total weight ≤ b, to minimise the sum of crossing times.

Brute Force

Think of placing "dividers" between vehicles to create groups. With n vehicles there are n−1 possible divider positions, each either present or absent. That gives 2ⁿ⁻¹ possible partitions. For each partition, check every group's weight constraint and compute total time. Time: O(2ⁿ · n). For n=1000, this is astronomically infeasible.

Where Overlapping Subproblems Appear

Consider a brute-force recursion that decides "where does the last group start?" If the last group is vehicles j+1 … n, then the remaining subproblem is "partition vehicles 1 … j optimally." Multiple different choices for the last-group boundary of a larger instance can lead to the same prefix-problem "optimise vehicles 1 … j." That is the overlap: different last-group choices for vehicles after j all need the answer for the first j vehicles.

State

dp[i] = minimum total crossing time for the first i vehicles

Once you know the optimal cost for the first j vehicles (for every j < i), you can try all possible last groups j+1 … i and pick the cheapest extension.

Transition

Story: "The last group to cross the bridge consists of vehicles j+1, j+2, …, i, for some j < i. Before this group, the first j vehicles were already partitioned optimally at cost dp[j]. The cost of this last group is 60·L / min(s[j+1..i]). I try all valid starting positions j and take the minimum."

dp[i] = min over valid j: dp[j] + 60·L / min(s[j+1], …, s[i])

The last group is "valid" if the total weight w[j+1] + … + w[i] ≤ b. As we scan j backward from i−1 to 0, we maintain a running total weight and running minimum speed. If the running weight exceeds b, we stop — all earlier j values would have even more weight.

Base Case

dp[0] = 0. Zero vehicles have already crossed, with zero time. This is the foundation that the recurrence builds on: the first group starts from position 1 and extends to some position i, with dp[0] as its prefix cost.

Final Answer

dp[n] — the minimum time to move all n vehicles across.

Code

#include <bits/stdc++.h>
using namespace std;
int main(){
    int n, b, L;
    cin >> n >> b >> L;
    vector<int> w(n+1), s(n+1);
    for(int i = 1; i <= n; i++) cin >> w[i] >> s[i];

    vector<double> dp(n+1, 1e18);
    vector<int> par(n+1, -1);   // reconstruction: dp[i] came from dp[par[i]]
    dp[0] = 0;

    for(int i = 1; i <= n; i++){
        int sumW = 0, minS = INT_MAX;
        for(int j = i; j >= 1; j--){   // try last group = vehicles j..i
            sumW += w[j];
            if(sumW > b) break;          // group too heavy, stop
            minS = min(minS, s[j]);
            double cost = 60.0 * L / minS;
            if(dp[j-1] + cost < dp[i]){
                dp[i] = dp[j-1] + cost;
                par[i] = j-1;            // record split point
            }
        }
    }
    printf("%.6f\n", dp[n]);

    // Reconstruct groups
    vector<pair<int,int>> groups;
    for(int pos = n; pos > 0; ){
        int prev = par[pos];
        groups.push_back({prev+1, pos});
        pos = prev;
    }
    reverse(groups.begin(), groups.end());
    for(auto [a,b2] : groups)
        printf("Group: vehicles %d..%d\n", a, b2);
}

Complexity

⏱

Time: O(n²) — for each of the n positions, the inner loop scans backward over at most n predecessors. Each (i,j) pair is visited at most once.

🗂

Space: O(n) — one dp array and one predecessor array.

Pitfalls

  • Forgetting to maintain the running minimum speed — recomputing min over j..i from scratch each time makes the solution O(n³).
  • Using greedy grouping (fill each group as full as possible) — fails because a smaller group with a fast minimum speed can be cheaper than a full group with a slow member.
  • Integer overflow when computing 60·L — use doubles or cast to long long before multiplying.
  • Not breaking when sumW > b — without this early exit, you waste time checking impossible groups.
🔗

The prefix-partition pattern: Any problem that says "partition a fixed-order sequence into contiguous groups to optimise some objective" is prefix-partition DP. Other examples: rod cutting (partition a rod into pieces to maximise profit), text justification (partition words into lines to minimise badness), and the painting fence problem.

9. MCM — Matrix Chain Multiplication INTERVAL DP

MCM is the prototype for interval DP — problems where the state is a contiguous subrange [l, r] and the transition tries all split points within that range. The table is triangular (only entries with l ≤ r are meaningful) and must be filled in order of increasing interval length. Understanding MCM deeply unlocks an entire class of problems: optimal BST, palindrome partitioning, balloon bursting, and more.

Problem

You have a chain of n matrices to multiply: A₁ × A₂ × … × Aₙ. Matrix Aᵢ has dimensions p[i-1] × p[i] (so consecutive matrices are compatible for multiplication). Matrix multiplication is associative — the result is the same regardless of parenthesisation — but the number of scalar multiplications depends enormously on the order. Find the parenthesisation that minimises the total number of scalar multiplications.

Example: A₁ is 10×30, A₂ is 30×5, A₃ is 5×60. Computing (A₁ × A₂) × A₃ costs 10·30·5 + 10·5·60 = 1500 + 3000 = 4500. Computing A₁ × (A₂ × A₃) costs 30·5·60 + 10·30·60 = 9000 + 18000 = 27000. Same result, but the first parenthesisation is 6× cheaper.

Brute Force

Try all possible parenthesisations. For n matrices, the number of distinct parenthesisations is the (n-1)-th Catalan number: C(n-1) ≈ 4ⁿ / (n^{3/2} · √π). For n=20, that is over a billion. The brute force simply tries every way to place the outermost split and recurses on both halves.

Where Overlapping Subproblems Appear

Consider the outer split for multiplying A₁…A₅ at position k=2: the subproblems are (A₁ × A₂) and (A₃ × A₄ × A₅). Now consider splitting at k=3: the subproblems are (A₁ × A₂ × A₃) and (A₄ × A₅). The sub-chain A₁ × A₂ appears in the first case directly and also within A₁ × A₂ × A₃ in the second case. As the chain grows, these overlapping sub-chains multiply rapidly. The same interval [l, r] is needed as a subproblem from many different parent splits.

State

dp[i][j] = minimum scalar multiplications to compute the product A_i × A_{i+1} × … × A_j

This state captures exactly the subproblem of optimally multiplying a contiguous sub-chain. The dimensions of the resulting matrix are p[i-1] × p[j].

Transition

Story: "To compute A_i × … × A_j, I must choose a split point k somewhere between i and j−1. This splits the chain into A_i × … × A_k (which costs dp[i][k]) and A_{k+1} × … × A_j (which costs dp[k+1][j]). After computing both sub-products, I multiply the two resulting matrices together, which costs p[i-1] · p[k] · p[j] scalar multiplications. I try all possible split points k and take the minimum."

dp[i][j] = min over i ≤ k < j of: dp[i][k] + dp[k+1][j] + p[i-1] · p[k] · p[j]

Base Case

dp[i][i] = 0 for all i. Multiplying a single matrix requires zero operations — there is nothing to multiply it with.

Fill Order — Why Length Matters

dp[i][j] depends on dp[i][k] and dp[k+1][j], both of which cover shorter intervals than [i, j]. So we must fill the table in order of increasing interval length: first all intervals of length 1 (base case, all zero), then length 2, then length 3, …, up to length n. Within each length, the order of starting position i does not matter.

Final Answer

dp[1][n] — the minimum cost to multiply the entire chain A₁ × … × Aₙ.

Step 1 — Recursion

Define cost(i, j) = "minimum cost to multiply matrices Aᵢ × Aᵢ₊₁ × … × Aⱼ". Try every possible split point k (where the outermost multiplication happens), recurse on both halves, and add the cost of the final multiplication.

vector<int> p;       // dimensions: matrix Aᵢ is p[i-1] × p[i]

long long cost(int i, int j) {
    if (i == j) return 0;                         // single matrix → 0 work
    long long best = LLONG_MAX;
    for (int k = i; k < j; k++) {
        long long c = cost(i, k) + cost(k+1, j)
                    + (long long)p[i-1] * p[k] * p[j];
        best = min(best, c);
    }
    return best;
}

Step 2 — Memoize

The state is two integers (i, j). Versioned-memoization template (see §16.5).

const int MAXN = 505;
long long memo[MAXN][MAXN];
int saved[MAXN][MAXN], sid;

long long cost(int i, int j) {
    if (i == j) return 0;                          // base — constant, no caching
    auto& ret = memo[i][j];
    if (saved[i][j] == sid) return ret;
    saved[i][j] = sid;
    long long best = LLONG_MAX;
    for (int k = i; k < j; k++) {
        long long c = cost(i, k) + cost(k+1, j) + (long long)p[i-1] * p[k] * p[j];
        best = min(best, c);
    }
    return ret = best;
}

Step 3 — Build (recover the actual parenthesisation)

The DP value tells us "the cheapest is 15125 multiplications" — the user wants the actual order: was it (A₁(A₂A₃))(A₄A₅) or ((A₁A₂)A₃)(A₄A₅) or…? The build function asks at each interval [i, j]: "Which split k produced memo[i][j]?" Then recursively prints the parenthesisation around that split.

void buildMCM(int i, int j) {
    if (i == j) { cout << "A" << i; return; }
    // Find the split k that produced the optimum
    for (int k = i; k < j; k++) {
        long long c = memo[i][k] + memo[k+1][j] + (long long)p[i-1] * p[k] * p[j];
        if (c == memo[i][j]) {
            cout << "(";
            buildMCM(i, k);          // left half
            cout << " x ";
            buildMCM(k+1, j);        // right half
            cout << ")";
            return;                  // pick exactly one split
        }
    }
}
// usage: cost(1, n);  buildMCM(1, n);

Step 4 — Iterate (full table) with cut[][] for fast reconstruction

Ordering question: dp[i][j] reads dp[i][k] and dp[k+1][j] — both cover shorter intervals than [i, j]. So we must fill the table in order of increasing interval length: first all intervals of length 1 (the base case, all zeros), then length 2, then length 3, … up to length n.

#include <bits/stdc++.h>
using namespace std;
int main(){
    int n;  // number of matrices
    cin >> n;
    vector<int> p(n+1);
    for(int i = 0; i <= n; i++) cin >> p[i];  // dimensions

    vector<vector<long long>> dp(n+1, vector<long long>(n+1, 0));
    vector<vector<int>> cut(n+1, vector<int>(n+1, 0));
    const long long INF = 1e18;

    // Fill by increasing interval length
    for(int len = 2; len <= n; len++){
        for(int i = 1; i + len - 1 <= n; i++){
            int j = i + len - 1;
            dp[i][j] = INF;
            // Try all split points k
            for(int k = i; k < j; k++){
                long long cost = dp[i][k] + dp[k+1][j]
                               + (long long)p[i-1] * p[k] * p[j];
                if(cost < dp[i][j]){
                    dp[i][j] = cost;
                    cut[i][j] = k;     // record best split — used by reconstruction
                }
            }
        }
    }
    cout << dp[1][n] << '\n';

    // Reconstruct parenthesisation (iterative version of buildMCM)
    function<void(int,int)> print = [&](int l, int r){
        if(l == r){ cout << "A" << l; return; }
        cout << "(";
        print(l, cut[l][r]);
        cout << " x ";
        print(cut[l][r]+1, r);
        cout << ")";
    };
    print(1, n);
    cout << '\n';
}

Step 5 — Reduce memory?

Interval DPs like MCM are not amenable to memory reduction. The recurrence reads from many (i, k) and (k+1, j) cells covering all shorter intervals; we cannot drop any of them while still computing dp[1][n]. The dp[][] table must remain O(n²). Reconstruction requires the cut[][] table too — also O(n²). Lesson: when dependencies have wide reach across the table, accept O(n²) memory and move on.

Complexity

⏱

Time: O(n³) — O(n²) intervals, each trying O(n) split points. Compare with exponential brute force (Catalan number).

🗂

Space: O(n²) — the dp and cut tables, both n×n.

Pitfalls

  • Filling by (i, j) in nested loops i=1..n, j=i+1..n instead of by increasing length — this causes dp[i][j] to read uninitialised values dp[k+1][j] where k+1 > i.
  • Off-by-one on dimensions: matrix Aᵢ has dimensions p[i-1]×p[i], not p[i]×p[i+1]. The multiplication cost is p[i-1]·p[k]·p[j].
  • Integer overflow: p values can be large (up to 500). p[i-1]·p[k]·p[j] can reach 1.25×10⁸ per step, and dp values can exceed int range. Use long long.
  • Not recording the split point cut[i][j] — without it, you can report the minimum cost but cannot reconstruct which parenthesisation achieves it.
🔗

The interval DP pattern: Any problem that says "operate on a contiguous range and the cost depends on how you split or shrink it" is interval DP. Other classic instances: optimal binary search tree, Stone Merging (CF 1527E), Treats for the Cows (choose from ends of a line), and palindrome partitioning.

10. Bitmask DP — Travelling Salesman Problem ICPC LEVEL

Bitmask DP is the technique that transforms problems involving subsets of a small set (typically n ≤ 20) from exponential brute force into manageable polynomial computation. The "bitmask" is simply an integer whose binary representation encodes which elements of the set have been included. Bit k is 1 if element k is in the subset, 0 otherwise. The Travelling Salesman Problem is the canonical bitmask DP problem — it is worth understanding in full because the exact same pattern appears in assignment problems, Hamiltonian path problems, and many combinatorial optimisation challenges.

💡

Plain-language intuition: We are not storing the whole path — we only store which cities were already visited and where we are right now. That pair (visited set + current city) is all the future needs. Two different trips that visited the same cities and ended at the same city are interchangeable from this point forward. This is still just DP — the "bitmask" is simply a compact integer encoding of the visited set, not a new idea.

Problem

Given n cities and the distance dist[i][j] between every pair, find the shortest route that starts at city 0, visits every other city exactly once, and returns to city 0. (If returning to city 0 is not required, the problem is called the Hamiltonian Path problem.)

Brute Force & Why It Fails

The brute-force approach enumerates all permutations of the n−1 cities to visit (we always start at city 0). There are (n−1)! such permutations. For n=20 that is 19! ≈ 1.2×10¹⁷ — completely infeasible. The brute-force time complexity is O(n! · n). Even with aggressive pruning it remains exponential in practice for n>15.

Where the Overlap Is

This is the key observation. Consider two partial tours: one visits cities {1,3,5} in order 1→3→5, and another visits them in order 3→1→5. Both end at city 5 with the same set of visited cities {0,1,3,5}. Any optimal continuation from city 5 that visits the remaining unvisited cities is identical for both partial tours — the continuation does not care about the order in which {1,3} were visited, only that they were visited and that we are currently at city 5. This means "current city + set of visited cities" fully determines all optimal future decisions. That observation defines the DP state.

State

The state keeps only two things: the set of cities already visited (encoded as a bitmask) and which city we are standing in right now. Everything else about the journey so far — the order, the route taken — is deliberately forgotten because the future does not need it.

dp[mask][u] = minimum cost to visit exactly the cities in 'mask', starting from city 0, and currently being at city u

Here mask is a bitmask of width n: bit k is 1 if city k has been visited. The state requires that city 0 is always the starting city and is always included in mask. So mask always has bit 0 set. There are 2ⁿ possible masks and n possible current cities, giving 2ⁿ · n states total — for n=20, that is about 20 million states, which is perfectly manageable.

Transition

Story: "I am currently at city u, having visited the cities in mask. I want to move to some unvisited city v. After moving, my new current city is v and my new visited set is mask with bit v set."

dp[mask | (1<<v)][v] = min(dp[mask | (1<<v)][v], dp[mask][u] + dist[u][v])

This transition is valid for every pair (mask, u) where bit u is set in mask, and for every v where bit v is NOT set in mask (unvisited city).

Base Case

dp[1][0] = 0 (only city 0 visited, currently at city 0, cost = 0)

All other dp[mask][u] are initialised to ∞.

Final Answer

After filling the table, the answer for the full TSP (return to start) is:

min over all u ≠ 0: dp[(1<<n)−1][u] + dist[u][0]

The mask (1<<n)−1 has all n bits set, meaning all cities have been visited. We add the cost of returning to city 0. For the Hamiltonian path variant (no return required), the answer is min over all u of dp[(1<<n)−1][u].

Step 3 — Build (recover the actual tour)

The DP value tells us "the cheapest tour costs 80" — the user wants the actual sequence of cities. The build pattern: at every state (mask, u) remember which previous city par[mask][u] we came from. After the iteration ends, find the best last city, then walk backward through par until you return to city 0.

// par[mask][u] = the city we came from when we first reached state (mask, u)

vector<int> buildTour(int fullMask, int lastCity, int par[1<<20][20]) {
    vector<int> tour;
    int mask = fullMask, u = lastCity;
    while (u != -1) {                  // walk backward through predecessors
        tour.push_back(u);
        int prev = par[mask][u];
        mask ^= (1 << u);              // remove u from the visited set
        u = prev;
    }
    reverse(tour.begin(), tour.end()); // now in forward order from city 0
    return tour;
}

This is the same Step 3 idea you used for LIS and Coin Change — store one extra "predecessor" piece at each cell, then walk backward at the end.

Step 4 — Iterate (full table) with par[][] for fast reconstruction

#include <bits/stdc++.h>
using namespace std;

int n;
int dist[20][20];
long long dp[1<<20][20];   // dp[mask][u]
int par[1<<20][20];         // for reconstruction: which city did we come from?

int main(){
    ios::sync_with_stdio(false); cin.tie(nullptr);
    cin >> n;
    for(int i=0;i<n;i++) for(int j=0;j<n;j++) cin>>dist[i][j];

    // Initialise all states to INF
    const long long INF = 1e18;
    for(auto& row:dp) fill(row,row+n,INF);
    for(auto& row:par) fill(row,row+n,-1);

    dp[1][0] = 0;   // Base case: only city 0 visited, at city 0, cost 0

    // Enumerate all masks in increasing order
    for(int mask=1; mask<(1<<n); mask++){
        for(int u=0;u<n;u++){
            if(!(mask&(1<<u))) continue;      // u must be in mask (we're at u)
            if(dp[mask][u]==INF) continue;    // unreachable state

            // Try extending tour to each unvisited city v
            for(int v=0;v<n;v++){
                if(mask&(1<<v)) continue;     // v already visited
                int newMask = mask|(1<<v);
                long long newCost = dp[mask][u] + dist[u][v];
                if(newCost < dp[newMask][v]){
                    dp[newMask][v] = newCost;
                    par[newMask][v] = u;      // came from u
                }
            }
        }
    }

    // Find best way to return to city 0 from any last city
    int fullMask = (1<<n)-1;
    long long ans = INF;
    int lastCity = -1;
    for(int u=1;u<n;u++){
        if(dp[fullMask][u]+dist[u][0] < ans){
            ans = dp[fullMask][u]+dist[u][0];
            lastCity = u;
        }
    }
    cout << ans << '\n';

    // Reconstruct tour
    vector<int> tour;
    int mask=fullMask, u=lastCity;
    while(u!=-1){
        tour.push_back(u);
        int prev=par[mask][u];
        mask^=(1<<u);
        u=prev;
    }
    reverse(tour.begin(),tour.end());
    for(int city:tour) cout<<city<<" ";
    cout<<"0\n";  // return to start
    return 0;
}

Complexity

⏱

Time: O(n² · 2ⁿ) — 2ⁿ masks × n current cities × n transitions = n²·2ⁿ. For n=20: about 400 million operations. Fast enough in ~2 seconds with optimisation.

🗂

Space: O(n · 2ⁿ) — the dp and par tables. For n=20: 20 million entries × 8 bytes ≈ 160 MB. Be careful with memory limits.

Hamiltonian cycle decision version

If the question is only "does a Hamiltonian cycle exist?", change dp to store boolean instead of cost: dp[mask][u] = can I visit exactly the cities in mask, starting from 0, and currently at u? The transition becomes dp[mask|(1<<v)][v] |= dp[mask][u]. The answer is any u with dp[(1<<n)−1][u] = true and dist[u][0] < ∞.

Pitfalls

  • Forgetting that bit 0 (city 0) must always be set — the mask always includes the starting city.
  • Confusing mask | (1<<v) with mask + (1<<v) — use bitwise OR, not addition.
  • Memory: for n=20 the table is large. Use short or int for the dp if distances are small.
  • Iterating masks in arbitrary order — masks must be processed in increasing order so that smaller masks (fewer cities visited) are computed before larger ones.

11. Digit DP ICPC LEVEL

Digit DP solves problems of the form "count how many integers in [0, N] satisfy some property about their digits." The key challenge is that checking each number individually is O(N) which can be up to 10¹⁸ — far too slow. Digit DP processes the number digit by digit from the most significant to least significant, maintaining a small amount of state that captures everything necessary about the numbers formed so far.

Intuition for Beginners

Imagine you want to count how many numbers from 0 to 375 have a digit sum divisible by 3. You could check each of the 376 numbers one by one — but what if N is 10¹⁸? You cannot loop that many times. Instead, think of building the number one digit at a time, left to right. You are writing a 3-digit number: _ _ _. For the first digit, you choose 0, 1, 2, or 3. For the second digit, your choice depends on what you chose first. For the third digit, your choice depends on the first two. The key insight: you do not need to remember the exact digits you chose — only the information that affects your future choices and the final answer. For the digit-sum-mod-3 problem, all you need is: which position am I at, what is the running digit sum mod 3, and am I still "tight" against N?

🔑

What does "tight" mean, simply? You are building a number that must be ≤ 375. If your first digit is 2, you are "free" — no matter what you put for the remaining digits, the number will be at most 299, which is always ≤ 375. But if your first digit is 3, you are "tight" — you must be careful: the second digit cannot exceed 7 (the second digit of 375). Once you choose a digit strictly less than N's digit at that position, you become "free" forever. Once tight becomes false, all later digits can be 0–9. This is why many different partial numbers lead to the same (position, state, tight) tuple — and that repetition is the overlapping subproblems that make DP work. The state remembers only what the future needs — position, the constraint-relevant summary, and the tight flag — not the actual digits chosen so far.

Tiny Walk-Through: Count numbers ≤ 375 with digit sum divisible by 3

N = 375. We process digit by digit. At position 0 (hundreds), tight=true, so d ∈ {0,1,2,3}. If we pick d=2: tight becomes false (2 < 3), running sum mod 3 = 2. At position 1 (tens), tight=false, so d ∈ {0,…,9}. If we pick d=5: sum mod 3 = (2+5) mod 3 = 1. At position 2 (units), tight=false, d ∈ {0,…,9}. We need a digit d where (1+d) mod 3 = 0, so d ∈ {2,5,8}. All three choices produce a valid number. The DP counts all such paths simultaneously, grouping together all partial numbers that share the same (position, sum_mod_3, tight) state.

Problem

Count the integers in [0, N] that have no two consecutive identical digits. For example, if N=25, the valid numbers include 0,1,2,...,9 (single digits, trivially valid), 10,12,13,...,20,21,23,... — but not 11, 22. We want to count efficiently without iterating over all numbers up to N.

Brute Force & Why It Fails

Check every integer from 0 to N: convert to string, scan adjacent digits, count valid ones. Time: O(N · digits(N)). For N = 10¹⁸ this is completely infeasible. The key insight is that the digit-by-digit structure of integers allows us to count entire classes of valid numbers at once.

The Tight Flag — The Central Idea

When building the number digit by digit from left to right, at each position we choose a digit d. The tight flag tracks whether the number being built is still bounded from above by the corresponding digit of N. If tight=true at position i, then the digit we choose at position i cannot exceed N's digit at position i (otherwise we'd exceed N). If tight=false, we can use any digit 0–9 freely because earlier choices have already made our number strictly smaller than N. Once a digit is chosen strictly less than N's digit, tight becomes false for all subsequent positions.

State

The state records only the information that future digit choices need: where we are in the number, a compact summary of what we have built so far (just enough for the constraint check), and whether we are still forced to match N's prefix.

dp[pos][last_digit][tight][leading_zero]

Here: pos = which digit position we are filling (0 = most significant); last_digit = the digit placed at position pos−1 (needed to check the "no consecutive identical digits" constraint); tight = whether we are still bounded by N at this position; leading_zero = whether only zeros have been placed so far (handles the fact that "007" is really "7" and its first digit is 7, not 0).

Transition

At each position we try every digit that is allowed and ask: "if I place this digit here, what does the state become for the next position?" The tight flag, the leading-zero flag, and the constraint state all update in a simple, predictable way.

At each position, try all possible digits d from 0 to (tight ? N[pos] : 9). The new tight flag is (old tight) AND (d == N[pos]). The new leading_zero is (old leading_zero) AND (d == 0). The constraint check (no two consecutive identical digits) applies only if leading_zero is false — we check d ≠ last_digit.

Code — count integers in [0, N] with no two consecutive identical digits

#include <bits/stdc++.h>
using namespace std;

string N_str;
int n_digits;
// memo[pos][last][tight][lz]
// last: 0-9 digit; tight: 0/1; lz (leading zero): 0/1
long long memo[19][11][2][2];   // last has 11 slots (0..9 plus sentinel for "no last digit")
int saved[19][11][2][2], sid;   // versioned memoization (see §16.5)

// Returns count of valid numbers from position pos onward,
// given that the last placed non-zero digit was 'last',
// tight=whether we are bounded by N, lz=leading zero
long long solve(int pos, int last, bool tight, bool lz) {
    if(pos == n_digits) return 1;  // successfully placed all digits → count this number

    auto& ret = memo[pos][last][tight][lz];
    if(saved[pos][last][tight][lz] == sid) return ret;     // cache hit for THIS test case
    saved[pos][last][tight][lz] = sid;

    int limit = tight ? (N_str[pos]-'0') : 9;
    ret = 0;
    for(int d = 0; d <= limit; d++) {
        bool new_tight = tight && (d == limit);
        bool new_lz    = lz && (d == 0);
        // Constraint: if not a leading zero, digit must differ from last
        if(!new_lz && !lz && d == last) continue;   // consecutive identical — skip
        int new_last   = new_lz ? last : d;           // update last only if not lz
        ret += solve(pos+1, new_last, new_tight, new_lz);
    }
    return ret;
}

long long count_up_to(long long N) {
    if (N < 0) return 0;
    N_str = to_string(N);
    n_digits = N_str.size();
    ++sid;                          // O(1) "logical reset" — no memset of visited[] needed
    return solve(0, 10, true, true); // start: no last digit (sentinel 10), tight=true, lz=true
}

int main(){
    long long L, R; cin >> L >> R;
    // Count [L, R] = count[0..R] - count[0..L-1]
    cout << count_up_to(R) - count_up_to(L-1) << '\n';
}
💡

Why this is the textbook example for versioned memoization. Digit DP queries are typically asked many times (count in [L,R] = count[0..R] − count[0..L−1] is already two queries per range, and contests often have q queries). The memo array memo[19][11][2][2] has 836 cells, so a memset per query is fast. But the same code template generalises to harder digit DP problems where the array is much larger (e.g., adding digit-sum or modular state) — there, memset dominates, while a single ++sid stays O(1). Always reach for the saved/sid pattern when writing recursive DP that runs many times in one program.

Complexity

⏱

Time: O(digits · 10 · 2 · 2 · 10) ≈ O(D · 400) where D ≤ 19 for 64-bit integers. Essentially O(1) in practice. Compare with O(N) = O(10¹⁸) brute force.

🗂

Space: O(D · 10 · 2 · 2) — the memoisation table. Tiny.

Generalising to other digit constraints

The "no consecutive identical digits" constraint is just one example. The digit DP template handles any constraint that can be captured in a small "digit state" that is updated as digits are placed. Common examples include: count of integers whose digit sum equals S (state = running sum mod S), count with a specific digit pattern, count where no digit exceeds k, count where digits are non-decreasing, and many others. The template (pos, tight, leading_zero, your_constraint_state) applies uniformly.

Pitfalls

  • Forgetting the leading zero flag — without it, "009" would be counted as having 0 as a significant digit.
  • Memoising with a stale tight=true state across different N values — always reset the memo when N changes.
  • Off-by-one: counting [L, R] = count[0..R] − count[0..L−1], not count[0..L].
  • Applying constraints when leading_zero is still true — a leading zero is not a real digit of the number.

Application — Count numbers in [1, N] with digit sum divisible by K

Story: Given N ≤ 10¹⁸ and K ≤ 100, count how many positive integers up to N have a digit sum that is a multiple of K.

Abstraction: The constraint depends only on digits placed so far — classic digit DP territory.

Why digit DP fits: The "digit sum mod K" state is tiny and updated digit-by-digit. The tight flag bounds us to ≤ N. Both are small finite states, so the table is tiny.

State: dp[pos][sum_mod_k][tight][leading_zero] — at most 19 × 100 × 2 × 2 = 7,600 entries.

12. Tree DP ICPC LEVEL

Tree DP is DP computed on a tree structure. The natural evaluation order is a post-order DFS: compute answers for all children before the parent. The prototype problem is maximum weighted independent set on a tree — a problem that cannot be solved greedily but admits a clean O(n) DP.

Problem — Maximum Weighted Independent Set

Given a tree with n nodes where each node u has weight w[u]. Select a subset of nodes (an "independent set") such that no two selected nodes are adjacent (connected by an edge), and the total weight is maximised.

Brute Force & Why It Fails

Try all 2ⁿ subsets of nodes. For each, check independence (no two are adjacent) and compute total weight. Time: O(2ⁿ · n). For n=100000 completely infeasible. Greedy fails too: greedily picking the heaviest unblocked node can block multiple lighter but collectively heavier nodes.

Overlap

Fix a root (say node 1). The optimal solution for the subtree rooted at u depends only on whether u itself is selected — the two choices are completely independent of what happens outside the subtree. Moreover, the choices for u's children are independent of each other once we know whether u is selected. This means the problem decomposes cleanly across subtrees — the classic tree DP structure.

State

dp[u][0] = maximum weight independent set in the subtree rooted at u, where u is NOT selected
dp[u][1] = maximum weight independent set in the subtree rooted at u, where u IS selected

Transition

Story: "If u is not selected (state 0), each child v can be either selected or not — take the better option for each child independently. If u is selected (state 1), no child can be selected — each child must be in state 0."

dp[u][0] = sum over children v of: max(dp[v][0], dp[v][1])
dp[u][1] = w[u] + sum over children v of: dp[v][0]

Base Case

For leaf nodes (no children): dp[u][0] = 0, dp[u][1] = w[u].

Code

#include <bits/stdc++.h>
using namespace std;

int n;
vector<int> adj[100005];
long long w[100005];
long long dp[100005][2];
bool visited[100005];

void dfs(int u, int parent) {
    dp[u][0] = 0;      // base: leaf with u not selected = 0
    dp[u][1] = w[u];   // base: leaf with u selected = w[u]

    for(int v : adj[u]) {
        if(v == parent) continue;  // don't go back up the tree
        dfs(v, u);                 // solve subtree rooted at v first

        // u NOT selected: each child can be selected or not — take best
        dp[u][0] += max(dp[v][0], dp[v][1]);
        // u IS selected: child v cannot be selected
        dp[u][1] += dp[v][0];
    }
}

int main(){
    ios::sync_with_stdio(false); cin.tie(nullptr);
    cin >> n;
    for(int i=1;i<=n;i++) cin >> w[i];
    for(int i=0;i<n-1;i++){
        int u,v; cin>>u>>v;
        adj[u].push_back(v); adj[v].push_back(u);
    }
    dfs(1, -1);
    cout << max(dp[1][0], dp[1][1]) << '\n';
    return 0;
}

Generalisation — Tree DP pattern

The dp[u][0/1] template appears in dozens of tree problems. The "0 or 1" often represents some binary decision about u — is u colored? is u a center? is u matched? is u a root of a chain? The recurrence always follows the same structure: combine children's DP values based on which choice was made for u. The evaluation order is always post-order DFS (children before parent).

Rerooting technique

Sometimes the answer depends on a root that we don't know in advance, or the problem asks for the answer for every possible root. The naive approach (run tree DP for every root) is O(n²). The rerooting technique computes all n answers in O(n) by first rooting the tree arbitrarily, computing dp for all nodes, then running a second DFS that "transfers the root" to each child by subtracting the child's contribution from the parent's answer and recomputing. This is an advanced technique worth learning after mastering the basic tree DP.

Complexity

⏱

Time: O(n) — each node is processed once in the DFS, and each edge is traversed exactly twice (once in each direction).

🗂

Space: O(n) — dp array and recursion stack (O(n) for a path-shaped tree, O(log n) for balanced). Use iterative DFS for very deep trees to avoid stack overflow.

Pitfalls

  • Recursing into the parent — always pass the parent node to DFS and skip edges back to parent.
  • Visiting nodes multiple times — use a visited array or parent-tracking.
  • Stack overflow for large n on linear trees — replace recursive DFS with iterative (topological order via BFS or explicit stack).
  • Forgetting that the choice for the root itself must also be optimised — the answer at the root is max(dp[root][0], dp[root][1]).

Application — Tree Matching

Story: Given a tree with n nodes, find the maximum matching — the largest set of edges such that no two edges share a node.

Abstraction: For each node u, decide: is u matched through one of its child-edges, or is it left unmatched?

Why tree DP fits: The decision for u depends only on its children's states, and the tree structure ensures no cycles. Post-order DFS naturally resolves all dependencies.

State: dp[u][0] = max matching in subtree of u where u is unmatched; dp[u][1] = where u is matched through one child edge.

13. Grid DP 2D PREFIX DP

Grid DP covers problems where the state is a cell (i, j) in a 2D matrix and movement is restricted (typically right and down only). The same grid structure supports counting paths, optimising a cost, or making feasibility decisions — the only difference is the operator. Grid DP is one of the most common interview and contest topics because it builds intuition for 2D state spaces while remaining visually concrete.

Problem 1 — Count Paths with Obstacles

Given an n × m grid where some cells are blocked ('*') and others are free ('.'). Starting from the top-left corner (0,0), you can move only right or down at each step. Count the number of distinct paths from (0,0) to (n-1,m-1). This count can be astronomically large, so report it modulo 10⁹+7.

Brute Force

At each cell, you have at most two choices: go right or go down. The total path length is (n−1) + (m−1) = n+m−2 steps, so the recursion tree has at most 2^(n+m-2) leaves. For a 1000×1000 grid, that is 2^1998 — completely impossible. Even without obstacles, the answer itself is C(n+m−2, n−1) which for large grids is a huge number.

Where Overlapping Subproblems Appear

Many different paths through the grid pass through the same cell (i, j). Once you arrive at cell (i, j), the number of paths from (i, j) to the destination depends only on i and j — not on how you got to (i, j). Different paths from the start reach cell (3, 4) via different routes, but they all face the same "subproblem of counting paths from (3, 4) to the end." This is the overlap that DP exploits.

State

dp[i][j] = number of distinct paths from (0,0) to cell (i,j)

Alternatively you could define it as "paths from (i,j) to end" and work backward — both work. The forward definition (paths from start to here) is more common and slightly more natural for the row-major fill order.

Transition

Story: "To reach cell (i, j), I must have come from either the cell directly above (i-1, j) or the cell directly to the left (i, j-1). There is no other way to arrive at (i, j) given the movement rules. So the number of paths to (i, j) is the sum of paths to those two predecessor cells."

dp[i][j] = dp[i-1][j] + dp[i][j-1] (if cell (i,j) is not blocked)

If cell (i, j) is blocked: dp[i][j] = 0 (no path can pass through a blocked cell).

Base Cases

dp[0][0] = 1 if the starting cell is free (there is exactly one way to be at the start: do nothing). If the start itself is blocked, dp[0][0] = 0 and the answer is immediately 0.

The first row (i=0) and first column (j=0) are special: a cell in the first row can only be reached from the left, and a cell in the first column can only be reached from above. If any cell in the first row is blocked, all cells to its right have dp = 0.

Final Answer

dp[n-1][m-1] — the number of paths reaching the bottom-right corner.

Code — Counting Paths

#include <bits/stdc++.h>
using namespace std;
const int MOD = 1e9 + 7;
int main(){
    int n, m;
    cin >> n >> m;
    vector<string> g(n);
    for(auto& row : g) cin >> row;

    vector<vector<long long>> dp(n, vector<long long>(m, 0));
    if(g[0][0] == '.') dp[0][0] = 1;

    for(int i = 0; i < n; i++)
        for(int j = 0; j < m; j++){
            if(g[i][j] == '*'){ dp[i][j] = 0; continue; }
            if(i == 0 && j == 0) continue;  // base already set
            if(i > 0) dp[i][j] = (dp[i][j] + dp[i-1][j]) % MOD;
            if(j > 0) dp[i][j] = (dp[i][j] + dp[i][j-1]) % MOD;
        }
    cout << dp[n-1][m-1] << '\n';
}

Problem 2 — Minimum Path Sum

Same grid, but each cell has a non-negative cost grid[i][j]. Find the path from top-left to bottom-right (moving only right or down) that minimises the total cost.

dp[i][j] = grid[i][j] + min(dp[i-1][j], dp[i][j-1])

Same dependency graph, different operator: sum replaces addition-of-counts, and min replaces sum-of-paths. Initialise unreachable cells with ∞. Base: dp[0][0] = grid[0][0].

// Minimum path sum
vector<vector<int>> dp(n, vector<int>(m, INT_MAX));
dp[0][0] = grid[0][0];
for(int i = 0; i < n; i++)
    for(int j = 0; j < m; j++){
        if(i == 0 && j == 0) continue;
        int top = (i > 0) ? dp[i-1][j] : INT_MAX;
        int left = (j > 0) ? dp[i][j-1] : INT_MAX;
        dp[i][j] = grid[i][j] + min(top, left);
    }
cout << dp[n-1][m-1] << '\n';

Complexity

⏱

Time: O(nm) — each cell is visited exactly once with O(1) work.

🗂

Space: O(nm) for the full table; O(m) with a rolling row since each row depends only on the current and previous rows.

Pitfalls

  • Forgetting to handle the first row and first column specially — they have only one predecessor direction, not two.
  • Not setting dp[i][j] = 0 for blocked cells in counting problems — a blocked cell with a non-zero value propagates wrong counts to all downstream cells.
  • Using max instead of min (or vice versa) — always check whether the problem asks for minimum cost, maximum value, or count.
  • Grid indexing: if the grid is 1-indexed in the problem statement but 0-indexed in your code, shift carefully.
💡

Grid DP is 2D prefix DP with geometric structure. The transition "from above or from the left" is just a spatial version of "from the previous row or the previous column in a 2D prefix table." Once you see this, grid problems feel like LCS/edit distance problems with a different story.

14. Profile DP / Broken Profile DP ICPC LEVEL

Profile DP (also called Broken Profile DP) is the technique used when a grid problem requires tracking the partial state of a frontier — typically a row boundary — as a bitmask. The classic application is counting the number of ways to tile a board with dominoes, but the same technique applies to any problem where the shape of what has been filled so far along a row/column boundary matters.

Intuition for Beginners

Imagine you are tiling a bathroom floor with 2×1 tiles. You fill the floor column by column, left to right. When you finish a column and move to the next, some cells in the new column might already be partially covered by horizontal tiles you started in the previous column. The "profile" or "frontier" is a snapshot of which cells in the next column are already taken. If the bathroom has 4 rows, this snapshot is a 4-bit number: each bit says "taken" (1) or "empty" (0) for one row of the next column.

🔑

Why is dp[row][col] not enough? Suppose you are at column 3 of a 4-row board. Two different placement histories might both have "filled everything up to column 2 and currently at column 3" — but in one case rows 0 and 2 of column 3 are already occupied by horizontal tiles from column 2, while in the other case rows 1 and 3 are occupied. These lead to completely different future tilings! The ordinary position (row, col) tells you where you are but not the shape of the boundary. The bitmask captures that shape — it is a compact way to remember the unfinished boundary, and nothing more. This is still ordinary DP: the state is just (column, frontier mask) instead of (row, column).

Tiny Example — 3×2 Board

Consider a 3-row, 2-column board. We process column 0, then column 1. At the start, the frontier mask entering column 0 is 000 (all empty). One way to fill column 0: place one vertical domino covering rows 0–1, and one horizontal domino covering row 2 (extending into column 1). The exit mask is 001 (row 2 protrudes into column 1). Another way: place a horizontal domino in row 0 and a vertical domino in rows 1–2. The exit mask is 100 (row 0 protrudes). Different placements, different exit masks — each becomes a separate state for the next column. When we process column 1, we handle each entering mask separately: mask 001 means "row 2 is already filled, fill rows 0 and 1 only." The DP counts all paths and sums them at the end where the exit mask must be 000 (nothing sticks out).

Problem — Domino Tiling

Given an n × m board, count the number of ways to completely tile it with 2×1 dominoes (horizontal or vertical). Every cell must be covered by exactly one domino, with no overlaps and no gaps.

Example: a 2×3 board has exactly 3 valid tilings. A 2×4 board has 5 valid tilings. A 4×7 board has 3149 valid tilings. The numbers grow rapidly — there is no closed-form formula, so we need a systematic counting method.

Brute Force

Place dominoes one cell at a time, scanning the board left-to-right, top-to-bottom. At each empty cell, try placing a horizontal domino (covering this cell and the one to its right) or a vertical domino (covering this cell and the one below it). Backtrack if a placement is impossible. The number of configurations explored is enormous — roughly O(2^(nm/2)) — because at each step the branching depends on what has already been placed.

Why Ordinary DP Fails

You might try dp[i][j] = "number of ways to tile the board up to cell (i,j)." But this does not capture enough information! When you are at cell (i, j), some cells in the current row below the current position may already be occupied by vertical dominoes placed from the row above. The number of tilings from this point depends on which cells in the current row are already occupied — not just on the position (i, j). A simple 2D index is not enough to distinguish these different frontier states.

The Frontier Mask Idea

The breakthrough insight: describe the "shape" of the boundary between the filled region and the unfilled region using a bitmask. As we process column by column (or cell by cell), we maintain a bitmask of width n (the number of rows) where each bit indicates whether the corresponding cell in the current column is already occupied by a domino placed earlier.

💡

Bitmask meaning: Consider an n=4 row board being filled column by column. The mask 0110 means: row 0 is empty, row 1 is filled (by a horizontal domino from the previous column), row 2 is filled, row 3 is empty. The next column placement must fill the empty rows (0 and 3) while possibly extending horizontal dominoes into the column after.

State

The state records which column we are about to fill and the frontier mask — a snapshot of which rows in that column are already occupied by tiles from the previous column. Nothing else about the tiling history matters.

dp[col][mask] = number of ways to tile all columns 0..col-1 completely, such that the profile (frontier) entering column col is described by mask

Here mask is a bitmask of width n. Bit r is 1 if row r in column col is already occupied by a horizontal domino extending from column col-1. Bit r is 0 if row r in column col is still empty and needs to be filled.

Transition — Filling a Column

Given the entering mask for column col, we process the column row by row (row 0 to row n-1). At each row r, the cell (r, col) is either already filled (bit r = 1 in the entering mask) or empty (bit r = 0). For each empty cell, we have choices:

Option A — Place a vertical domino

Cover cell (r, col) and cell (r+1, col). This requires that row r+1 is also empty. After placement, both bits r and r+1 become "handled" for this column. Neither extends into the next column, so both bits in the exit mask are 0.

Option B — Place a horizontal domino

Cover cell (r, col) and cell (r, col+1). This means bit r in the exit mask is 1 (it protrudes into the next column). This is only valid if col+1 < m (not the last column).

The transition is generated recursively (or iteratively) for each column: start at row 0 with the entering mask, process each row, and collect all valid exit masks with their counts. Each valid exit mask becomes the entering mask for the next column.

Overlapping Subproblems

Different placement sequences within the same column can produce the same exit mask. For example, placing a vertical domino in rows 0-1 and a horizontal domino in row 2 might produce the same exit mask as placing a horizontal domino in row 0 and a vertical domino in rows 1-2 (if the masks coincidentally match). All these different internal arrangements that produce the same exit mask lead to the same future subproblem — this is the overlap that DP exploits.

Base Case

dp[0][0] = 1. Before any column is processed, the frontier is entirely empty (mask = 0), and there is exactly one way to have "tiled nothing" — do nothing.

Final Answer

dp[m][0]. After processing all m columns, the exit mask must be 0 — meaning no domino protrudes beyond the last column. If any bit is 1 in the exit mask, those dominoes extend outside the board, which is invalid.

Code

#include <bits/stdc++.h>
using namespace std;

int n, m;               // n rows, m columns (use n ≤ ~20 for bitmask)
long long dp[1001][1 << 12]; // dp[col][mask], n ≤ 12 for practical limits

// Recursively generate all valid ways to fill column 'col'
// given that rows 0..row-1 are already handled.
// in_mask  = entering mask (which rows are pre-filled by horizontal from prev col)
// out_mask = exiting mask being built (which rows protrude into next col)
void generate(int col, int row, int in_mask, int out_mask, long long ways) {
    if(row == n) {
        // Finished processing all rows of this column
        dp[col + 1][out_mask] += ways;
        return;
    }
    if(in_mask & (1 << row)) {
        // Row 'row' is already filled by a horizontal domino from previous column
        // Just skip to next row; this row contributes 0 to out_mask
        generate(col, row + 1, in_mask, out_mask, ways);
    } else {
        // Row 'row' is empty — must place a domino here

        // Option A: horizontal domino → covers (row, col) and (row, col+1)
        // This sets bit 'row' in out_mask (protrudes right)
        if(col + 1 < m) {
            generate(col, row + 1, in_mask, out_mask | (1 << row), ways);
        }

        // Option B: vertical domino → covers (row, col) and (row+1, col)
        // Requires row+1 < n AND row+1 is also empty in in_mask
        if(row + 1 < n && !(in_mask & (1 << (row + 1)))) {
            // Both row and row+1 are filled within this column
            // Neither protrudes, so out_mask unchanged for these two rows
            generate(col, row + 2, in_mask, out_mask, ways);
        }

        // Note: if neither option is valid, this branch dies (no valid tiling
        // exists through this path). The recursion naturally handles this.
    }
}

int main() {
    cin >> n >> m;
    // Ensure n ≤ m for efficiency (bitmask on shorter dimension)
    if(n > m) swap(n, m);

    memset(dp, 0, sizeof(dp));
    dp[0][0] = 1;   // base case

    for(int col = 0; col < m; col++) {
        for(int mask = 0; mask < (1 << n); mask++) {
            if(dp[col][mask] == 0) continue;  // skip unreachable states
            generate(col, 0, mask, 0, dp[col][mask]);
        }
    }
    cout << dp[m][0] << '\n';
    return 0;
}

Complexity

⏱

Time: O(m · 2ⁿ · transitions) — m columns, 2ⁿ masks per column, and the generate function explores at most O(2ⁿ) branches per mask. Effectively O(m · 4ⁿ) in the worst case, but much less in practice due to early pruning.

🗂

Space: O(m · 2ⁿ) — the dp table. Can be compressed to O(2ⁿ) using rolling columns since each column depends only on the previous one.

This is exponential in n (the shorter dimension), so Profile DP is practical only when one dimension is small (typically n ≤ 12–15). Always use the shorter dimension as the mask width — if the board is 3×100, use n=3 (only 8 masks) and m=100.

Pitfalls

  • Using the longer dimension as the bitmask width — always mask the shorter dimension. A 3×1000 board with bitmask on 3 rows has only 8 masks; bitmask on 1000 columns is impossible.
  • Forgetting the final answer requires mask = 0 — any 1-bit in the exit mask means a domino sticks out beyond the board.
  • Not handling the "neither option works" case — if a cell is empty and no valid domino placement exists (e.g., last column, no horizontal; next row also filled, no vertical), the branch must simply die without contributing to any exit mask.
  • Counting tilings of boards where n·m is odd — such boards have an odd number of cells and cannot be perfectly tiled by 2-cell dominoes. The answer is always 0.

Recognition Guide

🎯

When to think Profile DP: the problem asks you to fill, tile, or cover a grid cell-by-cell; the future depends on the partial occupancy of a row/column frontier; and one dimension is small (≤ 15). Classic instances: domino/tromino tiling, placing non-attacking rooks/kings with row-state, and grid coloring with neighbor constraints.

Application — Domino Tiling (the standard Profile DP problem)

Story: Fill an n × m board completely with 2×1 dominoes. No overlaps, no gaps. Count the number of distinct tilings.

Abstraction: Process the board column by column. The frontier between "filled" and "unfilled" is a bitmask of width min(n, m).

Why profile DP fits: Ordinary dp[row][col] cannot distinguish different frontier shapes. The bitmask captures exactly which cells in the next column are already occupied — the minimum information the future needs.

State: dp[col][mask] — at most m × 2^min(n,m) entries. For n=4, m=100: 100 × 16 = 1,600 states.

15. DAG DP — Longest Path in a DAG GRAPH DP

DP on a Directed Acyclic Graph (DAG) is a powerful pattern that unifies many DP problems under a single framework. In fact, every DP problem can be viewed as a shortest/longest path problem on a DAG of states. Understanding this connection deepens your intuition for all of DP and makes graph-based DP problems feel natural.

Intuition for Beginners

Why is longest path easy on a DAG but hard on a general graph? The answer is one word: cycles. In a general graph, a cycle lets you visit the same node over and over — so "longest path" becomes meaningless (you could loop forever). In a DAG, there are no cycles. Every path must eventually end. This means you can line up all the nodes in a single queue — called a topological order — where every edge points forward. Once you have this queue, computing the longest path is just a single left-to-right scan: for each node, look at all incoming edges and take the best predecessor. The cycle-free guarantee means no dependency is circular, so this scan always works.

Tiny Example — 4-Node DAG

Consider nodes 1→2→4 and 1→3→4, with edge weights: 1→2 costs 5, 2→4 costs 3, 1→3 costs 2, 3→4 costs 7. Topological order: [1, 2, 3, 4]. We scan left to right, tracking dist[u] = longest path ending at u. Start: dist[1]=0. Process node 2: dist[2] = dist[1]+5 = 5. Process node 3: dist[3] = dist[1]+2 = 2. Process node 4: max(dist[2]+3, dist[3]+7) = max(8, 9) = 9. Answer: 9. The path 1→3→4 is longer than 1→2→4. No backtracking, no exponential exploration — just one linear scan because the DAG guarantees no node needs to be revisited.

Problem — Longest Path in a DAG

Given a directed acyclic graph (DAG) with n nodes and weighted edges, find the length of the longest path from any source node. (Alternatively: from a given source s to a given target t.)

Why DAGs Are Special

In general graphs, the longest path problem is NP-hard — there is no known polynomial algorithm. But DAGs have no cycles, which means there exists a topological ordering of the nodes: an ordering where every directed edge goes from an earlier node to a later node. This ordering gives a natural "fill order" for DP — process nodes left to right in topological order, and every dependency is guaranteed to be already computed.

Brute Force

Try all possible paths from each source node. In a dense DAG, the number of paths can be exponential (consider a layered DAG where each node connects to all nodes in the next layer). The overlap: different paths reaching the same node u face identical subproblems — "what is the longest path from u onward?"

State

dp[u] = length of the longest path starting from node u

Transition

Story: "From node u, I can go to any neighbor v via a directed edge u→v. The longest path from u is the edge weight w(u,v) plus the longest path from v. I take the maximum over all outgoing edges."

dp[u] = max over all edges u→v of: w(u,v) + dp[v]

If u has no outgoing edges (it is a sink), dp[u] = 0.

Fill Order

Process nodes in reverse topological order (sinks first, sources last). This ensures that when we compute dp[u], all successors v have already been computed. Alternatively, use memoised DFS — the recursion naturally computes dependencies before the caller.

Code

#include <bits/stdc++.h>
using namespace std;

const int MAXN = 100005;
int n, m;
vector<pair<int,int>> adj[MAXN];   // adj[u] = {(v, weight)}
int dp[MAXN];
int saved[MAXN], sid;               // versioned memoization (see §16.5)

// Memoised DFS approach — natural for DAG DP
int solve(int u) {
    auto& ret = dp[u];
    if(saved[u] == sid) return ret;            // cache hit for THIS test case
    saved[u] = sid;
    ret = 0;                                    // base: no outgoing edges → longest path = 0
    for(auto [v, w] : adj[u])
        ret = max(ret, w + solve(v));
    return ret;
}

int main() {
    int T; cin >> T;
    while (T--) {
        cin >> n >> m;
        for(int i = 1; i <= n; i++) adj[i].clear();   // graph differs per test case
        for(int i = 0; i < m; i++) {
            int u, v, w;
            cin >> u >> v >> w;
            adj[u].push_back({v, w});
        }

        ++sid;                                  // O(1) "logical reset" of dp/saved
        int ans = 0;
        for(int u = 1; u <= n; u++)
            ans = max(ans, solve(u));

        cout << ans << '\n';
    }
}

Alternative: Topological Sort + Relaxation

// Iterative version using Kahn's algorithm for topological order
vector<int> indeg(n+1, 0);
for(int u = 1; u <= n; u++)
    for(auto [v, w] : adj[u]) indeg[v]++;

queue<int> q;
for(int u = 1; u <= n; u++)
    if(indeg[u] == 0) q.push(u);

vector<int> topo;
while(!q.empty()) {
    int u = q.front(); q.pop();
    topo.push_back(u);
    for(auto [v, w] : adj[u])
        if(--indeg[v] == 0) q.push(v);
}

// DP in topological order (forward: longest path ending at each node)
vector<int> dist(n+1, 0);
for(int u : topo)
    for(auto [v, w] : adj[u])
        dist[v] = max(dist[v], dist[u] + w);

int ans = *max_element(dist.begin()+1, dist.end());

Complexity

⏱

Time: O(n + m) — each node and edge is processed once, either in the topological sort or in the relaxation pass.

🗂

Space: O(n + m) — adjacency list and dp array.

The Deep Connection

🔗

Every DP is a DAG shortest/longest path. The states of any DP form a directed graph where each transition is an edge. This graph is acyclic because DP states are always computed from "smaller" to "larger" (by index, length, mask value, etc.). The topological order of this DAG is exactly the fill order. Recognising this means that whenever you see a DAG, you should instinctively think "DP over topological order."

Pitfalls

  • Applying this approach to a graph with cycles — it only works on DAGs. If the graph has cycles, the longest path problem becomes NP-hard.
  • Confusing longest path with shortest path — longest path in a DAG uses max and initialises distances to 0; shortest path uses min and initialises to ∞.
  • Stack overflow on deep DAGs with memoised DFS — use the iterative topological sort version for large graphs.

16. Space Optimization OPTIMIZATION

Space optimization compresses a DP table by discarding states that can never be needed again. The rule is simple: after computing dp[i], ask "which earlier states could dp[i+1], dp[i+2], … possibly need?" If the answer is "only the previous k layers," then you can keep only those k layers and discard the rest.

The backward loop proof for 0/1 Knapsack

This is the most frequently misunderstood space optimization in competitive programming. In the 2D recurrence, dp[i][cap] reads from dp[i−1][cap] and dp[i−1][cap−w[i]]. When we compress to a single array and update in-place, both the "previous row" and "current row" share the same memory. The question becomes: when we compute the updated dp[cap] for item i, has dp[cap−w[i]] already been overwritten in this same pass? If cap increases from 0 to W (forward loop), then when we reach cap, the value at cap−w[i] (which is smaller than cap) has already been updated with item i's effect — meaning item i could be "taken" multiple times, silently turning 0/1 knapsack into unbounded. By looping cap from W down to w[i] (backward), when we compute dp[cap], the value at cap−w[i] (smaller index) has not yet been touched in this pass, so it correctly represents "row i−1."

✓ 0/1 Knapsack — backward (correct)
for(int i=1;i<=n;i++)
  for(int cap=W; cap>=w[i]; cap--)
    dp[cap]=max(dp[cap],
               v[i]+dp[cap-w[i]]);
// dp[cap-w[i]] is still "row i-1"
✗ Forward loop — wrong for 0/1
for(int i=1;i<=n;i++)
  for(int cap=w[i];cap<=W;cap++)
    dp[cap]=max(dp[cap],
               v[i]+dp[cap-w[i]]);
// dp[cap-w[i]] may be "row i" already

LCS rolling rows

dp[i][j] depends on dp[i−1][j−1], dp[i−1][j], and dp[i][j−1]. The first two are from the previous row; the third is from the current row. Keeping two arrays (prev_row and cur_row) and swapping them after each row reduces space from O(nm) to O(m). Warning: once you compress to two rows, walking the table for reconstruction becomes impossible — keep the full table if you need to print the actual LCS.

16.5 Versioned Memoization — the test-case reset trick PERFORMANCE

Every recursive memoization in this guide uses a small but powerful template that you should adopt unconditionally for ICPC-style problems. It is called versioned memoization (also "stamp memoization" or "epoch memoization"). The idea is simple, the speed-up is dramatic, and the code is clean. This section explains it once so the rest of the file can use it freely.

The problem it solves

Many competitive programming problems give you multiple test cases in a single program execution. A typical input looks like:

int T; cin >> T;
while (T--) {
    // read this test case's input
    // solve it
    // print the answer
}

If you solve each test case with a recursive memoization, you have a memo array that must be reset between test cases — otherwise stale values from a previous test case would be returned as if they were valid for the new one. The naïve fix is to clear the memo at the top of each test case:

long long memo[MAXN][MAXM];

while (T--) {
    memset(memo, -1, sizeof(memo));   // ← O(MAXN · MAXM) per test case
    // ... solve ...
}

That's correct, but it is also often the bottleneck that causes TLE. Imagine MAXN = MAXM = 5000 and T = 10⁵ test cases — the memset alone is 5000·5000·10⁵ = 2.5·10¹² writes, far beyond any time limit. Worse, each individual test case might only visit a tiny fraction of the cells. You spent more time clearing the cache than solving the problem.

The trick — stamp every cell with a "version number"

Add a parallel integer array saved[] the same shape as the memo, and a single global counter sid ("session id" or "stamp id"). When you compute a state, you write its result into the memo AND stamp the corresponding saved[] cell with the current sid. Before each test case, you simply do ++sid. A cached value is considered valid for the current test case only if saved[state] == sid. Any older stamp means the cell is logically empty.

memo[state] holds a value · saved[state] holds the test-case id that wrote it · sid is the current test-case id
// === The Versioned Memoization Template ===
T mem[N];               // the actual memoization values
int saved[N], sid;      // saved[i] = test-case id when mem[i] was last written

T solve(state) {
    auto& ret = mem[state];
    if (saved[state] == sid)        // valid for THIS test case?
        return ret;
    saved[state] = sid;             // mark this state as visited NOW
    // ... compute the answer using recursive calls ...
    // ... save the answer in ret ...
    return ret;
}

// in main():
int T; cin >> T;
while (T--) {
    // read this test case's input
    ++sid;                          // O(1) "logical reset" of the entire memo
    cout << solve(initial_state) << '\n';
}

Why this is dramatically faster

  • No bulk reset. A single ++sid takes O(1) instead of O(memo size).
  • Pay only for what you touch. If a test case visits 100 of 25,000,000 cells, you stamp 100 cells. The other 24,999,900 simply remain "old version" and are correctly treated as empty by the next test case's check.
  • Same correctness as memset. Stale cells are unreachable through solve because the if (saved[state] == sid) check rejects them.

Multi-dimensional version

The trick generalises immediately to any number of state dimensions — just match the shape of saved[] to the shape of memo[]:

T mem[N][M];
int saved[N][M], sid;

T solve(int i, int j) {
    auto& ret = mem[i][j];
    if (saved[i][j] == sid) return ret;
    saved[i][j] = sid;
    // ... compute ret using recursive calls ...
    return ret;
}

Three small but important details

  1. Initialize sid = 0 at program start. Both saved[] and the global sid start as zero. The first ++sid makes sid = 1, which differs from every saved[i] = 0, so all cells are correctly seen as "empty" on the first test case.
  2. Use auto& ret = mem[state]; so that writing ret = ... stores the result into the memo without typing the (often long) index expression twice. This is also why the template assigns the result to ret, not directly to mem[...].
  3. Base cases that are constants do NOT need caching. Return them before the cache lookup (e.g. if (n <= 1) return n; in Fibonacci). They cost nothing to recompute, and skipping the cache makes the code clearer.

When NOT to use it

  • If you only ever solve one test case per program run, there is nothing to reset between runs — the trick is not needed (though it doesn't hurt either).
  • If your DP is iterative bottom-up, you initialise the table as part of the algorithm itself. The trick is purely for top-down memoised recursion.
  • If sid could conceivably reach the int max in a single run (over 2·10⁹ test cases — never going to happen in practice), you would need to reset back to 1 and clear saved[] once. In ICPC, this concern is theoretical.

17. Reconstruction Guide — 5 Patterns RECONSTRUCTION

🔗

This section is a deep dive on Step 3 of the 5-Step Method. Every problem in this guide had a "Step 3 — Build" subsection that asked: "at each state, which option produced the stored value?" The five patterns below are simply the five common shapes that the build function takes, depending on the structure of the recurrence. They are not new ideas — they are templates for the same Step 3 logic.

Most non-trivial contest problems ask for the actual answer, not just its optimal value. Reconstruction means tracing back through the DP table to find which choices produced the optimum. There are five recurring patterns, each matched to a specific class of DP problems.

Pattern 1 — Predecessor pointer

Store par[state] = previous state. Follow backward from final state. Used in: LIS, minimum coins, To Europe, rod cutting.

Pattern 2 — Take/skip table

Store take[i][cap] = 1 if item i was taken. Walk backward over rows. Used in: all knapsack variants.

Pattern 3 — Split point table

Store cut[l][r] = best split position k. Recursively reconstruct both halves. Used in: MCM, palindrome interval problems.

Pattern 4 — Table walk

Keep full table; at each cell check which recurrence case matches current value and move accordingly. Used in: LCS, edit distance.

Pattern 5 — End choice

Store ch[l][r] = 0 (left) or 1 (right). Shrink interval by following ch. Used in: Treats for the Cows, palindrome variants.

⚠ Atomic update rule

Whenever the optimal value changes, update the witness (par/take/cut/ch) in the same if-block. Never update value and witness in separate passes — ties cause inconsistencies.

Common Beginner Confusions

DP ≠ "just memoization"

Memoization is one evaluation strategy for DP (the top-down variant). Tabulation (bottom-up) is another. DP is the problem-solving principle — recognising overlapping subproblems and optimal substructure. The strategy is secondary to the insight.

Greedy vs DP

Greedy makes one locally optimal choice and never reconsiders. DP considers all choices and picks the globally optimal one. If a greedy choice can be proven to always be part of some optimal solution (via exchange argument, etc.), greedy suffices and is faster. Otherwise, you need DP.

Backtracking vs DP

Backtracking explores the entire search tree with pruning. DP collapses repeated subtrees into single computations via memoization. If subproblems repeat, DP is exponentially faster than backtracking. If every subproblem is unique (e.g., N-Queens), DP offers no advantage.

"State too large" vs "State too weak"

"Too large" means the DP table exceeds memory or time — reduce dimensions. "Too weak" means the state does not capture enough information for a correct transition — add dimensions. Finding the right balance is the core DP design skill.

⚠

The answer is not always dp[n]. For LIS, it is max over all dp[i]. For bitmask TSP, it is min over all last cities in the full mask. For coin counting, it is dp[X]. For interval DP, it is dp[1][n]. Always identify which cell(s) contain the original problem's answer — this is step 5 of the design framework.

Common DP Misclassification Traps

In a contest, hasty pattern-matching can lead you to the wrong DP family. These traps catch even experienced programmers:

⚠ "capacity" ≠ always knapsack

A capacity constraint might appear in a graph (max-flow), a greedy (fractional knapsack), or a prefix-partition problem. Only add a capacity dimension to your DP if items are being selected independently with a cumulative weight constraint.

⚠ "two strings" ≠ always LCS

Two strings might require edit distance, interleaving, regex matching, or even digit DP if one string represents a number bound. Check the operation set before defaulting to LCS.

⚠ "try all subsets" ≠ always bitmask DP

Bitmask DP requires n ≤ 20 and a state that depends on which elements are in the subset. If n is large, you need a different approach entirely (greedy, flow, knapsack). If the subset has additional structure (e.g., it must be contiguous), bitmask is the wrong family.

⚠ "graph" ≠ always DP

DP on a graph requires a DAG, a tree, or a layered state structure. If the graph has arbitrary cycles, DP does not apply directly — you may need shortest-path algorithms, BFS/DFS, or network flow instead.

🔑

"Brute force with memo" is not enough unless the state is correct. Memoising a bad state gives a fast but wrong answer. Always verify: does the memoised state capture everything the future needs? If two calls with the same state can produce different optimal values, the state is incomplete.

19. Pitfalls & Debug Checklist MISTAKE BOOK

Most DP bugs fall into a small number of recurring categories. Learning to recognise the category from the symptom dramatically speeds up debugging.

Bug 1 — State stores too much or too little most common design error

Too much: Storing the entire subsequence or whole partial construction makes the state space exponential and defeats DP entirely. Always ask: "what is the minimum information the future decisions need?" For LIS it is the last element's position; for Array Description it is the last value; for LCS it is the two prefix lengths.

Too little: Dropping a variable that the transition actually requires makes the recurrence impossible to compute correctly. Symptom: two states look the same but have different optimal values — you need an extra state dimension.

Bug 2 — dp[0]=0 instead of 1 in counting DP kills the entire chain

For coin/sequence counting problems, dp[0] = 1 represents "one way to make sum 0: choose nothing." This is not a coincidence — it is the seeding value that propagates through the recurrence to give all other values. If dp[0]=0, then dp[1]=dp[2]=…=dp[X]=0 regardless of what coins are available. Every non-trivial result will be 0.

Bug 3 — Forward loop in 0/1 knapsack silent unbounded-knapsack conversion

Looping capacity forward in the 1D 0/1 knapsack allows each item to be "taken" multiple times within the same pass, silently converting it to unbounded knapsack. There is no error message — the code runs and produces a larger-than-correct answer that can look plausible. Always check: is each item allowed to be used more than once? If no, loop backward.

Bug 4 — Interval DP fill order violation reads uninitialised cell

dp[i][j] for interval DP depends on shorter intervals dp[i][k] and dp[k+1][j]. If you fill by (i from 1 to n, j from i+1 to n), dp[1][n] is attempted before dp[2][n] and dp[1][n−1] exist. The correct fill order is by increasing interval length: len from 2 to n, then i from 1 to n−len+1, with j=i+len−1.

Bug 5 — Answer in the wrong cell looks plausible but wrong

The answer is not always in the last cell. For LIS it is max over all dp[i]. For Array Description it is sum over all dp[n][x]. For Domino Tiling it is dp[last_row][mask=0]. For bitmask TSP it is min over all last cities. Always think explicitly about which cell(s) contain the answer to the original question.

Emergency debug checklist — run this before giving up

  • Say the state meaning out loud in one precise sentence. If you can't, the state is unclear.
  • Identify the answer type (decision / optimisation / counting / construction) and confirm the operator matches.
  • Check every base case: smallest valid state, impossible/empty state, boundary rows/columns.
  • Manually trace a tiny 3–5 element example and compare cell by cell with the expected values.
  • Confirm the final answer reads from the correct cell(s) — not always the last one.
  • Verify loop order is consistent with dependencies — every cell is computed only after all its predecessors.
  • Check default initialisation — ∞ for min problems, 0 for count problems, false for decision problems.
  • If reconstruction is wrong but value is correct, confirm that witness (par/take/cut/ch) is updated atomically with the value.
