Divide & Conquer

A complete treatment from first principles — what D&C is, why it works, how to derive recurrences from code, and how to implement every major algorithm. Beginner-friendly ramp, rigorous depth.

Overview PARADIGM

Divide and conquer is one of the three fundamental algorithm design paradigms in competitive programming, alongside dynamic programming and greedy. It drives the most important sorting algorithms, the fastest selection algorithm, the optimal closest-pair algorithm, and a large family of recursive construction problems in contests.

Every D&C algorithm varies on exactly three axes:
(1) How to split  ·  (2) What each recursive call returns  ·  (3) How to combine.
The recursion tree is the same structure every time.

1. Problem Definition PARADIGM

💡

In plain language — before anything else: Divide and conquer means three things in order: (1) Break the big problem into smaller independent copies of the same problem. (2) Solve each smaller copy recursively. (3) Combine the smaller answers into the answer for the big problem. That is all it is. Every algorithm in this tutorial is a specialization of exactly these three steps.

What does "independent" mean here? It means the two smaller copies do not share data and do not need each other's answers to run. Merge sort's left half and right half never interact during recursion — only during the combine step. This independence is what separates D&C from dynamic programming (where subproblems share work) and from backtracking (where the search explores state that depends on other choices).

What does "combine" really mean? It depends on the algorithm. In merge sort, combine means merging two sorted halves into one sorted array — this is O(n) work. In inversion count, combine means counting how many pairs have one index in each half — also O(n) during the merge. In closest pair, combine means scanning a narrow strip between the two halves to find cross-pairs — still O(n). The combine step is where the algorithm's real work happens. Splitting is almost always trivial.

The four paradigms compared — know the difference before you code

⚠

Why DP is not D&C: In D&C the subproblems are disjoint — the left half of your array appears in exactly one recursive call. In DP the same subproblem appears in many branches. Applying naive D&C to a DP problem gives exponential time (you solve the same subproblem repeatedly). Applying memoization to D&C does nothing useful (each subproblem already appears only once). If you see repeated subtrees in your recursion tree, switch to DP.

⚠

Why greedy is not D&C: Greedy never recurses on a full subproblem — it commits to one local choice and moves on. D&C recurses on full subproblems and combines their complete answers. Quickselect recurses on one subproblem (not a local choice) — it is D&C, not greedy.

Divide and conquer is a structural paradigm for algorithm design. It means: divide the problem into smaller subproblems of the same type, conquer them recursively, and combine their answers into the answer for the original problem.

✗

Common misconception: "Any recursive algorithm is divide and conquer." This is false. A recursive algorithm is D&C only when (a) the recursive calls are smaller versions of the same problem, and (b) the whole answer comes from combining those sub-answers in a structured way. DFS on a tree is recursive but is not D&C. Fibonacci with memoization is recursive DP, not D&C.

The three structural properties a problem must have to fit D&C:

Three Structural Requirements
①
Decomposability
The problem becomes easier as the input size shrinks. There exists a natural base case.
②
Same-Kind Subproblems
Recursive calls solve smaller instances of exactly the same kind of problem.
③
Combinability
The parent's answer can be assembled from the children's answers, often in sub-linear time.
🔑

The combine step is the heart. Splitting is almost always easy — just take the midpoint. The real challenge in D&C is almost always the combine step. In merge sort, combining is O(n) merge. In inversion count, combining counts cross-inversions. In closest pair, combining handles cross-strip pairs. The quality of a D&C solution is determined by the quality of its combine step.

2. Intuition from Scratch PARADIGM

The simplest example: find the maximum element of an array of 8 elements by divide and conquer.

Observation: The maximum of the whole array is the maximum of the left half vs the maximum of the right half. This is a valid decomposition because "find the maximum" is the same problem at both halves.

Recursion tree for max([2,1,7,3,5,6,4,8])

Notice the tree structure: the root is the original problem, each node is a smaller problem, leaves are trivially solved (size 1), and answers flow upward. This tree is the same structure for every D&C algorithm. Only the combine rule and the information flowing upward change.

💡

The key insight: In merge sort, the information flowing upward is "a sorted segment." In inversion count, it is "a sorted segment + an inversion count." In closest pair, it is "points sorted by y + a minimum distance." The recursion tree structure is identical every time. Only the payload changes.

Compare to linear scanning: linear scan is O(n) and D&C max is also O(n) for this problem, so D&C is not always faster than a simple approach. The paradigm shines when the combine step enables something impossible with a simple scan — like merge sort achieving stable O(n log n), or closest pair achieving O(n log n) instead of O(n²).

3. Visual Mental Model PARADIGM

Every D&C algorithm lives inside a binary tree. Learn to see that tree clearly before writing any code.

The standard picture

At each node, the algorithm:

  1. Receives a subproblem (typically an interval [l, r))
  2. Splits it at mid = (l+r)/2
  3. Recursively calls left child on [l, mid)
  4. Recursively calls right child on [mid, r)
  5. Combines the two results and returns
📌

Half-open intervals [l, r). This tutorial uses half-open ranges consistently. Length = r−l. Empty = l==r. One element = r−l==1. Left half = [l,mid). Right half = [mid,r). This avoids off-by-one errors.

What the tree tells you about complexity

Height of tree = O(log n) for balanced splits
Total nodes = O(n) for two-way split (sum: 1+2+4+…+n)
If combine costs O(1) → Total = O(n) [case 1 of Master Method]
If combine costs O(n) → Each level costs O(n) → Total = O(n log n) [case 2]

The tree makes complexity visible without algebra. Use it first, then verify with the Master Method.

4. Formal Abstraction PARADIGM

The universal skeleton

answer solve(int l, int r) { // [l, r)
    if (r - l <= base_size) return direct_answer(l, r);

    int mid = (l + r) >> 1;

    answer left  = solve(l, mid);
    answer right = solve(mid, r);

    return combine(left, right, l, mid, r);
}

5. Key Observations PARADIGM

💡

What "independent subproblems" means in practice: When you sort the left half of an array in merge sort, the answer for the right half does not change and the right half does not need to know anything about the left half's internal process — only its final sorted result. The two halves do not communicate during their recursion. This is what "independent" means. The moment subproblems start sharing internal state or reusing each other's partial results, you have moved into DP territory.

D&C vs Dynamic Programming vs Greedy

ParadigmSubproblemsDependencyKey conditionClassic examples
Divide & ConquerSmaller versions of same problemIndependentCombine is cheap enoughMerge sort, closest pair, quickselect
Dynamic ProgrammingOverlapping subproblemsShared — reuse saved answersOptimal substructureKnapsack, LCS, shortest paths
GreedyNo subproblems neededOne local choice at a timeLocally optimal is globally optimalInterval scheduling, Huffman, Dijkstra
🔑

The critical distinction between D&C and DP: In D&C, subproblems are disjoint and solved independently. In DP, subproblems overlap and the same subproblem is referenced by multiple parents, so we save and reuse its answer. Applying D&C to an overlapping-subproblem structure gives exponential time. Memoizing it gives DP. If your D&C call tree has repeated subtrees, you need DP instead.

The "three categories" pattern

In most D&C problems, the final answer (count, minimum, maximum, etc.) falls into one of three categories:

Entirely in left half

Handled by left recursive call

Entirely in right half

Handled by right recursive call

One index in each half (cross)

Handled by the combine step

This is the fundamental structure behind inversion count (pairs), closest pair (point pairs), and many other D&C problems. When you see a problem that involves counting or optimizing over pairs, this three-way decomposition is your first tool.

6. Complexity Analysis PARADIGM

Recurrences are not magic — they come directly from your code

Every recurrence is just a translation of what your code does, written as a formula. Nothing is invented. You read the code, you count the calls, you measure the non-recursive work, and you write it down. That is all.

Example: derive T(n) directly from merge sort code

Look at the merge sort skeleton:

void merge_sort(a, l, r) {
    if (r - l <= 1) return;          // base case: O(1)
    int mid = (l + r) >> 1;
    merge_sort(a, l, mid, ...);       // recursive call 1 — size n/2
    merge_sort(a, mid, r, ...);       // recursive call 2 — size n/2
    // merge two halves: O(n) work
}

Reading T(n) = 2T(n/2) + O(n) in plain language:

  • 2 = there are 2 recursive calls in the code
  • T(n/2) = each call gets half the data (r−l halves each time)
  • + O(n) = the merge loop outside the recursion does O(n) work

✓ Recurrence derivation checklist — 3 questions, every time

  1. How many recursive calls appear in the function body? → that number becomes the coefficient (a)
  2. What size is each recursive call? → if it is n/2, write T(n/2); if n−1, write T(n−1)
  3. How much work is done outside the recursive calls at this level? → that is f(n) in T(n) = a·T(n/b) + f(n)

If you can answer these three questions for your code, you have the recurrence. No guesswork needed.

How to build a recurrence from code

Look at the code and answer three questions: (1) How many recursive calls? (2) What size is each? (3) How much non-recursive work?

T(n) = (number of calls) × T(n / shrink factor) + (work per node)

The recurrence cookbook — five families

Family 1 — Binary search style
T(n) = T(n/2) + O(1)  →  O(log n)
One child, half size, constant work. Height = log n levels, each O(1). Sum = O(log n).
↓
Family 2 — Tree leaf dominance
T(n) = 2T(n/2) + O(1)  →  O(n)
Two children, half size, constant combine. The leaf count (n) dominates. Sum = 1+2+4+…+n = O(n).
↓
Family 3 — Merge sort (balanced levels)
T(n) = 2T(n/2) + O(n)  →  O(n log n)
Two children, half size, linear combine. Each of the O(log n) levels costs O(n). Every level does exactly the same amount of total work. Sum = O(n) × O(log n) = O(n log n).
↓
Family 4 — Balanced quickselect
T(n) = T(n/2) + O(n)  →  O(n)
One child, half size, linear work. Work shrinks geometrically: n + n/2 + n/4 + … = O(2n) = O(n).
↓
Family 5 — Bad quicksort (degenerate)
T(n) = T(n−1) + O(n)  →  O(n²)
One child of size n−1. Work is n + (n−1) + (n−2) + … + 1 = O(n²). This happens when the pivot is always the minimum or maximum.

Master Method from zero

The Master Method is just a shortcut for reading off the answer to T(n) = a·T(n/b) + f(n) without drawing the full recursion tree every time. You do not need to memorize it by rote — understand where it comes from and the cases become obvious.

💡

The intuition behind the three cases, in plain language: There are two sources of work in any D&C algorithm: (1) the work done at the leaves of the recursion tree (the leaf count = nlogba), and (2) the work done outside the recursion at each node (f(n) per level, times O(log n) levels). The Master Method just asks: which of these two sources is bigger?

Case 1 (leaves dominate): f(n) is small — the algorithm branches aggressively and the leaves pile up. Total work ≈ leaf count.
Case 2 (balanced): f(n) ≈ leaf-count-per-level — every level contributes equally, so you get log n equal-cost levels, adding a log factor.
Case 3 (root dominates): f(n) is large — most work happens at the top level, and the recursive calls are cheap by comparison.

The standard form: T(n) = a·T(n/b) + f(n) where a = number of children, b = shrink factor, f(n) = non-recursive work per node.

The comparison quantity is nlogb a — this is the size of the recursion tree if we ignore combine work. It equals the leaf count: alogb n = nlogb a.

The Master Method requires all subproblems to have the same size. It does not apply to:

7. Merge Sort SORT

Problem story: Sort an array. Sorting the whole array directly is hard. But if we sort the left half and sort the right half, then merging two sorted halves into one sorted array is easy — just compare the two front elements repeatedly.

✗ Why naive approaches fail for sorting

Selection sort / insertion sort: O(n²). For n = 10⁶ that is 10¹² operations — impossible in a contest.

Can we do better? Yes. If we recursively sort both halves, each half costs T(n/2). Then merging costs O(n). Total: T(n) = 2T(n/2) + O(n) = O(n log n). For n = 10⁶ that is about 20 × 10⁶ operations — fast enough.

🔑

Key invariant during merge: At every step, temp[l..k) already contains the smallest (k−l) elements from both halves, in sorted order. This invariant is why merge produces a sorted result.

Code — Merge Sort

#include <bits/stdc++.h>
using namespace std;

void merge_sort(vector<int>& a, int l, int r, vector<int>& temp) { // [l, r)
    if (r - l <= 1) return;

    int mid = (l + r) >> 1;
    merge_sort(a, l, mid, temp);
    merge_sort(a, mid, r, temp);

    int i = l, j = mid, k = l;

    while (i < mid && j < r) {
        if (a[i] <= a[j]) temp[k++] = a[i++];  // <= ensures stability
        else               temp[k++] = a[j++];
    }

    while (i < mid) temp[k++] = a[i++];
    while (j < r)   temp[k++] = a[j++];

    for (int p = l; p < r; ++p) a[p] = temp[p];
}

int main() {
    vector<int> a = {5, 2, 8, 1, 3, 7, 6, 4};
    vector<int> temp(a.size());
    merge_sort(a, 0, (int)a.size(), temp);
    for (int x : a) cout << x << ' ';
}
✓

Why <= makes it stable: When a[i] == a[j], we take from the left half first. Equal elements from the left half always precede equal elements from the right half. This preserves relative order → stable sort.

Complexity

T(n) = 2T(n/2) + O(n) → O(n log n) time, O(n) space

Best case = worst case = O(n log n). Unlike quicksort, merge sort has no bad input. Memory is O(n) for the temp array.

When merge sort matters in CP

  • You need a stable sort
  • You are counting inversions
  • You need offline geometric merges
  • You need to process cross-pairs during combine

Bottom-up iterative variant

Merge sort can also run iteratively: start with segments of size 1, then 2, 4, 8, … until the whole array is merged. Same O(n log n) time. No recursion stack. Slightly lower constant factor. The recursive version is usually clearer for understanding D&C structure.

8. Inversion Count MERGE APPLICATION

Problem: Given an array, count pairs (i, j) where i < j and a[i] > a[j]. These are called inversions because a[i] and a[j] are in the "wrong" relative order compared to a sorted array. The inversion count measures how far from sorted the array is.

✗ Brute force: O(n²)

long long inv = 0;
for (int i = 0; i < n; i++)
    for (int j = i+1; j < n; j++)
        if (a[i] > a[j]) inv++;
// O(n²) — too slow for n = 10⁵

Why D&C fits — the three-category argument: Every inversion is either both indices in the left half, both in the right half, or one in each. The first two are recursive. The third happens during the merge.

🔑

Cross-inversion counting trick: During merge, suppose we take element a[j] from the right half. Because the left half is sorted and a[i] > a[j], every remaining left element also exceeds a[j]. So we add mid − i at once. This is the single insight that makes the algorithm O(n log n).

Inversion count — merge phase cross-pair counting
The two halves [2, 3, 8] and [1, 6] are sorted. During merge, when we take 1 from the right half, all 3 remaining left elements exceed it → add 3 inversions at once.
#include <bits/stdc++.h>
using namespace std;

long long count_inversions(vector<int>& a, int l, int r, vector<int>& temp) { // [l, r)
    if (r - l <= 1) return 0;

    int mid = (l + r) >> 1;
    long long ans = 0;

    ans += count_inversions(a, l, mid, temp);
    ans += count_inversions(a, mid, r, temp);

    int i = l, j = mid, k = l;

    while (i < mid && j < r) {
        if (a[i] <= a[j]) {
            temp[k++] = a[i++];
        } else {
            temp[k++] = a[j++];
            ans += (mid - i);  // a[i..mid) all > a[j]
        }
    }

    while (i < mid) temp[k++] = a[i++];
    while (j < r)   temp[k++] = a[j++];

    for (int p = l; p < r; ++p) a[p] = temp[p];
    return ans;
}
⚠

Use long long: The answer can be up to n(n−1)/2 ≈ 5×10¹⁰ for n=10⁵. Always declare ans as long long.

Complexity: same as merge sort

T(n) = 2T(n/2) + O(n) → O(n log n)

The ans += (mid - i) line is O(1). The merge still costs O(n). The recurrence is unchanged from merge sort.

🎯

Contest significance: Inversion count is the model for all "count cross-pairs" D&C problems. Once you understand it, problems like "count pairs (i,j) where a[i] > 2·a[j]" become natural: they use the same merge structure, with the comparison condition changed.

9. Quicksort SORT

The big idea: Instead of splitting by position and fixing order later (merge sort), quicksort chooses a pivot — a reference value — and rearranges (partitions) the array so that smaller elements go left and larger elements go right. After this partitioning, the pivot is in its final sorted position. Recurse on both sides. The combine step is trivial: partition already guaranteed that every left element ≤ pivot ≤ every right element, so the two sorted sides just concatenate correctly. No merge needed.

💡

Merge sort vs quicksort — the structural difference: Merge sort does its real work after the recursive calls (during the merge). Quicksort does its real work before the recursive calls (during the partition). In merge sort the split is always balanced (midpoint) but the combine is O(n). In quicksort the split depends on the pivot (potentially unbalanced) but the combine is O(1). The complexity consequences of this difference are significant — see the pivot strategy table below.

Partition correctness invariant (Lomuto scheme)

At all times during partition: a[l..i) contains elements < pivot, a[i..j) contains elements ≥ pivot, a[j..r−1) is unprocessed.

3-way randomized quicksort — the recommended template

#include <bits/stdc++.h>
using namespace std;

mt19937 rng((uint32_t)chrono::steady_clock::now().time_since_epoch().count());

void quick_sort_3way(vector<int>& a, int l, int r) { // [l, r)
    if (r - l <= 1) return;

    int pivot = a[uniform_int_distribution<int>(l, r - 1)(rng)];

    // Partition into: [<pivot][==pivot][>pivot]
    int i = l, j = l, k = r;
    while (i < k) {
        if      (a[i] < pivot) { swap(a[i], a[j]); i++; j++; }
        else if (a[i] > pivot) { --k; swap(a[i], a[k]); }
        else                   { i++; }
    }
    // a[l..j) < pivot, a[j..k) == pivot, a[k..r) > pivot

    quick_sort_3way(a, l, j);
    quick_sort_3way(a, k, r);
}
✓

Why 3-way is better for duplicates: If the array has many equal elements, 2-way partition degenerates: elements equal to the pivot are processed inefficiently. 3-way partition places all equal elements in the middle region and never recurses on them again. For arrays with many duplicates, 3-way quicksort is O(n log k) where k is the number of distinct values — it can be faster than O(n log n) merge sort.

10. Quickselect SELECTION

Problem: Given n unsorted numbers, find the k-th smallest. You do not need them all sorted — just the one at rank k.

✗ Naive approach: sort first, then index

Sort the entire array (O(n log n)), then return a[k−1]. This works, but it does far too much work: sorting puts every element in its final position, but you only need the element at rank k.

Every element participates in every level of the merge sort recursion. You cannot escape paying O(n log n) if you sort.

✓ Quickselect: recurse into ONE side only

After partition around a pivot, you know exactly which side contains rank k. The pivot lands at some position p. If k < p, the answer is in the left half. If k > p, it is in the right half. If k == p, you found it.

Key insight: you discard half the data at each step. This changes the recurrence from 2T(n/2)+O(n) (both halves, like quicksort) to T(n/2)+O(n) (one half only). That single change goes from O(n log n) to O(n).

The key difference from quicksort: After partition, only one branch can contain the k-th element. Recurse into that branch only. This changes the recurrence from 2T(n/2)+O(n) (merge sort) to T(n/2)+O(n) (balanced quickselect).

n + n/2 + n/4 + ··· = O(2n) = O(n) ← geometric series, work shrinks each level

Contrast with sorting: in sorting you must recurse on both halves, so the work does not shrink — it stays O(n) per level for O(log n) levels = O(n log n). Quickselect escapes this because it throws away the irrelevant half.

🔑

Why the recurrence gives linear time: The partition work shrinks geometrically because each level handles half as much data (on average). This geometric convergence is why quickselect is faster than sorting — sorting must process all elements at every recursion level, but quickselect discards half the data each time.

#include <bits/stdc++.h>
using namespace std;

mt19937 rng((uint32_t)chrono::steady_clock::now().time_since_epoch().count());

int partition_rand(vector<int>& a, int l, int r) { // [l, r)
    int pivot_idx = uniform_int_distribution<int>(l, r - 1)(rng);
    swap(a[pivot_idx], a[r - 1]);

    int pivot = a[r - 1], i = l;
    for (int j = l; j < r - 1; ++j)
        if (a[j] < pivot) swap(a[i++], a[j]);
    swap(a[i], a[r - 1]);
    return i;
}

// Returns element with 0-based rank k in a[l..r)
int quick_select(vector<int>& a, int l, int r, int k) {
    while (true) {
        if (r - l == 1) return a[l];

        int p = partition_rand(a, l, r);

        if      (p == k) return a[p];
        else if (k < p)  r = p;
        else             l = p + 1;
    }
}

The idea: To guarantee O(n) even in the worst case, we need a pivot that always splits the array into at least a 30/70 ratio.

  1. Divide the array into groups of 5.
  2. Find the median of each group by sorting (each group has constant size, so this is O(n) total).
  3. Recursively find the median of those medians — this is T(n/5).
  4. Use that median as the pivot.
  5. The guarantee is that at least 3n/10 elements are < pivot and at least 3n/10 are > pivot.
T(n) = T(n/5) + T(7n/10) + O(n) → O(n)

The constant factor is high (~30–40×), so random-pivot quickselect is faster in practice. Median of medians is important for theory and for problems requiring a deterministic guarantee.

11. Closest Pair of Points GEOMETRY

Problem: Given n points in the plane, find the pair of distinct points with the smallest Euclidean distance between them. Return that minimum distance.

✗ Brute force: check every pair

Try every pair (i, j) with i < j. There are n(n−1)/2 ≈ n²/2 pairs. Each distance computation is O(1). Total cost: O(n²).

For n = 10⁵, that is about 5 × 10⁹ operations — far too slow. We need a way to skip pairs that clearly cannot be the closest without checking them.

// Naive O(n²)
long long ans = LLONG_MAX;
for (int i = 0; i < n; i++)
    for (int j = i+1; j < n; j++)
        ans = min(ans, dist2(pts[i], pts[j]));
🔑

Why the combine step is the real difficulty: Splitting is trivial — sort by x, take the midpoint. Solving each half recursively is also trivial (trust the recursion). The hard part is the combine step: after you know the best distance in the left half (d_L) and the best in the right half (d_R), how do you efficiently find whether any cross pair (one point in each half) does better? Without geometric insight, checking all cross pairs is O(n²) again — which destroys the whole algorithm. The entire cleverness of this algorithm is in proving the combine step can be done in O(n) time.

💡

Simple intuition before the formal argument: Let d = min(d_L, d_R). A cross pair can only improve d if both points are very close to the dividing line — within distance d horizontally. Everything farther away is already worse than d and can be ignored. This narrows the search from n² pairs to a narrow strip. Within that strip, sorting by y and scanning nearby neighbors in y turns out to need at most a constant number of checks per point. That is the O(n) combine.

⚠

Common confusion: "The hard part of closest pair is splitting." It is not. Splitting is just sort-by-x and take-the-midpoint. The hard part is proving the combine step — specifically, proving why only O(1) y-neighbors need to be checked in the strip. Without that proof you cannot claim O(n log n). Do not skip the packing argument below.

Why this is the deepest D&C problem in this tutorial: The split is trivial (sort by x, split at midpoint). The recursion is standard (trust both halves). The difficulty is entirely in the combine step — specifically, in proving that checking cross-border pairs does not blow up to O(n²). Everything below exists to establish that one fact.

The algorithm — five steps

Step 1
Sort points by x-coordinate
O(n log n) one-time cost before recursion.
↓
Step 2 — Recurse
Solve left half and right half
Get d_L = closest in left half, d_R = closest in right half. Set d = min(d_L, d_R).
↓
Step 3 — The strip observation
Any cross pair with distance < d must lie within a strip of width 2d around the dividing line
Points with |x − midx| ≥ d cannot improve d. So discard all other points — only strip points matter for the cross contribution.
↓
Step 4 — The packing argument
For each strip point, only check a constant number of y-neighbors
Inside the relevant 2d × d rectangle, no two points from the same half are closer than d. By a packing argument, at most 8 points can fit. So checking the next 7 y-neighbors suffices.
↓
Step 5 — Merge-sort trick
Maintain y-sorted order through recursion
If we re-sort by y at each level, combine costs O(n log n) → total O(n log² n). If we merge y-sorted lists from children (like merge sort), combine costs O(n) → total O(n log n).

What remains to prove: We must show that for each strip point, only a constant number of y-neighbors can possibly be closer than d. If that is true, the inner loop runs O(1) times per point, and the total combine cost is O(n). The packing argument below establishes exactly this.

🎯

Why only a constant number of strip points need to be checked per point: After narrowing to the strip, sort the strip by y. For any strip point p, you only look at strip points with y-coordinate within d of p's y-coordinate. Inside this d-wide y-window, draw two d×d squares (one on each side of the dividing line). Each square has the property that all points inside it are from the same recursive half — and that half already found its minimum distance d. So no two points from the same half can be closer than d in that square. By a packing argument (pack circles of radius d/2 in a 2d×d rectangle), at most 8 points can fit in that window. This is why the inner loop breaks early and the total combine cost is O(n).

⚠

Squared distances: Always compare squared distances to avoid square root precision issues. dx*dx + dy*dy is an integer if coordinates are integers. Only convert to floating point at the very end if needed.

#include <bits/stdc++.h>
using namespace std;

struct Point { long long x, y; };

long long dist2(const Point& a, const Point& b) {
    long long dx = a.x - b.x, dy = a.y - b.y;
    return dx*dx + dy*dy;
}

long long closest_dc(vector<Point>& pts, int l, int r, vector<Point>& tmp) {
    int len = r - l;
    if (len <= 3) {
        long long ans = (long long)4e18;
        for (int i = l; i < r; i++)
            for (int j = i+1; j < r; j++)
                ans = min(ans, dist2(pts[i], pts[j]));
        sort(pts.begin()+l, pts.begin()+r, [](const Point& a, const Point& b){
            return a.y != b.y ? a.y < b.y : a.x < b.x; });
        return ans;
    }

    int mid = (l + r) >> 1;
    long long midx = pts[mid].x;

    long long d = min(closest_dc(pts, l, mid, tmp),
                      closest_dc(pts, mid, r, tmp));

    // Merge by y (like merge sort, O(n) combine)
    int i = l, j = mid, k = l;
    while (i < mid && j < r)
        tmp[k++] = (pts[i].y < pts[j].y) ? pts[i++] : pts[j++];
    while (i < mid) tmp[k++] = pts[i++];
    while (j < r)   tmp[k++] = pts[j++];
    for (int p = l; p < r; p++) pts[p] = tmp[p];

    // Strip scan
    vector<Point> strip;
    for (int p = l; p < r; p++) {
        long long dx = pts[p].x - midx;
        if (dx*dx < d) strip.push_back(pts[p]);
    }
    int s = (int)strip.size();
    for (int a = 0; a < s; a++)
        for (int b = a+1; b < s; b++) {
            long long dy = strip[b].y - strip[a].y;
            if (dy*dy >= d) break;  // y-sorted: safe to break
            d = min(d, dist2(strip[a], strip[b]));
        }
    return d;
}

long long closest_pair_sq(vector<Point> pts) {
    int n = pts.size();
    sort(pts.begin(), pts.end(), [](const Point& a, const Point& b){
        return a.x != b.x ? a.x < b.x : a.y < b.y; });
    vector<Point> tmp(n);
    return closest_dc(pts, 0, n, tmp);
}
T(n) = 2T(n/2) + O(n) → O(n log n)

📌

Merge invariant: At every step of the merge loop, temp[l..k) contains the smallest (k−l) elements from the two halves, in sorted order. Pointers i and j point to the smallest unplaced element in each half.

📌

Partition invariant (Lomuto): At all times, a[l..i) ≤ pivot, a[i..j) > pivot, a[j..r−1) = unprocessed. When the loop ends, swapping a[i] and a[r−1] places the pivot at its final position i.

📌

Closest pair strip invariant: When scanning the strip sorted by y, once dy² ≥ d, no remaining point can improve d. This justifies the break in the inner loop.

C. Termination invariants

Every D&C algorithm here terminates because the input size strictly decreases at each recursive call (r−l decreases), and the base case handles size 1 (or size ≤ 3 for closest pair). The only potential issue is if mid == l or mid == r, which cannot happen when r−l ≥ 2 with the formula mid = (l+r)>>1.

13. Proof of Correctness CORRECTNESS

Every D&C proof follows the same four-step template. Memorize the structure, not the specific proofs.

The universal D&C proof template

  1. State the contract: What does the recursive function return? What postcondition holds when it returns?
  2. Prove the base case: When size ≤ threshold, direct computation is clearly correct.
  3. Apply the induction hypothesis: Assume both recursive calls satisfy their postcondition. Do not re-prove them — trust the contract.
  4. Prove the combine step is correct: Show that the combine step correctly assembles the parent's answer from the children's guaranteed outputs. Enumerate all answer categories and show each is covered.
Proof: Merge Sort is correct click to expand

Contract: merge_sort(a, l, r) sorts a[l..r) in non-decreasing order.

Base case: r−l ≤ 1. A segment of 0 or 1 elements is trivially sorted. ✓

Induction hypothesis: merge_sort(a, l, mid) correctly sorts a[l..mid), and merge_sort(a, mid, r) correctly sorts a[mid..r).

Combine step: After recursive calls, both halves are sorted. The merge procedure maintains the invariant: temp[l..k) is the sorted sequence of the smallest (k−l) elements from both halves. By induction on k, this is maintained at every step. When the merge terminates, temp[l..r) contains all elements of a[l..r) in sorted order. Copying back to a completes the proof. ∎

Proof: Inversion Count is correct click to expand

Contract: count_inversions(a, l, r) returns the number of inversions in a[l..r) and sorts a[l..r).

Base case: Size ≤ 1. No inversions, trivially sorted. ✓

Induction hypothesis: Both recursive calls return correct inversion counts for their respective halves and sort those halves.

Combine step: After recursive calls, both halves are sorted. We claim: every inversion in a[l..r) is either (a) within [l,mid), (b) within [mid,r), or (c) a cross inversion (i in left, j in right with a[i] > a[j]). Types (a) and (b) are counted by the recursive calls. Type (c): during merge, when we take a[j] from the right and a[i] > a[j] (so we're on the else branch), the sorted left half means a[i], a[i+1], …, a[mid−1] all exceed a[j]. We add exactly (mid−i) inversions. By the merge invariant, each such cross inversion is counted exactly once. ∎

Proof: Closest Pair is correct click to expand

Contract: closest_dc(pts, l, r) returns the minimum squared distance among pts[l..r) and leaves pts[l..r) sorted by y.

Base case: r−l ≤ 3. Brute force checks all pairs. Manual y-sort. ✓

Induction hypothesis: Both recursive calls return correct minimum distances and sort their halves by y.

Combine step: Any closest pair is either both in the left half, both in the right half, or a cross pair. The first two are handled by the recursive calls giving d_L and d_R. Let d = min(d_L, d_R). For a cross pair to improve d, both points must lie within the strip |x − midx| < sqrt(d). The strip is processed in y order. For each strip point p, we check subsequent strip points q. When (q.y − p.y)² ≥ d, the Euclidean distance already exceeds sqrt(d), and since strip is y-sorted, all later q's are even further in y. The packing argument: in the 2sqrt(d) × sqrt(d) rectangle relevant to p, at most 8 points fit (they all lie in two d × d squares, and within each square no two points from the same half are closer than d — the recursive calls guarantee that). So we check at most 7 subsequent points per strip point, giving O(n) combine cost. ∎

14. Template Philosophy TEMPLATES

All D&C templates in this tutorial form one unified family. The same naming conventions, the same invariants, and the same mental model apply everywhere.

Universal naming: l = left bound, r = right bound (exclusive), mid = (l+r)>>1
temp[] = auxiliary array, always allocated outside and passed in
All ranges are half-open [l, r). Base case: r−l ≤ 1.

The two template families

🔀 Merge-sort family

  • Split by position (mid)
  • Recurse on both halves
  • Combine by merging
  • Returns: sorted segment + extra info
  • Used for: merge sort, inversion count, offline cross-pair queries

🎯 Pivot family

  • Split by pivot value
  • Recurse on one or both halves
  • Combine is trivial or absent
  • Returns: void (sort) or value (select)
  • Used for: quicksort, quickselect, k-th element queries
⚠

Naming discipline: In the merge-sort family, always pass the temp array as a parameter — do not allocate it inside the recursive function. In the pivot family, always use a random pivot. Mixing families or deviating from these rules is the single largest source of bugs in D&C code.

void merge_sort(vector<int>& a, int l, int r, vector<int>& temp) {
    if (r - l <= 1) return;
    int mid = (l + r) >> 1;
    merge_sort(a, l, mid, temp);
    merge_sort(a, mid, r, temp);
    int i = l, j = mid, k = l;
    while (i < mid && j < r) temp[k++] = a[i] <= a[j] ? a[i++] : a[j++];
    while (i < mid) temp[k++] = a[i++];
    while (j < r)   temp[k++] = a[j++];
    for (int p = l; p < r; ++p) a[p] = temp[p];
}

long long count_inv(vector<int>& a, int l, int r, vector<int>& temp) {
    if (r - l <= 1) return 0;
    int mid = (l + r) >> 1;
    long long ans = count_inv(a, l, mid, temp) + count_inv(a, mid, r, temp);

    while (i < mid && j < r) {
        if (a[i] <= a[j]) temp[k++] = a[i++];
        else { temp[k++] = a[j++]; ans += mid - i; }
    }

    return ans;
}

void qsort3(vector<int>& a, int l, int r) {
    if (r - l <= 1) return;
    int pv = a[uniform_int_distribution<int>(l, r-1)(rng)];
    int i = l, j = l, k = r;
    while (i < k) {
        if      (a[i] < pv) swap(a[i++], a[j++]);
        else if (a[i] > pv) swap(a[i], a[--k]);
        else i++;
    }
    qsort3(a, l, j); qsort3(a, k, r);
}

Quickselect (returns k-th smallest, 0-indexed)

int qselect(vector<int>& a, int l, int r, int k) {
    while (true) {
        if (r - l == 1) return a[l];
        int pv = a[uniform_int_distribution<int>(l, r-1)(rng)];
        int i = l, j = l, hi = r;
        while (i < hi) {
            if      (a[i] < pv) swap(a[i++], a[j++]);
            else if (a[i] > pv) swap(a[i], a[--hi]);
            else i++;
        }
        // a[l..j) < pv, a[j..hi) == pv, a[hi..r) > pv
        if      (k <  j)  r = j;
        else if (k < hi)  return pv;
        else              l = hi;
    }
}

18. Common Pitfalls & Bugs PITFALLS

Pitfall 1 — Wrong base case

The base case must cover all small inputs that would cause infinite recursion or incorrect behavior. For merge sort, r − l ≤ 1 is correct. Using r == l misses the single-element case. For closest pair, len ≤ 3 is necessary because splitting a 2-element segment can give a 0-element half.

✗

if (l == r) return; — misses r−l=1, returns before solving the single element.

✓

if (r - l <= 1) return; — handles both empty and single-element correctly.

Pitfall 2 — Allocating temp inside recursion

A very common performance bug: declaring vector<int> temp(r - l) inside the recursive function. This allocates O(r−l) memory at every node in the recursion tree, totaling O(n log n) memory and significant allocation overhead.

✗

Allocate temp inside — O(n log n) memory, slow.

✓

Allocate once in main with size n, pass as parameter — O(n) memory.

Pitfall 3 — Naive quicksort on sorted input

Using first or last element as pivot on sorted input gives the degenerate T(n) = T(n−1) + O(n) = O(n²) recurrence. Contest problems are specifically designed to trigger this. Use a random pivot or 3-way partition.

Pitfall 4 — Confusing expected and worst case

"Randomized quicksort is O(n log n)" is imprecise. It is expected O(n log n). The worst case is still O(n²) — just astronomically unlikely with a good RNG. Never claim a randomized algorithm has a deterministic O guarantee unless you are using median-of-medians.

Pitfall 5 — Re-sorting in closest pair

If you sort the strip by y at every recursion level, the combine step costs O(n log n) instead of O(n). The total becomes O(n log² n). This is still a correct algorithm, but it is suboptimal. Use the merge-sort-style y-order propagation for O(n log n).

Pitfall 6 — Forgetting to sort before closest pair

The algorithm requires the input sorted by x before the first recursive call. The recursion maintains y-sorted order through the merge, but x-sorted order must be established before recursion starts. If you forget the initial sort, the strip argument breaks.

Pitfall 7 — Int overflow in inversion count

The maximum answer is n(n−1)/2. For n = 10⁵, this is about 5 × 10⁹, which overflows a 32-bit int (max ≈ 2.1 × 10⁹). Always use long long for the answer and the return type.

Pitfall 8 — Copy-back range error

In merge sort, the copy-back loop must use range [l, r), not [0, r) or [0, n). Copying the wrong range overwrites portions of the array that other recursive calls have already correctly sorted.

The recognition question

🎯

The single most useful question: "If I split the input in half, what information would I want each half to return so that the parent can finish in O(n) or O(n log n) time?" If you can answer this clearly, you have found the D&C solution. If you cannot, D&C may not be the right tool — consider DP or a different decomposition.

When NOT to use Divide & Conquer

✗

Overlapping subproblems → think DP instead: If your recursion tree has repeated subtrees (the same subproblem appears more than once), D&C without memoization is exponential. Classic tell: Fibonacci, coin change, shortest paths in a graph. Add memoization and you have DP, not D&C.

✗

Local irrevocable choice → think greedy instead: If you can show that the locally optimal choice is always globally safe, greedy is faster and simpler than D&C. Interval scheduling, Huffman coding, and Dijkstra's algorithm all fall here. D&C does more work than necessary by solving full subproblems when one choice suffices.

✗

State-space search → think backtracking instead: If the problem requires trying combinations of choices (permutations, subsets, valid assignments), backtracking with pruning is the right tool. D&C requires structuring subproblems independently, which is not possible when choices interact.

✗

Expensive combine → D&C may fail or degrade: D&C only helps if the combine step is cheaper than re-solving the original problem. If the combine costs O(n²) at every level, the total is O(n² log n) — worse than brute force for many inputs. Always verify the combine cost before committing to D&C.

Common wrong instincts — don't fall for these

  • "Recursive = D&C." No. DFS on a tree is recursive but not D&C. Fibonacci is recursive but is DP. D&C requires independent subproblems of the same type + a combine step that assembles the parent's answer.
  • "Split the array → it must be D&C." Splitting is only useful if the combine step can actually be done cheaply. If you split and then have to check all cross-pairs in O(n²), you have gained nothing over brute force.
  • "Sort both halves and concatenate." This only works if the two halves are known to be non-overlapping in value (like partition sort). In general, sorted left + sorted right is not a sorted whole — you need a real merge.
  • "Quicksort is worst-case O(n log n)." It is not. Quicksort with a fixed pivot is worst-case O(n²). Only randomized quicksort achieves expected O(n log n). Worst case is still O(n²) even with a random pivot — just with exponentially low probability.
  • "The recursion tree always has O(log n) levels." Only for balanced splits. If quicksort always picks the worst pivot (min or max), the tree has O(n) levels and complexity is O(n²).

5-second recognition rules — scan the problem statement for these signals

Sort / recurse / merge halves → merge-sort family Count pairs (i<j) with some property → inversion-count template Pick pivot, recurse on one side → quickselect family Answer = left part + right part + cross part → D&C candidate Geometry: split line + strip scan → closest pair family Recursive self-similar string/array construction → D&C construction T(n) = 2T(n/2) + O(n) derivable → likely O(n log n) D&C Same subproblem in both branches → NOT D&C, think DP One local choice, no recursion → NOT D&C, think greedy

Common confusions

⚠

D&C vs DP: If the same subproblem appears in multiple branches of the recursion tree, it is DP, not D&C. Fibonacci: fib(n−2) appears in the subtree of fib(n−1) AND in the direct call — this is DP with overlapping subproblems. Merge sort: the subproblem [l, mid) appears exactly once — this is D&C with independent subproblems.

⚠

D&C vs greedy: D&C solves all subproblems. Greedy makes one local choice at each step and never revisits. Quickselect recurses into one branch — is it greedy? No: it recurses on a subproblem of the same type, it does not make a permanent local choice. The distinction is about structure, not branching factor.

20. Variants & Extensions ADVANCED

Variant 1 — Print The Pattern (Recursive Construction)

Problem story: Build the string for level n: level 1 = "a", level n = level(n−1) + char('a'+n−1) + level(n−1). Print the result.

Why it fits D&C: The answer for level n is assembled from two copies of the answer for level n−1 with a separator. This is combine-by-concatenation.

What each call returns: The string for level k.

string build(int n) {
    if (n == 1) return "a";
    string prev = build(n - 1);
    char mid = char('a' + n - 1);
    return prev + mid + prev;
}
// Output length is 2^n - 1 (grows exponentially)

Complexity: The output length is 2ⁿ−1, so any algorithm must be at least Ω(2ⁿ). This is not about speed — it is about recognizing recursive self-similarity.

Problem link: Codeforces 420B — Online Meeting (similar recursive string construction)

Variant 2 — Masha and a Beautiful Tree (Recursive Block Swaps)

Problem story: An array of length 2ᵏ. A merge-sort-like procedure can swap the two halves of any recursive block. Find the minimum number of swaps to sort the array, or report impossible.

Why it fits D&C: The recursion tree of the procedure is the model. At each node you have exactly two choices: keep children as-is, or swap them. Swapping the two recursive blocks requires that the left block's maximum < right block's minimum (or vice versa if swapped).

What each call returns: Number of swaps needed, −1 if impossible.

int solve(vector<int>& a, int l, int r) { // [l, r)
    if (r - l == 1) return 0;
    int mid = (l + r) >> 1;
    int lc = solve(a, l, mid), rc = solve(a, mid, r);
    if (lc == -1 || rc == -1) return -1;

    // After recursion: left half sorted, right half sorted.
    // Two valid configurations: left_max < right_min (no swap)
    //                        or right_max < left_min (swap)
    int lmax = *max_element(a.begin()+l, a.begin()+mid);
    int lmin = *min_element(a.begin()+l, a.begin()+mid);
    int rmax = *max_element(a.begin()+mid, a.begin()+r);
    int rmin = *min_element(a.begin()+mid, a.begin()+r);

    if (lmax < rmin) return lc + rc;          // already correct order
    if (rmax < lmin) {                         // swap needed
        reverse(a.begin()+l, a.begin()+r);     // swap the two halves
        return lc + rc + 1;
    }
    return -1; // neither configuration works
}

Complexity: O(n log n) — T(n) = 2T(n/2) + O(n) due to min/max calls. Can be optimized to O(n) by returning (min, max) from each call instead.

Problem link: Codeforces 1464B — Masha and a Beautiful Tree

Variant 3 — Merge Sort? (Controlling the Recursion Tree)

Problem story: Build a permutation of 1..n such that a merge-sort-like procedure makes exactly k recursive calls (where a call on an already-sorted segment is a leaf). Report −1 if impossible.

Key observation: A leaf contributes 1 call. Expanding any internal node adds exactly 2 more calls (the two children) while the node itself still counts as 1. So every expansion adds 2 to the total. Starting from 1 (sorted = no recursion), every reachable k satisfies k ≡ 1 (mod 2). If k is even, output −1.

Construction: Start with sorted array. To force recursion at a segment, swap its two halves — this makes the segment unsorted while each half remains internally valid from previous construction.

vector<int> a;
int budget; // remaining calls to "spend"

void build(int l, int r) {
    if (r - l <= 1) return;
    if (budget <= 1) return; // this segment will be a leaf
    budget -= 2; // this node expands: costs 2 extra calls
    int mid = (l + r) >> 1;
    build(l, mid);
    build(mid, r);
    // Swap the two halves to make this segment unsorted
    // (forcing the procedure to recurse here)
    reverse(a.begin()+l, a.begin()+mid);
    reverse(a.begin()+mid, a.begin()+r);
    reverse(a.begin()+l, a.begin()+r);
}

// main: if k%2==0 → -1. Else set budget=k, build(0,n).
// If budget != 1 after build → -1 (k too large for n).

Core lesson: This problem is not about sorting at all — it is about controlling the structure of the recursion tree. Understanding that recursion tree is the key to solving this class of problem.

Problem link: Codeforces 1582F — Gregor and Two Painters (related recursion-tree structure)

Variant 4 — Count Pairs with a[i] > 2·a[j] (Modified Inversion Count)

Problem story: Count pairs (i, j) with i < j and a[i] > 2·a[j].

Why it fits: Same three-category argument. Cross pairs where i is in the left half and j in the right half can be counted efficiently during the merge step because both halves are sorted.

Cross-pair counting during merge: For each right-half element a[j], use a two-pointer on the sorted left half to find how many left elements exceed 2·a[j]. Since the left half is sorted, this is a monotone scan.

// During merge of sorted a[l..mid) and a[mid..r):
// For each j in [mid, r), count i in [l, mid) with a[i] > 2*a[j].
// Use pointer ptr starting at mid, moving left.
long long cross = 0;
int ptr = mid;
for (int j = mid; j < r; j++) {
    while (ptr > l && a[ptr-1] > 2LL * a[j]) ptr--;
    cross += (mid - ptr);
}
ans += cross;

Complexity: Same as inversion count — O(n log n).

Problem link: VJudge POJ-2299 — Ultra-QuickSort (inversion count)

21. Problem-Based Applications PRACTICE

Each entry is a compact mini-lesson: problem story → abstraction → why D&C fits → core idea → complexity → link. Focus on the "core idea" — that is the single piece of thinking that unlocks each problem.

Ultra-QuickSort — Inversion Count

Problem story: Given a permutation, find the minimum number of adjacent swaps (bubble sort swaps) needed to sort it.

Abstraction: Count pairs (i, j) where i < j and a[i] > a[j]. Each such pair requires exactly one adjacent swap.

Why D&C fits: Every inversion is either both indices in the left half, both in the right half, or one in each. The first two are recursive. The cross-inversions are counted during the merge step because both halves are sorted.

Core idea: During merge, when a[j] from the right half is taken (because a[i] > a[j]), every remaining left element also exceeds a[j] — so add mid − i inversions at once.

Complexity: O(n log n) — same recurrence as merge sort.

Link: VJudge POJ-2299

Closer — Closest Pair of Points

Problem story: Given n points in the plane, find the pair with the smallest Euclidean distance.

Abstraction: Minimum distance over all O(n²) pairs, solved in O(n log n).

Why D&C fits: Split points by x-coordinate. The closest pair is either entirely in the left half, entirely in the right half, or one point in each. The cross-pair case is handled by a strip scan near the dividing line.

Core idea: After recursion gives d = min(d_L, d_R), only points within distance d of the split line can improve the answer. Sort the strip by y and check at most 7 neighbors per point — the packing argument guarantees O(n) combine.

Complexity: O(n log n) with merge-sort-style y-order propagation.

Link: VJudge SPOJ-CLOPPAIR

Masha and a Beautiful Tree — Recursive Block Swaps

Problem story: Given an array of length 2ᵏ, a merge-sort-like procedure can swap the two halves of any recursive block. Find the minimum swaps to sort it, or report impossible.

Abstraction: At each recursion node, decide: keep or swap the two children. The answer is the total number of swap decisions.

Why D&C fits: The recursion tree of the procedure is the model. Each node has exactly two valid configurations: left_max < right_min (no swap) or right_max < left_min (swap). If neither holds, it is impossible.

Core idea: Recurse on both halves. After they are sorted, check the two configurations. If a swap is needed, reverse the block halves and add 1 to the count.

Complexity: O(n log n) — T(n) = 2T(n/2) + O(n).

Link: CF 1464B

Towers — K-th Order Statistic Queries

Problem story: Answer multiple queries asking for the k-th smallest element within a given subarray range.

Abstraction: Offline or persistent range order-statistic queries.

Why D&C fits: A merge sort tree stores the sorted state of each recursion level. For each query range, binary search across tree levels to find the k-th element. Alternatively, offline D&C on queries processes them by splitting on the array midpoint.

Core idea: Build a persistent sorted structure at each merge sort level, then binary-search the answer for each query across those levels.

Complexity: O(n log² n) for merge sort tree approach.

Link: SPOJ CTOWER

The Inversions Problem — Rotation Inversion Count

Problem story: Given a permutation of 0..n−1, compute the inversion count for every rotation (cyclically shift the first element to the end, repeat).

Abstraction: Compute the base inversion count once, then derive each rotation's count by a formula relating to how many elements the shifted element is greater/less than.

Why D&C fits: The base inversion count is computed with the standard merge-sort-based inversion count. Each subsequent rotation adjusts the count in O(1).

Core idea: Merge while counting cross-inversions for the base case. Then use the identity: shifting element a[0] to the end removes (a[0]) inversions and adds (n − 1 − a[0]) inversions.

Complexity: O(n log n) for the initial count, O(n) for all rotations.

Link: VJudge HDU-1394

K-th Largest in Unsorted Array — Quickselect

Problem story: Given n unsorted numbers, find the k-th smallest without fully sorting the array.

Abstraction: Order-statistic selection in expected O(n) time.

Why D&C fits: After partitioning around a random pivot, the pivot lands at some position p. If k = p, you found it. If k < p, recurse left. If k > p, recurse right. Only one branch is explored — half the data is discarded each time.

Core idea: Partition + recurse into one side only. The geometric shrinkage n + n/2 + n/4 + … converges to O(2n) = O(n).

Complexity: O(n) expected with random pivot; O(n) worst-case with median-of-medians.

Link: CF 1367E

Checklist: before submitting any D&C solution

  • Base case handles empty segment (r − l = 0) and single element (r − l = 1)
  • Mid computation uses (l + r) >> 1 — never (l + r) / 2 with signed integers if overflow possible
  • Temp array allocated once outside, passed as parameter
  • Copy-back range is exactly [l, r), not [0, r) or [0, n)
  • Inversion count uses long long
  • Quicksort / quickselect uses random pivot
  • Closest pair uses long long for squared distances

22. Final Summary SUMMARY

The entire paradigm reduces to seven questions. If you can answer them all clearly for a given problem, you have a correct D&C solution.

The Seven Questions
①
Can I split?
Can the input be divided into smaller subproblems of the same kind?
②
What does it return?
What exact information does each recursive call give back to its parent?
③
Three categories?
What are the exhaustive categories of the final answer? Which are recursive, which are cross?
④
Cheap combine?
Can I handle the cross category in O(n) or O(n log n) time?
⑤
Hidden log?
Am I accidentally adding an extra log by re-sorting or re-allocating inside the recursion?
⑥
What recurrence?
What recurrence does the finished code produce? Does the Master Method confirm the complexity?
⑦
Really D&C?
Is this actually D&C, or is it DP (overlapping subproblems), greedy (one local choice), or backtracking (search with pruning)?
Complexity quick reference
T(n/2)+O(1) → O(log n)
2T(n/2)+O(1) → O(n)
2T(n/2)+O(n) → O(n log n) ← most common
T(n/2)+O(n) → O(n) ← quickselect
T(n−1)+O(n) → O(n²) ← bad pivot

The learning order

  1. Understand the paradigm: what D&C is and is not
  2. Learn to derive recurrences from code
  3. Learn the recursion-tree method for complexity
  4. Learn the Master Method from the recurrence-tree picture, not by memorizing cases
  5. Implement merge sort — understand every line
  6. Extend merge sort to inversion counting — understand the cross-pair argument
  7. Implement quicksort — understand pivot strategies and their complexity consequences
  8. Implement quickselect — understand why recursing into one side gives O(n)
  9. Implement closest pair — understand the strip argument and the merge-sort y-order trick
  10. Practice recursive construction problems (patterns, block swaps, recursion tree control)
