1. Problem Definition FOUNDATION

Backtracking solves problems where the answer can be seen as a sequence of decisions. At every step you make one choice, then another, then another, until you either build a complete solution or prove the partial solution cannot ever work. That is the entire technique.

The kinds of decisions that arise include: choosing which items to include, arranging items in an order, assigning values to variables, building a path through a grid or graph, filling cells of a grid, or constructing numbers digit by digit.

answer = sequence of decisions d₁, d₂, …, dₖ that satisfies all constraints
📌

Why "sequence"? Because recursion processes decisions one at a time. Even though the final answer may be a set (like a subset), we generate it by deciding one element at a time. The tree structure of recursion mirrors exactly the tree of possible decision sequences.

💡

Core insight: Every backtracking problem is really about exploring a tree. Each internal node is a partial decision. Each leaf is a complete (or provably dead) assignment. The technique is about how intelligently we explore that tree.

Your Running Example CONCRETE

Before any theory, let's ground everything in one tiny example that we will reuse throughout the early sections.

💡

The problem

You have four items with weights [3, 5, 6, 7]. You want to find a subset whose total weight is exactly 10.

Possible subsets that work: {3, 7} (sum = 10). Also possibly others — let's find out by trying everything.

The decision tree

We process items left to right. For each item, we have exactly two choices: take it or skip it. This creates a binary tree with 24 = 16 leaves — one for every possible subset.

Here are the first few levels of the tree, showing what happens at each node:

This picture is the entire idea of complete search. We explore every path in this tree. Each path is one subset. Each leaf is one possible answer. The tree has 24 = 16 leaves because there are 4 items with 2 choices each.

📌

Key terms you just learned (without being told): The tree is the search tree. Each node is a state (the current index and running sum). Each arrow is a choice (take or skip). Reaching a leaf where sum = target is a valid solution. Reaching a leaf where sum ≠ target is a failed branch. This is all backtracking is.

We will keep coming back to this example. Whenever a new concept is introduced, ask yourself: "How would this look on the [3, 5, 6, 7] tree?"

Story: Given a set of items with positive weights and an interval [Cmin, Cmax], decide whether any subset's total weight falls within the interval.

Abstraction: Subset-sum decision search with an interval target.

Goal: Find any single subset with total weight in [Cmin, Cmax], or report none exists.

2. Intuition from Scratch — Complete Search FOUNDATION

You just saw the [3, 5, 6, 7] tree. That tree — trying every path without any shortcuts — is exactly what we call complete search. Let's now understand the idea more carefully.

Before backtracking, there is complete search. Complete search means: try every candidate. No pruning. No heuristics. No cleverness yet. This is the correct first step because backtracking is not a different universe — it is simply complete search plus early rejection.

If you do not understand the plain search tree first, the advanced pruning tricks will feel random.

Why complete search is not "bad"

Many students think brute force is weak, naive, or always too slow. That is false. Complete search is often the best contest solution when the search space is small enough, the implementation is short, the debugging is easy, and the constraints allow it.

The right first question is: "Can I search everything in time?"
Not: "Can I show off a harder algorithm?"
⚠

Contest wisdom: A correct simple solution beats a fancy broken one every time. Many problems have constraints specifically designed to allow complete search.

⚠ Common Mistakes

  • Jumping to DP or greedy without first checking if brute force fits the constraints
  • Calling complete search "wrong" or "lazy" — it is a legitimate first-class technique

Mini-checkpoint — you should now understand:

✓ Complete search = try every candidate, no shortcuts

✓ It is the correct starting point — we add intelligence later

✓ For our [3,5,6,7] example, complete search visits all 2⁴ = 16 subsets

3. Visual Mental Model — Recursion = Nested Loops FOUNDATION

We now know what complete search is. But how do we actually code it? The answer is recursion — and it is simpler than it looks if you think of it as automated nested loops.

Students often understand loops more easily than recursion. Consider a triple loop:

for (int i = 0; i < A; i++)
    for (int j = 0; j < B; j++)
        for (int k = 0; k < C; k++)
            work();

This means: choose i, then choose j, then choose k. A recursive complete-search function is the same idea — but the number and kind of loops depend on the state. So recursion is not magic. It is the natural way to generate a decision tree when the depth or branching is not fixed at compile time.

Back to our running example: with arr = [3, 5, 6, 7], the recursive function dfs(i, sumNow) at level 0 asks "take or skip 3?", at level 1 asks "take or skip 5?", and so on. Four levels of recursion generate 2⁴ = 16 leaves — one for each subset.

📌

Mental model: Every call to dfs(state) is one "level" of the tree. The for loop inside generates the children at that level. Returning from dfs is moving back up one level. The call stack is the current path from root to the node you are exploring.

  • Thinking recursion is fundamentally different from loops — it is just loops where the nesting depth is determined at runtime
  • Forgetting that returning from a recursive call is the same as going back up the tree — this is where undo happens

Rule of thumb: ~107–108 operations per second. If search space ≤ 107, pure search is comfortable. 220 ≈ 106 is trivial. 230 ≈ 109 needs pruning.

⚠ Common Mistakes

  • Starting to code without counting the search space first — you might waste an hour on a solution that cannot possibly run in time
  • Confusing 2n (manageable for n≤20) with n! (unmanageable beyond n≈12) — know which pattern applies

5. The Three Types of Search Problems CLASSIFICATION

Every complete-search problem should be classified as one of these three types before writing code, because the type determines how your search function behaves at the leaves.

Decision
Does a valid solution exist?

You only need one success. Return type is bool.

Examples: Subset-Range, graph colorability, Sudoku

Enumeration
List / count all valid solutions

You must explore all branches. Return type is void with a counter.

Examples: all subsets, Robots on Ice (counting paths)

Optimization
Find the best valid solution

Must prove no better answer exists. Maintain global best.

Examples: Pyramids (min count), Fence Rails (max rails)

🎯

Why this matters: Decision problems can short-circuit with return true. Optimization can use branch-and-bound. Enumeration gets neither shortcut.

Same tree, three different behaviors — using our running example

Consider our [3, 5, 6, 7] example with target sum 10. The same recursion tree looks different depending on what we want:

TypeQuestionWhat happens at {3,7}=10?After finding {3,7}?Extra machinery
DecisionDoes any subset sum to 10?Return true immediately!Stop. Done. Answer: yes.return true propagation
EnumerationList ALL subsets that sum to 10Print {3,7}, then keep goingExplore every remaining branchCounter or output list
OptimizationFewest items summing to 10?Record {3,7} (size=2) as current bestKeep searching — maybe a single item sums to 10?Global best + bound pruning

Notice: the search tree is identical in all three cases. What changes is the stopping rule and the extra bookkeeping at the leaves. Classifying your problem before coding tells you exactly which stopping rule to use.

⚠ Common Mistakes

  • Using return true short-circuit in an enumeration problem — you stop after the first answer and miss all others
  • Not maintaining a global best in optimization — you print the first valid answer instead of the best one

Mini-checkpoint — you should now understand:

✓ Decision = stop at first valid answer

✓ Enumeration = visit ALL leaves

✓ Optimization = track best, keep searching, use bounds to prune

6. The Pure Complete Search Template TEMPLATE

We have seen the tree, counted its size, and classified the problem type. Now let's write the general code skeleton. First, remember the concrete version from our running example:

// Concrete: subset-sum for [3,5,6,7]
void dfs(int i, int sumNow) {
    if (i == 4) {                    // all decisions made
        if (sumNow == 10) cout << "found!\n";
        return;
    }
    dfs(i + 1, sumNow + arr[i]);     // take arr[i]
    dfs(i + 1, sumNow);              // skip arr[i]
}

This specific code is just one instance of a general pattern. Here is the abstract version — the same idea, generalized:

// Generic: the universal complete search skeleton
void dfs(State& st) {
    if (complete(st)) {        // all decisions made?
        process_answer(st);    // record / print / update best
        return;
    }
    for (auto choice : all_choices(st)) {
        apply(st, choice);     // make the decision
        dfs(st);               // recurse with new state
        undo(st, choice);      // restore old state exactly
    }
}
📌

The apply/undo pattern: Every change you make before recursing must be undone after recursing. Not "close." Exactly. In our running example, if you add arr[i] to sumNow, you must subtract it back before the skip branch.

⚠

Important: This is a thinking template, not a paste-and-go snippet. You adapt it to each problem — the state, choices, and undo logic change every time. The template tells you the shape of the recursion; you fill in the content.

Line-by-line: what each part does and what breaks if removed

LinePurposeWhat breaks if missing
if (complete(st))Detect when all decisions are madeRecursion never stops → stack overflow
process_answer(st)Record / print / update global bestValid answers are found but ignored
for (auto choice : ...)Generate all branches at this levelNo exploration happens — returns immediately
apply(st, choice)Make the decision — change stateRecursive call sees the old state, not the new one
dfs(st)Explore the subtree created by this choiceOnly depth-1 exploration, misses deeper solutions
undo(st, choice)Restore state for the next sibling branchTHE #1 BUG: corrupted state → all subsequent branches are wrong

⚠ Common Mistakes

  • Forgetting the undo step — the state leaks from one branch into the next, producing wrong answers that are very hard to debug
  • Placing the base case after the choice loop instead of before — you try to generate choices for a complete state

7. Key Observations — When Pure Search Suffices INSIGHT

Before we add any pruning intelligence, there is one more crucial insight: sometimes you don't need pruning at all. Some problems look huge until you model them correctly.

Before pruning, first reduce the search space mathematically.
Sometimes the correct complete search is already tiny.

Party Lamps (see catalog #2)

Naively: four button choices repeated many times. Correctly modeled: each button matters only modulo 2. Total cases: 24 = 16. The partial solution at step k = parities of buttons 1..k; lamps affected by those buttons are already determined.

The Clocks (see catalog #3)

Naively: sequences of arbitrary moves. Correctly modeled: each of 9 moves used 0–3 times (mod 4). Total: 49 = 262,144. At depth m, moves 0..m−1 have fixed counts; the current clock configuration reflects all choices so far.

💡

Lesson: The first optimization is not code-level pruning — it is mathematical reduction. A different model can collapse exponential into constant.

8. The Turning Point — From Search to Backtracking PRUNING

We now know how to generate every possibility. But what if the search space is too large for brute force? This is where the real technique begins.

Now we move from "try every branch" to "try only branches that still have a chance."

bool dfs(State& st) {
    if (prune(st)) return false;      // kill dead branches
    if (complete(st)) { process(st); return true; }
    for (auto choice : ordered_choices(st)) {
        if (!legal(st, choice)) continue;
        apply(st, choice);
        if (dfs(st)) { undo(st, choice); return true; }
        undo(st, choice);
    }
    return false;
}

Three words changed: prune, ordered, legal. Those three words are where AC is won.

Concrete: adding pruning to our [3, 5, 6, 7] example

In the pure search, we visit all 16 leaves. Now add one prune: if (sumNow > 10) return false;

When we take 3, take 5, take 6 → sumNow = 14 > 10. We immediately return. We never visit the subtree below this node (which would have tried taking or skipping 7). That subtree had 2 leaves — both saved. Similarly, take 3, take 5, skip 6, take 7 → sumNow = 15 > 10 → pruned. The prune kills roughly 7 leaves out of 16.

Layer 1 — Complete Search
Try every branch
No intelligence. Explore the entire tree. Guaranteed correct but potentially slow.
↓ add pruning, ordering, legality checks
Layer 2 — Backtracking
Try only viable branches
Same recursion, same undo, but we prove dead branches can't contain useful answers.
↓ add optimization bounds
Layer 3 — Branch & Bound
Prune with optimality bounds
For optimization: maintain current best, compute lower bound, prune subtrees that can't beat best.

⚠ Common Mistakes

  • Adding a prune that is too aggressive — it incorrectly kills branches that contain valid solutions → Wrong Answer
  • Adding a prune without a mathematical proof — "it feels like this branch won't work" is not a valid reason to prune

9. The Four Invariants of Backtracking INVARIANTS

Why this section exists: We have now added pruning and ordering to our search. But how do we know the result is still correct? These four invariants are the mathematical backbone that guarantees correctness. If any one is violated, the algorithm silently produces wrong answers.

Think of them as a checklist: if you can verify all four hold in your code, your backtracking is correct.

1
State Invariant
The current state is consistent. E.g., sumNow equals the sum of chosen elements.
In plain language: "the numbers I'm tracking are correct right now." In our example, sumNow truly equals the total of items we've taken so far.
2
Transition Invariant
Applying a legal choice produces another valid partial state.
In our example: taking arr[i] when sumNow + arr[i] is non-negative and i < n produces a valid new state (index i+1, updated sum). We never create an impossible intermediate state.
3
Undo Invariant
After undo, the old state is restored exactly. Every modified variable is restored.
In our example: if we added arr[i] to sumNow, we subtract exactly arr[i] before trying skip. The program looks as if the take never happened.
4
Pruning Invariant
When you prune, you are proving: no useful solution exists in this subtree.
In our example: if sumNow is already 14 and we only add positive numbers, it can never come back down to 10. That's a proof, not a guess.
⚠

Critical: A wrong prune gives Wrong Answer. Every prune must have a mathematical argument.

Detailed correctness: feasibility pruning click to expand

Subset-range with positive weights sorted descending, suffix sums precomputed:

1. Invariant: At recursive depth d, sumNow equals the exact total of the items chosen so far (indices 0..d−1 that were taken), and suf[d] equals the total of all remaining items (indices d..n−1). Every item has positive weight.

2. Holds initially: At the root call dfs(0), no items have been chosen, so sumNow = 0 (set in main). suf[0] = sum of all items (built by the suffix loop). Both are trivially correct.

3. Preserved across steps: When we take item d, we execute sumNow += w[d] before recursing into dfs(d+1). Now sumNow includes w[d] and suf[d+1] excludes it — both remain correct. When we undo (sumNow -= w[d]), the original value is restored exactly. The skip branch does not modify sumNow, so it remains correct at the pre-take value. Therefore the invariant holds at every node.

4. Leads to correctness:

Prune 1 (sumNow > Cmax): By the invariant, sumNow is exact. All remaining weights are positive, so any descendant has sum ≥ sumNow > Cmax. No valid solution exists below → prune is safe.

Prune 2 (sumNow + suf[i] < Cmin): By the invariant, sumNow + suf[i] is the maximum achievable sum (taking every remaining item). If this maximum is below Cmin, no descendant can reach the target range → prune is safe.

Early accept (Cmin ≤ sumNow ≤ Cmax): By the invariant, sumNow is the exact total of the chosen subset. It lies within the target range. A valid solution exists → return true is correct.

Detailed correctness: branch-and-bound (Pyramids) click to expand

Choose distinct pyramids (cubes 1³, 2³, 3³, …) summing to c, minimising the count of pyramids used.

1. Invariant: At each recursive node, rem = exact remaining cubes needed (c minus cubes already chosen), curSz = count of pyramids chosen so far, mx = largest pyramid still available (candidates are tried in descending order), and bestSz = size of the best complete solution found so far.

2. Holds initially: At the root call, rem = c (nothing chosen yet), curSz = 0, mx = largest pyramid ≤ c, and bestSz = ∞. All are trivially correct.

3. Preserved across steps: When we choose pyramid k3, we set rem -= k³ and curSz++. On undo, we restore rem += k³ and curSz--. The next candidate is smaller (descending loop), so mx decreases monotonically. bestSz is updated only when a complete valid solution is found (rem reaches 0), so it only ever decreases. All quantities remain correct.

4. Leads to correctness: The prune condition is curSz + ⌈rem/mx⌉ ≥ bestSz. By the invariant, rem is exact and mx is the largest remaining pyramid. Even using pyramids of size mx exclusively (the most optimistic packing), we need at least ⌈rem/mx⌉ more. Adding these to curSz gives a lower bound on the final count. If this lower bound already meets or exceeds bestSz, no descendant can improve the best — the prune is safe. The lower bound is optimistic (it ignores the distinctness constraint), so the true count can only be higher, never lower.

Detailed correctness: distance lower bounds (Robots on Ice) click to expand

Setting: Hamiltonian path on an R×C grid. Checkpoints require visiting cell (rk, ck) at exactly step tk.

1. Invariant: At each recursive call, cur = exact number of cells visited so far (equivalently, the current step number). The current cell is (i, j). For every future checkpoint k, the Manhattan distance |i − r_k| + |j − c_k| is a hard lower bound on the minimum number of steps needed to reach checkpoint k from the current position, because each step moves exactly one cell in one direction.

2. Holds initially: At the root call, cur = 1 (cell (0,0) is visited) and (i,j) = (0,0). The Manhattan distance to each checkpoint is computed from the known starting position — trivially correct.

3. Preserved across steps: When we move from (i,j) to an adjacent cell (ni,nj), cur increments by 1. The Manhattan distance to any checkpoint changes by at most 1 (triangle inequality on the grid). The lower-bound property is intrinsic to the grid metric — it does not depend on which cells have been visited, only on the current position. Undo restores (i,j) and decrements cur, re-establishing the previous state exactly.

4. Leads to correctness: If manhattan(cur_pos, checkpoint_k) > t_k − cur, then even moving directly toward checkpoint k on every step cannot arrive by step tk. Since checkpoint timing is a hard constraint, every continuation of the current path violates it. No valid Hamiltonian path exists in this subtree → prune is safe. Conversely, if the distance check passes, the prune does not fire — it never kills a branch that might still be feasible.

Additional check (from code): if (i == ti[k] && j == tj[k]) — being at a future checkpoint's cell at the wrong step is also provably fatal: leaving and returning costs at least 2 extra steps (out and back), which may violate later checkpoints. The code prunes this case directly.

Detailed correctness: structural / forced-move pruning (Robots on Ice) click to expand

Setting: Hamiltonian path on a grid. Every cell must be visited exactly once.

1. Invariant: For every unvisited cell (i,j), the value unVisCnt[i][j] equals the exact number of unvisited neighbours of (i,j). A non-exit cell with unVisCnt == 1 has only one remaining path of entry — if the path does not use it immediately, the cell becomes permanently isolated.

2. Holds initially: Before the search begins, all interior cells are unvisited. unVisCnt[i][j] is initialised by counting unvisited neighbours (the boundary cells are pre-marked as visited). This loop runs in main() and produces correct counts.

3. Preserved across steps: The makeVis(i,j,1) call marks (i,j) as visited and decrements unVisCnt for all four neighbours of (i,j). The makeVis(i,j,0) undo call clears the visited flag and increments the neighbour counts back. These are exact inverses — the count is correct at every node. No approximation or deferred update is used.

4. Leads to correctness: When cnt == 1 (exactly one unvisited neighbour has unVisCnt == 1), that neighbour has one remaining access point — the current cell. If we do not go there next, its last access path gets used for something else or is blocked, and the cell becomes unreachable. A Hamiltonian path must visit every cell, so isolation is fatal → the forced move is the only viable choice, and going there is mandatory. When cnt ≥ 2, two or more neighbours are simultaneously forced, but we can only move to one cell per step — at least one forced cell will become isolated no matter which we choose → no Hamiltonian path exists → prune is safe.

Exit cell exemption (from code): The condition (ni != 1 || nj != 2) exempts the exit cell from forced-move detection. The exit cell is the last cell visited; having exactly one unvisited neighbour pointing to it at the end is the expected completion condition, not a forced-move crisis. Including it in forced detection would cause premature redirection toward the exit.

Detailed correctness: arithmetic pruning (Vampire Numbers) click to expand

Setting: Build two factors a and b digit by digit from LSD (least significant digit) upward so that a × b = target, using exactly the target's digits.

1. Invariant: After assigning the last k digits of both factors, p = 10k, and the partial values of a and b represent exactly those assigned low digits. The digit frequency array freq[0..9] contains exactly the digits not yet assigned. The quantity (a × b) mod p depends only on the last k digits of a and b — higher digits multiply by factors of p and vanish mod p.

2. Holds initially: Before the first recursive call, a = 0, b = 0, p = 1, and freq[] contains all digits of the target. Zero digits have been assigned, so the invariant holds vacuously: (0 × 0) mod 1 = 0 = target mod 1.

3. Preserved across steps: When we assign digit da to position k of a, we execute a += p × d_a and freq[d_a]--. Similarly for b. Then p *= 10. The new partial values of a and b now represent their last k+1 digits, and the new p = 10k+1. On undo: p /= 10, b -= p × d_b, freq[d_b]++, then a -= p × d_a, freq[d_a]++. These are exact inverses — every quantity is restored.

4. Leads to correctness: The prune checks (a × b) % p ≠ target % p. By the invariant, the left side depends only on the assigned low digits and cannot be changed by any assignment of higher digits (formally: let a = alo + ahi·p and b = blo + bhi·p; then a·b mod p = alo·blo mod p, with all cross-terms and high terms vanishing mod p). If this does not equal target mod p, no complete assignment of digits can make a × b = target. Every valid factorisation requires matching suffixes at every digit depth → the prune is safe.

Leading-zero flags (az, bz): The loop starts from i = az (and j = bz), where the flag is 1 if the factor so far is all zeros. This prevents leading zeros in the final factor, which would make it not a true n-digit number. This is a validity constraint derived from the code, not a heuristic.

Detailed correctness: constraint propagation / domain emptiness (Prime3) click to expand

Setting: Fill a 5×5 grid so all 12 lines (5 rows, 5 columns, 2 diagonals) are 5-digit primes with a given digit sum.

1. Invariant (domain consistency): At each recursive depth, for every unassigned line L, the candidate set adj[read(L)] contains exactly the 5-digit primes that are consistent with all currently-fixed cells along L. This is guaranteed by the base-11 encoding: read(L) produces a key where filled cells contribute their digit and unfilled cells contribute 10 (wildcard). The adj table was precomputed to map every such key to the correct candidate list.

2. Holds initially: Before the search, grid[0][0] is set to the given first digit and all other cells are 10 (wildcard). The adj table was built in main() by iterating over all 5-digit primes with the correct digit sum and inserting each into every wildcard-combination key it matches. Therefore adj[read(L)] is correct for every line at the start.

3. Preserved across steps: When swim() assigns a prime to line L, it calls write(L, prime, 10) which fills the 5 cells of L with the prime's digits. This changes the read() keys of all crossing lines — their candidate lists are implicitly narrowed because adj[new_key] returns only primes matching the now-fixed intersection cells. On undo, write(L, encoded, 11) restores each cell to its pre-assignment value (either the previous digit or 10/wildcard). The adj table itself is never modified — it is a static lookup. So read() keys revert, and all domain sizes return to their pre-assignment state. The invariant is preserved exactly.

4. Leads to correctness: If after assigning line L, some crossing line M has adj[read(M)].size() == 0, then no 5-digit prime with the correct digit sum matches the cells currently fixed along M. By the invariant, this means no matter how the remaining lines are assigned, line M cannot be filled with a valid prime. The grid is therefore unsolvable in this subtree → backtracking is safe. No valid complete grid is lost because the empty domain is a hard proof of infeasibility, not a heuristic. Conversely, if all domains are non-empty, the search continues — no viable branch is killed prematurely.

MRV ordering (from code): swim() picks the unassigned line with the smallest candidate list (Minimum Remaining Values). This is a pure ordering choice — it does not affect correctness because the set of valid complete grids is independent of assignment order. But it minimises the branching factor at each level, dramatically reducing nodes explored.

Space complexity

Backtracking uses the call stack as implicit storage. The maximum stack depth equals the maximum depth of the search tree. For subset problems this is O(n) — one level per item. For permutation problems it is also O(n). For grid-path problems like Robots on Ice it is O(R·C) — one level per cell. The adj[] table in Prime3 is the main outlier: it precomputes all 115 ≈ 161K entries mapping partial line patterns to candidate primes, requiring O(115 · average list size) space — several megabytes, but well within contest limits.

Template 2 — Subset Sum Decision (with Feasibility Pruning)

#include <bits/stdc++.h>
using namespace std;
const int N = 155;
int Cminn, Cmaxx, n, w[N], sumNow;
long long suf[N];
bool dfs(int i) {
    if (sumNow > Cmaxx) return false;          /* [PRUNE] prune(st): upper bound */
    if (sumNow + suf[i] < Cminn) return false;  /* [PRUNE] prune(st): lower bound */
    if (Cminn <= sumNow && sumNow <= Cmaxx) return true; /* [GOAL] goal(st)+process */
    if (i == n) return false;                    /* [GOAL] base case: exhausted items */
    /* [CHOICES] branch 1: take w[i] */
    sumNow += w[i];                              /* [APPLY] apply(st, take) */
    if (dfs(i+1)) { sumNow -= w[i]; return true; } /* recurse + [UNDO] on success */
    sumNow -= w[i];                              /* [UNDO] undo(st, take) */
    /* [CHOICES] branch 2: skip w[i] */
    if (dfs(i+1)) return true;                   /* recurse (no apply/undo for skip) */
    return false;
}
int main() {
    int S; scanf("%d",&S);
    while (S--) {
        scanf("%d%d%d",&Cminn,&Cmaxx,&n);
        for (int i=0;i<n;i++) scanf("%d",&w[i]);
        sort(w,w+n,greater<int>());
        suf[n]=0; for(int i=n-1;i>=0;i--) suf[i]=suf[i+1]+w[i];
        sumNow=0;
        puts(dfs(0)?"Sallow swallow swallows.":"Sallow swallow wallows in dust.");
    }
}

15a. Feasibility Pruning PRUNING

⚠

Naming note — Template 2 (§13) vs. Full Solution (§21): Template 2 uses pedagogical names (Cminn, Cmaxx, w[], sumNow, suf[], parameter i). The full AC-ready Subset-Range solution in §21 uses contest-shorthand names (a, b, arr[], sum, cum[], global ind). Both implement the same algorithm. The mapping is: Cminn↔a, Cmaxx↔b, w↔arr, sumNow↔sum, suf↔cum, i↔ind. The §21 version modifies ind as a global (increment/decrement) rather than passing i+1 as a parameter — same effect, different convention.

Why this exists: This is the most common and most intuitive pruning family. You already used it in our running example when you checked sumNow > target. Here we generalize the idea into a universal form:

If the current state + all remaining freedom cannot satisfy constraints → prune

The Subset-Range problem (catalog #1) demonstrates both directions: upper bound (sumNow > Cmax) and lower bound (sumNow + suf[i] < Cmin).

💡

Key pattern: Compute a bound on what is still achievable. Compare against the required range. Upper bounds prove "already too much." Lower bounds prove "can never reach enough."

15b. Branch & Bound PRUNING

Why this exists: Feasibility pruning kills branches that cannot be valid. But for optimization problems, we can go further: kill branches that cannot be better than what we already have. This is the idea behind branch and bound.

In our [3,5,6,7] optimization variant (fewest items summing to 10), if we already found {3,7} with 2 items, any branch needing 3+ items is pruned even if it could sum to 10.

if (curCost + optimistic_lower_bound ≥ bestCost) prune

The Pyramids problem (catalog #4) illustrates this: curSz + ⌈rem/mx⌉ ≥ bestSz → prune.

⚠

Ordering is critical: Try larger pyramids first so small-cardinality solutions are found early → tight bounds.

15c. Distance Lower Bounds PRUNING

Why this exists: Some problems involve spatial structure (grids, graphs). If you cannot physically reach a required location in the remaining steps, the branch is dead. This is a geometric version of feasibility pruning.

Central in Robots on Ice (catalog #5). A grid path can't reach a cell in fewer moves than its Manhattan distance.

if (manhattan_dist(current, checkpoint) > steps_remaining) → prune
📌

Not branch & bound. This is feasibility pruning from geometry. It applies to all three problem types, not just optimization.

15d. Structural / Forced-Move Pruning PRUNING

Why this exists: This may feel complex, but it is just an extension of what we already built. Instead of checking numbers (sums, distances), we check the topology of the remaining search space. If the structure itself proves impossibility, we prune.

Also from Robots on Ice (catalog #9). If an unvisited cell has exactly one remaining way to be entered later, it's a forced move. If two incompatible forced moves exist → prune.

🎯

Why deep: Structural pruning reasons about topology, not numbers. It detects dead ends from connectivity, appearing in Hamiltonian paths, puzzle solvers, and CSPs.

15e. Arithmetic Pruning PRUNING

Why this exists: When building numbers digit by digit, we can check partial results at each step. This is yet another flavor of the same idea: if the partial result already contradicts the target, stop early.

From Vampire Numbers (catalog #6). Build factors digit by digit from LSD. After fixing k digits:

if ((a × b) mod 10k ≠ target mod 10k) → prune
💡

General principle: When building numbers digit by digit, check partial product/sum mod 10k at each step. Modular arithmetic is a one-way constraint.

15f. Constraint Propagation PRUNING

Why this exists: In problems with many interacting constraints (like filling a grid where rows, columns, and diagonals all matter), assigning one variable can immediately eliminate options for others. This is the most sophisticated pruning family — essentially turning backtracking into a mini constraint-satisfaction solver.

The essence of Prime3 (catalog #7). Treat 12 lines as variables, each assigned a 5-digit prime. Lines overlap at shared cells. Assigning one constrains all crossing lines.

CSP backtracking = assign variable + propagate constraints + check domains + backtrack if empty domain
📌

MRV heuristic: Choose the remaining variable with the smallest domain. This minimises branching factor at each level.

Subset-Range / "Sallow Swallow…" FULL SOLUTION Decision Subset-Sum

Story: Given positive-weight items and interval [Cmin, Cmax], decide if any subset's total weight falls within the interval.

Abstraction: Subset-sum decision search. Two choices per item (take/skip). Positive weights.

Why this modeling is correct: Every subset corresponds to a unique path in the binary decision tree. The take/skip framework ensures every subset is generated exactly once.

📌

State: (index i, sumNow). At depth d, items [0..d−1] have been decided. sumNow = total of chosen items. Items [d..n−1] remain undecided. The recursive function tries to complete the sum into [Cmin, Cmax].

Branching: Two branches per item — take (+weight) or skip (unchanged).

Partial solution at depth d: The current state already fixed which of the first d items are in the subset. sumNow is their total. When you backtrack from "take", you undo sumNow -= w[d].

Pruning: (1) sumNow > Cmax → dead. (2) sumNow + suffix[i] < Cmin → dead. (3) Cmin ≤ sumNow ≤ Cmax → accept.

Ordering: Sort weights descending → triggers upper-bound prune earlier.

Common dead branch: Taking a large item pushes sum past Cmax immediately → pruned. Skipping everything remaining leaves sum below Cmin → pruned.

Complexity: O(2n) worst case, dramatically pruned on structured inputs.

Why in this tutorial: The textbook introduction to feasibility pruning — both directions, with suffix sums.

Link: VJudge — UVALive 5121

Full C++ Code Subset-Range 
#include <bits/stdc++.h>
#define ll long long
using namespace std;

const int N = 151;
int a, b, n;
int arr[N], sum, ind;
ll cum[N];

bool solve() {
    if (sum > b || sum + cum[ind] < a) return 0;  // feasibility prune
    if (sum >= a && sum <= b) return 1;            // accept
    if (ind == n) return 0;

    sum += arr[ind++];          // take
    bool ret = solve();
    sum -= arr[--ind];          // undo take
    if (ret) return 1;

    ind++;                      // skip
    ret = solve();
    ind--;                      // undo skip
    return ret;
}

int main() {
    int t;
    scanf("%d", &t);
    while (t--) {
        scanf("%d%d%d", &a, &b, &n);
        for (int i = 0; i < n; i++) scanf("%d", arr + i);

        sort(arr, arr + n, greater<int>());      // descending order
        cum[n] = 0;
        if (n) cum[n - 1] = arr[n - 1];
        for (int i = n - 2; i >= 0; i--)
            cum[i] = arr[i] + cum[i + 1];         // suffix sums

        sum = 0; ind = 0;
        printf(solve() ? "Sallow swallow swallows."
                       : "Sallow swallow wallows in dust.");
        if (t) puts("");
    }
    return 0;
}

The Clocks — Complete Search FULL SOLUTION Decision kn Search

⚠

Important: This solution is presented as complete search, not BFS. We enumerate all 49 = 262,144 combinations of move counts and check which one solves the puzzle.

Story: A 3×3 grid of clocks showing 12, 3, 6, or 9. Nine predefined "moves" each rotate certain clocks by 90° clockwise. Find a shortest sequence of moves that sets all clocks to 12.

Abstraction: Complete search over move-count assignments. Each move is applied 0, 1, 2, or 3 times.

Why this modeling is correct: Each move rotates clocks by 90°. Applying any move 4 times is equivalent to 0 times (360° = full circle). So each move's useful count is exactly 0, 1, 2, or 3. The order of distinct moves does not matter (they commute: all are additive mod 4 on a fixed set of clocks). Therefore, the only thing that matters is how many times each move is used, not the order. This reduces the problem from potentially infinite sequences to exactly 49 = 262,144 move-count vectors.

📌

State: (move_index, counts[0..8], clock_values[0..8]). Each counts[k] ∈ {0,1,2,3}. Clock values are mod 4 (12→0, 3→1, 6→2, 9→3). At depth m, moves 0..m−1 have fixed counts. Moves m..8 are still undecided.

One recursive step means: For move index m, try applying it 0, 1, 2, or 3 times. After trying all 4 options (0 through 3 applications), we have applied it a total of 0+1+2+3 = 6 times = 4+2 times. But we just need to note: inside the loop, after the recursive call returns, we apply the move one more time to move from count t to count t+1. After the loop (which did 4 iterations total), the move has been applied 4 times total — which restores the original state exactly (mod 4 cancellation). This is a clever self-resetting undo.

Partial solution at depth d: Moves 0..d−1 have their counts decided and applied to the clock state. The clock values reflect all those applications. Moves d..8 are undecided. When the loop for move d finishes all 4 iterations, the 4 total applications auto-undo.

Pruning / reduction: The entire search space is only 262,144 — pure enumeration runs instantly. No pruning needed.

Common dead branch: Most of the 262K assignments leave at least one clock not at 12. The function checks allTwelve() only at leaves (when all 9 moves have been assigned). Non-12 leaves are simply discarded.

Complexity: O(49) = O(262,144) — trivial.

Why in this tutorial: Teaches the most important lesson in complete search: mathematical reduction of the search space via modular arithmetic. Also demonstrates the self-resetting undo pattern.

TASK: clocks
LANG: C++
*/

#include <bits/stdc++.h>
using namespace std;

int clk[9];
int cnt[9];
int bestPresses = INT_MAX;
vector<int> bestSeq;

const int moves[9][9] = {
    {1,1,0,1,1,0,0,0,0}, // 1: ABDE
    {1,1,1,0,0,0,0,0,0}, // 2: ABC
    {0,1,1,0,1,1,0,0,0}, // 3: BCEF
    {1,0,0,1,0,0,1,0,0}, // 4: ADG
    {0,1,0,1,1,1,0,1,0}, // 5: BDEFH
    {0,0,1,0,0,1,0,0,1}, // 6: CFI
    {0,0,0,1,1,0,1,1,0}, // 7: DEGH
    {0,0,0,0,0,0,1,1,1}, // 8: GHI
    {0,0,0,0,1,1,0,1,1}  // 9: EFHI
};

void applyMove(int id) {
    for (int i = 0; i < 9; ++i)
        if (moves[id][i])
            clk[i] = (clk[i] + 1) % 4;
}

bool allTwelve() {
    for (int i = 0; i < 9; ++i)
        if (clk[i] != 0) return false;
    return true;
}

vector<int> buildSequence() {
    vector<int> seq;
    for (int i = 0; i < 9; ++i)
        for (int j = 0; j < cnt[i]; ++j)
            seq.push_back(i + 1);
    return seq;
}

void dfs(int idx, int usedPresses) {
    if (idx == 9) {
        if (!allTwelve()) return;
        vector<int> seq = buildSequence();
        if (usedPresses < bestPresses ||
            (usedPresses == bestPresses && seq < bestSeq)) {
            bestPresses = usedPresses;
            bestSeq = seq;
        }
        return;
    }

    // Try move idx exactly 0,1,2,3 times.
    // After the loop, 4 total applications restore original state.
    for (int times = 0; times < 4; ++times) {
        cnt[idx] = times;
        dfs(idx + 1, usedPresses + times);
        applyMove(idx);  // apply once more → moves to next count
    }
    // After 4 applications total: state is restored (mod 4)
    cnt[idx] = 0;
}

    freopen("clocks.in", "r", stdin);
    freopen("clocks.out", "w", stdout);

    for (int i = 0; i < 9; ++i) {
        int x; cin >> x;
        clk[i] = (x / 3) % 4; // 12->0, 3->1, 6->2, 9->3
    }

    dfs(0, 0);

    for (int i = 0; i < (int)bestSeq.size(); ++i) {
        if (i) cout << ' ';
        cout << bestSeq[i];
    }
    cout << '\n';
    return 0;
}

Party Lamps FULL SOLUTION Enumeration Parity Reduction

Story: N lamps, 4 buttons. Each button toggles a specific pattern. Given C total presses and lamps known on/off, list all achievable configurations.

Abstraction: Tiny-state parity reduction. Each button matters only modulo 2 (pressing twice = not pressing). Only 24 = 16 states.

Why this modeling is correct: Button k pressed an even number of times has no net effect (XOR cancels). So only parity matters. The constraint is: the number of odd-pressed buttons must have the same parity as C and be ≤ C.

📌

State: A 4-bit mask. Bit k = 1 means button k was pressed an odd number of times. For each mask, we check: (1) parity of popcount matches parity of C, (2) popcount ≤ C, (3) resulting lamp configuration satisfies all on/off constraints.

Partial solution at depth d: Buttons 0..d−1 have their parity assigned. Lamp positions affected by those buttons are determined. Remaining buttons are undecided.

Pruning: None needed — only 16 candidates. Constraint checking filters out invalid ones.

Common dead branch: A mask where button 1 (all lamps) is pressed but lamp 2 is required ON — the toggled state of lamp 2 depends on the combination of all buttons affecting it. If the combination forces lamp 2 OFF while the constraint requires ON, that mask is rejected.

Complexity: O(16 × N) — trivially fast.

Why in this tutorial: The ultimate lesson in mathematical reduction. The "search" is 16 iterations, not a deep recursion tree.

Full C++ Code Party Lamps 
#include <bits/stdc++.h>
using namespace std;

static void apply_button(array<int, 6>& lamps, int button) {
    if (button == 0) {
        for (int i = 0; i < 6; ++i) lamps[i] ^= 1;            // all
    } else if (button == 1) {
        for (int i = 0; i < 6; i += 2) lamps[i] ^= 1;         // odd positions
    } else if (button == 2) {
        for (int i = 1; i < 6; i += 2) lamps[i] ^= 1;         // even positions
    } else {
        for (int i = 0; i < 6; i += 3) lamps[i] ^= 1;         // 1,4,7,...
    }
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);
    int n, c; cin >> n >> c;
    vector<int> mustOn, mustOff;
    int x;
    while (cin >> x && x != -1) mustOn.push_back(x);
    while (cin >> x && x != -1) mustOff.push_back(x);
    set<string> ans;

    for (int mask = 0; mask < 16; ++mask) {
        int presses = __builtin_popcount((unsigned)mask);
        if (presses > c) continue;
        if ((c - presses) % 2 != 0) continue;  // parity check

        array<int, 6> lamps; lamps.fill(1);    // all ON initially
        for (int b = 0; b < 4; ++b)
            if ((mask >> b) & 1) apply_button(lamps, b);

        bool ok = true;
        for (int pos : mustOn)
            if (lamps[(pos-1)%6] != 1) { ok = false; break; }
        if (!ok) continue;
        for (int pos : mustOff)
            if (lamps[(pos-1)%6] != 0) { ok = false; break; }
        if (!ok) continue;

        string s; s.reserve(n);
        for (int i = 0; i < n; ++i) s.push_back(char('0'+lamps[i%6]));
        ans.insert(s);
    }
    if (ans.empty()) cout << "IMPOSSIBLE\n";
    else for (const string& s : ans) cout << s << '\n';
    return 0;
}

Pyramids FULL SOLUTION Optimization Branch & Bound

Story: Given c cubes, build distinct pyramids (high or low type) using all cubes. Minimise pyramid count.

Abstraction: Subset-sum optimization with distinct elements. Branch and bound.

📌

State: (pyramid_index, remaining_cubes, current_count). bestSz = globally best count. Pyramid candidates are pre-generated and sorted by size (largest first).

Partial solution at depth d: Pyramids from the largest down to current index have been decided. The count of chosen pyramids and remaining cubes are tracked. When you backtrack from "use," you restore the remaining cubes.

Pruning: ⌈remaining / largest_remaining⌉ + currentCount ≥ bestSz → prune. This is the branch-and-bound core.

Ordering: Candidates sorted by size descending. Large pyramids tried first → small-count solutions found early → tight bound.

Common dead branch: Trying a small pyramid when even using only the largest remaining pyramid gives a lower bound that exceeds bestSz → pruned.

Complexity: Exponential worst case, but B&B makes it fast for c ≤ 106.

Link: VJudge — UVALive 5137

Full C++ Code Pyramids 
#include <bits/stdc++.h>
using namespace std;

struct pyramid {
    int base, cnt; char typ;
    bool operator < (const pyramid& a) const {
        return make_pair(-cnt, typ) < make_pair(-a.cnt, a.typ);
    }
};

pyramid pyramids[330];
int ind = 0;

void genPyramids() {
    for (int base = 2; ; base++) {
        pyramid high = {base, 0, 'H'};
        for (int i = base; i > 0; i--) high.cnt += i * i;
        if (high.cnt <= 1000000) pyramids[ind++] = high;

        pyramid low = {base, 0, 'L'};
        for (int i = base; i > 0; i -= 2) low.cnt += i * i;
        if (low.cnt <= 1000000) {
            if (base > 2) pyramids[ind++] = low;
        } else break;
    }
    sort(pyramids, pyramids + ind);
}

int cur[7], best[7], curSz, bestSz;

void backTrack(int rem, int fst) {
    if (!rem) {
        if (curSz < bestSz) {
            bestSz = curSz;
            memcpy(best, cur, curSz * sizeof(cur[0]));
        }
        return;
    }
    if (fst == ind) return;
    // Branch & bound: optimistic lower bound on remaining pyramids
    if ((rem + pyramids[fst].cnt - 1) / pyramids[fst].cnt + curSz >= bestSz) return;

    for (int i = fst; i < ind; i++) {
        if (rem < pyramids[i].cnt) continue;
        cur[curSz++] = i;
        backTrack(rem - pyramids[i].cnt, i + 1);
        curSz--;  // undo
    }
}

int main() {
    genPyramids();
    int c, tc = 0;
    while (scanf("%d", &c), c) {
        printf("Case %d:", ++tc);
        bestSz = 7;
        backTrack(c, 0);
        if (bestSz == 7) puts(" impossible");
        else {
            for (int i = 0; i < bestSz; i++) {
                int idx = best[i];
                printf(" %d%c", pyramids[idx].base, pyramids[idx].typ);
            }
            puts("");
        }
    }
    return 0;
}

Robots on Ice FULL SOLUTION Enumeration Hamiltonian

Story: A robot visits every cell of an R×C grid exactly once (Hamiltonian path), starting from (1,1). Certain cells must be visited at exact time steps. Count all valid paths.

Abstraction: Grid Hamiltonian path enumeration with time-constrained checkpoints.

📌

State: (row, col, time_step, vis[][], unVisCnt[][]). vis[i][j] tracks visited cells. unVisCnt[i][j] counts unvisited neighbours — decremented when a neighbour is visited, restored on undo. Checkpoints: cell (ti[k], tj[k]) must be visited at time ts[k].

One recursive step means: Move to an adjacent unvisited cell. Call makeVis to mark it visited and decrement all neighbours' unVisited counts. Recurse. Then makeVis(..., 0) to undo.

Partial solution at depth d: The path from time 0 to time d is fixed. All d+1 cells are marked visited. unVisCnt reflects current topology. When you backtrack, the cell is unvisited and all neighbour counts are restored.

Pruning: (1) If at a checkpoint cell but time doesn't match → return. (2) Manhattan distance to any future checkpoint > steps available → return. (3) If exactly one forced-move neighbour exists (unVisCnt==1 and not the exit cell) → must go there. If zero forced neighbours and some exist with unVisCnt==1 that we can't reach → structural dead-end.

Common dead branch: Moving right when a cell to the left has only one remaining unvisited neighbour (us), and we're moving away → that cell becomes permanently isolated → prune. This structural reasoning is what makes the solution fast despite exponential worst case.

Complexity: Exponential, but heavy pruning makes it feasible for contest grid sizes (typically ≤ 8×8).

Link: VJudge — UVA 1098

Full C++ Code Robots on Ice 
#include <bits/stdc++.h>
using namespace std;

int ti[4], tj[4], ts[4];
bool vis[10][10];
int unVisCnt[10][10], cur = 1, nxt = 0, r, c, ans = 0;

int manDist(int i, int j, int ni, int nj) {
    return abs(i - ni) + abs(j - nj);
}

void makeVis(int i, int j, int v) {
    cur += v ? 1 : -1;
    vis[i][j] = v;
    int di = 0, dj = -1;
    for (int k = 0; k < 4; ++k) {
        int ni = i + di, nj = j + dj;
        unVisCnt[ni][nj] += v ? -1 : 1;
        swap(di, dj);
        if (k == 1) dj = 1;
    }
}

void bt(int i, int j) {
    // Checkpoint check
    if (i == ti[nxt] && j == tj[nxt]) {
        if (ts[nxt] != cur) return;      // wrong time → prune
        if (nxt == 3) { ans++; return; } // last checkpoint = done
        nxt++; bt(i, j); nxt--; return;
    }

    // Distance lower-bound prune for all future checkpoints
    for (int k = nxt; k < 4; ++k)
        if (manDist(i, j, ti[k], tj[k]) > ts[k] - cur
            || (i == ti[k] && j == tj[k]))
            return;

    makeVis(i, j, 1);  // visit cell

    // Structural / forced-move detection
    int cnt = 0, ii, jj;
    int di = 0, dj = -1;
    for (int k = 0; k < 4; ++k) {
        int ni = i + di, nj = j + dj;
        if (!vis[ni][nj]) {
            if (unVisCnt[ni][nj] == 1 && (ni != 1 || nj != 2)) {
                ii = ni; jj = nj; cnt++;
            }
        }
        swap(di, dj);
        if (k == 1) dj = 1;
    }

    switch (cnt) {
        case 1: bt(ii, jj); break;  // forced move
        case 0:                      // no forced → try all
            di = 0; dj = -1;
            for (int k = 0; k < 4; ++k) {
                int ni = i + di, nj = j + dj;
                if (!vis[ni][nj]) bt(ni, nj);
                swap(di, dj);
                if (k == 1) dj = 1;
            }
            break;
        // cnt >= 2: multiple forced neighbours → contradiction → prune
    }

    makeVis(i, j, 0);  // undo visit
}

int main() {
    int t = 0;
    while (scanf("%d%d", &r, &c), r || c) {
        for (int i = 0; i < 3; ++i) {
            scanf("%d%d", ti + i, tj + i);
            ++ti[i]; ++tj[i];
            ts[i] = (i + 1) * r * c / 4;
        }
        ti[3] = 1; tj[3] = 2; ts[3] = r * c;

        memset(vis, 0, sizeof(vis));
        memset(unVisCnt, 0, sizeof(unVisCnt));
        for (int i = 1; i <= r; ++i) vis[i][0] = vis[i][c+1] = 1;
        for (int j = 1; j <= c; ++j) vis[0][j] = vis[r+1][j] = 1;

        for (int i = 1; i <= r; ++i)
            for (int j = 1; j <= c; ++j) {
                int di = 0, dj = -1;
                for (int k = 0; k < 4; ++k) {
                    int ni = i+di, nj = j+dj;
                    unVisCnt[i][j] += !vis[ni][nj];
                    swap(di, dj); if (k==1) dj=1;
                }
            }

        ans = 0; bt(1, 1);
        printf("Case %d: %d\n", ++t, ans);
    }
    return 0;
}

Vampire Numbers FULL SOLUTION Decision Arithmetic

Story: Determine if a number's digits can be rearranged into two "fangs" whose product equals the original.

Abstraction: Digit construction with arithmetic constraints. Build factors from LSD upward.

📌

State: (partial_a, partial_b, power_of_10, freq[0..9]). As digits are assigned from LSD to MSD, partial_a and partial_b grow. The digit frequency pool shrinks. Undo returns digits to the pool.

One recursive step means: Choose a digit for position k of a (from available digits), then a digit for position k of b. This assigns two digits per level and multiplies p by 10.

Partial solution at depth d: The last d digits of both factors are fixed. (a × b) mod 10d must equal target mod 10d. Higher digits remain undecided.

Pruning: Arithmetic suffix: if (a×b) mod p ≠ target mod p, prune. Higher digits cannot fix lower-digit mismatches. Also skip leading zeros via the az/bz flags.

Common dead branch: Assigning digits that make the partial product's last k digits differ from target's last k digits → immediately pruned by the modular check.

Complexity: Exponential in digit count, but arithmetic pruning kills most branches very early.

Link: VJudge — UVALive 5123

Full C++ Code Vampire Numbers 
#include <bits/stdc++.h>
#define ll long long
using namespace std;

ll target, p = 1, a, b;
int freq[10];

bool solve(bool az, bool bz) {
    if (a * b == target) return true;
    if ((a * b) % p != target % p) return false;  // arithmetic prune

    bool ret = 0;
    for (int i = az; i < 10 && !ret; i++) {       // digit for a
        if (!freq[i]) continue;
        freq[i]--;
        a += p * i;
        for (int j = bz; j < 10 && !ret; j++) {   // digit for b
            if (!freq[j]) continue;
            freq[j]--;
            b += p * j;
            p *= 10;
            ret |= solve(i == 0, j == 0);
            p /= 10;     // undo power
            b -= p * j;  // undo b
            freq[j]++;   // restore digit
        }
        a -= p * i;  // undo a
        freq[i]++;   // restore digit
    }
    return ret;
}

int main() {

Prime Square / Prime3 FULL SOLUTION CSP Constraint Propagation

Story: Fill a 5×5 grid so every row, column, and diagonal is a 5-digit prime with a specified digit sum.

Abstraction: CSP with 12 overlapping line variables. Each assigned a 5-digit prime.

📌

State: The 5×5 grid (partially filled, using 10 as wildcard in base-11 encoding) + for each of the 12 lines, a precomputed list of candidate primes consistent with currently-filled cells. adj[encoded_line] maps a partially-filled line (with wildcards) to all matching 5-digit primes. rem[] tracks which lines are still unassigned.

One recursive step (the swim function): Find the unassigned line with the smallest candidate list (MRV). Try each candidate prime for that line. Assigning a prime fills the 5 cells of that line, which constrains all crossing lines. Undo restores wildcards.

Partial solution at depth d: d lines have assigned primes. Cells at intersections of assigned lines are fixed digits. Unassigned lines' candidate lists are implicitly narrowed by the base-11 lookup (only primes matching fixed cells are returned). When backtracking, the line's cells are restored to their previous values.

Pruning: MRV ordering + constraint propagation via the adj table. If any unassigned line's candidate list is empty → backtrack (domain emptiness).

Common dead branch: Assigning a row prime whose digit at column 3 causes column 3's domain to become empty → immediate backtrack and try next row prime.

Key modeling insight: The naive approach assigns 25 cells (1025 possibilities). The correct model assigns 12 lines. With constraint propagation, the effective search is tiny.

Complexity: Depends heavily on digit-sum constraint; typically very fast due to MRV + propagation.

    for (int i=3; i*i<=x; i+=2) if (x%i==0) return false;
    return true;
}
int digsm(int x) { int s=0; while(x){s+=x%10;x/=10;} return s; }

void addNumber(int x) {
    // Insert x into adj for all 2^5 wildcard combinations
    for (int msk=0; msk<(1<<5); msk++) {
        int y=x, p=1, s=0;
        for (int i=0; i<5; i++) {
            int d=((msk>>i)&1)?10:y%10;
            y/=10; s+=d*p; p*=11;
        }
        adj[s].push_back(x);
    }
}

    int res=0, i=si[ind], j=sj[ind];
    for (int k=0;k<5;k++){res*=11;res+=grid[i][j];i+=di[ind];j+=dj[ind];}
    return res;
}

}
vector<vector<int> > getGrid() {
    vector<vector<int> > r(5);
    for (int i=0;i<5;i++) for (int j=0;j<5;j++) r[i].push_back(grid[i][j]);

void swim(int depth) {
    if (depth==12) { sol.push_back(getGrid()); return; }

    // MRV: choose unassigned line with smallest domain
    int mn=INT_MAX, mnx, mni;
    for (int i=0;i<remsz;i++) {
        int x=read(rem[i]);
        if ((int)adj[x].size()<mn) { mn=adj[x].size(); mnx=x; mni=i; }
    }
    // If domain is empty (mn==0), backtrack automatically (loop doesn't execute)

    swap(rem[mni], rem[--remsz]);     // remove line from unassigned
    for (int i=0; i<(int)adj[mnx].size(); i++) {
        write(rem[remsz], adj[mnx][i], 10);  // assign prime (fill cells)
        swim(depth+1);
    }
    write(rem[remsz], mnx, 11);       // undo: restore wildcards
    swap(rem[mni], rem[remsz++]);     // restore line to unassigned
}

Fence Rails FULL SOLUTION Optimization Binary Search + Feasibility

Story: A farmer has N boards and R rails. Maximise how many rails can be cut.

Abstraction: Optimization → decision via binary search. Each decision call: "Can we cut the first k rails?" → backtracking feasibility test.

📌

State: (rail_index, curBoards[], currentWaste). Rails are sorted ascending. We process them from largest to smallest (index k−1 down to 0). For each rail, try assigning it to a board with enough remaining length.

One recursive step means: For rail r, try each board i with curBoards[i] ≥ rails[r]. Subtract rails[r] from curBoards[i]. If the leftover is too small for any remaining rail, count it as waste. Recurse for rail r−1. Undo restores curBoards[i].

Partial solution at depth d: Rails from the largest down to d+1 are assigned to specific boards. Each board's remaining length is reduced. currentWaste tracks unusable leftover wood.

Pruning: (1) Symmetry: if two boards have the same remaining length, only try the first (skip duplicates). (2) Waste: if currentWaste > maxWaste (= totalBoards − neededSum), prune. (3) Perfect fit: if curBoards[i] == rails[r], no need to try other boards (the rail fits perfectly in this one).

Binary search wrapper: Binary search on k (number of rails). For each k, call canMake(k) which runs the backtracking feasibility check.

Common dead branch: Assigning a rail to a board that creates waste exceeding the slack → immediately pruned by waste tracking.

Complexity: O(log R) binary search calls × exponential per call, heavily pruned.

Link: VJudge — 计蒜客 T3496

Full C++ Code Fence Rails 
#include <bits/stdc++.h>
using namespace std;

class FenceRailsSolver {
public:
    int n=0, r=0;
    vector<int> boards, rails;
    vector<int> curBoards;
    vector<long long> prefixRails;
    int targetCount=0;
    long long totalBoards=0, maxWaste=0, currentWaste=0;

    bool dfs(int railIdx, int startBoard) {
        if (railIdx < 0) return true;  // all rails assigned
        int need = rails[railIdx];

        int begin = 0;
        // Symmetry: if consecutive rails are equal, avoid duplicate board assignments
        if (railIdx+1 < targetCount && rails[railIdx]==rails[railIdx+1])
            begin = startBoard;

        int previousTriedBoardLength = -1;
        for (int i = begin; i < n; ++i) {
            if (curBoards[i] < need) continue;
            if (curBoards[i] == previousTriedBoardLength) continue; // skip dup boards

            int oldLen = curBoards[i];
            curBoards[i] -= need;  // assign rail to board

            long long addedWaste = 0;
            if (curBoards[i] < rails[0]) {  // leftover too small for any rail
                addedWaste = curBoards[i];
                currentWaste += addedWaste;
            }

            if (currentWaste <= maxWaste && dfs(railIdx-1, i))
                return true;

            if (addedWaste > 0) currentWaste -= addedWaste;  // undo waste
            curBoards[i] = oldLen;  // undo assignment

            previousTriedBoardLength = oldLen;
            if (oldLen == need) break;  // perfect fit → no point trying others
        }
        return false;
    }

    bool canMake(int k) {
        if (k == 0) return true;
        long long needSum = prefixRails[k];
        if (needSum > totalBoards) return false;
        targetCount = k;
        curBoards = boards;
        currentWaste = 0;
        maxWaste = totalBoards - needSum;
        return dfs(k-1, 0);
    }

    int solve() {
        sort(boards.begin(), boards.end());
        sort(rails.begin(), rails.end());
        totalBoards = 0;
        for (int x : boards) totalBoards += x;
        while (!rails.empty() && rails.back() > boards.back()) rails.pop_back();
        prefixRails.assign(rails.size()+1, 0);
        for (size_t i=0; i<rails.size(); ++i)
            prefixRails[i+1] = prefixRails[i] + rails[i];
        while (!rails.empty() && prefixRails[rails.size()] > totalBoards) {
            rails.pop_back(); prefixRails.pop_back();
        }
        int low=0, high=(int)rails.size(), ans=0;
        while (low <= high) {
            int mid = low + (high-low)/2;
            if (canMake(mid)) { ans=mid; low=mid+1; }
            else high = mid-1;
        }
        return ans;
    }
};

int main() {
    ios::sync_with_stdio(false); cin.tie(nullptr);
    FenceRailsSolver solver;
    cin >> solver.n;
    solver.boards.resize(solver.n);
    for (int i=0;i<solver.n;++i) cin >> solver.boards[i];
    cin >> solver.r;
    solver.rails.resize(solver.r);
    for (int i=0;i<solver.r;++i) cin >> solver.rails[i];
    cout << solver.solve() << '\n';
    return 0;
}
