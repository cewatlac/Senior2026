Hirschberg Reconstruction

Memory-Efficient Knapsack Subset Reconstruction — Full Master Tutorial
Theme

Overview FRAMEWORK

This tutorial teaches one advanced technique: reconstructing an optimal subset from a 0/1 knapsack-style DP while keeping memory at O(k) — the same as a rolling DP — instead of the O(nk) that a parent-pointer table would need. The technique is due to Hirschberg and appears in LCS, sequence alignment, and grid DP; here we study its knapsack form, which is the most contest-relevant variant.

Hirschberg = rolling DP on each half + optimal capacity split at the middle + divide-and-conquer reconstruction
What this tutorial covers
  1. Why rolling DP cannot reconstruct without a parent table
  2. The packed-score encoding that collapses two objectives into one
  3. The solve() function: rolling knapsack on any segment
  4. The hirschberg() function: divide-and-conquer reconstruction
  5. Proofs of correctness, complexity, five variants, and edge-case audit
✓ What you can do after this
  • Reconstruct optimal knapsack subsets with only O(k) memory
  • Encode lexicographic objectives into a single packed scalar
  • Recognize when Hirschberg applies vs simpler approaches
  • Extend the technique to grids, LCS, and value-knapsack variants
Prerequisites
  • 0/1 knapsack DP (both O(nk) and O(k) rolling versions)
  • Basic recursion and divide-and-conquer
  • Big-O notation and logarithms

1. Problem Definition FOUNDATION

We are given n items, each with a weight w[i], and a knapsack of capacity k. We want to choose a subset satisfying three goals simultaneously.

🎯

First, the chosen total weight must be as large as possible without exceeding k. Second, among all subsets with that same maximum total weight, we want the one with the fewest items. Third, we want to print the actual chosen items, not just report the answer values.

Maximize total chosen weight W ≤ k,
then minimize item count C among all maximum-W subsets,
then print one optimal subset.

The first two goals form a lexicographic objective: (maximize W, then minimize C). The third goal — printing the subset — is what makes this problem interesting. Without the print requirement, a simple rolling O(k) DP would suffice. With it, the question becomes how to reconstruct without blowing up memory.

💡

Why this is not trivial. A standard parent-pointer table would store for every (item, capacity) pair whether the item was taken, requiring O(nk) memory. On large inputs this can be gigabytes. Hirschberg reconstruction keeps memory at O(k) while recovering an optimal subset exactly.

2. Intuition from Scratch FOUNDATION

Let us think about the problem from the ground up, before introducing any technique. You have seen the rolling 0/1 knapsack: you process items one by one, and at each step you only need the previous DP row to compute the current one. This gives O(k) memory. But consider what you lose: once you overwrite row i−1 to produce row i, the information about which specific choices were made is gone.

Why a parent table costs O(nk)

The standard fix is a boolean table take[i][r]: for each item i and capacity r, did the optimal solution with items 1..i and capacity r take item i? This table has n × k entries. For n = 10000 and k = 75000, that is 750 million booleans — around 750 MB just for the reconstruction table. That is usually too large.

The key insight: any solution decomposes at the middle

Here is the central observation that makes Hirschberg work. Suppose the items are numbered 0 to n−1 and we split them at the middle index mid = (0+n−1)/2. The left half is items 0..mid and the right half is items mid+1..n−1.

Now think about any valid chosen subset for our problem. It consists of some items from the left half and some items from the right half. These two groups use some total capacity — say the left group uses capacity r and the right group uses capacity k−r. If we knew the value of r, we could reconstruct the two halves independently: find an optimal subset in the left half with capacity r, and an optimal subset in the right half with capacity k−r. We do not need to consider them together at all.

💡

The algorithm in one sentence: Run rolling DP on the left half (collecting the best score for every capacity), run rolling DP on the right half (collecting the best score for every capacity), try every possible split of the remaining capacity, choose the one giving the best combined score, and then recurse on each half independently.

This is the entire technique. Everything else — the packed-score encoding, the details of the DP, the base case — is in service of this single idea.

3. Visual Mental Model FOUNDATION

Let us picture the algorithm for a concrete example: items with weights [3, 5, 6, 7] and capacity k = 10.

The diagram shows the full recursion tree for our example. At every non-leaf call, the segment of items is split in half. Rolling DP is run on each half independently. Then the best capacity split is found by trying all possible values of r. The two halves are then solved recursively with their assigned capacities. At the leaves, single-item decisions are trivial. The output is {3, 7} with total weight 10.

📌

The memory magic. At every level of recursion, only two DP arrays of size O(k) are needed — one for the left half and one for the right. These arrays are reused across calls. The recursion depth is O(log n). So the total memory is O(k), not O(nk).

4. Formal Abstraction FOUNDATION

The algorithm has two parts that need to be formalized: the optimization objective (which drives the DP) and the reconstruction procedure (which recovers the subset).

The packed-score encoding

Our objective is lexicographic: maximize total weight W, then minimize item count C. One clean way to handle a lexicographic objective in a DP is to encode it as a single scalar. We define a large constant scale and assign each chosen item of weight x a score of:

contribution of item x = scale × x − 1

If a subset chooses items with total weight W and item count C, its packed score is:

packed score = scale × W − C

Now, if scale is large enough — specifically if scale > n (the maximum possible item count) — then maximizing the packed score is exactly equivalent to first maximizing W and then minimizing C. This is because a single unit of additional weight adds scale to the score, which dominates the maximum possible change in the C component (which is at most n). So no C improvement can compensate for losing even one unit of W, as long as scale > n.

💡

In the template, scale is set to 20001, which comfortably exceeds 20000 (the maximum n). This is tight but correct. If your n is larger, you must adjust scale accordingly.

What the DP computes

We run a standard 0/1 knapsack DP, but instead of maximizing the usual profit, we maximize the packed score. The state dp[r] after processing items s..i holds the best packed score achievable by choosing a subset of items s..i with total weight at most r. The transition is the standard knapsack transition with the packed contribution:

dp[r] = max(dp[r], dp[r − w[i]] + scale × w[i] − 1) if w[i] ≤ r

5. Key Observations INSIGHT

Observation 1: the packed score is strictly monotone in W

Suppose two subsets have total weights W₁ > W₂. Their packed scores differ by at least scale × (W₁ − W₂) − n ≥ scale − n > 0. So the subset with larger weight always has strictly larger packed score, regardless of item counts. This means: maximizing packed score correctly picks the heaviest subset first.

Observation 2: among same-weight subsets, larger score means fewer items

If W₁ = W₂, then the score difference is C₂ − C₁. Larger score corresponds to smaller C. So the packed score breaks ties in favour of fewer items, exactly as required.

Observation 3: the best split of capacity is the critical unknown

In the reconstruction phase, the key unknown at each call hirschberg(s, e, rem) is: how much of the remaining capacity rem does the left half of the items use in an optimal solution? Call this value r*. Once we know r*, the two halves are completely independent: the left-half problem is hirschberg(s, mid, r*) and the right-half problem is hirschberg(mid+1, e, rem−r*).

Observation 4: r* can be found by trying all splits

The best split r* satisfies: d1[r*] + d2[rem − r*] is maximized over all r from 0 to rem, where d1 and d2 are the DP rows returned by solving the left and right halves respectively. Why? Because any feasible solution of the whole problem is a feasible solution of the left half for some capacity r plus a feasible solution of the right half for capacity rem − r. The combination of the two best-per-capacity solutions, maximized over all splits, gives the globally optimal combined score. Choosing the maximizing split gives us an r* that belongs to some optimal global solution.

6. The solve() Function ALGORITHM

The function solve(dp, s, e, rem) runs a 0/1 knapsack DP over items s..e with maximum capacity rem. It returns a pointer to the final DP row — not just a single value, but the full array of best scores for every capacity 0..rem. This is what makes the reconstruction possible.

// Returns dp[0..rem] where dp[r] = best packed score
// using a subset of items s..e with total weight ≤ r.
long long *solve(long long dp[2][MXW], int s, int e, int rem) {
    // Initialize the "previous" row (before processing item s).
    // dp[!(s&1)] is the row that item s's transition reads from.
    memset(dp[!(s & 1)], 0, (rem + 1) * sizeof(dp[0][0]));

    for (int i = s; i <= e; ++i) {
        for (int r = 1; r <= rem; ++r) {
            long long &res = dp[i & 1][r];   // current row, capacity r
            int id = !(i & 1);               // previous row index
            res = dp[id][r];                 // option 1: skip item i
            if (w[i] <= r) {
                // option 2: take item i — adds packed contribution
                res = max(res, dp[id][r - w[i]] + scale * w[i] - 1);
            }
        }
    }
    return dp[e & 1];  // return the row for the last processed item
}
📌

The parity trick. Items are processed from s to e. We use a two-row rolling array indexed by i & 1. When i is even, row 0 is current and row 1 is previous; when i is odd, the roles flip. Before starting, the row that will serve as "previous" for item s — which is dp[!(s&1)] — must be set to all zeroes to represent choosing nothing.

The function is completely general: it works on any contiguous subrange [s..e] of items. This generality is what allows the hirschberg() function to call it independently on the two halves at each level of recursion.

What the returned row means

After the loop, dp[e & 1][r] holds the maximum packed score achievable by choosing a subset of items s..e with total weight at most r. This is defined for every r from 0 to rem. Returning the entire row (not just the value at rem) is essential because the reconstruction will probe every possible capacity split.

7. The hirschberg() Function ALGORITHM

This is where reconstruction happens. The function hirschberg(s, e, rem) prints all items that belong to one optimal subset of items s..e with remaining capacity rem. It uses only the two global DP buffers of size O(k).

void hirschberg(int s, int e, int rem) {
    // Base case 1: nothing to do
    if (e < s || rem == 0) return;

    // Base case 2: single item — the decision is trivial
    if (s == e) {
        if (w[s] <= rem) printf("%d\n", w[s]);
        return;
    }

    // Divide: split items at the middle
    int mid = (s + e) / 2;

    // Solve left and right halves independently
    long long *d1 = solve(dp1, s, mid, rem);
    long long *d2 = solve(dp2, mid + 1, e, rem);

    // Find the capacity split that gives the best combined score
    long long best = -1;
    int bestr = 0;
    for (int r = 0; r <= rem; ++r) {
        long long v = d1[r] + d2[rem - r];
        if (v > best) { best = v; bestr = r; }
    }

    // Conquer: recurse on each half with its assigned capacity
    hirschberg(s, mid, bestr);
    hirschberg(mid + 1, e, rem - bestr);
}

Why the single-item base case is correct

When the recursion reaches a single item s == e, it arrived there along a chain of optimal capacity splits. Each split was chosen to be consistent with some global optimum. So the subproblem "choose an optimal subset of {w[s]} with capacity rem" has exactly one non-trivial answer: take item s if it fits, skip it otherwise. The function prints it and returns. This is not a guess — it is provably the correct decision for this leaf of the recursion tree.

⚠

Note on initializing best to −1. The variable best is initialized to −1, not 0. This matters because a valid combined score of 0 exists (when no items can be chosen from either half). Initializing to 0 would cause the split r=0 to never be chosen if all scores are 0, corrupting the recursion. With −1, any valid combined score of ≥ 0 is immediately recognized as better.

8. Invariants INSIGHT

Invariant 1: DP state invariant

After solve() processes item i (for any starting item s), the following holds for every capacity r from 0 to rem:

dp[i & 1][r] = max packed score achievable by choosing a subset of items s..i with total weight ≤ r

This invariant is established at i = s by the base-row initialization (zeroes, meaning "choose nothing"), and preserved at each step i by the standard knapsack transition (take or skip item i).

Invariant 2: split-optimality invariant

For any call hirschberg(s, e, rem), the chosen split bestr satisfies: there exists an optimal solution of the subproblem (s, e, rem) in which the chosen items from the left half use exactly bestr total capacity, and the chosen items from the right half use exactly rem − bestr total capacity. In other words, the chosen split is always consistent with at least one globally optimal solution of the current subproblem.

Invariant 3: termination

Each recursive call reduces the segment length by at least half: mid − s < e − s and e − (mid+1) < e − s. Since segment length starts at e − s + 1 and halves at each level, the recursion terminates after at most ⌈log₂(e − s + 1)⌉ levels.

9. Proof of Correctness PROOF

Lemma 1: Packed score correctly encodes the objective click to expand

For any two subsets with (W₁, C₁) and (W₂, C₂), the packed score of subset 1 is larger if and only if W₁ > W₂, or W₁ = W₂ and C₁ < C₂.

Proof. If W₁ > W₂, the score difference is scale × (W₁ − W₂) − (C₁ − C₂). Since W₁ − W₂ ≥ 1 and |C₁ − C₂| ≤ n < scale, the difference is strictly positive. If W₁ = W₂, the score difference is C₂ − C₁, which is positive iff C₁ < C₂. □

Lemma 2: solve() computes correct DP values click to expand

For every segment [s..e], every rem, and every r from 0 to rem, the value returned by solve() at index r equals the maximum packed score achievable by a subset of items s..e with total weight at most r.

Proof. By induction on i from s to e. Base: before processing item s, the row is all zeroes (choosing nothing gives score 0 for any capacity). Inductive step: the transition correctly considers both choices (skip item i: inherit previous row; take item i: inherit previous row at r − w[i] plus contribution). The maximum of these two is exactly the optimal score with items s..i. □

Lemma 3: the chosen split bestr belongs to some optimal solution click to expand

For any call hirschberg(s, e, rem), if bestr maximizes d1[r] + d2[rem − r] over r = 0..rem, then there exists an optimal solution of (s, e, rem) whose left-half chosen items use exactly bestr total capacity.

Proof. Every feasible solution of (s, e, rem) decomposes into a left-half solution using some capacity r and a right-half solution using capacity rem − r. By Lemma 2, d1[r] and d2[rem−r] are the best scores for those two halves. So the best combined score over all valid splits equals max over r of d1[r] + d2[rem−r]. Any split achieving this maximum corresponds to a pair of optimal half-solutions, and together they form an optimal solution of the whole. □

Lemma 4: recursive calls remain on valid optimal subproblems click to expand

If hirschberg(s, e, rem) is called on a valid optimal subproblem, then both recursive calls hirschberg(s, mid, bestr) and hirschberg(mid+1, e, rem−bestr) are also called on valid optimal subproblems.

Proof. By Lemma 3, bestr belongs to some optimal solution X of (s, e, rem). The items of X from the left half form an optimal solution of (s, mid, bestr) — if a better one existed, combining it with the right-half portion of X would beat X, a contradiction. Similarly for the right half. □

Lemma 5: base case prints exactly the correct choice click to expand

When s == e, the algorithm prints w[s] if w[s] ≤ rem, and prints nothing otherwise. This is the correct decision for the leaf subproblem.

Proof. By Lemma 4, the call arrived here along a chain of valid optimal splits. So the subproblem is: find an optimal subset of the single item {w[s]} with capacity rem. If w[s] > rem, no subset exists except the empty one. If w[s] ≤ rem, the optimal subset is {w[s]} (since taking the item gives non-zero packed score, which is strictly better than the empty subset's score of 0, and the item was exactly the one delivered by the optimal split chain). □

📐

Final theorem. The algorithm prints exactly one optimal subset — one with maximum total weight W ≤ k and minimum item count C among all maximum-weight subsets. By Lemma 2, the initial DP computes the globally optimal packed score. By Lemmas 3 and 4, each recursive split follows some optimal decomposition. By Lemma 5, leaves print exactly the correct choices. By Lemma 1, this corresponds to the correct lexicographic optimum. □

10. Complexity Analysis INSIGHT

Time complexity

The initial call to solve() over all n items costs O(nk) time. The reconstruction phase calls solve() on segments at each level of the recursion tree. At depth d of the recursion tree, the total length of all item segments across all active calls sums to n (since the segments at each level partition the items). Each call at that level also performs a O(k) loop to find the best capacity split. So each level of the recursion tree costs O(nk) in total.

The recursion depth is O(log n) because segments halve at each level. However, the segment lengths decrease geometrically, so the total work across all levels is dominated by the top few levels. By a geometric series argument, the total reconstruction time is O(nk) — the same as the initial DP.

Total time = O(nk)

Memory complexity

The two global DP buffers dp1 and dp2, each of size 2 × (k+1), are the dominant memory cost. The recursion stack has depth O(log n) with O(1) state per frame. The item array w[] costs O(n). Total:

Total memory = O(k + n) ≈ O(k) when k ≥ n
💡

The key trade-off. Compared to storing a parent table (O(nk) memory, O(nk) time), Hirschberg reconstruction uses O(k) memory and the same O(nk) time. You give up nothing in time; you gain a factor of n in memory. That is the entire value of the technique.

Why reconstruction time is still O(nk)

At depth 0: two solve() calls on n/2 items each, so O(nk/2 + nk/2) = O(nk). At depth 1: four calls on n/4 items each, so O(4 × nk/4) = O(nk). Each depth level costs O(nk), and there are O(log n) levels. So total is O(nk log n) in the naive accounting. But in practice: the find-best-split loops cost O(k) per call, and at depth d there are 2^d calls each with segments of length n/2^d, each costing O(k × n/2^d) for its solve() calls. The total across level d is O(nk). Summing O(log n) levels: O(nk log n). This is a constant factor in practice and still faster than methods using O(nk) memory, which often have worse cache behavior.

11. Core Template Philosophy DESIGN

The template makes several deliberate design choices that are worth understanding before you use or adapt it.

Global DP buffers dp1 and dp2

Two separate global arrays are used, one for left-half solves and one for right-half solves. This avoids aliasing bugs that would occur if both halves wrote to the same buffer. They are passed as parameters to solve() so the function is reusable on any segment.

scale = 20001, not a power of 2

The scale must strictly exceed the maximum number of choosable items (at most n ≤ 20000 in the template). Using 20001 is the minimal correct value. Using a power of 2 or any other "round" number is fine as long as it exceeds n.

Parity indexing, not a copy

The two-row rolling trick uses i & 1 to determine which row is current. This is cheaper than copying one row to another at each item step, and it works correctly across any contiguous segment [s..e].

The empty-solution guard

If no item fits within capacity k, the packed score v returned by the initial DP is 0. The decoding formulas assume v > 0. An explicit guard must print "0 0" and exit early. Without this guard, the decoded W and C values are nonsensical.

12. Master Implementation CODE

The complete, ready-to-submit C++ template. Read this after the explanations in §6–7 for full context.

#include <bits/stdc++.h>
using namespace std;

// MXW must exceed the maximum capacity k
const int MXW = 75005;
// Two global rolling DP buffers: dp1 for left-half solves, dp2 for right-half
long long dp1[2][MXW], dp2[2][MXW];
// scale must strictly exceed the maximum number of choosable items (n)
const long long scale = 20001;
int w[20001];

// Runs 0/1 knapsack on items s..e with max capacity rem.
// Returns dp[0..rem]: best packed score for every capacity.
long long *solve(long long dp[2][MXW], int s, int e, int rem) {
    // Zero out the "previous" row before item s is processed
    memset(dp[!(s & 1)], 0, (rem + 1) * sizeof(dp[0][0]));
    for (int i = s; i <= e; ++i) {
        for (int r = 1; r <= rem; ++r) {
            long long &res = dp[i & 1][r];
            int id = !(i & 1);
            res = dp[id][r];                       // skip item i
            if (w[i] <= r)
                res = max(res, dp[id][r - w[i]] + scale * w[i] - 1);  // take item i
        }
    }
    return dp[e & 1];
}

// Prints the items belonging to one optimal subset of items s..e
// using at most rem total capacity.
void hirschberg(int s, int e, int rem) {
    if (e < s || rem == 0) return;          // nothing to reconstruct
    if (s == e) {                            // base case: single item
        if (w[s] <= rem) printf("%d\n", w[s]);
        return;
    }
    int mid = (s + e) / 2;
    long long *d1 = solve(dp1, s, mid, rem);      // left half DP row
    long long *d2 = solve(dp2, mid + 1, e, rem);  // right half DP row

    // Find the capacity split that gives the best combined packed score
    long long best = -1;
    int bestr = 0;
    for (int r = 0; r <= rem; ++r) {
        long long v = d1[r] + d2[rem - r];
        if (v > best) { best = v; bestr = r; }
    }
    hirschberg(s, mid, bestr);           // reconstruct left half
    hirschberg(mid + 1, e, rem - bestr); // reconstruct right half
}

int main() {
    int n, k;
    scanf("%d %d", &n, &k);
    for (int i = 0; i < n; ++i) scanf("%d", w + i);

    // Compute optimal packed score over all items
    long long v = solve(dp1, 0, n - 1, k)[k];

    if (v == 0) {          // No item fits within capacity k
        printf("0 0\n");
        return 0;
    }

    // Decode the packed score back into (bestWeight, minItems)
    long long bestWeight = v / scale + 1;
    long long minItems   = scale - (v % scale);

    printf("%lld %lld\n", bestWeight, minItems);
    hirschberg(0, n - 1, k);
    return 0;
}

13. Line-by-Line Explanation CODE

solve(): base row initialization memset(dp[!(s & 1)], 0, ...)

Before processing item s, we need the "previous" row to be zeroed. The previous row for item s is dp[!(s & 1)]. The memset writes only the first rem+1 entries (not the full MXW), which is correct and faster when rem << MXW.

Why the specific row index? When i = s, the current row is dp[s & 1] and the previous row is dp[!(s & 1)]. If we initialized the wrong row, we would use garbage data as the "no items chosen" baseline, corrupting all subsequent values.

solve(): transition res = max(skip, take) the knapsack core

The reference long long &res = dp[i & 1][r] writes directly into the current row. It is first set to dp[id][r] (skip item i). If w[i] ≤ r, it is then updated to the maximum with dp[id][r − w[i]] + scale × w[i] − 1 (take item i, add packed contribution). The −1 counts one more chosen item.

solve(): return dp[e & 1] the last row

After processing all items s..e, the final DP row is dp[e & 1]. This pointer is returned. The caller (hirschberg) uses it to probe any capacity r from 0 to rem. Do not copy this row — return the pointer directly to avoid an O(k) copy.

hirschberg(): bestr = 0, best = -1 initialization matters

Using best = -1 ensures that even a combined score of 0 (both halves choose nothing) is properly recorded at r = 0. If best were 0, the split r = 0 would never be chosen, which is wrong when all scores are 0 (the empty-solution case). The guard if (v == 0) return early in main prevents most such cases, but robustness in hirschberg() itself is still important.

main(): decoding v into bestWeight and minItems the inverse formula

The packed score v = scale × W − C. Since 1 ≤ C < scale, the quantity v % scale equals scale − C (because v mod scale = (scale × W − C) mod scale = scale − C when C > 0). Thus:

W = v / scale + 1 (integer division, the +1 compensates for C ≥ 1)
C = scale − (v % scale)

Verify: packed score of a subset with W = 10, C = 2, scale = 20001 is 200010 − 2 = 200008. Then: W = 200008 / 20001 + 1 = 10 + 1 = ... wait. 200008 / 20001 = 9 (integer division). So W = 9 + 1 = 10. ✓ And C = 20001 − (200008 % 20001) = 20001 − (200008 − 9×20001) = 20001 − (200008 − 180009) = 20001 − 19999 = 2. ✓

14. Decoding the Answer CODE

Understanding the decoding formulas is important enough to deserve its own section, because they are easy to get wrong.

The packed score for a subset with total weight W and item count C is v = scale × W − C. We know that C satisfies 1 ≤ C < scale (every chosen item contributes at least 1, and there are fewer than scale items). From this, two properties follow:

ExpressionValueWhy
v / scale (integer)W − 1scale×W/scale = W, but the −C term causes one unit of borrowing since C < scale
v % scalescale − C(scale×W − C) mod scale = (0 − C) mod scale = scale − C
v / scale + 1Wthe +1 corrects the borrowing
scale − v % scaleCscale − (scale − C) = C
⚠

This decoding only works when v > 0. If v = 0 (no item can be chosen), both formulas produce nonsense. Always guard against v = 0 before applying the formulas.

15. Edge-Case Audit AUDIT

These are the eight concrete corners where the algorithm fails if not handled correctly.

EC1: No item fits (v = 0)

The initial DP returns v = 0. The decoding formulas are undefined for v = 0. Print "0 0" and return immediately. Forgetting this guard produces outputs like W = 1, C = 20001 which are completely wrong.

EC2: scale too small

If scale ≤ n, then it is possible for the C-component to overflow into the W-component, corrupting the lexicographic ordering. Always verify scale > max-possible-n, not just "approximately large enough".

EC3: Wrong parity row initialized

The memset must target dp[!(s & 1)], not dp[0]. If s is odd, the "previous" row is row 0, so row 0 must be zeroed. Zeroing the wrong row produces silently wrong DP values.

EC4: best = 0 instead of −1

If best is initialized to 0, split r = 0 is never selected even when all valid scores are 0. This corrupts reconstruction when no items can be chosen in either half.

EC5: integer overflow

The packed score can reach scale × k ≈ 20001 × 75000 ≈ 1.5 × 10⁹. This fits in a 32-bit int. But if k or scale are larger, use long long throughout. The template already uses long long.

EC6: duplicate weights

If two items have equal weight, the output (printing weights) is ambiguous. If the problem wants item indices, store and print the original index, not w[s], in the base case.

EC7: rem = 0 at leaf

The base case guard if (e < s || rem == 0) return must come before the single-item check, because rem = 0 should produce no output even for a leaf segment.

EC8: recursion stack depth

Depth is O(log n), which is at most 17 for n = 20000. Stack overflow is not a practical concern. However, do not confuse this with algorithms that may recurse O(n) deep — this one is safely shallow.

16. Common Pitfalls PITFALLS

Pitfall 1: "This is just divide and conquer" missing the key step

Divide and conquer alone does not help. What makes Hirschberg work is that we solve each half with a complete DP row (best score for every capacity), not just for one target capacity. This is the step that lets us try all possible capacity splits and find the one consistent with a global optimum. Without it, the halves cannot be correctly linked.

Pitfall 2: thinking the halves are independent they are not, until bestr is chosen

The two halves share the total remaining capacity rem. They are not independent until the split bestr is decided. Only after choosing bestr are they truly independent subproblems. A common mistake is to run each half with the full rem, which over-counts the available capacity.

Pitfall 3: using a pair instead of packed score, naively max comparison on pairs

It is possible to use pair<int,int> with the comparison (larger first component, then smaller second) instead of the packed score. This is pedagogically cleaner. However, adding pairs is not defined by default — you would need custom logic. The packed scalar is simpler and faster in practice.

Pitfall 4: confusing "capacity at most r" with "capacity exactly r" the DP semantics

The DP computes the best score for capacity at most r, not exactly r. This means dp[r] ≥ dp[r-1] always. The split search correctly uses this: d1[r] + d2[rem−r] asks "best from left using ≤ r, and best from right using ≤ rem − r". The total used weight may be less than rem, which is correct since we want to maximize weight without exceeding k.

Pitfall 5: using the same buffer for both halves aliasing bug

Both solve() calls inside hirschberg() must use different buffers (dp1 and dp2). If the same buffer is used, the right-half computation overwrites the data from the left-half computation, making the split search impossible. The template uses separate global arrays for exactly this reason.

17. Decision / Recognition Guide GUIDE

The four rules for when to use Hirschberg

Rule 1: Rolling DP exists
REQUIRED

The optimal value can be computed with a rolling (low-memory) DP. This means the DP state can be expressed with only two adjacent rows or a single row, not the entire table.

Rule 2: Output required
REQUIRED

The problem asks you to print an actual solution (chosen items, path, sequence), not just the optimal value. Without this requirement, rolling DP alone suffices.

Rule 3: Parent table too large
TRIGGER

A full parent table would cost O(nk) or O(nm) memory, which exceeds the memory limit. This is the real reason to use Hirschberg over a simple parent table.

Rule 4: Clean middle split
REQUIRED

The optimal solution can be decomposed into two independent optimal subproblems through a single "middle boundary" parameter (capacity split, row crossing, string split).

✓ Use Hirschberg
when all 4 hold

If all four rules are satisfied, Hirschberg reconstruction is a strong candidate. Examples: knapsack with output, LCS with reconstruction, grid shortest path with output.

✗ Do not use
common exceptions

If parent table fits in memory: use the simpler table. If only the value is needed: use rolling DP. If you need all optimal solutions: Hirschberg finds one. If the recurrence does not split cleanly: the method may not apply.

Quick reflex guide

In a contest, scan for these signals: "n, k large," "memory limit tight," "print the chosen items," "find minimum/maximum number of items." If you see all of these, reach for Hirschberg. Also look for the words "restoration," "reconstruction," or "find the actual subset" alongside a large constraint on the DP dimensions.

⚠

Common confusion: Many people think Hirschberg is only for LCS or sequence alignment. It is not. The technique is about any DP where reconstruction can be driven by discovering the correct middle split. The knapsack variant is equally valid and perhaps more contest-frequent because of how often large-capacity problems demand output of the chosen items.

18. Variants & Extensions VARIANTS

Variant 1: Print item indices instead of weights most practical change

Problem story

The judge wants the 1-indexed positions of the chosen items in the original input order, not their weight values. Items may have repeated weights, so printing weights is ambiguous.

What changes

In the base case of hirschberg(), instead of printing w[s], print s + 1 (or the stored original index). If items were sorted before the DP, store the mapping from sorted position to original index in an auxiliary array, and print the original index at the leaf.

// In the base case, replace:
if (w[s] <= rem) printf("%d\n", w[s]);
// With:
if (w[s] <= rem) printf("%d\n", originalIdx[s]);

Complexity: unchanged. When to use: whenever the judge identifies items by their input position, not their weight.

Variant 2: Use a pair instead of packed score pedagogically cleaner

Why

The packed scalar is fast and compact, but it obscures intent. An alternative is to store pairs (total_weight, -item_count) and use lexicographic comparison. This makes the DP semantics more readable at the cost of more complex DP state and potentially slower pair operations.

What changes

Replace long long dp[2][MXW] with pair<int,int> dp[2][MXW], change the transition to use the pair ordering, and remove the packed-score encoding/decoding. The hirschberg() function's split loop compares pairs instead of scalars.

Complexity: same asymptotically; pair comparisons are O(1) but involve two integer comparisons. When to use: for pedagogy or when the scale trick is unavailable (e.g., the count dimension is not bounded).

Variant 3: Standard value knapsack (weight + profit) most general form

Problem story

Each item has both a weight and a profit value. We want to maximize total profit (not total weight) subject to the capacity constraint. The output is the list of chosen items.

What changes

Replace the packed contribution scale × w[i] − 1 with value[i] (or scale × value[i] − 1 if you still want a tie-break on item count). The rest of the algorithm is unchanged. The solve() function now computes maximum total profit, and the split search still finds the optimal capacity split.

// In the take-item branch, replace:
dp[id][r - w[i]] + scale * w[i] - 1
// With:
dp[id][r - w[i]] + value[i]

If you do not need the tie-break on item count, the decoding formulas also simplify: v itself is the maximum profit, not a packed representation.

Variant 4: Classical grid / path DP the original Hirschberg context

Problem story

You have a grid and want a minimum-cost path from top-left to bottom-right. A parent pointer table costs O(nm) memory. You want to reconstruct the path with only O(n + m) memory.

How Hirschberg applies

Split the grid at the middle column. Run the path DP forward from the left to accumulate costs reaching each cell of the middle column. Run the path DP backward from the right to accumulate costs for the continuation from each middle cell. The row in the middle column that minimizes total cost is the crossing row of an optimal path. Recurse on the left and right rectangles. This is the classic Hirschberg grid algorithm and is the technique that Hirschberg originally proposed.

Complexity: O(nm) time, O(n + m) memory — a factor of min(n, m) savings over the parent table.

Variant 5: LCS / sequence alignment reconstruction the string form

Problem story

Find one longest common subsequence of strings A (length m) and B (length n). A full LCS table costs O(mn) memory. You want reconstruction with O(m + n) memory.

How Hirschberg applies

Split string A at the middle. Compute the forward LCS score row (how much of B matches the first half of A). Compute the backward LCS score row (how much of B matches the reversed second half of A, also reversed). Find the split position in B that maximizes their sum. Recurse on the two halves of A, each matched to its portion of B. This is the original Hirschberg LCS algorithm from 1975.

Complexity: O(mn) time, O(m + n) memory.

19. Problem Applications PROBLEMS

The Hirschberg knapsack reconstruction is a relatively rare but distinct technique. Here are the problem families where it appears or becomes necessary.

Subset restoration with tight memory

The canonical fit: n items, capacity k, print chosen items. The usual flag when this needs Hirschberg is n × k > 10⁷ in the context of a memory limit of 256 MB.

Practice: search for "knapsack reconstruction" on competitive programming platforms, or subset-sum restoration problems where the product nk is large.

LCS / edit distance reconstruction

When m × n is large and the problem asks to print the actual LCS or alignment, Hirschberg's string version applies. This is the original use case.

Practice: Codeforces problems on LCS with large n, where memory is the binding constraint.

Grid path restoration

Find and print a minimum-cost path in an n × m grid where storing the full DP table is too large. Split the grid at the middle column and apply the grid variant.

Practice: grid DP problems where path output is required and n × m > 10⁷.

Maximize used capacity problems

The exact template in this tutorial: choose items to maximize used weight, minimize count, print subset. Appears in packing, scheduling, and bin-covering variants.

Practice: VJudge — search for "0/1 knapsack" with "print" or "output" in the problem statement.

How to recognize the pattern in a contest

The pattern has a clear signature: the problem gives large n and k (so n × k is at most around 10⁸), it asks you to find some optimal subset or path, and it wants you to print the actual elements (not just the value). If the memory limit is 256 MB or less and n × k bytes exceeds that, you cannot afford a parent table and should reach for Hirschberg. The four recognition rules in §17 operationalize this check.

20. Final Summary SUMMARY

Hirschberg is not about strings, grids, or knapsacks specifically — it is about reconstructing an optimal DP solution by discovering the correct middle split while keeping memory as small as the value DP itself.

Here are the five ideas you should leave this tutorial with, each building on the previous one.

1
Rolling DP loses reconstruction information.

Storing only two rows is memory-efficient but throws away the parent pointers needed to trace back the optimal solution. A parent table costs O(nk). Hirschberg recovers the solution without ever building that table.

2
Any optimal solution decomposes at the middle.

Split the items at mid. The optimal solution uses some capacity r in the left half and rem−r in the right. Running rolling DP on each half and searching over all r finds the correct split. This is the key insight.

3
The packed score encodes lexicographic objectives.

Using score = scale × W − C where scale > n collapses two objectives (maximize W, minimize C) into one scalar that a standard DP can maximize. Decoding recovers both values with a division and a modulo.

4
Recursion depth is O(log n), so memory stays O(k).

At every recursion level, only two arrays of size O(k) are needed. These are reused across all calls at the same level. The O(log n) stack frames add negligible memory. Total: O(k + n).

5
This is a general principle, not a special trick.

The same divide-at-the-middle idea works for knapsack, LCS, grid paths, sequence alignment, and any DP where solutions decompose cleanly through a single split parameter. Learning the principle, not just the template, is what lets you apply it in new settings.
