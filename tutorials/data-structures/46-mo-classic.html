Mo's Algorithm — Classic Array

Offline Range Queries · Sqrt Decomposition by Locality · The Master Template of the Family

🧩 Problem Definition

We are given a static array a[0], a[1], …, a[n-1] and many range queries of the form

query i = [l_i, r_i]
answer_i = F(a[l_i], a[l_i+1], ..., a[r_i])

The aggregator F can be simple or complicated. Mo's algorithm becomes useful when F depends on the multiset of values inside the range, not just the prefix sums.

Typical aggregators Mo handles well

Examples that fit naturally include the number of distinct values, the sum of value · frequency², the count of values whose frequency equals some target x, the number of pairs with XOR equal to k, the mode of an interval, and the number of colours appearing at least k times in a subtree (after an Euler-tour transformation). Each of these reduces to maintaining a frequency table while one element enters or leaves the active window.

When Mo's algorithm is the right tool

Mo applies when four conditions hold together. First, all queries must be known offline so they can be reordered. Second, the array is static, unless we add the time dimension and use the update-aware variant. Third, we can maintain an answer for the current range as a single piece of state. Fourth, adding or removing one element to the active range is much cheaper than recomputing the answer from scratch.

Mo is rarely the first tool you should reach for. It is the tool you reach for when the answer is easy to update locally, but hard to merge from two halves.

💡 Intuition from Scratch — Why Mo Exists

Naive idea: recompute every range

For each query [l, r], scan the whole subarray. Each query costs up to O(n), and with q queries that is O(nq). For n, q ≈ 2·10⁵ this is far too slow.

Why prefix sums may fail

Prefix sums work when the answer decomposes as

answer(l, r) = prefix[r] − prefix[l-1]

This is fine for sums, XOR, or counts of one fixed value. But it does not work for the number of distinct values in [l, r], because duplicates interact across the boundary. Knowing the distinct count of [1, 5] and [6, 10] does not tell you the distinct count of [1, 10] — the same value can appear in both halves and gets double-counted.

Why segment trees may fail

A segment tree needs a clean merge: combining the answer of the left half and the answer of the right half must give the answer of the whole interval. Many frequency-based answers refuse to merge cheaply. The mode of the left half and the mode of the right half do not determine the mode of the whole; the merge would essentially need to look at both frequency tables, which kills the logarithmic advantage.

Mo's different idea — do not merge, walk

Mo refuses to merge anything. Instead it says: keep one current interval and move it slowly from one query to the next. If query A is [10, 20] and query B is [12, 25], do not recompute B from scratch. Remove positions 10 and 11, then add positions 21 through 25. The cost is just the distance the endpoints travel.

This turns the whole problem into a routing problem: in what order should we visit all the query intervals so the endpoints move as little as possible? Mo's sort is a simple, locality-preserving heuristic, not the optimal route.

🎯 Visual Mental Model — Two Fingers on the Array

Imagine two fingers, called curL and curR, on the array. The current interval is exactly what lies between them. The data structure (a frequency table, a counter, or whatever the problem needs) always describes that interval and nothing else.

🧠 Formal Abstraction — Two Layers

Mo cleanly separates the algorithm into two independent layers. This is what makes the master template reusable: only the answer layer changes between problems.

Movement layer — knows nothing about the answer

The movement layer only knows how to transform the current interval into the next query interval. It is exactly four while loops:

while (curL > L) add(--curL);
while (curR < R) add(++curR);
while (curL < L) remove_(curL++);
while (curR > R) remove_(curR--);

These four lines are identical for every Mo problem on a static array. They are the "engine" of the algorithm.

Answer layer — the problem-specific brain

The answer layer knows what state must be kept so that adding or removing one index updates the answer correctly. Some common shapes are listed below.

Query typeMaintained state
distinct valuesfreq[value], scalar distinct
Powerful Array (∑ x·cnt[x]²)freq[value], scalar current_answer
mode of intervalfreq[value] plus a structure over frequencies
XOR pairscntPrefix[xor], scalar current_pairs
tree path distinctactive[node], freq[color], scalar distinct

The single design question in every Mo problem is: what must I store so that adding or removing one endpoint updates the answer correctly? Once that is solved, the movement layer is copy-paste.

🔭 Key Observations

Observation 1 — Query order does not affect correctness

Queries are independent. If all queries are known beforehand, we may answer them in any order and store each answer by its original index id. This is what unlocks reordering.

Observation 2 — Adjacent ranges are cheap

Moving from [L, R] to [L+1, R] costs exactly one removal. Moving to [L, R+1] costs exactly one addition. So the total work depends on the total endpoint travel distance, not on the number of queries times the range length.

Observation 3 — A good route, not the optimal route

Think of every query as a point (l, r) in a 2-D grid. Moving from query (l₁, r₁) to (l₂, r₂) costs |l₁ − l₂| + |r₁ − r₂|. The optimal visiting order is a Manhattan TSP — too hard. Mo's heuristic is dead simple: group queries by the block of l, and inside each block sort by r. The two pointers then never wander randomly.

Observation 4 — The update must be reversible

Classic Mo needs both add and remove. If the problem has easy insertion but hard deletion (the canonical example is connectivity with a DSU), classic Mo may not directly fit. In that case, consider rollback Mo, DSU-on-tree, or offline divide-and-conquer instead — these are covered in the variants file.

⚙️ The Algorithm — Just Four Loops

The whole Mo engine reads like this. After this loop body, the maintained data structure exactly represents the elements of [q.l, q.r], and we can read off the answer for query q.

for each query q (in sorted order) {
    while (curL > q.l) add(--curL);   // grow left
    while (curR < q.r) add(++curR);   // grow right
    while (curL < q.l) remove(curL++); // shrink left
    while (curR > q.r) remove(curR--); // shrink right

    ans[q.id] = current_answer;
}

Why exactly these four loops, in this order

The four loops are designed so that at every moment the active interval is a valid (non-broken) interval, and the maintained state describes it. The two growing loops (curL-- and curR++) come first because they enlarge the interval to a superset of both the current and target ranges — this guarantees we never "skip over" elements that should be included. The two shrinking loops then trim the interval down to exactly the target range. The growth-before-shrink order also avoids transient negative-length intervals, which would corrupt the frequency table.

Pre-decrement (--curL) and pre-increment (++curR) are critical inside the growth loops: we move the pointer first, then add the newly-included index. In the shrink loops the order is reversed (curL++, curR--) so that we remove the index that is leaving before the pointer leaves it.

🔒 Invariants — What is Always True

State invariant

At all times during the main loop, the maintained data structure represents exactly the elements indexed by [curL, curR]. For the initial empty interval we use curL = 0, curR = -1, which is a valid representation of "no elements yet".

Transition invariant

Every pointer movement updates the state consistently. If an index enters the interval, add(index) is called exactly once. If an index leaves, remove(index) is called exactly once. Nothing is called twice, and nothing is skipped.

Query invariant

Immediately before storing ans[q.id], we have curL == q.l and curR == q.r. The maintained state therefore represents exactly the query range, and get_answer() returns its answer.

Termination invariant

Each of the four while loops strictly moves one pointer one step closer to its target on every iteration, so all four loops terminate in finite time, with no chance of livelock.

📜 Proof of Correctness — Three Lemmas

Lemma 1 — Each pointer move preserves the state invariant

Assume the maintained state represents exactly [curL, curR] before a single step. Four cases:

If we execute add(--curL), the interval becomes [curL−1, curR] and the newly-included index is added exactly once. If we execute add(++curR), the interval becomes [curL, curR+1] and the new right end is added once. If we execute remove(curL++), the old curL is removed before the pointer leaves; the interval becomes [curL+1, curR]. If we execute remove(curR--), the old curR is removed first; the interval becomes [curL, curR−1]. In all four cases, after the step the state still matches the new interval.

Lemma 2 — After the four loops, the current interval equals the query interval

The first loop decreases curL until curL ≤ q.l. The third loop increases curL until curL ≥ q.l. Therefore curL == q.l. Symmetrically, the second loop pushes curR up to at least q.r, and the fourth loop pulls it back to exactly q.r. So after all four loops, [curL, curR] = [q.l, q.r].

Lemma 3 — The stored answer for every query is correct

By Lemma 2, before storing ans[q.id] the current interval is exactly [q.l, q.r]. By Lemma 1, the maintained state at this moment represents this interval. Therefore get_answer() returns the correct answer for the query.

Theorem

The algorithm outputs the correct answer for every query, in original input order. Each query is processed exactly once; by Lemma 3 each stored answer is correct; the answers are stored in an array indexed by id, so the final output respects the original order even though queries were processed in Mo order.

⏱ Complexity Analysis

Let n be the array length, q the number of queries, B the block size, and F the cost of one add / remove call.

Left-pointer cost

Within a fixed left block, all queries have l inside a range of length B. So moving curL from one query to the next costs at most O(B). Across q queries, the left pointer pays O(qB) in total.

Right-pointer cost

There are about n / B left blocks. Inside each block, queries are sorted by r (or reverse-r in odd blocks), so the right pointer mostly moves in one direction across the whole array; for one block that costs O(n). Across all blocks: O(n · (n/B)) = O(n²/B).

Total work

movement = O(qB + n²/B)
total    = O((qB + n²/B) · F)

With B = √n this collapses to the familiar O((n + q) · √n · F).

Better theoretical block size when q differs greatly from n

Balancing the two terms gives qB = n²/B, i.e. B = n / √q. So when q is much smaller or much larger than n, the tuned choice is:

BLOCK = max(1, (int)(n / sqrt(q)));

In contests many people still hard-code a block size by experiment because cache behaviour can matter more than the asymptotic constant.

📄 Master Template — Classic Array Mo

The skeleton below is 0-based and uses odd-even ordering by default. To solve a new problem, edit only the marked section. Everything outside the marked region is untouched across the entire variant family.

#include <bits/stdc++.h>
using namespace std;

struct Query {
    int l, r, id;
};

int BLOCK;

bool mo_cmp(const Query& a, const Query& b) {
    int block_a = a.l / BLOCK;
    int block_b = b.l / BLOCK;

    if (block_a != block_b) return block_a < block_b;

    // Odd-even ordering: reduce right-pointer backtracking.
    if (block_a & 1) return a.r > b.r;
    return a.r < b.r;
}

// ============================================================
// Problem-specific state begins here.
// Replace this section for each problem.
// ============================================================

vector<int> a;
long long current_answer = 0;

void add_index(int idx) {
    // Add a[idx] to the maintained interval.
}

void remove_index(int idx) {
    // Remove a[idx] from the maintained interval.
}

long long get_answer() {
    return current_answer;
}

// ============================================================
// Generic Mo solver. Do not modify.
// ============================================================

vector<long long> solve_mo(int n, vector<Query> queries) {
    BLOCK = max(1, (int)sqrt(n));

    sort(queries.begin(), queries.end(), mo_cmp);

    vector<long long> ans(queries.size());

    int curL = 0;
    int curR = -1; // empty interval

    for (const Query& q : queries) {
        while (curL > q.l) add_index(--curL);
        while (curR < q.r) add_index(++curR);
        while (curL < q.l) remove_index(curL++);
        while (curR > q.r) remove_index(curR--);

        ans[q.id] = get_answer();
    }

    return ans;
}

🔎 Line-by-Line Explanation

struct Query

Each query carries its left and right endpoints, plus the field id recording its original input position. Storing id is non-negotiable: Mo answers queries in a permuted order, and we need id to put each answer back in the correct output slot.

The block size variable

BLOCK is a global so the comparator can read it. It is initialized once, just before sorting. sqrt(n) is the safe default; n / sqrt(q) is the tuned version when q is far from n.

The comparator

block_a = a.l / BLOCK maps each query's left endpoint to a block index. Queries in earlier blocks are processed first. Inside the same block, even blocks sort right endpoints ascending and odd blocks sort them descending — the "snake" or "boustrophedon" pattern, which avoids the right pointer jumping back to a small r at the start of each new block.

The initial interval

curL = 0, curR = -1 is the canonical representation of an empty interval. It is consistent: the set of indices {i : curL ≤ i ≤ curR} is empty when curR < curL. The very first query will simply expand curR upward and (if q.l > 0) shrink curL upward.

Expanding left — while (curL > q.l) add_index(--curL);

The pre-decrement runs first, so curL becomes the new index. Then that new index is added to the maintained state. After this loop, curL ≤ q.l.

Expanding right — while (curR < q.r) add_index(++curR);

Symmetric: pre-increment first, then add. After this loop, curR ≥ q.r.

Shrinking left — while (curL < q.l) remove_index(curL++);

The element at curL is leaving the interval, so we remove it before the post-increment moves the pointer away. After this loop, curL = q.l.

Shrinking right — while (curR > q.r) remove_index(curR--);

Same idea on the right side. After this loop, curR = q.r.

Storing the answer

At this point, [curL, curR] = [q.l, q.r] and the maintained state is correct. We store get_answer() into ans[q.id], not ans[i], so the answer ends up in the right output slot.

🔀 Ordering Strategies — Plain, Odd-Even, Hilbert

Plain block ordering

The simplest comparator sorts by left-block, then by right endpoint:

bool cmp(const Query& a, const Query& b) {
    if (a.l / BLOCK != b.l / BLOCK) return a.l < b.l;
    return a.r < b.r;
}

Plain ordering is correct, but at the boundary between blocks the right pointer may have to jump backward from a very high r in the previous block to a very low r in the next block. Asymptotically this is fine; in practice it wastes work.

Odd-even ordering — the default

Inside odd-indexed blocks, reverse the secondary key:

bool cmp(const Query& a, const Query& b) {
    int ba = a.l / BLOCK;
    int bb = b.l / BLOCK;
    if (ba != bb) return ba < bb;
    if (ba & 1) return a.r > b.r;   // odd block: r decreasing
    return a.r < b.r;               // even block: r increasing
}

The visit pattern becomes a snake: block 0 sweeps r upward, block 1 sweeps it downward, block 2 upward again, and so on. The right pointer continues smoothly across block boundaries instead of resetting. The asymptotic bound is unchanged but the constant factor typically drops 20-40 percent.

Hilbert order — when even odd-even is not enough

Hilbert order treats each query as a 2-D point (l, r) and sorts by a locality-preserving space-filling curve. Use it when n and q are both large, add/remove is expensive, and odd-even Mo barely passes the time limit. The Hilbert key is computed once per query at read-time, then queries are sorted by that key.

long long hilbert_order(int x, int y, int pow = 21, int rot = 0) {
    if (pow == 0) return 0;

    int hpow = 1 << (pow - 1);
    int seg;

    if (x < hpow) {
        seg = (y < hpow) ? 0 : 3;
    } else {
        seg = (y < hpow) ? 1 : 2;
    }

    seg = (seg + rot) & 3;

    static const int rotate_delta[4] = {3, 0, 0, 1};

    int nx = x & (x ^ hpow);
    int ny = y & (y ^ hpow);
    int nrot = (rot + rotate_delta[seg]) & 3;

    long long sub_square_size = 1LL << (2 * pow - 2);
    long long ans = 1LL * seg * sub_square_size;
    long long add = hilbert_order(nx, ny, pow - 1, nrot);

    ans += (seg == 1 || seg == 2) ? add : (sub_square_size - add - 1);
    return ans;
}

Then add a field long long ord to Query, set query.ord = hilbert_order(query.l, query.r, 21) while reading input, and sort by ord. The recursion depth pow must be large enough that 2^pow exceeds the maximum coordinate; 21 covers up to about 2 million.

🧪 Answer-Maintenance Patterns

These are the four most useful answer-layer recipes. They are the only things that change between problems; the master template above is reused verbatim.

1) Distinct values scalar + freq[]

Maintain freq[x] and a scalar distinct_count. The invariant is distinct_count = |{x : freq[x] > 0}|.

void add_value(int x) {
    if (freq[x]++ == 0) distinct_count++;
}

void remove_value(int x) {
    if (--freq[x] == 0) distinct_count--;
}

The trick is that distinct_count only changes at the moments when a frequency transitions through zero — exactly when the post-increment in add was previously zero, or the pre-decrement in remove just reached zero.

2) Powerful Array — sum of freq[x]² · x algebraic delta

If value x currently has count c, its contribution is c² · x. Adding one x changes the contribution from c²·x to (c+1)²·x, a delta of (2c+1)·x. Removing gives a delta of −(2c−1)·x. Compute the delta before mutating freq:

void add_value(int x) {
    current_answer += 1LL * (2 * freq[x] + 1) * x;
    freq[x]++;
}

void remove_value(int x) {
    current_answer -= 1LL * (2 * freq[x] - 1) * x;
    freq[x]--;
}

Order matters: in add the delta uses the old count c; in remove the delta also uses the old count, which is the count before the decrement.

3) Count values whose frequency equals (or exceeds) something double-indexed

Maintain two arrays. freqValue[x] is the frequency of value x, and freqCount[f] is the number of distinct values that currently have frequency exactly f. Both transitions are three lines:

// add x:
freqCount[freqValue[x]]--;
freqValue[x]++;
freqCount[freqValue[x]]++;

// remove x:
freqCount[freqValue[x]]--;
freqValue[x]--;
freqCount[freqValue[x]]++;

This unlocks three different queries with no extra cost: "how many values appear exactly k times?" is just freqCount[k]; the MEX of frequencies is the first f ≥ 1 with freqCount[f] == 0; the count of values with frequency at least k is a suffix sum, supported with a Fenwick tree over freqCount.

4) XOR-pair counting via prefix XOR index transform

Build pref[0] = 0 and pref[i] = a[0] ⊕ a[1] ⊕ … ⊕ a[i−1]. Then xor([l, r]) = pref[r+1] ⊕ pref[l]. Counting subarrays inside [l, r] with XOR equal to k becomes counting prefix-index pairs inside [l, r+1] whose XOR is k. The state is cntPrefix[x] and a scalar current_pairs:

void add(int idx) {
    int x = pref[idx];
    current_pairs += cntPrefix[x ^ k];
    cntPrefix[x]++;
}

void remove_(int idx) {
    int x = pref[idx];
    cntPrefix[x]--;
    current_pairs -= cntPrefix[x ^ k];
}

The ordering inside add and remove is asymmetric on purpose: when adding, count pairs against existing partners first, then bump the counter; when removing, bump the counter down first, then subtract pairs against the remaining partners. This avoids self-pairing.

🪜 Centralized Edge-Case Audit

Before submitting any Mo solution, walk this checklist. Each item is a real bug that has lost real submissions.

#CheckWhy it matters
1Queries 0-based or 1-based?SPOJ, Codeforces, and your code may all disagree. Convert at read-time, then stay 0-based internally.
2Is r inclusive?The template assumes inclusive [l, r]. If the problem gives [l, r), decrement r at read-time.
3Does the initial interval represent empty?curL = 0, curR = -1 is the canonical empty state. Any other initial value will mis-count the first query.
4Values coordinate-compressed if needed?If values reach 10⁹, you cannot index freq[a[i]] directly. Compress to [0, distinct_count).
5Can the answer overflow int?Powerful-array-style answers can reach 10¹⁸. Use long long for the running total.
6Is remove exactly the inverse of add?Asymmetric implementations are the most common silent bug. Test on a long sequence of add(x); remove(x); pairs.
7Does freq[0] mean something?In frequency-of-frequency problems, freqCount[0] is meaningless. Skip it when querying.
8Are queries stored by original id?Writing to ans[i] after sorting prints answers in the wrong order. Always use ans[q.id].
9Is the block size nonzero?sqrt(0) is 0 and division-by-zero crashes. Use max(1, (int)sqrt(n)).
10Does Hilbert pow cover max coordinate?pow = 21 covers 2²¹ ≈ 2·10⁶. For larger n, bump it.
11Tree-Mo: LCA handled exactly once?The doubled-Euler trick toggles the LCA exactly once when needed and untoggles after. Forgetting either side gives off-by-one in the answer for many queries.

⚠️ Common Pitfalls

Pitfall 1 — Removing after changing the pointer

Wrong code (removes the new position, not the leaving one):

curL++;
remove_index(curL);   // BUG: removes a[curL+1_old], not a[curL_old]

Correct:

remove_index(curL++);   // remove first, then move

Pitfall 2 — Forgetting coordinate compression

If values can reach 10⁹, then freq[a[i]]++ silently allocates an array of impossible size. Compress to [0, K) where K is the number of distinct values.

Pitfall 3 — Overflow in Powerful Array

freq² · value can overflow int well before the array length does. Use long long and write the multiplications as 1LL * ... * ... to force 64-bit arithmetic.

Pitfall 4 — Coordinate-compressing the values needed by the answer

Some answer formulas literally use the original numeric value of a[i] (e.g. Powerful Array multiplies by x). If you compress a[i] to a small index, keep the original value alongside, or skip compression and use a frequency map.

Pitfall 5 — Wrong order of pair-counting in XOR

In add, count pairs first, then bump cntPrefix[x]. In remove, decrement cntPrefix[x] first, then subtract pairs. Doing it the other way pairs each element with itself.

Pitfall 6 — Hilbert order with too-small pow

If 2^pow is smaller than the maximum coordinate, the Hilbert key collapses many queries to the same value and the sort becomes random, losing all locality.

Quick reflex rules

Reach for Mo when you see many range queries (typically 10⁵ to 3·10⁵), a static array or offline-able operations, an answer that depends on frequencies inside the range, easy local add/remove, and merging two halves that looks hard or impossible. Do not use Mo just because the problem has a [l, r]. Ask first whether a simpler tool (prefix sums, segtree, sparse table) handles it.

Common confusion

People sometimes pick Mo because it "always works for range queries". This is wrong. Mo's sqrt factor is the worst useful complexity in the family — slower than segment trees, Fenwick trees, and sparse tables for problems they cover. Pick Mo specifically when those tools cannot maintain the answer, not as a default.

This file covers classic array Mo. The rest of the family lives in three sibling files, each one a controlled mutation of this master template. The variant table tells you exactly what changes and why.

VariantWhat changesWhyWhen to useFile
Mo on Trees Map tree to array via Euler tour. For subtree queries the interval is [tin[v], tout[v]]. For path queries use doubled Euler and a node-toggle instead of an index-add. Subtrees and paths are not natively contiguous; Euler tours linearize them. Subtree counts, path distincts, tree-color queries. 23-mo-on-tree.html
Mo with Updates (3D Mo) Add a time dimension curT. Sort by (block_l, block_r, t). Block size becomes n^(2/3). Point updates between queries must be applied/undone offline. Range queries interleaved with point updates, all offline. 24-mo-with-updates.html
Rollback Mo (with DSU) Replace remove with snapshot+rollback. Per left-block: reset the structure, sweep right forward permanently, snapshot, add left temporarily, answer, rollback. Some data structures (DSU) support add but not delete. Range-of-edges connectivity, monotonic-add answers. 25-mo-other-variants.html
4D Mo Two intervals [l₁, r₁] and [l₂, r₂] moving in tandem. Some problems compare two ranges (e.g. two subtrees). Two-subtree comparisons, multi-range aggregates. 25-mo-other-variants.html
Online block-precomputed Precompute structures for every full block; answer online queries by combining a full-block result with boundary adds. Some problems forbid offline reordering. Online range queries where Mo reordering is not allowed. 25-mo-other-variants.html

Related sqrt-decomposition techniques

Mo belongs to the sqrt-decomposition family, but not every sqrt solution is Mo. Sorted blocks (each block keeps a sorted copy) handle count ≤ x with updates. Block-Fenwick keeps a Fenwick per block for dynamic inversions. 2-D sliding-block plus monotonic queues handles matrix-window aggregates. These appear briefly in the Other Variants file, marked as adjacent techniques and not as "Mo".

🚀 Problem-Based Learning — Classic Array Mo

Each problem here is included because it reveals a different answer-layer pattern. The movement layer is identical to the master template; only add, remove, and the state change.

23.1 — SPOJ DQUERY: distinct values in a range scalar + freq[]

VJudge: https://vjudge.net/problem/SPOJ-DQUERY

Story. Given an array and many queries [l, r], answer how many distinct numbers occur in that interval.

Abstraction. Static offline range query whose answer depends on the multiset of values in the range. Not merge-friendly.

Why Mo fits. Adding or removing one value updates the distinct count in O(1) through a single integer comparison.

Key observation. A value contributes to the distinct count iff its frequency is positive. So distinct changes only at the zero-to-one and one-to-zero transitions.

#include <bits/stdc++.h>
using namespace std;

struct Query { int l, r, id; };
int BLOCK;

bool cmp(const Query& a, const Query& b) {
    int ba = a.l / BLOCK, bb = b.l / BLOCK;
    if (ba != bb) return ba < bb;
    if (ba & 1) return a.r > b.r;
    return a.r < b.r;
}

int main() {
    ios::sync_with_stdio(false); cin.tie(nullptr);

    int n; cin >> n;
    vector<int> a(n), all; all.reserve(n);
    for (int i = 0; i < n; i++) { cin >> a[i]; all.push_back(a[i]); }
    sort(all.begin(), all.end());
    all.erase(unique(all.begin(), all.end()), all.end());
    for (int& x : a) x = lower_bound(all.begin(), all.end(), x) - all.begin();

    int q; cin >> q;
    vector<Query> queries(q);
    for (int i = 0; i < q; i++) {
        int l, r; cin >> l >> r; --l; --r;
        queries[i] = {l, r, i};
    }

    BLOCK = max(1, (int)sqrt(n));
    sort(queries.begin(), queries.end(), cmp);

    vector<int> freq(all.size(), 0), ans(q);
    int distinct = 0, curL = 0, curR = -1;

    auto add = [&](int idx) {
        int x = a[idx];
        if (freq[x] == 0) distinct++;
        freq[x]++;
    };
    auto remove_ = [&](int idx) {
        int x = a[idx];
        freq[x]--;
        if (freq[x] == 0) distinct--;
    };

    for (const Query& qu : queries) {
        while (curL > qu.l) add(--curL);
        while (curR < qu.r) add(++curR);
        while (curL < qu.l) remove_(curL++);
        while (curR > qu.r) remove_(curR--);
        ans[qu.id] = distinct;
    }
    for (int x : ans) cout << x << '\n';
    return 0;
}

Complexity. O((N + Q) · √N).

Pitfalls. SPOJ uses 1-based input. Values can be large, so coordinate-compress to make freq[] small.

23.2 — Codeforces 86D: Powerful Array algebraic delta

VJudge: https://vjudge.net/problem/CodeForces-86D

Story. For each query [l, r], compute the sum over values x of cnt[x]² · x.

Why Mo fits. The answer depends on frequencies, not on a simple linear sum, so prefix sums fail. But adding or removing one occurrence of x changes only its contribution, and the delta has a clean closed form.

Key observation. Adding x with old count c changes the answer by (2c+1)·x; removing changes it by −(2c−1)·x.

#include <bits/stdc++.h>
using namespace std;

struct Query { int l, r, id; };
int BLOCK;

bool cmp(const Query& a, const Query& b) {
    int ba = a.l / BLOCK, bb = b.l / BLOCK;
    if (ba != bb) return ba < bb;
    if (ba & 1) return a.r > b.r;
    return a.r < b.r;
}

int main() {
    ios::sync_with_stdio(false); cin.tie(nullptr);

    int n, q; cin >> n >> q;
    vector<int> a(n);
    int maxValue = 0;
    for (int i = 0; i < n; i++) { cin >> a[i]; maxValue = max(maxValue, a[i]); }

    vector<Query> queries(q);
    for (int i = 0; i < q; i++) {
        int l, r; cin >> l >> r; --l; --r;
        queries[i] = {l, r, i};
    }

    BLOCK = max(1, (int)sqrt(n));
    sort(queries.begin(), queries.end(), cmp);

    vector<long long> ans(q);
    vector<int> freq(maxValue + 1, 0);
    long long curAns = 0;
    int curL = 0, curR = -1;

    auto add = [&](int idx) {
        int x = a[idx];
        curAns += 1LL * (2 * freq[x] + 1) * x;
        freq[x]++;
    };
    auto remove_ = [&](int idx) {
        int x = a[idx];
        curAns -= 1LL * (2 * freq[x] - 1) * x;
        freq[x]--;
    };

    for (const Query& qu : queries) {
        while (curL > qu.l) add(--curL);
        while (curR < qu.r) add(++curR);
        while (curL < qu.l) remove_(curL++);
        while (curR > qu.r) remove_(curR--);
        ans[qu.id] = curAns;
    }
    for (long long x : ans) cout << x << '\n';
    return 0;
}

Complexity. O((N + Q) · √N).

Pitfalls. Use long long for the running total; the value x appears in the formula, so if you compress, keep the original value alongside.

23.3 — Codeforces 220B: Little Elephant and Array contribution flip

VJudge: https://vjudge.net/problem/CodeForces-220B

Story. For each range [l, r], count values x with freq(x) == x.

Why Mo fits. The answer depends on per-value frequencies. Only values x ≤ n can ever satisfy freq(x) == x, because a value cannot appear more often than the array length.

Pattern. Before mutating freq[x], subtract the old contribution; mutate; add the new contribution. This "remove-old, change, add-new" pattern is reusable any time an answer has per-value contribution.

#include <bits/stdc++.h>
using namespace std;

struct Query { int l, r, id; };
int BLOCK;

bool cmp(const Query& a, const Query& b) {
    int ba = a.l / BLOCK, bb = b.l / BLOCK;
    if (ba != bb) return ba < bb;
    if (ba & 1) return a.r > b.r;
    return a.r < b.r;
}

int main() {
    ios::sync_with_stdio(false); cin.tie(nullptr);

    int n, q; cin >> n >> q;
    vector<int> a(n);
    for (int i = 0; i < n; i++) cin >> a[i];

    vector<Query> queries(q);
    for (int i = 0; i < q; i++) {
        int l, r; cin >> l >> r; --l; --r;
        queries[i] = {l, r, i};
    }

    BLOCK = max(1, (int)sqrt(n));
    sort(queries.begin(), queries.end(), cmp);

    vector<int> freq(n + 1, 0), ans(q);
    int good = 0;

    auto remove_contribution = [&](int x) {
        if (1 <= x && x <= n && freq[x] == x) good--;
    };
    auto add_contribution = [&](int x) {
        if (1 <= x && x <= n && freq[x] == x) good++;
    };
    auto add = [&](int idx) {
        int x = a[idx];
        if (1 <= x && x <= n) {
            remove_contribution(x); freq[x]++; add_contribution(x);
        }
    };
    auto remove_ = [&](int idx) {
        int x = a[idx];
        if (1 <= x && x <= n) {
            remove_contribution(x); freq[x]--; add_contribution(x);
        }
    };

    int curL = 0, curR = -1;
    for (const Query& qu : queries) {
        while (curL > qu.l) add(--curL);
        while (curR < qu.r) add(++curR);
        while (curL < qu.l) remove_(curL++);
        while (curR > qu.r) remove_(curR--);
        ans[qu.id] = good;
    }
    for (int x : ans) cout << x << '\n';
    return 0;
}

Pitfalls. Ignore values greater than n early — testing x ≤ n at the top of add avoids a needless array write. Always remove the old contribution before mutating, then add the new contribution.

23.4 — Codeforces 617E: XOR and Favorite Number prefix-XOR index transform

VJudge: https://vjudge.net/problem/CodeForces-617E

Story. For each query [l, r], count subarrays inside that range with XOR equal to k.

Abstraction. Subarray XOR equals a pair of prefix XORs. Original query [l, r] (1-based) becomes prefix-index query [l−1, r].

#include <bits/stdc++.h>
using namespace std;

struct Query { int l, r, id; };
int BLOCK;

bool cmp(const Query& a, const Query& b) {
    int ba = a.l / BLOCK, bb = b.l / BLOCK;
    if (ba != bb) return ba < bb;
    if (ba & 1) return a.r > b.r;
    return a.r < b.r;
}

int main() {
    ios::sync_with_stdio(false); cin.tie(nullptr);

    int n, q, k; cin >> n >> q >> k;
    vector<int> pref(n + 1, 0);
    for (int i = 1; i <= n; i++) {
        int x; cin >> x; pref[i] = pref[i - 1] ^ x;
    }

    vector<Query> queries(q);
    for (int i = 0; i < q; i++) {
        int l, r; cin >> l >> r;
        queries[i] = {l - 1, r, i};
    }

    BLOCK = max(1, (int)sqrt(n + 1));
    sort(queries.begin(), queries.end(), cmp);

    const int MAXX = 1 << 21;
    vector<int> cnt(MAXX, 0);
    vector<long long> ans(q);
    long long curAns = 0;
    int curL = 0, curR = -1;

    auto add = [&](int idx) {
        int x = pref[idx];
        curAns += cnt[x ^ k];   // count pairs before bumping
        cnt[x]++;
    };
    auto remove_ = [&](int idx) {
        int x = pref[idx];
        cnt[x]--;               // decrement first
        curAns -= cnt[x ^ k];
    };

    for (const Query& qu : queries) {
        while (curL > qu.l) add(--curL);
        while (curR < qu.r) add(++curR);
        while (curL < qu.l) remove_(curL++);
        while (curR > qu.r) remove_(curR--);
        ans[qu.id] = curAns;
    }
    for (long long x : ans) cout << x << '\n';
    return 0;
}

Pitfalls. Convert the original query to [l−1, r] over prefix indices; in remove decrement first, then subtract pairs, to avoid pairing the leaving element with itself.

23.5 — Adjacent technique: SPOJ DISTNUM3 / mode of range block over frequencies

Idea. For range mode, maintain freq[value] and a structure valuesByFrequency[f] (or a block decomposition over values) so the maximum frequency can be queried in O(√max_freq). The movement layer is identical to the master template; only the answer layer changes.

This is the natural exercise after distinct count and Powerful Array. It teaches that the answer state can itself be a sqrt structure layered on top of the frequency table, without changing the engine.

📝 Final Summary

Mo's algorithm is an offline range-query framework. Its core idea fits on one line:

Do not answer each range from scratch.
Sort the queries so consecutive ranges are similar.
Move two pointers and maintain the answer incrementally.

The master invariant is that the maintained data structure exactly represents [curL, curR]. The four pointer loops are the entire engine; everything else is answer design.

while (curL > L) add(--curL);
while (curR < R) add(++curR);
while (curL < L) remove_(curL++);
while (curR > R) remove_(curR--);

Use Mo when queries are offline, ranges are static (or updates can be made offline), add and remove are easy, and merging answers is hard. Avoid Mo when prefix sums solve it, when a segment or Fenwick tree handles it online, when the answer has a clean associative merge, when deletion is impossible and rollback is not available, or when constraints or constants are simply too large.

The most important skill is not memorizing the template. It is learning to design the maintained state so that one element entering or leaving the window can update the answer correctly. Once you can do that, the rest of the family — Mo on Trees, Mo with Updates, Rollback Mo, 4D Mo — is a small step away.
