Mo on Trees — Subtree & Path Queries

Euler Tour Linearization · Doubled Euler · LCA Toggle · The Tree Variant of the Master Template

🧩 Problem Definition

We are given a rooted tree on n nodes. Each node carries a value (or color, or weight). Many queries arrive in one of two shapes. A subtree query asks for some aggregate F over all values in the subtree of a given node v. A path query asks for the same kind of aggregate, but over the values on the path between two nodes u and v. The aggregator F is exactly the kind of frequency-based answer that Mo's algorithm handles well on flat arrays — distinct counts, colors with frequency at least k, mode, and so on.

The whole point of this file is that we never invent a new algorithm. We transform the tree into a flat array, reduce every subtree (or path) query to an interval over that array, and then run the same master template from the classic Mo file. The transformation is the only thing that is new.

💡 Intuition from Scratch — Why Direct Mo Fails on Trees

A tree is not an array. Mo's whole engine assumes we can describe the current "active set" by two pointers curL, curR on a flat array and step from one query to the next by adding or removing one element. On a tree there is no obvious flat array to point into.

The naive idea is to recompute each query from scratch with a DFS. For a single subtree query this costs the size of the subtree; for a single path query it costs the path length. With q queries we are back to O(nq) in the worst case.

The right move is to linearize the tree. There exists a well-known DFS traversal — the Euler tour — that turns the tree into an array with two essential properties. First, the subtree of any node corresponds to a contiguous interval in this array. Second, the path between any two nodes can be embedded in a slightly different Euler array (the doubled Euler tour) so that the path's nodes are precisely the indices that appear an odd number of times in some interval. Once the tree becomes an array, Mo is a black box: we sort intervals, move two pointers, and call our problem-specific add/remove hooks.

So the Mo-on-tree algorithm is two layers stacked on top of the master template: an Euler-tour linearization on the bottom, and the same movement + answer layer from classic Mo on top. The only new code in this file builds the Euler array correctly and routes the right interval into Mo for each query.

🎯 Visual Mental Model — The Euler Tour

Imagine running a DFS starting at the root. Every time the DFS enters a node, write the node's label into a growing array. For the path version, we also write the label a second time when the DFS exits. The first behavior gives the preorder Euler array used for subtree queries; the second gives the doubled Euler array used for path queries.

The two important facts you should be able to read off the diagram are these: in the preorder Euler array, the subtree of any node v occupies a single contiguous block of indices, from tin[v] to tout[v] inclusive. In the doubled Euler array, each node appears exactly twice (once on entry, once on exit), and the parity of how many times a node has appeared so far determines whether the node is "currently active" on the implicit path.

🧠 Formal Abstraction — Tree to Array Embedding

The Euler tour is produced by a simple DFS that maintains a counter, called timer. Each time we enter a node, we record tin[u] = timer and write u into euler[timer], then bump the timer. For preorder, that is all we do. For the doubled Euler tour, we also write u a second time on the way out and record tout[u]. The Euler array therefore has length n in preorder, or length 2n in doubled form.

Subtree-Mo lifts this to a Mo problem directly. Map the query "aggregate over the subtree of v" to the array query "aggregate over euler[tin[v] .. tout[v]]". Then plug into the master template with the same add(idx) = "include the value at euler[idx] into the maintained state" and remove(idx) = "remove the value at euler[idx] from the maintained state".

Path-Mo needs one extra layer of indirection: a toggle. Because each node appears twice in the doubled Euler array, we cannot just call add for every index entering the interval. Instead, we maintain an active[u] bit per node. Whenever the pointer movement causes node u to be touched, we flip the bit: if it was inactive, we activate it (add_node(u)); if it was active, we deactivate it (remove_node(u)). The intuition is that the second visit to the same node cancels the first.

🔭 Key Observations

Observation 1 — Subtree intervals exist for free

In preorder, the descendants of any node u are visited consecutively right after u. That is exactly what makes [tin[u], tout[u]] a valid contiguous interval. No fancy structure is needed beyond tin and tout arrays.

Observation 2 — Path intervals need both endpoints of doubled Euler

Without loss of generality, assume tin[u] ≤ tin[v] (swap otherwise), and let w = LCA(u, v). There are two cases. If w == u, then u is an ancestor of v, and the path nodes appear with odd count in the doubled-Euler interval [tin[u], tin[v]]. If w ≠ u, then u and v are in separate subtrees of w, and the doubled-Euler interval to use is [tout[u], tin[v]]. In this second case, w itself is not in that interval, so we must add it as a temporary node when answering the query and remove it afterwards.

Observation 3 — Toggle invariant on the doubled Euler tour

In the doubled Euler interval, a node u is active (i.e., belongs to the implicit current path-set) iff u appears an odd number of times in that interval. Each pointer move touches exactly one cell; that cell carries some node label u; we flip active[u] and call add_node or remove_node depending on the direction of the flip.

Observation 4 — Block size scales with the Euler array, not with the tree size

For subtree-Mo, the Euler array has length n, so the natural block size is B = √n. For path-Mo, the doubled Euler array has length 2n, so the block size used in the comparator should be B = √(2n). This is a small detail that matters in practice.

🌿 Subtree Algorithm — Preorder Euler

The subtree case is almost identical to classic array Mo. We replace the array a[] with the preorder Euler array euler[], build the tin/tout tables once, and rewrite every subtree query as the interval [tin[v], tout[v]]. From that moment on, the master template runs unchanged.

void dfs(int u, int p) {
    tin[u] = timer;
    euler[timer] = u;
    timer++;

    for (int v : adj[u]) {
        if (v != p) dfs(v, u);
    }

    tout[u] = timer - 1;   // last index of the subtree of u
}

// subtree(u) == euler[tin[u] .. tout[u]]

The add(idx) hook takes the index idx in the Euler array, fetches euler[idx] (a node), reads its color/value, and bumps the maintained state. remove(idx) is the exact inverse. The pointer-movement layer is identical to classic Mo.

📄 Subtree Master Template

This template is structurally identical to the classic Mo master template. The only differences are the DFS preprocessing block, the indirection through euler[], and the query construction. Everything inside solve_mo is reused verbatim.

#include <bits/stdc++.h>
using namespace std;

struct Query { int l, r, id; /* plus problem-specific fields */ };
int BLOCK;

bool mo_cmp(const Query& a, const Query& b) {
    int ba = a.l / BLOCK, bb = b.l / BLOCK;
    if (ba != bb) return ba < bb;
    if (ba & 1) return a.r > b.r;
    return a.r < b.r;
}

int n;
vector<vector<int>> adj;
vector<int> tin, tout_, euler, val;   // val[u] = value/color of u
int timer_ = 0;

void dfs(int u, int p) {
    tin[u] = timer_;
    euler[timer_] = u;
    timer_++;
    for (int v : adj[u]) if (v != p) dfs(v, u);
    tout_[u] = timer_ - 1;
}

// =====================================================
// Problem-specific state (replace per problem)
// =====================================================
vector<int> freq;
long long current_answer = 0;

void add_index(int idx) {
    int node = euler[idx];
    int c = val[node];
    // update freq[c] and current_answer
}

void remove_index(int idx) {
    int node = euler[idx];
    int c = val[node];
    // inverse of add_index
}

long long get_answer(const Query& q) { return current_answer; }

// =====================================================
// Generic Mo driver (unchanged from classic Mo)
// =====================================================
vector<long long> solve_mo(vector<Query> queries) {
    BLOCK = max(1, (int)sqrt(n));
    sort(queries.begin(), queries.end(), mo_cmp);
    vector<long long> ans(queries.size());

    int curL = 0, curR = -1;
    for (const Query& q : queries) {
        while (curL > q.l) add_index(--curL);
        while (curR < q.r) add_index(++curR);
        while (curL < q.l) remove_index(curL++);
        while (curR > q.r) remove_index(curR--);
        ans[q.id] = get_answer(q);
    }
    return ans;
}

// Then for each subtree query on node v: push {tin[v], tout_[v], id}.

🛤 Path Algorithm — Doubled Euler + LCA

Path queries are the more delicate case because a path between two arbitrary nodes is not contiguous in the simple DFS preorder. The trick is to build a doubled Euler tour: write each node into the array twice, once on the way in (entry) and once on the way out (exit). The resulting array has length 2n, and every node appears exactly twice. The key combinatorial fact, made precise in the toggle invariant below, is that the set of nodes appearing an odd number of times inside any sub-interval of this array equals the set of nodes on a specific tree-path. Choosing the right interval for the query gives exactly the path from u to v.

Building the doubled Euler tour

void dfs(int u, int p) {
    tin[u] = timer_;
    euler[timer_++] = u;           // entry

    up[0][u] = p;
    for (int j = 1; j < LOG; j++) up[j][u] = up[j-1][up[j-1][u]];

    for (int v : adj[u]) {
        if (v == p) continue;
        depth[v] = depth[u] + 1;
        dfs(v, u);
    }

    tout_[u] = timer_;
    euler[timer_++] = u;           // exit
}

Notice that tin[u] stores the position of the entry occurrence and tout[u] stores the position of the exit occurrence. Both are needed when building each query's interval.

Picking the interval for a query

Without loss of generality, assume tin[u] ≤ tin[v] (swap them otherwise). Compute w = LCA(u, v) with binary lifting. Two cases follow.

Case 1, ancestor case: if w == u, then u is an ancestor of v. The interval is [tin[u], tin[v]]; no extra LCA correction is needed because u already appears with odd count inside this interval (entry only).

Case 2, separate-subtrees case: if w ≠ u, then the interval is [tout[u], tin[v]], and w is not in that interval. Before reading the answer we toggle w in (so it counts), and after reading we toggle it back out. We record lca = w in the second case and lca = -1 in the first case so the main loop knows whether to apply the temporary toggle.

🔁 The Toggle Invariant

The central piece of theory in path-Mo lives in a single sentence, which is worth memorizing.

A node u is active in the current Mo state if and only if u appears an odd number of times in the doubled-Euler interval [curL, curR].

This invariant is what makes the toggle trick legal. Each time a pointer moves over a cell holding node u, the parity of u's occurrences inside the interval flips. We mirror that flip in our active[u] bit and trigger exactly one add_node(u) or one remove_node(u).

bool active[N];

void toggle_node(int u) {
    if (!active[u]) add_node(u);
    else            remove_node(u);
    active[u] = !active[u];
}

Why "appears odd times" matches "on the path"

Consider what happens during the DFS for a single node u. Between tin[u] and tout[u] (inclusive), the DFS visits exactly the subtree of u. Any node x in that subtree contributes both its entry and its exit inside that interval, an even count, and therefore is not considered active. Any node x on the path from u downward to v (when u is an ancestor of v) has its entry inside the interval but its exit outside, an odd count, and is therefore active. When u and v are in separate subtrees of their LCA w, a similar accounting on the interval [tout[u], tin[v]] picks up exactly the union of the two root-to-node paths, minus w itself — which is why we add w back temporarily.

📄 Path Master Template — Distinct Colors on a Tree Path

This is the SPOJ COT2 accepted template, taken directly from the source corpus, reformatted to match the family naming. The only thing that changes when you adapt it to another path problem is the add_node / remove_node pair and the distinct scalar.

#include <bits/stdc++.h>
using namespace std;

struct Query { int l, r, id, lca; };
int BLOCK;

bool cmp(const Query& a, const Query& b) {
    int ba = a.l / BLOCK, bb = b.l / BLOCK;
    if (ba != bb) return ba < bb;
    if (ba & 1) return a.r > b.r;
    return a.r < b.r;
}

int main() {
    ios::sync_with_stdio(false); cin.tie(nullptr);

    int n, q; cin >> n >> q;
    vector<int> val(n + 1), all;
    for (int i = 1; i <= n; i++) { cin >> val[i]; all.push_back(val[i]); }
    sort(all.begin(), all.end());
    all.erase(unique(all.begin(), all.end()), all.end());
    for (int i = 1; i <= n; i++)
        val[i] = lower_bound(all.begin(), all.end(), val[i]) - all.begin();

    vector<vector<int>> adj(n + 1);
    for (int i = 0; i < n - 1; i++) {
        int u, v; cin >> u >> v;
        adj[u].push_back(v); adj[v].push_back(u);
    }

    int LOG = 1; while ((1 << LOG) <= n) LOG++;
    vector<vector<int>> up(LOG, vector<int>(n + 1, 0));
    vector<int> depth(n + 1, 0), tin(n + 1), tout_(n + 1), euler(2 * n);
    int timer_ = 0;

    function<void(int,int)> dfs = [&](int u, int p) {
        tin[u] = timer_; euler[timer_++] = u;
        up[0][u] = p;
        for (int j = 1; j < LOG; j++) up[j][u] = up[j - 1][up[j - 1][u]];
        for (int v : adj[u]) {
            if (v == p) continue;
            depth[v] = depth[u] + 1; dfs(v, u);
        }
        tout_[u] = timer_; euler[timer_++] = u;
    };
    dfs(1, 1);

    auto lca = [&](int a, int b) {
        if (depth[a] < depth[b]) swap(a, b);
        int diff = depth[a] - depth[b];
        for (int j = 0; j < LOG; j++)
            if (diff & (1 << j)) a = up[j][a];
        if (a == b) return a;
        for (int j = LOG - 1; j >= 0; j--)
            if (up[j][a] != up[j][b]) { a = up[j][a]; b = up[j][b]; }
        return up[0][a];
    };

    vector<Query> queries(q);
    for (int i = 0; i < q; i++) {
        int u, v; cin >> u >> v;
        if (tin[u] > tin[v]) swap(u, v);
        int w = lca(u, v);
        if (w == u) queries[i] = {tin[u],  tin[v], i, -1};
        else        queries[i] = {tout_[u], tin[v], i, w};
    }

    BLOCK = max(1, (int)sqrt(2 * n));
    sort(queries.begin(), queries.end(), cmp);

    vector<int> freq(all.size(), 0);
    vector<char> active(n + 1, false);
    vector<int> ans(q);
    int distinct = 0;

    auto add_node = [&](int u) {
        int x = val[u];
        if (freq[x] == 0) distinct++;
        freq[x]++;
    };
    auto remove_node = [&](int u) {
        int x = val[u];
        freq[x]--;
        if (freq[x] == 0) distinct--;
    };
    auto toggle = [&](int u) {
        if (active[u]) remove_node(u);
        else add_node(u);
        active[u] = !active[u];
    };

    int curL = 0, curR = -1;
    for (const Query& qu : queries) {
        while (curL > qu.l) toggle(euler[--curL]);
        while (curR < qu.r) toggle(euler[++curR]);
        while (curL < qu.l) toggle(euler[curL++]);
        while (curR > qu.r) toggle(euler[curR--]);

        if (qu.lca != -1) toggle(qu.lca);   // include LCA temporarily
        ans[qu.id] = distinct;
        if (qu.lca != -1) toggle(qu.lca);   // and undo
    }

    for (int x : ans) cout << x << '\n';
    return 0;
}

🔒 Invariants

State invariant (subtree case)

At every moment during the main loop, the maintained state represents exactly the multiset of values { val[euler[i]] : curL ≤ i ≤ curR }. This is the direct lift of the classic Mo state invariant to a preorder Euler array.

State invariant (path case)

At every moment during the main loop, the maintained state represents exactly the multiset of values of active nodes, where active[u] = true iff u appears an odd number of times in euler[curL..curR]. This is the toggle invariant.

Transition invariant

Each pointer movement touches one cell and triggers exactly one toggle on the node label in that cell. The toggle then routes to either add_node or remove_node; in either case, exactly one state mutation happens per pointer step.

Query invariant

Immediately before storing ans[q.id], the interval matches the query's stored [l, r]. If q.lca != -1, the LCA has been toggled in once before reading and toggled out once after, so the temporary correction is balanced and does not corrupt subsequent queries.

📜 Proof of Correctness — Path Case

Lemma A — Parity argument on the doubled Euler tour

Fix a node x. Inside any sub-interval [L, R] of the doubled-Euler array, x appears either zero, one, or two times. It appears zero times iff x's entry and exit are both outside [L, R], equivalently tout[x] < L or tin[x] > R. It appears twice iff both tin[x] and tout[x] lie inside [L, R]; in that case the subtree of x is wholly inside the interval, which means x is not on the path being queried. It appears exactly once otherwise — which is the only case where x sits on the boundary of containment, i.e., on the path.

Lemma B — Ancestor case

If u is an ancestor of v, the interval [tin[u], tin[v]] contains tin[x] for every x on the path from u down to v, but contains tout[x] only for nodes strictly inside the subtree of x that are visited completely before tin[v]. A careful case analysis shows that for nodes on the path, the entry is inside and the exit is outside, giving odd count, and for nodes not on the path either both occurrences are inside (subtrees of off-path nodes visited fully before v) or both are outside.

Lemma C — Separate-subtrees case

If w = LCA(u, v) ≠ u, the interval [tout[u], tin[v]] spans the gap between leaving u's subtree and entering v's subtree. The nodes on the path are partitioned into (i) u itself, whose exit is inside and entry is outside (odd, included), (ii) ancestors of u strictly below w, whose entries and exits straddle the interval (odd, included), (iii) w itself, whose entry is strictly before tout[u] and whose exit is strictly after tin[v], giving an even count of zero inside the interval — therefore w is missing and must be toggled in, then out, around the answer read.

Theorem

Combining Lemmas A, B, C with the transition invariant from Mo, the maintained state at each query exactly represents the multiset of values of the path nodes. Hence every stored answer is correct.

⏱ Complexity Analysis

For subtree-Mo, the Euler array has length n, so the complexity is O((n + q) · √n · F), where F is the cost of one add_index or remove_index. The "375D"-style problem additionally pays a log n factor for Fenwick lookups on frequency-of-frequency tables.

For path-Mo, the doubled Euler array has length 2n, and each pointer step is O(F). The Mo bound becomes O((n + q) · √n · F). LCA queries are O(log n) each via binary lifting, paid only once per query during input reading, so they add O(q · log n) to the running total — usually dwarfed by the Mo factor unless F is itself O(1).

The block-size choice is B = max(1, ⌊√(2n)⌋) for path-Mo, because the comparator sorts indices into the doubled-Euler array. Using √n instead works but is slightly suboptimal.

🔎 Line-by-Line — Path Code Dissected

Reading the values and coordinate-compressing

The values on the nodes can be huge; we compress them into a contiguous range starting from zero so that freq[] can be a flat vector of size equal to the number of distinct values, rather than the value range. The all vector is sorted, deduplicated, and then each val[i] is replaced by its position in all.

Building the doubled Euler array and binary-lifting table together

The DFS writes the node both on entry and on exit. The same DFS computes up[j][u], the 2^j-th ancestor of u, used by the LCA function. Recording tin and tout in the global timer also gives us a constant-time test for "is a an ancestor of b?" via the interval-containment check tin[a] ≤ tin[b] && tout[a] ≥ tout[b], although the binary-lifting LCA is the more general tool used in this template.

Choosing the query interval

After ensuring tin[u] ≤ tin[v], the code computes w = lca(u, v). If w == u, the query interval is [tin[u], tin[v]] and no LCA correction is needed. Otherwise, the query interval is [tout[u], tin[v]] and w is stored so it can be toggled in temporarily.

The toggle wrapper

toggle(u) consults active[u] and dispatches to either add_node or remove_node, then flips active[u]. This is the single point of indirection that makes the doubled-Euler array work with the classic Mo movement loop.

The four-loop engine — unchanged from classic Mo

The four loops are syntactically identical to the classic Mo file. The only difference is that each step calls toggle(euler[idx]) instead of add_index(idx)/remove_index(idx). The pre/post-increment ordering is the same as before.

The LCA correction sandwich

If qu.lca != -1, the code toggles the LCA in before reading distinct, then toggles it out again. This pair of calls is symmetric and balanced, so the active set is exactly the same after the read as it was before the read — which is critical for the next query to start from a correct state.

🪜 Centralized Edge-Case Audit

The classic-Mo audit applies in full, plus the additional checks below that are specific to running Mo on top of a tree linearization.

#CheckWhy it matters
1Is the tree 1-indexed in input but you're using 0-indexed Euler positions internally?Off-by-one bugs in the tin/tout tables show up as queries returning answers for the wrong subtree.
2Did you size the Euler array to n for subtree, or 2n for path?Mixing the two corrupts the very last few entries silently; the longest test cases will fail.
3For path-Mo, did you ensure tin[u] ≤ tin[v] before picking the interval?The interval-selection rule is asymmetric in u and v; swapping them is mandatory.
4For path-Mo, did you toggle the LCA both times (in and out)?Forgetting the second toggle pollutes the active set for the next query, breaking everything downstream.
5Is the block size based on the Euler length, not the tree size?B = √(2n) for path-Mo; B = √n for subtree-Mo.
6Are values coordinate-compressed if they can be large?Without compression, freq[val[u]] can allocate an unworkable array.
7Does the binary-lifting LOG cover the deepest node?If 2^LOG ≤ n, the LCA function returns wrong answers for deep nodes; pick LOG so 2^LOG > n.
8Is the DFS iterative or recursive deep enough?Recursive DFS on a tree of 200k nodes can blow the stack. Use an iterative DFS or raise the stack limit explicitly.
9Are subtree-Mo queries sorted by tin[v] blocks, not by v?Mo sorts by interval endpoints, not by node IDs. Storing {tin[v], tout[v], …} at read-time keeps the comparator pure.
10For subtree-Mo with frequency-of-frequency, did you avoid storing freqCount[0]?freqCount[0] is the number of "missing" values; it's a Fenwick noise source if you index it.
11If the problem has values on edges rather than nodes, did you fold edge values onto the deeper endpoint?Trees with edge values can be reduced to trees with node values by pushing each edge's value to its child; this lets you reuse the same template with only the LCA's value adjustment.

⚠️ Common Pitfalls

Pitfall 1 — Using the wrong interval in path-Mo

The two cases w == u and w ≠ u require different intervals. Mixing them up leads to mostly-correct answers that fail on the trickiest tests. Always normalize by swapping so that tin[u] ≤ tin[v], then check the LCA case.

Pitfall 2 — Forgetting the LCA correction

Even after picking the right interval, the separate-subtrees case excludes the LCA. The temporary toggle around the answer read is non-optional. Without it, you produce wrong answers exactly on the queries whose LCA is neither u nor v.

Pitfall 3 — Forgetting the second toggle of the LCA

The LCA must be toggled out after reading the answer. If you forget, the next query starts with the LCA still active, which corrupts every subsequent answer. The bug is easy to introduce when refactoring and very hard to find without targeted tests.

Pitfall 4 — Re-entering a node through toggle in subtree-Mo

Subtree-Mo does not use the toggle trick — each node appears exactly once in the preorder Euler array. If you accidentally borrow the path-Mo template for a subtree problem and keep the active[] bit, you will under-count nodes because the bit will get stuck. Use add_index/remove_index directly for subtree problems.

Pitfall 5 — Confusing entry-time and exit-time when storing intervals

Subtree-Mo uses [tin[v], tout[v]]. Path-Mo case 1 uses [tin[u], tin[v]]. Path-Mo case 2 uses [tout[u], tin[v]]. Three different rules, all using mixtures of tin and tout. Write them in one place and never inline them.

Pitfall 6 — Stack overflow on recursive DFS

For n = 2·10⁵ a chain-shaped tree has DFS depth 200000. Many judges' default stack size handles 10k but not 200k. Solutions either rewrite the DFS iteratively with an explicit stack, or pre-allocate a larger stack at process start.

🧭 Decision Guide — Mo on Tree vs Alternatives

SituationPrefer
Subtree frequency-style queries (e.g., colors with frequency ≥ k)subtree-Mo via preorder Euler; this file.
Path frequency-style queries (e.g., distinct colors on a path)path-Mo via doubled Euler + LCA toggle; this file.
Subtree queries where the answer has a clean mergesmall-to-large / sack / DSU-on-tree, often O(n log n).
Single-element subtree sums or countsEuler + Fenwick tree, online and faster.
Path sums or path XORLCA + prefix-sum on root paths, O(log n) per query.
Queries with updates on treeHLD + segtree, or 3D Mo on a flattened tree (advanced).
The answer combines well for subtrees but not for pathsDSU-on-tree (small-to-large) is usually preferable for subtree problems with merge.

Reflex rules

For subtree queries, ask first: "does this problem have a merge that DSU-on-tree could handle in O(n log² n) or better?" If yes, prefer DSU-on-tree; it is usually faster and has smaller constants. Pick subtree-Mo when the answer fundamentally needs a frequency table that is hard to merge — counts of colors with frequency at least k being the canonical example.

For path queries, ask first: "does this aggregate decompose as F(u, v) = F(root, u) + F(root, v) − F(root, LCA) − F(root, parent(LCA))?" If yes, prefix-on-root is online and O(log n) per query. Pick path-Mo only when the aggregate has the frequency-based, non-mergeable flavor — distinct colors, mode, threshold frequencies.

🌿 Variants & Extensions

Edge values instead of node values

If the problem assigns values to edges instead of nodes, "fold" each edge's value onto its child endpoint. Then the path from u to v in the new node-valued tree contains exactly the edge values of the original path, plus the value that was folded onto the LCA. Since the LCA itself is on both sides of the boundary, we adjust by not toggling the LCA (i.e., always treat qu.lca = -1 in the answer-read step, but be careful that u and v themselves are treated correctly — usually meaning the values of u and v are their incoming edges, so they automatically participate).

Why it fits: the bijection between original-edge sets and "edges below a node" sets is exact, so the toggle invariant transfers.

Algorithm choice: reuse the path-Mo template; change only the value-folding step at input time.

Problem link suggestions: classical problems on tree edge weights with frequency-based answers; the SPOJ COT2 setup adapts trivially.

Mixed subtree + path queries

If the same problem has both subtree and path queries, prefer two separate solvers in the same offline pass. Both use Euler tours, but the path solver doubles the Euler array. Mixing them in a single Mo over the doubled Euler is possible if and only if the subtree intervals can be re-expressed in the doubled Euler — and they can, using [tin[v], tout[v] − 1] with care — but the simpler choice in contests is to run two passes.

Block-precomputed online tree-Mo

The online variant precomputes structures for full Euler blocks and answers each query by combining a full-block answer with boundary toggles. It is uncommon in contests but appears in the extracted source files (covered in the Other Variants file). Use only when the problem forbids offline reordering.

23.5 — Codeforces 375D: Tree and Queries subtree · color frequency ≥ k

VJudge: https://vjudge.net/problem/CodeForces-375D

Story. Each node has a color. For each query (v, k), count how many colors appear at least k times in the subtree of v.

Abstraction. The subtree of v becomes the interval [tin[v], tout[v]] in the preorder Euler array. Within that interval we maintain a frequency-of-frequency table: colorFreq[c] tells how often color c appears, and a Fenwick tree over frequencies supports the "sum of freqCount[f] for f ≥ k" suffix query.

Why Mo fits. The answer needs frequencies of frequencies — there is no clean associative merge — and add/remove of one Euler position changes exactly one color frequency.

Algorithm choice. Subtree-Mo + Fenwick over frequencies. The change_color_frequency helper performs the standard "remove old contribution, mutate, add new contribution" pattern.

#include <bits/stdc++.h>
using namespace std;

struct Query { int l, r, k, id; };

struct Fenwick {
    int n; vector<int> bit;
    Fenwick(int n = 0) { init(n); }
    void init(int n_) { n = n_; bit.assign(n + 1, 0); }
    void add(int idx, int val) {
        for (; idx <= n; idx += idx & -idx) bit[idx] += val;
    }
    int sumPrefix(int idx) const {
        int res = 0;
        for (; idx > 0; idx -= idx & -idx) res += bit[idx];
        return res;
    }
    int sumRange(int l, int r) const {
        if (l > r) return 0;
        return sumPrefix(r) - sumPrefix(l - 1);
    }
};

int BLOCK;
bool cmp(const Query& a, const Query& b) {
    int ba = a.l / BLOCK, bb = b.l / BLOCK;
    if (ba != bb) return ba < bb;
    if (ba & 1) return a.r > b.r;
    return a.r < b.r;
}

int main() {
    ios::sync_with_stdio(false); cin.tie(nullptr);

    int n, q; cin >> n >> q;
    vector<int> color(n + 1);
    for (int i = 1; i <= n; i++) cin >> color[i];

    vector<vector<int>> adj(n + 1);
    for (int i = 0; i < n - 1; i++) {
        int u, v; cin >> u >> v;
        adj[u].push_back(v); adj[v].push_back(u);
    }

    vector<int> tin(n + 1), tout_(n + 1), euler(n);
    int timer_ = 0;
    function<void(int,int)> dfs = [&](int u, int p) {
        tin[u] = timer_; euler[timer_++] = u;
        for (int v : adj[u]) if (v != p) dfs(v, u);
        tout_[u] = timer_ - 1;
    };
    dfs(1, 0);

    vector<Query> queries(q);
    for (int i = 0; i < q; i++) {
        int v, k; cin >> v >> k;
        queries[i] = {tin[v], tout_[v], k, i};
    }

    BLOCK = max(1, (int)sqrt(n));
    sort(queries.begin(), queries.end(), cmp);

    int maxColor = *max_element(color.begin() + 1, color.end());
    vector<int> colorFreq(maxColor + 1, 0);
    Fenwick fw(n + 2);
    vector<int> ans(q);
    int curL = 0, curR = -1;

    auto change_color_frequency = [&](int c, int delta) {
        int oldFreq = colorFreq[c];
        if (oldFreq > 0) fw.add(oldFreq, -1);
        colorFreq[c] += delta;
        int newFreq = colorFreq[c];
        if (newFreq > 0) fw.add(newFreq, +1);
    };
    auto add = [&](int idx) {
        int node = euler[idx];
        change_color_frequency(color[node], +1);
    };
    auto remove_ = [&](int idx) {
        int node = euler[idx];
        change_color_frequency(color[node], -1);
    };

    for (const Query& qu : queries) {
        while (curL > qu.l) add(--curL);
        while (curR < qu.r) add(++curR);
        while (curL < qu.l) remove_(curL++);
        while (curR > qu.r) remove_(curR--);

        if (qu.k > n) ans[qu.id] = 0;
        else ans[qu.id] = fw.sumRange(qu.k, n);
    }
    for (int x : ans) cout << x << '\n';
    return 0;
}

Complexity. O((N + Q) · √N · log N) — the log comes from the Fenwick updates and the suffix query.

Pitfalls. Do not let freqCount[0] enter the Fenwick. If color values are huge, coordinate-compress; otherwise colorFreq can be indexed directly by raw color.

23.6 — SPOJ COT2: Count on a Tree II path · distinct values

VJudge: https://vjudge.net/problem/SPOJ-COT2

Story. Each node has a value. For each query (u, v), count the number of distinct values on the path from u to v.

Abstraction. Build the doubled Euler tour. Compute each query's interval via the LCA rules, store the LCA only in the separate-subtrees case, and route to the Mo template. The state is the same distinct-count state from the classic file, but updated through toggle rather than direct add/remove.

Why Mo fits. Distinct count is the canonical example of a frequency-based answer with no associative merge, and the path can be embedded into a 1-D doubled-Euler interval.

Algorithm choice. Path-Mo with binary-lifting LCA and the toggle wrapper. The full template code in the Path Master Template section above is the accepted solution to this problem.

Complexity. O((N + Q) · √N · log N). The Mo movement is O(1) per step; the log factor comes from preprocessing-time LCA queries.

Pitfalls. Use a doubled Euler array of size 2N. Toggle the LCA exactly once when needed (case 2), and exactly once again to undo it after reading. Sort queries by l-block of size √(2N).

📝 Final Summary

Mo on trees is the master template plus an Euler-tour translation. For subtrees, the preorder Euler array makes each subtree a contiguous interval and the classic template applies unchanged. For paths, the doubled Euler array plus the toggle invariant turns each path into a sub-interval, with the LCA correction handled by a balanced pair of toggles around the answer read.

The single new theorem worth carrying in your head:

A node is on the current Mo path iff it appears
an odd number of times in the doubled-Euler interval.

Everything else — the four-loop engine, the comparator, the answer-state design — is the same as classic Mo. Continue with 24-mo-with-updates.html for the time-dimension extension and 25-mo-other-variants.html for rollback and 4D Mo.
