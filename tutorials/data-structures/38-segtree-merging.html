Lesson 9 — Segment Tree Merging

Combine Two Sparse Trees · Small-to-Large · DSU-on-Tree Backbone

🧩 Problem Definition

Imagine a rooted tree of N vertices. Each vertex carries a color (a value in [0, U)). For every vertex v, you want some answer that depends on the multiset of colors in the subtree rooted at v: the most frequent color, the sum of distinct colors, the number of colors appearing exactly k times, and so on.

A natural plan: at each vertex maintain a frequency segment tree over [0, U), where leaf c stores how many times color c appears in the subtree. A point query on this segment tree answers any “histogram” question. The challenge is building all N of these segment trees efficiently — a naive copy at each vertex costs O(N²).

Segment tree merging solves the problem by reusing nodes. The segment tree at vertex v is built by merging the segment trees at v's children, with the cost of the merge equal to the number of shared-overlap nodes, not the sum of the sizes. Total work across the whole tree turns out to be O(N log U).

💡 Intuition from Scratch

Take two dynamic sparse segment trees A and B, each representing a multiset of values. We want to merge them into one tree (call it A after the merge) that represents A ∪ B as a multiset. The structure of segment trees gives us a free divide-and-conquer for this: at the root, three things can happen.

If B is null (a ghost), we are done — A already represents the union. If A is null, we steal B entirely: A := B. Otherwise, both are real and we recurse into the corresponding left children, then into the right children, and finally we re-pull the aggregate (e.g. sum) at the current node.

The crucial cost insight: the recursion only enters a sub-call when both children are non-null — and the total such pairs across all levels is bounded by the number of nodes that actually exist in both trees. Each node in either tree is “consumed” at most once across all merges done over the lifetime of the algorithm, which is what gives us the global O(N log U) bound.

🌳 Visual Mental Model

Picture two trees drawn side-by-side over the same coordinate range. Draw a vertical line of correspondence between them — left of A's root corresponds to left of B's root, and so on. The merge walks both trees in lockstep down the correspondence line. At every position there are three cases:

Case (a): A child exists, B child is ghost. → keep A child unchanged, no recursion. Case (b): A child is ghost, B child exists. → A child := B child ("steal"), no recursion. Case (c): Both children exist. → recurse, then pull(A current).

Cases (a) and (b) cost O(1). Only Case (c) recurses, and only into both children. So the total cost of a single merge is proportional to the number of internal nodes where Case (c) fires — equivalently the size of the “intersection skeleton” of the two trees.

📐 Formal Abstraction

The merge function operates on the same dynamic sparse segment tree from Lesson 7: nodes live in a shared pool with integer child pointers, -1 for ghost. The signature is:

int merge(int a, int b, long long lo, long long hi); // returns the index of the merged tree's root.

The return value matters because merge reuses nodes destructively: if both a and b are non-null, we mutate a in place to become the merged root, and we orphan b (its nodes are no longer reachable from any live root). If only one of the two is non-null, we return whichever one is — no allocation needed.

Because the same pool is shared, the result is automatically a valid dynamic sparse tree that the rest of the program can query and update.

🔬 Key Observations

Observation 1 (per-merge cost). The work of a single merge(A, B) is proportional to the number of nodes u such that both A and B have a node at u's interval position. Call these the overlap nodes.

Observation 2 (global amortized bound). If we merge all sub-tree segment trees together up the rooted tree (each node merged exactly once into its parent's collection), then every node in any of the original segment trees participates as an overlap at most as many times as the tree depth — so the total work across the whole algorithm is O(total leaf insertions · log U), which is O(N log U) for N insertions.

Observation 3 (destructive but safe). Merging destroys the input trees by repurposing their nodes for the output. If you need to keep the input, you must use the persistent variant (Variant 1 below), which costs more memory.

⚙️ Algorithm

merge(a, b, lo, hi)

Standard divide-and-conquer over the coordinate range. Base cases handle the three case-(a)/(b)/(c) options:

  1. If a == -1, return b.
  2. If b == -1, return a.
  3. If lo == hi (leaf), combine pool[a].sum += pool[b].sum; return a.
  4. Otherwise, recurse on left children, recurse on right children, then call pull(a); return a.

tree-aggregation pattern (typical usage)

Given a rooted tree on N vertices, build a leaf segment tree at each vertex by inserting that vertex's color. Then do a post-order traversal: at each internal vertex v, repeatedly merge the segment tree of each child c into v's segment tree using seg[v] = merge(seg[v], seg[c], 0, U-1). After all children have been folded in, seg[v] represents the multiset of colors in the subtree of v.

🛡️ Invariants

✅ Invariant — Sum invariant

Immediately after merge(a, b, lo, hi) returns, the node it returns is a valid sparse-tree node whose sum equals the sum of the two input multisets over the interval [lo, hi].

✅ Invariant — Disjointness invariant

After the merge, the nodes originally rooted at b are orphaned — no live segment-tree root in the program reaches them. They become dead memory in the pool. This is harmless but is the reason segment-tree merging is one-shot: you cannot merge B into two different trees.

✅ Invariant — Reuse invariant

Each pool node is consumed (i.e., participates as the <both children present> case) in at most one merge across the lifetime of the algorithm. This is the key fact that makes the global running time linear in the number of inserted points.

🧮 Proof of Correctness

🧮 Proof — Correctness of the recursive merge

Setup. Both inputs are valid dynamic sparse segment trees for the interval [lo, hi]. Each maintains its own sum invariant: every node's stored sum equals the sum of its interval's leaves.

Case A null. The conceptual multiset of A is empty over [lo, hi], so the union equals B's multiset. Returning b directly is correct.

Case B null. Symmetric.

Leaf case. Both a and b are leaves at the same coordinate lo. The union has pool[a].sum + pool[b].sum at this coordinate, exactly what the merge writes.

Recursive case. By induction, the recursive calls on left and right return valid merged subtrees. The pull(a) at the end recomputes pool[a].sum from the (possibly new) children, so the sum invariant is restored. ∎

🧮 Proof — Global O(N log U) bound

Define a “token” for every pool node ever created. Each token can be consumed at most once: a node is consumed when merge reaches it from both inputs simultaneously (the Case-c “both present” line). Once consumed, one of the two duplicates is unreachable forever.

If we insert N total leaves over the whole algorithm, the pool contains at most N · ⌈log₂ U⌉ + 1 nodes (each insertion materializes one path). The total work across all merges is proportional to the number of token consumptions, which is at most the total node count. Hence total merge work is O(N log U). ∎

Despite individual merges potentially being expensive (e.g. when the two trees are nearly identical, an entire log-depth path is traversed), the global cost telescopes. This is similar in spirit to small-to-large with std::set::merge, but the segment-tree version supports more complex queries on the merged multiset (range counts, kth-smallest, etc.).

🧰 Core Template Philosophy

Segment tree merging is a small merge function appended to the dynamic sparse segment tree from Lesson 7. There is no new data structure; only a new operation. The discipline that makes it efficient is “reuse, don't allocate”: whenever one of the two inputs is a ghost, return the other's index unchanged. Allocations happen only during the initial leaf-insertion phase.

Persistence — i.e., the desire to keep both inputs intact — is a separate, more expensive option (Variant 1). If you don't need it, don't pay for it.

💻 Master Template (C++)

#include <bits/stdc++.h>
using namespace std;

struct DynamicSparseSumSegTree {
    using T = long long;
    using Coord = long long;

    struct Node {
        T sum = 0;
        int left = -1;
        int right = -1;
    };

    Coord lo_root, hi_root;
    vector<Node> pool;

    DynamicSparseSumSegTree(Coord lo, Coord hi) : lo_root(lo), hi_root(hi) {}

    int new_root() {
        int u = (int)pool.size();
        pool.push_back({});
        return u;
    }

    // SAFE: do NOT take an int& into pool here. new_root()/push_back may
    // reallocate the pool and invalidate any prior reference. We re-read
    // the parent's child slot by index after the allocation and write the
    // new child id then.
    int materialize_left(int parent) {
        if (pool[parent].left == -1) {
            int child = new_root();          // may reallocate pool
            pool[parent].left = child;       // re-read parent by index
        }
        return pool[parent].left;
    }

    int materialize_right(int parent) {
        if (pool[parent].right == -1) {
            int child = new_root();
            pool[parent].right = child;
        }
        return pool[parent].right;
    }

    void point_add(int& root, Coord pos, T value) {
        if (root == -1) root = new_root();
        point_add(root, lo_root, hi_root, pos, value);
    }

    void point_add(int u, Coord lo, Coord hi, Coord pos, T value) {
        if (lo == hi) { pool[u].sum += value; return; }
        Coord mid = lo + (hi - lo) / 2;
        if (pos <= mid) {
            int c = materialize_left(u);
            point_add(c, lo, mid, pos, value);
        } else {
            int c = materialize_right(u);
            point_add(c, mid + 1, hi, pos, value);
        }
        T lS = pool[u].left  == -1 ? 0 : pool[pool[u].left ].sum;
        T rS = pool[u].right == -1 ? 0 : pool[pool[u].right].sum;
        pool[u].sum = lS + rS;
    }

    int merge(int a, int b, Coord lo, Coord hi) {
        if (a == -1) return b;
        if (b == -1) return a;

        if (lo == hi) {
            pool[a].sum += pool[b].sum;
            return a;
        }

        Coord mid = lo + (hi - lo) / 2;
        pool[a].left  = merge(pool[a].left,  pool[b].left,  lo,     mid);
        pool[a].right = merge(pool[a].right, pool[b].right, mid+1, hi);

        T lS = pool[a].left  == -1 ? 0 : pool[pool[a].left ].sum;
        T rS = pool[a].right == -1 ? 0 : pool[pool[a].right].sum;
        pool[a].sum = lS + rS;
        return a;
    }

    T range_sum(int u, Coord ql, Coord qr) const {
        return range_sum(u, lo_root, hi_root, ql, qr);
    }

    T range_sum(int u, Coord lo, Coord hi, Coord ql, Coord qr) const {
        if (u == -1 || qr < lo || hi < ql) return 0;
        if (ql <= lo && hi <= qr) return pool[u].sum;
        Coord mid = lo + (hi - lo) / 2;
        return range_sum(pool[u].left,  lo,     mid, ql, qr)
             + range_sum(pool[u].right, mid+1, hi,  ql, qr);
    }
};

🔎 Line-by-Line Explanation

merge — the only new routine click to expand/collapse

The function takes two pool indices and the current coordinate range. The first two short-circuits cover the only-one-side-present case: return whichever is real. At a leaf with both sides present, add the two sums. In the recursive case, recurse on both sides, store the returned indices back into pool[a], and re-pull the sum at a. The function never allocates a new node — it only reads existing pool nodes and writes child pointers.

point_add overload accepting a root reference click to expand/collapse

Because each vertex in the outer algorithm has its own root, the segment tree class exposes a version of point_add that takes a root by reference. If the root is still -1 (no nodes yet), the call allocates one before descending.

Typical usage in DFS click to expand/collapse

The outer algorithm runs a post-order DFS. For each leaf vertex, it inserts the vertex's color into a fresh segment tree. For each internal vertex, it folds each child's segment tree into the current vertex's segment tree using seg[v] = merge(seg[v], seg[c], 0, U-1);. After all children are folded in, seg[v] is the multiset of subtree colors and any answer for v can be read off (count of color c, k-th smallest color, etc.).

⚠️ Common Pitfalls

⚠️ Pitfall — Holding an int& into the pool across new_root()

A tempting helper for the point-add path is int materialize(int& slot){ if(slot == -1){ slot = new_root(); } }. It's unsafe because new_root() ends in push_back and can reallocate the pool's buffer, invalidating slot before the assignment runs. The fix used in the template above is to pass the parent index by value and write the new child id back via pool[parent].left/right after the allocation.

⚠️ Pitfall — Using a merged tree as if both inputs were intact

After seg[v] = merge(seg[v], seg[c], ...), the original seg[c] is no longer a valid root — its nodes have been incorporated into seg[v]. Don't query seg[c] afterwards. The standard discipline is to merge children into the parent in a post-order DFS and only query at the parent.

⚠️ Pitfall — Forgetting to re-pull sum at the parent

After the two recursive merge calls return, the current node's sum may be stale. Always recompute it from the (possibly new) children before returning.

⚠️ Pitfall — Using merge with lazy propagation but no push-before

If your segment tree carries lazy tags, the merge function must push any pending lazy state at both input roots before descending. Otherwise the merge will combine stale subtree summaries with fresh ones. This is the hidden gotcha when extending merging to lazy variants.

⚠️ Pitfall — Calling merge on roots with different coordinate ranges

The merge function only makes sense when both trees share the same coordinate range [lo_root, hi_root]. Mixing ranges silently produces wrong sums. If two trees were built for different ranges, you must rebuild one before merging.

⚠️ Pitfall — Mistaking the bound for <per-merge O(log U)>

A single merge can be expensive (linear in the smaller of the two trees, in the worst case). The O(N log U) bound is global, summed over all merges in the algorithm. Don't promise “merge is O(log U)” in a problem statement — it isn't.

🎛️ Variants & Extensions

Variant 1 — Persistent merging

To merge while keeping both inputs intact, allocate a fresh node at every both-children step, mimicking path-copying. Cost rises to O(N log² U) per total work, and memory to O(N log U). Used when you need to query both pre-merge trees later.

Variant 2 — Weighted merge

Each node carries multiple aggregates (sum, max, count of distinct, etc.). The merge function combines them according to each aggregate's combine rule. Standard pattern for “at each subtree, give me the color with the highest weighted frequency” (CF 600E).

Variant 3 — Mergeable with lazy propagation

If the underlying tree supports range updates with lazy tags, the merge function must push pending tags on both inputs before descending. Otherwise the merge will combine stale subtree summaries.

Variant 4 — Split + Merge

For some problems, you want the inverse: split one tree at a coordinate into two trees. Implement split(u, key) mirroring merge; together with merge, you get a “sortable” structure used in problems like “sort a permutation by range chmin/chmax”.

🏁 Final Summary

Segment tree merging is a single additional routine on top of the dynamic sparse segment tree from Lesson 7. It walks two trees in parallel, with three cases per node: keep A unchanged when B is a ghost, steal B when A is a ghost, recurse and re-pull when both are real. Because the merge consumes pool nodes destructively, the total work across all merges in an algorithm is linear in the total number of pool nodes — O(N log U) for N inserted points.

The killer application is tree DP where each subtree maintains a histogram or multiset of values. The classic CF-600E (Lomsat gelral) is the textbook example: at every vertex, store the color frequencies in the subtree, and aggregate from leaves to root by repeatedly merging children. The next lesson (XOR segment tree) takes a similar “walk two roots in parallel” idea but applies it to a permuted virtual array.
