Lesson 6 — Merge Sort Tree

Sorted Vector per Node · O(log² n) Order Queries · Static Arrays

🧩 Problem Definition

Given a static array a[0..n−1], support many queries of the form: given a range [l, r] and a threshold x, return the number of elements in a[l..r] with value ≤ x. The array is never updated.

This is the classic range order statistic primitive. With it, by binary-searching on x we can answer k-th smallest in a range (Lesson 5's problem) — though typically slower than the persistent solution. The merge-sort tree's real attraction is its generality: count of values in [lo, hi] with one subtraction, sum of values ≤ x, lower-bound rank inside any range, etc.

Build: O(n log n) Range count: O(log² n) Memory: O(n log n) Static (no updates)

💡 Intuition from Scratch

The recursive segment tree from Lesson 1 covers any [l, r] with O(log n) fully-covered nodes plus O(log n) boundary nodes. Each fully-covered node represents some prefix-like block of the array, but its order information is lost — the node only stores a sum, or a min, or a max.

Suppose instead we store, at every node, the sorted vector of values in that node's interval. Then for a fully-covered node, "how many values are ≤ x" is a single upper_bound, costing O(log(interval size)). Summing over the O(log n) fully-covered nodes gives O(log² n) total.

Storing sorted vectors at every level looks expensive, but each level holds n values total (each array element appears once per level), and there are log n levels — so the whole structure is O(n log n) space. The build itself is just merge sort done bottom-up: hence the name.

🌳 Visual Mental Model

Picture the segment tree as a pyramid. Each node holds a sorted vector. A leaf holds a one-element vector. An internal node's vector is the sorted merge of its two children's vectors. As we walk up, the vectors get longer; at the root, the vector is the entire array a sorted.

a = [5, 2, 7, 1, 9, 3, 8, 4] leaves: [5][2][7][1][9][3][8][4] level 1: [2,5] [1,7] [3,9] [4,8] level 2: [1,2,5,7] [3,4,8,9] root: [1,2,3,4,5,7,8,9]

A query (l, r, x) walks the tree exactly like Lesson 1, but on every full-overlap node it does upper_bound(node.vec.begin(), node.vec.end(), x) − node.vec.begin() instead of returning a precomputed scalar.

📐 Formal Abstraction

The tree uses the standard 0-based heap indexing. Each node stores:

struct Node { vector vals; // sorted values in this node's interval };

Build: vals at leaf i contains the single element a[i]; at internal node i it equals merge(left.vals, right.vals). Use std::merge in O(L + R).

The query is the recursive segment tree's standard descent:

  • No overlap: contribute 0.
  • Full overlap: contribute upper_bound(vals, x) − vals.begin().
  • Partial overlap: recurse into both children, sum the results.

🔬 Key Observations

Observation 1. Each array element a[i] lives in exactly one vector per level (the level's nodes partition [0, n−1]). So the total storage is exactly (levels) · n = O(n log n).

Observation 2. A query visits O(log n) full-overlap nodes, each costing O(log n) for one binary search. Total: O(log² n) per query.

Observation 3. The build is just merge sort done top-down — every internal node's vector is a merge of its two children. Total build cost is O(n log n), matching merge sort.

Observation 4. Because we only ever read from the sorted vectors, the tree is not updatable in O(log n). Inserting one element requires touching O(log n) vectors with O(n) shifts each. So this is a static-only structure unless paired with a BIT (see Variants).

⚙️ Algorithm

Build (top-down recursion)

At node i with interval [l, r]: if l == r (and l < n), set vals = {a[l]} and return. Otherwise recurse into both children, then set vals = merge(left.vals, right.vals). The merge is std::merge(left.begin(), left.end(), right.begin(), right.end(), back_inserter(vals)).

count_leq(ql, qr, x)

Standard recursive descent. No overlap → 0. Full overlap → upper_bound(vals.begin(), vals.end(), x) − vals.begin(). Partial → recurse left + recurse right. Total O(log² n).

count_in_range(ql, qr, lo, hi)

Same skeleton: upper_bound(hi) − lower_bound(lo) at each full-overlap node.

kth_smallest(ql, qr, k)

Binary search on the answer value using count_leq: find the smallest x such that count_leq(ql, qr, x) ≥ k. Two layers of binary search: O(log V · log² n) per query.

🛡️ Invariants

✅ Invariant — Sorted invariant

For every node i, vals[i] is sorted in non-decreasing order. This is preserved at construction by std::merge and never mutated thereafter.

✅ Invariant — Multiset invariant

For every node i with interval [l, r], vals[i] is exactly the multiset of values {a[l], a[l+1], ..., a[r]} (sorted). Duplicates are preserved.

✅ Invariant — Level-total invariant

The sum of |vals[i]| over all nodes at depth d is exactly n (or base, counting padded leaves with 0-length vectors).

🧮 Proof of Correctness

🧮 Proof — Why summing upper_bound over fully-covered nodes is correct

The standard segment-tree descent partitions [ql, qr] into O(log n) disjoint, fully-covered subintervals — call them I_1, I_2, ..., I_k. By the Multiset invariant, each vals[I_j] is the sorted multiset of values in I_j. The number of values in I_j that are ≤ x is exactly upper_bound(vals[I_j].begin(), vals[I_j].end(), x) − vals[I_j].begin().

Since the I_j are disjoint and their union is exactly [ql, qr], the total count of values ≤ x in [ql, qr] is the sum of these binary-search counts. The recursive descent visits exactly these I_j as its full-overlap leaves. ∎

Use the merge sort tree when the queries are not k-th but something like "how many values are between 100 and 200 in this range" (where the persistent solution would need two descents).

🧰 Core Template Philosophy

The template is the recursive segment tree from Lesson 1 with two changes: the per-node payload is a sorted vector<int> instead of a scalar, and the per-node "query summary" is replaced with a binary search. Everything else — the three-cases descent, the base-rounding, the 0-based heap indexing — is identical.

For any new query you want to support, you only need to answer: "given a sorted vector and a query parameter x, what does my query contribute?" If the answer is a single STL call (lower_bound, upper_bound, equal_range, partial sum prefix etc.), the merge sort tree gives you that query in O(log² n).

💻 Master Template (C++)

#include <bits/stdc++.h>
using namespace std;

struct MergeSortTree {
    int n = 0;
    int base = 1;
    vector<vector<int>> vals;

    MergeSortTree() {}

    MergeSortTree(const vector<int>& a) {
        init(a);
    }

    void init(const vector<int>& a) {
        n = (int)a.size();

        base = 1;
        while (base < max(1, n)) {
            base <<= 1;
        }

        vals.assign(2 * base, {});

        build(0, 0, base - 1, a);
    }

    void build(int i, int l, int r, const vector<int>& a) {
        if (l == r) {
            if (l < n) vals[i] = { a[l] };
            return;
        }

        int left = 2 * i + 1;
        int right = left + 1;
        int mid = l + (r - l) / 2;

        build(left, l, mid, a);
        build(right, mid + 1, r, a);

        vals[i].resize(vals[left].size() + vals[right].size());
        merge(vals[left].begin(), vals[left].end(),
              vals[right].begin(), vals[right].end(),
              vals[i].begin());
    }

    int count_leq(int ql, int qr, int x) const {
        return count_leq(ql, qr, x, 0, 0, base - 1);
    }

    int count_leq(int ql, int qr, int x, int i, int l, int r) const {
        if (qr < l || r < ql) return 0;

        if (ql <= l && r <= qr) {
            return (int)(upper_bound(vals[i].begin(), vals[i].end(), x)
                         - vals[i].begin());
        }

        int left = 2 * i + 1;
        int right = left + 1;
        int mid = l + (r - l) / 2;

        return count_leq(ql, qr, x, left, l, mid)
             + count_leq(ql, qr, x, right, mid + 1, r);
    }

    int count_in_range(int ql, int qr, int lo, int hi) const {
        return count_leq(ql, qr, hi) - count_leq(ql, qr, lo - 1);
    }
};

🔎 Line-by-Line Explanation

build(i, l, r, a) — merge sort done bottom-up click to expand/collapse

A leaf at l == r with l < n stores the single element a[l] as its sorted vector. Padded leaves remain empty. An internal node recursively builds its children, then uses std::merge to produce its own sorted vector in one linear pass. The merge writes into a pre-resized vector, so the total work at each level is exactly n, and across all log n levels the build is O(n log n).

count_leq(ql, qr, x, i, l, r) — three cases with binary search at full click to expand/collapse

If [l, r] doesn't overlap [ql, qr], return 0. If it's fully inside, do one upper_bound on vals[i]; the iterator-difference gives the count of values ≤ x. Otherwise recurse into both children and return the sum. The descent itself is identical to Lesson 1's structure; the difference is purely what happens at full overlap.

count_in_range(ql, qr, lo, hi) click to expand/collapse

Two calls to count_leq: one with hi and one with lo − 1. Subtract. This handles closed-interval value queries naturally, including unbounded sides (use INT_MAX or INT_MIN).

⚠️ Common Pitfalls

⚠️ Pitfall — Using lower_bound instead of upper_bound (or vice versa)

For "count of values ≤ x", use upper_bound(x) - begin. For "count of values < x", use lower_bound(x) - begin. Mixing them produces off-by-one bugs that disappear on small inputs.

⚠️ Pitfall — Allocating vectors in the wrong order

If you build a node before its children, the child vectors don't exist yet to merge. Always recurse into children first; only then merge upward. The provided template does this naturally because the recursion is a post-order traversal.

⚠️ Pitfall — Storing pointers/iterators across builds

The vectors are inside a vector<vector<int>>. Iterators into the inner vectors are invalidated if the outer vector reallocates. Always use vals.assign(2 · base, {}) first, never push_back.

⚠️ Pitfall — Pretending the tree supports updates

An update would require shifting O(log n) sorted vectors with O(n) work each — totally unacceptable. If updates are needed, switch to a merge-sort tree with BIT (offline only), or to a persistent segment tree, or to a sorted set per node with order-statistic indexing (slow and memory-heavy).

⚠️ Pitfall — Forgetting padded leaves contribute zero

It's tempting to skip the empty vectors as a special case, but the upper_bound on an empty vector correctly returns 0. The only thing to watch is the full-overlap case for padded leaves: the descent will not reach them in practice because the query is clamped to [0, n−1], but if you allow qr ≥ n by mistake, the answer will be wrong silently. Always clamp qr = min(qr, n − 1).

🎛️ Variants & Extensions

Variant 1 — Range count of values in [lo, hi]

One subtraction of two count_leq calls. O(log² n) per query.

Variant 2 — Range sum of values ≤ x

Store both vals (sorted) and prefSum at each node. At a full-overlap node, the contribution is prefSum[upper_bound − begin]. Still O(log² n).

Variant 3 — Range k-th smallest by binary-search outside

Outer binary search on x in the value range; inner uses count_leq. Total O(log V · log² n). Slower than the persistent approach but very easy to code if you already have count_leq.

Variant 4 — Fractional cascading

Store, at every internal node, two parallel pointer arrays into the children's sorted vectors. Then at a full-overlap node the binary search starts from a known position — saving a log n factor. Theoretical O(log n) per query but implementation-heavy; rarely worth it in practice unless contest limits demand it.

🏁 Final Summary

A merge sort tree is a segment tree whose nodes hold sorted vectors of their intervals. The build is just merge sort done top-down on the segment tree shape. Queries that ask order-statistic questions on a range — count, count-in-range, sum-of-values-≤-x — become a recursive descent that runs upper_bound (or lower_bound) at every full-overlap node, costing O(log² n) per query.

The structure is static, but its versatility for order queries (especially count-in-value-range) makes it a daily-driver for static problems. When you need updates, switch to a persistent variant or to BIT-of-sorted-vectors.
