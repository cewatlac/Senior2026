Mo with Updates — 3D Mo

Adding a Time Dimension · Apply / Undo Hooks · Codeforces 940F Machine Learning

🧩 Problem Definition

Classic Mo solves offline range queries on a static array. In this file we relax that assumption: between the queries, point updates may also arrive. The input stream now contains two kinds of operations. A type-1 operation is a query: given l and r, compute some frequency-based aggregate over the current contents of the range a[l..r]. A type-2 operation is an update: given a position pos and a new value val, overwrite a[pos] = val. The two kinds are interleaved in arbitrary order, and the algorithm is still allowed to be offline — we may read everything, sort, and produce answers in input order.

The canonical illustration in this file is Codeforces 940F (Machine Learning). It asks for the smallest positive integer f such that no value occurs exactly f times in the queried range, while supporting point updates of array values. We will use 940F throughout this file as the running example for the master template, the line-by-line walk, and the tracer.

💡 Intuition — Why Classic Mo Breaks

The whole reason classic Mo is correct is that the maintained state is uniquely determined by two integers (curL, curR). If you tell me where the pointers are, I can tell you the multiset of values they cover. Pointer movement is reversible because the array contents under the pointers never change.

Add even one update to the mix and that uniqueness collapses. The multiset of values inside [l, r] now depends on which updates have already been applied. Two visits to the same [l, r] at different points in input order can yield different answers. The classic two-pointer engine cannot recover from this on its own.

The fix is to extend the state with a third coordinate, curT — the number of updates that have currently been applied to the array. The full state is now (curL, curR, curT), and the engine grows a third loop that moves time forward (by applying updates) and backward (by undoing them). The classic four-loop pointer engine survives unchanged; we wrap it inside a sixth and seventh loop for the time axis.

Because we are still offline, we can sort queries by a 3-D key — block of l, block of r, time t — so that the total pointer travel along all three axes stays bounded.

🎯 Visual Mental Model — A 3-D State

Picture a cube. The x axis is the left pointer, the y axis is the right pointer, and the z axis is time. Each query is a point inside that cube, and our task is to walk from point to point with a total path length that is sub-quadratic in N · Q.

Two pieces of structure make this tractable. First, classic Mo's comparator still organizes l into √N blocks; we now do the same for r, creating a 2-D block grid in the (l, r) plane. Inside each cell of that grid, we sort by t. Second, the time axis is monotone within each cell: we only ever apply or undo updates one at a time, and a balanced amount of forward/backward motion is acceptable because each update touches at most one cell of the array.

The right block size for this geometry is no longer √N. It is N^(2/3), which balances pointer travel across all three axes and yields a total complexity of O(Q · N^(2/3)). We will derive that more carefully later in the file.

🧠 Formal Abstraction — Updates as Reversible Hooks

The update queue is stored as a vector of records. Each record contains pos (the position that is being overwritten), oldVal (the value that was there before this update), and newVal (the value that will be there after). Both values are recorded so that the algorithm can apply or undo the update symmetrically.

struct Query  { int l, r, t, id; };
struct Update { int pos, oldVal, newVal; };

The single most important rule about oldVal is that it must be the value of a[pos] after all previous updates have been applied, not the value from the original input array. We compute it during a single forward pass through the input operations, keeping a "current array" that mirrors what the array would look like at the moment each update was issued.

The hooks have a satisfying symmetry. apply_update(id) reads newVal from updates[id] and writes it into a[pos]. undo_update(id) reads oldVal from the same record and writes it back. Both hooks also patch the maintained Mo state when the updated position is inside the current pointer interval — and crucially, they leave the maintained state alone when the position is outside. This is the entire trick that makes 3D Mo correct.

🔭 Key Observations

Observation 1 — Position-vs-range branching is the only new logic

When curL ≤ pos ≤ curR, the value at pos participates in the maintained Mo state. Changing it means doing one remove_value(oldVal) immediately followed by one add_value(newVal). When pos is outside the pointer interval, the maintained state is unaffected and we only patch the underlying array a[pos]. This single branch is the entire conceptual addition in 3D Mo.

Observation 2 — Time is monotone within a cell, not globally

Across the whole sort, curT goes up and down many times. Inside a single (block_l, block_r) cell, however, curT is monotonically non-decreasing because the comparator's tertiary key is t. This local monotonicity is exactly what keeps the time-axis travel within each cell linear in the total number of updates.

Observation 3 — Block size is N^(2/3), not √N

In 2D Mo, balancing the cost of moving the right pointer (linear inside a block) against the cost of jumping the left pointer (one block-step per query) gives B = √N. In 3D Mo, balancing left-pointer travel, right-pointer travel, and time-pointer travel gives B = N^(2/3). The CF 940F template uses max(1, ⌊N^(2/3)⌋).

Observation 4 — Time-axis loop comes before the four pointer loops

It is not a hard constraint, but it is a sane convention: move time first to synchronize the array, then move the pointers. This keeps the position-vs-range check in apply_update/undo_update meaningful — the pointers still describe the interval at the previous query, which is exactly the interval that should react to in-range updates.

🔧 Apply & Undo Updates — The Heart of the Variant

Both hooks have the same shape. They patch the underlying array and, only when the updated position sits inside the current pointer interval, they patch the maintained Mo state by removing the old value and adding the new one.

void apply_update(int id) {
    int pos    = updates[id].pos;
    int oldVal = updates[id].oldVal;
    int newVal = updates[id].newVal;

    if (curL <= pos && pos <= curR) {
        remove_value(oldVal);
        add_value(newVal);
    }
    a[pos] = newVal;
}

void undo_update(int id) {
    int pos    = updates[id].pos;
    int oldVal = updates[id].oldVal;
    int newVal = updates[id].newVal;

    if (curL <= pos && pos <= curR) {
        remove_value(newVal);
        add_value(oldVal);
    }
    a[pos] = oldVal;
}

Why is this symmetric pair correct? In apply_update, the array currently has a[pos] = oldVal (the value before this particular update was applied). When pos is in range, oldVal is part of the Mo state, so we remove it. We then add newVal because after the write to a[pos], that is the new participant. undo_update is the mirror image: the array has a[pos] = newVal, we remove that from the Mo state, add oldVal back, and overwrite the array. The order of the two remove/add calls and the assignment is interchangeable here because the Mo state and the array are decoupled within the hook's body — but write the hook in this canonical order anyway, so future-you can pattern-match it instantly.

🔀 3D Ordering — (block_l, block_r, t)

The comparator is a tiny extension of classic Mo's comparator. Primary key: block of l. Secondary key: block of r. Tertiary key: t. There is no odd-even snake on the right axis in the canonical 3D template; the wins from doing one come from a smaller constant, and most contest setups already pass without it.

bool cmp(const Query& a, const Query& b) {
    int al = a.l / BLOCK, bl = b.l / BLOCK;
    if (al != bl) return al < bl;

    int ar = a.r / BLOCK, br = b.r / BLOCK;
    if (ar != br) return ar < br;

    return a.t < b.t;
}

Why two blocks of size B = N^(2/3) on both l and r? Each query incurs at most O(B) left-pointer travel within its (block_l, block_r) cell, and the total left-pointer travel across all cells is bounded by O(Q · B + (N/B)² · N) — the second term coming from jumps between cells. Setting Q · B = N³/B² yields B = (N³/Q)^(1/3); when Q ≈ N, this is N^(2/3). The time-axis cost is bounded by U (the number of updates) times the number of cells, and the same B balances it.

📄 Master Template — CF 940F Machine Learning

This is the accepted Codeforces 940F template, preserved verbatim from the source corpus and lightly reformatted. The answer state uses two arrays: cntValue[x] is the frequency of value x in the current range, and cntFreq[f] is the number of values whose frequency equals exactly f. The answer is the smallest f ≥ 1 with cntFreq[f] = 0. The MEX is found by a linear scan — perfectly fine here because the answer is bounded by the square root of the range length.

#include <bits/stdc++.h>
using namespace std;

struct Query {
    int l, r, t, id;
};

struct Update {
    int pos, oldVal, newVal;
};

int BLOCK;

bool cmp(const Query& a, const Query& b) {
    int al = a.l / BLOCK;
    int bl = b.l / BLOCK;
    if (al != bl) return al < bl;

    int ar = a.r / BLOCK;
    int br = b.r / BLOCK;
    if (ar != br) return ar < br;

    return a.t < b.t;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, m;
    cin >> n >> m;

    vector<int> initial(n + 1);
    vector<array<int, 3>> operations;
    vector<int> allValues;

    for (int i = 1; i <= n; i++) {
        cin >> initial[i];
        allValues.push_back(initial[i]);
    }

    operations.reserve(m);
    for (int i = 0; i < m; i++) {
        int type, x, y;
        cin >> type >> x >> y;
        operations.push_back({type, x, y});
        if (type == 2) allValues.push_back(y);
    }

    sort(allValues.begin(), allValues.end());
    allValues.erase(unique(allValues.begin(), allValues.end()), allValues.end());

    auto compress = [&](int x) {
        return (int)(lower_bound(allValues.begin(), allValues.end(), x) - allValues.begin()) + 1;
    };

    vector<int> a(n + 1);
    for (int i = 1; i <= n; i++) a[i] = compress(initial[i]);

    vector<int> currentForReading = a;
    vector<Query> queries;
    vector<Update> updates(1); // 1-based updates

    int updateCount = 0;
    int queryCount = 0;

    for (auto op : operations) {
        int type = op[0], x = op[1], y = op[2];

        if (type == 1) {
            queries.push_back({x, y, updateCount, queryCount++});
        } else {
            int pos = x;
            int newVal = compress(y);
            int oldVal = currentForReading[pos];
            updates.push_back({pos, oldVal, newVal});
            currentForReading[pos] = newVal;
            updateCount++;
        }
    }

    BLOCK = max(1, (int)pow(n, 2.0 / 3.0));
    sort(queries.begin(), queries.end(), cmp);

    vector<int> cntValue(allValues.size() + 2, 0);
    vector<int> cntFreq(n + 2, 0);
    vector<int> ans(queryCount);

    int curL = 1, curR = 0, curT = 0;

    auto add_value = [&](int x) {
        cntFreq[cntValue[x]]--;
        cntValue[x]++;
        cntFreq[cntValue[x]]++;
    };

    auto remove_value = [&](int x) {
        cntFreq[cntValue[x]]--;
        cntValue[x]--;
        cntFreq[cntValue[x]]++;
    };

    auto add_index = [&](int idx) {
        add_value(a[idx]);
    };

    auto remove_index = [&](int idx) {
        remove_value(a[idx]);
    };

    auto apply_update = [&](int id) {
        int pos = updates[id].pos;
        int oldVal = updates[id].oldVal;
        int newVal = updates[id].newVal;

        if (curL <= pos && pos <= curR) {
            remove_value(oldVal);
            add_value(newVal);
        }

        a[pos] = newVal;
    };

    auto undo_update = [&](int id) {
        int pos = updates[id].pos;
        int oldVal = updates[id].oldVal;
        int newVal = updates[id].newVal;

        if (curL <= pos && pos <= curR) {
            remove_value(newVal);
            add_value(oldVal);
        }

        a[pos] = oldVal;
    };

    auto get_answer = [&]() {
        int f = 1;
        while (cntFreq[f] > 0) f++;
        return f;
    };

    for (const Query& qu : queries) {
        while (curT < qu.t) apply_update(++curT);
        while (curT > qu.t) undo_update(curT--);

        while (curL > qu.l) add_index(--curL);
        while (curR < qu.r) add_index(++curR);
        while (curL < qu.l) remove_index(curL++);
        while (curR > qu.r) remove_index(curR--);

        ans[qu.id] = get_answer();
    }

    for (int x : ans) cout << x << '\n';
    return 0;
}

🔒 Invariants

State invariant

At every moment during the main loop, the maintained data structures (cntValue, cntFreq in 940F; more generally, whatever your answer layer needs) exactly represent the multiset of values { a[i] : curL ≤ i ≤ curR } in the array's snapshot after curT updates.

Array-mirror invariant

At every moment during the main loop, a[] equals the array's snapshot after exactly curT updates. This is a stronger claim than just "Mo state is correct"; it says the array storage itself has been brought to the right time. Without this, the next pointer step would add the wrong value to the Mo state.

Transition invariant

Each time-step inside the two curT loops touches exactly one update record and performs at most one remove/add pair (when the update position is in range), plus a write to a[pos]. Each step inside the four pointer loops touches exactly one array cell and performs exactly one add/remove call.

Query invariant

Immediately before storing ans[qu.id], all three pointers match the query: (curL, curR, curT) = (qu.l, qu.r, qu.t). The maintained state therefore represents exactly the queried multiset at the queried point in time.

📜 Proof of Correctness — Why Undo Works

The classic Mo proof shows that the four pointer loops preserve the state invariant. We need to show that the two extra time loops preserve it as well, and that apply_update and undo_update are exact inverses of each other.

Lemma A — apply_update preserves the state invariant

Suppose the array-mirror invariant holds and Mo state is correct for snapshot curT. After apply_update(curT+1), we want the array-mirror invariant to hold for snapshot curT+1 and the Mo state to be correct for it. By construction, the underlying array has a[pos] = oldVal right before the hook runs (because we are at snapshot curT, and the offline pre-pass recorded oldVal as the value at pos immediately before this update). After a[pos] = newVal in the hook, the array equals snapshot curT+1; that closes the array-mirror part. The Mo state changed by exactly one remove(oldVal)/add(newVal) pair iff pos ∈ [curL, curR], which is exactly the difference between the multiset at snapshot curT and at snapshot curT+1 restricted to [curL, curR].

Lemma B — undo_update preserves the state invariant

The same argument runs in reverse. At entry, the array is at snapshot curT with a[pos] = newVal; on exit, it is at snapshot curT-1 with a[pos] = oldVal. The Mo state mutation removes newVal and adds oldVal, undoing what apply_update(curT) did. Crucially, the in-range check uses the current curL/curR, not the values at the moment apply_update was called; this is fine because the invariant requires only that the Mo state match the current (curL, curR, curT), not the historical one.

Theorem

Combining Lemmas A and B with the classic Mo lemmas for pointer movement, the state invariant and array-mirror invariant hold at every moment of the main loop. In particular they hold immediately before each get_answer call. Therefore every stored answer is correct.

⏱ Complexity Analysis

Let N be the array length, Q be the number of queries, and U be the number of updates. Let B be the block size used by the comparator, and let F be the cost of one add_value/remove_value call.

Across all queries, the left pointer travels O(Q · B) within cells and O((N/B)² · N) = O(N³/B²) across cell boundaries. The right pointer travels O(N · N/B) = O(N²/B) for each block_l bucket, times N/B buckets, giving O(N³/B²) in total — same as the cross-cell left jumps. The time pointer travels O(U) per cell, times (N/B)² cells, giving O(U · N²/B²).

Setting Q · B = N³/B² gives B = (N³/Q)^(1/3). For Q ≈ N this collapses to B = N^(2/3) and the total complexity becomes O(Q · N^(2/3) · F). With F = O(1) as in 940F, the practical running time is roughly N^(5/3). For N = 10⁵ that is on the order of 2 · 10⁸ elementary operations — fast enough on a typical 2 s judge limit, often with comfortable margin.

🔎 Line-by-Line — CF 940F Dissected

Coordinate compression over both sources

The vector allValues collects the initial array values and every update's new value. After sort+unique, compress(x) maps any value that will ever appear into a small contiguous range. Missing this step — for example, only compressing the initial values — leaves the update values uncompressed and either crashes cntValue or produces wrong frequencies.

Offline pre-pass to record oldVal correctly

The local variable currentForReading is a mirror of the array as it would look at the moment we are reading an update from the input stream. When we encounter a type-2 operation, the value we record as oldVal is exactly currentForReading[pos], not initial[pos]. If two consecutive updates touch the same position, the second one's oldVal is the first one's newVal — which is exactly what currentForReading captures.

Initial pointers

This template uses 1-based indexing throughout, so the canonical empty interval is curL = 1, curR = 0. curT = 0 means no updates have yet been applied. These three values together represent the "empty Mo state at the beginning of time".

Block size pow(n, 2/3)

The integer block size is computed by max(1, (int)pow(n, 2.0/3.0)). The max guard prevents division by zero when n < 2. Using cbrt(n*n) is mathematically equivalent and often slightly faster in practice.

The double-indexed answer layer

The lambdas add_value and remove_value form the canonical "frequency-of-frequency" pattern: bump cntFreq down for the old frequency, mutate cntValue, and bump cntFreq back up for the new frequency. The MEX in get_answer is a tiny linear scan because the answer is bounded by ⌈√N⌉ (no more than that many distinct frequencies can be active simultaneously without overshooting the range length).

The seven-loop engine

The main loop now has seven phases per query. First the two time loops bring the array to the right snapshot. Then the four pointer loops bring curL/curR to the right interval. Finally get_answer reads off the result. The time loops come first by convention; switching the order is correct but requires more care, because the in-range check inside the update hooks then uses the not-yet-updated pointers.

🪜 Centralized Edge-Case Audit

The classic-Mo audit still applies; the items below are additional checks that specifically concern the time dimension.

#CheckWhy it matters
1Did you compress both original and update values?Skipping one collection silently breaks the frequency layer.
2Is oldVal recorded from the post-previous-updates state?Two consecutive updates to the same position have chained oldVal; using initial[pos] is wrong.
3Do apply_update and undo_update patch a[pos] unconditionally?The array-mirror invariant requires updating a[pos] even when the position is outside [curL, curR].
4Are time loops placed before pointer loops?Putting them after still works but flips the meaning of the in-range check; do not mix conventions across the same codebase.
5Did you initialize curT = 0 and 1-base the updates vector?The template uses updates(1) so that updates[1] is the first real update. Off-by-one here corrupts every query.
6Is BLOCK = N^(2/3), not √N?Using √N works but is asymptotically suboptimal — typically times out on the largest tests.
7Are the sizes of cntValue and cntFreq correct?cntValue is sized by the number of distinct values; cntFreq by the max possible frequency, which is N.
8Did you store qu.t = updateCount at the moment of reading the query?Each query's t is the number of updates that have already been applied before it appears in the input stream.
9Is the MEX scan bounded?The linear MEX is safe because the answer is at most ⌈√N⌉+1; if you ever see it spinning past that bound, an invariant has broken.
10Are queries read with type==1 and stored only then?Mixing query and update records in a single vector and post-filtering is a common bug; keep them in separate vectors.
11Did you set curL = 1, curR = 0 (1-based)?If you use 0-based pointers but 1-based array storage, every add_index reads from the wrong cell.
12Are ans writes by qu.id, not by sorted index?Same trap as classic Mo; sorted iteration must still write back into the original output position.

⚠️ Common Pitfalls

Pitfall 1 — Using the wrong oldVal

If two updates target the same position, the second update's oldVal must be the first update's newVal. Compute oldVal by maintaining a running mirror of the array during the offline pre-pass, not by reading from initial[].

Pitfall 2 — Forgetting to mutate a[pos] when out of range

The position-vs-range branch decides whether to touch the Mo state. The write to a[pos] happens unconditionally, because the array-mirror invariant must hold whether or not the update affected the maintained state.

Pitfall 3 — Compressing only initial values

If you forget to include update values in the compression pool, calls like cntValue[compress(newVal)] read garbage or out-of-bounds memory. Include every value that will ever land in the array.

Pitfall 4 — √N block size

Plenty of submissions use √N out of habit and pass when constraints are gentle. On the official 940F bounds (N, M ≤ 10⁵), √N typically TLEs while N^(2/3) is comfortable.

Pitfall 5 — Forgetting that the comparator depends on BLOCK

If you sort queries before assigning BLOCK, the comparator uses whatever value BLOCK had at the time — usually zero — and division-by-zero crashes silently inside sort. Always set BLOCK first.

Pitfall 6 — Mixing time-axis convention across hooks

If apply_update swaps the order of "mutate a[pos]" and "patch Mo state" while undo_update does not, you will produce wrong answers on a non-trivial fraction of tests. Keep the structure of the two hooks symmetric and obvious.

🧭 Decision Guide — 3D Mo vs Alternatives

SituationPrefer
Offline range queries, no updatesclassic Mo (see file 1).
Offline range queries, point updates allowed3D Mo (this file).
Online range queries with updates and a clean mergesegment tree.
Online range queries with updates, frequency-based answeronline sqrt with block-precomputed structures (see file 4).
Tree queries with updatesHLD + segment tree, or Euler tour + 3D Mo flattening (advanced).
Range updates instead of point updates3D Mo does not apply directly — use offline difference arrays + segtree.

Reflex rules

The reflex rule for 3D Mo is short: choose it when the problem is offline, the queries are range queries on a 1-D array, the updates are point updates, and the answer would already have been a Mo problem if the updates were absent. If any of these conditions fails, the right tool is something else.

🚀 Problem-Based Learning — Codeforces 940F

24.1 — Codeforces 940F: Machine Learning MEX over freq-of-freq

VJudge: https://vjudge.net/problem/CodeForces-940F

Story. Given an array and a sequence of operations, each one either queries the smallest f ≥ 1 with no value appearing exactly f times in some range, or overwrites one position with a new value.

Abstraction. This is 3D Mo with the double-indexed frequency answer layer. The full template above is the accepted solution. The block size is N^(2/3), the time axis is the third sort key, and the MEX is read in O(√N) per query.

Why Mo fits. The answer is fundamentally a frequency-of-frequency aggregate with no clean associative merge, the updates are point updates that can be made offline, and the queries are range queries on a 1-D array.

Algorithm choice. 3D Mo with cntValue + cntFreq. The MEX scan is bounded by ⌈√N⌉ + 1 so its total contribution across all queries is dominated by the Mo movement cost.

Complexity. O((Q + N) · N^(2/3)) for the Mo engine; the MEX scan adds at most O(Q · √N) which is dominated.

Pitfalls. Compress both the initial values and the update values into a single pool; record each update's oldVal from the running mirror, not from the original initial[]; use N^(2/3) for the block size; place the time loops before the pointer loops.

📝 Final Summary

3D Mo is classic Mo plus one new axis. The state is (curL, curR, curT), the engine has seven loops per query instead of four, and the block size is N^(2/3). The hooks apply_update and undo_update are exact inverses; both patch a[pos] unconditionally but patch the Mo state only when the position is inside the current pointer interval.

Three rules earn most of the correctness:

1. oldVal is the value AFTER previous updates, not from initial[].
2. The position-vs-range check is the only conditional in the hooks.
3. Block size N^(2/3) balances all three axes.
