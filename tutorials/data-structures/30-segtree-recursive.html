Lesson 1 — Standard Recursive Segment Tree

Range Sum · Point Update · The Three Query Cases · 0-Based Heap

🧩 Problem Definition

You are given an array a of n integers, indexed from 0 to n−1. We need a structure that supports the following two operations efficiently:

1. Point Set

Given pos and x, replace a[pos] with x.

2. Range Sum Query

Given l, r with 0 ≤ l ≤ r < n, return a[l] + a[l+1] + ... + a[r].

Build: O(n) Point Set: O(log n) Range Sum: O(log n) Memory: O(n)

Brute force gives O(1) for the point set but O(n) per query. Prefix sums flip this and break on any update. The segment tree is the smallest data structure that makes both operations logarithmic.

💡 Intuition from Scratch

The slow part of a range sum is having to touch every element in [l, r]. If r − l = 10^5 and we have 10^5 queries, that's 10^10 operations — completely infeasible.

Idea: pre-compute partial sums for many intervals, not just one element at a time. Then a query [l, r] is answered by stitching together a small number of pre-computed intervals that together exactly cover it.

But pre-computing for every possible interval costs O(n²) space. We need a smarter choice of intervals.

The segment tree's choice is elegant:

Cover [0, n−1] with one big interval. Split it in half — recursively, down to single elements. Store the sum at every interval we visit.

That gives us about 2n intervals total (a binary tree of leaves and internal nodes), each query needs only O(log n) of them, and each update only changes O(log n) stored sums.

How indexing works

We use a flat array to hold all nodes. Children are at predictable offsets:

0-based heap convention

root         = 0
left  child  = 2 * i + 1
right child  = 2 * i + 2 = left + 1
parent of i  = (i - 1) / 2

Array size

base = smallest power of 2 with base >= n
used nodes = 2 * base - 1
recommended size = 2 * base

n = 5  →  base = 8  →  16 cells
n = 8  →  base = 8  →  16 cells
n = 9  →  base = 16 →  32 cells

The padding to a power of 2 keeps the tree perfect, which makes all index math trivial. Padded leaves store the identity for the operation (0 for sum).

📐 Formal Abstraction

Each node i has a fixed interval [l, r] that is determined entirely by its position in the heap. The root is [0, base−1]. For any non-leaf node i with interval [l, r]:

  • mid = l + (r − l) / 2 (integer division)
  • left child 2i+1 covers [l, mid]
  • right child 2i+2 covers [mid+1, r]

Each node stores a value tree[i]. The contract is:

✅ Node Invariant

tree[i] equals the merge of all real array values whose index lies inside this node's interval. Padded positions (indices ≥ n) contribute the identity for the operation: 0 for sum, +∞ for min, −∞ for max, 0 for GCD.

The set of all node intervals forms a balanced binary tree of intervals. Crucially, every interval [ql, qr] can be expressed as the disjoint union of at most 2 ⌈log₂ n⌉ node intervals — that is the source of all logarithmic bounds.

🔬 Key Observation — The Three Query Cases

When the query function visits a node with interval [l, r] while answering a range query [ql, qr], exactly one of three cases applies:

CaseConditionActionWhy
No overlap qr < l or r < ql Return identity Nothing in this subtree contributes to the answer.
Full overlap ql ≤ l and r ≤ qr Return tree[i] The whole subtree is inside the query, so the cached value is the exact answer for this piece.
Partial overlap Neither of the above Recurse into both children and merge their answers The query boundary cuts through this interval, so we have to look closer.

The recursion shape is: at each level of the tree we visit at most two partial nodes on the boundary; the rest are either fully inside (taken whole) or fully outside (skipped). This is the geometric heart of the O(log n) bound.

⚙️ Algorithm — Three Recursive Routines

All operations are recursive on the heap node i together with its interval [l, r]. The signatures are uniform:

build(i, l, r)

Recursively builds the entire tree. Leaves copy a[l]; internal nodes merge their two children.

point_set(pos, x, i, l, r)

Recurses toward the leaf at pos. Skips the irrelevant subtree. Re-merges from leaf back to root.

query(ql, qr, i, l, r)

Three cases: no-overlap returns identity; full-overlap returns tree[i]; partial recurses both children and merges.

🛡️ Invariants

✅ Invariant — Node Invariant

tree[i] stores the merge of all real array values inside [l, r]. Padded positions contribute identity.

✅ Invariant — Build Invariant

After building children, the parent is recomputed as tree[i] = merge(tree[left], tree[right]). There is no other rule.

✅ Invariant — Update Invariant

After a point update at position pos, every ancestor of the leaf storing pos is recomputed from its children. No other node is touched.

✅ Invariant — Query Invariant

The recursive query returns the merge of all real values that lie inside [l, r] ∩ [ql, qr]. Outside that intersection it returns identity.

🧮 Proof of Correctness

🧮 Proof — O(log n) for Both Update and Query

Point update. The recursion descends one path from root to a leaf. The tree has 1 + log₂ base ≤ 1 + ⌈log₂ n⌉ + 1 levels, and we do O(1) work per level (descend and re-merge). Total: O(log n).

Range query. Call a node relevant if its interval intersects but is not contained in [ql, qr] — these are the partial-overlap nodes. We claim that at every level of the tree there are at most two relevant nodes.

Sketch: at depth d the tree's intervals are a partition of [0, base-1] into equal blocks. Any query interval [ql, qr] crosses each block boundary at most twice (left boundary and right boundary). So at most two blocks at each level can be "cut" by the query. Every other block is either entirely inside (taken whole) or entirely outside (skipped).

With O(log n) levels and ≤ 2 partial nodes per level, the recursion visits O(log n) nodes total. Inside each, we do O(1) work. Total: O(log n). ∎

⏱ Complexity Analysis

Time

build: O(n)

point_set: O(log n)

query: O(log n)

Space

Allocated array: 2 × base ≤ 4n

Recursion stack: O(log n)

Total work for q queries

O((n + q) log n)

Easily handles n, q ≤ 10⁶ within typical 1–2 s limits.

💻 Master Template — Range Sum + Point Set

🧰 Template Philosophy

One pattern, four operations, swappable identity and merge:

  • ID — the identity element for the merge operation (0 for sum, +∞ for min, etc).
  • merge_values(a, b) — combines two children. Must be associative.
  • build / point_set / query — recursive routines that are the same regardless of operation.

To switch from sum to min, change only ID and merge_values. Nothing else.

Full C++ Template click to expand/collapse 
#include <bits/stdc++.h>
using namespace std;

struct SumSegTree {
    using T = long long;

    static constexpr T ID = 0;

    int n = 0;
    int base = 1;
    vector<T> tree;

    SumSegTree() {}

    SumSegTree(const vector<T>& a) {
        init(a);
    }

    static T merge_values(T left_value, T right_value) {
        return left_value + right_value;
    }

    void init(const vector<T>& a) {
        n = (int)a.size();

        base = 1;
        while (base < max(1, n)) {
            base <<= 1;
        }

        tree.assign(2 * base, ID);

        build(0, 0, base - 1, a);
    }

    void build(int i, int l, int r, const vector<T>& a) {
        if (l == r) {
            if (l < n) {
                tree[i] = a[l];
            } else {
                tree[i] = ID;
            }
            return;
        }

        int left = 2 * i + 1;
        int right = left + 1;
        int mid = l + (r - l) / 2;

        build(left, l, mid, a);
        build(right, mid + 1, r, a);

        tree[i] = merge_values(tree[left], tree[right]);
    }

    void point_set(int pos, T value) {
        point_set(pos, value, 0, 0, base - 1);
    }

    void point_set(int pos, T value, int i, int l, int r) {
        if (pos < l || r < pos) {
            return;
        }

        if (l == r) {
            tree[i] = value;
            return;
        }

        int left = 2 * i + 1;
        int right = left + 1;
        int mid = l + (r - l) / 2;

        point_set(pos, value, left, l, mid);
        point_set(pos, value, right, mid + 1, r);

        tree[i] = merge_values(tree[left], tree[right]);
    }

    T query(int ql, int qr) const {
        return query(ql, qr, 0, 0, base - 1);
    }

    T query(int ql, int qr, int i, int l, int r) const {
        if (qr < l || r < ql) {
            return ID;
        }

        if (ql <= l && r <= qr) {
            return tree[i];
        }

        int left = 2 * i + 1;
        int right = left + 1;
        int mid = l + (r - l) / 2;

        T left_answer = query(ql, qr, left, l, mid);
        T right_answer = query(ql, qr, right, mid + 1, r);

        return merge_values(left_answer, right_answer);
    }
};
0-based indexing Heap children = 2i+1, 2i+2 Identity ID = 0 Operation = sum

🔎 Line-by-Line Walkthrough

The constructors and init

SumSegTree(const vector<T>& a) { init(a); }

void init(const vector<T>& a) {
    n = (int)a.size();
    base = 1;
    while (base < max(1, n)) base <<= 1;
    tree.assign(2 * base, ID);
    build(0, 0, base - 1, a);
}

The loop computes the smallest power of two that is at least n. Then we allocate 2 * base cells (used cells are 2*base − 1; the extra cell costs nothing and simplifies bookkeeping). Every cell starts as identity (0) — that is what handles padding automatically: padded leaves will never be overwritten by build's if (l < n) branch.

build — bottom-up via recursion

void build(int i, int l, int r, const vector<T>& a) {
    if (l == r) {                                // leaf
        if (l < n) tree[i] = a[l];
        else       tree[i] = ID;                 // padded leaf
        return;
    }
    int left = 2*i + 1, right = left + 1;
    int mid = l + (r - l) / 2;
    build(left, l, mid, a);
    build(right, mid + 1, r, a);
    tree[i] = merge_values(tree[left], tree[right]);
}

A clean post-order recursion: build children first, then merge into the parent. The base case is a leaf. Note that mid = l + (r-l)/2 avoids the classic overflow bug of (l + r) / 2.

point_set — descend, then re-merge

void point_set(int pos, T value, int i, int l, int r) {
    if (pos < l || r < pos) return;              // pruned: pos not in this subtree
    if (l == r) { tree[i] = value; return; }      // leaf — write the new value
    int left = 2*i + 1, right = left + 1;
    int mid = l + (r - l) / 2;
    point_set(pos, value, left, l, mid);
    point_set(pos, value, right, mid + 1, r);
    tree[i] = merge_values(tree[left], tree[right]);
}

The first line is a pruning guard — only one of the two children will actually contain pos, so the other call returns immediately. After both recursive calls (one of which did real work) we re-merge into tree[i]. Total work: one path from root to leaf, plus O(1) per level for the trivial return on the other side.

query — the three cases

T query(int ql, int qr, int i, int l, int r) const {
    if (qr < l || r < ql) return ID;             // NONE → identity
    if (ql <= l && r <= qr) return tree[i];      // FULL → cached
    int left = 2*i + 1, right = left + 1;         // PARTIAL → recurse both
    int mid = l + (r - l) / 2;
    return merge_values(query(ql, qr, left, l, mid),
                        query(ql, qr, right, mid + 1, r));
}

This is the heart of the structure. The first two early returns capture all the work-saving; the recursive case happens only on the at most two partial-overlap nodes per level that we proved exist.

🔄 Adapt to Min / Max / GCD — Change Two Things

The recursion is operation-agnostic. You only ever need to change two members:

OperationIDmerge_values(a, b)
Sum0a + b
Minnumeric_limits<T>::max() or LLONG_MAXmin(a, b)
Maxnumeric_limits<T>::lowest() or LLONG_MINmax(a, b)
GCD0 (because gcd(x, 0) = x)std::gcd(a, b)
XOR0a ^ b
Product (mod p)1a * b % p

The operation must be associative. It need not be commutative for sum-like trees, but commutativity makes the code symmetric. We will see non-commutative merges in Lesson 2 (max-subarray composite node).

🎛️ Variants & Extensions

Variant 1 — Range Min Query (RMQ), Point Update

What changes: ID = LLONG_MAX, merge_values = min.

Why: identical structure; the segment tree does not care whether we are aggregating with + or with min.

When to use: classic RMQ with updates (Sparse Table is faster for static RMQ but cannot update).

Problem: SPOJ RMQSQ

Variant 2 — Range GCD

What changes: ID = 0, merge_values = std::gcd.

Why: GCD is associative, and gcd(x, 0) = x makes 0 a perfect identity.

When to use: queries about common factors over ranges, dynamic.

Problem: CF 474F — Ant Colony

Variant 3 — Range XOR

What changes: ID = 0, merge_values = ^.

Why: XOR is associative and x ^ 0 = x.

When to use: dynamic XOR queries on subarrays; counting-in-pairs problems.

Variant 4 — Range Max + Index

What changes: the node value becomes pair<T, int> = (value, position). merge_values picks the pair with the larger value (or smaller position on ties).

Why: we need not just the maximum but where it occurs.

When to use: reporting the argmax inside a range, e.g. for sweep-line problems.

Variant 5 — Multiplicative product modulo p

What changes: ID = 1, merge_values = (a * b) % p.

Why: multiplication is associative; 1 is its identity.

When to use: dynamic modular product queries.

Variant 6 — Tuple Aggregate (sum + count + min)

What changes: store a struct { sum, cnt, mn } at each node. Merge component-wise.

Why: some problems need multiple aggregates at once (e.g. sum and minimum for medians).

When to use: custom aggregates; precursor to Lesson 2.

🧯 Edge Cases — Centralized Audit

Edge caseWhat the template does
n = 1base = 1; root is the only node, also the leaf. query(0,0) returns tree[0].
n not a power of 2Padding ensures base is a power of 2; padded leaves hold ID, so any query that reaches them returns identity correctly.
Query full range [0, n-1]Root is fully inside the query; the very first call returns tree[0] in O(1).
Query a single position [pos, pos]Recurses one path; returns the leaf value in O(log n).
Update first or last elementSymmetric; one path from root to that leaf.
Negative valuesSum works as-is. For max-subarray (Lesson 2) the identity must be carefully chosen.
Repeated updates at the same positionEach update overwrites the leaf; ancestors are recomputed. No accumulation drift.
Out-of-range query (ql < 0 or qr ≥ n)Template assumes valid input. Clamp at the caller.
Empty array (n = 0)Avoid this by guarding at the caller; or change init to skip build.

⚠️ Common Pitfalls & Bugs

PitfallSymptomFix
Mixing 0-based and 1-based heap formulasOff-by-one indices; wrong answersPick one convention and stick with it. This course uses 0-based: left = 2i+1, right = 2i+2.
Allocating only 2n cells when n is not a power of 2Out-of-bounds writesAlways allocate 2 × base where base is the next power of two.
Computing mid = (l + r) / 2Overflow when l, r ~ 2³⁰Use mid = l + (r - l) / 2.
Forgetting to re-merge after updateStale ancestor valuesThe last line of point_set must be tree[i] = merge_values(...).
Wrong identityMin returns 0 forever; max returns INT_MAX foreverUse the operation's true identity: +∞ for min, −∞ for max.
Using int for sums of large arraysSilent overflowUse long long when sum may exceed 2³¹.
Forgetting that operation must be associativeRandom failuresIf your merge isn't associative, segment tree won't work — pick a different structure.

🎯 Decision / Recognition Guide

Use this lesson's template when…

  • You need range queries and updates change a single element at a time.
  • The aggregate (sum/min/max/gcd/xor) is associative.
  • You don't need range updates (use Lesson 4 — Lazy Propagation for those).
  • You don't need order statistics inside ranges (use Lesson 6 — Merge Sort Tree).
  • You don't need to keep old versions (use Lesson 5 — Persistent).

Quick reflex rules

"Range sum + point set" → this lesson "Range min/max + point update" → change ID & merge "Range add + range sum" → Lesson 4 (Lazy) "Range chmin/chmax" → Lesson 8 (Beats) "k-th smallest in [l,r]" → Lesson 5 or 6

Common confusion

People sometimes reach for segment tree when a prefix-sum array suffices. If the array is static (no updates), a prefix sum gives O(1) per query in O(n) time — strictly better. Segment tree shines when there are updates.

🏁 Final Summary

The standard recursive segment tree is the foundation of the entire family. Every other variant in this course is built by changing exactly one of the following three things:

  1. What a node stores — a single number for Lesson 1, a composite struct for Lesson 2, a sorted vector for Lesson 6, a separate inner segment tree for Lesson 12.
  2. How children merge — addition for sum, the four-rule combine for max-subarray, multiset union for merging, etc.
  3. What can be applied to a node without recursing — for ordinary trees, only leaf writes; for lazy and beats, much more.

Once you've internalized the three query cases and the heap-based recursion above, every other lesson is "the same machine, with one slot swapped out".
