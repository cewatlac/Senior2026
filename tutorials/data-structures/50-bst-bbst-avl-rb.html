BST → BBST → AVL → Red-Black

From the plain Binary Search Tree to its balanced descendants. Why naive BST collapses on sorted input, how rotations repair the height, and how AVL and Red-Black trees automate that repair with two different bookkeeping schemes.
Theme

📌 01 · Problem Definition

Maintain an ordered set (or multiset) of comparable elements under arbitrary update sequences and answer the standard menu of ordered-set queries efficiently. Specifically, the structure must support insertion of a new element, deletion of an existing element, membership tests, and the order-statistic family of queries: predecessor and successor of a given value, the k-th smallest element, and the rank of a value defined as the number of elements strictly less than it. Every operation should run in O(log n) time, where n is the current size of the set, and the structure must remain robust against any input order that an adversarial user might supply, including sorted or reverse-sorted sequences.

A plain sorted array gives O(log n) search via binary search, but insertion and deletion are O(n) because we must shift the tail. A linked list gives O(1) insertion at a known position but O(n) search. A hash table gives O(1) average insert and lookup but no order-statistic queries at all. The balanced binary search tree resolves the tension by giving O(log n) for the whole menu of operations, including the order-statistic ones.

🧠 02 · Intuition from Scratch

Imagine we want to search for a number in a sorted array. Binary search visits the middle element, compares, and recurses into either the left half or the right half. The recursion tree of binary search is itself a binary tree, where each internal node represents a comparison and the children represent the two possible directions. The binary search tree is the same idea reified into a data structure: we make the comparisons explicit by storing them as nodes, and we maintain the invariant that at every node, everything in the left subtree is smaller and everything in the right subtree is larger.

The advantage over a sorted array is that we no longer need contiguous memory or position-based indexing. Insertion does not have to shift elements; we just walk down the tree to find the right leaf and attach the new node there. Deletion does not have to compact the array; we just rewire pointers locally. As long as the tree stays roughly balanced — meaning its height is close to log of the number of elements — every operation takes the same logarithmic time as binary search on a sorted array, while allowing dynamic updates.

The danger, which the rest of this tutorial spends most of its energy addressing, is that a plain BST does not automatically stay balanced. Adversarial input orders such as inserting elements in already-sorted sequence produce a chain-shaped tree whose height equals n, reducing every operation to O(n). Balanced BSTs add some bookkeeping that detects when the tree is becoming too tall and repairs it through local restructuring called rotations.

🎨 03 · Visual Mental Model

A binary search tree is drawn with the root at the top, smaller values descending to the left, larger values descending to the right. Reading the values in inorder traversal (left subtree, then current node, then right subtree) produces them in sorted order. Below is a small balanced BST for the set {2, 4, 5, 7, 8}, followed by the same five values inserted in sorted order, which produces a worst-case chain. Both trees satisfy the BST invariant, but only the first one supports fast operations.

Two BSTs holding the same five values

The left tree has height 2; any descent from root to leaf visits at most three nodes. The right tree, born from the sorted insertion order 2, 4, 5, 7, 8, has height 4. Even though both store exactly the same set and both satisfy the BST invariant, their performance profiles differ by an entire complexity class. The job of a balanced BST is to refuse to look like the second tree no matter what order the user supplies.

📐 04 · Formal Abstraction

The node and the BST invariant

A binary search tree is a binary tree where every node carries a key drawn from a totally ordered universe. The BST invariant states that for every node, every key in its left subtree compares strictly less than the node's key, and every key in its right subtree compares strictly greater. We will say strictly less and strictly greater throughout; if the structure must accept duplicates as separate elements, we either break ties with a secondary key such as insertion order, or we collapse duplicates into a single node with a frequency counter (the approach the key treap takes in tutorial 27).

For every node u in the tree: every node v in subtree(u.left): v.key < u.key every node v in subtree(u.right): v.key > u.key Consequence: inorder traversal produces the keys in sorted order.

Augmentation: subtree size

A useful augmentation is to store at every node the size of its subtree, which is the number of nodes that descend from it including itself. This single extra integer per node enables the order-statistic queries: the k-th smallest descent uses subtree sizes to decide left or right at each step, and the rank query order_of_key uses subtree sizes to count elements strictly less than a given value during the descent. We will adopt this augmentation throughout the rest of the tutorial.

For every node u in the tree: u.sz = sz(u.left) + 1 + sz(u.right)

When duplicates collapse into a single node with a frequency counter, the formula becomes u.sz = sz(u.left) + u.frq + sz(u.right), and the k-th descent and the rank query both work in terms of occurrences rather than distinct values. This is the form the key treap uses.

🔍 05 · Key Observations

Inorder is the sorted order
The BST invariant is exactly the condition needed for inorder traversal to produce the keys in sorted order. This is the basis of all order-statistic queries.
Every operation walks one root-to-leaf path
Insert, delete, find, predecessor, successor, kth, and rank all descend from the root following a single path that is at most h levels deep, where h is the height of the tree. So the cost of every operation is O(h).
Height can be Θ(n) on adversarial input
A naive BST built by inserting elements in sorted order becomes a chain of length n. Every operation then degrades to O(n), which is no better than a linked list.
Rotations preserve inorder
A rotation is a local restructuring that swaps a parent with one of its children while preserving the BST invariant globally. Rotations are the elementary repair operation used by every balanced BST.
Balanced BSTs maintain height O(log n)
By detecting imbalance after each insert or delete and performing a small number of rotations, balanced BSTs guarantee that the height never grows beyond a constant factor of log n. Every operation then runs in O(log n) worst-case time.

⚙️ 06 · BST Insertion

To insert a key x into a BST, we start at the root and walk downward. At each node we compare x with the node's key. If x is smaller we descend into the left child, otherwise we descend into the right child. When we reach a null pointer, that null pointer is where the new node belongs. We allocate a fresh leaf carrying the key x and link it in place of the null pointer. The descent is at most h levels deep, so insertion costs O(h) time.

The stepper below illustrates inserting the sequence 5, 3, 8, 4, 9 into an initially empty tree. At each step the highlighted node is the one we are currently comparing against, and the green outline marks the newly inserted leaf.

Building a BST by inserting 5, 3, 8, 4, 9

Notice that the final shape of the tree depends entirely on the order in which the keys arrive. Inserting the same five values in the order 3, 4, 5, 8, 9 would have produced a right-leaning chain of depth 4. This is the fundamental fragility of the plain BST: the user, not the data structure, controls the shape.

🗑️ 07 · BST Deletion

Deletion is more involved than insertion because removing an internal node leaves a hole in the tree that has to be filled in a way that preserves the BST invariant. There are three cases to consider, depending on how many children the doomed node has. We walk through them in increasing order of complexity.

Case 1: doomed node is a leaf

If the node we want to delete has no children, we simply detach it by setting its parent's child pointer to null. There is no other rearrangement to do; the BST invariant is preserved trivially because we removed a value from a position that the invariant did not constrain beyond emptiness.

Case 2: doomed node has exactly one child

If the doomed node has one child (whether left or right), we splice the child up into the doomed node's position. That is, we set the doomed node's parent to point at the doomed node's only child instead of at the doomed node itself. The subtree rooted at that child remains a valid BST, and the BST invariant continues to hold with respect to the parent because every key in the surviving subtree was already on the correct side of the parent's key — it was on the correct side of the doomed node's key, and the doomed node's key was on the correct side of the parent's.

Case 3: doomed node has two children

The hard case. We cannot simply splice up either child because doing so would leave the other child without a parent slot. Instead, we replace the doomed node's key with the smallest key in its right subtree (the inorder successor) and then recursively delete that successor from the right subtree. The successor is guaranteed to have at most one child (it is the leftmost node of the right subtree, so it has no left child), so the recursive deletion falls into Case 1 or Case 2 and terminates immediately. A symmetric approach uses the largest key in the left subtree (the inorder predecessor); both choices are equally valid.

BST deletion: the three cases in sequence

🔄 08 · Rotations: The Repair Primitive

A rotation is a local restructuring that moves a child up over its parent without changing the inorder of the tree. Rotations are the elementary repair operation used by every self-balancing BST: AVL trees use rotations to restore the height balance, red-black trees use them to restore the color invariants, and treaps use them to restore the heap-on-priority invariant. Understanding what a rotation does and why it preserves inorder is the foundation for understanding all three.

Right rotation

A right rotation is performed at a node p whose left child is c. After the rotation, c becomes the parent and p becomes c's right child. The middle subtree, which was c's right subtree before the rotation, becomes p's new left subtree. The left rotation is the mirror image: it is performed at a node whose right child becomes the new parent.

Right rotation step-by-step

Why rotation preserves inorder

Consider the right rotation we just visualised. Before the rotation, the inorder of the subtree at p is inorder(A), c, inorder(B), p, inorder(C), where A, B, C are the three subtrees that hang off c and p. After the rotation, the inorder of the subtree at c is inorder(A), c, inorder(B), p, inorder(C) — exactly the same sequence. The BST invariant, which depends only on inorder, is preserved automatically. The rotation has changed the depths of nodes but not their order.

Cost of one rotation

A rotation only rewires three child pointers (and, if we maintain them, three parent pointers and updates to subtree size on the two affected nodes). It is therefore an O(1) operation. Balanced BSTs perform at most a constant number of rotations per insert or delete, so the total work after an update is O(log n) for the descent plus O(1) for the rotations, giving O(log n) overall.

🌲 09 · AVL Tree

The AVL tree is the first self-balancing BST in the literature, published by Adelson-Velsky and Landis in 1962. It maintains a strict height-balance invariant: at every node, the heights of the left and right subtrees differ by at most one. The difference (left height minus right height) is called the balance factor and must always lie in the set {-1, 0, +1}. When an insertion or deletion temporarily makes some node's balance factor equal to +2 or -2, the AVL repair procedure performs one or two rotations at that node to restore the invariant.

Why the height stays O(log n)

The minimum number of nodes in an AVL tree of height h obeys the recurrence N(h) = N(h-1) + N(h-2) + 1, with base cases N(0) = 1 and N(1) = 2. This is essentially the Fibonacci recurrence offset by one, so N(h) grows like φ^h where φ is the golden ratio. Inverting the inequality, an AVL tree of n nodes has height at most about 1.44 log₂(n+2), which means every operation costs O(log n) in the worst case.

The four imbalance cases

After an insertion, exactly one node along the path from the new leaf back to the root may have its balance factor jump to ±2. We call that node u. The repair depends on where the new node was inserted relative to u's grandchildren, giving four cases: left-left (LL), left-right (LR), right-right (RR), and right-left (RL). LL and RR are mirror images of each other and need a single rotation; LR and RL are mirror images and need a double rotation, which is a rotation at the child followed by a rotation at u.

AVL imbalance cases LL, RR, LR, RL

Bookkeeping

Each AVL node stores either the subtree height or just the balance factor; both work, and the choice is mostly a matter of taste. After an insert the rebalance procedure walks back up from the new leaf, updating heights at each ancestor, and stops at the first ancestor whose balance factor reaches ±2 — that is where one of the four repair cases applies. After a delete the rebalance procedure must continue up to the root, because deletions can propagate balance changes upward, occasionally requiring multiple rotations along the path.

🎨 10 · Red-Black Tree

The red-black tree relaxes the strict AVL height balance into a softer color-based invariant. Every node carries a single bit of information, its color, which is either red or black. The five red-black properties guarantee that the longest root-to-leaf path is at most twice as long as the shortest one, which is enough to bound the height by 2 log₂(n+1). The trade-off is that red-black trees are typically taller in practice than AVL trees (because they allow more imbalance), but they rebalance with fewer rotations on average — usually at most three rotations per insert or delete, versus up to log n rotations for AVL.

The five red-black properties

First, every node is either red or black. Second, the root is black. Third, every leaf (the conventional null sentinel that terminates each branch) is black. Fourth, no red node has a red child; equivalently, if a node is red then both of its children are black. Fifth, every path from a given node down to any descendant null leaf contains the same number of black nodes; this number, counted excluding the starting node, is called the black-height of the node.

Why the height is bounded

The fifth property forces every root-to-leaf path to have the same number of black nodes, call it bh. The fourth property forbids two reds in a row, so at most half the nodes on any path can be red. The total path length is therefore at most 2 bh. A standard counting argument shows that any subtree of black-height bh has at least 2^bh − 1 internal nodes, so bh ≤ log₂(n+1) and the total height is at most 2 log₂(n+1).

Why we describe rebalance without showing code

Red-black rebalance has many cases (the textbook treatment has five distinct insert cases and six delete cases), and writing them out fully would consume more space than it deserves in a foundations tutorial. The practical takeaway is that most major C++ standard-library implementations use red-black-tree-like structures for std::set and std::map, although the C++ standard specifies behavior and complexity requirements rather than mandating a specific tree type; either way, any time you reach for these containers you are already benefiting from a balanced ordered structure with logarithmic operations. For contest purposes the relevant facts are the height bound, the choice between AVL and red-black being a constant-factor trade-off, and the fact that the policy-based data structure (PBDS) ordered set is a red-black tree augmented with subtree-size information to support order-statistic queries.

Each balanced BST flavor makes a different trade-off between height, rotation count, and implementation complexity. Below is a quick comparison of the three approaches you will see most often in practice.

For contest use the rule of thumb is straightforward. When the standard library's std::set or std::map already gives you what you need, use it. When you also need order-statistic queries (k-th element or rank-of-value), reach for the policy-based data structure tree from __gnu_pbds, which is a red-black tree internally with subtree sizes attached. When you need anything more exotic, such as set merge and split, range reverse on a sequence, or persistence, hand-roll a treap; the code is shorter than AVL or red-black and the family extends naturally to all the variants covered in tutorials 27 through 29.

📜 13 · Correctness Proofs

For a BST, correctness reduces to showing that every operation preserves the BST invariant on every node. For a balanced BST, we additionally show that the balance invariant is preserved and that the height bound follows from the balance invariant.

BST insertion preserves the invariant

The insertion descent visits each node and goes left if the new key is smaller, right if larger. At the moment the descent reaches a null pointer, every ancestor on the path has been compared with the new key and the descent direction was consistent with the BST invariant. Attaching the new leaf at the null pointer places it in a position where the invariant holds at every ancestor: it is smaller than every ancestor we descended right from and larger than every ancestor we descended left from. The local invariant at the new leaf is vacuous because it has no children.

BST deletion preserves the invariant

Case 1 (leaf) removes a value that the invariant only constrained through emptiness, so the invariant is preserved trivially. Case 2 (one child) splices the surviving subtree up; the BST invariant continues to hold because the surviving subtree was already on the correct side of every ancestor of the doomed node. Case 3 (two children) is handled by reducing it to Case 1 or Case 2 via the inorder successor substitution; the substitution preserves the BST invariant because the successor was the smallest key in the right subtree, hence still strictly greater than every key in the left subtree.

Rotations preserve inorder

We sketched this in section 8. A right rotation at a parent p with left child c transforms a subtree whose inorder is inorder(A), c, inorder(B), p, inorder(C) into a subtree whose inorder is the same string with the same A, B, C, c, p. Because the inorder is unchanged, the BST invariant (which is equivalent to the inorder being sorted) is preserved globally.

AVL height bound

Let N(h) be the minimum number of nodes in an AVL tree of height h. By the balance invariant the two subtrees of the root differ in height by at most one, so the shorter subtree has height at least h − 2 and the taller has height h − 1. The minimum is therefore N(h) = 1 + N(h-1) + N(h-2), with N(0) = 1 and N(1) = 2. This is essentially the Fibonacci recurrence, giving N(h) ≥ φ^h for the golden ratio φ. Inverting, h ≤ log_φ(n) ≈ 1.44 log₂ n.

Red-Black height bound

By the black-height counting argument we sketched in section 10, the height of a red-black tree of n internal nodes is at most 2 log₂(n+1). The fact that red-black rebalancing performs at most three rotations per update is a non-trivial case analysis that we omit; it is covered in any standard textbook treatment.

🧩 14 · Edge Cases (Centralized Audit)

A consolidated list of the boundary conditions every BBST implementation must handle correctly. Most contest bugs in tree code trace back to missing one of these.

  • Empty tree. The root is null. Find returns nothing, insert creates the root, delete returns immediately.
  • Insert into empty tree. The new node becomes the root and may or may not need to be coloured (red-black) or have its balance factor initialised (AVL).
  • Delete the root. Special care because the root has no parent; the rebalance must update the root pointer afterward.
  • Insert a duplicate. Either reject, attach as a right child (treating equal as greater), or bump a frequency counter — pick one convention and document it.
  • Delete a key that is not present. Walk to a null pointer and return; the tree is unchanged.
  • Delete a node with two children where the inorder successor is the immediate right child. The recursive successor-substitution must still update the right pointer correctly; an unguarded swap can leave a dangling pointer.
  • kth(k) with k out of range. Either return a sentinel value or assert; never silently walk off the tree into a null child.
  • Predecessor of the minimum or successor of the maximum. Returns "nothing"; callers must check.
  • All keys identical when duplicates are stored. The tree degenerates to a chain or to a single node with a large frequency; either is fine but tests must cover it.
  • Insert-then-immediately-delete the same key. Memory must be freed; pointers must not dangle.
  • Long sorted-insert sequence into a plain BST. Produces a chain. Useful as a stress test to verify that a balanced BST stays shallow under this adversary.

🪲 15 · Common Pitfalls and Bugs

Pitfall 1 — Forgetting to update the parent pointer after a rotation
A rotation rewires three child pointers but also two parent pointers, plus a possible update at the grandparent's child slot. Any one of these forgotten leaves the tree in an inconsistent state that may not crash until much later, when an unrelated descent walks through a stale link.
Pitfall 2 — Forgetting to update subtree size or height after every mutation
If you maintain subtree size for order-statistic queries or height for AVL balance, every place that changes a child pointer must also update the parent's augmentation. Routing all mutations through a single link(p, c, dir) helper that also calls pull(p) is the cleanest fix.
Pitfall 3 — Using ≤ in BST descent when you intended distinct keys
If your descent goes left on x ≤ u->key instead of strict less-than, equal keys go to the left subtree and the resulting tree no longer satisfies the strict BST invariant. Subsequent operations may produce wrong answers in subtle ways. Pick a convention and stick to it everywhere.
Pitfall 4 — Deleting the inorder successor without removing the original
In Case 3 of BST deletion, after copying the successor's key into the doomed node, you must recursively delete the successor from the right subtree. Forgetting the second step leaves the successor's value in two places at once, breaking the invariant immediately.
Pitfall 5 — Stack overflow on deep recursion
A plain BST on adversarial sorted input has depth n. Recursive operations may blow the stack on n = 10^6 even though the algorithm is technically O(n). Either iterate, or switch to a balanced BST so the height stays O(log n).
Pitfall 6 — Confusing the inorder successor with the in-place "next"
For Case 3 we want the smallest key in the right subtree, which is found by going right once and then left until a null is reached. A surprisingly common bug is to go left immediately instead, which returns either the doomed node itself or a node from the wrong subtree.
Pitfall 7 — Returning a stale pointer after rebalance
After a rotation at a node u, the new root of the local subtree may be different from u. Recursive insert and delete must return the new root so the caller can update its child pointer. Returning u blindly leaves the parent pointing at the wrong subtree root.

🧭 16 · Decision Guide

Which balanced BST should I reach for?

just insert / find / lower_bound ⇒ std::set (typically a balanced BBST, red-black-like in major implementations)
kth-smallest or rank-of-value ⇒ PBDS tree or hand-rolled key treap
multiset duplicates with frequency counts ⇒ key treap with frq (tutorial 27)
sequence operations (insert at position, range reverse) ⇒ implicit treap (tutorial 28)
historical versions / persistence ⇒ persistent treap (tutorial 29)
need set merge / split ⇒ treap with split / merge primitives
teaching / writing from scratch ⇒ AVL is the cleanest to derive; treap is the shortest to code

🎯 18 · Final Summary

One sentence: a binary search tree gives O(h) for every ordered-set operation, where h depends on insertion order; balanced BSTs use rotations to keep h at O(log n) regardless of input.

Five things to remember

First, the BST invariant says left subtree is strictly smaller, right subtree is strictly larger, and inorder traversal is sorted. Second, every operation walks one root-to-leaf path, so the cost is exactly O(h). Third, the height of a plain BST is whatever the user's input order makes it; sorted input gives a chain of height n. Fourth, rotations are the elementary repair operation: they preserve inorder while changing depths. Fifth, AVL maintains strict height balance ±1 with a tighter height bound, red-black uses softer color invariants with fewer rotations, and treap uses random priorities to make balance probabilistic — each is a different way to enforce O(log n) height.

Mastery checklist

  • I can write BST insert, find, and the three cases of delete from scratch in under ten minutes.
  • I can perform a right rotation by hand on a small tree and verify that inorder is preserved.
  • I can recognise which of the four AVL imbalance cases applies given the position of the new leaf.
  • I can recite the five red-black properties and explain why they bound the height by 2 log₂(n+1).
  • I know when to reach for std::set, when for PBDS, and when to hand-roll a treap.
  • I can prove that BST insertion and the three deletion cases preserve the BST invariant.
