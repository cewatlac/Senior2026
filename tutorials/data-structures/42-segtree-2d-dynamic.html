Lesson 13 — 2D Dynamic Sparse Segment Tree

Huge 2D Coordinates · Lazy Allocation in Both Dimensions · O(log² U) per Op

🧩 Problem Definition

Coordinates can be as large as U = 10⁹ in both dimensions, but the number of operations is at most ~10⁵ or ~10⁶. Support:

point_add(r, c, v)

Add v to the cell (r, c).

rectangle_sum(r1, c1, r2, c2)

Return the sum of all cells inside the rectangle.

Allocating a dense U × U grid is impossible. The 2D dynamic sparse segment tree materializes outer and inner nodes only when they are touched by an update — memory grows with the number of operations, not with U.

💡 Intuition from Scratch

Take the dense 2D segment tree from Lesson 12 and apply Lesson 7's lazy-allocation discipline twice: once to the outer (row) tree, once to each inner (column) tree. The outer tree starts with just a root; when an update at (r, c) arrives, it walks the outer path from root to outer-leaf-r, materializing outer nodes as needed. At each outer node it visits, it also materializes (if needed) that node's inner tree's root, then performs a 1D dynamic sparse update at column c in that inner tree.

The result: each point_add allocates at most O(log² U) nodes total — O(log U) outer nodes, each contributing O(log U) inner nodes. With N updates, total memory is O(N log² U).

🌳 Visual Mental Model

Two pools. The outer pool holds row-tree nodes. Each outer node has a child pointer to its inner tree's root (in the inner pool), along with the usual left/right outer children pointers.

struct OuterNode { int innerRoot = -1; // index into inner pool, -1 = inner ghost int outerLeft = -1; int outerRight = -1; }; struct InnerNode { long long sum = 0; int innerLeft = -1; int innerRight = -1; };

When an outer node's innerRoot is -1, the rectangle sum over that outer node's row band is zero — both the outer node itself and its inner tree are ghosts. Queries can short-circuit there.

📐 Formal Abstraction

The 2D structure is the composition of two 1D dynamic sparse trees. Conceptually:

(a) Build a 1D dynamic sparse tree over rows, with universe [0, U_R − 1]. Each row-tree node owns an inner column tree.

(b) Each inner column tree is a 1D dynamic sparse tree over columns, with universe [0, U_C − 1].

A 2D point update materializes an outer path of length O(log U_R) and, at every outer node on the path, materializes an inner path of length O(log U_C).

A 2D rectangle query descends the outer tree as usual; for every fully-row-covered outer node, queries its inner column tree over the column range.

🔬 Key Observations

Observation 1 (memory). One point_add allocates at most O(log U_R · log U_C) total nodes — O(log U_R) outer plus up to O(log U_C) inner per outer node on the path.

Observation 2 (query cost). A rectangle query touches O(log U_R) outer nodes. At each fully-row-covered outer node, it descends the inner tree for O(log U_C) work. Partial-row outer nodes just recurse. Total: O(log U_R · log U_C).

Observation 3 (ghosts short-circuit twice). If an outer node is a ghost, the entire rectangle's contribution from that row band is zero — no inner descent. If an inner node along an inner descent is a ghost, contribute zero for that column-band × row-band.

⚙️ Algorithm

point_add(r, c, v)

Recurse on the outer tree, materializing outer nodes along the row path. At every outer node visited (including all those on the path, not just the leaf), perform a 1D point_add at column c with value v on that outer node's inner tree, materializing the inner tree's root and path as needed. After all the inner updates, the inner sums of every outer node on the path correctly reflect v being added at (r, c).

rectangle_sum(r1, c1, r2, c2)

Outer recursive descent: no-row-overlap returns 0; full-row-overlap returns the inner-column-sum from c1 to c2 on this outer node's inner tree (which is a recursive 1D dynamic sparse query). Partial recurses into both outer children.

🛡️ Invariants

✅ Invariant — Outer-ghost-is-identity invariant

If outerLeft or outerRight is -1, the entire row band on that side is empty (no cells with non-zero value). The rectangle sum for any rectangle confined to that row band is zero.

✅ Invariant — Inner-ghost-is-identity invariant

Within an inner tree, a ghost child means the column sub-band has no contribution. The 1D dynamic sparse rules apply unchanged.

✅ Invariant — Per-outer-node consistency

For every outer node i with row range [r1, r2], the inner tree rooted at i.innerRoot represents the multiset of cells with row in [r1, r2], keyed by column.

✅ Invariant — Update materializes only the path

A point_add(r, c, v) materializes at most O(log U_R) outer nodes (the row path) and, at each, at most O(log U_C) inner nodes (the column path). Off-path nodes in both dimensions are untouched.

🧮 Proof of Correctness

🧮 Proof — Why the 2D dynamic structure returns correct rectangle sums

By Lesson 12's argument, a rectangle query is decomposed by the outer descent into O(log U_R) disjoint fully-row-covered outer sub-bands. For each sub-band, the inner column query over [c1, c2] returns the correct sum of cells with row in that sub-band and column in [c1, c2], by Lesson 7's argument applied to the inner tree.

The dynamic part adds: at no-overlap outer nodes (or outer ghost), we return 0. The Outer-ghost invariant guarantees this is exactly the contribution of the empty band. The inner descent uses the Inner-ghost invariant the same way. Each layer's correctness is independent and they compose. ∎

For U_R = U_C = 10⁹, both logs are ~30, giving roughly 900 work units per operation. With N = 10⁵ updates, total memory is up to ~10⁷ nodes — manageable. For more updates, switch to offline coordinate compression + dense 2D BIT.

🧰 Core Template Philosophy

The 2D dynamic sparse segment tree is exactly “a 1D dynamic sparse tree (Lesson 7) over rows, with each outer node owning a 1D dynamic sparse tree (Lesson 7) over columns.” The discipline is to compose the two layers cleanly: when the outer code needs to update or query a particular outer node's inner tree, it delegates to the inner 1D routines.

In practice, this means writing two separate node pools and two separate sets of 1D routines (or templating one over the other). Either way, the outer routines call into the inner routines at the right moments.

💻 Master Template (C++)

#include <bits/stdc++.h>
using namespace std;

struct SegTree2D_DynamicSparse_Sum {
    using T = long long;
    using Coord = long long;

    struct InnerNode {
        T sum = 0;
        int left = -1;
        int right = -1;
    };
    struct OuterNode {
        int innerRoot = -1;
        int outerLeft  = -1;
        int outerRight = -1;
    };

    Coord loR, hiR, loC, hiC;
    vector<OuterNode> outerPool;
    vector<InnerNode> innerPool;
    int root = 0;

    SegTree2D_DynamicSparse_Sum(Coord lr, Coord hr, Coord lc, Coord hc)
        : loR(lr), hiR(hr), loC(lc), hiC(hc) {
        outerPool.push_back({});
    }

    int innerNew() {
        int u = (int)innerPool.size();
        innerPool.push_back({});
        return u;
    }

    // SAFE: do NOT take an int& into the pool here.  innerNew() calls
    // push_back which may reallocate the vector and invalidate any prior
    // reference into innerPool[*].  Instead, we re-read the parent slot
    // by index AFTER the allocation and store the new child id then.
    int materializeInnerLeft(int parent) {
        if (innerPool[parent].left == -1) {
            int child = innerNew();           // may reallocate innerPool
            innerPool[parent].left = child;   // re-read parent by index
        }
        return innerPool[parent].left;
    }
    int materializeInnerRight(int parent) {
        if (innerPool[parent].right == -1) {
            int child = innerNew();
            innerPool[parent].right = child;
        }
        return innerPool[parent].right;
    }

    void inner_add(int u, Coord lo, Coord hi, Coord c, T v) {
        if (lo == hi) { innerPool[u].sum += v; return; }
        Coord mid = lo + (hi - lo) / 2;
        if (c <= mid) {
            int ch = materializeInnerLeft(u);
            inner_add(ch, lo, mid, c, v);
        } else {
            int ch = materializeInnerRight(u);
            inner_add(ch, mid + 1, hi, c, v);
        }
        T ls = innerPool[u].left  == -1 ? 0 : innerPool[innerPool[u].left ].sum;
        T rs = innerPool[u].right == -1 ? 0 : innerPool[innerPool[u].right].sum;
        innerPool[u].sum = ls + rs;
    }
    T inner_sum(int u, Coord lo, Coord hi, Coord c1, Coord c2) const {
        if (u == -1 || c2 < lo || hi < c1) return 0;
        if (c1 <= lo && hi <= c2) return innerPool[u].sum;
        Coord mid = lo + (hi - lo) / 2;
        return inner_sum(innerPool[u].left,  lo,     mid, c1, c2)
             + inner_sum(innerPool[u].right, mid + 1, hi,  c1, c2);
    }

    int outerNew() {
        int u = (int)outerPool.size();
        outerPool.push_back({});
        return u;
    }

    // SAFE: same discipline as the inner side.  Never hold an int& into
    // outerPool across outerNew() / innerNew() calls.
    int materializeOuterLeft(int parent) {
        if (outerPool[parent].outerLeft == -1) {
            int child = outerNew();           // may reallocate outerPool
            outerPool[parent].outerLeft = child;
        }
        return outerPool[parent].outerLeft;
    }
    int materializeOuterRight(int parent) {
        if (outerPool[parent].outerRight == -1) {
            int child = outerNew();
            outerPool[parent].outerRight = child;
        }
        return outerPool[parent].outerRight;
    }
    int materializeInnerRoot(int outerIdx) {
        if (outerPool[outerIdx].innerRoot == -1) {
            int ir = innerNew();              // may reallocate innerPool
            outerPool[outerIdx].innerRoot = ir;
        }
        return outerPool[outerIdx].innerRoot;
    }

    void point_add(Coord r, Coord c, T v) {
        point_add_outer(root, loR, hiR, r, c, v);
    }
    void point_add_outer(int u, Coord lo, Coord hi, Coord r, Coord c, T v) {
        int ir = materializeInnerRoot(u);
        inner_add(ir, loC, hiC, c, v);
        if (lo == hi) return;
        Coord mid = lo + (hi - lo) / 2;
        if (r <= mid) {
            int ch = materializeOuterLeft(u);
            point_add_outer(ch, lo, mid, r, c, v);
        } else {
            int ch = materializeOuterRight(u);
            point_add_outer(ch, mid + 1, hi, r, c, v);
        }
    }

    T rectangle_sum(Coord r1, Coord c1, Coord r2, Coord c2) const {
        return rectangle_sum_outer(root, loR, hiR, r1, c1, r2, c2);
    }
    T rectangle_sum_outer(int u, Coord lo, Coord hi,
                          Coord r1, Coord c1, Coord r2, Coord c2) const {
        if (u == -1 || r2 < lo || hi < r1) return 0;
        if (r1 <= lo && hi <= r2) {
            return inner_sum(outerPool[u].innerRoot, loC, hiC, c1, c2);
        }
        Coord mid = lo + (hi - lo) / 2;
        return rectangle_sum_outer(outerPool[u].outerLeft,  lo,     mid,
                                    r1, c1, r2, c2)
             + rectangle_sum_outer(outerPool[u].outerRight, mid + 1, hi,
                                    r1, c1, r2, c2);
    }
};

🔎 Line-by-Line Explanation

Two pools, one each click to expand/collapse

outerPool holds row-tree nodes; each carries an innerRoot pointer into innerPool, plus the usual outer children pointers. innerPool holds plain 1D column-tree nodes with their sum and inner children pointers. Both pools start with the necessary root: the constructor pushes the outer root, and inner roots are allocated lazily.

point_add_outer — climb both axes click to expand/collapse

At every outer node on the row path, we materialize the outer node's inner-tree root if needed, then perform a 1D inner-tree point_add at column c. After the inner update, we move down the outer tree to the appropriate child. The total work per point_add is O(log U_R · log U_C).

rectangle_sum_outer — short-circuit on outer ghosts and at full-row click to expand/collapse

If the outer node is a ghost (u == -1) or has no overlap with [r1, r2], we contribute zero. If it's fully covered in the row range, we delegate to inner_sum for the column range — this is where the inner descent happens. Otherwise we recurse into both outer children.

materializeInnerLeft / materializeOuterLeft (and the right-side counterparts) click to expand/collapse

The materialize helpers look up a child slot by parent index and side, not by an int& reference into the pool vector. This matters because innerNew() and outerNew() both end with push_back, which can reallocate the underlying buffer and invalidate any reference that pointed into it. If we had written materializeInnerChild(int& slot) and called slot = innerNew();, the write to slot would target the old (freed) memory, silently dropping the new child id. The safe pattern is: allocate first (which may reallocate), then re-read the parent by index and store the new child id into innerPool[parent].left / .right.

inner_add / inner_sum — exactly Lesson 7 click to expand/collapse

The inner 1D routines are unchanged from Lesson 7. They take a node index (or -1 for ghost) and descend the column tree, materializing only what's needed during inner_add and short-circuiting at ghosts during inner_sum.

⚠️ Common Pitfalls

⚠️ Pitfall — Holding an int& into a vector across push_back

Do not pass child slots by reference if creating the child can push_back on the same vector. A pattern like materializeChild(int& slot){ if(slot == -1) slot = innerNew(); } looks innocent, but innerNew() calls push_back on innerPool, which may reallocate and invalidate slot before the assignment runs. The write then lands on freed memory, the parent never sees the new child id, and the next query returns 0. Safe pattern: store the parent index and child direction; allocate first; then write the new id back via innerPool[parent].left/right.

⚠️ Pitfall — Forgetting to materialize an inner root before inner_add

If you call inner_add(outerPool[u].innerRoot, ...) directly without materializing, you may pass -1 as the first arg — undefined behavior on first access. Always materialize then add.

⚠️ Pitfall — Off-by-one in the row/column dispatch

Easy to swap row and column variables, especially when adapting from a 1D template. Adopt a clear convention (outer = row, inner = column) and stick with it.

⚠️ Pitfall — Materializing during a query

Query routines must never grow the pool. Test for u == -1 (both outer and inner) at the top of every recursive query call and return 0.

⚠️ Pitfall — Not reserving the pools

Vector reallocations during operations destroy locality and can cause measurable slowdowns. Reserve roughly N · log² U for each pool up front. (Note: reserve is an optimization, not the actual fix for the int& trap above — the correctness fix is to remove the unsafe reference pattern.)

⚠️ Pitfall — Using 32-bit Coord with large universes

Use long long for both coordinates and midpoint computation. Otherwise lo + hi overflows for U ≥ 2³¹.

🎛️ Variants & Extensions

Variant 1 — 2D persistent dynamic

Apply path copying to both the outer and inner trees. Memory per update is O(log² U); supports historical queries. Used in some range-of-snapshots problems.

Variant 2 — 2D merging

Combine the merging idea (Lesson 9) with 2D. Used for “each subtree on a tree, build a 2D histogram of (depth, color) and merge children's 2D structures into parent's”.

Variant 3 — K-D-tree alternative

For some workloads, a K-D tree is simpler and faster. Note that K-D-tree rectangle queries are O(√N) in the worst case, not O(log² U) — so the segment tree wins when N is large and the rectangles are unconstrained.

Variant 4 — 2D dynamic with lazy

Lazy on the outer tree (for row-band updates) is tractable; lazy on the inner tree alone is rare. Beware: lazy + persistence + dynamic + 2D is research-grade.

🏁 Final Summary — End of the Segment Tree Course

The 2D dynamic sparse segment tree is the natural composition of the dynamic sparse pattern (Lesson 7) with the 2D pattern (Lesson 12). Each axis materializes nodes only when touched, so memory scales with the number of operations rather than with the coordinate universe. Each point_add and rectangle_sum costs O(log U_R · log U_C).

Looking back at the whole course

We started in Lesson 1 with the recursive segment tree, the core idea that drives everything else: associate each node with an interval and a precomputed summary, and decompose any range into O(log n) full-overlap pieces. From there, each lesson layered on one new idea: composite nodes (L2), iterative implementation (L3), lazy propagation (L4), persistence (L5), per-node sorted vectors (L6), lazy allocation (L7), the three-case beats discipline (L8), tree merging (L9), bit-level permutations (L10), the carry pattern for queries (L11), nested trees for 2D (L12), and finally the two-level dynamic structure of this lesson.

Every variation rests on the original abstraction: a binary partition of the universe into intervals, with summaries that compose. Once you internalise this skeleton — full overlap returns the summary, no overlap returns identity, partial recurses with possible push/pull — the rest is choosing what to store, when to clone, when to materialise, when to push, when to break. The pattern is one of the most general data-structure patterns in competitive programming; it shows up far beyond just “range sum”.

You now have a complete map of the segment-tree landscape. The natural next step is practice: pick a hard problem from any lesson's list, identify which segment-tree variant fits, and implement it from this course's templates. After ten or fifteen such problems, the variants become muscle memory and the choice of which variant to use becomes immediate.
