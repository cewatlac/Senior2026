Lesson 11 — Persistent Lazy Segment Tree

Path Copying · Range Updates · Carry Pattern · Past Versions Stay Intact

🧩 Problem Definition

Given an array a[0..n−1], support an online sequence of operations where each operation either:

Modifying op (creates a new version)

range_add(version_idx, ql, qr, x): produce a new version equal to version_idx after applying +x to every element of [ql, qr].

Read-only op (any version)

range_sum(version_idx, ql, qr): return the sum over [ql, qr] in the snapshot identified by version_idx.

Past versions remain queryable forever. Combining lesson-4 lazy propagation with lesson-5 path copying gives this in O(log n) per operation and O(N · log n) total memory for N modifying operations.

💡 Intuition from Scratch

Plain lazy propagation mutates two things during an update: the sum/state of the node and its lazy tag. To make the structure persistent, no mutation is allowed on existing (published) nodes — both must happen on freshly cloned nodes. The pattern is Lesson 5's path-copying applied to a node-state that now includes a lazy tag.

For updates, this is straightforward: every node on the path is cloned before being modified. Before a partial-cover descent, the parent's pending lazy must also be pushed into freshly cloned children — otherwise the later pull would lose it. So the update does push, but only into clones.

The harder question is the query. In Lesson 4, the query had to push pending tags down before recursing partially. But a query that pushed would mutate published nodes — corrupting older versions. The standard solution is the carry pattern: instead of pushing tags into the nodes, we carry them as a function argument while descending. The carry accumulates each ancestor's lazy tag along the query path, and contributes to the running sum at every full-overlap node.

The carry pattern (for queries) and push-into-clones (for updates) together are the discipline of persistent lazy. Once both clicks, the rest is mechanical.

🌳 Visual Mental Model

Every node now stores both a state field (e.g. sum) and a lazy tag (e.g. add_lazy). The state is what would be visible if we pushed all ancestors' tags down to and including this node — it is correct for the node's interval. The lazy tag is what is still owed to the node's children.

An update follows the path-copying pattern: for each level of the descent, if the node is fully inside the query range, we clone it once and apply the operation directly on the clone (no recursion). Otherwise, we clone the current node, push its pending lazy into freshly cloned left/right children, recurse into both (each may itself clone or share), update the clone's pointers, and re-pull. Off-path subtrees are shared between old and new versions.

A query follows the carry pattern: at every recursion level, the function receives an extra argument carry (cumulative pending add inherited from ancestors that contain this node's interval). At a full-overlap node, the contribution is node.sum + carry · (interval length). At a partial node, the function recurses into both children passing carry + node.add_lazy to each, never modifying the node.

📐 Formal Abstraction

Nodes live in a single pool. Each node has:

struct Node { long long sum; // correct for this interval given ancestors' applied tags long long add_lazy; // pending add owed to children, not yet pushed int left = -1; // pool index of left child int right = -1; // pool index of right child };

Update primitives: clone(u) allocates a fresh copy of node u; apply_add(u, l, r, x) directly modifies u (we only call it on fresh clones); pull(u) recomputes sum[u] from children.

Query primitive: sum_with_carry(u, l, r, ql, qr, carry) returns the sum of [ql, qr] over the subtree rooted at u, given that all ancestors contribute a total pending add of carry per element to this subtree. The function reads u but never writes to it.

🔬 Key Observations

Observation 1 (clone-on-write). Every update materializes O(log n) fresh nodes along its path. Off-path subtrees are shared. Memory grows by O(log n) per update.

Observation 2 (no mutation during queries). The carry pattern lets queries respect immutability. By passing the cumulative pending add as an argument, we avoid having to push tags into the (frozen) tree.

Observation 3 (full-overlap formula). At a full-overlap node u reached with carry c, the contribution is sum[u] + c · (r − l + 1). The node's own add_lazy is already “baked into” sum[u] for this node's interval — only ancestors' tags need to be carried.

Observation 4 (carry composes). When recursing past a partial node with its own pending add_lazy, both children receive carry + add_lazy[u]. The carry simply accumulates.

⚙️ Algorithm

range_add(prev_root, ql, qr, x)

Returns the root of a new version. Recurse with clone-on-write:

  1. If [l, r] is disjoint from [ql, qr], return prev_u unchanged (no clone).
  2. Allocate u as a copy of prev_u.
  3. If [l, r] is fully inside [ql, qr], call apply_add(u, l, r, x) and return u.
  4. Otherwise call push(u, l, r) first — this clones u's children, applies u's pending add_lazy to the clones, and clears u.add_lazy. Then recurse into both freshly cloned children, updating u.left and u.right; finally pull(u).

Why push is mandatory before partial descent. If u carries a pending lazy from a previous full-cover update, and we now do a partial update that recurses into u's children, the children's sum values do not yet reflect u's pending lazy. After the partial recursion, pull(u) overwrites u.sum with sum[left] + sum[right], which silently loses u's own full-cover contribution. Push fixes this by transferring the pending lazy into freshly cloned children before any partial recursion. Old versions remain intact because we only ever mutate freshly cloned nodes — the original children are never touched.

range_sum(root, ql, qr)

Call sum_with_carry(root, 0, base − 1, ql, qr, 0). The carry starts at 0 (no ancestor above the root). Recursion: at no-overlap return 0; at full-overlap return sum[u] + carry · (r − l + 1); at partial recurse with carry + add_lazy[u]. Queries themselves never mutate, so the carry covers the case where a full-cover ancestor still holds add_lazy that has not (yet) been pushed by any update.

🛡️ Invariants

✅ Invariant — Immutability invariant

Once a node has been published (its index appears as a root or as a child of a published node), its fields are never modified. All changes happen via cloning.

✅ Invariant — Sum invariant (per node)

For every node u with interval [l, r], sum[u] is correct for [l, r] assuming all updates that fully covered this node have been applied to u, but not any updates that fully covered an ancestor without fully covering u. The latter are tracked only in the ancestor's add_lazy and contribute via the carry.

✅ Invariant — Carry invariant (during a query)

When the query descends into a subtree rooted at u with carry value c, the actual answer for that subtree includes an extra c per element on top of the node's stored sum. The recursion passes c + add_lazy[u] down to children.

✅ Invariant — Push-on-partial-update invariant

Whenever an update needs to descend partially into a node u, the node is first cloned and then its pending add_lazy is pushed into freshly cloned children via apply_add; u.add_lazy is then cleared. This is the only time the structure mutates lazy state, and it always happens on a freshly cloned (unpublished) node, preserving the immutability of older versions.

✅ Invariant — Queries never push

Range-sum queries are read-only. They use the carry pattern to handle any add_lazy still sitting on an ancestor along the query path, without mutating any node. This is essential — pushing during a query would corrupt older versions.

🧮 Proof of Correctness

🧮 Proof — Why the carry pattern returns the same value as plain lazy

Compare the persistent lazy tree to a hypothetical plain lazy tree built by replaying all updates non-persistently. Define the “effective value” of element i in the plain tree as the value a[i] would have if every pending lazy tag in the plain tree were pushed all the way down to its leaf.

Claim: in the persistent tree, the effective value of i equals sum[leaf_i] + (Σ add_lazy[ancestor] over ancestors of leaf_i). By induction on the number of updates, every modifying operation preserves this property because applying +x to a fully-covered node u adds x to sum[u] (and conceptually x to add_lazy[u] for children, but in our convention we choose to add it to sum[u] directly for u's own interval and only record x in add_lazy[u] for descendants).

The carry pattern reproduces the sum Σ_{i ∈ [ql, qr]} effective(i) level by level. At a full-overlap node u, the contribution is sum[u] + carry · (r − l + 1), which equals Σ effective(i) for i ∈ [l, r]: sum[u] covers the node's own tags and the lower-level pending tags within the subtree, and carry covers ancestors' pending tags. ∎

🧰 Core Template Philosophy

Persistent lazy = Path copying (Lesson 5) + Lazy state on nodes (Lesson 4) + Carry pattern for queries. The single most important discipline is never mutate during a query. The carry pattern is the standard solution: turn the “push” step into a function argument.

For updates, the discipline is clone before writing. Every node that the update touches in a fully-covered way is cloned; every node that the update merely passes through (partial overlap) is cloned and its children pointers are updated to the new cloned children.

💻 Master Template (C++)

#include <bits/stdc++.h>
using namespace std;

struct PersistentLazyAddSumSegTree {
    using T = long long;

    struct Node {
        T sum = 0;
        T add_lazy = 0;
        int left = -1;
        int right = -1;
    };

    int n = 0;
    int base = 1;
    vector<Node> pool;
    vector<int> roots;   // roots[v] = root index of version v

    void init(const vector<T>& a) {
        n = (int)a.size();
        base = 1;
        while (base < max(1, n)) base <<= 1;
        pool.reserve(max(4 * base, 4 * 1024));
        roots.clear();
        roots.push_back(build(0, base - 1, a));
    }

    int build(int l, int r, const vector<T>& a) {
        int u = (int)pool.size();
        pool.push_back({});
        if (l == r) {
            if (l < (int)a.size()) pool[u].sum = a[l];
            return u;
        }
        int mid = l + (r - l) / 2;
        int L = build(l, mid, a);
        int R = build(mid + 1, r, a);
        pool[u].left = L;
        pool[u].right = R;
        pool[u].sum = pool[L].sum + pool[R].sum;
        return u;
    }

    int clone(int prev) {
        int u = (int)pool.size();
        pool.push_back(pool[prev]);
        return u;
    }

    void apply_add(int u, int l, int r, T x) {
        pool[u].sum += x * (T)(r - l + 1);
        pool[u].add_lazy += x;
    }

    void pull(int u) {
        int L = pool[u].left, R = pool[u].right;
        pool[u].sum = pool[L].sum + pool[R].sum;
    }

    // Push u's pending add_lazy down to freshly cloned children.
    // CRITICAL: must be called before any partial descent into u, otherwise
    // a later pull() would discard u's own pending lazy contribution.
    // u must already be a freshly cloned node owned by the new version.
    void push(int u, int l, int r) {
        if (pool[u].add_lazy == 0) return;
        if (l == r) { pool[u].add_lazy = 0; return; }
        int mid = l + (r - l) / 2;
        int newL = clone(pool[u].left);
        int newR = clone(pool[u].right);
        apply_add(newL, l,     mid, pool[u].add_lazy);
        apply_add(newR, mid+1, r,   pool[u].add_lazy);
        pool[u].left  = newL;
        pool[u].right = newR;
        pool[u].add_lazy = 0;
    }

    int range_add(int prev_root, int ql, int qr, T x) {
        int newRoot = range_add(prev_root, 0, base - 1, ql, qr, x);
        roots.push_back(newRoot);
        return newRoot;
    }

    int range_add(int prev, int l, int r, int ql, int qr, T x) {
        if (qr < l || r < ql) return prev;
        int u = clone(prev);
        if (ql <= l && r <= qr) {
            apply_add(u, l, r, x);
            return u;
        }
        // PARTIAL descent: push u's pending lazy into fresh cloned children
        // BEFORE recursing, so pull() does not lose the parent's full-cover lazy.
        push(u, l, r);
        int mid = l + (r - l) / 2;
        pool[u].left  = range_add(pool[u].left,  l,      mid, ql, qr, x);
        pool[u].right = range_add(pool[u].right, mid + 1, r,  ql, qr, x);
        pull(u);
        return u;
    }

    T range_sum(int root, int ql, int qr) const {
        return range_sum(root, 0, base - 1, ql, qr, 0);
    }

    T range_sum(int u, int l, int r, int ql, int qr, T carry) const {
        if (qr < l || r < ql || u == -1) return 0;

        if (ql <= l && r <= qr) {
            return pool[u].sum + carry * (T)(r - l + 1);
        }

        int mid = l + (r - l) / 2;
        T newCarry = carry + pool[u].add_lazy;

        return range_sum(pool[u].left,  l,      mid, ql, qr, newCarry)
             + range_sum(pool[u].right, mid + 1, r,  ql, qr, newCarry);
    }
};

🔎 Line-by-Line Explanation

apply_add(u, l, r, x) and pull(u) click to expand/collapse

apply_add directly mutates u's sum by x · length and accumulates x into add_lazy. We only call this on a freshly cloned u — never on a node that some other root might reach. pull recomputes sum[u] as sum[left] + sum[right]; this is correct only if every ancestor's pending lazy that should have flowed into u's subtree has already been pushed into the cloned children — otherwise pull would overwrite sum[u] with a value that silently drops the parent's full-cover update.

push(u, l, r) — clone-and-apply before partial descent click to expand/collapse

Before descending partially from u, any add_lazy sitting on u must be pushed into u's children — but in a persistent setting we cannot mutate existing children, so we clone each child, apply_add the parent's lazy to the clone, and re-point u at the new children. Then u.add_lazy is cleared. This is the single most important discipline of persistent lazy: without it, pull after the partial recursion would lose u's own pending full-cover lazy.

range_add — clone-on-write descent with push click to expand/collapse

The function takes the previous version's node and produces a new node. No-overlap returns prev verbatim (shared). Full-overlap clones once and calls apply_add. Partial-overlap clones, then calls push (which itself clones the children and applies the pending lazy to them), recurses into both freshly cloned children, and finally pulls. Off-path subtrees remain shared with older versions.

range_sum with carry — never mutates click to expand/collapse

The carry parameter starts at zero at the root. At a full-overlap node, the contribution is pool[u].sum + carry · length: the node's stored sum plus the inherited tags from any ancestor that still carries lazy. At a partial-overlap node, the recursion passes carry + pool[u].add_lazy down to children. The node is never written to — that is why queries never affect older versions.

⚠️ Common Pitfalls

⚠️ Pitfall — Pushing lazy tags during a query

The most common bug: copy-pasting the push step from Lesson 4 into the persistent query. This mutates published nodes — silently corrupts older versions. Always use the carry pattern instead.

⚠️ Pitfall — Forgetting to push during a partial update

If you go partial without first pushing u's pending add_lazy into the cloned children, then pull(u) at the end recomputes sum[u] from children that don't reflect u's own lazy — silently losing the parent's full-cover update. Symptom: a sequence like full-cover add, then partial-cover add, then query returns a number smaller than expected. The fix is push(u, l, r) before recursing partially.

⚠️ Pitfall — Pushing into published children

The push step must clone the children before applying the parent's lazy to them. If you apply directly to the existing children, you've mutated published nodes — corrupting older versions. Inside push, the only writes go to freshly cloned nodes.

⚠️ Pitfall — Forgetting to clone in the partial-overlap case

Even when the update doesn't apply directly at a node, the node must be cloned because its children pointers are about to change. Forgetting this clone makes the new version's child pointer leak into the old version's parent.

⚠️ Pitfall — Double-counting tags in pull

If you write pull as sum[u] = sum[L] + add_lazy[L]·left_len + sum[R] + add_lazy[R]·right_len, you double-count: sum[L] already reflects L's own interval given L's own tags. The correct pull is just sum[u] = sum[L] + sum[R] — same as Lesson 4. The piece that pull cannot capture (the parent's own pending lazy) is what push handles before pull runs.

⚠️ Pitfall — Not reserving the pool

The pool needs roughly N · log₂(base) + n nodes (initial build + path clones plus push-time child clones). For N = 10⁵ and n = 10⁵, that's ~4 · 10⁶ nodes. Reserve up front.

🎛️ Variants & Extensions

Variant 1 — Persistent set/add lazy

Add a second lazy field set_lazy with has_set flag. The carry argument now becomes a pair (carry_set, carry_add), applied in the right order (set first, then add). Same structure, just more bookkeeping.

Variant 2 — Persistent affine

Lazy state is a pair (b, c) for the affine map x ↦ b · x + c. The carry is also a pair, composed by function composition. Slick when you understand the composition rule.

Variant 3 — Range KTH on a versioned sorted multiset

Combine persistent lazy with a sorted-keys representation. Useful for problems like “in the multiset as of time t, what is the k-th smallest?”.

Variant 4 — Lazy beats with persistence

Mainly research; the carry now has to handle chmin caps, which break the simple additive carry. Rarely seen in contests.

🏁 Final Summary

The persistent lazy segment tree combines two ideas: clone-on-write for updates (Lesson 5) and a carry pattern for queries (this lesson's novelty). The carry pattern is the key insight: by passing the cumulative pending add as an argument during the query descent, we avoid mutating nodes — which would otherwise corrupt older versions.

Once you can write a single line of carry-aware query code (full-overlap returns sum[u] + carry · len; partial passes carry + add_lazy[u] down), the rest is just Lesson 5's path-copying machinery applied to a richer node state. The result is a structure that supports range updates and arbitrary historical queries, each in O(log n).
