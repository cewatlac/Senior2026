Persistent Treap (Path Copying)

The implicit treap of tutorial 28, augmented so that every mutation clones the spine instead of overwriting it. The old root remains a valid handle to the pre-mutation version forever; mutations produce new roots while sharing untouched subtrees with prior versions. Result: a time-travel-capable sequence container in O(log n) per operation.
Theme

📌 01 · Problem Definition

Maintain a sequence (or an ordered set) under a stream of mutating operations, but additionally support a snapshot operation that captures a handle to the current state, and a time-travel operation that re-issues any past query against any past snapshot. Once captured, a snapshot must answer queries about the state of the structure at the moment of capture, even after many subsequent mutations have been applied to the live version. Both the live version and every snapshot must support insertions, deletions, range queries, and range updates in expected O(log n) time per operation. Memory usage may grow with the number of recorded versions, but each operation may only contribute O(log n) words to the total memory.

Classical examples include partial-persistence problems (queries on past states only), full-persistence problems (mutations may branch new versions off any past snapshot), and the snapshot semantics needed in functional programming. The persistent treap supports all of these. We focus on full persistence in this tutorial because it is the strictly more general setting.

🧠 02 · Intuition from Scratch

If we want to preserve every past state, the naive approach is to clone the entire structure before every mutation. The cost is O(n) per snapshot, which destroys the logarithmic bound. The path-copying insight is that any mutation to a balanced tree changes at most O(log n) nodes along the path from the root to the mutation site. Every node off that path is unchanged; the old version and the new version can share those untouched subtrees by pointer.

Path copying in one sentence

Before modifying a node, clone it; then perform the modification on the clone, not on the original. The original remains immutable and addressable from the old root. The clone has a fresh pointer that the new root will refer to. Recursing this rule down the path from the root produces a fresh spine of clones for the new version and leaves every subtree off the path shared with the old version.

Why an implicit treap is the ideal candidate

The implicit treap from tutorial 28 has two properties that make path copying natural. First, it has no parent pointers, so cloning a node does not require updating links from anywhere else. Second, all mutations route through split and merge, which already descend exactly one root-to-leaf path. If we modify split and merge to clone before mutating, we have a fully persistent implicit treap without changing the public API.

📐 04 · Formal Abstraction

Versions are root pointers

A version is just a pointer to the root of a treap at some moment in time. We maintain a vector versions of root pointers; calling save() pushes the current live root into the vector and returns its index. Mutations consume a version index, compute a new root, and append it to the vector.

Path-copying invariant

A node is mutable only at the moment of its creation. After insertion into any version vector, the node becomes immutable forever. Subsequent operations that need to "modify" a node must clone it first.

The clone is a shallow copy of the node\u2019s fields, including the value, priority, size, info, and the two child pointers. The clone\u2019s child pointers initially point at the same children as the original; subsequent operations on the clone may overwrite those child pointers (since the clone is still mutable from the operation that created it), but the children themselves remain shared with the original until the operation descends into them and clones them in turn.

What lives in the clone, what gets shared

When split or merge descends into a child and modifies it, the child must be cloned first. The descendants off the descent path are not visited by the current operation, so they are not cloned; the cloned spine\u2019s child pointers continue to point at them. After the operation returns, the new root is a fresh clone, every node on the descent path is a fresh clone, and every other node is shared with the previous version.

Lazy fields under persistence

If we support lazy range updates, the lazy fields are part of the node\u2019s state. Pushing a lazy at an old node would mutate it, which is forbidden. The fix: push creates clones too. When we descend into a node, we clone it first (because we are about to push its lazy or recurse), and the cloned spine carries the cleared lazy while the originals retain their pre-push state. The price is that read queries also clone the descent path; this is acceptable because the cost per read remains O(log n).

🔍 05 · Key Observations

Old root → old structure, forever
The root pointer of version k continues to address a valid treap that holds the sequence as it was at the time of save. No later mutation can change what that root reaches.
Only the spine is cloned
Every mutation visits at most O(log n) nodes; only those O(log n) nodes are cloned. The other Θ(n) nodes are shared by pointer with the previous version.
Memory grows by O(log n) per operation
Each operation creates at most O(log n) fresh nodes. After q operations the total memory is O((n + q) log n), which is acceptable for n, q up to about 10^5 with care for node size.
Garbage collection is non-trivial
A node is reachable from some root if some version vector entry transitively references it. Counting references with full bookkeeping is expensive. In contest code we typically allocate from a pool and never free; the test sizes are small enough that this is acceptable.
Even reads may need to clone
If we use lazy range updates, a query that descends into a node with a pending lazy must push it; pushing mutates the node. In a persistent setting, the push routine therefore clones the descent path. This is why every operation in the master template returns a new root rather than performing the work in place.

⚙️ 06 · Algorithm

The clone primitive

The clone function takes a node pointer and returns a pointer to a fresh node whose fields are a shallow copy. The two child pointers in the clone initially point at the same children as the original. Whoever called clone now owns the clone and may freely mutate its child pointers.

split with cloning

The split function descends one level per recursive call and may modify a node\u2019s child pointer. Under persistence, we clone the node before modifying it: the modification happens on the clone, and the clone is what gets returned. The recursive call into one of the children may itself clone that child; the parent\u2019s clone then receives the recursively-cloned child as one of its pointers, while the parent\u2019s other child pointer still points at the original (which is shared with the previous version).

merge with cloning

Merge similarly clones before mutating. At each level we choose one of the two input roots as the merged root, clone it, and recurse into one of its children. The cloned root\u2019s child pointer on the recursive side gets the recursive merge result; the cloned root\u2019s child pointer on the other side still points at the original child (shared).

Persistent insert as the canonical example

To insert value v at position k in version u and produce a new version, we call split(versions[u], k) to get (L, R); these are roots into structures shared with version u. We allocate a new single-node treap n. We compute new_root = merge(merge(L, n), R). We push new_root into versions as a new index. The old root versions[u] still reaches a valid pre-insertion tree.

📊 07 · Invariants and Transitions

State invariants

For every version index i, the root versions[i] addresses a treap satisfying the heap invariant on priorities, the size invariant on subtree sizes, and the lazy invariants of the implicit treap (push order rev, set_val, add). Every node reachable from versions[i] is in the state it was at the moment the version was saved.

Immutability invariant

Once a node has been added to the version vector (transitively reachable from some saved root), the node is immutable. Any subsequent operation that needs to alter that node\u2019s fields must first clone it; the alteration happens on the clone.

Termination

Every operation reduces to a constant number of split and merge calls, plus the clone primitive which is O(1). Each split or merge descends one level per recursive call and visits at most O(h) nodes total; the expected height is O(log n), so the expected total work per operation is O(log n).

📜 08 · Proof of Correctness

Old roots remain valid

Claim: after any operation on the live root, the old roots saved in versions still address treaps with the contents they held at the time of save. Proof by induction on the number of operations applied. Base: zero operations after a save means nothing has changed. Inductive step: assume the claim holds before some operation, and consider the operation in isolation. By the immutability invariant, the operation only writes to nodes it has just cloned. These clones are not reachable from any old root by construction (they did not exist when the old root was saved). Therefore every node reachable from the old root is in the same state it was before the operation. By induction, the claim holds after any number of operations.

The new root holds the expected post-operation contents

The split, merge, and clone primitives in the persistent variant compute exactly the same logical result as their non-persistent counterparts. The only difference is that they write to clones instead of to the originals. Since the cloned spine inherits the children of the original spine (sharing the off-spine subtrees), and the clone\u2019s own fields are updated as required by the algorithm, the new root logically represents the same treap that the non-persistent variant would have produced.

Memory bound

Each operation visits at most O(h) nodes (one descent path, possibly two for the double-split in range operations). Each node visited is cloned exactly once. The clone is a constant-size object. So each operation contributes O(h) new nodes, which is O(log n) in expectation. Over q operations the total memory is O((n + q) log n).

🧱 10 · Template Philosophy

The persistent template differs from the implicit treap template on five points.

Choice 1: clone before every write

Every function that may write to a node\u2019s fields first calls clone on that node. The clone is the only object the function may mutate; the original is read-only.

Choice 2: split and merge return new roots

Both functions take the inputs as const root pointers and return fresh root pointers. The caller stores the result somewhere (typically the version vector or a local variable used to chain further calls).

Choice 3: push also clones

When push descends into a node\u2019s children, the children must be cloned because we are about to write their lazy fields. The clones of the children become the new children of the parent\u2019s clone. The originals remain part of the older version.

Choice 4: no parent pointers, no rotations

Same as the implicit treap. Parent pointers would create a circular reference web that makes cloning much harder, and rotations are not needed because split and merge are sufficient. The persistent variant inherits the split/merge style verbatim.

Choice 5: pool allocator

Allocate nodes from a contiguous arena rather than calling new per node. This makes allocation O(1) without a system call and avoids fragmentation. The free list is empty by default (we never free in contest code); the arena pointer increments on each allocation.

🎯 11 · Master Template — Ready to Submit

A full persistent implicit treap built on path copying, around 280 lines. Supports versioned insert at position, erase at position, point get, range sum query, range add, and range reverse. Snapshots are addressable by integer version index.

// Persistent Implicit Treap with path copying
#include <bits/stdc++.h>
using namespace std;

mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());

struct Node {
    long long v;
    int pri, sz;
    long long sum;
    Node *l = nullptr, *r = nullptr;
    long long add = 0;
    bool rev = false;
};

int sz(Node* t){ return t ? t->sz : 0; }
long long sum_of(Node* t){ return t ? t->sum : 0; }

// Clone is the heart of persistence: returns a fresh node with the same fields,
// including child pointers which are shared with the original until the caller
// overwrites them.
Node* clone(Node* t){
    Node* x = new Node{};
    *x = *t; // shallow copy all fields including l, r pointers
    return x;
}

Node* make_node(long long v){
    Node* x = new Node{};
    x->v = v; x->pri = (int) rng(); x->sz = 1; x->sum = v;
    return x;
}

// Apply a pending add to a (just-cloned) node.
void apply_add(Node* t, long long delta){
    if (!t) return;
    t->v += delta;
    t->sum += delta * (long long) t->sz;
    t->add += delta;
}

// Apply a pending reverse to a (just-cloned) node.
void apply_rev(Node* t){
    if (!t) return;
    swap(t->l, t->r);
    t->rev ^= 1;
}

// Push: propagate lazy fields to children. Children must be cloned first since
// applying the lazy mutates their fields.
void push(Node* t){
    if (!t) return;
    if (t->rev){
        if (t->l){ t->l = clone(t->l); apply_rev(t->l); }
        if (t->r){ t->r = clone(t->r); apply_rev(t->r); }
        t->rev = false;
    }
    if (t->add){
        if (t->l){ t->l = clone(t->l); apply_add(t->l, t->add); }
        if (t->r){ t->r = clone(t->r); apply_add(t->r, t->add); }
        t->add = 0;
    }
}

void pull(Node* t){
    if (!t) return;
    t->sz = 1 + sz(t->l) + sz(t->r);
    t->sum = t->v + sum_of(t->l) + sum_of(t->r);
}

// Persistent split: returns two new roots without modifying t.
void split(Node* t, int k, Node*& a, Node*& b){
    if (!t){ a = b = nullptr; return; }
    Node* nt = clone(t);
    push(nt);
    int lc = sz(nt->l);
    if (k <= lc){
        split(nt->l, k, a, nt->l);
        pull(nt); b = nt;
    } else {
        split(nt->r, k - lc - 1, nt->r, b);
        pull(nt); a = nt;
    }
}

// Persistent merge: returns a new root without modifying a or b.
Node* merge(Node* a, Node* b){
    if (!a) return b;
    if (!b) return a;
    if (a->pri > b->pri){
        Node* na = clone(a);
        push(na);
        na->r = merge(na->r, b);
        pull(na);
        return na;
    } else {
        Node* nb = clone(b);
        push(nb);
        nb->l = merge(a, nb->l);
        pull(nb);
        return nb;
    }
}

struct PersistentTreap {
    vector<Node*> versions; // versions[i] is the root at the moment version i was saved

    // Initial empty version.
    PersistentTreap(){ versions.push_back(nullptr); }

    int latest(){ return (int) versions.size() - 1; }
    Node* root_of(int u){ return versions[u]; }
    int size_of(int u){ return sz(versions[u]); }

    // Build a fresh version from a vector of values.
    int build(const vector<long long>& a){
        Node* t = nullptr;
        for (long long v : a) t = merge(t, make_node(v));
        versions.push_back(t);
        return latest();
    }

    // Insert v at position k in version u; produce a new version.
    int insert(int u, int k, long long v){
        Node *L, *R; split(versions[u], k, L, R);
        Node* nr = merge(merge(L, make_node(v)), R);
        versions.push_back(nr);
        return latest();
    }

    // Erase position k in version u; produce a new version.
    int erase(int u, int k){
        Node *L, *M, *R;
        split(versions[u], k, L, M); split(M, 1, M, R);
        // M is a freshly cloned single-node treap; we discard it. No physical free
        // needed under contest pool allocation. The old version still references
        // the original node via the un-cloned subtree off the spine.
        Node* nr = merge(L, R);
        versions.push_back(nr);
        return latest();
    }

    // Point read at position k in version u; returns the value.
    long long get(int u, int k){
        Node *L, *M, *R;
        split(versions[u], k, L, M); split(M, 1, M, R);
        long long ans = M->v;
        // We do not push the cloned spine back into versions because the query
        // does not need to be persisted. The clones become garbage immediately.
        return ans;
    }

    // Range sum query on [l, r) in version u.
    long long range_sum(int u, int l, int r){
        Node *L, *M, *R;
        split(versions[u], l, L, M); split(M, r - l, M, R);
        long long ans = sum_of(M);
        return ans;
    }

    // Range add on [l, r) in version u; produce a new version.
    int range_add(int u, int l, int r, long long d){
        Node *L, *M, *R;
        split(versions[u], l, L, M); split(M, r - l, M, R);
        if (M){ Node* M2 = clone(M); apply_add(M2, d); M = M2; }
        Node* nr = merge(L, merge(M, R));
        versions.push_back(nr);
        return latest();
    }

    // Range reverse on [l, r) in version u; produce a new version.
    int reverse(int u, int l, int r){
        Node *L, *M, *R;
        split(versions[u], l, L, M); split(M, r - l, M, R);
        if (M){ Node* M2 = clone(M); apply_rev(M2); M = M2; }
        Node* nr = merge(L, merge(M, R));
        versions.push_back(nr);
        return latest();
    }
};

📝 12 · Line-by-Line Walkthrough

The Node struct and clone

The Node struct is the same as the implicit treap of tutorial 28, minus the has_set / set_val machinery (omitted for clarity; add it back if your problem needs range-assign). The clone function is a one-liner: Node* x = new Node{}; *x = *t; return x; The shallow copy copies all fields including the two child pointers, which means immediately after clone, the cloned node and the original both point at the same children. That sharing is desirable: subsequent mutation by the caller may overwrite one of the clone\u2019s child pointers, but the original\u2019s child pointers continue to point at the originals.

push under persistence

The push function does something subtle. When we want to push a lazy flag from a parent to its children, we must mutate the children\u2019s fields (their own lazy, their value, their subtree sum). Mutating an old child would corrupt an old version. So we clone each child first and apply the lazy to the clone. The parent (which is already a clone by the time push is called) replaces its child pointers with the freshly cloned children.

split returns two new roots

At the top of split we clone the current node into nt. We push lazy on nt (which may further clone nt\u2019s children inside push). We compute lc. The recursive split call writes its second output into nt->l or nt->r directly; this is safe because nt is a fresh clone owned by the current call. After the recursive call, we pull nt and assign nt to either a or b depending on whether the descent went left or right. The other output is whatever the recursive call returned.

merge mirrors split

Merge picks one of the two input roots based on priorities and clones it. Push lazy on the clone. Recurse into the appropriate child slot. Assign the recursive result. Pull. Return the clone. The non-recursive side\u2019s pointer is shared with the previous version, which is exactly the path-copying behaviour we want.

The PersistentTreap public API

The class holds a vector versions indexed by version number. The constructor seeds it with a single nullptr entry representing the empty initial version. Every public mutator (build, insert, erase, range_add, reverse) takes a base version index, performs the operation against versions[u], pushes the resulting root into versions, and returns the new version index. Reads (get, range_sum) do not push a new version; they compute the answer and discard the clones produced as a byproduct of split.

What about memory leaks from queries that clone?

Each query produces O(log n) garbage clones. In a contest setting we typically do not free; the allocator just keeps growing. For a tighter memory budget, swap the global new for an arena allocator and reset the arena between test cases. For long-running services, use a smart-pointer-based variant or explicit garbage collection; both are out of scope for this tutorial.

🧩 13 · Edge Cases (Centralized Audit)

  • Initial empty version. versions[0] = nullptr. Every operation handles nullptr inputs by returning whichever side is non-null, so insert into v[0] works without special casing.
  • Insert at position 0 in any version. Split returns (nullptr, root); merge(new, root) gives a tree with the new node at position 0.
  • Insert at position n in any version. Split returns (root, nullptr); merge(root, new) puts the new node at the back.
  • Erase at the boundaries. Same structure; the middle is a freshly cloned single-node treap that becomes garbage immediately.
  • Range with l == r. The middle split returns nullptr; the operation has no effect; merge reassembles the original.
  • Query on the initial empty version. versions[0] is nullptr; the split returns nullptrs; sum_of(M) returns 0; size queries return 0.
  • Multiple branches off the same version. Calling insert(0, ...) twice produces two distinct new versions; they share the off-spine subtrees of version 0 but not with each other.
  • Pushing a lazy on a node with one null child. The push function checks if (t->l) and if (t->r) before cloning, so nullptr children are not dereferenced.
  • Range reverse with non-commutative aggregates. The current template uses only sum; if your problem uses a non-commutative aggregate, augment the Info bundle and call apply_rev on the aggregate as well as the children pointers.
  • Memory exhaustion on very long version histories. Each operation creates O(log n) nodes; n × q operations create O(n q log n) total nodes. Stop accepting test cases or implement garbage collection if this exceeds the memory limit.

🪲 14 · Common Pitfalls and Bugs

Pitfall 1 — Mutating a node that is reachable from an old root
Every write to a node\u2019s fields must happen on a clone. If you accidentally write to an original node, the corresponding old version silently breaks. Discipline: clone at the top of every function that may write, and never call apply_* or push on an un-cloned node.
Pitfall 2 — Forgetting to clone children inside push
Push writes the children\u2019s lazy fields. If the children are old (shared with a prior version), they must be cloned first. The master template handles this explicitly with t->l = clone(t->l); apply_*(t->l);
Pitfall 3 — Pull without pushing first
Pull reads the children\u2019s aggregates. If the parent has a pending lazy that has not been propagated, the children\u2019s aggregates are pre-lazy and the parent\u2019s pull recomputes a wrong aggregate. Always push before pull in any function that descends.
Pitfall 4 — Writing to versions vector mid-operation
Some implementations are tempted to write the new root into the version vector as soon as it is computed, even before the operation returns. This is fine, but it must happen exactly once per public mutator; double-pushing creates dead version indices.
Pitfall 5 — Garbage from query clones
Pure read queries (get, range_sum) clone nodes along the descent path because of lazy pushes. These clones become garbage at function return. Use an arena allocator and reset between test cases to keep memory bounded.
Pitfall 6 — Using parent pointers
A parent pointer would force you to update the parent of every cloned child to point at the clone, and the parent of every shared subtree might still legitimately point at the old version\u2019s parent. The cleanest solution is to abandon parent pointers; the split / merge style does not need them.
Pitfall 7 — Mixing live mutations with persistent calls
If you also use a non-persistent treap in the same program, do not let the two share node pointers. Persistent code may assume that any pointer it holds is safe to clone-and-modify; if a non-persistent code path holds the same pointer and mutates it directly, both copies corrupt each other.

🧭 15 · Decision Guide

When the persistent treap is the right tool

need queries on past states ⇒ persistent treap (or persistent segment tree)
need to branch off any past state ⇒ persistent treap (segment tree is partial-persistence only)
need range reverse with persistence ⇒ persistent implicit treap (only choice)
need confluent persistence (merge two histories) ⇒ persistent treap with split/merge
need partial persistence only, no rich range ops ⇒ persistent segment tree (shorter)
no time-travel needed ⇒ implicit treap (tutorial 28)

🎭 16 · Variants and Extensions

Variant A — confluent persistence

Two versions can be merged into a third by treating each as a piece of a longer sequence. This is just a regular merge of two persistent roots; the result is a fresh root that uses both inputs as shared substructures.

Variant B — snapshot semantics with reference counting

In a long-running setting where memory matters, attach a refcount to every node. Increment on clone, decrement on free. When a refcount drops to zero, recursively free children. The cost per clone is O(1), and the memory bound becomes proportional to live versions only.

Variant C — arena allocator with checkpointing

Allocate all nodes from a single contiguous arena and remember the arena pointer at each save point. When discarding a version, rewind the arena pointer to that version\u2019s checkpoint. This is much faster than per-node free and handles bulk discards naturally.

Variant D — persistent key treap

The key treap of tutorial 27 can also be made persistent, but the rotate-up / rotate-down style with parent pointers is awkward under cloning. The cleanest persistent key treap is built on split / merge primitives that key off value comparisons, mirroring this tutorial\u2019s structure with a value-keyed split.

Variant E — undo / redo as version stack

Every mutation pushes a new version; the undo operation rewinds the version index; redo advances it. The version vector serves as a linear undo stack. Branching from a non-tip version produces a new line of history; the original tip is still addressable but no longer the live version.

🎯 18 · Final Summary

One sentence: a persistent treap is the implicit treap of tutorial 28 where every function clones nodes before writing to them, leaving old root pointers as valid handles into the pre-mutation state and producing new root pointers that share O(n − log n) of their nodes with the previous version.

Five things to remember

First, path copying is the whole idea: clone the descent path before mutating, share everything off the path. Second, every public operation returns a new root; the version vector stores roots, and each version is just an index. Third, push must also clone, because pushing mutates the children\u2019s fields. Fourth, the cost per operation is O(log n) time and O(log n) extra nodes; total memory after q operations is O((n + q) log n), which limits practical sizes to about 10^5 of each on a typical 256 MB budget. Fifth, no parent pointers, no rotations, no shared mutable state with non-persistent code; the discipline must be airtight or old versions silently corrupt.

Mastery checklist

  • I can explain why clone is a one-line shallow copy and what subtleties that hides.
  • I can list the three places in split, merge, and push where a new clone is created.
  • I can convert the implicit treap of tutorial 28 into a persistent variant by adding clone calls and a version vector.
  • I can compute the memory bound O((n + q) log n) and decide whether the problem fits.
  • I know that pure-read queries also clone under lazy semantics, and I have a plan for the garbage.
