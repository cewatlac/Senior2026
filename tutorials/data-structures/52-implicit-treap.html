Implicit Treap (Sequence Treap)

A treap where position is the implicit key. The same node type that gave us an ordered multiset now gives us a dynamic sequence supporting insert-at-position, erase-at-position, range add, range assign, range reverse, range sum, and cyclic shift — all in expected O(log n) per operation.
Theme

📌 01 · Problem Definition

Maintain a dynamic sequence A[0], A[1], ..., A[n-1] of values under a rich menu of position-based operations. The structure must support insertion of a new value at an arbitrary position, deletion of the value at a given position, point read and point update, range query (sum, min, max, or any commutative-associative aggregate), range update (add a constant to every element of a range, or assign a constant to every element of a range), range reverse (reverse a contiguous block in place), and cyclic shift of a contiguous block. Every operation must complete in expected O(log n) time, where n is the current length of the sequence.

No comparison ordering exists between values. Two adjacent elements can be in any order; the only structure is positional. This rules out the key treap of tutorial 27, which depended on the BST invariant on values. The implicit treap solves this by treating position itself as the key, with a clever twist: positions are not stored at all. Each node\u2019s position is implicit, derived from the size of its left subtree at the moment of access.

🧠 02 · Intuition from Scratch

If we wanted to maintain a sequence using a balanced BST keyed by position, we would have a problem: inserting a value at position k would shift the positions of every subsequent value by one. Storing the positions explicitly would force us to rewrite up to n - k keys per insertion, defeating the logarithmic bound. The implicit treap sidesteps this by never storing the position at all. Instead, the inorder rank of each node within the tree is its position. The leftmost node is at position 0, the second-leftmost at position 1, and so on. When we walk down the tree, we infer the position of the current node from the size of its left subtree.

The price we pay, the price we save

The price we pay: we cannot answer a query like "what position is the value 42 at?" without scanning. We can only answer position-based queries. The price we save: insertion and deletion at any position cost O(log n), with no shift cost, because shifting is implicit in the tree structure. A node at position k has k − 1 inorder predecessors; if we rewire the tree so it now has k inorder predecessors, that node\u2019s position becomes k + 1 for free.

From rotation to split/merge

The key treap of tutorial 27 used rotate-up after insertion and rotate-down before deletion. The implicit treap uses two different primitives: split and merge. Split takes a treap and a position k and returns two treaps: one holding the first k elements (in order) and one holding the remaining elements. Merge takes two treaps and concatenates them into one. Every other operation — insert, erase, range query, range update, range reverse — is expressed as a short sequence of splits and merges with a small piece of work in between.

🎨 03 · Visual Mental Model

Visualise a sequence of five values [a, b, c, d, e] stored in an implicit treap. The tree has five nodes, one per value, and the inorder reads a, b, c, d, e. The shape of the tree is determined by the random priorities. To find the element at position 2 (which is c), we descend from the root: at each step we compare the requested position with the size of the left subtree, and either descend left, return the current node, or subtract sz(left) + 1 from the position and descend right.

📐 04 · Formal Abstraction

Node fields

Each node stores an Info bundle that carries the value at this position plus any aggregate derived from the subtree it roots. The minimal Info has the value v and the subtree sum sum; richer variants add min, max, or any commutative-associative aggregate. The node also holds a random priority pri, a subtree size sz counting the number of elements (each contributes one regardless of value), a pair of child pointers l and r, and three lazy flags: add for pending range-add, set_val with a boolean has_set for pending range-assign, and rev for pending range-reverse.

The three invariants

heap invariant: for every node u, u.pri >= u.l.pri AND u.pri >= u.r.pri size invariant: for every node u, u.sz = sz(u.l) + 1 + sz(u.r) push invariant: for every node u that does not have a pending lazy update, u.info is the correct aggregate of every value below it with all pending updates applied.

What is not in the invariants

There is no BST invariant on values. Two adjacent elements in the sequence can carry any values; the treap does not impose an ordering on them. The only ordering that matters is the inorder, which is the sequence order.

The lazy bundle

The lazy fields encode pending updates that have been promised to a subtree but not yet pushed to its descendants. When we descend into a node\u2019s children, we must first push the parent\u2019s lazy updates into both children\u2019s lazy fields, then clear the parent\u2019s lazy. The push order matters: rev swaps the children\u2019s subtrees in addition to flipping their rev flags, set_val overrides any pending add, and add accumulates with any pending add. We push in the order rev, then set_val, then add.

🔍 05 · Key Observations

Position is implicit, never stored
The position of a node in the sequence is its inorder rank in the tree. We compute it on demand by looking at the left subtree size of every ancestor we descended through.
Every operation is split-then-merge
Insert at position k: split at k, build a single-node treap, merge three times. Erase at position k: split at k, split at k+1, drop the middle, merge twice. Range reverse on [l, r): split-split, set rev on the middle, merge-merge. The split/merge primitives are the only mutators of tree structure.
Random priority gives expected O(log n) height
Same argument as the key treap: the tree shape depends only on the priorities, which are random, so the expected height is O(log n) regardless of operation order.
Push order matters
Lazy updates of different kinds do not commute. set_val after add means the final value is the assigned constant; add after set_val means the final value is constant + add. The push routine must apply them in a fixed order; we use rev first, then set_val, then add.

⚙️ 06 · Algorithm

Split at position k

Given a treap rooted at t, the operation split(t, k) returns a pair (L, R) of treaps such that L holds the first k elements of t in order and R holds the remaining elements. The recursion is: push any pending lazy updates at t down to its children. Compute lc = sz(t->l). If k <= lc, recursively split the left subtree at k, getting (L\u2032, R\u2032); then return (L\u2032, t) after replacing t->l with R\u2032. Otherwise, recursively split the right subtree at k - lc - 1, getting (L\u2032, R\u2032); replace t->r with L\u2032 and return (t, R\u2032). After every replacement, recompute the aggregate at t via pull(t).

Splitting a 5-element sequence at position 3

Merge two treaps

Given two treaps a and b, the operation merge(a, b) assumes every element of a is positioned before every element of b and returns one treap. The recursion is: if either side is empty, return the other. Push any pending lazy on the side we are about to descend. Compare a->pri with b->pri. If a->pri > b->pri, then a is the root of the merged treap; recursively merge a->r with b and store the result as a->r. Otherwise b is the root; recursively merge a with b->l and store the result as b->l. After every replacement, pull.

Merging a 2-element treap with a 3-element treap

Range reverse via the lazy flag

To reverse the contiguous block A[l..r-1], we split twice. First, split(t, l) gives (L, mid_plus_R). Then split(mid_plus_R, r - l) gives (mid, R). Now the middle treap mid holds exactly the block we want to reverse. We toggle mid->rev = !mid->rev, which flips a lazy bit but does not yet swap any children. Finally we merge L with mid with R in that order to reassemble the sequence. The reverse only materialises when we later push the flag down on a descent.

When we push the rev flag at a node, we swap that node\u2019s two children pointers and toggle the rev flag on each child. The flag thus propagates lazily down the tree exactly when a descent forces it. Subsequent split or merge calls into a node with a pending rev first push the flag, then proceed normally.

Range reverse on positions [1, 4): the lazy flag toggles, the swap is deferred

Other operations as split / merge sequences

Insertion of value x at position k is: split(t, k), build single-node treap n with value x, merge L with n with R. Erasure at position k is: split at k, split at k+1, discard the middle, merge the two remaining pieces. Range query is: split, split, read mid->info, merge back. Range add is: split, split, set mid->add += delta and update mid->info accordingly, merge back. Range assign is: split, split, set mid->has_set = true with mid->set_val = c, clear mid->add, update mid->info, merge back. Cyclic shift left by one on a range: split into three, then reassemble in shifted order.

📊 07 · Invariants and Transitions

State invariants between operations

Between operations the implicit treap maintains the heap invariant on priorities, the size invariant on subtree sizes, and a weaker version of the push invariant: every node\u2019s info is the correct aggregate of every value below it with all pending lazy applied. The lazy fields may carry deferred work that has not yet reached the descendants, but the aggregate already reflects the deferred work.

Transition: push

The push operation at a node t propagates the lazy fields at t to its two children and clears them at t. The order is: first apply t->rev to the children (swap their children pointers, toggle their rev), then apply t->set_val if t->has_set is true (set each child\u2019s set_val and has_set, clear each child\u2019s add, update each child\u2019s info), then apply t->add (accumulate into each child\u2019s add if not has_set, or into each child\u2019s set_val if has_set, and update each child\u2019s info). Finally clear t->rev, t->has_set, t->add.

Transition: pull

The pull operation at a node t recomputes t->sz and t->info from t->l and t->r. It assumes the two children\u2019s info is correct and aggregates the children\u2019s contributions plus t\u2019s own value into t\u2019s info. Every mutation of a child pointer must be followed by a pull on the parent.

Termination of split and merge

Both recursions descend one level into the tree at each step, so both terminate after at most h levels, where h is the expected O(log n) height.

📜 08 · Proof of Correctness

Split is correct

By induction on the height of the input. Base: an empty tree splits into two empty trees. Inductive step: at a node t with left size lc, if k <= lc then the first k elements are entirely contained in t->l; the inductive hypothesis on the recursive call at t->l produces the correct (L\u2032, R\u2032) partition of t->l, and gluing R\u2032 back as t\u2019s left child yields a treap whose first k elements are L\u2032 and whose remainder is t with its other elements. Symmetric for k > lc. Lazy is pushed at the start to ensure the position computations use up-to-date subtree sizes.

Merge is correct

By induction on the sum of the two heights. If either side is empty, the other is returned unchanged, which is correct. Otherwise, compare priorities: if a->pri > b->pri, the heap invariant forces a to be the root of the merged treap. The merged treap\u2019s left subtree is a->l unchanged, and the merged treap\u2019s right subtree is the recursive merge of a->r with b. By induction the recursive merge is correct, so the whole result is correct.

The split-merge identity

For any treap t and any valid position k, merge(split(t, k)) reproduces t as a multiset of (value, priority) pairs. The shape may differ because the recursive merge may pick different roots than the original tree did; the inorder sequence and the multiset of priorities are preserved.

Lazy push preserves the aggregate

By construction, pushing a lazy flag from a parent to its children sets the children\u2019s lazy fields to encode the same pending update, sets the children\u2019s info to the value it would have if the update were applied, and clears the parent\u2019s lazy. The parent\u2019s info remains correct because it was already aggregating the children\u2019s post-push views.

⏱️ 09 · Complexity Analysis

Every implicit-treap operation reduces to a constant number of split and merge calls plus a constant amount of post-processing per node visited. Each split or merge call descends one root-to-leaf path, so each costs O(h). The expected height of the treap is O(log n) because priorities are random, giving expected O(log n) per operation.

🧱 10 · Template Philosophy

The template choices for the implicit treap differ from the key treap on three points.

Choice 1: no parent pointers

All operations work via recursive split and merge that pass and return pointers. There is no rotate operation, so parent pointers earn nothing and cost a word per node. We drop them.

Choice 2: nullptr instead of an EMPTY sentinel

The recursive split / merge style branches on if (!t) at the start of every function, which is exactly what we want for nullptr. A sentinel would force us to either check pointer equality against the sentinel or push lazy fields into the sentinel, neither of which gains us anything.

Choice 3: an Info bundle for clean augmentation

Rather than scatter sum, min, max, and other aggregates across the node body, we group them into an Info struct with a combine and an apply_add and an apply_set method. Add a new aggregate by extending Info; the rest of the template does not change. The combine function is associative and commutative; if you want non-commutative aggregates, the rev flag must also flip the aggregate, which Info should expose as apply_rev.

Choice 4: a strict lazy-push order

The push function pushes rev first, then set_val, then add. The reasoning: rev is a structural rearrangement that does not change values, so it can be applied before any value mutation; set_val overrides any pending add, so it must come before add; add is the only thing left at that point. Any other order silently corrupts the aggregate.

🎯 11 · Master Template — Ready to Submit

A complete implicit treap with subtree-sum aggregate, range add, range assign, range reverse, and cyclic shift, around 250 lines. The Info bundle is sized for sum and count; extend it for min, max, or any commutative-associative aggregate by adding fields and updating combine, apply_add, apply_set.

// Implicit Treap with lazy: range add, range assign, range reverse, cyclic shift
#include <bits/stdc++.h>
using namespace std;

mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());

struct Info {
    long long sum = 0;  // sum of values in this subtree
    int cnt = 0;        // number of nodes in this subtree
};

Info combine(const Info& a, const Info& b){
    Info r; r.sum = a.sum + b.sum; r.cnt = a.cnt + b.cnt; return r;
}

struct node {
    long long v;       // the value stored at this position
    int pri, sz;
    Info info;          // aggregate over the subtree
    node *l = nullptr, *r = nullptr;
    long long add = 0; // pending range-add
    long long set_val = 0;
    bool has_set = false;  // pending range-assign?
    bool rev = false;       // pending range-reverse?
};

int sz(node* t){ return t ? t->sz : 0; }
Info info_of(node* t){ return t ? t->info : Info{}; }

void apply_add(node* t, long long delta){
    if (!t) return;
    t->v += delta;
    t->info.sum += delta * (long long) t->sz;
    if (t->has_set) t->set_val += delta; else t->add += delta;
}

void apply_set(node* t, long long c){
    if (!t) return;
    t->v = c;
    t->info.sum = c * (long long) t->sz;
    t->has_set = true; t->set_val = c;
    t->add = 0;
}

void apply_rev(node* t){
    if (!t) return;
    swap(t->l, t->r);
    t->rev ^= 1;
}

void push(node* t){
    if (!t) return;
    // push order: rev, then set, then add
    if (t->rev){
        apply_rev(t->l); apply_rev(t->r);
        t->rev = false;
    }
    if (t->has_set){
        apply_set(t->l, t->set_val);
        apply_set(t->r, t->set_val);
        t->has_set = false; t->set_val = 0;
    }
    if (t->add){
        apply_add(t->l, t->add);
        apply_add(t->r, t->add);
        t->add = 0;
    }
}

void pull(node* t){
    if (!t) return;
    t->sz = 1 + sz(t->l) + sz(t->r);
    Info self; self.sum = t->v; self.cnt = 1;
    t->info = combine(combine(info_of(t->l), self), info_of(t->r));
}

// Split t into (a, b) where a holds the first k elements, b holds the rest.
void split(node* t, int k, node*& a, node*& b){
    if (!t){ a = b = nullptr; return; }
    push(t);
    int lc = sz(t->l);
    if (k <= lc){
        split(t->l, k, a, t->l);
        b = t;
    } else {
        split(t->r, k - lc - 1, t->r, b);
        a = t;
    }
    pull(t);
}

// Merge two treaps a and b where every position in a precedes every position in b.
node* merge(node* a, node* b){
    if (!a) return b;
    if (!b) return a;
    if (a->pri > b->pri){
        push(a);
        a->r = merge(a->r, b);
        pull(a);
        return a;
    } else {
        push(b);
        b->l = merge(a, b->l);
        pull(b);
        return b;
    }
}

node* make_node(long long v){
    node* x = new node{};
    x->v = v; x->pri = (int) rng(); x->sz = 1;
    x->info.sum = v; x->info.cnt = 1;
    return x;
}

struct ImplicitTreap {
    node* root = nullptr;

    int size(){ return sz(root); }

    void build(const vector<long long>& a){
        root = nullptr;
        for (long long v : a) root = merge(root, make_node(v));
    }

    void insert(int k, long long v){
        node *L, *R; split(root, k, L, R);
        root = merge(merge(L, make_node(v)), R);
    }

    void erase(int k){
        node *L, *M, *R;
        split(root, k, L, M); split(M, 1, M, R);
        delete M;
        root = merge(L, R);
    }

    long long get(int k){
        node *L, *M, *R;
        split(root, k, L, M); split(M, 1, M, R);
        long long ans = M->v;
        root = merge(merge(L, M), R);
        return ans;
    }

    void set_value(int k, long long v){
        node *L, *M, *R;
        split(root, k, L, M); split(M, 1, M, R);
        apply_set(M, v);
        root = merge(merge(L, M), R);
    }

    Info query(int l, int r){
        node *L, *M, *R;
        split(root, l, L, M); split(M, r - l, M, R);
        Info ans = info_of(M);
        root = merge(merge(L, M), R);
        return ans;
    }

    void range_add(int l, int r, long long d){
        node *L, *M, *R;
        split(root, l, L, M); split(M, r - l, M, R);
        apply_add(M, d);
        root = merge(merge(L, M), R);
    }

    void range_set(int l, int r, long long c){
        node *L, *M, *R;
        split(root, l, L, M); split(M, r - l, M, R);
        apply_set(M, c);
        root = merge(merge(L, M), R);
    }

    void reverse(int l, int r){
        node *L, *M, *R;
        split(root, l, L, M); split(M, r - l, M, R);
        apply_rev(M);
        root = merge(merge(L, M), R);
    }

    // Cyclic shift: rotate A[l..r-1] left by one position; A[l] moves to A[r-1].
    void cyclic_shift(int l, int r){
        if (r - l <= 1) return;
        node *L, *M, *R, *first, *rest;
        split(root, l, L, M); split(M, r - l, M, R);
        split(M, 1, first, rest);
        root = merge(L, merge(merge(rest, first), R));
    }
};

📝 12 · Line-by-Line Walkthrough

The Info bundle and aggregation

The Info struct holds the subtree aggregate. For this template we maintain sum and a count cnt. The combine function adds two Infos componentwise; it is associative and commutative. To add a minimum aggregate, extend Info with a mn field, update combine to take the minimum, and update apply_add and apply_set to push the lazy into mn as well.

The three lazy appliers

The function apply_add applies a pending add to a subtree: it updates the node\u2019s own value, adjusts the subtree sum by delta * sz, and merges the delta into the lazy field (into set_val if the subtree already has a pending assign, otherwise into add). The function apply_set applies a pending assign: it overwrites the node\u2019s value, recomputes the subtree sum as c * sz, sets has_set and set_val, and clears any pending add (because the assign supersedes the add). The function apply_rev applies a pending reverse: it swaps the node\u2019s two children pointers and toggles the rev flag. The actual swap propagates lazily as descendants get pushed.

The push order

The push function does three things in fixed order. First, push rev by calling apply_rev on both children and clearing the flag at t; this is purely structural and does not change values. Second, push has_set by calling apply_set on both children with t->set_val; this overrides any pending add the children might have. Third, push add by calling apply_add on both children with t->add; this accumulates into the children\u2019s pending add (or into their pending assign if they have one). Any other order produces wrong aggregates.

The pull function

The pull function recomputes sz and info at a node from its children. Subtree size is one plus the children\u2019s sizes. The aggregate is the children\u2019s aggregates combined with a single-node aggregate built from the current node\u2019s value. The call order in combine is combine(combine(left, self), right); the order matters for non-commutative aggregates but is irrelevant here.

Split and merge

The split function descends one level per recursive call. At each level it pushes lazy, computes the left subtree size lc, and either splits the left subtree (if k is in the left subtree) or splits the right subtree (with the adjusted index k - lc - 1). The references a and b are written before returning; the assignments overwrite t->l or t->r with the appropriate part of the recursive split. Final pull ensures the aggregate is correct on the way back up. The merge function picks the higher-priority root, pushes lazy on it, recurses on the appropriate side, and pulls on the way back up.

The public API

Every public operation follows the same shape: split-split, work on the middle, merge-merge. The build function repeatedly merges single-node treaps to initialise the structure from a vector; it is O(n log n) on the average random input but can be improved to O(n) by sorting the priorities and using a Cartesian-tree-style build. Cyclic shift is a slight generalisation: split into three (prefix, middle, suffix), then split the middle to peel off its first element, then reassemble in shifted order.

🧩 13 · Edge Cases (Centralized Audit)

  • Empty tree. split returns two nullptrs; merge returns whichever side is non-null. All operations handle the empty case without special-casing.
  • Insert at position 0. split(root, 0) returns (nullptr, root); merge(new, root) gives a tree with the new node at position 0.
  • Insert at position n. split(root, n) returns (root, nullptr); merge(root, new) gives a tree with the new node at the back.
  • Erase at the boundaries. Same split-split structure; the middle is the single node to remove. Be sure to delete it before merging.
  • Range [l, r) with r == l. The middle split returns nullptr; range operations on it are no-ops. Merging back is safe.
  • Range that covers the whole sequence. Reduces to lazy on the root and trivial merges.
  • Reverse on a range of length 1. The middle treap is a single node; toggling its rev flag has no effect because pushing on a leaf still leaves no descendants to reorder.
  • Cyclic shift on a range of length 0 or 1. Guarded explicitly at the top of cyclic_shift.
  • Range assign then range add interaction. set_val first wins; subsequent adds accumulate into set_val. The push order in apply_add handles this by checking has_set.
  • Forgetting to push before descending. Every descent through a node (in split, merge, get without split) must push first; the master template enforces this at the top of split and merge.

🪲 14 · Common Pitfalls and Bugs

Pitfall 1 — Wrong push order
If you push add before set_val, the assigned constant overwrites your add and the value silently drops the delta. If you push set_val before rev with non-commutative aggregates, the reversed view sees a sum that was computed before the reversal. The strict order rev / set_val / add is mandatory.
Pitfall 2 — Forgetting to push at the top of split or merge
Both functions read child pointers and subtree sizes to decide the recursion direction. If the parent has a pending rev that has not been pushed, the children pointers are still in the unrotated layout and the split goes into the wrong subtree. Always push at the top of split and merge.
Pitfall 3 — Forgetting to pull after a recursive call
Each recursive split or merge call may change a child pointer; the parent\u2019s subtree size and aggregate must be refreshed afterward. The cleanest fix is a strict discipline: every function that changes a child pointer ends with a pull.
Pitfall 4 — Mutating a child pointer without going through a helper
In the key treap of tutorial 27, we routed all child writes through link. The implicit treap does not have parent pointers, so we cannot make link a strict requirement, but the recursive split and merge functions enforce pull after every mutation; ad-hoc rewrites outside these functions tend to forget pull and produce subtle aggregation bugs.
Pitfall 5 — Range bounds confusion (inclusive vs exclusive)
The master template uses half-open ranges [l, r). The split call split(M, r - l, M, R) produces a middle whose size is exactly r - l. If you accidentally pass r - l + 1 you grab one extra element; r - l - 1 grabs one too few. Pin the convention and test all four boundary cases.
Pitfall 6 — Random priority quality
Same warning as the key treap: use mt19937, not rand(). On Windows rand() has 15 bits which gives ties with high probability on n > 30000 and the treap degenerates.
Pitfall 7 — Memory leaks on erase
After split-split isolates the middle node, you must delete it before merging the two surviving pieces. Forgetting the delete is harmless for correctness on a single problem but leaks memory on stress tests with many erases.

🧭 15 · Decision Guide

When the implicit treap is the right tool

need insert at position k ⇒ implicit treap
need erase at position k ⇒ implicit treap
need range reverse ⇒ implicit treap (segment tree cannot)
need cyclic shift ⇒ implicit treap
need range query + range update (no insert/delete) ⇒ segment tree with lazy (simpler, faster)
need historical versions of the sequence ⇒ persistent implicit treap (tutorial 29)
need order-statistic queries on values ⇒ key treap (tutorial 27)

🎭 16 · Variants and Extensions

Variant A — non-commutative aggregates

If the aggregate is non-commutative (matrix product, prefix-sum, GCD over an ordered range), then range reverse must also reverse the aggregate. Add an apply_rev method on Info that flips its internal ordering, and call it inside apply_rev on the node. Without this, the aggregate is silently wrong after a reverse.

Variant B — persistent implicit treap

Clone before mutating: every split and every merge that descends into a subtree first creates a fresh copy of the node it is about to modify. Old versions remain immutable and addressable through stored root pointers. The cost is O(log n) extra allocations per operation. See tutorial 29 for the full persistent treatment.

Variant C — 2D implicit treap (treap of treaps)

For sequence-of-sequences problems where each row is itself a dynamic sequence: store an implicit treap of pointers to implicit treaps. Outer-treap operations split and merge rows; inner-treap operations split and merge within a row. The combined cost is O(log n × log m).

Variant D — offline batched build in O(n)

Instead of merging n single-node treaps in O(n log n), generate n random priorities, sort the priorities, and build a Cartesian tree in O(n). The result is a valid implicit treap whose inorder is the input array; useful when the initial sequence is provided up front.

Variant E — Sliding-window queries

If the workload is a fixed-size sliding window over a long stream, an implicit treap supports insert at the back and erase at the front, each in O(log n). For some aggregates (sum, max) a monotonic deque is faster, but the implicit treap handles arbitrary commutative-associative aggregates uniformly.

🎯 18 · Final Summary

One sentence: an implicit treap replaces the value-key BST invariant of tutorial 27 with implicit position-as-key, giving a dynamic sequence container with split / merge as the only primitives and expected O(log n) for insert at position, erase at position, range query, range update, and range reverse.

Five things to remember

First, position is implicit. We never store positions; we infer them on each descent from left-subtree sizes. Second, every operation reduces to split-then-merge. Insert is split + 2 merges; erase is 2 splits + 2 merges; range operations are 2 splits + 2 merges with work on the middle. Third, lazy push order is rev, then set_val, then add — any other order corrupts the aggregate. Fourth, the Info bundle is the single place to extend the template: add a new field, update combine, update apply_add and apply_set, and every operation continues to work. Fifth, no parent pointers, no rotations, no sentinel — the recursive split / merge style needs none of them.

Mastery checklist

  • I can write split and merge from memory in under 10 minutes.
  • I can hand-trace split(t, k) on a 5-element treap to show which pieces go left and which go right.
  • I can explain why range reverse needs a lazy flag and not just a structural swap.
  • I can extend Info with a min field, update combine, apply_add, apply_set, and have the template still work.
  • I know which problems prefer an implicit treap over a segment tree (anything with insert / erase / reverse / cyclic shift).
