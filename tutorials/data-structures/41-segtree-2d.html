Lesson 12 — 2D Segment Tree (Dense)

Outer over Rows · Inner over Columns · Rectangle Queries in O(log² n)

🧩 Problem Definition

Given a 2D grid a[0..R-1][0..C-1], support:

Cell update

set(r, c, v): replace a[r][c] with v.

Rectangle sum

sum(r1, c1, r2, c2): return the sum of a[r][c] for r1 ≤ r ≤ r2, c1 ≤ c ≤ c2.

When R and C are both moderate (≈ 10³), a 2D segment tree gives O(log R · log C) per operation. For static problems, 2D prefix sums are O(1) per query; the 2D segment tree shines when updates are needed.

💡 Intuition from Scratch

Build a segment tree over the rows. Each outer node corresponds to a contiguous range of rows. Inside that outer node, store a segment tree over the columns: the inner tree's leaves represent column indices, and its values aggregate the cells of the outer node's row range at that column.

So if outer node O covers rows [r1, r2] and inner node I within O covers columns [c1, c2], then I.sum = Σ a[r][c] over r ∈ [r1, r2], c ∈ [c1, c2] — a rectangle sum.

A rectangle query sum(r1, c1, r2, c2) then runs the outer descent for the row range and, at each fully-covered outer node, makes an inner column-range query for [c1, c2]. Total: O(log R) outer nodes, each costing O(log C), giving O(log R · log C).

🌳 Visual Mental Model

Imagine the grid drawn as a rectangle. Slice it into row-bands by the outer segment tree: the root spans all rows; each level of the tree splits the band in half; leaves are single rows. For every band, the corresponding inner tree is built over the columns, with leaves being the sum of the band's cells in each column.

An outer-leaf's inner tree is the row's column tree. An outer-internal-node's inner tree is the elementwise sum of its two children's inner trees — same shape, summed values. The shape is identical across all outer nodes; only the values differ.

📐 Formal Abstraction

Storage: a 2D array of segment-tree nodes. With both R and C rounded up to powers of two baseR and baseC, we allocate tree[2·baseR][2·baseC] — total 4 · baseR · baseC nodes. For R = C = 1024 this is ~4 · 10⁶ integers, which fits in memory.

An outer node at index i in the outer tree owns an inner array tree[i][·]. For an outer leaf i corresponding to row r, the inner array is a column segment tree over a[r][·]. For an outer internal node i with children 2i+1, 2i+2, the inner array is the elementwise sum of tree[2i+1][·] and tree[2i+2][·].

🔬 Key Observations

Observation 1 (rectangle as composition). A rectangle [r1, r2] × [c1, c2] is the intersection of a row range and a column range. The outer descent over rows produces O(log R) fully-covered row sub-bands; for each, we ask the inner tree for the column range — O(log C). Total O(log R · log C).

Observation 2 (sum-of-inner-trees). When two outer subtrees merge into a parent, their inner trees combine elementwise. Sum is associative and commutative, so this is well-defined; for min, max, gcd, xor, it works similarly. For non-additive composite states (like Lesson 2's max subarray sum), the merge is more delicate.

Observation 3 (update is O(log R · log C)). A cell update set(r, c, v) must update the inner trees of every outer node whose row range contains r — that's O(log R) outer nodes, each requiring an O(log C) inner update. Total: O(log R · log C) per cell update.

⚙️ Algorithm

build

Build the inner column trees of all outer leaves first (one per row). Then build the outer-internal nodes bottom-up: each outer internal node's inner tree is the elementwise sum of its two children's inner trees. Total work: O(R · C) for the leaf trees, plus O(R · C) for the internal sums.

cell update set(r, c, v)

Compute the delta d = v − a[r][c]. Then for every outer node on the path from outer-root to outer-leaf-r (there are O(log R) such nodes), update column c in that node's inner tree by +d. Each inner update is O(log C). Total: O(log R · log C).

rectangle query sum(r1, c1, r2, c2)

Descend the outer tree as usual. For a fully-covered outer node i, ask its inner tree for sum_inner([c1, c2]) and add to the answer. For partial overlap, recurse into both outer children. For no overlap, contribute 0.

🛡️ Invariants

✅ Invariant — Outer-inner consistency

For every outer node i covering row range [r1, r2] and every inner node j in i's inner tree covering column range [c1, c2], tree[i][j].sum equals the sum of a[r][c] over r ∈ [r1, r2], c ∈ [c1, c2].

✅ Invariant — Outer-merge invariant

For every outer internal node i with children 2i+1 and 2i+2, the inner array tree[i][·] is the elementwise sum of tree[2i+1][·] and tree[2i+2][·].

✅ Invariant — Update path invariant

After a cell update, every inner tree along the outer path has had its column c updated. No inner tree off the outer path is changed.

🧮 Proof of Correctness

🧮 Proof — Why the two-level descent returns the correct rectangle sum

The outer descent over rows decomposes [r1, r2] into O(log R) disjoint fully-covered sub-bands S_1, S_2, ..., S_k. Each S_j corresponds to an outer node with stored inner column tree representing the sum Σ_{r ∈ S_j, c} a[r][c] per column range.

For each S_j, the inner query over [c1, c2] returns Σ_{r ∈ S_j, c ∈ [c1, c2]} a[r][c] by the standard 1D segment tree proof. Summing over j gives the rectangle sum because the S_j partition [r1, r2]. ∎

🧰 Core Template Philosophy

A 2D segment tree is just “a segment tree where each node holds a segment tree”. The discipline is to write the outer routines in terms of inner-tree operations: outer build calls inner build for each leaf row, then outer build calls inner sum-merge for each internal outer node. Outer update calls inner update on every outer node on the row path. Outer query calls inner query on every fully-covered outer node.

The flat 2D array layout makes this straightforward: tree[outer_idx][inner_idx] with both indices using 0-based heap addressing.

💻 Master Template (C++)

#include <bits/stdc++.h>
using namespace std;

struct SegTree2D_Dense_Sum {
    using T = long long;

    int R = 0, C = 0;
    int baseR = 1, baseC = 1;
    vector<vector<T>> tree;  // tree[outer_idx][inner_idx]

    void init(const vector<vector<T>>& a) {
        R = (int)a.size();
        C = R ? (int)a[0].size() : 0;
        baseR = 1; while (baseR < max(1, R)) baseR <<= 1;
        baseC = 1; while (baseC < max(1, C)) baseC <<= 1;
        tree.assign(2 * baseR, vector<T>(2 * baseC, 0));

        for (int r = 0; r < R; r++) {
            int outer = baseR - 1 + r;
            for (int c = 0; c < C; c++) {
                tree[outer][baseC - 1 + c] = a[r][c];
            }
            for (int j = baseC - 2; j >= 0; j--) {
                tree[outer][j] = tree[outer][2*j+1] + tree[outer][2*j+2];
            }
        }

        for (int i = baseR - 2; i >= 0; i--) {
            int L = 2*i + 1, R2 = 2*i + 2;
            for (int j = 0; j < 2 * baseC; j++) {
                tree[i][j] = tree[L][j] + tree[R2][j];
            }
        }
    }

    void set(int r, int c, T value) {
        int outer = baseR - 1 + r;
        T delta = value - read_leaf(outer, c);
        for (int i = outer; ; i = (i - 1) / 2) {
            // Update column c in tree[i][·]
            int j = baseC - 1 + c;
            tree[i][j] += delta;
            while (j > 0) {
                j = (j - 1) / 2;
                tree[i][j] = tree[i][2*j+1] + tree[i][2*j+2];
            }
            if (i == 0) break;
        }
    }

    T read_leaf(int outer, int c) {
        return tree[outer][baseC - 1 + c];
    }

    T rectangle_sum(int r1, int c1, int r2, int c2) {
        return outer_query(0, 0, baseR - 1, r1, r2, c1, c2);
    }

    T outer_query(int i, int l, int r, int r1, int r2, int c1, int c2) {
        if (r2 < l || r < r1) return 0;
        if (r1 <= l && r <= r2) {
            return inner_query(i, 0, 0, baseC - 1, c1, c2);
        }
        int mid = l + (r - l) / 2;
        return outer_query(2*i+1, l,      mid, r1, r2, c1, c2)
             + outer_query(2*i+2, mid+1, r,   r1, r2, c1, c2);
    }

    T inner_query(int outer_i, int j, int l, int r, int c1, int c2) {
        if (c2 < l || r < c1) return 0;
        if (c1 <= l && r <= c2) return tree[outer_i][j];
        int mid = l + (r - l) / 2;
        return inner_query(outer_i, 2*j+1, l,      mid, c1, c2)
             + inner_query(outer_i, 2*j+2, mid+1, r,   c1, c2);
    }
};

🔎 Line-by-Line Explanation

init — build inner trees first, then merge them click to expand/collapse

The build is two passes. The first pass builds the inner column tree for every outer leaf (one per row of a): place the row's values at column leaves, then sweep up the column tree. The second pass walks the outer tree bottom-up; for each outer internal node, set its inner array to the elementwise sum of its two outer children's inner arrays. Total O(R · C).

set(r, c, v) — climb the outer path and update each inner tree click to expand/collapse

We compute the delta from the current leaf value. Then we walk from the outer leaf to the outer root, and at every outer node we apply a 1D segment tree point update at column c on the inner tree (climbing the column tree from leaf to root, adding delta at each step). Two nested 1D climbs: O(log R) outer · O(log C) inner.

rectangle_sum and outer_query / inner_query click to expand/collapse

The outer query is a standard 1D descent over rows. The novelty is what happens at full-overlap: instead of returning a precomputed scalar, we make a 1D column query inner_query(i, ...) on the inner tree of outer node i. The inner query is just Lesson 1's range-sum descent.

⚠️ Common Pitfalls

⚠️ Pitfall — Confusing the two tree indices

The outer and inner trees both use 0-based heap addressing, and it's easy to swap them. Conventional naming: use i for outer index and j for inner index, and always write tree[outer][inner].

⚠️ Pitfall — Forgetting to update the inner tree at every outer ancestor

A cell update at (r, c) must update column c in the inner tree of every outer node on the row path. Forgetting any one of them leaves stale data in that node's inner tree — queries that hit that node return wrong sums.

⚠️ Pitfall — Not initializing padded inner trees

If R isn't a power of 2, the padded outer leaves' inner trees should be zero everywhere. The default assign(...) with 0 handles this, but if you replace it with manual initialization, remember to handle the padded portion.

⚠️ Pitfall — Trying to use 2D segment tree for sparse coordinates

For grids where most cells are zero (or for huge coordinates like 10⁹), the dense version is wasteful. Switch to a 2D dynamic sparse segment tree (Lesson 13).

🎛️ Variants & Extensions

Variant 1 — Min / Max / GCD / XOR

Replace the elementwise sum-merge in the outer build with the appropriate operation. The inner tree's merge changes correspondingly. Identity values must be chosen carefully (LLONG_MAX for min, LLONG_MIN for max, 0 for gcd/xor).

Variant 2 — Iterative inner tree

Each inner tree can be an iterative segment tree (Lesson 3) for a faster constant. The outer tree usually stays recursive.

Variant 3 — Row updates + column queries (special case)

If updates are entire rows (not cells), only one inner tree needs updating per row update — just modify the row's inner tree, and the outer ancestors' inner trees automatically reflect via the build-time merging. (Not, however, after the fact — you must re-merge.)

Variant 4 — Persistent 2D segment tree

Apply path-copying to the outer tree, sharing inner trees that don't change. Memory O(log R · log C) per update version.

🏁 Final Summary

A 2D segment tree is a segment tree over rows whose every node holds a segment tree over columns. Rectangle queries decompose as an outer descent over rows; at each fully-covered outer node, an inner column query gives the rectangle sum for that row band. Total cost: O(log R · log C) per operation.

The structure is simple to write but memory-hungry: O(R · C) total. For sparse grids or huge coordinates, switch to the dynamic sparse 2D variant (Lesson 13), which materializes inner nodes on demand. The dense version is the right tool when both dimensions are moderate (≤ 10³) and updates are frequent.
