Fenwick Tree / Binary Indexed Tree

Complete ICPC handbook — visual block diagrams, step-by-step interactive tracers, correctness proofs, the full master template, and 11 variants with 10 fully solved contest problems.
Theme

§0 — Problem Definition

Given an array a[1..n], support two operations simultaneously and efficiently: modify one element, and query the sum of a contiguous prefix or range. A plain prefix-sum array handles queries in O(1) but each update costs O(n). A Fenwick Tree gives O(log n) for both — without any pointers, heap allocation, or complex structure. It is just a cleverly indexed plain array.

The tension

A static prefix array pref[i] = a[1]+…+a[i] answers sum(l,r) in O(1) via pref[r]−pref[l−1]. But updating a[i] forces rebuilding every pref[j] for j ≥ i — O(n) per update. We cannot have both. Or can we?

BIT's answer

Instead of storing every prefix, store O(log n) carefully chosen canonical blocks that together cover any prefix. Updating one position touches only O(log n) blocks. Querying any prefix reads only O(log n) blocks. Both in O(log n) with tiny constants.

§1 — Intuition from Scratch

The lowbit operation

Everything in a Fenwick Tree flows from one tiny operation: lowbit(i) = i & (-i). This isolates the least significant set bit of i. For example, i=6 is 0110₂, so lowbit(6) = 2 (the 0010₂ bit). For i=4 (0100₂), lowbit(4) = 4. For i=7 (0111₂), lowbit(7) = 1.

Why does i & (-i) work? Two's complement negation flips all bits and adds 1. The lowest set bit survives because all lower bits become 0 and the negation carry propagates exactly to that bit. This is a beautiful trick that costs one instruction.

What each cell stores

Each index i in the BIT "owns" a block of size lowbit(i) ending at i:

bit[i] = a[i - lowbit(i) + 1] + a[i - lowbit(i) + 2] + ... + a[i]

How prefix queries work

To compute sumPrefix(13), note that 13 = 1101₂. The query loop strips one bit per iteration:

i = 13   → add bit[13] (covers [13,13]), then i -= lowbit(13) = 1  → i = 12
i = 12   → add bit[12] (covers [9,12]),  then i -= lowbit(12) = 4  → i = 8
i = 8    → add bit[8]  (covers [1,8]),   then i -= lowbit(8)  = 8  → i = 0
i = 0    → stop

The three blocks [1,8], [9,12], [13,13] partition [1,13] exactly — no overlap, no gap.

How updates work

When we update a[6], which bit[i] cells need to change? Only those whose stored range contains position 6. The update loop climbs upward by adding the lowbit:

i = 6   → bit[6] covers [5,6]: includes 6 ✓. Update, then i += lowbit(6) = 2  → i = 8
i = 8   → bit[8] covers [1,8]: includes 6 ✓. Update, then i += lowbit(8) = 8  → i = 16
i = 16  → if n < 16, stop. Otherwise continue.

Each step adds one bit, climbing to the next "ancestor" block. The ascent terminates in O(log n) steps because each step sets a strictly higher bit.

Mental model: Each bit[i] is a "manager" responsible for a block ending at i of size lowbit(i). Querying decomposes a prefix into non-overlapping manager blocks (strip bits downward). Updating notifies every manager whose block covers the changed position (add bits upward). The binary representation does the decomposition automatically.

§3 — Formal Abstraction

A Fenwick Tree is a dynamic prefix-decomposition structure over a group with an invertible binary operation. The standard version uses addition over integers (or long long), but the idea extends to any group where the operation has an inverse — this is precisely why sums work and minimum does not (minimum has no inverse).

The canonical-block property

For every integer r in [1, n], the sequence obtained by repeatedly stripping the least significant bit:

r → r - lowbit(r) → r - lowbit(r) - lowbit(r - lowbit(r)) → ... → 0

produces a set of disjoint intervals that partition [1, r] exactly. This is a theorem about binary representations: the set bits of r encode exactly the block sizes needed. The partition has at most ⌊log₂ r⌋ + 1 ≤ log₂ n + 1 blocks.

The ancestor property

For any position pos, the sequence obtained by repeatedly adding the least significant bit:

pos → pos + lowbit(pos) → pos + lowbit(pos) + lowbit(pos + lowbit(pos)) → ... → (first power of 2 > n)

visits exactly the indices i such that bit[i]'s block contains pos. Each step sets a strictly higher bit, so the sequence terminates in O(log n) steps and visits O(log n) indices.

§4 — Key Observations

① Block Ownership
bit[i] stores the sum of exactly the lowbit(i) elements ending at i. No more, no less. The block [i-lowbit(i)+1 .. i] is its sole responsibility.
② Query strips bits
i -= i & -i removes the last set bit, jumping from the end of a block to the end of the remaining prefix. This is the query descent — at each step we read one block and remove it.
③ Update adds bits
i += i & -i sets the next higher bit, climbing to the next ancestor block that also covers the updated position. All ancestors are visited exactly once.
④ Logarithmic depth
Both loops touch at most one index per bit of n. Since n has at most ⌊log₂ n⌋+1 bits, each operation is O(log n). For n=10⁶ that is at most 20 iterations.

§5 — Invariants

Storage Invariant (the master invariant)

For every index i in [1, n]:
bit[i] = a[i - lowbit(i) + 1] + a[i - lowbit(i) + 2] + … + a[i]
This invariant must hold at all times. Every BIT operation is designed to maintain it.

Query Invariant

During sumPrefix(r), maintain the predicate: ans + sum(a[1..i]) = sum(a[1..r]), where i is the current loop variable. Initially i=r and ans=0. Each iteration adds bit[i] (which equals sum(a[i-lowbit(i)+1..i])) to ans and subtracts that block from i, keeping the invariant. When i=0 the uncovered prefix is empty, so ans equals the full prefix sum.

Update Invariant

During add(pos, delta), the loop visits exactly the set of indices i such that bit[i]'s block contains pos. Adding delta to each visited bit[i] restores the storage invariant because the sum stored in each such cell increases by exactly delta. Cells whose blocks do not contain pos are not visited and their invariant is unaffected.

Termination

Both loops terminate because: in the query loop, i -= lowbit(i) strictly decreases i (it clears a set bit), so i reaches 0 in finitely many steps. In the update loop, i += lowbit(i) strictly increases i (it sets a higher bit), so i exceeds n in finitely many steps.

§6 — Proof of Correctness

Theorem: BIT correctly answers prefix sum queries after any sequence of point additions. Click to expand/collapse

Base case: After initializing all cells to zero, the storage invariant holds trivially since all a[i] = 0.

Preservation under add(pos, delta): The set of indices i satisfying "the block of bit[i] contains pos" is exactly the set visited by i += lowbit(i) starting from pos. This is because bit[i]'s block is [i-lowbit(i)+1, i]; it contains pos iff i-lowbit(i) < pos ≤ i, which characterizes exactly the ancestor chain in binary. Adding delta to each such bit[i] increases by delta the sum of each block containing pos, which is exactly what happens to a[pos]. All other cells are untouched. Therefore the storage invariant is restored.

Correctness of sumPrefix(r): At each loop step with current index i, the algorithm reads bit[i] which equals the sum of block [i-lowbit(i)+1, i]. Then i is moved to i-lowbit(i), the index just before this block. The blocks read are non-overlapping (each new i is strictly before the start of the previous block) and they cover [1, r] in reverse order. Therefore their sum equals a[1]+a[2]+…+a[r]. ∎

Corollary: sumRange(l, r) is correct. Click to expand

sumRange(l, r) = sumPrefix(r) − sumPrefix(l−1). This works because addition is a group operation with subtraction as inverse. The identity a[l]+…+a[r] = (a[1]+…+a[r]) − (a[1]+…+a[l−1]) is elementary. This is also exactly why Fenwick Trees work naturally for sums but not minimum — min has no inverse. ∎

Theorem: O(n) build is correct. Click to expand

The O(n) build avoids calling add n times (which would be O(n log n)). Instead, for each i, it adds a[i] directly to bit[i] and then propagates to the parent i + lowbit(i). This is a bottom-up aggregation: each block's value is accumulated from smaller blocks that partition it. By induction on the block sizes, when index i is processed, all smaller blocks whose ranges are subsets of [i-lowbit(i)+1, i] have already been processed and their sums propagated to i. Therefore the final bit[i] correctly equals the sum of its block. ∎

Constant factor analysis

In practice BIT is roughly 2–4× faster than a segment tree for point-update range-sum problems. The reasons are: no pointer chasing, perfect cache locality (a short stride along a flat array), minimal branches, and only one or two loop variables. Each "iteration" is literally: read/write one cell, one bitwise AND, one addition or subtraction. This fits in a handful of CPU instructions.

2D extension

The 2D BIT (variant §V4) runs two nested loops — O(log n · log m) per operation — using O(n·m) memory. For n=m=5000, that is 25M entries of 8 bytes = 200 MB. Dense 2D BIT requires care about memory limits; use compression when coordinates are large but the number of updates is small.

§8 — Template Philosophy

All Fenwick Tree variants in this document derive from a single parameterized struct Fenwick<T>. The template type T controls the value domain (int, long long, double, modular int). Higher-level structures like RangeAddRangeSum or Fenwick2D wrap or compose the base struct. This keeps naming, loop logic, and indexing identical across every variant — reducing bugs and mental load under contest pressure.

What never changes
• i += i & -i in add
• i -= i & -i in sumPrefix
• 1-indexed externally
• bit[0] unused
• T{} as zero value
What changes per variant
• Number of BIT instances (1, 2, or nested)
• The outer wrapper logic
• How indices map to semantic positions
• Whether we call add directly or wrap it
1-based vs 0-based
This tutorial uses 1-based externally throughout. Inputs in [0, n-1] need idx++. Half-open intervals [l, r) use sumRange(l+1, r). Never mix conventions in the same file.

§9 — Master Template (AC-Ready)

#include <bits/stdc++.h>
using namespace std;

template <class T>
struct Fenwick {
    int n;
    vector<T> bit;

    Fenwick() : n(0) {}

    Fenwick(int n_) {
        init(n_);
    }

    void init(int n_) {
        n = n_;
        bit.assign(n + 1, T{});
    }

    // Add delta to a[idx].  idx is 1-indexed.
    void add(int idx, T delta) {
        for (int i = idx; i <= n; i += i & -i)
            bit[i] += delta;
    }

    // Return a[1] + a[2] + ... + a[idx].  idx is 1-indexed.
    T sumPrefix(int idx) const {
        T ans{};
        for (int i = idx; i > 0; i -= i & -i)
            ans += bit[i];
        return ans;
    }

    // Return a[l] + ... + a[r].  l and r are 1-indexed, inclusive.
    T sumRange(int l, int r) const {
        if (l > r) return T{};
        return sumPrefix(r) - sumPrefix(l - 1);
    }

    // Build from a 1-indexed array: a[0] ignored, a[1..n] used.
    void build(const vector<T>& a) {
        init((int)a.size() - 1);
        for (int i = 1; i <= n; ++i) {
            bit[i] += a[i];
            int parent = i + (i & -i);
            if (parent <= n) bit[parent] += bit[i];
        }
    }

    // First index pos where sumPrefix(pos) >= target.
    // REQUIRES: all values in the BIT are nonnegative.
    // Returns n+1 if no such pos exists.
    int lowerBound(T target) const {
        if (target <= T{}) return 1;
        int idx = 0;
        T current{};
        int step = 1;
        while ((step << 1) <= n) step <<= 1;
        for (; step > 0; step >>= 1) {
            int next = idx + step;
            if (next <= n && current + bit[next] < target) {
                idx = next;
                current += bit[next];
            }
        }
        return idx + 1;
    }

    // First index pos where sumPrefix(pos) > target.
    // REQUIRES: all values in the BIT are nonnegative.
    int upperBound(T target) const {
        int idx = 0;
        T current{};
        int step = 1;
        while ((step << 1) <= n) step <<= 1;
        for (; step > 0; step >>= 1) {
            int next = idx + step;
            if (next <= n && current + bit[next] <= target) {
                idx = next;
                current += bit[next];
            }
        }
        return idx + 1;
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    // Demo — 1-indexed array of size 5: a=[5,1,3,7,2]
    vector<long long> a = {0, 5, 1, 3, 7, 2};
    Fenwick<long long> fw;
    fw.build(a);

    cout << fw.sumRange(2, 4) << '\n'; // 1+3+7 = 11
    fw.add(3, 10);                       // a[3] += 10
    cout << fw.sumRange(2, 4) << '\n'; // 1+13+7 = 21
    cout << fw.lowerBound(12) << '\n'; // first pos with prefix >= 12

    return 0;
}

§10 — Line-by-Line Explanation

template <class T> struct Fenwick Type flexibility

The template parameter T allows Fenwick<int>, Fenwick<long long>, Fenwick<double>, or Fenwick<ModInt>. In contest code, always use long long unless values are guaranteed small — sums of n=10⁵ elements each up to 10⁹ can reach 10¹⁴, overflowing int.

bit.assign(n + 1, T{}) Why n+1?

The BIT is 1-indexed: valid positions are 1..n. We allocate n+1 cells so that bit[n] is valid. bit[0] is allocated but never read or written. The expression T{} value-initializes T to its zero — 0 for int/long long, 0.0 for double, the identity element for any custom type.

for (int i = idx; i <= n; i += i & -i) in add The ascent

i & -i extracts the lowest set bit of i. Adding it to i clears that bit and sets the next higher bit (or creates a new one). This is the "climb to ancestor" operation. Each iteration visits one ancestor block. The loop terminates when i exceeds n.

for (int i = idx; i > 0; i -= i & -i) in sumPrefix The descent

Subtracting the lowest set bit clears it, moving i to the position just before the last covered block. This decomposes the prefix into non-overlapping canonical blocks. The loop terminates when all bits are cleared (i = 0).

sumRange(l, r) = sumPrefix(r) - sumPrefix(l-1) The range trick

This works because addition is invertible: a[l]+…+a[r] = (a[1]+…+a[r]) - (a[1]+…+a[l-1]). The subtraction undoes the prefix up to l-1. This is the deepest reason why BIT works for sums and not minimum — minimum has no inverse operation.

build — why O(n) instead of O(n log n)? The parent push

Calling add(i, a[i]) for each i would be O(n log n). Instead, the O(n) build directly deposits a[i] into bit[i] and then pushes the partial sum to the parent i + lowbit(i). Because we process i in increasing order and each parent index is strictly greater than i, every block receives contributions from all its sub-blocks exactly once — a one-pass bottom-up sweep.

lowerBound — binary lifting on the BIT O(log n) kth element

This replaces a binary search over sumPrefix(mid) (which would be O(log²n)) with a direct O(log n) descent that walks the BIT tree from the highest power of 2 downward. At each step we speculatively add a block: if the running sum plus bit[idx+step] is still below the target, we commit that jump; otherwise we skip it. After all powers of 2 are tried, idx+1 is the answer. Critical: this is only valid when all BIT values are nonnegative, because prefix sums must be monotone for binary search to be valid.

V1 Point Add + Range Sum

The baseline. Use the master template directly.

When to use

Point updates, range sum queries, dynamic arrays where values change one position at a time. Classic example: AtCoder Library Practice Problem B.

When NOT to use

Static arrays (use prefix sum). Range minimum with arbitrary updates (use segment tree). Range assignment (use lazy segment tree).

// V1: Point add + range sum — use Fenwick<long long> directly.
Fenwick<long long> fw(n);
fw.add(i, x);           // point update
fw.sumRange(l, r);      // range sum
fw.sumPrefix(r);        // prefix sum
O(log n) update and query
Memory O(n)

V2 Range Add + Point Query

Store the difference array in a BIT. Range add becomes two point updates; point query becomes a prefix sum.

Key idea

Let diff[i] = a[i] - a[i-1]. Then a[i] = diff[1]+diff[2]+…+diff[i]. Adding x to [l, r] means diff[l] += x, diff[r+1] -= x. Point query at i is a prefix sum of diff up to i.

template <class T>
struct RangeAddPointQuery {
    int n;
    Fenwick<T> fw;

    RangeAddPointQuery(int n_ = 0) { init(n_); }

    void init(int n_) { n = n_; fw.init(n); }

    void rangeAdd(int l, int r, T delta) {
        if (l > r) return;
        fw.add(l, delta);
        if (r + 1 <= n) fw.add(r + 1, -delta);
    }

    T pointQuery(int idx) const {
        return fw.sumPrefix(idx);
    }
};

// Usage:
RangeAddPointQuery<long long> ds(n);
ds.rangeAdd(l, r, x);    // add x to a[l..r]
ds.pointQuery(k);         // retrieve a[k]
O(log n) both operations
Don't forget the r+1 sentinel

V3 Range Add + Range Sum

Two BITs. One stores coefficients of p; the other stores constants. This derivation is the deepest trick in BIT theory.

Derivation

Adding x to [l, r] contributes to sumPrefix(p):

0                 if p < l
x*(p - l + 1)     if l <= p <= r
x*(r - l + 1)     if p > r

This factors as p * B1.prefix(p) - B2.prefix(p) where:

B1[l]     += x;    B1[r+1] -= x;
B2[l]     += x*(l-1);  B2[r+1] -= x*r;
prefix(p) = p * B1.prefix(p) - B2.prefix(p)
template <class T>
struct RangeAddRangeSum {
    int n;
    Fenwick<T> b1, b2;

    RangeAddRangeSum(int n_ = 0) { init(n_); }

    void init(int n_) { n = n_; b1.init(n); b2.init(n); }

    void rangeAdd(int l, int r, T x) {
        if (l > r) return;
        b1.add(l, x);       b2.add(l, x * (l - 1));
        if (r+1 <= n) { b1.add(r+1, -x); b2.add(r+1, -x * r); }
    }

    T sumPrefix(int p) const {
        return b1.sumPrefix(p) * p - b2.sumPrefix(p);
    }

    T sumRange(int l, int r) const {
        if (l > r) return T{};
        return sumPrefix(r) - sumPrefix(l - 1);
    }
};
O(log n) all operations
Use long long — multiplications overflow int easily

V4 2D Fenwick Tree

Nest two BIT loops: outer over x-axis, inner over y-axis. Point update and rectangle sum both in O(log n · log m).

template <class T>
struct Fenwick2D {
    int n, m;
    vector<vector<T>> bit;

    Fenwick2D(int n_ = 0, int m_ = 0) { init(n_, m_); }

    void init(int n_, int m_) {
        n = n_; m = m_;
        bit.assign(n+1, vector<T>(m+1, T{}));
    }

    void add(int x, int y, T delta) {
        for (int i = x; i <= n; i += i & -i)
            for (int j = y; j <= m; j += j & -j)
                bit[i][j] += delta;
    }

    T sumPrefix(int x, int y) const {
        T ans{};
        for (int i = x; i > 0; i -= i & -i)
            for (int j = y; j > 0; j -= j & -j)
                ans += bit[i][j];
        return ans;
    }

    T sumRectangle(int x1, int y1, int x2, int y2) const {
        if (x1 > x2 || y1 > y2) return T{};
        return sumPrefix(x2,y2) - sumPrefix(x1-1,y2)
             - sumPrefix(x2,y1-1) + sumPrefix(x1-1,y1-1);
    }
};
O(log n log m) per op
Memory O(nm) — check limits carefully
5000×5000 BIT of long long ≈ 200 MB — too large without care

V6 Order Statistic Multiset

BIT + compression

If all possible values are known in advance, compress them to [1..m] and use a BIT. Insert = add(rank, +1); erase = add(rank, -1); count ≤ x = sumPrefix(rank[x]); kth = lowerBound(k). Handles duplicates naturally.

PBDS instead when

Values are online and unknown in advance; you cannot compress. Or when you need direct ordered iteration. __gnu_pbds::tree gives O(log n) rank and kth without compression but with a larger constant factor.

V7 Inversion Counting

Scan right to left. For each element, query how many already-inserted elements are strictly smaller. That count is exactly the inversions contributed by the current element. Then insert the current element.

long long countInversions(vector<long long> a) {
    // Coordinate compress
    vector<long long> vals = a;
    sort(vals.begin(), vals.end());
    vals.erase(unique(vals.begin(), vals.end()), vals.end());

    Fenwick<long long> fw((int)vals.size());
    long long ans = 0;

    for (int i = (int)a.size() - 1; i >= 0; --i) {
        int rank = (int)(lower_bound(vals.begin(), vals.end(), a[i]) - vals.begin()) + 1;
        ans += fw.sumPrefix(rank - 1);  // count elements < a[i] already inserted
        fw.add(rank, 1);
    }
    return ans;
}
O(n log n)
Use rank-1 for strict inversions (a[i] > a[j])
Use long long for answer — up to n²/2 ≈ 5×10¹⁰

V8 Euler Tour + BIT on Trees

The Euler tour flattens tree structure to a contiguous array interval. BIT then handles subtree and root-path queries via standard point-update / range-add patterns.

Subtree sum

DFS to compute tin[u] and tout[u]. Subtree of u = interval [tin[u], tout[u]]. Update node u = point add at tin[u]. Sum of subtree = sumRange(tin[u], tout[u]). Uses V1 (point add + range sum).

Root-to-node path sum

Value at node v contributes to every descendant's root-path sum. So update v = rangeAdd(tin[v], tout[v], delta). Query path to u = pointQuery(tin[u]). Uses V2 (range add + point query).

V9 Offline BIT Sweeps

Read all queries upfront, reorder processing for efficiency, then output answers in original order. Two canonical patterns:

KQUERY pattern

Count elements > k in a[l..r]. Sort array values descending. Sort queries by k descending. For each query, insert positions of values > k that haven't been inserted yet. Query BIT sum over [l, r].

DQUERY pattern

Count distinct values in a[l..r]. Process positions left to right. For each position i, remove previous occurrence of a[i] from BIT, add current occurrence. Answer queries grouped by right endpoint r = sumRange(l, i).

V10 DP with BIT

When a DP transition is "sum of all dp[j] where value[j] < value[i]", this is a prefix query over compressed value ranks — exactly what a BIT answers in O(log n).

// Counting increasing subsequences of length K
// For each element x with rank r:
//   for len from K down to 2:
//     ways[len] = dp[len-1].sumPrefix(r - 1)
//     dp[len].add(r, ways[len])
//   dp[1].add(r, 1)
//
// Final answer: dp[K].sumPrefix(m)
// where m = number of distinct values

The inner loop goes downward from K to 2 to prevent using the same element twice in one subsequence — the same logic as 0/1 knapsack. This pattern appears in LIS counting, rank-based DP, and offline subsequence problems.

V11 Min BIT — Limitations

A BIT can support prefix minimum queries, but ONLY when all updates are monotone decreasing (i.e., you only take min, never increase values). There is no general range minimum BIT.

struct FenwickMin {
    int n;
    const long long INF = 1LL << 60;
    vector<long long> bit;

    FenwickMin(int n_ = 0) { init(n_); }

    void init(int n_) { n = n_; bit.assign(n+1, INF); }

    // Update: take min. Values can only decrease.
    void updateMin(int idx, long long value) {
        for (int i = idx; i <= n; i += i & -i)
            bit[i] = min(bit[i], value);
    }

    // Prefix min: minimum of a[1..idx].
    long long prefixMin(int idx) const {
        long long ans = INF;
        for (int i = idx; i > 0; i -= i & -i)
            ans = min(ans, bit[i]);
        return ans;
    }
};
Cannot do range min with arbitrary updates — use segment tree
prefixMin only, NOT rangeMin(l,r) — min has no inverse
Useful for offline DP where values only decrease

P4 — MATSUM (SPOJ MATSUM) V4 2D BIT with point-set

Problem: Multiple test cases. N×N matrix, 0-indexed. SET x y val sets a cell; SUM x1 y1 x2 y2 sums a rectangle; END ends the test case. VJudge link

Key insight: BIT supports add, not direct set. Store current matrix values to compute delta. All 0-based input coordinates are incremented to 1-based before BIT calls.

#include <bits/stdc++.h>
using namespace std;

template <class T>
struct Fenwick2D {
    int n, m;
    vector<vector<T>> bit;
    Fenwick2D(int n_=0,int m_=0){init(n_,m_);}
    void init(int n_,int m_){
        n=n_;m=m_;
        bit.assign(n+1,vector<T>(m+1,T{}));
    }
    void add(int x,int y,T d){
        for(int i=x;i<=n;i+=i&-i)
            for(int j=y;j<=m;j+=j&-j)
                bit[i][j]+=d;
    }
    T sum(int x,int y) const {
        T a{};
        for(int i=x;i>0;i-=i&-i)
            for(int j=y;j>0;j-=j&-j)
                a+=bit[i][j];
        return a;
    }
    T rect(int x1,int y1,int x2,int y2) const {
        return sum(x2,y2)-sum(x1-1,y2)-sum(x2,y1-1)+sum(x1-1,y1-1);
    }
};

int main(){
    ios::sync_with_stdio(false);cin.tie(nullptr);
    int tc;cin>>tc;
    while(tc--){
        int n;cin>>n;
        Fenwick2D<long long> fw(n,n);
        vector<vector<long long>> a(n+1,vector<long long>(n+1,0));
        string cmd;
        while(cin>>cmd){
            if(cmd=="END") break;
            if(cmd=="SET"){
                int x,y;long long val;cin>>x>>y>>val;
                ++x;++y;                     // 0-based → 1-based
                long long delta=val-a[x][y];
                a[x][y]=val;
                fw.add(x,y,delta);
            } else { // SUM
                int x1,y1,x2,y2;cin>>x1>>y1>>x2>>y2;
                ++x1;++y1;++x2;++y2;         // 0-based → 1-based
                cout<<fw.rect(x1,y1,x2,y2)<<'\n';
            }
        }
        if(tc) cout<<'\n';
    }
}

Every input coordinate from the problem is 0-based; each is incremented by 1 before being passed to the 1-based Fenwick2D. The current matrix a[][] tracks values so SET can compute a delta instead of assigning directly.

P5 — Multiset kth Deletion (CF 1354D) V5 lowerBound

Problem: Given n initial values and q queries. Positive query x inserts x into the multiset. Negative query −k deletes the kth smallest element. At the end, print the smallest remaining element or 0 if empty. Values are in [1..n]. VJudge link

#include <bits/stdc++.h>
using namespace std;

template <class T>
struct Fenwick {
    int n; vector<T> bit;
    Fenwick(int n_=0){init(n_);}
    void init(int n_){n=n_;bit.assign(n+1,T{});}
    void add(int i,T d){for(;i<=n;i+=i&-i)bit[i]+=d;}
    T sum(int i){T a{};for(;i>0;i-=i&-i)a+=bit[i];return a;}
    // First pos where prefix sum >= target. All values must be nonneg.
    int lowerBound(T target) const {
        if(target<=T{})return 1;
        int idx=0; T cur{};
        int step=1;
        while((step<<1)<=n)step<<=1;
        for(;step>0;step>>=1){
            int nxt=idx+step;
            if(nxt<=n && cur+bit[nxt]<target){idx=nxt;cur+=bit[nxt];}
        }
        return idx+1;
    }
};

int main(){
    ios::sync_with_stdio(false);cin.tie(nullptr);
    int n,q;cin>>n>>q;
    Fenwick<int> fw(n);
    // Load initial multiset
    for(int i=0;i<n;i++){int x;cin>>x;fw.add(x,1);}
    // Process queries
    while(q--){
        int x;cin>>x;
        if(x>0){
            fw.add(x,1);           // insert x
        } else {
            int k=-x;
            int pos=fw.lowerBound(k); // find kth smallest
            fw.add(pos,-1);        // remove it
        }
    }
    // Output answer
    if(fw.sum(n)==0) cout<<0<<'\n';
    else             cout<<fw.lowerBound(1)<<'\n';
}

Values fit in [1..n] so no coordinate compression is needed. lowerBound is valid because all frequencies are nonnegative (we only delete existing elements).

P9 — Path Queries (CSES 1138) V8 Euler Tour + V2

Sum of node values on path from root (node 1) to node s. Update a single node's value. The key insight: a node v's value contributes to every descendant's root-path sum. So updating v by delta is a range add over its subtree interval [tin[v], tout[v]], and querying node s is a point query at tin[s]. CSES link

#include <bits/stdc++.h>
using namespace std;

// Difference-array BIT: range add + point query
template<class T>struct Fenwick{
    int n;vector<T>bit;
    Fenwick(int n_=0){init(n_);}
    void init(int n_){n=n_;bit.assign(n+1,T{});}
    void add(int i,T d){for(;i<=n;i+=i&-i)bit[i]+=d;}
    T sum(int i){T a{};for(;i>0;i-=i&-i)a+=bit[i];return a;}
    // range add [l,r] by delta
    void rangeAdd(int l,int r,T d){
        add(l,d);
        if(r+1<=n)add(r+1,-d);
    }
    // point query at idx
    T pointQuery(int idx){return sum(idx);}
};

int main(){
    ios::sync_with_stdio(false);cin.tie(nullptr);

    int n,q;cin>>n>>q;
    vector<long long>val(n+1);
    for(int i=1;i<=n;i++)cin>>val[i];

    vector<vector<int>>adj(n+1);
    for(int i=0;i<n-1;i++){
        int u,v;cin>>u>>v;
        adj[u].push_back(v);
        adj[v].push_back(u);
    }

    // Euler tour using iterative DFS to avoid recursion depth issues.
    vector<int>tin(n+1),tout(n+1);
    int timer=0;

    struct Frame{int u,p,ci;};
    stack<Frame>st;
    st.push({1,0,0});
    tin[1]=++timer;

    while(!st.empty()){
        Frame&f=st.top();
        if(f.ci<(int)adj[f.u].size()){
            int v=adj[f.u][f.ci++];
            if(v==f.p)continue;
            tin[v]=++timer;
            st.push({v,f.u,0});
        } else {
            tout[f.u]=timer;
            st.pop();
        }
    }

    Fenwick<long long>fw(n);

    // Each node's value contributes to all its descendants' root-path sums
    for(int u=1;u<=n;u++)
        fw.rangeAdd(tin[u],tout[u],val[u]);

    while(q--){
        int t;cin>>t;
        if(t==1){
            int s;long long x;cin>>s>>x;
            long long delta=x-val[s];
            val[s]=x;
            fw.rangeAdd(tin[s],tout[s],delta);
        } else {
            int s;cin>>s;
            cout<<fw.pointQuery(tin[s])<<'\n';
        }
    }
}

The iterative DFS uses a struct-stack with a child-index counter to correctly compute tin/tout without recursion, avoiding stack overflow for n up to 2×10⁵. rangeAdd over [tin[u], tout[u]] encodes that u's value affects every descendant's path sum.

§ Edge Case Audit

📌
1-indexed vs 0-indexed: BIT uses 1-indexed positions internally. If input is 0-indexed, add 1 to every index before calling BIT. If query is half-open [l, r), use sumRange(l+1, r).
⚠️
Overflow: Always use long long for sums. Values up to 10⁹ summed over n=10⁵ elements reach 10¹⁴, overflowing 32-bit int. In V3 (range add range sum), the intermediate product x * (l-1) can also overflow if x and l are both large — use long long consistently.
🚫
lowerBound validity: Only call lowerBound when all values added to the BIT are nonnegative. This ensures prefix sums are monotone. Calling lowerBound on a BIT with negative values produces wrong answers silently.
📌
Range add sentinel: In V2 and V3, always write if (r+1 <= n) fw.add(r+1, -delta). Forgetting this causes the subtraction to never happen, making every subsequent point query wrong for positions after r.
⚠️
Coordinate compression: When values are in [0, 10⁹], the BIT size must be the number of distinct values, not the max value. Compress with sort + unique + lower_bound. Always get ranks in [1, m] not [0, m-1].
📌
Duplicate handling in inversion count: For strict inversions (a[i] > a[j]), query sumPrefix(rank-1), not sumPrefix(rank). Using rank would count equal elements as inversions.
⚠️
2D memory: A dense Fenwick2D<long long> of size 5000×5000 uses 25M entries × 8 bytes = 200 MB. Check the problem memory limit. For large coordinate spaces with few updates, use coordinate compression or a map-based BIT.
📌
Euler Tour tout: Confirm whether tout[u] is inclusive or exclusive in your DFS implementation and be consistent when calling sumRange or rangeAdd on the interval [tin[u], tout[u]].
🚫
Min BIT validity: The prefix-min BIT only works when all updateMin calls are monotone decreasing. If you mix increasing and decreasing updates, the storage invariant breaks and results are wrong.

§ Common Pitfalls

Calling sumRange with l > r Returns wrong value if unguarded

If l > r, sumPrefix(r) - sumPrefix(l-1) returns a negative or meaningless value. Always guard with if (l > r) return T{} at the start of sumRange — the master template does this, but custom code may forget it.

Forgetting to reinitialize between test cases Classic SPOJ multi-test bug

SPOJ problems often have multiple test cases. Always call fw.init(n) at the start of each test case, or declare the Fenwick object inside the test-case loop. Reusing a BIT from the previous test case gives WA silently.

Using int when long long is needed Silent overflow

The BIT compiles fine with int; the overflow is silent. Always check: (max value) × n. If this exceeds 2³¹ ≈ 2×10⁹, use long long. In V3, intermediate products like x*(l-1) need long long even if individual values are small.

Wrong answer from offline sort order KQUERY / DQUERY

In KQUERY, sort array values descending AND queries by k descending. Both must be descending — sorting one ascending and one descending is a common bug. Always insert elements with value > query.k, not >= query.k, since the problem asks "strictly greater than k".

lowerBound returning n+1 when target is reachable Off-by-one in step initialization

The step initialization loop while ((step << 1) <= n) step <<= 1 must produce the highest power of 2 that is ≤ n. Starting with step=0 causes an infinite loop. Starting with step=n causes wrong results if n is not a power of 2. The correct starting value is 1 with the while loop.

The deepest recognition insight: BIT works when the aggregate is a group operation with an inverse (sum, XOR, etc.) AND the problem decomposes into prefix queries. When the aggregate is a monoid without inverse or the queries are not prefix-decomposable, use a segment tree instead. The one exception taught in V11 is the prefix-min BIT, which supports monotone decreasing updates and prefix minimum — but only under that strict monotonicity condition, and it does not support arbitrary range minimum or updates that increase values.

§ Final Summary

A Fenwick Tree is a compact structure for dynamic prefix information. Its core idea is binary decomposition: every prefix decomposes into O(log n) canonical blocks, each block is owned by exactly one BIT cell, and every point update touches exactly the blocks that contain it. The master invariant is bit[i] = sum(a[i-lowbit(i)+1 .. i]). Everything else follows.

The two micro-operations i += i & -i (climb ancestors for update) and i -= i & -i (strip bits for query) are the heartbeat of the entire structure. Once you see them clearly — as binary representations being built up and torn down — every BIT variant becomes a natural extension.

For range queries built from prefix subtraction, BIT works when the aggregate has an inverse (sum, XOR). Without an inverse, a standard BIT cannot answer arbitrary range queries — reach for a segment tree instead. The restricted exception taught in V11 is the prefix-min BIT: it works only for monotone decreasing updates and prefix-minimum queries, not arbitrary range minimum. Outside that narrow case, the inverse requirement is absolute. Everything else is a template adaptation.
