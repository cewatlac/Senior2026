Lesson 5 — Persistent Segment Tree

Path Copying · K-th Smallest in Range · Frozen History · O(log n) per Update

🧩 Problem Definition

You are given a static array a[0..n−1]. You must answer many queries of the form: given l, r, k, what is the k-th smallest value among a[l], a[l+1], ..., a[r]? Both n and the number of queries can be up to 10⁵ or larger.

Sorting each subrange costs O((r−l+1) log(r−l+1)) per query — far too slow. We need an O(log n) per query data structure that captures the distribution of values inside any subrange.

Build: O(n log n) k-th query: O(log n) Memory: O(n log n) for n persistent versions Static array — no updates

The persistent segment tree pattern generalises beyond k-th smallest. It is the workhorse for "range count of values in [lo, hi]", "range distinct count", "range majority", "range sum of values ≤ x", and many others.

💡 Intuition from Scratch

Compress the values: replace each a[i] with its rank among all distinct values, so we are working with indices into a sorted vector vals of size m. Build, for every prefix length p ∈ {0, 1, ..., n}, a frequency segment tree cnt_p[v] = number of i < p with a[i] = v. Call its root root[p].

Two crucial properties.

(P1) Subtraction. The frequency of value v in a[l..r] is cnt_{r+1}[v] − cnt_l[v]. So node-by-node, the difference tree root[r+1] − root[l] is the frequency tree of a[l..r]. We never materialize it — we walk both roots in parallel.

(P2) Path copying. cnt_{p+1} differs from cnt_p in exactly one leaf, so we only need to allocate O(log n) new nodes for the new version: the path from the root to the changed leaf. Every other subtree is shared by pointer with cnt_p.

Total memory: O(n log n) instead of O(n · n). Each query is just a descent down two trees in parallel, O(log n).

🌳 Visual Mental Model

Imagine n + 1 root pointers root[0], root[1], ..., root[n]. root[0] points to a fully zero tree. root[p+1] is constructed by walking from root[p] down to the leaf for value a[p], allocating a fresh node at every step, but sharing the sibling subtree on the other side at each step.

root[0] → fully zero tree root[1] = update(root[0], a[0]) root[2] = update(root[1], a[1]) ... root[n] = update(root[n-1], a[n-1])

So the persistent forest is a DAG: at most O(n log n) nodes total, organized as n+1 rooted "views" that share most of their internal structure.

When we query k-th in [l, r], we descend root[l] and root[r+1] simultaneously, starting at value-range [0, m−1]. At each internal node, the count in the left subtree of the difference tree is L.cnt(r+1) − L.cnt(l). If this is at least k, the answer lives in the left half. Otherwise it lives in the right half and we subtract that count from k.

📐 Formal Abstraction

The persistent count tree is built over the value range [0, m−1], not over array positions. Each node stores:

struct Node { int left; // pointer (index in pool) to left child, -1 if absent int right; // pointer to right child int cnt; // total count of values in this node's interval };

Nodes are allocated from a vector<Node> pool; child pointers are integer indices into the pool, never raw pointers. This is the standard, cache-friendly way to do persistence in competitive programming.

Immutability is the entire game: once a node is published in some root, its three fields never change. Every "update" allocates new nodes along the path; the old roots remain valid forever.

🔬 Key Observations

Observation 1. The frequency tree is additive over array positions: insert one occurrence of value v ⇒ increment exactly one leaf and increment cnt on the path of length O(log m) from root to that leaf.

Observation 2. "Path copying" turns the additive structure into a persistent one. Each new version's path of O(log m) nodes is allocated fresh; the off-path subtrees (one per level) are shared.

Observation 3. To query a[l..r] we descend two roots in lockstep and use cnt(rightSubtree of root[r+1]) − cnt(rightSubtree of root[l]) as the working count. Whichever side contains the k-th element receives the recursion.

Observation 4. The trees are perfectly balanced over [0, m−1], so every descent is exactly ⌈log₂ m⌉ deep.

⚙️ Algorithm

Setup

Compress the input. Let m be the number of distinct values, and rank[v] the index of v in sorted distinct order. Set base = smallest power of 2 ≥ m. Build root[0] as a tree of zeros.

update(prev_root, position_in_array, value_rank)

Recurse over the value range [lo, hi] starting at [0, base−1]. At each step, if lo == hi, allocate a new leaf with cnt = pool[prev_root].cnt + 1. Otherwise, compute mid, decide which child holds value_rank, recurse into that child (passing the old child of prev_root), and reuse the other child of prev_root unchanged. Allocate a new internal node whose two children are: (1) the new recursive result, (2) the reused sibling from prev_root. Its cnt is the sum of the two new children's cnt — equivalently prev cnt + 1.

kth(root_l, root_rp1, k)

Descend both nodes simultaneously starting from value range [0, base−1]. At each step compute left_cnt_diff = pool[L_left].cnt − pool[L_left_old].cnt. If left_cnt_diff ≥ k, recurse left. Otherwise k -= left_cnt_diff and recurse right. When lo == hi, return vals[lo].

🛡️ Invariants

✅ Invariant — Immutability invariant

Once a node has been published (i.e., its index is reachable from any root[p]), its fields are never modified. Update only allocates new nodes along the path; everything off-path is shared by index.

✅ Invariant — Count invariant

For every node u with interval [lo, hi] in version p, pool[u].cnt equals the number of array positions i < p with rank(a[i]) ∈ [lo, hi]. Therefore cnt_p(node) − cnt_l(node) counts how many elements of a[l..p−1] have rank in [lo, hi].

✅ Invariant — Path-length invariant

Every update allocates exactly ceil(log₂(base)) + 1 new nodes — one per level from root to leaf. No more, no fewer.

✅ Invariant — K-th descent invariant

At every step of the k-th descent, the variable k equals the rank (in 1-based order) of the answer within the current value sub-range. Going right subtracts the size of the left side from k; going left leaves k unchanged.

🧮 Proof of Correctness

🧮 Proof — Why the k-th descent always lands on the correct value

Setup. Let F = freq_{[l, r]} denote the implicit frequency tree of a[l..r], which we never construct but is well-defined per node as cnt_{r+1}(node) − cnt_l(node). The Count invariant guarantees this is non-negative everywhere and totals exactly r − l + 1 at the root.

Induction. We prove that after descending to a node with interval [lo, hi], the answer's rank inside [lo, hi] is exactly the current k. Base: at the root, [lo, hi] = [0, base − 1] spans all values, so the answer's rank is k by definition. Step: let L = F.left.cnt be the number of subrange elements with rank in the left half. If L ≥ k, the answer lies in the left half and its 1-based rank within the left half is still k (since the left half is a prefix). If L < k, the answer lies in the right half; its rank within the right half is k − L. Either way the inductive invariant is preserved.

Termination. The descent halves the value range every step and terminates at a leaf in ⌈log₂(base)⌉ steps. At a leaf with lo = hi, the rank-k element of a constant-value list is that value, so we return vals[lo]. ∎

When sizing the node pool, reserve (n + 1) · (ceil_log2(base) + 2) nodes — a small constant times n log n. Underestimating triggers vector reallocation that can move all node indices; you don't want that during contest time.

🧰 Core Template Philosophy

The persistent template is the smallest piece of code that captures the "never mutate, allocate on the path, share the rest" pattern. Three rules govern everything:

(1) The node pool is an append-only vector<Node>. Children are int indices, never pointers. Reserve up front to avoid relocation.

(2) Every recursion that produces a new version returns an index into the pool — the root of the freshly built subtree. The caller installs that index as one of its own children.

(3) For aggregations beyond cnt (sum, weighted sum, etc.), add the field to the struct and recompute it in the same place where cnt is updated. The path-copying machinery doesn't change.

💻 Master Template (C++)

#include <bits/stdc++.h>
using namespace std;

struct PersistentSegTree {
    struct Node {
        int left = -1;
        int right = -1;
        int cnt = 0;
    };

    int base = 1;
    vector<Node> pool;
    vector<int> root;
    vector<int> vals;

    void build(const vector<int>& a) {
        vals = a;
        sort(vals.begin(), vals.end());
        vals.erase(unique(vals.begin(), vals.end()), vals.end());

        int m = (int)vals.size();
        base = 1;
        while (base < max(1, m)) {
            base <<= 1;
        }

        int n = (int)a.size();
        int per_path = 0;
        for (int b = base; b > 0; b >>= 1) per_path++;
        pool.reserve((long long)(n + 2) * (per_path + 1));

        // root[0] = empty tree
        root.assign(n + 1, -1);
        root[0] = build_empty(0, base - 1);

        for (int i = 0; i < n; i++) {
            int r = (int)(lower_bound(vals.begin(), vals.end(), a[i])
                            - vals.begin());
            root[i + 1] = update(root[i], 0, base - 1, r);
        }
    }

    int build_empty(int lo, int hi) {
        int u = (int)pool.size();
        pool.push_back({});
        if (lo == hi) return u;
        int mid = lo + (hi - lo) / 2;
        int L = build_empty(lo, mid);
        int R = build_empty(mid + 1, hi);
        pool[u].left = L;
        pool[u].right = R;
        return u;
    }

    int update(int prev, int lo, int hi, int pos) {
        int u = (int)pool.size();
        pool.push_back(pool[prev]);

        if (lo == hi) {
            pool[u].cnt += 1;
            return u;
        }

        int mid = lo + (hi - lo) / 2;
        if (pos <= mid) {
            pool[u].left = update(pool[prev].left, lo, mid, pos);
        } else {
            pool[u].right = update(pool[prev].right, mid + 1, hi, pos);
        }

        pool[u].cnt = pool[pool[u].left].cnt + pool[pool[u].right].cnt;
        return u;
    }

    int kth(int l, int r, int k) const {
        // Returns the actual value (decompressed)
        return descend(root[l], root[r + 1], 0, base - 1, k);
    }

    int descend(int uL, int uR, int lo, int hi, int k) const {
        if (lo == hi) return vals[lo];

        int leftDiff = pool[pool[uR].left].cnt - pool[pool[uL].left].cnt;
        int mid = lo + (hi - lo) / 2;

        if (leftDiff >= k) {
            return descend(pool[uL].left, pool[uR].left, lo, mid, k);
        } else {
            return descend(pool[uL].right, pool[uR].right,
                           mid + 1, hi, k - leftDiff);
        }
    }
};

🔎 Line-by-Line Explanation

build(a) — coordinate compression and forest construction click to expand/collapse

We sort and deduplicate a into vals, so each distinct value gets a rank in [0, m − 1]. The base of the segment tree is the smallest power of 2 ≥ m. We reserve the pool generously to prevent reallocation, then build the empty version root[0] recursively. The main loop walks the input left-to-right, computing the rank of each a[i] by binary search and producing the next version via update.

build_empty(lo, hi) — initial zero tree click to expand/collapse

A trivial recursion that allocates 2·base − 1 nodes representing the zero-count tree over [0, base − 1]. We do this only once. All future versions branch off from this skeleton.

update(prev, lo, hi, pos) — the heart of path copying click to expand/collapse

We allocate a new node u as a copy of prev (so its left and right children initially point to the same children as prev). If lo == hi, this is a leaf — increment cnt by one and return u. Otherwise decide which side pos belongs to. Only that side recurses, producing a new index; the other side keeps its inherited shared pointer. Finally, u.cnt is recomputed as the sum of its (possibly new) children's counts.

kth(l, r, k) and descend(uL, uR, lo, hi, k) click to expand/collapse

We pass l and r + 1 into the helper so that cnt_{r+1} − cnt_l is exactly the frequency over a[l..r]. At every internal node, the left-child count difference tells us how many subrange elements have ranks in the left half. The comparison leftDiff ≥ k drives the descent direction. When we reach a leaf at lo == hi, we decompress and return vals[lo].

⚠️ Common Pitfalls

⚠️ Pitfall — Mutating shared nodes by accident

The fastest way to break persistence is to assign into a node that some other root still points to. Always allocate a copy first (pool.push_back(pool[prev]); int u = ...) and modify only that copy. Never write to pool[prev] in update.

⚠️ Pitfall — Using prefix length p instead of array index l

The subtraction is root[r+1] − root[l], not root[r] − root[l] or root[r+1] − root[l−1]. The off-by-one comes from the prefix convention: root[p] counts the first p elements a[0..p−1].

⚠️ Pitfall — Forgetting to reserve the pool

Reallocation while pool grows is correct but slow and disastrous if you keep raw references. Always pool.reserve(...) up front; computing the right bound is a small exercise but worth the discipline.

⚠️ Pitfall — Sorting the wrong vector

The decompression vector vals is the sorted-deduplicated original. Don't accidentally sort a in place — you need to remember the original order so you can insert a[0], a[1], ... in sequence.

⚠️ Pitfall — Mixing 1-based and 0-based k

The descent expects k in 1-based (the smallest is k=1). If the problem gives 0-based k, add 1 before calling kth.

🎛️ Variants & Extensions

Variant 1 — Range count of values in [lo_val, hi_val]

Instead of a descent, range-query root[l] and root[r+1] for the count in [rank(lo_val), rank(hi_val)] and subtract. O(log m) per query.

Variant 2 — Range distinct count (one common form)

Set the persistent tree's leaf i = "1 if a[i] is the last occurrence of its value in the prefix, else 0", maintained as you sweep. The range distinct count is sum over [l, r] in root[r+1].

Variant 3 — Range sum of the smallest k values

Store sum alongside cnt at each node. During the k-th descent, accumulate the entire sum of the left half when going right, and continue with k − leftDiff on the right.

Variant 4 — Persistent set of integers (online)

Replace "prefix of array" with "current version of a multiset". Each insertion produces a new root. Useful for offline algorithms that branch on a value or for online algorithms that must answer queries about past states.

🏁 Final Summary

A persistent segment tree is a segment tree whose every "update" preserves the previous version by copying only the affected path. With O(log n) new nodes per version, the entire forest of n + 1 prefix versions costs O(n log n) memory.

The k-th smallest in a subarray problem reduces to a descent over root[l] and root[r + 1] in parallel, using the count difference of the left subtree at each node to choose direction. The descent is O(log n), fast enough for the largest contest inputs.

Two disciplines from this lesson carry to every other persistent variant: never mutate published nodes, and only the path is allocated fresh. Lesson 11 (persistent lazy) and Lesson 13 (persistent dynamic) both lean on exactly these two rules.
