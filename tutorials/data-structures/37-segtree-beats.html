Lesson 8 — Segment Tree Beats

Range chmin + range add + range sum · O((n+q) log² n) amortized · The Ji Driver Tree

🧩 Problem Definition

Given an integer array a[0..n−1], the master template in this lesson supports three kinds of operations on subranges in O(log² n) amortized time:

1. chmin(l, r, x)

For every i in [l, r], set a[i] = min(a[i], x).

2. add(l, r, x)

For every i in [l, r], set a[i] += x.

3. sum(l, r)

Return a[l] + ... + a[r].

The symmetric chmax(l, r, x) — set a[i] = max(a[i], x) over the range — is a natural mirror of chmin and is discussed in the Variants section, but the shipped master template here implements range chmin, range add, and range sum only. Adding chmax requires symmetric mn1, mn2, mnc fields and a second three-case rule. Each of these operations alone is solvable by Lesson-4-style lazy propagation. The hard part is combining them — mixing chmin with add in particular, since the lazy tag for chmin cannot simply be “the minimum value to take” once additions are interleaved. Segment Tree Beats is the standard solution and the gateway to many advanced lazy variants.

💡 Intuition from Scratch

Consider a single segment-tree node covering an interval. If we knew the exact multiset of values inside, then chmin(x) would split that multiset into “values ≤ x stay” and “values > x become x”. The sum changes by an amount that depends only on the values strictly greater than x.

We cannot afford to store the whole multiset at every node. But we do not need to. We only need to store enough so that, for any incoming chmin(x), we can either apply it in O(1) at this node or recurse into children. The smallest piece of summary information that lets us make this decision is the maximum, the strict second-maximum, and the count of elements equal to the maximum.

With those three quantities — call them mx1, mx2, mxc — the chmin operation has three crystal-clear cases. If x ≥ mx1, nothing happens. If mx2 < x < mx1, exactly the mxc maximum-value elements drop to x, so the sum decreases by mxc · (mx1 − x) and the new mx1 becomes x. If x ≤ mx2, our four fields are not enough to summarize the effect and we have to recurse into both children — the famous “break” case. The amortized analysis shows that “cannot summarize” does not happen often.

🌳 Visual Mental Model

Every node stores four numbers plus a lazy add tag:

struct Node { long long sum; // sum of all elements in the interval long long mx1; // strict maximum value in the interval long long mx2; // strict second maximum (NEG_INF if all equal) int mxc; // number of elements equal to mx1 long long add_lazy;// pending add for children };

The chmin operation is summarized as an implicit “cap”: when we apply it at a node where mx2 < x < mx1, we set sum -= mxc · (mx1 − x) and mx1 = x. The chmin is recorded implicitly because the new mx1 is the cap. No separate chmin_lazy field is needed — but on push, we propagate the cap to each child by calling that child's apply_chmin with the parent's current mx1.

📐 Formal Abstraction

The four-field state is closed under all three update operations provided the node's interval is “homogeneous enough” relative to the operation. Formally:

apply_add(node, x). Add x to sum (scaled by length), to mx1, and to mx2 if mx2 is finite; accumulate into add_lazy.

apply_chmin(node, x). Three cases. If x ≥ mx1: no effect. If mx2 < x < mx1: update sum -= mxc · (mx1 − x), mx1 = x; this is “stop here, tag is implicit”. If x ≤ mx2: this case is not handled at this node; the caller must recurse.

push(node). For each child, apply the parent's add_lazy via apply_add, then apply the chmin cap via apply_chmin(child, mx1[parent]). The chmin cap is propagated using each child's own mx1, mx2, mxc, so the three-case rule applies again at the child.

🔬 Key Observations

Observation 1 (the three cases). A chmin(x) on a node with state (mx1, mx2, mxc) has three possible relationships to x:

CaseConditionWhat happensAction
1x ≥ mx1Nothing — every value is already ≤ xReturn immediately (“skip”)
2mx2 < x < mx1Exactly mxc elements (the maxes) drop to xApply in O(1) and stop
3x ≤ mx2Multiple distinct values are affected — cannot summarizePush and recurse into both children (“break”)

Observation 2 (cap propagation). When a parent has been “capped” at value v (so its mx1 = v after a Case-2 application), a future push to children must cap each child at v. This is done by calling apply_chmin(child, v), which itself respects the three-case rule.

Observation 3 (amortized complexity). A “break” descent costs O(log n) per visited node. The total number of break descents across all operations is bounded by O((n + q) log n) via a potential argument that counts distinct values in the tree. Total cost: O((n + q) log² n).

⚙️ Algorithm

build / pull

At a leaf: sum = mx1 = a[l], mx2 = NEG_INF, mxc = 1. At an internal node, pull merges the children with a small case analysis:

sum = left.sum + right.sum if left.mx1 == right.mx1: mx1 = left.mx1 mx2 = max(left.mx2, right.mx2) mxc = left.mxc + right.mxc elif left.mx1 > right.mx1: mx1 = left.mx1 mx2 = max(left.mx2, right.mx1) mxc = left.mxc else: mx1 = right.mx1 mx2 = max(left.mx1, right.mx2) mxc = right.mxc

apply_add(node, x)

sum += x · length, mx1 += x, mx2 += x (only if mx2 ≠ NEG_INF), add_lazy += x.

apply_chmin(node, x)

If x ≥ mx1: return. Else (the caller guarantees mx2 < x): sum -= mxc · (mx1 − x), mx1 = x.

range_chmin(ql, qr, x)

Standard recursion. Return on no-overlap or when x ≥ mx1[i]. Apply in O(1) when fully covered and x > mx2[i]. Otherwise push, recurse into both children, then pull — this is the break case.

🛡️ Invariants

✅ Invariant — Field invariants (true outside the middle of an operation)

For every node u with interval [l, r]:
(a) sum[u] is the total of all current values in [l, r].
(b) mx1[u] is the largest current value in [l, r].
(c) mx2[u] is the largest value strictly less than mx1[u], or NEG_INF if all values are equal.
(d) mxc[u] is the count of values equal to mx1[u].

✅ Invariant — Add-lazy invariant

add_lazy[u] is a uniform pending addition already applied to (sum, mx1, mx2) of u but not yet to its children. On push, each child receives apply_add(child, add_lazy[u]), then add_lazy[u] is cleared.

✅ Invariant — Implicit chmin tag

There is no separate chmin_lazy field. The implicit tag is mx1[u]: any child whose mx1[child] exceeds mx1[u] after the add propagation must be capped. On push, call apply_chmin(child, mx1[u]) for each child.

✅ Invariant — Push order

On push, propagate apply_add before apply_chmin on each child — this matches the conceptual order in which the operations were applied to the parent. Add first to bring the child's mx1 up, then cap with chmin.

🧮 Proof of Correctness

🧮 Proof — Correctness of the three-case rule

Consider a node covering [l, r] with state (sum, mx1, mx2, mxc), and an incoming chmin(x).

Case 1 (x ≥ mx1). Every value in [l, r] is at most mx1 ≤ x, so the operation has no effect. Returning immediately preserves all four field invariants.

Case 2 (mx2 < x < mx1). Every value equal to mx1 drops to x; every value at most mx2 < x is unchanged. So mxc values change, each by mx1 − x. The new sum is sum − mxc · (mx1 − x); the new mx1 is x (greater than the unchanged mx2, equal to the dropped values); mx2 and mxc are unchanged. These match the apply rule.

Case 3 (x ≤ mx2). Multiple distinct values are affected and the four fields are insufficient to summarize. We break by pushing and recursing into both children; at each child, the three-case rule applies again. Termination is guaranteed because each recursion strictly reduces the size of the interval, and at a leaf the case is always 1 or 2. ∎

🧮 Proof — Amortized complexity (sketch)

Define the potential Φ = Σ_u (number of distinct values in u's interval). Each Case-2 application at a node decreases the number of distinct values in that node's interval by exactly one (the old mx1 collapses onto the new mx1 = x). Each Case-3 break visits both children and triggers their three-case rule; if a child hits Case 2, the potential drops by one there. The initial potential is O(n log n) (each element appears in log n nodes), so the total number of Case-2 events across all operations is O((n + q) log n). Combined with the O(log n) overhead per operation, the total cost is O((n + q) log² n). (The full proof uses a refined potential to handle chmax and add interleaving, but the conclusion is the same.) ∎

🧰 Core Template Philosophy

Beats is the same lazy-propagation skeleton as Lesson 4 with two key differences. First, the per-node state grows to four fields (or five, for chmax). Second, the chmin operation implements the three-case rule explicitly: skip on Case 1, apply-and-stop on Case 2, break-and-recurse on Case 3. The break case is the surprising one — a lazy tree that sometimes refuses to stop at a fully-covered node.

The mental discipline that makes beats work: at every node, ask “can my current summary capture this operation's effect?” If yes, apply it. If no, push down and recurse. The three-case classifier is the canonical answer to that question for chmin.

💻 Master Template (C++)

#include <bits/stdc++.h>
using namespace std;

struct SegTreeBeats {
    using T = long long;
    static constexpr T NEG_INF = LLONG_MIN / 4;

    int n = 0;
    int base = 1;

    vector<T> sum, mx1, mx2;
    vector<int> mxc;
    vector<T> add_lazy;

    SegTreeBeats() {}

    SegTreeBeats(const vector<T>& a) { init(a); }

    void init(const vector<T>& a) {
        n = (int)a.size();
        base = 1;
        while (base < max(1, n)) base <<= 1;
        sum.assign(2*base, 0);
        mx1.assign(2*base, NEG_INF);
        mx2.assign(2*base, NEG_INF);
        mxc.assign(2*base, 0);
        add_lazy.assign(2*base, 0);
        build(0, 0, base-1, a);
    }

    void build(int i, int l, int r, const vector<T>& a) {
        if (l == r) {
            if (l < n) {
                sum[i] = mx1[i] = a[l];
                mx2[i] = NEG_INF;
                mxc[i] = 1;
            }
            return;
        }
        int left = 2*i+1, right = left+1;
        int mid = l + (r-l)/2;
        build(left, l, mid, a);
        build(right, mid+1, r, a);
        pull(i);
    }

    void pull(int i) {
        int left = 2*i+1, right = left+1;
        sum[i] = sum[left] + sum[right];
        if (mx1[left] == mx1[right]) {
            mx1[i] = mx1[left];
            mx2[i] = max(mx2[left], mx2[right]);
            mxc[i] = mxc[left] + mxc[right];
        } else if (mx1[left] > mx1[right]) {
            mx1[i] = mx1[left];
            mx2[i] = max(mx2[left], mx1[right]);
            mxc[i] = mxc[left];
        } else {
            mx1[i] = mx1[right];
            mx2[i] = max(mx1[left], mx2[right]);
            mxc[i] = mxc[right];
        }
    }

    void apply_add(int i, int l, int r, T x) {
        sum[i] += x * (r - l + 1);
        mx1[i] += x;
        if (mx2[i] != NEG_INF) mx2[i] += x;
        add_lazy[i] += x;
    }

    void apply_chmin(int i, T x) {
        if (x >= mx1[i]) return;
        sum[i] -= (T)mxc[i] * (mx1[i] - x);
        mx1[i] = x;
    }

    void push(int i, int l, int r) {
        if (l == r) { add_lazy[i] = 0; return; }
        int left = 2*i+1, right = left+1;
        int mid = l + (r-l)/2;

        if (add_lazy[i] != 0) {
            apply_add(left, l, mid, add_lazy[i]);
            apply_add(right, mid+1, r, add_lazy[i]);
            add_lazy[i] = 0;
        }

        apply_chmin(left, mx1[i]);
        apply_chmin(right, mx1[i]);
    }

    void range_chmin(int ql, int qr, T x) {
        range_chmin(ql, qr, x, 0, 0, base-1);
    }

    void range_chmin(int ql, int qr, T x, int i, int l, int r) {
        if (qr < l || r < ql || x >= mx1[i]) return;
        if (ql <= l && r <= qr && x > mx2[i]) {
            apply_chmin(i, x);
            return;
        }
        push(i, l, r);
        int left = 2*i+1, right = left+1;
        int mid = l + (r-l)/2;
        range_chmin(ql, qr, x, left, l, mid);
        range_chmin(ql, qr, x, right, mid+1, r);
        pull(i);
    }

    void range_add(int ql, int qr, T x) {
        range_add(ql, qr, x, 0, 0, base-1);
    }

    void range_add(int ql, int qr, T x, int i, int l, int r) {
        if (qr < l || r < ql) return;
        if (ql <= l && r <= qr) { apply_add(i, l, r, x); return; }
        push(i, l, r);
        int left = 2*i+1, right = left+1;
        int mid = l + (r-l)/2;
        range_add(ql, qr, x, left, l, mid);
        range_add(ql, qr, x, right, mid+1, r);
        pull(i);
    }

    T range_sum(int ql, int qr) {
        return range_sum(ql, qr, 0, 0, base-1);
    }

    T range_sum(int ql, int qr, int i, int l, int r) {
        if (qr < l || r < ql) return 0;
        if (ql <= l && r <= qr) return sum[i];
        push(i, l, r);
        int left = 2*i+1, right = left+1;
        int mid = l + (r-l)/2;
        return range_sum(ql, qr, left, l, mid)
             + range_sum(ql, qr, right, mid+1, r);
    }
};

🔎 Line-by-Line Explanation

pull(i) — three-case merge of children click to expand/collapse

This is where the four-field state composes from the children. The three cases depend on which child has the larger mx1. If they tie, the union has the same mx1 with combined count mxc, and mx2 is the larger of the two children's mx2. If one child's mx1 is strictly larger, the union takes that child's mx1 and mxc; mx2 becomes the larger of that child's mx2 and the other child's mx1 (which is now the second-largest).

apply_chmin(i, x) — Case-2 application click to expand/collapse

The function implements Case 2. The Case 1 short-circuit (x ≥ mx1) is here as defense-in-depth. The caller (range_chmin) is responsible for guaranteeing x > mx2 before calling — that is exactly what its full-coverage condition x > mx2[i] ensures.

push(i, l, r) — propagate add then chmin click to expand/collapse

Order matters. First apply the pending add to each child, which moves their mx1 values; then apply chmin(mx1[i]) to each child as the implicit cap. The child's own apply_chmin resolves the three cases internally — if the cap is already loose, it does nothing; if tight, it applies in O(1).

range_chmin(ql, qr, x, i, l, r) — the break-stop-tag descent click to expand/collapse

Three short-circuits at the top: no overlap, full Case 1 (x ≥ mx1[i]), and full Case 2 (covered range and x > mx2[i]). If none apply, we are in the break case: push, recurse into both children, pull. The order is identical to Lesson 4's lazy template.

⚠️ Common Pitfalls

⚠️ Pitfall — Forgetting to guard mx2 add against NEG_INF

If mx2 = NEG_INF (all values equal), naively doing mx2 += x in apply_add will give a meaningless value. Always check if (mx2 != NEG_INF) mx2 += x;.

⚠️ Pitfall — Using non-strict inequality in the Case-2 guard

The full-coverage Case-2 condition is x > mx2[i], with strict greater-than. Using x >= mx2[i] can apply Case 2 when there are actually two distinct largest values equal to x, breaking the count mxc.

⚠️ Pitfall — Wrong order in push (chmin before add)

The conceptual order is “add was applied first, then chmin cap”. So push must apply add first, then chmin. Reversing the order gives sums that don't match.

⚠️ Pitfall — Missing pull after recurse

The break case must end with pull(i) to refresh the four fields from the children after they have changed. Forgetting this leaves the parent with stale mx1, mx2, mxc, sum.

⚠️ Pitfall — Treating beats as O(log n) per operation

Beats is amortized O(log² n) per chmin (and per chmax in the symmetric variant), not O(log n). On adversarial inputs (e.g. repeatedly making all values distinct, then collapsing) the worst-case sequence is genuinely O((n + q) log² n) — not per-operation O(log n).

🎛️ Variants & Extensions

Variant 1 — chmax / min added

Mirror image: add fields mn1, mn2, mnc. apply_chmax(node, x) with the symmetric three-case rule. Push must propagate both implicit chmin cap and chmax floor.

Variant 2 — Range max query

Trivial: just read mx1[i] at full-coverage nodes. O(log n) per query.

Variant 3 — Historic maximum

Add fields hmx1, hmx2, hmxc tracking the maximum value ever held by any position in the interval. Updates copy hmx from mx with care for pending lazy tags. Used for “maximum prefix sum ever” and similar.

Variant 4 — Beats over 2D points

Combined with KD-tree-like structures for some specialised problems. Rare in contests; mostly research-grade.

🏁 Final Summary

Segment Tree Beats extends lazy propagation to operations that cannot be summarized by a single tag — most notably chmin, chmax, and their combinations with range add and range sum. The trick is to enrich each node's summary with the strict second-maximum and the count of maxes, which together let us decide for each incoming chmin whether the operation can be applied in O(1) at this node or whether we must break and recurse into the children.

The three-case rule (skip / apply-stop / break-recurse) is mechanical once internalised; the amortized analysis based on a potential counting distinct values proves the total cost is O((n + q) log² n). Beats is the gateway to a family of advanced lazy structures including historic-max trees and assign-and-add lazy beats.
