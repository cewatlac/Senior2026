Monotonic Stack & Deque

Stack & Deque With Order Discipline — Complete Tutorial
Theme

What You'll Learn

A monotonic stack or monotonic deque is not a new magical data structure. It is an ordinary stack or deque used with one extra rule: keep the stored candidates in increasing or decreasing order by removing candidates that are no longer useful. From that single discipline emerge linear-time algorithms for an entire family of problems where naive scans are quadratic.

Core Promise: Each element is pushed at most once and popped at most once. The "nested while" inside a "for" is amortized linear. You will leave this tutorial able to recognize a monotonic problem at sight, pick the correct strictness, and write the template from memory.

1. Problem Definition

Monotonic structures are mainly used when the problem reduces to one of these patterns: nearest greater element, nearest smaller element, previous greater/smaller, next greater/smaller, first warmer day, first cheaper or equal price, the boundary of a rectangle, visibility until a taller blocker, contribution of an element as minimum or maximum of subarrays, sliding-window maximum or minimum, DP transition over a moving range, or prefix-sum range optimization.

The reason they are powerful is that many brute-force solutions look like this:

for each i:
    scan left or right until you find the first useful element

That is often O(n²). A monotonic stack/deque keeps only the candidates that may still matter, so each element is inserted once and removed at most once, giving O(n).

The two questions every monotonic problem asks

Question A — Direction & relation

Are we looking left or right? Are we looking for something greater or smaller? These four combinations cover almost all stack problems.

Question B — Strictness

Does equal count? "Strictly greater" vs "greater or equal" decides whether the comparison uses < or <= in the pop condition. Subtle, but it changes correctness.

TermEquality allowed?Example
strictly increasingNo[1,2,3]
non-decreasingYes[1,2,2,3]
strictly decreasingNo[5,4,3]
non-increasingYes[5,4,4,2]
⚠

Vocabulary discipline: never say "strictly increasing including equal" — that is contradictory. Pick one of the four exact phrases above and stay consistent throughout the problem.

2. Intuition from Scratch

Ordinary stack vs monotonic stack

An ordinary stack is LIFO — you push and pop without any rule about the values. A monotonic stack is still a stack, but before pushing a new element, we pop elements that break the desired order. That is the entire definition.

Suppose we want an increasing stack and the input is [2,1,6,3,5,4,2]. We process left to right:

push 2        stack = [2]
1 pops 2      stack = [1]
push 6        stack = [1, 6]
3 pops 6      stack = [1, 3]
push 5        stack = [1, 3, 5]
4 pops 5      stack = [1, 3, 4]
2 pops 4,3    stack = [1, 2]

The final stack [1,2] is not the answer by itself. The important part is what happened during popping.

💡

Core teaching phrase: what pops you is your next answer on the right; what remains below you is your previous answer on the left.

Why popping is safe — the deep intuition

Suppose we are solving Next Greater Element: for every index i, we want the first index j > i with a[j] > a[i]. Brute force scans inwards from each i and is O(n²).

Now imagine scanning left to right and keeping unresolved elements on a stack. With [5, 3, 2, 9, 4]:

5 waits for a greater element
3 waits for a greater element
2 waits for a greater element
Stack: [5, 3, 2]

Now we see 9.
9 > 2  → 9 is the next greater for 2.
9 > 3  → 9 is the next greater for 3.
9 > 5  → 9 is the next greater for 5.
After this, 9 itself waits for a future greater element.

This is the magic moment: one current value can resolve many previous unresolved values in a single sweep. The total amount of work is bounded by the number of pushes plus the number of pops, both of which are at most n. That is the entire reason monotonic structures are linear.

3. Visual Mental Model

Think of the stack as a line of candidates waiting for help. Each candidate stays only as long as it is still potentially useful. The moment a new element proves the candidate cannot be the answer for anyone again, the candidate is destroyed.

Decreasing-stack snapshot for Next Greater Element
candidates still alive popped (resolved by current) current element

What just happened in that snapshot:

stack before:  [10, 7, 3]    current = 8

8 pops 3   →  3's next greater is 8
8 pops 7   →  7's next greater is 8
10 stays   →  10 is the previous greater of 8
push 8

stack after:   [10, 8]

The popped elements got their answer from the right (the current element). The element that survived gave the new arrival its answer from the left. This is exactly the dual nature of the technique: the same single sweep simultaneously answers next-side and previous-side queries.

4. Formal Abstraction

The four canonical relations

Almost every stack problem reduces to one of these four nearest-element queries. The pop condition is dictated by which one you want and whether equality counts. Memorize the structure, not the table.

Wanted answerPop condition (strict)Pop condition (or-equal)Meaning of pop
Next Greatera[top] < a[i]a[top] <= a[i]current is greater → resolves popped
Next Smallera[top] > a[i]a[top] >= a[i]current is smaller → resolves popped
Previous Greatera[top] <= a[i]a[top] < a[i]top is invalid for current
Previous Smallera[top] >= a[i]a[top] > a[i]top is invalid for current
📐

Why the asymmetry between Next-side and Previous-side rules? Next-side scans resolve popped indices, so you pop while the popped element has not yet found its answer. Previous-side scans read the top after cleanup, so you pop while the top fails to be a valid answer for the current. The two phrasings produce different comparison directions for the same problem.

Two equivalent scan styles

The same problem can be implemented two ways. Both are correct; pick the one whose mental model matches the question.

Style A — Query style (read from stack)

For previous greater: scan left → right, pop until top is strictly greater than current, then read the top.

for (int i=0;i<n;i++){
    while(!st.empty() && a[st.back()]<=a[i])
        st.pop_back();
    ans[i] = st.empty()?-1:st.back();
    st.push_back(i);
}

Style B — Resolution style (current pops)

For next greater: scan left → right, current pops smaller elements and writes their answer.

for (int i=0;i<n;i++){
    while(!st.empty() && a[st.back()]<a[i]){
        ans[st.back()] = a[i];
        st.pop_back();
    }
    st.push_back(i);
}

Stack stores indices, not values

By default, store indices. From an index you can recover the value, the position, the distance, the width, and you avoid duplicate-key collisions when values repeat. Store raw values only when (a) values are guaranteed unique and (b) the output is values, not positions — for example, the value-to-value map in LeetCode 496.

5. Key Observations — Two Ways an Answer Appears

5.1 Current resolves popped elements

This is the natural style for next-side queries. The current element i is the first valid answer for everything it pops:

while (!st.empty() && a[st.back()] < a[i]) {
    ans[st.back()] = a[i];   // i is the next greater for st.back()
    st.pop_back();
}
st.push_back(i);

5.2 Current reads its own answer from the remaining top

This is the natural style for previous-side queries. After cleaning up invalid candidates, the surviving top is the answer for i:

while (!st.empty() && a[st.back()] <= a[i]) {
    st.pop_back();
}
ans[i] = st.empty() ? -1 : a[st.back()];   // nearest previous greater
st.push_back(i);

5.3 The histogram pattern: pop yields BOTH boundaries at once

For rectangle and contribution problems we exploit a beautiful trick: when the current bar pops a previous bar, the popped bar's right boundary is the current index, and the popped bar's left boundary is whatever is still on the stack underneath it. Two answers, one pop.

while (!st.empty() && h[st.back()] > h[i]) {
    int mid   = st.back(); st.pop_back();
    int left  = st.empty() ? -1 : st.back();   // previous smaller
    int right = i;                             // next smaller (current)
    long long width = right - left - 1;
    long long area  = (long long)h[mid] * width;
    best = max(best, area);
}

The same idea drives Trapping Rain Water, Largest Rectangle, and Sum of Subarray Minimums. Once you see "boundary on both sides as a popped element collapses", you have understood the deepest move in the entire technique.

6. Algorithm / Method Explanation

The master monotonic stack template

vector<int> st; // stores indices

for (int i = 0; i < n; i++) {
    while (!st.empty() && condition(a[st.back()], a[i])) {
        int idx = st.back();
        st.pop_back();
        // optionally process idx because current i resolved it
    }
    // optionally read st.back() as previous boundary/answer

    st.push_back(i);
}

The skeleton is identical across every stack problem in this tutorial. You only change three things: the comparison operator (strict vs or-equal), what you do to a popped index, and what you read from the new top after popping. That's it.

The master monotonic deque template

A deque is needed when elements can expire from the front — that is, become invalid simply because the window has moved past them. A stack cannot remove its bottom efficiently, so we upgrade to a deque.

deque<int> dq; // stores indices, values monotone

for (int i = 0; i < n; i++) {
    // (1) expire indices that fell out of the window
    while (!dq.empty() && dq.front() <= i - k)
        dq.pop_front();

    // (2) pop dominated candidates from the back
    while (!dq.empty() && a[dq.back()] <= a[i])
        dq.pop_back();

    dq.push_back(i);

    // (3) once the window is full, the front is the answer
    if (i >= k - 1)
        ans.push_back(a[dq.front()]);
}

The three deque operations correspond to three logical jobs: expire stale indices, kill dominated candidates, and read the answer from the front. Every sliding-window monotonic problem is some recombination of these three jobs.

7. Invariants

An invariant is a property that holds at every iteration boundary of the algorithm. For monotonic structures, three invariants completely characterize correctness.

7.1 State invariant — values are monotone

For Next Greater (Style B), after each iteration the stack stores indices whose values are non-increasing from bottom to top:

a[st[0]] ≥ a[st[1]] ≥ … ≥ a[st.back()]

If the relation is "or-equal", duplicates are allowed; if it is "strict", duplicates are forbidden. Whichever choice you make, the invariant must hold after every iteration — including after a sequence of pops, before the push.

7.2 Unresolved invariant — every stack element is still waiting

In the resolution style, every index currently on the stack has not yet found its next greater. The moment an index finds its answer, it is popped. This is what makes the work amortized linear: an index pays a constant amount of work when it enters and a constant amount when it leaves, and it leaves at most once.

7.3 Pop invariant — popping is always optimal

When index x is popped by current index i, the current i is the first greater element to the right of x. Not "some" greater element, not "an eventual" greater element — the very first. The proof is given in the next section.

7.4 Termination invariant — leftover elements have no answer

At the end of the scan, every index still in the stack has no next greater element to the right. The empty-stack check at the end is therefore correct: those indices' answers are -1 (or whatever sentinel you chose).

7.5 Deque invariants

For sliding window maximum the three invariants are: (1) every index in the deque is inside the current window, (2) the values are decreasing from front to back, and (3) the front is the maximum of the window. Each pop_front maintains (1), each pop_back maintains (2), and the conjunction implies (3).

8. Proof of Correctness — Next Greater Element

We prove that the resolution-style algorithm correctly assigns the next greater element to every index.

Theorem & proof click to collapse

Claim

Let the algorithm process the array a[0..n-1] left to right with a stack of unresolved indices and pop condition a[st.back()] < a[i]. Then for every index x: the value written to ans[x] equals the first index j > x with a[j] > a[x], or -1 if no such index exists.

Step 1 — When ans[x] is written, it is a valid greater element

Index x is popped exactly when some current index i satisfies a[i] > a[x] (the strict pop condition). So whatever value gets assigned to ans[x], it is greater than a[x] and lies to the right.

Step 2 — It is the FIRST such element

Suppose, for contradiction, that there exists an index k with x < k < i and a[k] > a[x]. Consider the moment the scan reached k. At that moment x was still on the stack (since it remained until i by assumption). The pop condition would have triggered: a[x] < a[k] means we pop x at iteration k, contradicting "x stayed until i". Therefore no such k exists, and i is the first greater element to the right of x. ∎

Step 3 — Indices that are never popped have no answer

If index x remains on the stack at the end, then for every j > x the pop condition a[x] < a[j] never triggered (otherwise x would have been popped). Hence no j > x satisfies a[j] > a[x], and ans[x] = -1 is correct.

Conclusion

Together, Steps 1–3 establish that ans[x] equals the first index with a strictly greater value to the right, for every x. The same argument, with reversed comparisons or scanning direction, proves the other three nearest-relation variants.

9. Complexity Analysis

Why nested while is still O(n)

The code looks like this:

for (int i = 0; i < n; i++) {
    while (!st.empty() && condition) {
        st.pop_back();
    }
    st.push_back(i);
}

At first glance the nested while inside a for looks like O(n²). It is not. The trick is to count operations on the stack rather than nesting depth:

total work ≤ n pushes + n pops = 2n = O(n)

Each index is pushed exactly once. Each index is popped at most once. Therefore the total number of inner-loop iterations across the entire run is at most n, regardless of how the iterations distribute across outer steps. The technique behind this argument is called amortized analysis — specifically, the aggregate method.

⚠

"Average case" vs "amortized": these are not the same. Average case requires a probability distribution over inputs. Amortized analysis is a worst-case statement about a sequence of operations. Monotonic stacks are worst-case amortized O(n), not "average O(n)".

Deque complexity

The same argument applies. Each index is push_back-ed once and removed once (either by pop_back during dominance pruning or by pop_front during expiry). Total work is bounded by 2n.

Space

Both structures use O(n) auxiliary space in the worst case (think strictly increasing input for a "next greater" stack, or a strictly increasing window for sliding-max deque).

10. Core Template Philosophy

One family, four nearest variants

Rather than memorizing four separate algorithms, treat them as four configurations of one template. The skeleton is shared; the differences are surgical. This is the same philosophy that runs through Coach Academy's Dijkstra and DSU tutorials: internalize the shape, not the special cases.

VariantScanStack monotonicityPop whileRead answer from
Previous Greaterleft → rightdecreasinga[top] <= a[i]remaining top
Previous Smallerleft → rightincreasinga[top] >= a[i]remaining top
Next Greaterright → leftdecreasinga[top] <= a[i]remaining top
Next Smallerright → leftincreasinga[top] >= a[i]remaining top

Notice the perfect symmetry: previous and next simply reverse the scan direction; greater and smaller simply flip the comparison. Once you see this, you stop memorizing and start deriving.

🧭

Naming conventions used everywhere in this tutorial: a[] for the input, st for the stack of indices, dq for the monotonic deque, ans[] for the output, and i for the current index. Keeping naming consistent is half of being able to write these from memory in a contest.

11. Master Implementations

11.1 Stack: four nearest-element templates

#include <bits/stdc++.h>
using namespace std;

// Returns indices; -1 if no answer.
vector<int> previousGreaterIndex(const vector<int>& a) {
    int n = (int)a.size();
    vector<int> ans(n, -1), st;
    for (int i = 0; i < n; i++) {
        while (!st.empty() && a[st.back()] <= a[i])
            st.pop_back();
        if (!st.empty()) ans[i] = st.back();
        st.push_back(i);
    }
    return ans;
}

vector<int> previousSmallerIndex(const vector<int>& a) {
    int n = (int)a.size();
    vector<int> ans(n, -1), st;
    for (int i = 0; i < n; i++) {
        while (!st.empty() && a[st.back()] >= a[i])
            st.pop_back();
        if (!st.empty()) ans[i] = st.back();
        st.push_back(i);
    }
    return ans;
}

vector<int> nextGreaterIndex(const vector<int>& a) {
    int n = (int)a.size();
    vector<int> ans(n, -1), st;
    for (int i = n - 1; i >= 0; i--) {
        while (!st.empty() && a[st.back()] <= a[i])
            st.pop_back();
        if (!st.empty()) ans[i] = st.back();
        st.push_back(i);
    }
    return ans;
}

vector<int> nextSmallerIndex(const vector<int>& a) {
    int n = (int)a.size();

            st.pop_back();

11.2 Deque: sliding window maximum and minimum

#include <bits/stdc++.h>
using namespace std;

vector<int> slidingWindowMaximum(const vector<int>& a, int k) {
    int n = (int)a.size();
    vector<int> ans;
    deque<int> dq; // indices, values strictly decreasing-or-equal-aware

    for (int i = 0; i < n; i++) {
        // (1) expire from the front
        while (!dq.empty() && dq.front() <= i - k)
            dq.pop_front();
        // (2) dominance prune from the back
        while (!dq.empty() && a[dq.back()] <= a[i])
            dq.pop_back();
        dq.push_back(i);
        // (3) emit answer once the window is full
        if (i >= k - 1)
            ans.push_back(a[dq.front()]);
    }
    return ans;
}

vector<int> slidingWindowMinimum(const vector<int>& a, int k) {
    int n = (int)a.size();
    vector<int> ans;
    deque<int> dq; // values increasing

    for (int i = 0; i < n; i++) {
        while (!dq.empty() && dq.front() <= i - k)
            dq.pop_front();
        while (!dq.empty() && a[dq.back()] >= a[i])
            dq.pop_back();
        dq.push_back(i);
        if (i >= k - 1)
            ans.push_back(a[dq.front()]);
    }
    return ans;
}

12. Line-by-Line Code Explanation

The stack template, decoded

vector<int> ans(n, -1), st;

Two vectors: the answer is initialized to -1 (the "no valid answer" sentinel), and st stores indices, never values. Storing indices gives access to value, position, distance, width, expiry, and avoids any aliasing trouble when values repeat.

while (!st.empty() && a[st.back()] <= a[i])
    st.pop_back();

This is the "kill invalid candidates" step. For previous greater, any value less than or equal to the current cannot be the previous greater for the current — and crucially, it cannot be the previous greater for any future element that the current dominates either. Removing it now costs nothing later.

if (!st.empty()) ans[i] = st.back();

After cleanup, the surviving top is the nearest previous greater. If the stack is empty, there is no such element and we leave the sentinel.

st.push_back(i);

The current index becomes a candidate for future elements.

The deque template, decoded

while (!dq.empty() && dq.front() <= i - k)
    dq.pop_front();

Indices older than i - k + 1 are outside the window of size k and must go. The condition dq.front() <= i - k means "front is at position i - k or earlier", i.e. strictly before the window, so it must be discarded.

while (!dq.empty() && a[dq.back()] <= a[i])
    dq.pop_back();

Any value at the back that is less than or equal to the new value is dominated: the new index is both newer and at least as good, so the old back can never again be the maximum of any window where the new element is also present.

if (i >= k - 1) ans.push_back(a[dq.front()]);

Once the first full window has formed (at index k - 1), every subsequent step yields an answer. The front of the deque is, by the invariant, the maximum of the current window.

13. Edge Cases — Centralized Audit

Stack edge cases

Edge caseWhat to check
empty arrayReturn empty answer; the for-loop naturally does nothing.
one elementNo next/previous answer exists; sentinel stays.
strictly increasing inputFor "next greater": many pops late; for "previous greater": all answers are -1.
strictly decreasing inputFor "next greater": all answers are -1; for "previous greater": every answer is the immediate predecessor.
all equal valuesStrictness flips the result entirely. Decide whether equal counts.
duplicates anywhere< vs <= matters — choose one and document it.
large area / productUse long long. Histogram and contribution problems overflow int easily.
circular arrayUse i % n over a doubled scan; in unresolved-style templates, push only during the first pass to avoid duplicates.
output typeDistinguish value, index, distance, count — they are NOT the same answer.

Deque edge cases

Edge caseWhat to check
k = 1Every element is its own window; answer equals the input.
k = nOne answer: the global max/min.
k > nNo full window; define behavior explicitly (often: return empty).
duplicatesUse < or <= consistently in dominance pruning.
stale indexAlways expire the front BEFORE reading the answer, never after.
negative valuesNo issue — comparisons work the same.
large DP valuesUse long long if accumulating sums or scores.

The duplicate policy for contribution problems

This deserves its own warning. For "Sum of Subarray Minimums" and similar problems, exactly one side should own equality. Two valid configurations exist:

PolicyLeft boundaryRight boundary
Option Aprevious strictly smallernext smaller-or-equal
Option Bprevious smaller-or-equalnext strictly smaller

If both sides are strict, you under-count by missing subarrays whose minimum repeats. If both sides are non-strict, you over-count by counting the same subarray once for each occurrence of the minimum. The asymmetry is essential.

14. Common Pitfalls / Bugs

⚠ Stack pitfalls

  • Storing values when output requires indices or distances.
  • Using <= when "equal" is not a valid answer (or vice versa).
  • Forgetting to initialize ans with -1 (or whatever your sentinel is).
  • Forgetting long long on histogram/contribution multiplications.
  • In circular templates, pushing during both passes — leads to duplicate unresolved entries.
  • Pushing in the wrong place (must be after pops, not before).
  • Confusing "next" with "previous" — "next" always means right, "previous" always means left.

⚠ Deque pitfalls

  • Forgetting to expire stale indices from the front before reading the answer.
  • Mixing the dominance comparator (< vs <=) — pick one, stay consistent.
  • Reading dq.front() before the window is full (use i >= k - 1 guard).
  • Using a plain queue when you also need to remove dominated candidates from the back.
  • Using a plain stack when oldest elements must expire — that's the deque-only signal.
  • Forgetting that dp[dq.front()] reads from dp, not a, in DP-over-deque problems.

"Increasing stack" depends on orientation. Always say "values from bottom to top are non-decreasing" rather than vague "increasing stack". The same physical stack can be called increasing or decreasing depending on which end you look from. Be explicit.

⚠

Stack vs deque: if any element can become invalid simply because time passed (i.e., the window moved past it), you cannot use a stack. The signal is the word window or any phrase like "last k", "within distance d", or "while xj − xi ≤ k".

16. Variants & Extensions

A. Value-based monotonic queue with FIFO pop

Sometimes you know the outgoing value exactly (a true FIFO stream). Then a value-based monotonic queue works without storing indices. This is the right tool for a streaming maximum where elements expire by FIFO order, not by sliding-window index distance.

struct MonotonicMaxQueue {
    queue<int> q;
    deque<int> dq;

    void push(int x) {
        q.push(x);
        while (!dq.empty() && dq.back() < x) dq.pop_back();
        dq.push_back(x);
    }
    void pop() {
        if (q.empty()) return;
        if (!dq.empty() && q.front() == dq.front()) dq.pop_front();
        q.pop();
    }
    int maxValue() const { return dq.front(); }
};

When to use: elements pop in exact FIFO order, you do not need indices, distances, or positions. When NOT to use: classical sliding-window problems — there, the index deque is preferred because it handles duplicates and expiry by position automatically.

B. Queue with two monotonic stacks

This supports queue max/min with amortized O(1) movement, useful for problems where you need a queue aggregate but the structure is not a simple sliding window. The trick: each stack tracks its own max as it is pushed, and the queue is two stacks (in/out) joined together.

struct MaxStack {
    vector<pair<int,int>> st; // (value, running max)
    void push(int x) {
        int mx = st.empty() ? x : max(x, st.back().second);
        st.push_back({x, mx});
    }
    void pop()         { st.pop_back(); }
    bool empty() const { return st.empty(); }
    int  top()   const { return st.back().first; }
    int  getMax()const { return st.back().second; }
};

struct MaxQueue {
    MaxStack in, out;
    void move() {
        if (!out.empty()) return;
        while (!in.empty()) { int x = in.top(); in.pop(); out.push(x); }
    }
    void push(int x) { in.push(x); }
    void pop()       { move(); out.pop(); }
    int  getMax() {
        if (in.empty())  return out.getMax();
        if (out.empty()) return in.getMax();
        return max(in.getMax(), out.getMax());
    }
};

Each element moves at most twice (push to in, then once to out), so total work is amortized O(1) per operation across any sequence.

C. 2D sliding window minimum

Given an n × m matrix, find the minimum in every a × b submatrix. Reduce to two passes: first apply 1D sliding-min along each row with width b, then apply 1D sliding-min along each column of the result with height a. Total time O(n · m).

vector<int> windowMin1D(const vector<int>& row, int k) {
    int n = (int)row.size();
    vector<int> ans;
    deque<int> dq;
    for (int i = 0; i < n; i++) {
        while (!dq.empty() && dq.front() <= i - k) dq.pop_front();
        while (!dq.empty() && row[dq.back()] >= row[i]) dq.pop_back();
        dq.push_back(i);
        if (i >= k - 1) ans.push_back(row[dq.front()]);
    }
    return ans;
}

vector<vector<int>> submatrixMinimums(const vector<vector<int>>& mat, int a, int b) {
    int n = (int)mat.size(), m = (int)mat[0].size();
    vector<vector<int>> rowMin(n);
    for (int i = 0; i < n; i++) rowMin[i] = windowMin1D(mat[i], b);

    int newM = m - b + 1;
    vector<vector<int>> ans(n - a + 1, vector<int>(newM));
    for (int col = 0; col < newM; col++) {
        deque<int> dq;
        for (int row = 0; row < n; row++) {
            while (!dq.empty() && dq.front() <= row - a) dq.pop_front();
            while (!dq.empty() && rowMin[dq.back()][col] >= rowMin[row][col]) dq.pop_back();
            dq.push_back(row);
            if (row >= a - 1) ans[row - a + 1][col] = rowMin[dq.front()][col];
        }
    }
    return ans;
}

D. Circular next greater (virtual doubling)

For a circular array, every index i can search up to 2n − 1 virtual positions before wrapping all the way around. Use i % n on a doubled scan. The careful bit: in the unresolved-style template, push only during the first pass (or your stack will fill with duplicates).

// Right-to-left candidate-stack version (push during both halves is fine here):
class Solution {
public:
    vector<int> nextGreaterElements(vector<int>& nums) {
        int n = (int)nums.size();
        vector<int> ans(n, -1), st;
        for (int t = 2 * n - 1; t >= 0; t--) {
            int i = t % n;
            while (!st.empty() && nums[st.back()] <= nums[i]) st.pop_back();
            if (t < n && !st.empty()) ans[i] = nums[st.back()];
            st.push_back(i);
        }
        return ans;
    }
};

E. Online / streaming patterns — when there is no array to index

Every template above assumed you have the full input upfront and can store indices on the stack. In an online or streaming setting that assumption breaks: values arrive one at a time, you must answer each query immediately, and there is no permanent array to point into. The monotonic logic still works — you just shift what the stack stores.

The classic example is Online Stock Spanner (LeetCode 901). The query at each new price is "how many consecutive prior days, including today, had a price less than or equal to today?" — a textbook left-extending span until a strictly greater previous price. Offline this is "previous strictly greater" on a monotonic stack of indices. Online, we have no indices to store, so we store {value, span_count} pairs instead: when the new price dominates an old top, we add the popped entry's span into our own span and discard it. The popped entry is permanently absorbed into the new entry's span, so its individual value is no longer needed.

class StockSpanner {
    // Each pair: {price, span ending at that price}.
    // Stack is strictly decreasing in price from bottom to top.
    vector<pair<int,int>> st;
public:
    int next(int price) {
        int span = 1;                      // today itself counts as 1
        while (!st.empty() && st.back().first <= price) {
            span += st.back().second;      // absorb the popped entry's span
            st.pop_back();                 // it is now dominated forever
        }
        st.push_back({price, span});       // push the merged candidate
        return span;
    }
};

The amortized analysis is identical to the offline case: every price is pushed exactly once and popped at most once, so a long run of next() calls runs in total linear time even though a single call can occasionally do a lot of pops. The same compression trick — store the aggregate, not the individual indices — generalizes to any online problem where popped elements can be safely merged into the survivor: streaming "next greater so far", running maximum-of-tail, and online range queries with a known monotone window.

📡

Recognition cue. If the problem says "design a class with a method that is called once per element" or "values arrive one at a time and each query must be answered immediately", you cannot use indices into a future-unknown array. Switch to {value, aggregate} pairs and let domination merge them.

17a. Problem-Based Applications — Classical Stack Problems

Problem 1 — Next Greater Element (foundation) VJudge

Story. For every number, find the first greater number to its right.

Why monotonic stack fits. If the current value is greater than the stack top, it resolves that index. Each unresolved index waits on the stack until a future greater value arrives.

vector<int> nextGreaterValues(const vector<int>& a) {
    int n = (int)a.size();
    vector<int> ans(n, -1), st;
    for (int i = 0; i < n; i++) {
        while (!st.empty() && a[st.back()] < a[i]) {
            ans[st.back()] = a[i];
            st.pop_back();
        }
        st.push_back(i);
    }
    return ans;
}

Complexity. O(n) time, O(n) space. Pitfall: using <= when "equal" is not greater; storing values when the output requires indices.

Problem 2 — LeetCode 496: NGE I (subset query) VJudge

Story. Given nums2 (full, unique values) and nums1 (subset), find the next greater value in nums2 for each value in nums1.

Approach. Preprocess nums2 with a value-to-NGE map (uniqueness of values makes value-keying safe), then answer each query in O(1).

class Solution {
public:
    vector<int> nextGreaterElement(vector<int>& nums1, vector<int>& nums2) {
        unordered_map<int,int> nxt;
        vector<int> st; // values are safe here because nums2 is unique
        for (int x : nums2) {
            while (!st.empty() && st.back() < x) {
                nxt[st.back()] = x;
                st.pop_back();
            }
            st.push_back(x);
        }
        while (!st.empty()) { nxt[st.back()] = -1; st.pop_back(); }

        vector<int> ans;
        for (int x : nums1) ans.push_back(nxt[x]);
        return ans;
    }
};

Complexity. O(|nums1| + |nums2|). Pitfall: if values are not unique, switch to indices.

Problem 3 — Daily Temperatures VJudge

Story. For each day, return how many days until a strictly warmer temperature; 0 if none.

Why < not <=? Equal temperature is not warmer, so equal does not resolve a waiting index.

class Solution {
public:
    vector<int> dailyTemperatures(vector<int>& temperatures) {
        int n = (int)temperatures.size();
        vector<int> ans(n, 0), st;
        for (int i = 0; i < n; i++) {
            while (!st.empty() && temperatures[st.back()] < temperatures[i]) {
                int j = st.back(); st.pop_back();
                ans[j] = i - j;
            }
            st.push_back(i);
        }
        return ans;
    }
};

Complexity. O(n) time, O(n) space.

Problem 4 — Final Prices With a Special Discount VJudge

Story. For each prices[i] find the first prices[j] to the right with prices[j] <= prices[i]. Subtract it.

Why > not >=? Equal price IS a valid discount, so we must NOT pop equal values during the right-to-left scan.

class Solution {
public:
    vector<int> finalPrices(vector<int>& prices) {
        int n = (int)prices.size();
        vector<int> ans(n), st;
        for (int i = n - 1; i >= 0; i--) {
            while (!st.empty() && prices[st.back()] > prices[i])
                st.pop_back();
            int discount = st.empty() ? 0 : prices[st.back()];
            ans[i] = prices[i] - discount;
            st.push_back(i);
        }
        return ans;
    }
};

Complexity. O(n) time, O(n) space.

Problem 5 — Circular Next Greater Element VJudge

Story. The array is circular: after the last element, we continue from the first.

Approach. Iterate 2n − 1 virtual positions, mapping i = t % n. For this right-to-left candidate-stack template, pushing during both halves is correct because we are storing candidates, not unresolved indices.

class Solution {
public:
    vector<int> nextGreaterElements(vector<int>& nums) {
        int n = (int)nums.size();
        vector<int> ans(n, -1), st;
        for (int t = 2 * n - 1; t >= 0; t--) {
            int i = t % n;
            while (!st.empty() && nums[st.back()] <= nums[i])
                st.pop_back();
            if (t < n && !st.empty()) ans[i] = nums[st.back()];
            st.push_back(i);
        }
        return ans;
    }
};
Problem 6 — Largest Rectangle in Histogram VJudge

Story. Given bar heights, find the largest axis-aligned rectangle that fits inside the histogram.

Why monotonic stack fits. For every bar i, the rectangle limited by h[i] can extend until the first smaller bar on each side. The pop event of an increasing-stack template hands you both boundaries simultaneously: the right boundary is the current index (because it forced the pop), and the left boundary is whatever sits below the popped element on the stack.

class Solution {
public:
    int largestRectangleArea(vector<int>& heights) {
        vector<int> h = heights;
        h.push_back(0); // Sentinel value: forces all remaining bars to be popped and the stack to flush at the end.
        vector<int> st;
        long long best = 0;
        for (int i = 0; i < (int)h.size(); i++) {
            while (!st.empty() && h[st.back()] > h[i]) {
                int mid = st.back(); st.pop_back();
                int left  = st.empty() ? -1 : st.back();
                int right = i;
                long long width = right - left - 1;
                long long area  = (long long)h[mid] * width;
                best = max(best, area);
            }
            st.push_back(i);
        }
        return (int)best;
    }
};

Complexity. O(n) time, O(n) space.

Problem 7 — SPOJ HISTOGRA VJudge

Story. Multiple test cases; for each histogram output the largest rectangle area. Width × height can overflow int for large inputs, so the entire computation runs in long long.

#include <bits/stdc++.h>

long long largestRectangleArea(const vector<long long>& h) {

    vector<int> st;
    long long best = 0;
    for (int i = 0; i <= n; i++) {
        long long cur = (i == n ? 0 : h[i]); // virtual zero sentinel
        while (!st.empty() && h[st.back()] > cur) {

            int right = i;

    }

}

Problem 8 — Trapping Rain Water VJudge

Story. Given bar heights, compute trapped water above the bars.

Approach. When the current bar is taller than the stack top, the popped bar is the bottom of a valley. The new stack top is the left wall, the current bar is the right wall, and the water above the bottom is bounded by min(left, right) − height[bottom].

class Solution {
public:
    int trap(vector<int>& height) {
        int n = (int)height.size();
        vector<int> st;
        long long water = 0;
        for (int i = 0; i < n; i++) {
            while (!st.empty() && height[i] > height[st.back()]) {
                int bottom = st.back(); st.pop_back();
                if (st.empty()) break;
                int left = st.back();
                int width = i - left - 1;
                int boundedHeight = min(height[left], height[i]) - height[bottom];
                if (boundedHeight > 0)
                    water += (long long)width * boundedHeight;
            }
            st.push_back(i);
        }
        return (int)water;
    }
};

17b. Problem-Based Applications — Contribution Problems

The contribution trick is a viewpoint shift. Rather than enumerating all O(n²) subarrays and computing each minimum, we ask: for how many subarrays is each element the minimum? Each element contributes value × L × R, where L and R are the left and right reach until a bound. The two reaches come from two monotonic stacks.

Problem 9 — Sum of Subarray Minimums VJudge

Story. Sum the minimums of every contiguous subarray, modulo 10⁹ + 7.

Duplicate policy. Use previous strictly smaller and next smaller-or-equal. This asymmetry assigns each subarray's minimum to the rightmost occurrence among equal minima, preventing double counting when the minimum repeats.

class Solution {
public:
    int sumSubarrayMins(vector<int>& arr) {
        const long long MOD = 1000000007LL;
        int n = (int)arr.size();
        vector<int> left(n), right(n), st;

        // previous strictly smaller — left reach
        for (int i = 0; i < n; i++) {
            while (!st.empty() && arr[st.back()] >= arr[i]) st.pop_back();
            left[i] = st.empty() ? i + 1 : i - st.back();
            st.push_back(i);
        }
        st.clear();
        // next smaller-or-equal — right reach
        for (int i = n - 1; i >= 0; i--) {
            while (!st.empty() && arr[st.back()] > arr[i]) st.pop_back();
            right[i] = st.empty() ? n - i : st.back() - i;
            st.push_back(i);
        }

        long long ans = 0;
        for (int i = 0; i < n; i++)
            ans = (ans + (long long)arr[i] * left[i] % MOD * right[i]) % MOD;
        return (int)ans;
    }
};
Problem 10 — Sum of Subarray Ranges VJudge

Story. Sum (max − min) across all subarrays. Trick. Sum of ranges equals sum of "max contributions" minus sum of "min contributions" — two independent contribution computations. Same asymmetric strictness rule applies, mirrored for max.

class Solution {
public:
    long long subArrayRanges(vector<int>& nums) {
        int n = (int)nums.size();

        auto contributionAsMax = [&](){
            vector<int> left(n), right(n), st;
            for (int i = 0; i < n; i++) {
                while (!st.empty() && nums[st.back()] <= nums[i]) st.pop_back();
                left[i] = st.empty() ? i + 1 : i - st.back();
                st.push_back(i);
            }
            st.clear();
            for (int i = n - 1; i >= 0; i--) {
                while (!st.empty() && nums[st.back()] < nums[i]) st.pop_back();
                right[i] = st.empty() ? n - i : st.back() - i;
                st.push_back(i);
            }
            long long s = 0;
            for (int i = 0; i < n; i++) s += (long long)nums[i] * left[i] * right[i];
            return s;
        };
        auto contributionAsMin = [&](){
            vector<int> left(n), right(n), st;
            for (int i = 0; i < n; i++) {
                while (!st.empty() && nums[st.back()] >= nums[i]) st.pop_back();
                left[i] = st.empty() ? i + 1 : i - st.back();
                st.push_back(i);
            }
            st.clear();
            for (int i = n - 1; i >= 0; i--) {
                while (!st.empty() && nums[st.back()] > nums[i]) st.pop_back();
                right[i] = st.empty() ? n - i : st.back() - i;
                st.push_back(i);
            }
            long long s = 0;
            for (int i = 0; i < n; i++) s += (long long)nums[i] * left[i] * right[i];
            return s;
        };

        return contributionAsMax() - contributionAsMin();
    }
};
Problem 11 — Codeforces 1913D Array Collapse VJudge

Story. The array has distinct values. One operation replaces a chosen subarray by its minimum. Count the number of distinct arrays reachable after any sequence of operations, modulo a prime.

Approach. Define dp[i] as the number of reachable arrays whose last surviving element is a[i]. To place some j immediately before i in the final result, the subarray a[j..i] must have a terminal minimum — its min must lie at j or at i; an interior minimum would refuse to vanish. The two cases give a sum over the monotonic-increasing stack (where each element is a possible left terminal minimum) and a sum over the range strictly between the previous smaller and i (where i itself is the terminal minimum). Maintaining a running stack-sum and a prefix sum of dp reduces both queries to O(1) and gives an O(n) overall solution. What sumStack really is: it is the running sum of dp values for exactly the elements currently surviving on the monotonic stack — every push adds dp[i] into sumStack and every pop subtracts the popped dp value back out, so it always equals the answer to "sum of dp[j] over all j still in the stack". That is precisely the left-terminal-minimum query, computed in O(1) instead of O(stack depth) — the single trick that turns an O(n²) DP transition into O(n). The final answer counts only suffix minima, which are exactly the final-position candidates.

#include <bits/stdc++.h>
using namespace std;
const long long MOD = 998244353;

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);
    int T; cin >> T;
    while (T--) {
        int n; cin >> n;
        vector<int> a(n);
        for (int i = 0; i < n; i++) cin >> a[i];

        vector<long long> dp(n, 0), pref(n + 1, 0);
        vector<int> st;
        long long sumStack = 0;
        int prefMin = INT_MAX;

        for (int i = 0; i < n; i++) {
            while (!st.empty() && a[st.back()] > a[i]) {
                sumStack = (sumStack - dp[st.back()] + MOD) % MOD;
                st.pop_back();
            }
            int prevSmaller = st.empty() ? -1 : st.back();

            long long rightTerminal = 0;
            int L = prevSmaller + 1, R = i - 1;
            if (L <= R) rightTerminal = (pref[R + 1] - pref[L] + MOD) % MOD;

            long long leftTerminal = sumStack;
            dp[i] = (rightTerminal + leftTerminal) % MOD;
            if (a[i] < prefMin) dp[i] = (dp[i] + 1) % MOD;
            prefMin = min(prefMin, a[i]);

            st.push_back(i);
            sumStack    = (sumStack + dp[i]) % MOD;
            pref[i + 1] = (pref[i] + dp[i]) % MOD;
        }

        long long ans = 0;
        int suffixMin = INT_MAX;
        for (int i = n - 1; i >= 0; i--) {
            if (a[i] < suffixMin) ans = (ans + dp[i]) % MOD;
            suffixMin = min(suffixMin, a[i]);
        }
        cout << ans % MOD << '\n';
    }
    return 0;
}
⚠

Teach this last. The Array Collapse trick layers four ideas — monotonic stack, contribution counting, prefix-sum DP optimization, and suffix-minimum filtering. Each prerequisite must be solid first.

17c. Problem-Based Applications — Greedy Monotonic Stack

A different family: not "find nearest" but "build the lexicographically best output". The stack stores a partial answer, and we pop whenever a previous choice is strictly worse than what is now possible. The bookkeeping (counter limits, repetition tracking) varies per problem, but the core idea — pop while the top is dominated by the new arrival — is the same.

Problem 12 — Remove K Digits VJudge

Story. Remove exactly k digits from a numeric string so the remaining number is smallest.

Why monotonic stack fits. To make a number small, the leftmost digits should be as small as possible. Whenever a previously kept digit is larger than the next digit, removing it improves the result. After processing, if removals remain, drop them from the end (which hurts least once digits are non-decreasing). Strip leading zeros at the end.

class Solution {
public:
    string removeKdigits(string num, int k) {
        string st;
        for (char c : num) {
            while (!st.empty() && k > 0 && st.back() > c) {
                st.pop_back(); k--;
            }
            st.push_back(c);
        }
        while (!st.empty() && k > 0) { st.pop_back(); k--; }
        int pos = 0;
        while (pos < (int)st.size() && st[pos] == '0') pos++;
        string ans = st.substr(pos);
        return ans.empty() ? "0" : ans;
    }
};
Problem 13 — Remove Duplicate Letters VJudge

Story. Remove duplicate letters so each letter appears once and the result is lexicographically smallest.

Extra conditions. A character may be popped only if it appears again later (otherwise removing it loses it forever) and the current character is not already used. Track remaining[c] and used[c] alongside the stack.

class Solution {
public:
    string removeDuplicateLetters(string s) {
        vector<int> remaining(256, 0);
        vector<bool> used(256, false);
        for (char c : s) remaining[c]++;
        string st;
        for (char c : s) {
            remaining[c]--;
            if (used[c]) continue;
            while (!st.empty() && st.back() > c && remaining[st.back()] > 0) {
                used[st.back()] = false;
                st.pop_back();
            }
            st.push_back(c);
            used[c] = true;
        }
        return st;
    }
};
Problem 14 — Maximum Width Ramp VJudge

Story. Find the maximum j − i with i ≤ j and nums[i] ≤ nums[j].

Why monotonic stack fits. Only strictly decreasing left-candidates matter — any element that is greater than or equal to an earlier value is a worse left endpoint. Build the candidate stack in one forward pass, then scan from the right; whenever the right element can pair with the top, that is the farthest possible right for that left, so pop it.

class Solution {
public:
    int maxWidthRamp(vector<int>& nums) {
        int n = (int)nums.size();
        vector<int> st;
        for (int i = 0; i < n; i++)
            if (st.empty() || nums[i] < nums[st.back()]) st.push_back(i);

        int ans = 0;
        for (int j = n - 1; j >= 0; j--) {
            while (!st.empty() && nums[st.back()] <= nums[j]) {
                ans = max(ans, j - st.back());
                st.pop_back();
            }
        }
        return ans;
    }
};
Problem 15 — 132 Pattern VJudge

Story. Decide whether there exist i < j < k with nums[i] < nums[k] < nums[j].

Approach. Scan right to left, maintaining a decreasing stack of candidates for the "3" value. A separate variable candidate2 stores the best "2" value found so far (the largest value ever popped). If a current value falls below candidate2, we have found the "1".

class Solution {
public:
    bool find132pattern(vector<int>& nums) {
        int n = (int)nums.size();
        vector<int> st;
        int candidate2 = INT_MIN;
        for (int i = n - 1; i >= 0; i--) {
            if (nums[i] < candidate2) return true;
            while (!st.empty() && nums[i] > st.back()) {
                candidate2 = max(candidate2, st.back());
                st.pop_back();
            }
            st.push_back(nums[i]);
        }
        return false;
    }
};
Problem 16 — Number of Visible People in a Queue VJudge

Story. A person sees shorter people to their right until a taller-or-equal blocker, who is also visible. Approach. Right-to-left scan with a monotonic stack of strictly decreasing heights. The popped people are visible; if anyone remains after popping, that one taller-or-equal blocker is also visible.

class Solution {
public:
    vector<int> canSeePersonsCount(vector<int>& heights) {
        int n = (int)heights.size();
        vector<int> ans(n, 0), st;
        for (int i = n - 1; i >= 0; i--) {
            while (!st.empty() && st.back() < heights[i]) {
                ans[i]++;
                st.pop_back();
            }
            if (!st.empty()) ans[i]++;
            st.push_back(heights[i]);
        }
        return ans;
    }
};

17d. Problem-Based Applications — Monotonic Deque

Problem 17 — Sliding Window Maximum VJudge

Story. For every window of size k, return its maximum. Why deque, not heap? A heap with lazy deletion is O(n log k); the monotonic deque is true O(n). The deque also has trivially small constants — no log factors, no allocator churn.

class Solution {
public:
    vector<int> maxSlidingWindow(vector<int>& nums, int k) {
        int n = (int)nums.size();
        vector<int> ans;
        deque<int> dq;
        for (int i = 0; i < n; i++) {
            while (!dq.empty() && dq.front() <= i - k) dq.pop_front();
            while (!dq.empty() && nums[dq.back()] <= nums[i]) dq.pop_back();
            dq.push_back(i);
            if (i >= k - 1) ans.push_back(nums[dq.front()]);
        }
        return ans;
    }
};

Problem 19 — HackerRank Queries With Fixed Length VJudge

Story. For each query d, compute the maximum of every window of length d, then return the minimum among those maximums.

#include <bits/stdc++.h>
using namespace std;

vector<int> solveQueries(const vector<int>& a, const vector<int>& queries) {
    int n = (int)a.size();
    vector<int> output;
    for (int k : queries) {
        deque<int> dq;
        int best = INT_MAX;
        for (int i = 0; i < n; i++) {
            while (!dq.empty() && dq.front() <= i - k) dq.pop_front();
            while (!dq.empty() && a[dq.back()] <= a[i]) dq.pop_back();
            dq.push_back(i);
            if (i >= k - 1) best = min(best, a[dq.front()]);
        }
        output.push_back(best);
    }
    return output;
}

Problem 20 — Jump Game VI VJudge

Story. From index i you can jump to any of i + 1, …, i + k. Maximize the sum of values collected from 0 to n − 1.

DP. dp[i] = nums[i] + max(dp[j]) for i − k ≤ j < i. The "max over a window of size k of past dp values" is exactly sliding-window maximum. The deque stores indices with decreasing dp values, and the front is the best previous state.

class Solution {
public:
    int maxResult(vector<int>& nums, int k) {
        int n = (int)nums.size();
        vector<int> dp(n);
        deque<int> dq;
        dp[0] = nums[0];
        dq.push_back(0);
        for (int i = 1; i < n; i++) {
            while (!dq.empty() && dq.front() < i - k) dq.pop_front();
            dp[i] = nums[i] + dp[dq.front()];
            while (!dq.empty() && dp[dq.back()] <= dp[i]) dq.pop_back();
            dq.push_back(i);
        }
        return dp[n - 1];
    }
};
Problem 21 — Shortest Subarray With Sum at Least K VJudge

Story. Possibly negative numbers, find the shortest subarray with sum at least k. Plain sliding-window fails because expanding/shrinking is no longer monotonic with negatives.

Trick. Use prefix sums: a subarray sum is pref[r] − pref[l]. For each right endpoint r, we want some l < r with pref[l] ≤ pref[r] − k minimizing r − l. A deque of candidate prefix indices, kept increasing in pref values from front to back, supports both pops correctly. Front pop: when pref[r] − pref[dq.front()] ≥ k, the front already yields a valid subarray ending at r; we record its length and pop it because the front is the oldest index in the deque, and for any future r' > r the same front would only produce a longer subarray r' − front ≥ r − front, never a better one — so it can be safely retired. Back pop: when pref[dq.back()] ≥ pref[r], the new index r dominates the old back: r is later and has a smaller-or-equal prefix sum, so as a future left endpoint it will satisfy the threshold sooner and produce a strictly shorter subarray. The old back is therefore useless from now on.

class Solution {
public:
    int shortestSubarray(vector<int>& nums, int k) {
        int n = (int)nums.size();
        vector<long long> pref(n + 1, 0);
        for (int i = 0; i < n; i++) pref[i + 1] = pref[i] + nums[i];
        deque<int> dq;
        int ans = n + 1;
        for (int r = 0; r <= n; r++) {
            while (!dq.empty() && pref[r] - pref[dq.front()] >= k) {
                ans = min(ans, r - dq.front());
                dq.pop_front();
            }
            while (!dq.empty() && pref[dq.back()] >= pref[r])
                dq.pop_back();
            dq.push_back(r);
        }
        return ans == n + 1 ? -1 : ans;
    }
};
Problem 22 — Longest Continuous Subarray With Absolute Diff ≤ Limit VJudge

Story. Find the longest subarray with max − min ≤ limit. Approach. Two deques — a decreasing one for max, an increasing one for min. Slide the right endpoint, and whenever the constraint is violated, advance the left endpoint and synchronize both deques.

class Solution {
public:
    int longestSubarray(vector<int>& nums, int limit) {
        int n = (int)nums.size();
        deque<int> maxDq, minDq;
        int l = 0, ans = 0;
        for (int r = 0; r < n; r++) {
            while (!maxDq.empty() && nums[maxDq.back()] <= nums[r]) maxDq.pop_back();
            maxDq.push_back(r);
            while (!minDq.empty() && nums[minDq.back()] >= nums[r]) minDq.pop_back();
            minDq.push_back(r);
            while (!maxDq.empty() && !minDq.empty() &&
                   nums[maxDq.front()] - nums[minDq.front()] > limit) {
                if (maxDq.front() == l) maxDq.pop_front();
                if (minDq.front() == l) minDq.pop_front();
                l++;
            }
            ans = max(ans, r - l + 1);
        }
        return ans;
    }
};
Problem 23 — Max Value of Equation VJudge

Story. Points sorted by x, maximize yi + yj + |xi − xj| with xj − xi ≤ k and i < j.

Decomposition. Since xj > xi, the expression equals (yi − xi) + (yj + xj). For each current point j, we want the maximum of (yi − xi) among earlier points within distance k in x. That is sliding-window maximum on the value y − x, with the window expressed in x-distance rather than index distance.

class Solution {
public:
    int findMaxValueOfEquation(vector<vector<int>>& points, int k) {
        deque<int> dq;
        int ans = INT_MIN;
        for (int j = 0; j < (int)points.size(); j++) {
            int xj = points[j][0], yj = points[j][1];
            while (!dq.empty() && xj - points[dq.front()][0] > k) dq.pop_front();
            if (!dq.empty()) {
                int i = dq.front();
                ans = max(ans, yj + xj + points[i][1] - points[i][0]);
            }
            int valJ = yj - xj;
            while (!dq.empty()) {
                int back = dq.back();
                int valBack = points[back][1] - points[back][0];
                if (valBack >= valJ) break;
                dq.pop_back();
            }
            dq.push_back(j);
        }
        return ans;
    }
};

18. Final Summary & Checklist

Monotonic stack and monotonic deque are not exotic data structures. They are ordinary stacks and deques used with one extra discipline: candidate survival. A candidate stays only as long as it can still become an answer for someone. The instant the current element proves that the candidate is forever dominated, the candidate is destroyed. This single rule produces linear-time algorithms across an enormous range of problems that look quadratic at first glance.

The questions to ask before any monotonic problem

The recognition reflex you should drill is a sequence of seven small decisions. What is each element trying to find? Decides whether you need a stack at all. Is the answer on the left or right? Decides scan direction. Greater or smaller? Decides the comparison direction. Does equality count? Decides strict vs or-equal. Does the window expire old elements over time? Decides stack vs deque. What output do I need — value, index, distance, width, area, or count? Decides what you store and how you read. What does a popped element mean? What does the surviving top mean? Decides where you write the answer.

✓ Pre-submit checklist

  • Stored indices, not values
  • Initialized ans with the right sentinel
  • Comparison strictness is correct for the problem
  • Equality handling is consistent on both sides for contribution
  • Scan direction matches "next" vs "previous"
  • Used deque (not stack) when window expires
  • Removed expired indices from the front before reading the answer
  • Handled k = 1, k = n, empty input
  • Used long long for area/contribution arithmetic
  • Avoided duplicate pushes in circular unresolved-style templates
  • Handled empty result for Remove K Digits and similar
  • Final stack/deque drain processed if the algorithm requires it

→ Core phrases to remember

  • Current resolves popped elements (next-side)
  • Remaining stack top is the nearest previous valid candidate (prev-side)
  • Deque front is the current window's answer
  • Pop dominated candidates from the back
  • Expire invalid old candidates from the front
  • Count total pushes and pops, not nested loop shape
  • For rectangles: pop yields BOTH left and right boundary at once
  • For contribution: one side strict, the other or-equal — never both the same

Everything else is local variations on the same skeleton.
