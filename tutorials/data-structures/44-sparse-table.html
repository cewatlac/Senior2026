Sparse Table · 2D · Disjoint

Static range queries · O(n log n) build · O(1) query · associativity vs idempotence

🧩 Problem Definition

Given an array a[0..n-1], we receive many range queries of the form:

minimum on [l, r] maximum on [l, r] gcd on [l, r] bitwise AND / OR on [l, r] range sum / product (special — see Disjoint ST)

Concrete example

a = [5, 2, 7, 3, 6, 1]

query (minimum, l=1, r=4):
    a[1..4] = 2, 7, 3, 6
    minimum = 2

Naïve solution

int ans = a[l];
for (int i = l + 1; i <= r; i++)
    ans = min(ans, a[i]);

Cost per query: O(r − l + 1). With n = q = 10⁵, total work reaches 10¹⁰ — far too slow.

We need preprocessing so that each query becomes essentially free. The Sparse Table family delivers exactly this:

Build: O(n log n) Query: O(1) No updates allowed

💡 Intuition from Scratch

Think of three escalating ideas.

Idea 1 — Memoize every interval

Precompute the answer for every interval [l, r]. Each query is O(1), but we store O(n²) answers — too much memory for n = 10⁵.

Idea 2 — Memoize only power-of-two intervals

Store answers only for intervals whose length is a power of two: 1, 2, 4, 8, 16, … That is only O(n log n) intervals. The question becomes: can we still answer an arbitrary query in O(1)?

Idea 3 — Cover any query with two power-of-two blocks

Take a query of length L. Let k = ⌊log₂(L)⌋, so 2ᵏ ≤ L < 2ᵏ⁺¹. Place one block of length 2ᵏ at the left end and another at the right end:

query of length 6  →  largest power of two ≤ 6 is 4

indices:        0  1  2  3  4  5
left  block:   [0  1  2  3]
right block:         [2  3  4  5]
                   overlap = {2, 3}

The two blocks cover the whole query. They may overlap — and that is exactly where the deepest design choice of the data structure lives:

For min / max / gcd / AND / OR — overlap is harmless For sum / product — overlap double-counts

The first family is solved by the classic Sparse Table. The second is solved by the Disjoint Sparse Table, which trades the overlap trick for a clever per-block prefix/suffix layout.

How a cell is built

A block of length 2ᵏ is the union of two adjacent half-blocks of length 2ᵏ⁻¹:

st[k][i] = op( st[k-1][i],
               st[k-1][i + 2ᵏ⁻¹] )

🧠 Formal Abstraction

Sparse Table is a monoid query structure. The operation op must satisfy at least one of two algebraic properties.

Property 1 — Associativity (always required)

op( op(a, b), c )  =  op( a, op(b, c) )

Associativity lets us build big intervals from smaller intervals — order of grouping does not matter. All Sparse Table variants need this.

Property 2 — Idempotence (required for the classic 1D/2D Sparse Table)

op( x, x )  =  x

Idempotence means duplicating a value does not change the answer. This is what makes overlap harmless.

🔍 Key Observations

Observation 1 — Build uses disjoint halves

When we build st[k][i] from st[k-1][i] and st[k-1][i + 2ᵏ⁻¹], the two halves sit side by side and do not overlap. Because of this, the build phase only ever requires associativity. It works correctly for sum, product, matrix multiplication, or any other associative merge — the operation never has to be idempotent.

Observation 2 — Query uses (possibly) overlapping blocks

When we answer [l, r) with k = ⌊log₂(r-l)⌋, the left block [l, l+2ᵏ) and the right block [r-2ᵏ, r) may share elements in the middle. The query is correct only when op is idempotent so that those shared elements contribute the same whether they appear in one block or in both.

Observation 3 — The Disjoint Sparse Table escape hatch

If we cannot afford overlap because our operation is not idempotent, we change the storage layout entirely. At each level we cut the array into blocks and store suffixes inside the left half of every block and prefixes inside the right half. Any query [l, r] straddles the midpoint of exactly one block, and we glue together one suffix and one prefix that are guaranteed to be disjoint. This is the trick that allows the Disjoint Sparse Table to handle any associative operation.

Observation 4 — log table beats log2()

Precomputing lg[i] = ⌊log₂(i)⌋ in pure integer arithmetic is both faster and safer than calling log2(double). The floating-point function can return 1.9999999 for an exact input of 2.0 and round down to 1 when we wanted 2. This is a real and famous source of wrong-answer verdicts.

⚙️ Algorithm Explanation

The Sparse Table algorithm has two phases that mirror each other. The build phase fills the table level by level using disjoint halves, and the query phase answers any range using two power-of-two blocks placed at the two ends of the range. Both phases share the same algebraic engine: the merge function op.

Build (preprocessing)

// level 0 = original array
for i in 0..n:
    st[0][i] = a[i]

// climb levels
for k in 1..⌊log₂ n⌋:
    half = 2^(k-1)
    for i in 0..n - 2^k:
        st[k][i] = op( st[k-1][i],
                       st[k-1][i + half] )

Each level reuses the previous level's results. Building goes from short blocks to long blocks, and the work at level k touches at most n − 2ᵏ + 1 cells.

Query (O(1) answer)

// half-open: answer for [l, r)
len = r - l
k   = ⌊log₂ len⌋          // from lg[]
blk = 1 << k

return op( st[k][l],
           st[k][r - blk] )

Only two table lookups and one merge operation. No loop, no branch, no recursion. This is the heart of the O(1) query.

📐 Invariants

Every contest-grade data structure earns its correctness from a small set of invariants that hold at every moment of its life. Naming them keeps the implementation honest.

A) State invariant — "what each cell means"

For every valid k and i,
    st[k][i]  stores the correct answer for the half-open interval
              [i, i + 2ᵏ)
    where "valid" means  0 ≤ i  and  i + 2ᵏ ≤ n.

B) Transition invariant — "how a level is built"

If level k-1 is correct, then for every valid (k, i):
    st[k][i] = op( st[k-1][i],  st[k-1][i + 2ᵏ⁻¹] )
holds, and the two halves are disjoint and adjacent.

C) Termination invariant — "build stops at the largest needed level"

Let K = ⌊log₂ n⌋ + 1.
The loop fills levels k = 1, 2, ..., K - 1.
After it ends, every query length 1 ≤ L ≤ n maps to some level 0 ≤ k ≤ K - 1
                     with  2ᵏ ≤ L < 2ᵏ⁺¹.

D) Failure modes (when the invariants are violated)

SymptomRoot cause
Wrong answer on overlapping blocks for sum/productIdempotence missing. Use Disjoint ST.
Out-of-bounds read at high levelBuilt st[k][i] for invalid i + 2ᵏ > n.
Off-by-one on the right blockForgot half-open vs inclusive. Use query(l, r+1) when the problem says [l, r].
Silent wrong answer after an updateSparse Table is static. Any update needs a rebuild or a different structure.
Crash on __builtin_clz(0)Empty query. Always guard l < r before calling.

📜 Proof of Correctness

Theorem 1 — Build correctness

For every valid (k, i), after the build loop the cell st[k][i] equals the value of op applied to the multiset a[i], a[i+1], …, a[i + 2ᵏ − 1].

Proof — induction on k click to collapse

Base case (k = 0). The interval has length 2⁰ = 1. The code sets st[0][i] = a[i], which is the unique element of the interval. The base case holds.

Inductive step. Assume the claim holds for level k − 1. Consider the interval [i, i + 2ᵏ). Split it into two adjacent halves:

L = [i, i + 2ᵏ⁻¹)
R = [i + 2ᵏ⁻¹, i + 2ᵏ)

By the inductive hypothesis, st[k-1][i] correctly reduces L and st[k-1][i + 2ᵏ⁻¹] correctly reduces R. Because op is associative, the reduction over L ∪ R equals op(reduce(L), reduce(R)). The build sets exactly that into st[k][i]. ∎

Theorem 2 — Query correctness (classic Sparse Table)

For every non-empty query [l, r) and k = ⌊log₂(r − l)⌋, the returned value op(st[k][l], st[k][r − 2ᵏ]) equals the reduction of op over a[l..r-1], provided op is idempotent.

Proof — coverage + idempotence click to collapse

Let L = [l, l + 2ᵏ) and R = [r − 2ᵏ, r). Both are valid intervals of length 2ᵏ ≤ r − l, so Theorem 1 says they are correctly stored.

Coverage. Every index i ∈ [l, r) lies in L ∪ R. If i < r − 2ᵏ, then i ∈ L (since L extends up to l + 2ᵏ − 1, and l + 2ᵏ − 1 ≥ r − 2ᵏ − 1 follows from 2 · 2ᵏ ≥ r − l, i.e. 2ᵏ⁺¹ > r − l − 1, which is true by the choice of k). Otherwise i ∈ R.

Overlap is harmless. The overlap L ∩ R may be non-empty, but its elements are present in both reductions. Because op is idempotent, duplicating a value during the final merge does not change the result. Therefore op(reduce(L), reduce(R)) = reduce(L ∪ R) = reduce([l, r)). ∎

Theorem 3 — Why sum is not allowed in the classic table

For sum, op(x, x) = 2x ≠ x. Whenever L ∩ R ≠ ∅, the overlap is counted twice, producing a strictly larger answer than the true sum. So the classic Sparse Table is incorrect by construction for sum — see Disjoint Sparse Table below.

⏱ Complexity Analysis

Build

O(n log n) time and memory. Level k processes n − 2ᵏ + 1 cells, and there are ⌊log₂ n⌋ + 1 levels.

Query

O(1) — one lookup in lg[], two table reads, one merge.

Updates

Not supported. A single point update would require rebuilding all levels above it.

2D Sparse Table

Build: O(n · m · log n · log m). Query: O(1). Memory dominates everything — be careful past n, m ≈ 500.

Disjoint Sparse Table

Build: O(n log n). Query: O(1). Same asymptotics as classic ST; the constant factor is a touch higher.

Compared with Segment Tree

Segment Tree: build O(n), query/update O(log n). Lose query speed, gain updates.

🧩 Core Template Philosophy

Across the Coach Academy series we keep a unified template family so that every variant feels familiar after the first one is learned. The Sparse Table family follows the same discipline. The classic 1D and 2D tables share one shape, and the Disjoint Sparse Table is a sibling that swaps the storage strategy without changing the public surface of the data structure.

VariantWhat it storesWhat stays the sameWhat changes
1D Sparse Tablest[k][i] = reduction of [i, i+2ᵏ)operator op, log table, half-open queries—
2D Sparse Tablest[kx][ky][x][y] = reduction of a rectanglesame op, same query shapefour corner blocks instead of two endpoints
Disjoint Sparse Tabletable[k][i] = suffix or prefix inside a level-k blocksame op contract, half-open queriesstorage layout; query uses MSB of l XOR r

Rules of the family

We always pass the operation as a small functor so the same struct can serve min, max, gcd, argmin, or any fully custom merge. We always use half-open intervals internally, so query(l, r) means [l, r). We always precompute a log table rather than relying on log2(double). And we name things consistently across files: n, K, st, lg, op. Adopting these names everywhere makes switching between tutorials cost-free.

📄 Master / Base Implementation — 1D Sparse Table

The template below is the canonical 1D Sparse Table for the Coach Academy series. It uses half-open intervals throughout, so when a problem says "minimum over [l, r]" we call query(l, r + 1). The merge is delegated to a tiny Op functor, so the same struct serves min, max, gcd, AND, OR, or any idempotent associative operation. The struct also keeps no global state, so it is safe to instantiate several copies in the same program (for example one min-table and one max-table on the same array).

Full C++ — 1D Sparse Table click to expand/collapse 
#include <bits/stdc++.h>
using namespace std;

template <class T, class Op>
struct SparseTable {
    int n;
    vector<int> lg;
    vector<vector<T>> st;
    Op op;

    SparseTable() {}

    SparseTable(const vector<T>& a, Op op = Op()) : op(op) {
        build(a);
    }

    void build(const vector<T>& a) {
        n = (int)a.size();

        lg.assign(n + 1, 0);
        for (int i = 2; i <= n; i++) {
            lg[i] = lg[i / 2] + 1;
        }

        int K = lg[n] + 1;
        st.assign(K, vector<T>(n));

        st[0] = a;

        for (int k = 1; k < K; k++) {
            int len  = 1 << k;
            int half = len >> 1;

            for (int i = 0; i + len <= n; i++) {
                st[k][i] = op(st[k - 1][i],
                              st[k - 1][i + half]);
            }
        }
    }

    // query the half-open interval [l, r) — must be non-empty.
    T query(int l, int r) const {
        int len   = r - l;
        int k     = lg[len];
        int block = 1 << k;

        return op(st[k][l], st[k][r - block]);
    }
};

struct MinOp { int operator()(int a, int b) const { return min(a, b); } };
struct MaxOp { int operator()(int a, int b) const { return max(a, b); } };
struct GcdOp { int operator()(int a, int b) const { return std::gcd(a, b); } };
half-open [l, r) templated T and Op no recursion · no I/O do not call query(l, l)

🔬 Line-by-Line Code Explanation

To make every choice in the template explicit, we walk through it block by block. For each block we name what the code does, why it is written that way, and what would go wrong if that decision were changed.

1. Template parameters

template <class T, class Op>
struct SparseTable

The struct is generic in the element type T and the merge operation Op. This is what lets the same struct serve int arrays, long long arrays, pair<int,int> arrays, custom Node structs, and any merge that is associative plus (for the classic table) idempotent. If we removed templating, every new operation would require a full rewrite of the struct.

2. Data members

int n;
vector<int> lg;
vector<vector<T>> st;
Op op;

The integer n is the array length, lg[i] stores ⌊log₂ i⌋, st[k][i] stores the answer for the block [i, i + 2ᵏ), and op is the user-supplied merge functor. Keeping lg as a vector lets us answer every query in pure integer arithmetic without ever touching floating point. Reading lg[0] would be meaningless, but we never do because the query length is always at least one.

3. Log table

lg.assign(n + 1, 0);
for (int i = 2; i <= n; i++)
    lg[i] = lg[i / 2] + 1;

This computes lg[i] = ⌊log₂ i⌋ using the recurrence lg[i] = lg[i/2] + 1, anchored by the manual values lg[0] = lg[1] = 0. Every query then needs k = ⌊log₂(r − l)⌋ instantly. If we replaced this loop with a call to log2(double) we would risk getting back something like 1.9999999 for the integer input 2 and rounding down to the wrong level — a real source of wrong-answer verdicts.

4. Number of levels

int K = lg[n] + 1;
st.assign(K, vector<T>(n));

The largest block length we will ever need is the largest power of two not exceeding n, which is 2^lg[n]. So we need levels 0, 1, …, lg[n], which is lg[n] + 1 levels in total. We allocate width n at every level even though some high-level cells will not be filled, because uniform sizing keeps indexing simple and the memory cost is well within budget. An off-by-one here would cause the build loop to skip the top level or write out of bounds.

5. Base level

st[0] = a;

Level zero stores intervals of length 2⁰ = 1, which means each cell is exactly one array element. Whole-vector assignment is the clearest possible statement of the base case and lets the compiler emit a single memcpy.

6. Climbing the levels

for (int k = 1; k < K; k++) {
    int len  = 1 << k;          // 2^k
    int half = len >> 1;        // 2^(k-1)

    for (int i = 0; i + len <= n; i++) {
        st[k][i] = op(st[k - 1][i],
                      st[k - 1][i + half]);
    }
}

For each level k ≥ 1 we combine two side-by-side blocks of length 2ᵏ⁻¹ into one block of length 2ᵏ. The bound i + len ≤ n ensures that the block [i, i + 2ᵏ) fits entirely inside the array. Loosening this bound would write past the boundary, corrupt neighbouring vectors, and produce wrong queries at the high end of the array.

7. Query

T query(int l, int r) const {
    int len   = r - l;
    int k     = lg[len];
    int block = 1 << k;
    return op(st[k][l], st[k][r - block]);
}

We compute the highest power of two not exceeding r − l, look up the two endpoint blocks of that length, and merge them. The two blocks together cover every index of [l, r) (this is Theorem 2 above), and overlap is harmless under idempotence. Calling query(l, l) would set len = 0, then lg[0] = 0, then block = 1, then r − block = l − 1, and we would read garbage — so we always guard l < r at the call site.

📄 2D Sparse Table

When the input is a matrix and queries are rectangles, the same idea generalises naturally. A 1D table indexes blocks by their starting index i and their length 2ᵏ. A 2D table indexes rectangles by their top-left corner (x, y) and their dimensions 2ᵏˣ × 2ᵏʸ. The query splits a rectangle into four overlapping power-of-two rectangles, one anchored at each corner, and then merges all four.

Storage

The cell st[kx][ky][x][y] stores the reduction of the rectangle whose top-left corner is (x, y), whose height is 2ᵏˣ, and whose width is 2ᵏʸ. The build proceeds in two independent dimensions: first we expand horizontally at kx = 0, then we expand vertically for every kx ≥ 1 using the already-built lower row level.

Query as four corners

For a query rectangle [x1, x2) × [y1, y2) with h = x2 − x1 and w = y2 − y1, we choose kx = ⌊log₂ h⌋ and ky = ⌊log₂ w⌋. Then four rectangles of size 2ᵏˣ × 2ᵏʸ, one in each corner of the query, together cover the entire query. They may overlap, but as in 1D the operation must be idempotent so that overlap is harmless.

#include <bits/stdc++.h>
using namespace std;

template <class T, class Op>
struct SparseTable2D {
    int n, m;
    vector<int> lgN, lgM;
    vector<vector<vector<vector<T>>>> st;
    Op op;

    SparseTable2D() {}

    SparseTable2D(const vector<vector<T>>& a, Op op = Op()) : op(op) {
        build(a);
    }

    void build(const vector<vector<T>>& a) {
        n = (int)a.size();
        m = (int)a[0].size();

        lgN.assign(n + 1, 0);
        lgM.assign(m + 1, 0);
        for (int i = 2; i <= n; i++) lgN[i] = lgN[i / 2] + 1;
        for (int j = 2; j <= m; j++) lgM[j] = lgM[j / 2] + 1;

        int KX = lgN[n] + 1;
        int KY = lgM[m] + 1;

        st.assign(KX, vector<vector<vector<T>>>(
                       KY, vector<vector<T>>(n, vector<T>(m))));

        st[0][0] = a;

        for (int kx = 0; kx < KX; kx++) {
            for (int ky = 0; ky < KY; ky++) {
                if (kx == 0 && ky == 0) continue;

                int height = 1 << kx;
                int width  = 1 << ky;

                if (kx == 0) {
                    int half_w = width >> 1;
                    for (int x = 0; x + height <= n; x++)
                        for (int y = 0; y + width <= m; y++)
                            st[kx][ky][x][y] =
                                op(st[kx][ky - 1][x][y],
                                   st[kx][ky - 1][x][y + half_w]);
                } else {
                    int half_h = height >> 1;
                    for (int x = 0; x + height <= n; x++)
                        for (int y = 0; y + width <= m; y++)
                            st[kx][ky][x][y] =
                                op(st[kx - 1][ky][x][y],
                                   st[kx - 1][ky][x + half_h][y]);
                }
            }
        }
    }

    // half-open rectangle [x1, x2) × [y1, y2)
    T query(int x1, int y1, int x2, int y2) const {
        int h = x2 - x1, w = y2 - y1;
        int kx = lgN[h],  ky = lgM[w];
        int H = 1 << kx, W = 1 << ky;

        T a = st[kx][ky][x1][y1];
        T b = st[kx][ky][x2 - H][y1];
        T c = st[kx][ky][x1][y2 - W];
        T d = st[kx][ky][x2 - H][y2 - W];
        return op(op(a, b), op(c, d));
    }
};

struct MaxOp { int operator()(int a, int b) const { return max(a, b); } };
struct MinOp { int operator()(int a, int b) const { return min(a, b); } };

Memory caution

The total number of stored cells is n · m · log n · log m. For n = m = 1000 this approaches 10⁸ integers — roughly 400 MB before any overhead. A 2D Sparse Table is powerful, but it must be sized carefully. When memory is tight, prefer 2D prefix sums (for sum) or a 2D segment tree (when updates are needed).

📄 Disjoint Sparse Table

The Disjoint Sparse Table removes the idempotence requirement. It still answers queries in O(1) and still builds in O(n log n), but its trick is to make the two queried pieces always disjoint, so the operation only has to be associative — never idempotent. Examples that need this structure are range sum, range product, matrix multiplication, function composition, and the maximum-subarray merge state used in problems like SPOJ GSS1.

The layout

At level k we conceptually cut the array into blocks of length 2ᵏ⁺¹. Each block has a left half (the first 2ᵏ positions) and a right half (the last 2ᵏ positions). Inside the left half we store, at every position i, the reduction of the suffix from i up to the midpoint inclusive. Inside the right half we store, at every position j, the reduction of the prefix from the midpoint up to j inclusive.

How the query splits

For a query [l, r] with l ≠ r, we look at the binary representations of l and r and find the most significant bit where they differ — call it k. At level k the indices l and r land in the same block of length 2ᵏ⁺¹, with l in the left half and r in the right half. We then glue together the stored suffix at l and the stored prefix at r. The two pieces are disjoint by construction. The MSB calculation is done with __builtin_clz on l XOR r, which is a single CPU instruction.

#include <bits/stdc++.h>
using namespace std;

template <class T, class Op>
struct DisjointSparseTable {
    int n;
    int K;
    vector<T> a;
    vector<vector<T>> table;
    Op op;

    DisjointSparseTable() {}

    DisjointSparseTable(const vector<T>& values, Op op = Op()) : op(op) {
        build(values);
    }

    void build(const vector<T>& values) {
        a = values;
        n = (int)a.size();

        K = 1;
        while ((1 << K) < n) K++;

        table.assign(K, vector<T>(n));

        for (int k = 0; k < K; k++) {
            int half = 1 << k;
            int len  = half << 1;

            for (int block_start = 0; block_start < n; block_start += len) {
                int mid       = min(block_start + half, n);
                int block_end = min(block_start + len,  n);

                if (mid == block_start) continue;

                // left half: suffixes ending at the midpoint
                table[k][mid - 1] = a[mid - 1];
                for (int i = mid - 2; i >= block_start; i--)
                    table[k][i] = op(a[i], table[k][i + 1]);

                // right half: prefixes starting at the midpoint
                if (mid < block_end) {
                    table[k][mid] = a[mid];
                    for (int i = mid + 1; i < block_end; i++)
                        table[k][i] = op(table[k][i - 1], a[i]);
                }
            }
        }
    }

    // half-open [l, r) — non-empty
    T query(int l, int r) const {
        if (r - l == 1) return a[l];

        int left  = l;
        int right = r - 1;
        int k = 31 - __builtin_clz(left ^ right);

        return op(table[k][left], table[k][right]);
    }
};
works for any associative op order matters: op(left, right) single-element query short-circuits to a[l] do not call with r == l

🧪 Edge Cases — Centralized Audit

Edge cases are where Sparse Table implementations quietly fail in contest. The list below collects every boundary the structure can encounter, why it deserves attention, and what the template above does about it.

Boundaries of the input array

When n = 1, the build still works correctly: the log table holds lg[1] = 0, the number of levels is K = 1, and only level zero is filled. Any call to query(0, 1) returns a[0] after a single trivial merge. Implementations that index lg from lg[2] instead of lg[1] must double-check this case.

Boundaries of the query

The most common boundary is query(l, l), which we never call because the structure assumes non-empty ranges. The next is query(l, l+1), which sets len = 1, k = lg[1] = 0, block = 1, and returns op(a[l], a[l]) — correct under idempotence and equal to a[l]. The full-array query query(0, n) uses level lg[n] and is always safe.

Power-of-two lengths

When the query length is exactly a power of two, the two endpoint blocks line up perfectly with no overlap. When the query length is just below a power of two (for example 2ᵏ − 1) the two blocks have length 2ᵏ⁻¹ and they overlap by 2ᵏ − 2 indices. When the query length is just above a power of two (for example 2ᵏ + 1) the two blocks have length 2ᵏ and they overlap by 2ᵏ − 1 indices. In all three cases the math works out — there is no special handling required.

Values

Duplicate values, negative values, and large values are all handled by the underlying integer types. For 64-bit data make sure the template parameter T is long long. For argmin-style queries make sure ties are broken deterministically (return the smaller index) so that the same query always produces the same index.

2D corners

A query rectangle with height or width equal to one is valid: the corresponding kx or ky is zero, and the four corner rectangles collapse to two or one effective rectangles, but all four lookups are still well-defined. A query rectangle with height or width zero is empty and must never be passed in.

Disjoint Sparse Table single-element queries

When the query is a single element, the l XOR r trick degenerates because l == r and the XOR is zero. The template short-circuits this case explicitly by returning a[l]. Removing that guard causes __builtin_clz(0) to be invoked, whose behaviour is undefined.

Quick reflex rules

When you read a problem and want to commit to a structure within a few seconds, run through these checks in order. First, are there any updates? If yes, the Sparse Table family is out. Second, what is the operation? If it is one of min, max, gcd, AND, or OR, jump to the classic Sparse Table. Third, if the operation is sum, prefer prefix sums and forget the table. Fourth, if the operation is custom and only associative, reach for the Disjoint Sparse Table. Fifth, if memory is tight, fall back to a segment tree or sqrt decomposition.

Common confusion note

Beginners often reach for a Sparse Table the moment they see the words range and query. The first instinct should instead be the operation itself. A Sparse Table is the right answer for static queries with an idempotent merge. If either condition fails, look at a different structure first and use Disjoint or 2D Sparse Table only when the problem genuinely fits.

Recognition triggers

Phrases like find the minimum in, find the gcd of, find the maximum value among in a problem that says the array is fixed almost always point to a Sparse Table. Phrases like find the sum on a fixed array point to prefix sums. Phrases that combine custom merges, like find the maximum sum of a contiguous subarray within, point to either a segment tree or a Disjoint Sparse Table depending on whether updates appear.

🔀 Variants and Extensions

The Sparse Table family supports a handful of small but useful extensions. Each one keeps the same outer interface and changes only the type or the comparison stored at each cell.

Argmin / Argmax — store indices, not values

When the problem asks for the position of the minimum (not its value), the table should store indices. We supply a tiny comparator that takes two indices and returns the one whose underlying value is smaller, with a deterministic tiebreaker on the index. This is how the histogram divide-and-conquer in Problem 5 works.

struct ArgMinOp {
    const vector<int>* a;
    int operator()(int i, int j) const {
        if ((*a)[i] != (*a)[j]) return ((*a)[i] < (*a)[j]) ? i : j;
        return min(i, j);
    }
};

Pair table — min and max in one structure

If the problem needs both the minimum and the maximum on every query (for example to compute their difference), we can store a small struct and reduce the two together. This halves the memory pressure compared with two separate tables.

struct MinMax {
    int mn, mx;
};
struct MinMaxOp {
    MinMax operator()(MinMax a, MinMax b) const {
        return { min(a.mn, b.mn), max(a.mx, b.mx) };
    }
};

Skip the log table with __builtin_clz

Instead of storing lg we can compute the highest bit of len directly: int k = 31 - __builtin_clz(len); for 32-bit lengths and 63 - __builtin_clzll(len) for 64-bit. The compiler emits a single CPU instruction. The catch is that calling these on zero is undefined behaviour, so the caller must still guarantee l < r.

Memory layout — st[i][k] versus st[k][i]

Both layouts are correct. We use st[k][i] in the family because the level is the outer dimension during teaching ("first all of level 1, then all of level 2") and because it lets us write st[0] = a; as a one-liner. Swapping the order can improve cache behaviour on very large inputs, at the cost of a slightly less didactic implementation.

Custom associative merges with Disjoint Sparse Table

The Disjoint Sparse Table accepts any operation that is associative. This includes matrix multiplication, function composition f ∘ g, and custom DP transitions where each "value" is a small struct containing several summary fields. The order of the merge must be preserved because such operations are usually non-commutative.

🚀 Application 1 — CSES Static Range Minimum Queries

Problem story. You are given an array of n integers and q queries. Each query is a pair (l, r) with 1-based inclusive indices, and you must report the minimum value of the subarray a[l..r]. There are no updates between queries.

Abstraction. This is a textbook static range-minimum problem. The operation is min, which is associative and idempotent, so the classic 1D Sparse Table is the perfect fit. The conversion from the 1-based inclusive input to the template's 0-based half-open interface is a simple shift.

Why it fits. Static array, no updates, idempotent operation, large query count — every box is ticked. The asymptotic profile is also exactly what the time limits demand.

Algorithm choice. Build a SparseTable<int, MinOp> once and answer each query in O(1) by calling st.query(l - 1, r) after the input shift.

#include <bits/stdc++.h>
using namespace std;

template <class T, class Op>
struct SparseTable {
    int n;
    vector<int> lg;
    vector<vector<T>> st;
    Op op;
    SparseTable() {}
    SparseTable(const vector<T>& a, Op op = Op()) : op(op) { build(a); }
    void build(const vector<T>& a) {
        n = (int)a.size();
        lg.assign(n + 1, 0);
        for (int i = 2; i <= n; i++) lg[i] = lg[i / 2] + 1;
        int K = lg[n] + 1;
        st.assign(K, vector<T>(n));
        st[0] = a;
        for (int k = 1; k < K; k++) {
            int len = 1 << k, half = len >> 1;
            for (int i = 0; i + len <= n; i++)
                st[k][i] = op(st[k - 1][i], st[k - 1][i + half]);
        }
    }
    T query(int l, int r) const {
        int len = r - l, k = lg[len], block = 1 << k;
        return op(st[k][l], st[k][r - block]);
    }
};
struct MinOp { int operator()(int a, int b) const { return min(a, b); } };

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, q;
    cin >> n >> q;

    vector<int> a(n);
    for (int i = 0; i < n; i++) cin >> a[i];

    SparseTable<int, MinOp> st(a);

    while (q--) {
        int l, r;
        cin >> l >> r;
        // CSES uses 1-based inclusive ranges.
        --l; --r;
        cout << st.query(l, r + 1) << '\n';
    }
    return 0;
}

Complexity. The build is O(n log n), each query is O(1), and the memory is O(n log n). With n, q ≤ 2·10⁵ this fits well within typical CSES limits.

Pitfalls specific to this problem. CSES uses 1-based inclusive ranges, so we must shift both endpoints down by one and then convert to half-open by querying r + 1 after the shift. Forgetting either step is a quiet wrong-answer source.

🚀 Application 3 — POJ-3264 Lineup

Abstraction. Two range queries combined: one minimum and one maximum. Both are idempotent, so the classic Sparse Table fits both, and we instantiate two independent tables on the same array.

🚀 Application 4 — UVA-11235 Frequent Values

Problem story. You are given a non-decreasing array. For each query [l, r] you must find the maximum frequency of any single value within that subarray.

Abstraction. A direct Sparse Table on values would not work — we are asked about frequencies, not values. But because the array is non-decreasing, equal values appear in contiguous blocks. We compress the array into runs of equal values, store the length of each run, and then a range-maximum Sparse Table on the run lengths handles the bulk of every query.

Why it fits. The query splits into three parts: a partial left run, a partial right run, and a number of full runs in the middle. The middle part is exactly the range maximum of run lengths, and the two partial pieces are computed by direct arithmetic on the block boundaries.

Algorithm choice. Compress the array into runs. Build a SparseTable<int, MaxOp> on the run lengths. For each query, identify the runs containing l and r, compute the lengths of the two partial pieces explicitly, and take the maximum of those two pieces with the table answer on the full middle runs (when any exist).

    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, q;
    while (cin >> n && n) {
        cin >> q;
        vector<int> a(n);
        for (int i = 0; i < n; i++) cin >> a[i];

        // Compress into runs of equal values (the array is non-decreasing).
        vector<int> block_id(n), block_left, block_right, block_freq;
        int id = -1;
        for (int i = 0; i < n; ) {
            int j = i;
            while (j < n && a[j] == a[i]) j++;
            ++id;
            block_left.push_back(i);
            block_right.push_back(j - 1);
            block_freq.push_back(j - i);
            for (int k = i; k < j; k++) block_id[k] = id;
            i = j;
        }

        SparseTable<int, MaxOp> st(block_freq);

        while (q--) {
            int l, r;
            cin >> l >> r;
            // UVA uses 1-based inclusive indices.
            --l; --r;

            int lb = block_id[l], rb = block_id[r];
            if (lb == rb) { cout << r - l + 1 << '\n'; continue; }

            int leftPart  = block_right[lb] - l + 1;
            int rightPart = r - block_left[rb] + 1;
            int ans = max(leftPart, rightPart);

            int firstFull = lb + 1, lastFull = rb - 1;
            if (firstFull <= lastFull)
                ans = max(ans, st.query(firstFull, lastFull + 1));

            cout << ans << '\n';
        }
    }
    return 0;
}

Pitfalls specific to this problem. The technique relies on the array being non-decreasing. If equal values were scattered, the block compression would be invalid and the technique would fail. We also must guard against an empty middle range, since firstFull > lastFull happens whenever l and r lie in adjacent blocks.

🚀 Application 5 — POJ-2559 Largest Rectangle in a Histogram (via RMQ)

Problem story. You are given the bar heights of a histogram and must find the largest axis-aligned rectangle that fits inside it.

Abstraction. For any interval [l, r), let m be the index of the minimum height in that interval. The best rectangle inside [l, r) is one of three options: a rectangle that spans the whole interval at height h[m], the best rectangle in the left subinterval [l, m), or the best rectangle in the right subinterval [m + 1, r). This is a divide-and-conquer on the minimum position. A Sparse Table that stores indices of minimum gives the position in O(1) per query, making the whole algorithm O(n log n).

Why it fits. The argmin operation is idempotent (with a deterministic tiebreak by index), so the classic Sparse Table works perfectly. The technique is not the simplest solution to histograms — a monotonic stack reaches O(n) — but it is a beautiful demonstration of how RMQ can guide divide-and-conquer.

Algorithm choice. Store an array of indices [0, 1, …, n-1] and build a Sparse Table whose operation returns the index of the smaller-value element. Run an iterative divide-and-conquer (a stack of subintervals) to avoid recursion overflow on adversarial inputs.

    return 0;
}

Complexity. The table build is O(n log n), each argmin query is O(1), and the divide-and-conquer processes O(n) subintervals, giving O(n log n) in total. Memory is O(n log n).

Pitfalls specific to this problem. The Sparse Table must store indices, not values, so that we can locate the minimum's position. Ties in heights must be broken deterministically by index, otherwise the divide-and-conquer can revisit the same subinterval and recurse forever. A recursive implementation is also a common source of stack overflow on sorted input — the iterative version above sidesteps that.

🚀 Application 6 — LightOJ-1081 A Easy Game (2D Rectangle Max)

Problem story. You are given a square matrix. Each query consists of a top-left corner and a square size, and you must report the maximum value inside that square.

Abstraction. A static 2D rectangle maximum query. The operation max is associative and idempotent, so a 2D Sparse Table is the perfect match.

Why it fits. The matrix never changes. Many queries arrive. The operation is idempotent. The 2D Sparse Table answers in O(1) per query after an O(n² log² n) build.

Algorithm choice. Build a SparseTable2D<int, MaxOp> on the matrix. For each query, convert the 1-based corner and the size into a half-open rectangle and call query(x, y, x + s, y + s).

Full AC-ready code click to expand 
#include <bits/stdc++.h>
using namespace std;

template <class T, class Op>
struct SparseTable2D {
    int n, m;
    vector<int> lgN, lgM;
    vector<vector<vector<vector<T>>>> st;
    Op op;
    SparseTable2D() {}
    SparseTable2D(const vector<vector<T>>& a, Op op = Op()) : op(op) { build(a); }

    void build(const vector<vector<T>>& a) {
        n = (int)a.size();
        m = (int)a[0].size();
        lgN.assign(n + 1, 0);
        lgM.assign(m + 1, 0);
        for (int i = 2; i <= n; i++) lgN[i] = lgN[i / 2] + 1;
        for (int j = 2; j <= m; j++) lgM[j] = lgM[j / 2] + 1;
        int KX = lgN[n] + 1, KY = lgM[m] + 1;
        st.assign(KX, vector<vector<vector<T>>>(
                       KY, vector<vector<T>>(n, vector<T>(m))));
        st[0][0] = a;
        for (int kx = 0; kx < KX; kx++) {
            for (int ky = 0; ky < KY; ky++) {
                if (kx == 0 && ky == 0) continue;
                int H = 1 << kx, W = 1 << ky;
                if (kx == 0) {
                    int hw = W >> 1;
                    for (int x = 0; x + H <= n; x++)
                        for (int y = 0; y + W <= m; y++)
                            st[kx][ky][x][y] =
                                op(st[kx][ky - 1][x][y],
                                   st[kx][ky - 1][x][y + hw]);
                } else {
                    int hh = H >> 1;
                    for (int x = 0; x + H <= n; x++)
                        for (int y = 0; y + W <= m; y++)
                            st[kx][ky][x][y] =
                                op(st[kx - 1][ky][x][y],
                                   st[kx - 1][ky][x + hh][y]);
                }
            }
        }
    }

    T query(int x1, int y1, int x2, int y2) const {
        int h = x2 - x1, w = y2 - y1;
        int kx = lgN[h], ky = lgM[w];
        int H = 1 << kx, W = 1 << ky;
        T a = st[kx][ky][x1][y1];
        T b = st[kx][ky][x2 - H][y1];
        T c = st[kx][ky][x1][y2 - W];
        T d = st[kx][ky][x2 - H][y2 - W];
        return op(op(a, b), op(c, d));
    }
};
struct MaxOp { int operator()(int a, int b) const { return max(a, b); } };

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int T;
    cin >> T;
    for (int tc = 1; tc <= T; tc++) {
        int n, q;
        cin >> n >> q;
        vector<vector<int>> mat(n, vector<int>(n));
        for (int i = 0; i < n; i++)
            for (int j = 0; j < n; j++)
                cin >> mat[i][j];

        SparseTable2D<int, MaxOp> st(mat);

        cout << "Case " << tc << ":\n";
        while (q--) {
            int x, y, s;
            cin >> x >> y >> s;
            --x; --y;     // input is 1-based
            cout << st.query(x, y, x + s, y + s) << '\n';
        }
    }
    return 0;
}

Complexity. Build is O(n² log² n), each query is O(1), and memory is O(n² log² n). The memory budget is the binding constraint here; in practice n ≤ 500 is safe.

Pitfalls specific to this problem. The biggest trap is memory limit exceeded — for larger n the table grows beyond what is reasonable. The other classic mistake is reusing a 2D Sparse Table for rectangle sums, where the four corner blocks overlap and double-count.

🚀 Application 7 — SPOJ-GSS1 Maximum Subarray Sum (Disjoint Sparse Table)

Problem story. You are given an array and asked, for each query [l, r], for the maximum sum of any contiguous subarray that lies inside [l, r]. There are no updates.

Abstraction. The merge state at each node is a small struct holding the total sum, the best prefix sum, the best suffix sum, and the best inner subarray sum. Merging two adjacent states is associative — combining three pieces from left to right gives the same result regardless of grouping — but it is not idempotent, because merging a state with itself doubles the totals. The classic Sparse Table would silently produce wrong answers on overlapping queries, so we use a Disjoint Sparse Table.

Why it fits. The array is static, queries are numerous, and the merge is associative-but-not-idempotent. This is exactly the niche the Disjoint Sparse Table fills.

Algorithm choice. Define a Node with sum, pref, suff, and best. Define an operation that combines two nodes preserving left-to-right order. Wrap it in a DisjointSparseTable<Node, MaxSubarrayOp> and answer each query by returning the best field of the result.

Full AC-ready code click to expand 
#include <bits/stdc++.h>
using namespace std;

struct Node {
    long long sum, pref, suff, best;
};
struct MaxSubarrayOp {
    Node operator()(const Node& L, const Node& R) const {
        Node res;
        res.sum  = L.sum + R.sum;
        res.pref = max(L.pref, L.sum + R.pref);
        res.suff = max(R.suff, R.sum + L.suff);
        res.best = max({ L.best, R.best, L.suff + R.pref });
        return res;
    }
};

template <class T, class Op>
struct DisjointSparseTable {
    int n, K;
    vector<T> a;
    vector<vector<T>> table;
    Op op;
    DisjointSparseTable() {}
    DisjointSparseTable(const vector<T>& v, Op op = Op()) : op(op) { build(v); }

    void build(const vector<T>& v) {
        a = v;
        n = (int)a.size();
        K = 1;
        while ((1 << K) < n) K++;
        table.assign(K, vector<T>(n));

        for (int k = 0; k < K; k++) {
            int half = 1 << k, len = half << 1;
            for (int bs = 0; bs < n; bs += len) {
                int mid = min(bs + half, n);
                int be  = min(bs + len,  n);
                if (mid == bs) continue;
                table[k][mid - 1] = a[mid - 1];
                for (int i = mid - 2; i >= bs; i--)
                    table[k][i] = op(a[i], table[k][i + 1]);
                if (mid < be) {
                    table[k][mid] = a[mid];
                    for (int i = mid + 1; i < be; i++)
                        table[k][i] = op(table[k][i - 1], a[i]);
                }
            }
        }
    }

    T query(int l, int r) const {
        if (r - l == 1) return a[l];
        int left = l, right = r - 1;
        int k = 31 - __builtin_clz(left ^ right);
        return op(table[k][left], table[k][right]);
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    cin >> n;
    vector<Node> a(n);
    for (int i = 0; i < n; i++) {
        long long x;
        cin >> x;
        a[i] = { x, x, x, x };
    }

    DisjointSparseTable<Node, MaxSubarrayOp> dst(a);

    int q;
    cin >> q;
    while (q--) {
        int l, r;
        cin >> l >> r;
        // SPOJ GSS1 uses 1-based inclusive queries.
        --l; --r;
        Node ans = dst.query(l, r + 1);
        cout << ans.best << '\n';
    }
    return 0;
}

Complexity. Build is O(n log n) with a small constant per merge, each query is O(1), and memory is O(n log n).

Pitfalls specific to this problem. Order matters: the merge is not commutative, so we must always merge the left piece into the right piece, never the other way around. A single-element range must return the original element directly. And using a classic Sparse Table here is the classic mistake — it gives wrong answers as soon as the two query blocks share even one index.

19. Final Summary — Cheat Sheet

At the end of a long tutorial it helps to compress the whole landscape into one page that a contestant can re-read in two minutes before a round. The three structures we built share a single mental skeleton — preprocess answers for blocks whose lengths are powers of two, then answer any query in O(1) by gluing together a tiny constant number of those blocks — but they differ in how they glue, and the differences are dictated entirely by what kind of operation the problem asks about.

The three structures at a glance

Structure Storage Query mechanism Op contract Build · Query · Memory
Classic 1D Sparse Table st[k][i] = answer for [i, i + 2ᵏ) Two overlapping power-of-two blocks anchored at l and r Associative and idempotent O(n log n) · O(1) · O(n log n)
2D Sparse Table st[kx][ky][x][y] = answer for [x, x+2ᵏˣ) × [y, y+2ᵏʸ) Four overlapping power-of-two rectangles, one anchored at each corner Associative and idempotent O(nm log n log m) · O(1) · O(nm log n log m)
Disjoint Sparse Table table[k][i] = suffix (left half) or prefix (right half) inside a level-k block One suffix at l + one prefix at r−1, level chosen by MSB(l XOR (r−1)) Associative only — idempotence is not required O(n log n) · O(1) · O(n log n)

The decision in one paragraph

If the array is fixed and the operation is one of min, max, gcd, bitwise AND, or bitwise OR, reach for the classic 1D Sparse Table. If the same kind of operation is asked over rectangles of a fixed matrix, switch to the 2D Sparse Table — but first do a memory check, because n · m · log n · log m grows alarmingly fast. If the operation is custom — a struct merge, a maximum-subarray combine, matrix multiplication, function composition, anything that is associative but where op(x, x) ≠ x — switch to the Disjoint Sparse Table. If the operation is plain sum on a fixed array, do not use any of these at all: a prefix sum array is shorter, faster, and uses linear memory. And if there are any updates, the entire Sparse Table family is out — segment trees and Fenwick trees own that regime.

Golden recipe — classic 1D Sparse Table

Build a log table lg[1..n] using the recurrence lg[i] = lg[i/2] + 1. Allocate st[K][n] where K = lg[n] + 1, copy the input into st[0], and fill higher levels with the relation st[k][i] = op(st[k-1][i], st[k-1][i + 2ᵏ⁻¹]). To answer a half-open query [l, r), set k = lg[r − l] and return op(st[k][l], st[k][r − 2ᵏ]). The whole query is one log-table lookup, two table reads, one merge, and absolutely nothing else.

Golden recipe — Disjoint Sparse Table

At each level k, conceptually cut the array into blocks of length 2ᵏ⁺¹. Inside each block, fill the left half from right to left with suffix reductions ending at the midpoint, and fill the right half from left to right with prefix reductions starting at the midpoint. To answer a half-open query [l, r), set k = ⌊log₂(l XOR (r − 1))⌋ (computed in one CPU instruction by 31 − __builtin_clz), and return op(table[k][l], table[k][r − 1]). A single-element query bypasses the table and returns a[l] directly.

The three pitfalls that bite even experienced contestants

The first is using a classic Sparse Table for sum: the two query blocks overlap, sum is not idempotent, and the answer is silently wrong on any non-power-of-two length. The fix is either a prefix sum or a Disjoint Sparse Table. The second is forgetting that the template uses half-open intervals: inputs that hand you inclusive [l, r] must be converted to query(l, r + 1), otherwise off-by-one errors corrupt every query. The third is memory in 2D: the four-dimensional table grows as the product of two logs times the matrix area, and a 1000 × 1000 matrix already pushes hundreds of megabytes — many problems that look like a 2D Sparse Table fit are actually 2D prefix sums in disguise.

The Sparse Table family is a beautiful example of how a single idea — answers for power-of-two intervals — can be reshaped to answer many different question types. Once the mental model is locked in, recognising which variant a problem needs becomes almost instant, and the implementation collapses to a short, reusable template you can paste into any contest.
