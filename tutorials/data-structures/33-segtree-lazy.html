Lesson 4 — Lazy Propagation

Range Add · Range Set · Range Sum · Push-Before-Recurse · O(log n)

🧩 Problem Definition

An array a of size n needs to support three operations, each in O(log n):

1. Range Add

For every i in [l, r], set a[i] += x.

2. Range Set

For every i in [l, r], set a[i] = x.

3. Range Sum

Return a[l] + a[l+1] + ... + a[r].

Without lazy propagation, range updates that touch k elements take O(k) per operation, and with k potentially as large as n, the total cost can blow up to O(n · q). Lazy propagation gives us O(log n) per operation by deferring the work for nodes the update fully covers — exactly the trick that makes range updates as cheap as point updates.

💡 Intuition from Scratch

Imagine you owe ten people each one dollar. Instead of paying them one at a time, you write a single sticky note saying "owe one dollar to each of these ten people, will pay later" and put it on the group. As long as nobody asks for their money individually, the sticky note is a perfect summary; you don't have to touch each person. When someone does ask, you peel the note off the group, give one dollar to each person under it, and now their personal accounts are correct again.

That's lazy propagation. The "people" are the children of a segment tree node. The "sticky note" is a lazy tag attached to the parent. The "asking for money individually" is a future query or update that needs to recurse past the parent into its children — and that's the only moment we must push the tag down, paying the deferred work.

🌳 Visual Mental Model

Each node now stores not one number but three pieces of state:

sum[i] ← the correct total for interval [l, r], including all pending tags that fully cover this node. add_lazy[i] ← pending add that the children still need to apply. set_lazy[i] ← pending set that the children still need to apply, has_set[i] and whether such a set is actually pending.

The crucial point: the node itself is always correct. The lazy tag is owed to the children, not to the node. So you can always read sum[i] immediately when the node is fully inside the query, without pushing anything.

You only push when you must recurse past i into one of its children — because then the children themselves need to be correct, and the pending tag needs to be transferred so the recursion sees up-to-date subtrees.

📐 Formal Abstraction

Define three primitives that every lazy segment tree implements:

apply(i, l, r, op)

Apply operation op directly to node i as if its entire interval [l, r] were fully covered. Updates sum[i] and merges the new op into the lazy state of i.

push(i, l, r)

Transfer the pending lazy state from i to its two children using apply, then clear the state at i. Called before recursing into children for a partial overlap.

pull(i)

Recompute sum[i] from sum[left] + sum[right] after the children may have changed. Called after returning from recursion.

The recursive update and recursive query both follow the same skeleton: check none → check full → otherwise push, recurse into both children, then pull. That's the entire pattern.

🔬 Key Observations

Three observations that distinguish lazy propagation from naive range updates.

Observation 1 (the apply trick). If a node is fully covered by an update, we can update its sum in O(1) using only the interval length and the operation value. For range add, sum += x · length. For range set, sum = x · length. No recursion needed.

Observation 2 (set beats add). When a set is pending and we get a new add, the add applies after the set. So we can fold the add into the pending set value: set_lazy += x. But when a set arrives and there was a pending add, the add becomes irrelevant — the set overwrites everything. We zero the add immediately. Order matters.

Observation 3 (push is one-way). A push hands the parent's pending state to the children using apply, and clears it at the parent. After a push, the parent itself is still correct (sum[i] doesn't change), but the children are now "newer", carrying any tags the parent had.

⚙️ Algorithm

The three public operations all follow the same template once the primitives are in place.

range_add(ql, qr, x)

Recurse from the root. If the current node has no overlap with [ql, qr], return. If it is fully inside, call apply_add and return. Otherwise, push the pending tag down, recurse into both children, then pull.

range_set(ql, qr, x)

Identical structure with apply_set.

range_sum(ql, qr)

Identical structure. Read sum[i] directly when fully inside. Otherwise push and recurse — even a query must push, because the query has to look at the children's true values when only partially overlapping.

🛡️ Invariants

✅ Invariant — Value invariant

sum[i] is correct for the entire interval of node i, including all updates that fully covered this node. The lazy tag describes what still has to be propagated to children — not what's missing from sum[i].

✅ Invariant — Lazy invariant

If has_set[i] is true, then after the pending set is pushed, the entire interval of i should become exactly set_lazy[i] (and any pending add should fold into it). If has_set[i] is false and add_lazy[i] ≠ 0, then the children still need a +add_lazy[i].

✅ Invariant — Push invariant (before recurse)

Before recursing into either child of i due to a partial overlap, the function must call push(i, l, r) so the children carry the deferred work and are once again correct snapshots.

✅ Invariant — Pull invariant (after recurse)

After returning from a partial-overlap recursion into both children, the function must call pull(i) to refresh sum[i] from the children's now-current values.

🧮 Proof of Correctness

🧮 Proof — Why lazy propagation is correct and O(log n)

Correctness. By the Value invariant, every fully-covered node already carries the correct sum. The Push invariant guarantees that whenever we descend past a node with a pending tag, the children become correct before the recursion observes them. The Pull invariant guarantees that we never leave a parent stale after its children may have changed. Together these maintain the Value invariant after every operation — so the answer reported by range_sum is exactly the sum of the underlying conceptual array.

Complexity. Each operation visits O(log n) nodes for the same reason as Lesson 1: full overlaps are answered immediately, no-overlaps return immediately, and the only nodes that recurse into both children are the O(log n) "boundary" nodes that straddle the endpoints of the query range. push and pull are O(1), so the total cost is O(log n) per operation. ∎

🧰 Core Template Philosophy

A lazy segment tree is most cleanly written by isolating three primitives:

apply(i, l, r, op) applies an operation directly to a node. It changes sum[i] and merges op into the node's lazy state. This is the only place where the meaning of the operation lives. Everything else is plumbing.

push(i, l, r) delegates the pending lazy state to children via two apply calls and clears the state at i. The two apply calls handle their own composition — they do not assume the child was clean.

pull(i) rebuilds sum[i] from children. Trivial for sum (sum[i] = sum[left] + sum[right]), but for composite nodes (next lesson's flavor of lazy) it becomes tree[i] = merge_nodes(tree[left], tree[right]).

Once those three primitives are correct, the public operations are mechanical: check none / full / push-recurse-pull.

💻 Master Template (C++)

#include <bits/stdc++.h>
using namespace std;

struct LazySetAddSumSegTree {
    using T = long long;

    int n = 0;
    int base = 1;

    vector<T> sum;
    vector<T> add_lazy;
    vector<T> set_lazy;
    vector<char> has_set;

    LazySetAddSumSegTree() {}

    LazySetAddSumSegTree(const vector<T>& a) {
        init(a);
    }

    void init(const vector<T>& a) {
        n = (int)a.size();

        base = 1;
        while (base < max(1, n)) {
            base <<= 1;
        }

        sum.assign(2 * base, 0);
        add_lazy.assign(2 * base, 0);
        set_lazy.assign(2 * base, 0);
        has_set.assign(2 * base, false);

        build(0, 0, base - 1, a);
    }

    void build(int i, int l, int r, const vector<T>& a) {
        if (l == r) {
            if (l < n) {
                sum[i] = a[l];
            }
            return;
        }

        int left = 2 * i + 1;
        int right = left + 1;
        int mid = l + (r - l) / 2;

        build(left, l, mid, a);
        build(right, mid + 1, r, a);

        pull(i);
    }

    void pull(int i) {
        int left = 2 * i + 1;
        int right = left + 1;
        sum[i] = sum[left] + sum[right];
    }

    void apply_set(int i, int l, int r, T value) {
        sum[i] = value * (r - l + 1);
        has_set[i] = true;
        set_lazy[i] = value;
        add_lazy[i] = 0;
    }

    void apply_add(int i, int l, int r, T value) {
        sum[i] += value * (r - l + 1);

        if (has_set[i]) {
            set_lazy[i] += value;
        } else {
            add_lazy[i] += value;
        }
    }

    void push(int i, int l, int r) {
        if (l == r) {
            has_set[i] = false;
            add_lazy[i] = 0;
            return;
        }

        int left = 2 * i + 1;
        int right = left + 1;
        int mid = l + (r - l) / 2;

        if (has_set[i]) {
            apply_set(left, l, mid, set_lazy[i]);
            apply_set(right, mid + 1, r, set_lazy[i]);
            has_set[i] = false;
        }

        if (add_lazy[i] != 0) {
            apply_add(left, l, mid, add_lazy[i]);
            apply_add(right, mid + 1, r, add_lazy[i]);
            add_lazy[i] = 0;
        }
    }

    void range_set(int ql, int qr, T value) {
        range_set(ql, qr, value, 0, 0, base - 1);
    }

    void range_set(int ql, int qr, T value, int i, int l, int r) {
        if (qr < l || r < ql) {
            return;
        }

        if (ql <= l && r <= qr) {
            apply_set(i, l, r, value);
            return;
        }

        push(i, l, r);

        int left = 2 * i + 1;
        int right = left + 1;
        int mid = l + (r - l) / 2;

        range_set(ql, qr, value, left, l, mid);
        range_set(ql, qr, value, right, mid + 1, r);

        pull(i);
    }

    void range_add(int ql, int qr, T value) {
        range_add(ql, qr, value, 0, 0, base - 1);
    }

    void range_add(int ql, int qr, T value, int i, int l, int r) {
        if (qr < l || r < ql) {
            return;
        }

        if (ql <= l && r <= qr) {
            apply_add(i, l, r, value);
            return;
        }

        push(i, l, r);

        int left = 2 * i + 1;
        int right = left + 1;
        int mid = l + (r - l) / 2;

        range_add(ql, qr, value, left, l, mid);
        range_add(ql, qr, value, right, mid + 1, r);

        pull(i);
    }

    T range_sum(int ql, int qr) {
        return range_sum(ql, qr, 0, 0, base - 1);
    }

    T range_sum(int ql, int qr, int i, int l, int r) {
        if (qr < l || r < ql) {
            return 0;
        }

        if (ql <= l && r <= qr) {
            return sum[i];
        }

        push(i, l, r);

        int left = 2 * i + 1;
        int right = left + 1;
        int mid = l + (r - l) / 2;

        return range_sum(ql, qr, left, l, mid)
             + range_sum(ql, qr, right, mid + 1, r);
    }
};

🔎 Line-by-Line Explanation

apply_set(i, l, r, value) click to expand/collapse

This is the operation in its purest form: the node now becomes a block of (r − l + 1) copies of value. sum[i] is updated immediately to value · (r − l + 1). The lazy state is replaced wholesale: has_set = true, set_lazy = value, and add_lazy = 0 — because any previously-pending add is now meaningless, the entire interval is being overwritten.

apply_add(i, l, r, value) click to expand/collapse

Adding value to every element of the interval changes the sum by value · length. The interesting part is what we do with the lazy state. If a set is already pending, then conceptually the interval is currently about to become some target value set_lazy[i], and then the add will follow — so we can fold the add into the set value: set_lazy[i] += value. Otherwise, there is just a pending add, and we accumulate into it.

push(i, l, r) click to expand/collapse

Push is a one-way transfer. If a set is pending, both children receive apply_set with set_lazy[i]; this also clears any add the children might have had, which is correct because the set defines an entirely new state. After the set is pushed, we clear has_set[i]. Then if an add is pending, both children receive apply_add with add_lazy[i], and we clear it. At a leaf (l == r) there are no children, so we just clear the state.

range_set / range_add / range_sum click to expand/collapse

All three follow the universal lazy template: check no-overlap (return identity), check full-overlap (call apply or return sum[i]), otherwise push, recurse into both children, then pull. The only differences are which apply they call and how the result is combined. Note that the query also has to push — if it didn't, partial overlap would observe stale children.

⚠️ Common Pitfalls

⚠️ Pitfall — Forgetting that set destroys add

If you write apply_set as a simple set without zeroing add_lazy, then a sequence +5 then =9 will incorrectly produce 14 on push — the set was supposed to overwrite the add. Always set add_lazy[i] = 0 at the end of apply_set.

⚠️ Pitfall — Pushing after recursing

The push must come before recursing into children, not after. If you push after, the children processed by the recursion saw stale data. This is the most common silent bug in lazy propagation.

⚠️ Pitfall — Forgetting to pull after a partial update

After a partial-overlap recursion, the children may have changed but the parent's sum still reflects the old children. You must call pull to recompute it. Forgetting this gives correct answers only if you happen never to query a node whose children just changed underneath it — a fragile property that breaks at random.

⚠️ Pitfall — Forgetting to multiply by length in apply_add / apply_set

The sum of (r − l + 1) copies of x is x · (r − l + 1), not just x. A common typo is sum[i] += value; instead of sum[i] += value · (r − l + 1); — only correct when the node is a single leaf.

⚠️ Pitfall — Using int when values can overflow

With range add of x = 10⁹ over n = 10⁵ elements, the sum can hit 10¹⁴. Always use long long for sum, add_lazy, and set_lazy when values are large.

Quick reflex

If you see "add x to every element from l to r" together with "what's the sum from l' to r'" — that's the lazy lesson, no need to think further.

🎛️ Variants & Extensions

Variant 1 — Add-only lazy (range add + range sum)

Drop set_lazy and has_set entirely. The only primitive is apply_add. Simplest and fastest of all lazy variants.

Variant 2 — Set-only lazy

Drop add_lazy. apply_set sets, push pushes a set if pending.

Variant 3 — Range flip (XOR with 1) + count of ones

Each node stores cnt (number of ones in its interval). apply_flip(i, l, r) sets cnt = (r − l + 1) − cnt and toggles a boolean flip_lazy[i]. push propagates flips by applying flip to both children.

Variant 4 — Range affine (mul · x + add) + range sum

Lazy state is a pair (b, c) representing x ↦ b·x + c. Composition is well-known function composition: (b1, c1) ∘ (b2, c2) = (b1·b2, b1·c2 + c1). Applying to a node updates sum = b · sum + c · length.

Variant 5 — Range assign + range min

Nodes store min. apply_set sets min = value. No length multiplication needed because min doesn't aggregate by count.

🏁 Final Summary

Lazy propagation extends the segment tree to range updates without losing the O(log n) guarantee. The trick is to defer work on fully-covered nodes by storing a lazy tag, and push that tag down to children only when a future operation needs to recurse past the parent.

The mental model — sum is always correct on the node; the tag describes what's owed to the children — is the most important single idea in this lesson. The three primitives apply, push, pull express this idea in code. The set-vs-add composition rule (set destroys pending add; add folds into pending set) is the most error-prone detail; trace it twice when you write your own template.

With this base in place, the next harder lazy tasks — affine updates, chmin/chmax/add (beats), persistent lazy, 2D rectangles — all become variations on the same three primitives.
