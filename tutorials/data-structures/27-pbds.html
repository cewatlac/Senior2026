PBDS: Zero to Hero

GCC's secret data-structure library — the one that gives you indexed sets and unhackable hash maps.
📘 Beginner-friendly ⚡ ICPC weapon

Part 0 · Why PBDS Exists

You're doing a problem and need to answer:

  • "How many elements in my set are less than x?"
  • "What's the 5th smallest element in my set?"

With a regular std::set you can't do either in O(log n). You'd have to scan, or layer a Fenwick tree on top, or write a balanced BST by hand.

Or — you add two lines:

#include <ext/pb_ds/assoc_container.hpp>
using namespace __gnu_pbds;

And suddenly you have ordered_set: a balanced tree that answers both questions in O(log n).

💡
Analogy Imagine your std::set grew two new superpowers overnight: a built-in counter that tells you how many elements are smaller than any value, and an indexer that lets you jump to the k-th element instantly. That's PBDS.

What PBDS gives you

⚠️
GCC-only. PBDS lives in GCC's __gnu_pbds namespace. It works on Codeforces, AtCoder, and any judge using GCC/Clang. It does NOT work on MSVC. Don't use it for production code.

#include <ext/pb_ds/assoc_container.hpp>
#include <ext/pb_ds/tree_policy.hpp>

using namespace __gnu_pbds;

int main() {
    ordered_set<int> s;
    s.insert(5);
    s.insert(2);
    s.insert(8);
    s.insert(3);
    s.insert(9);
    // Sorted: {2, 3, 5, 8, 9}

    cout << *s.find_by_order(0) << '\n';   // 2  (smallest)
    cout << *s.find_by_order(2) << '\n';   // 5  (3rd smallest)

    cout << s.order_of_key(5) << '\n';    // 2  (number of elements < 5)
    cout << s.order_of_key(7) << '\n';    // 3  (2,3,5 are < 7)
}

Read Parts 0–2 first — that's the foundation everyone uses. Part 3 (multiset trick) is essential for sliding-window-of-distinct-values problems. Parts 5–7 (hash, mutable PQ, trie) are situational power tools.

Part 1 · ordered_set — Set with Order Statistics

The full type breakdown

tree<
    Key,                                  // type stored
    null_type,                            // "no mapped value" — this is a SET, not a MAP
    less<Key>,                            // comparator (use greater<Key> for descending)
    rb_tree_tag,                          // red-black tree (other: splay_tree_tag)
    tree_order_statistics_node_update     // the magic that gives find_by_order/order_of_key
>

The standard alias

Type all of that once and forget it. From now on use ordered_set<T>:

template<class T>
using ordered_set =
    tree<T, null_type, less<T>, rb_tree_tag, tree_order_statistics_node_update>;

// Descending variant
template<class T>
using ordered_set_desc =
    tree<T, null_type, greater<T>, rb_tree_tag, tree_order_statistics_node_update>;

The two superpowers

OperationWhat it doesComplexity
s.find_by_order(k)Iterator to the k-th smallest element (0-indexed). Returns end() if k ≥ size.O(log n)
s.order_of_key(x)Number of elements strictly less than x. (x doesn't have to be in the set.)O(log n)
ordered_set<int> s = {10, 20, 30, 40, 50};

// Indexing into a sorted "array"
cout << *s.find_by_order(0);   // 10  (1st smallest)
cout << *s.find_by_order(3);   // 40  (4th smallest)
if (s.find_by_order(5) == s.end()) cout << "out of range";

// Counting
cout << s.order_of_key(25);    // 2  (10 and 20 are < 25)
cout << s.order_of_key(10);    // 0  (nothing is < 10)
cout << s.order_of_key(99);    // 5  (everyone is < 99)

All other operations (inherited from std::set)

ordered_set<int> s;

s.insert(5);                                  // O(log n)
s.erase(5);                                   // erase by value
s.erase(s.find(5));                          // erase by iterator

auto it = s.find(5);                          // O(log n)
if (it != s.end()) cout << *it;

cout << s.size();
cout << s.empty();

// lower_bound / upper_bound
auto lo = s.lower_bound(10);
auto hi = s.upper_bound(20);

// Iterate in sorted order
for (int x : s) cout << x << ' ';

Killer use #1 — count elements in a value range

ordered_set<int> s = {3, 7, 12, 15, 20, 25, 30};

// How many elements are in [L, R]?
int L = 10, R = 22;
int cnt = s.order_of_key(R + 1) - s.order_of_key(L);
// cnt = 3  (12, 15, 20 are in [10, 22])

Killer use #2 — counting inversions in O(n log n)

// Count pairs (i, j) with i < j and a[i] > a[j]
long long countInversions(vector<int>& a) {
    ordered_set<int> s;
    long long inv = 0;
    for (int i = (int)a.size() - 1; i >= 0; --i) {
        inv += s.order_of_key(a[i]);   // how many already-seen values are < a[i]
        s.insert(a[i]);
    }
    return inv;
}

Part 2 · ordered_map — Map with Order Statistics

Replace null_type with the value type, and you've got the map version.

template<class Key, class Value>
using ordered_map =
    tree<Key, Value, less<Key>, rb_tree_tag, tree_order_statistics_node_update>;

// Use it like std::map
ordered_map<string, int> m;
m["alice"] = 30;
m["bob"]   = 25;
m["carol"] = 28;

cout << m["bob"];               // 25

// Order statistics work on KEYS
cout << m.find_by_order(0)->first;   // "alice" (lex-smallest key)
cout << m.order_of_key("bob");        // 1 (only "alice" is < "bob")

Use case: leaderboard with rank lookup

// score → player_name. Sorted by score.
ordered_map<int, string> board;

board[95] = "alice";
board[88] = "bob";
board[72] = "carol";

// What's the score of the 2nd-lowest?
cout << board.find_by_order(1)->first;   // 88

// How many players have score < 90?
cout << board.order_of_key(90);          // 2

Part 3 · ordered_multiset (the pair-with-counter trick)

⚠️
PBDS doesn't have an ordered_multiset out of the box. The tree template requires uniqueness — if you try to insert a duplicate it silently fails. The trick is to make every key artificially unique by pairing it with a counter or timestamp.

The trick

template<class T>
using ordered_multiset =
    tree<pair<T, int>, null_type, less<pair<T, int>>,
         rb_tree_tag, tree_order_statistics_node_update>;

Each insert gets a unique tie-breaker integer. Pairs are compared lexicographically — same value with different tie-breakers count as distinct nodes, but they sort together.

ordered_multiset<int> ms;
int tick = 0;                         // monotonically increasing tie-breaker

ms.insert({5, tick++});                // (5, 0)
ms.insert({5, tick++});                // (5, 1) — counts as distinct node
ms.insert({3, tick++});                // (3, 2)
ms.insert({5, tick++});                // (5, 3)

cout << ms.size();                  // 4 — three 5s and one 3

The four "must-know" multiset operations

1) Count of elements strictly less than x

// Smallest possible tie-breaker → ignore everything with value == x
int cnt_less = ms.order_of_key({x, INT_MIN});

2) Count of elements ≤ x (i.e., strictly less than x+1)

int cnt_le = ms.order_of_key({x, INT_MAX});

3) Erase exactly one copy of x

// lower_bound on (x, INT_MIN) lands on the first node with key x (any tie-breaker)
auto it = ms.lower_bound({x, INT_MIN});
if (it != ms.end() && it->first == x) ms.erase(it);

4) Get the k-th smallest value (ignoring duplicates? No, including them)

int kth = ms.find_by_order(k)->first;

Putting it together — a wrapper class

template<class T>
struct Multiset {
    tree<pair<T, int>, null_type, less<pair<T, int>>,
         rb_tree_tag, tree_order_statistics_node_update> t;
    int tick = 0;

    void insert(T x)             { t.insert({x, tick++}); }
    void erase_one(T x) {
        auto it = t.lower_bound({x, INT_MIN});
        if (it != t.end() && it->first == x) t.erase(it);
    }
    int  count_less(T x)        { return t.order_of_key({x, INT_MIN}); }
    int  count_le(T x)          { return t.order_of_key({x, INT_MAX}); }
    int  count_eq(T x)          { return count_le(x) - count_less(x); }
    T    kth(int k)              { return t.find_by_order(k)->first; }
    int  size()                 { return t.size(); }
};

Part 4 · Custom Comparators

The third template parameter of tree is the comparator. Anything that works for std::set works here.

Descending order

template<class T>
using ordered_set_desc =
    tree<T, null_type, greater<T>, rb_tree_tag, tree_order_statistics_node_update>;

ordered_set_desc<int> s = {3, 1, 5, 9};
cout << *s.find_by_order(0);     // 9 (largest is now "first")

Custom struct comparator: sort pairs by .second

struct BySecond {
    bool operator()(const pair<int,int>& a,
                    const pair<int,int>& b) const {
        if (a.second != b.second) return a.second < b.second;
        return a.first < b.first;       // tie-break by first
    }
};

tree<pair<int,int>, null_type, BySecond,
     rb_tree_tag, tree_order_statistics_node_update> s;

s.insert({10, 5});  s.insert({3, 2});  s.insert({7, 8});
// Sorted by .second: (3,2), (10,5), (7,8)

cout << s.find_by_order(0)->first;   // 3  (.second=2 is smallest)
⚠️
Comparator gotcha The comparator must define a strict weak ordering — same rule as std::sort. Returning true for equal elements (using <= instead of <) silently corrupts the tree.

Part 5 · gp_hash_table — A Faster, Harder unordered_map

💡
Analogy gp_hash_table is std::unordered_map on espresso. Same interface (mostly), faster constants, and trivial to harden against adversarial test cases.

Basic use — drop-in replacement

#include <ext/pb_ds/assoc_container.hpp>
using namespace __gnu_pbds;

gp_hash_table<int, int> mp;

mp[5] = 100;
mp[3] = 200;
cout << mp[5];                // 100
if (mp.find(5) != mp.end()) cout << "present";
mp.erase(3);
cout << mp.size();             // 1

for (auto& [k, v] : mp) cout << k << "->" << v << '\n';

Why it's faster

std::unordered_mapgp_hash_table
Collision strategySeparate chaining (linked lists)Open addressing (probing)
Cache localityBad — list nodes are scatteredGood — flat array
Constant factorBaseline~2–4× faster typically
Memory overheadOne pointer per elementOne slot per element

Hardening against anti-hash attacks (CRUCIAL)

Codeforces test setters know exactly which inputs blow up GCC's default integer hash to O(n). Defend yourself in 3 lines:

struct custom_hash {
    static uint64_t splitmix64(uint64_t x) {
        x += 0x9e3779b97f4a7c15ULL;
        x = (x ^ (x >> 30)) * 0xbf58476d1ce4e5b9ULL;
        x = (x ^ (x >> 27)) * 0x94d049bb133111ebULL;
        return x ^ (x >> 31);
    }
    size_t operator()(uint64_t x) const {
        static const uint64_t SEED =
            chrono::steady_clock::now().time_since_epoch().count();
        return splitmix64(x + SEED);
    }
};

gp_hash_table<long long, int, custom_hash> safe_mp;

Hashing pairs

struct pair_hash {
    size_t operator()(const pair<int,int>& p) const {
        return custom_hash{}(((long long)p.first << 32) | (unsigned)p.second);
    }
};

gp_hash_table<pair<int,int>, int, pair_hash> coord_count;
coord_count[{3, 5}]++;

Use case: 2D coordinate frequency in a grid problem

// Count visited cells
gp_hash_table<long long, int, custom_hash> cells;
for (auto& [x, y] : moves)
    cells[((long long)x << 32) | (unsigned)y]++;
cout << cells.size() << " distinct cells\n";
✅
The default vs custom hash speedup In contest benchmarks, gp_hash_table<long long, int, custom_hash> is often 5–10× faster than unordered_map<long long, int> on adversarial inputs that would otherwise TLE.

Part 6 · PBDS priority_queue — A Mutable Heap

💡
Analogy std::priority_queue is a black box — you can only push, pop, and peek the top. PBDS's version gives you handles (iterators) into the heap that survive across pushes and pops, and a modify() operation to change a node's priority in place.

Basic use

#include <ext/pb_ds/priority_queue.hpp>
using namespace __gnu_pbds;

// Max-heap (default), pairing-heap implementation
__gnu_pbds::priority_queue<int, less<int>, pairing_heap_tag> pq;

pq.push(5);  pq.push(2);  pq.push(9);
cout << pq.top();    // 9
pq.pop();
cout << pq.top();    // 5
⚠️
Use the full name __gnu_pbds::priority_queue to avoid clashing with std::priority_queue.

The killer feature: modify() and erase(it)

__gnu_pbds::priority_queue<int> pq;
auto h1 = pq.push(5);     // returns an iterator (handle)
auto h2 = pq.push(2);
auto h3 = pq.push(9);

// Increase the priority of node h2 from 2 to 100
pq.modify(h2, 100);
cout << pq.top();         // 100

// Erase a specific node (not just the top)
pq.erase(h3);             // removes the 9

Heap implementations (the 5 tags)

TagpushpopmodifyBest for
pairing_heap_tagO(1)O(log n)O(log n)Default — usually fastest in practice
thin_heap_tagO(1)O(log n)O(1) if priority increasesDijkstra (many decrease-key)
binomial_heap_tagO(log n)O(log n)O(log n)Theoretical interest only
rc_binomial_heap_tagO(1)O(log n)O(log n)Same; rarely used
binary_heap_tagO(log n)O(log n)not supportedPlain heap, no handles

Use case: Dijkstra with proper decrease-key

using PQ = __gnu_pbds::priority_queue<pair<int,int>,
                                       greater<pair<int,int>>,
                                       thin_heap_tag>;

vector<int> dijkstra(vector<vector<pair<int,int>>>& g, int src) {
    int n = g.size();
    vector<int> dist(n, INT_MAX);
    vector<PQ::point_iterator> handle(n);    // stable handles
    PQ pq;

    dist[src] = 0;
    handle[src] = pq.push({0, src});

    while (!pq.empty()) {
        auto [d, u] = pq.top();  pq.pop();
        for (auto& [v, w] : g[u]) {
            if (dist[u] + w < dist[v]) {
                dist[v] = dist[u] + w;
                if (handle[v] == PQ::point_iterator())
                    handle[v] = pq.push({dist[v], v});
                else
                    pq.modify(handle[v], {dist[v], v});  // no stale entries!
            }
        }
    }
    return dist;
}

Part 7 · PBDS Trie — Built-in Prefix Search

PBDS ships a trie with a built-in prefix-search policy. Less commonly used than a hand-rolled trie but extremely concise.

#include <ext/pb_ds/assoc_container.hpp>
#include <ext/pb_ds/trie_policy.hpp>
using namespace __gnu_pbds;

typedef trie<string, null_type, trie_string_access_traits<>,
             pat_trie_tag, trie_prefix_search_node_update> pref_trie;

pref_trie t;
t.insert("apple");
t.insert("app");
t.insert("application");
t.insert("banana");

// All strings starting with "app"
auto [first, last] = t.prefix_range("app");
for (auto it = first; it != last; ++it)
    cout << *it << '\n';          // "app", "apple", "application"

Use case: autocomplete

vector<string> complete(pref_trie& t, string q) {
    vector<string> out;
    auto [a, b] = t.prefix_range(q);
    for (auto it = a; it != b; ++it) out.push_back(*it);
    return out;
}
📘
When to prefer a hand-rolled trie The PBDS trie has overhead. For tight inner loops (e.g., XOR-tries with millions of operations), a hand-rolled int children[2] array trie is often 3–5× faster. PBDS trie is best when you want to write code fast, not when you want code that runs fast.

Pattern 1 · K-th smallest in a dynamic set

ordered_set<int> s;
for (auto& q : queries) {
    if (q.type == INSERT) s.insert(q.x);
    else if (q.type == ERASE)  s.erase(q.x);
    else if (q.type == KTH)    cout << *s.find_by_order(q.k - 1) << '\n';
}
// Each operation: O(log n)

Pattern 2 · Count inversions in O(n log n)

Already shown in Part 1. Reverse-scan with order_of_key.

Pattern 3 · "Number of elements in [L, R]" rolling queries

ordered_set<int> s;
auto count_in_range = [&](int L, int R) {
    return s.order_of_key(R + 1) - s.order_of_key(L);

Pattern 4 · Sliding-window median

// Maintain a multiset; median = element at position k/2 or average of two middles
Multiset<int> ms;
for (int i = 0; i < n; ++i) {
    ms.insert(a[i]);
    if (i >= k) ms.erase_one(a[i - k]);
    if (i >= k - 1) {
        int mid = ms.kth(k / 2);
        cout << mid << ' ';
    }
}

Pattern 5 · Distinct elements seen so far (online)

gp_hash_table<long long, int, custom_hash> seen;
int distinct = 0;
for (long long x : stream) {
    if (seen.find(x) == seen.end()) ++distinct;
    seen[x]++;
}

Pattern 6 · Order of first occurrence (insertion-order map)

// "Was x seen before? If so, when?"
gp_hash_table<int, int, custom_hash> first_seen;
for (int i = 0; i < n; ++i)
    if (first_seen.find(a[i]) == first_seen.end())
        first_seen[a[i]] = i;

// Three operation types: insert salary, change salary, count in [L, R]
Multiset<int> ms;
ms.insert(salary);
ms.erase_one(old_salary);  ms.insert(new_salary);

Pattern 8 · Top-K queries with mutable PQ

__gnu_pbds::priority_queue<pair<int,int>> pq;
vector<decltype(pq)::point_iterator> handle(n);

for (int i = 0; i < n; ++i) handle[i] = pq.push({score[i], i});

// Bump player i's score

// Drop player j
pq.erase(handle[j]);

Pattern 9 · Anti-hash-attack hash map

// One-line replacement that survives Codeforces anti-hash test cases

for (long long x : a) mp[x]++;

pref_trie t;
for (auto& w : dictionary) t.insert(w);

Part 9.2 · Top 10 PBDS Pitfalls

  1. Forgetting the headers. <bits/stdc++.h> does NOT pull in PBDS. Add <ext/pb_ds/assoc_container.hpp> and <ext/pb_ds/tree_policy.hpp> explicitly.
  2. Forgetting using namespace __gnu_pbds; — or qualifying half the types with it and not the other half.
  3. tree rejects duplicates silently. Insert returns a pair<iterator, bool> — bool is false when duplicate. Use the pair-trick for multiset behavior.
  4. order_of_key is STRICTLY less. To count "≤ x" do order_of_key(x + 1) for ints, or use the multiset pair trick.
  5. find_by_order(size()) returns end() — always check before dereferencing.
  6. Erasing in a multiset by VALUE does not remove all duplicates from a tree-based multiset (unlike std::multiset::erase) — and on the pair-trick version it doesn't even compile because keys aren't of type T. Always erase via lower_bound.
  7. gp_hash_table with default hash is hackable. ALWAYS use a custom hash with a random seed for long long keys on Codeforces.
  8. __gnu_pbds::priority_queue handle becomes invalid after pop() if it pointed at the popped element. Track which ones you've popped manually.
  9. binary_heap_tag does NOT support modify() or erase(it). If you need them, pick pairing_heap_tag or thin_heap_tag.
  10. PBDS is GCC-only. Don't paste it into MSVC code or production projects. It's purely a contest weapon.
