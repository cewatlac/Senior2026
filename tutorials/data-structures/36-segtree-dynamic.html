Lesson 7 — Dynamic Sparse Segment Tree

Huge Coordinates · No Compression · Create-on-Demand · O(log U) per op

🧩 Problem Definition

The input coordinates can be as large as U = 10⁹ or 10¹⁸, but the number of operations q is at most ~10⁵ or ~10⁶. We need to support point updates and range queries on values indexed by these huge coordinates — without allocating an array of size U.

Two classical options compete here:

(a) Coordinate compression. Collect all coordinates that appear in the offline list of operations, sort, deduplicate, replace each coordinate by its rank. Resulting universe is at most q. Build a normal segment tree.

(b) Dynamic sparse segment tree. Keep coordinates as-is. Allocate tree nodes only when a path through them is first touched. Each operation allocates at most O(log U) new nodes; total memory is O(q log U).

Option (a) is simpler and tighter when all operations are known in advance. Option (b) is essential when operations arrive online (you don't know the coordinates ahead of time), or when the algorithm spawns subtrees during execution (see Segment Tree Merging, Lesson 9, and 2D Dynamic, Lesson 13).

💡 Intuition from Scratch

A normal segment tree over [0, U − 1] would have 2U nodes. For U = 10⁹ that's eight gigabytes of memory — clearly unworkable. But notice: most of those nodes carry the identity value (zero, in a sum tree), because we only ever touch a tiny fraction of them. If a node and all its descendants store identity, the node is informationally trivial — we can pretend it doesn't exist.

That's the entire idea. We represent the root explicitly (it always exists). Every other node is either explicit (allocated) or implicit (treated as a "ghost" carrying identity). A ghost's left child is also a ghost; same for the right. When an update or query needs to descend past a ghost, we materialize it: allocate a new node with identity value and two new ghost children, then proceed normally.

🌳 Visual Mental Model

Picture an infinite segment tree over [0, U − 1] drawn faintly in gray. Initially every node is gray (a ghost). A single point update at coordinate p walks from the root toward the leaf at p, turning each ghost on the path into a solid node along the way. After the update, exactly log₂ U nodes have been materialized; everything else is still gray.

The next update materializes more nodes — but only along its own path. If the new path branches off the first one halfway down, only the second half of the path is freshly materialized; the prefix nodes are reused.

After q updates, at most q · log U nodes exist. That's 10⁵ · 60 = 6 · 10⁶ for U = 10¹⁸ — fits comfortably in memory.

📐 Formal Abstraction

Nodes live in a single vector<Node> pool. A node has two integer child pointers and a value field. -1 means "ghost child":

struct Node { long long sum = 0; int left = -1; int right = -1; };

The root has index 0 and is always present. The tree's coordinate range is fixed at construction: [lo_root, hi_root], typically [0, U − 1]. Every recursive call carries (node_idx, lo, hi) where [lo, hi] is the coordinate interval covered by node_idx in the implicit balanced tree.

Ghost semantics: a ghost node's value is identity, its sum over its full interval is identity, and its children are also ghosts. When we need a real child during an operation, we call materialize to allocate one with default fields and update the parent's pointer.

🔬 Key Observations

Observation 1. A point update touches one root-to-leaf path of length ⌈log₂ U⌉. At most that many new nodes are materialized per operation.

Observation 2. Range queries follow the same three-cases descent as Lesson 1. When the descent meets a ghost child, the contribution is identity, so we return immediately — no allocation needed for queries.

Observation 3. Because the tree is balanced over the coordinate range (not over the touched points), depth is always exactly ⌈log₂ U⌉, regardless of how many points we've touched. This bounds operations at O(log U).

Observation 4. The "implicit tree" model is equivalent to a normal segment tree where unspecified leaves carry identity. The dynamic tree only exposes the parts that differ from identity.

⚙️ Algorithm

point_add(pos, value)

Recurse from the root over [lo, hi] = [0, U − 1]. At each step, if lo == hi, add value to the current node's sum. Otherwise compute mid, decide which child contains pos, materialize that child if it's currently a ghost (-1), recurse into it, and finally update the current node's sum as the sum of its two children's sum (where a ghost contributes 0).

range_sum(ql, qr)

Standard three-cases descent. If [lo, hi] is disjoint from [ql, qr], return 0. If fully inside, return node.sum. Otherwise recurse into both children — but if a child is a ghost, treat its contribution as 0 without allocating.

Why we don't materialize during queries

Queries are read-only. A ghost's sum is exactly 0 by invariant, so there's nothing to gain from creating a real node. Allocating in queries would multiply memory use by the query count, defeating the entire purpose.

🛡️ Invariants

✅ Invariant — Ghost-as-identity invariant

A ghost node (pointer value -1) represents an entire subtree whose conceptual sum is identity (0 for sum, +∞ for min, etc.). The actual sum is never materialized; the operation handler treats it as identity by convention.

✅ Invariant — Root invariant

The root is always present at index 0 in the pool. Its coordinate range [lo_root, hi_root] is fixed at construction.

✅ Invariant — Sum invariant

For every explicit (non-ghost) node, sum = sum(left child) + sum(right child), where a ghost child contributes 0. This holds outside the middle of an update.

✅ Invariant — Materialization invariant

An update creates at most one new node per level, and only when descending past a ghost child. Queries never materialize.

🧮 Proof of Correctness

🧮 Proof — Why ghost children are safe to skip in queries

By the Ghost-as-identity invariant, every ghost subtree has conceptual sum 0. So during a recursive query, descending into a ghost child is equivalent to descending into an explicit subtree of zero sums — both contribute 0 to the answer. We can return 0 without allocating anything.

For updates the situation is different: the update may turn some leaf's value from 0 into a non-zero number, which means the subtree containing that leaf must become explicit. So we materialize one node per level on the path. Off-path siblings remain ghosts and continue to satisfy the invariant. ∎

🧮 Proof — Depth and per-operation cost

The implicit tree is always a balanced binary tree over the fixed range [0, U − 1], so every root-to-leaf path has length exactly ⌈log₂ U⌉. Each operation traverses at most two such paths (the partial-overlap boundary), so the worst-case work per operation is O(log U) regardless of how many points have been touched so far. ∎

🧰 Core Template Philosophy

The template is a small variation on the recursive segment tree from Lesson 1, with three differences. First, the tree is stored in a vector<Node> pool rather than a fixed array. Second, child pointers can be -1 meaning "ghost". Third, the operation handlers consult those pointers and either materialize (in updates) or short-circuit to identity (in queries).

Critically: the value -1 for a child pointer is the entire reason the structure is small. Treat it as a contract: anywhere you see node.left = -1, you must guarantee the left subtree behaves as identity. Anywhere you intend to use a child, either materialize or check for -1 first.

💻 Master Template (C++)

#include <bits/stdc++.h>
using namespace std;

struct DynamicSparseSumSegTree {
    using T = long long;
    using Coord = long long;

    struct Node {
        T sum = 0;
        int left = -1;
        int right = -1;
    };

    Coord lo_root, hi_root;
    vector<Node> pool;

    DynamicSparseSumSegTree(Coord lo, Coord hi) : lo_root(lo), hi_root(hi) {
        pool.push_back({});  // root at index 0
    }

    int new_node() {
        int u = (int)pool.size();
        pool.push_back({});
        return u;
    }

    // SAFE: do NOT take an int& into pool here. new_node() ends in
    // push_back, which can reallocate pool and invalidate any prior
    // reference. We re-read parent's child slot by index after the
    // allocation and write the new child id then.
    int materialize_left(int parent) {
        if (pool[parent].left == -1) {
            int child = new_node();          // may reallocate pool
            pool[parent].left = child;       // re-read parent by index
        }
        return pool[parent].left;
    }

    int materialize_right(int parent) {
        if (pool[parent].right == -1) {
            int child = new_node();
            pool[parent].right = child;
        }
        return pool[parent].right;
    }

    void point_add(Coord pos, T value) {
        point_add(0, lo_root, hi_root, pos, value);
    }

    void point_add(int u, Coord lo, Coord hi, Coord pos, T value) {
        if (lo == hi) {
            pool[u].sum += value;
            return;
        }

        Coord mid = lo + (hi - lo) / 2;

        if (pos <= mid) {
            int c = materialize_left(u);
            point_add(c, lo, mid, pos, value);
        } else {
            int c = materialize_right(u);
            point_add(c, mid + 1, hi, pos, value);
        }

        T leftSum  = pool[u].left  == -1 ? 0 : pool[pool[u].left].sum;
        T rightSum = pool[u].right == -1 ? 0 : pool[pool[u].right].sum;
        pool[u].sum = leftSum + rightSum;
    }

    T range_sum(Coord ql, Coord qr) const {
        return range_sum(0, lo_root, hi_root, ql, qr);
    }

    T range_sum(int u, Coord lo, Coord hi, Coord ql, Coord qr) const {
        if (u == -1 || qr < lo || hi < ql) return 0;
        if (ql <= lo && hi <= qr) return pool[u].sum;

        Coord mid = lo + (hi - lo) / 2;
        return range_sum(pool[u].left,  lo,     mid, ql, qr)
             + range_sum(pool[u].right, mid+1, hi,  ql, qr);
    }
};

🔎 Line-by-Line Explanation

Constructor, new_node, and materialize_left / materialize_right click to expand/collapse

The constructor records the fixed coordinate range and pushes a single root node. new_node() appends a default node to the pool and returns its index. The two materialize helpers take the parent's index by value and a direction: each checks whether the parent's child slot is -1 and, if so, allocates a new node and writes its id into pool[parent].left or pool[parent].right. The critical detail is that we never hold an int& into the pool across new_node(), because push_back can reallocate the underlying buffer and invalidate any reference into it. The safe pattern is: allocate first, then re-read the parent by index and write the new child id then.

point_add — descend with materialization click to expand/collapse

At a leaf we add the value to sum and return. At an internal node we compute mid, decide direction by pos, materialize the chosen child if necessary, recurse, then update sum from both children's sums (treating a ghost child as 0). Note that we only materialize the child we descend into, never the sibling.

range_sum — short-circuit on ghosts click to expand/collapse

The first guard checks both u == -1 (ghost) and the no-overlap condition; both return identity. If fully inside, return pool[u].sum. Otherwise recurse into both children — even if a child is -1, the recursive call handles it by returning 0 via the first guard. No allocation happens anywhere in the query.

⚠️ Common Pitfalls

⚠️ Pitfall — Using lo + hi instead of lo + (hi - lo) / 2

For U = 10¹⁸, lo + hi overflows long long. The standard safe midpoint is lo + (hi - lo) / 2. Always use that form.

⚠️ Pitfall — Forgetting to handle u == -1 in queries

If a recursive query forgets to check u == -1, it will dereference pool[-1] with disastrous results. The cleanest pattern is a single guard at the top of range_sum covering both no-overlap and ghost cases.

⚠️ Pitfall — Materializing during a query

Pure queries should never grow the pool. If you accidentally call materialize_left/materialize_right in the query path, your memory usage will scale with the number of queries — exactly what we wanted to avoid.

⚠️ Pitfall — Holding an int& into the pool across new_node()

A tempting helper is int materialize(int& slot){ if(slot == -1){ slot = new_node(); } return slot; }. It is unsafe: new_node() ends in push_back, which can reallocate the pool's underlying buffer and invalidate slot. The assignment then writes to freed memory; the parent never sees the new child id, and subsequent queries return zero. The fix is to take the parent index by value, allocate first, then write the new child id back via pool[parent].left / .right.

⚠️ Pitfall — Storing iterators or pointers across operations

The pool is a vector; reallocation invalidates pointers. Always use integer indices, even for the root. Don't store Node* or auto& n = pool[u] across recursive descents that might allocate.

⚠️ Pitfall — Pool reallocation slowness

When the vector grows past its capacity, it doubles and copies everything. For large pools this is a noticeable hiccup. Reserve up front with a generous upper bound: pool.reserve(20'000'000) is fine for typical contest budgets. (Note: reserve is an optimization, not the actual fix for the int& trap above.)

🎛️ Variants & Extensions

Variant 1 — Range count of points (frequency tree)

Same template, value stored is the count. point_add(pos, +1) when a point appears, point_add(pos, -1) when removed. range_sum(ql, qr) counts points in the range.

Variant 2 — Range update + range query with lazy

Combine with Lesson 4's lazy machinery. The push step must materialize children before pushing to them. Both apply calls work on materialized nodes as in the dense version.

Variant 3 — Min/Max instead of Sum

Ghost identity becomes +∞ (or −∞), so a ghost child contributes +∞ (or −∞) to a min (or max) merge. Adjust the early return in range_sum to return the appropriate identity.

Variant 4 — Merging two dynamic trees (Lesson 9)

If two dynamic sparse trees represent two disjoint multisets, they can be merged in O(total node count) by walking both in parallel and combining. This is the backbone of the small-to-large DSU-on-tree pattern.

🏁 Final Summary

The dynamic sparse segment tree extends the recursive segment tree to coordinates as large as 10¹⁸ by materializing nodes only when their interval is first touched. Each operation costs O(log U) time and allocates at most log U new nodes. Queries never allocate.

The structure is the natural choice when you can't compress coordinates ahead of time, or when an algorithm dynamically spawns subtrees. It also serves as a building block for segment-tree merging (next lesson) and for the persistent and 2D variants in later lessons. Once you can write the materialize helper, the rest of the template is mechanically the same as Lesson 1.
