Mo's Other Variants

Rollback Mo · 4D Mo · Online Block-Precomputed · Adjacent Sqrt Techniques

🗺 Overview — What This File Covers

The three earlier files in this series build the master template and then specialize it: classic array Mo, Mo on trees, and Mo with updates. This file collects the remaining variants of the same family, plus a short tour of adjacent sqrt-decomposition techniques that competitive programmers tend to confuse with Mo.

The first variant is Rollback Mo with DSU, the right tool when the data structure can add but cannot delete. The second is 4D Mo, used when a single query controls two ranges instead of one. The third is the online block-precomputed variant, which gives up offline reordering in exchange for online query handling. After the three variants we add a brief tour of three adjacent sqrt techniques — sorted blocks, block-Fenwick hybrids, and 2-D sliding windows — that look like Mo but are not Mo. Knowing where the family ends is as important as knowing what is inside it.

Every variant in this file inherits the comparator pattern, the block-sorted query stream, the pointer-movement engine, and the answer-maintenance philosophy from the master template. The only thing that changes is how the maintained state reacts to additions and removals, and how the comparator extends from one dimension to several.

🧩 Variant 1 — Why Rollback Mo

The classic Mo engine requires add and remove to be exact inverses. Plenty of useful data structures break that requirement. The most important example is the disjoint-set union with union-by-size or union-by-rank. Adding an edge is an O(α) union; removing an arbitrary old edge is hard in the general case because path compression has rewired many ancestors and several unions later may have re-rooted entire trees.

Rollback Mo sidesteps the problem rather than solving it. Instead of designing a true delete, we arrange the work so that every "delete" is actually a rollback to a previous snapshot. Pointer movement is reorganized so the right pointer is only ever added to (and never deleted from) within a left-block, and the left pointer's contributions are kept in a separate, fully-rolled-back layer.

The canonical illustration in this section is CodeChef GERALD07. The input gives an ordered list of edges; each query asks how many connected components remain after adding all edges with indices in [l, r]. Adding an edge to DSU is one union; removing one is what the rollback machinery is for.

⚙️ Rollback Mo — Algorithm

Small versus large dichotomy

Queries are partitioned by length. If r − l + 1 ≤ B (where B = √m), the range is "small" — at most B edges. Such queries are answered directly by snapshotting the DSU, adding all edges l..r, reading the component count, and rolling back. Each small query costs O(B · α), which is acceptable.

Large queries, grouped by left-block

Larger queries are sorted by block_l, then by r ascending. For each new left-block we reinitialize the DSU and place the right pointer at blockRight — the last index of the left-block. Inside the block we move the right pointer forward monotonically. This makes right-pointer movement permanent: an edge added on the way right is never removed within this block.

The left part of the interval is handled separately. For each query [l, r] in the current block, after the right pointer has advanced to r, we take a snapshot of the DSU, then add the edges from l up to blockRight (one block's worth at most), read the component count, and roll back to the snapshot. The right-pointer state survives across queries; the left-pointer state is rebuilt from scratch every time. That asymmetry is the whole point.

Rollback DSU mechanics

The DSU needs three primitives: a regular unite(a, b) that records enough information to be reversible, a snapshot() returning the current history-length, and a rollback(snap) that pops history entries until the length is back to snap, undoing each recorded change. Because we cannot use path compression (it would rewire many pointers that the history struct does not track), find is the iterative while (parent[x] != x) x = parent[x]; with union-by-size, the worst-case find cost is O(log n).

Every call to unite pushes one history entry, even if the two endpoints are already in the same component. That dummy entry stores the unchanged components value so the rollback bookkeeping stays consistent. Forgetting the no-op push is a classic bug that makes a non-trivial fraction of submissions wrong on edge cases.

📄 Master Template — CodeChef GERALD07

This is the accepted CodeChef-GERALD07 template from the source corpus, preserved with its original logic and variable names. The two important contracts are documented at the top of RollbackDSU: every unite pushes exactly one history entry, and rollback(snap) brings history.size() back to snap.

#include <bits/stdc++.h>
using namespace std;

struct RollbackDSU {
    struct Change {
        int childRoot;
        int parentRoot;
        int sizeParentBefore;
        int componentsBefore;
    };

    vector<int> parent, sz;
    vector<Change> history;
    int components = 0;

    RollbackDSU(int n = 0) { init(n); }

    void init(int n) {
        parent.resize(n + 1);
        sz.assign(n + 1, 1);
        history.clear();
        components = n;
        for (int i = 1; i <= n; i++) parent[i] = i;
    }

    int find(int x) {
        while (parent[x] != x) x = parent[x];
        return x;
    }

    int snapshot() const {
        return (int)history.size();
    }

    void rollback(int snap) {
        while ((int)history.size() > snap) {
            Change ch = history.back();
            history.pop_back();

            if (ch.childRoot == -1) {
                components = ch.componentsBefore;
                continue;
            }

            parent[ch.childRoot] = ch.childRoot;
            sz[ch.parentRoot] = ch.sizeParentBefore;
            components = ch.componentsBefore;
        }
    }

    void unite(int a, int b) {
        int ra = find(a);
        int rb = find(b);

        if (ra == rb) {
            history.push_back({-1, -1, -1, components});
            return;
        }

        if (sz[ra] > sz[rb]) swap(ra, rb);

        history.push_back({ra, rb, sz[rb], components});
        parent[ra] = rb;
        sz[rb] += sz[ra];
        components--;
    }
};

struct Edge {
    int u, v;
};

struct Query {
    int l, r, id;
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int T;
    cin >> T;

    while (T--) {
        int n, m, q;
        cin >> n >> m >> q;

        vector<Edge> edges(m + 1);
        for (int i = 1; i <= m; i++) {
            cin >> edges[i].u >> edges[i].v;
        }

        int B = max(1, (int)sqrt(m));

        vector<int> answer(q);
        vector<Query> bigQueries;
        bigQueries.reserve(q);

        RollbackDSU dsu(n);

        for (int id = 0; id < q; id++) {
            int l, r;
            cin >> l >> r;

            if (r - l + 1 <= B) {
                int snap = dsu.snapshot();
                for (int e = l; e <= r; e++) {
                    dsu.unite(edges[e].u, edges[e].v);
                }
                answer[id] = dsu.components;
                dsu.rollback(snap);
            } else {
                bigQueries.push_back({l, r, id});
            }
        }

        sort(bigQueries.begin(), bigQueries.end(), [&](const Query& a, const Query& b) {
            int ba = a.l / B;
            int bb = b.l / B;
            if (ba != bb) return ba < bb;
            return a.r < b.r;
        });

        int lastBlock = -1;
        int rightPtr = 0;
        int blockRight = 0;

        for (const Query& qu : bigQueries) {
            int block = qu.l / B;

            if (block != lastBlock) {
                dsu.init(n);
                blockRight = min(m, (block + 1) * B - 1);
                rightPtr = blockRight;
                lastBlock = block;
            }

            while (rightPtr < qu.r) {
                rightPtr++;
                dsu.unite(edges[rightPtr].u, edges[rightPtr].v);
            }

            int snap = dsu.snapshot();
            for (int e = qu.l; e <= blockRight; e++) {
                dsu.unite(edges[e].u, edges[e].v);
            }

            answer[qu.id] = dsu.components;
            dsu.rollback(snap);
        }

        for (int x : answer) cout << x << '\n';
    }

    return 0;
}

⚠️ Rollback Mo — Pitfalls

No path compression. Path compression rewires arbitrary ancestor pointers and the rollback stack cannot recover them efficiently. Use the iterative find shown above; the O(log n) per find is fine.

Every unite pushes a history entry — even no-ops. The case ra == rb still pushes a dummy {-1, -1, -1, components} entry. Without it, the count of entries in the stack does not match the number of unite calls, and rollback(snap) will undo a different boundary than the one we snapshotted.

Small ranges are answered separately. If you tried to handle ranges shorter than B in the main loop, you would end up with left and right pointers in the same block, and the "right pointer moves only forward" invariant would collapse.

Re-initialize the DSU between left-blocks. Forgetting dsu.init(n) at the start of each new left-block carries stale unions into the next block; answers will be too small.

🧩 Variant 2 — 4D Mo: Two Intervals at Once

Some problems naturally involve two ranges per query, for example "given two subtrees, compare the multisets of their colors." The state of Mo doubles in dimension: instead of one pair (curL, curR), we maintain two pairs (cl1, cr1) and (cl2, cr2). Each pair has its own four-loop movement block.

The comparator extends accordingly. The primary key is the block of l1, the secondary is the block of r1, the tertiary is the block of l2, and the final key is r2. Some authors use odd-even snakes on the inner keys to reduce constant factors; this is optional.

The answer structure has to tolerate additions and removals from both intervals. Most often the maintained state is two independent frequency tables, one per interval, plus whatever aggregate ties them together — for instance, the number of values whose counts differ between the two tables, or the Jaccard-style overlap of the two multisets.

📄 4D Mo — Movement Skeleton

The skeleton below shows the generic shape. Plug in your own add1/remove1 for the first interval and add2/remove2 for the second. The maintained answer can read from both tables in get_answer().

struct Query4D {
    int l1, r1, l2, r2, id;
};

int BLOCK;

bool cmp4(const Query4D& a, const Query4D& b) {
    int al1 = a.l1 / BLOCK, bl1 = b.l1 / BLOCK;
    if (al1 != bl1) return al1 < bl1;
    int ar1 = a.r1 / BLOCK, br1 = b.r1 / BLOCK;
    if (ar1 != br1) return ar1 < br1;
    int al2 = a.l2 / BLOCK, bl2 = b.l2 / BLOCK;
    if (al2 != bl2) return al2 < bl2;
    return a.r2 < b.r2;
}

// Maintained state, problem-specific.
int cl1 = 0, cr1 = -1;
int cl2 = 0, cr2 = -1;
// freq1[], freq2[], current_answer ...

void solve_4d(vector<Query4D>& queries) {
    BLOCK = max(1, (int)sqrt(n));
    sort(queries.begin(), queries.end(), cmp4);

    for (const Query4D& q : queries) {
        while (cl1 > q.l1) add1(--cl1);
        while (cr1 < q.r1) add1(++cr1);
        while (cl1 < q.l1) remove1(cl1++);
        while (cr1 > q.r1) remove1(cr1--);

        while (cl2 > q.l2) add2(--cl2);
        while (cr2 < q.r2) add2(++cr2);
        while (cl2 < q.l2) remove2(cl2++);
        while (cr2 > q.r2) remove2(cr2--);

        ans[q.id] = get_answer();
    }
}

The complexity is harder to state cleanly because it depends on how the two intervals are correlated. In the worst case the total pointer travel scales as O(N · √N + Q · √N) per dimension, multiplied across four axes — but the constant factors are large and the comparator tuning matters. Use 4D Mo only when no simpler reduction exists.

⚠️ 4D Mo — Pitfalls

Double counting on overlap. If the two intervals can share cells, the add/remove hooks may credit the same element twice. Decide once whether the answer should treat the intervals as multisets (allow overlap) or as sets (deduplicate), and document the choice next to the maintained state.

Comparator order matters. Different orderings of the four sort keys yield very different total pointer-travel constants. Start with (block_l1, block_r1, block_l2, r2) and tune if necessary.

Memory is doubled. Two frequency tables, two pointer pairs, two of every problem-specific scalar. Watch the memory budget — for n = 10⁵ with non-trivial answer states, 4D Mo can hit a few hundred MB without care.

🧩 Variant 3 — Online Block-Precomputed

Classic Mo's central trick is offline reordering. When the problem statement forbids reordering — when each query must be answered before the next one is read — Mo's amortization argument no longer applies. The online block-precomputed variant is the sqrt-decomposition workaround.

The idea is to precompute, for every full-block interval [i·B, (i+1)·B − 1], the answer structure needed by the problem. A query [l, r] is then answered by combining the full-block structure spanning l's block to r's block with manual additions for the boundary elements on both sides.

If point updates are present, every update touches all precomputed structures whose block contains the updated position; with a block size of B, that is N/B structures, each costing O(B) to patch, giving O(N) per update — too slow for hard instances. In practice the online block-precomputed variant is best suited to problems with relatively few updates and many queries, or to settings where the per-block structure has a cheaper update path (e.g. Fenwick-per-block).

⚙️ Online Block-Precomputed — Algorithm

Block layout

Split the array into N/B blocks of size B. For each pair of block indices (i, j) with i ≤ j, precompute the answer structure for the union of blocks i..j. The naive precomputation is O((N/B)² · N) — typically O(N²/B) — and is the dominant preprocessing cost.

Query handling

Given [l, r], locate the inner full-block range [i, j] where i = ⌈l/B⌉ and j = ⌊(r+1)/B⌋ − 1. Read the precomputed answer for [i·B, (j+1)·B − 1]. Then apply the boundary corrections: the prefix from l to i·B − 1 and the suffix from (j+1)·B to r. Each correction touches at most B elements, so the per-query work is O(B) outside the precomputed lookup.

Update handling

If updates are allowed and inexpensive, each point update at position pos touches every block structure whose interval contains pos. There are N/B such structures on each side of pos's own block. Patching a single structure costs whatever the structure costs to update — often O(log N) if it is a Fenwick tree.

This variant is the right answer when the problem says "queries are interleaved with updates and must be processed online, but the answer is frequency-based with a clean block layout." That phrasing is rare in pure Mo problems but common in semi-online sqrt problems. The accepted-code section below for SPOJ-RACETIME is a worked-out version of exactly this pattern.

📐 Adjacent Technique — SPOJ RACETIME (Sorted Blocks)

VJudge: https://vjudge.net/problem/SPOJ-RACETIME

The problem maintains an array under point updates and answers online "count values ≤ x in [l, r]" queries. Each block keeps a sorted copy of its values; full blocks are answered via upper_bound, and partial-block boundaries are scanned linearly. Updates re-sort the affected block. The query cost is O(√N · log N) and the update cost is O(√N log √N) or O(√N) depending on insertion strategy. This is online, not Mo.

// Accepted submission - SPOJ-RACETIME (preserved verbatim, blank lines collapsed)
#include <bits/stdc++.h>
using namespace std;

const int N = 1 << 18, M = ceil(sqrt(N));

int arr[N], n, q, sq;
vector<int> blocks[M];

int countLess(int s, int e, int x) {
    int cnt = 0;
    for (int i = s; i <= e; i++)
        cnt += (arr[i] <= x);
    return cnt;
}

int main() {
    scanf("%d%d", &n, &q);
    sq = sqrt(n);
    for (int i = 0; i < n; i++) {
        scanf("%d", arr + i);
        blocks[i / sq].push_back(arr[i]);
    }
    for (int i = 0; i < n; i += sq)
        sort(blocks[i / sq].begin(), blocks[i / sq].end());

    while (q--) {
        char c;
        scanf(" %c", &c);
        if (c == 'C') {
            int s, e, x;
            scanf("%d%d%d", &s, &e, &x);
            s--, e--;
            int cnt = 0;
            if (s / sq == e / sq) cnt = countLess(s, e, x);
            else {
                cnt = countLess(s, (s / sq + 1) * sq - 1, x);
                cnt += countLess(e / sq * sq, e, x);
                for (int i = s / sq + 1; i < e / sq; ++i)
                    cnt += upper_bound(blocks[i].begin(), blocks[i].end(), x) - blocks[i].begin();
            }
            printf("%d\n", cnt);
        }
        else {
            int p, x;
            scanf("%d%d", &p, &x);
            p--;
            int b = p / sq;
            auto it = lower_bound(blocks[b].begin(), blocks[b].end(), arr[p]);
            *it = x;
            sort(blocks[b].begin(), blocks[b].end());
            arr[p] = x;
        }
    }
    return 0;
}

Family relation: shares the block-decomposition skeleton with Mo but does not reorder queries. The maintained "answer per block" is the sorted vector; the boundary scan is the partial-block correction.

📐 Adjacent Technique — SPOJ SWAPS (Block + Fenwick Hybrid)

VJudge: https://vjudge.net/problem/SPOJ-SWAPS

The problem maintains an array under point assignments and reports the total inversion count after each. Each block keeps a Fenwick tree indexed by value; counting inversions contributed by one index uses three pieces — left side of the same block (linear), right side of the same block (linear), and the full Fenwicks of the other blocks (one Fenwick query per block). Updates patch the single block's Fenwick and recompute the contribution of one index.

// Accepted submission - SPOJ-SWAPS (preserved verbatim from source; blank lines collapsed)
#include <bits/stdc++.h>
using namespace std;

#define F first
#define S second
#define pb push_back
#define ll long long
#define all(x) x.begin() , x.end()
#define rep(i,s,e) for (int i = s; i < e; ++i)
#define rev(i,s,e) for (int i = s; i > e; --i)

const int N = 5e4 + 12, M = 3e5;

struct BIT {
    int bit[N];
    void init() { memset(bit, 0, sizeof bit); }
    int f(int n) { return (n & -n); }
    void add(int idx, int val) {
        for (++idx; idx < N; idx += f(idx)) bit[idx] += val;
    }
    int sum(int idx) {
        int ret = 0;
        for (++idx; idx > 0; idx -= f(idx)) ret += bit[idx];
        return ret;
    }
};

BIT bucket[515];
BIT invBIT;

int n, q, sq, mx, a[M];

int countInversion(int idx) {
    int curBucket = idx / sq;
    int ret = 0;
    for (int i = idx - 1; (i / sq) == curBucket and i >= 0; --i) ret += (a[i] > a[idx]);
    for (int i = idx + 1; (i / sq) == curBucket and i <  n; ++i) ret += (a[i] < a[idx]);
    for (int i = curBucket - 1; i >= 0; --i) {
        ret += bucket[i].sum(N - 5) - bucket[i].sum(a[idx]);
    }
    for (int i = curBucket + 1; i <= mx; ++i) {
        ret += bucket[i].sum(a[idx] - 1);
    }
    return ret;
}

int main() {
    ios::sync_with_stdio(0); cin.tie(0); cout.tie(0);
    cin >> n;
    if (n == 0) return 0;
    sq = sqrt(n);
    mx = (n / sq) + (n % sq ? 1 : 0);
    rep(i, 0, mx + 1) bucket[i].init();

    ll invCnt = 0;
    rep(i, 0, n) {
        cin >> a[i];
        bucket[i / sq].add(a[i], 1);
        invCnt += invBIT.sum(N - 5) - invBIT.sum(a[i]);
        invBIT.add(a[i], 1);
    }

    cin >> q;
    while (q--) {
        int idx, val;
        cin >> idx >> val; --idx;
        invCnt -= countInversion(idx);
        bucket[idx / sq].add(a[idx], -1);
        a[idx] = val;
        bucket[idx / sq].add(a[idx], +1);
        invCnt += countInversion(idx);
        cout << invCnt << "\n";
    }
    return 0;
}

Family relation: sqrt-decomposition on positions with a Fenwick layer per block. Online and update-supportive — neither of which Mo gives you. The contribution computation in countInversion is the "boundary plus block sweep" pattern at its purest.

📐 Adjacent Technique — CodeChef CHSQARR (2-D Sliding Window)

VJudge: https://vjudge.net/problem/CodeChef-CHSQARR

This problem is not Mo at all. It belongs here because it illustrates the same competitive-programming theme: "avoid recomputing every window from scratch by maintaining reusable local structure." A monotonic queue gives sliding maxima in O(1) amortized; combining row-wise sliding windows with column-wise sliding windows answers each a × b rectangle query in linear time per query.

// Accepted submission - CodeChef-CHSQARR (preserved verbatim from source; blank lines collapsed)
#include "bits/stdc++.h"
using namespace std;

const int N = 1e3 + 5;
int mat[N][N];
pair<int,int> row[N][N];
int n, m, q, a, b;

struct mQueue {
    deque<int> dq;
    queue<int> q;

    int size() { return q.size(); }
    void push(int x) {
        q.push(x);
        while (dq.size() && dq.back() < x) {
            dq.pop_back();
        }
        dq.push_back(x);
    }
    void pop() {
        if (dq.front() == q.front()) {
            dq.pop_front();
        }
        q.pop();
    }
    int& front() { return q.front(); }
    int getMax() { return dq.front(); }
    bool empty() { return q.empty(); }
};

int main() {
    scanf("%d %d", &n, &m);
    for (int i = 0; i < n; i++) {
        for (int j = 0; j < m; j++) {
            scanf("%d", &mat[i][j]);
        }
    }
    scanf("%d", &q);
    while (q--) {
        scanf("%d %d", &a, &b);
        int res = 2e9;
        for (int i = 0; i < n; i++) {
            mQueue qe;
            int sum = 0;
            for (int j = 0; j < b; j++) {
                sum += mat[i][j];
                qe.push(mat[i][j]);
            }
            row[i][0] = {sum, qe.getMax()};
            for (int j = b; j < m; j++) {
                sum += mat[i][j] - mat[i][j - b];
                qe.pop();
                qe.push(mat[i][j]);
                row[i][j - b + 1] = {sum, qe.getMax()};
            }
        }
        for (int j = 0; j <= m - b; j++) {
            mQueue qe;
            int sum = 0;
            for (int i = 0; i < a; i++) {
                sum += row[i][j].first;
                qe.push(row[i][j].second);
            }
            res = min(res, qe.getMax() * a * b - sum);
            for (int i = a; i < n; i++) {
                sum += row[i][j].first - row[i - a][j].first;
                qe.pop();
                qe.push(row[i][j].second);
                res = min(res, qe.getMax() * a * b - sum);
            }
        }
        cout << res << endl;
    }
    return 0;
}

Family relation: none, really. CHSQARR illustrates the "maintain reusable local structure as the window moves" mindset that Mo also embodies, but the algorithm itself is a textbook 2-D sliding window with monotonic queues and is faster than any sqrt approach would be for the same problem. It is included here as a contrast: not every sqrt-like problem deserves a sqrt-shaped solution.

Hard reflex rules

Use rollback Mo only when the data structure is add-only and add is genuinely O(α) or O(log n); the overhead of snapshots is otherwise not worth it. Reach for 4D Mo only when the problem cannot be re-cast as a sequence of two 2D Mo passes; in many "two subtree" problems you can answer each subtree independently and combine at the end. Use the online block-precomputed variant when reordering is explicitly forbidden; if the problem is not online, prefer regular Mo because its constants are smaller.

📝 Final Summary — End of the Family

This series has now covered the entire Mo family: classic array Mo, Mo on trees, Mo with updates, rollback Mo, 4D Mo, and the online block-precomputed variant. The master template stayed structurally identical throughout; the variants are all controlled mutations of the same engine. Block size, comparator dimensionality, and the answer layer adapt; the four-loop pointer movement does not.

The three checkpoints to internalize before submitting any sqrt-decomposition solution:

1. Can the data structure be both add'd to and removed from in O(F)?
   No → rollback Mo (delete = rollback) or another sqrt technique.

2. Are the queries offline and is reordering allowed?
   No → online block-precomputed; do not use Mo.

3. How many axes does the query state have?
   1 axis (range)         → classic Mo (or Mo on trees if the data is a tree).
   2 axes (range + time)  → 3D Mo.
   2 ranges (4 endpoints) → 4D Mo.

That decision tree, applied honestly, will pick the right tool every time.
