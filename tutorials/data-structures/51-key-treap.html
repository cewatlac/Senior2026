Key Treap with Rotations and frq

A randomized balanced BST in fewer than 250 lines. Combine the BST invariant on values with a heap invariant on random priorities, and you get expected O(log n) for insert, erase, count, kth, and order_of_key, all without AVL or Red-Black logic.
Theme

📌 01 · Problem Definition

Maintain an ordered multiset of integers under arbitrary update sequences and answer rich order-statistic queries in expected logarithmic time. The data structure must support insertion of a new element, removal of one occurrence of an element, membership test, count of occurrences of a specific value, k-th smallest extraction, the rank query order_of_key(x) giving the number of elements strictly less than x, and predecessor and successor lookups. Every operation must run in expected O(log n) time, where n is the current cardinality of the multiset, and the structure must remain robust against adversarial input orders such as monotonically sorted insertions.

A plain BST collapses to a chain on sorted input. A strict balanced BST like AVL or Red-Black is correct but cumbersome to write under contest pressure. The key treap with rotations is the middle path: it gives the same expected bounds with far less code, and a single random priority per node replaces all of the height bookkeeping or color logic.

🧠 02 · Intuition from Scratch

A classical result on random BSTs says that if n keys are inserted into an initially empty BST in a uniformly random order, the expected height of the resulting tree is O(log n) and the expected depth of any individual key is about 2 ln n. The problem with the plain BST in practice is not the structure itself but the fact that the user controls the insertion order, and an adversarial order such as a monotonically increasing sequence produces a chain. The treap turns this around: it gives each node a random priority the moment the node is created, and then maintains an additional invariant that the tree is also a max-heap on priorities. Because the priorities are random, the shape of the tree depends only on the random priorities, not on the order in which keys arrived.

The name

The word treap is a portmanteau of tree and heap. Each node has a key value v obeying the BST order on values, and a priority pri obeying the max-heap order on priorities. The same node sits in the same tree under two simultaneous orderings.

Why two invariants pin down the shape uniquely

Given a set of (key, priority) pairs with all priorities distinct, exactly one binary tree satisfies both orderings simultaneously. The proof is constructive: the node with the highest priority must be the root by the heap property, and the BST property then partitions the remaining nodes into those with smaller keys (left subtree) and those with larger keys (right subtree). Recursing into each side produces a unique tree. If we pick the priorities uniformly at random, this unique tree is exactly the tree we would get by inserting the keys into a plain BST in the order of decreasing priority, which is a uniformly random order. Hence the expected height is O(log n) regardless of the order the user provided.

🎨 03 · Visual Mental Model

Visualise each node as a labelled circle carrying two numbers: the key on top and the priority below it. The BST invariant orders the keys left-to-right along the inorder direction. The heap invariant orders the priorities top-to-bottom along the parent-child direction. Both invariants live in the same tree at the same time.

A treap for values 3, 5, 8 with priorities 30, 50, 80

A key treap stores each distinct value as one node and uses an integer field frq to remember how many copies of that value the multiset contains. Inserting 5 three times produces a single node with v = 5 and frq = 3. The subtree-size field sz sums frequencies of all descendants, so kth continues to work correctly because it locates the k-th occurrence rather than the k-th distinct value.

📐 04 · Formal Abstraction

Node fields

Each node carries six pieces of information that together encode both invariants and enable order-statistic queries. The value v is the key stored at this node. The priority pri is a random integer drawn once at creation and never changed. The frequency frq counts how many copies of v exist in the multiset and is at least one for every live node. The subtree size sz sums frequencies across all descendants by the rule sz = sz(left) + frq + sz(right). The parent pointer p and child pointers ch[0], ch[1] wire the tree together; missing pointers reference a single static EMPTY sentinel rather than null.

The three invariants

BST invariant: for every node u, every node in ch[0]'s subtree has v < u.v every node in ch[1]'s subtree has v > u.v heap invariant: for every node u, u.pri >= ch[0].pri AND u.pri >= ch[1].pri size invariant: for every node u, u.sz = ch[0].sz + u.frq + ch[1].sz

Sentinel design

Rather than checking for nullptr everywhere, we use a single static EMPTY node whose children point to itself, whose sz and frq are zero, and whose pri is zero. Every leaf points its children at EMPTY, so reads like u->ch[L]->sz are always safe and we never branch on a null check.

🔍 05 · Key Observations

Random priority defeats adversarial input
The user controls the keys but not the priorities. Since the shape of the unique valid treap depends only on the priorities, the user cannot craft a worst-case input, and the expected height is O(log n) for any input order.
Rotations preserve inorder
Repair after insert and during erase happens through left or right rotations, which we already proved preserve inorder in the BST tutorial. The BST invariant survives every rotation automatically.
frq is a compression trick
Storing one node per distinct value rather than one per occurrence keeps the tree small when duplicates are common. The sz field counts occurrences (frequency-weighted) so kth still locates the k-th occurrence, not the k-th distinct value.
Locality of rebalancing
After inserting a new node, only nodes on the path from that node to the root can possibly violate the heap invariant. We rotate the new node upward until its parent's priority dominates or until it becomes the root, performing at most O(h) rotations.

⚙️ 06 · Algorithm

Insert: walk down, plant, rotate up

To insert a value x, we walk down the tree following BST descent. If we encounter a node whose value equals x, we increment that node's frq and update sz on the ancestors. If we walk off the tree without finding x, we create a new node with a freshly drawn random priority and link it in place of the null pointer we reached. The new node is at the correct leaf position for the BST invariant but may violate the heap invariant if its priority is larger than its parent's. To repair, we rotate the new node upward repeatedly: while the new node's priority exceeds its parent's, we perform a rotation that lifts the new node one level. Each rotation preserves the BST invariant (as shown in tutorial 26) and reduces the heap violation by one level. We stop when the parent's priority is at least the child's or when the child has no parent.

Inserting value 5 with priority 80 into a tree whose root has priority 50

Erase: rotate the doomed node down to a leaf, then detach

To erase one occurrence of x, we walk down to find the node carrying x. If that node has frq > 1 we simply decrement frq and update sz on the ancestors. If frq equals one, we cannot simply splice up a child because the doomed node may have two children, and splicing either up would break the heap invariant at the other side. The treap trick: rotate the doomed node downward until it becomes a leaf. At each step, the rotation chooses the child with the higher priority so that the heap invariant is restored in the local subtree after the doomed node moves down. Once the doomed node has no children, we detach it from its parent. The descent and the rotation chain together still take O(h) work.

Erasing the root by rotating it down to a leaf, then detaching

kth: descend using subtree sizes

To find the k-th smallest occurrence (k is one-indexed), we descend from the root maintaining the convention that k is the rank we are still looking for within the current subtree. At each node we compare k against the size of the left subtree. If k ≤ sz(left) we descend into the left subtree with k unchanged. If sz(left) < k ≤ sz(left) + frq the answer is the current node's value. Otherwise we subtract sz(left) + frq from k and descend into the right subtree. The descent is O(h) and uses only one comparison per level.

kth descent using subtree sizes

order_of_key, first_ge, last_le

The order_of_key query for a value x returns the number of multiset elements strictly less than x. We descend from the root: at each node, if x ≤ u.v we descend left without accumulating anything; if x > u.v we add sz(left) + frq to the running total and descend right. The descent is O(h) and the running total at the end is the answer. The lower-bound queries first_ge(x) and last_le(x) are analogous BST descents that remember the most recent candidate that satisfies the inequality.

📊 07 · Invariants and Transitions

State invariants

At every quiescent moment the treap satisfies the three invariants of section 4. The BST invariant constrains the relative order of values, the heap invariant constrains the relative order of priorities, and the size invariant says sz aggregates frequencies along subtrees. Any algorithm that mutates the tree must restore all three by the time it returns.

Transition: a single rotation

A rotation at a node is a local rewiring of three child pointers (and the matching parent pointers, plus a child slot update at the grandparent). The transition preserves the BST invariant globally because inorder is unchanged. The transition preserves the size invariant locally if we call a pull(u) recomputation on the two affected nodes (the one that descended and the one that rose) in the correct order: first pull the node that descended, then pull the node that rose. The transition may temporarily break the heap invariant at the new local root if its parent's priority is smaller; that violation is what insert chases up the tree until it disappears.

Termination of insert

The insert procedure rotates the new node upward as long as the new node has a parent whose priority is strictly smaller. Each rotation reduces the new node's depth by one. The depth is bounded below by zero (root), so insert performs at most h rotations, where h is the height of the tree at the moment of insertion. The expected height is O(log n), so the expected number of rotations is O(log n).

Termination of erase

The erase procedure rotates the doomed node downward as long as the doomed node has at least one child whose priority is strictly larger. Each rotation increases the doomed node's depth by one and either leaves the number of its children unchanged or reduces it by one. Eventually the doomed node has no children and is detached. The total number of rotations is again bounded by the height of the original tree, giving expected O(log n) work.

📜 08 · Proof of Correctness

Insert preserves the BST invariant

The descent phase of insert places the new leaf at a position consistent with the BST invariant by the same argument used in the plain BST: we go left on strict-less-than and right on strict-greater-than, so the new leaf is smaller than every ancestor we went right from and larger than every ancestor we went left from. The rotation phase preserves the BST invariant by the inorder-preservation property of rotations.

Insert restores the heap invariant

After the descent, the only place the heap invariant might be violated is at the new node and its parent. The rotate-up loop rotates the new node above any parent whose priority is smaller. After each rotation, the new node has a new parent (its grandparent before the rotation) and the loop continues. The loop terminates when the parent's priority is at least the new node's or when the new node has no parent. At termination, the heap invariant holds at the new node. The heap invariant elsewhere in the tree is unchanged because the rotation only affected the two nodes involved.

Erase preserves all three invariants

If the doomed node had frq > 1 before erase, decrementing frq and pulling sz up the ancestors preserves everything. If frq dropped to zero, the rotate-down loop rotates the doomed node toward the higher-priority child at each step. The rotation preserves BST and reduces the local heap violation: the higher-priority child rises above the doomed node, which means the heap invariant is restored at the rising node (because its other child still has lower priority by the heap invariant at the higher-priority child before rotation). The doomed node now sits in the position where the rising child was, with possibly a new subtree to descend into. After O(h) rotations the doomed node becomes a leaf and is detached.

kth, order_of_key, first_ge, last_le are correct by structural induction

Each of these descents uses only the BST invariant and the size invariant. The kth descent's case analysis exactly mirrors the inorder layout of the subtree (left subtree first, then the current node's frq occurrences, then the right subtree). The order_of_key descent counts every occurrence of a value strictly less than x by accumulating sz(left) + frq whenever the descent goes right and ignoring elements on the right whenever the descent goes left. The first_ge and last_le descents remember the most recent candidate and update it when crossing the comparison boundary.

⏱️ 09 · Complexity Analysis

The expected height of a treap on n nodes is O(log n) when priorities are chosen uniformly at random. The expected depth of any specific key is approximately 2 ln n, the same bound as for a random BST. From this single fact every operation cost follows: each operation walks one root-to-leaf path (possibly with a chain of rotations along it), so its expected cost is O(log n).

OperationExpected timeWorst caseWhat happens
insert(x)O(log n)O(n)BST descent then rotate-up
erase(x)O(log n)O(n)BST descent then rotate-down to leaf
count(x)O(log n)O(n)BST descent
kth(k)O(log n)O(n)size-driven descent
order_of_key(x)O(log n)O(n)BST descent accumulating
first_ge(x), last_le(x)O(log n)O(n)BST descent remembering candidate
inorder traversalO(n)O(n)visit every node once

The worst case occurs with vanishing probability and only when an adversary somehow learns the priority sequence in advance. In contest contexts this is impossible; the priorities come from a thread-local random source seeded by the system clock. The amortised constants in the expected case are small enough that the key treap competes with red-black trees on real workloads.

🧱 10 · Template Philosophy

The template style we adopt has four design choices that make the code short and bug-resistant.

Choice 1: single static sentinel EMPTY

Rather than checking for nullptr at every pointer dereference, we maintain a single static EMPTY node whose children point to itself, with sz = 0, frq = 0, pri = 0. Every leaf points its children at EMPTY. Reads like u->ch[L]->sz are always safe and produce zero on an empty subtree, which is exactly what the size formulas need. Writes still must avoid mutating EMPTY; the only invariant is that its fields stay at their zero values.

Choice 2: rotate(child) instead of rotate(parent, direction)

The classical formulation rotates at a parent and asks for a direction. We use the alternative formulation: rotate(c) rotates the child c above its parent without passing an explicit direction. The direction is recovered from the parent's child slots. This eliminates one source of bugs (passing the wrong direction) and makes the rotate-up loop in insert read like while (c->p->pri < c->pri) rotate(c); which is one of the shortest possible expressions of the rotate-up phase.

Choice 3: parent pointers

We maintain a parent pointer on every node. The cost is one extra word per node and a few extra assignments per rotation; the benefit is that the rotate-up loop in insert and the rotate-down loop in erase become straightforward without an explicit stack. Implicit treaps (tutorial 28) drop parent pointers because the rebalance logic there is structurally different; for the key treap, parent pointers are the right trade-off.

Choice 4: a link(p, c, dir) helper that pulls automatically

Every pointer mutation goes through link(p, c, dir), which sets p->ch[dir] = c, sets c->p = p if c is not the sentinel, and finally calls pull(p) to recompute p->sz. By funneling all writes through one helper, we cannot forget to maintain parent pointers or to refresh subtree sizes. The whole class becomes nearly write-correct by construction.

🎯 11 · Master Template — Ready to Submit

The complete key treap with rotations and frequency-counted duplicates, written in the rotate(child) / link(p,c,dir) style described above. Around 220 lines including I/O scaffolding. Drop into a contest and the only thing you may need to change is the type of the value (templated on a single integer-comparable type).

// Key Treap with frq and rotations — rotate(child) style
#include <bits/stdc++.h>
using namespace std;

mt19937 rng(chrono::steady_clock::now().time_since_epoch().count());

struct node {
    int v;        // the key stored here
    int pri;      // random priority drawn once at creation
    int sz;       // sum of frq in subtree
    int frq;      // number of occurrences of v (key treap of multiset)
    node *p, *ch[2]; // parent, left child (0), right child (1)
};

static node EMPTY_STORAGE;
node* const EMPTY = &EMPTY_STORAGE;

void init_empty(){
    EMPTY->v = 0; EMPTY->pri = 0; EMPTY->sz = 0; EMPTY->frq = 0;
    EMPTY->p = EMPTY; EMPTY->ch[0] = EMPTY; EMPTY->ch[1] = EMPTY;
}

struct KeyTreap {
    node* root;
    KeyTreap() : root(EMPTY) { init_empty(); }

    node* make_node(int v){
        // Real-node priorities live in [1, INT_MAX] so they always strictly beat the
        // sentinel's pri=0. Casting raw rng() to int risks negative priorities, which
        // would let EMPTY beat a real child during the erase rotate-down loop.
        static uniform_int_distribution<int> dist(1, INT_MAX);
        node* u = new node{v, dist(rng), 1, 1, EMPTY, {EMPTY, EMPTY}};
        return u;
    }

    // Recompute sz from children plus own frq.
    void pull(node* u){
        if (u == EMPTY) return;
        u->sz = u->ch[0]->sz + u->frq + u->ch[1]->sz;
    }

    // Set p's child slot dir to c, set c's parent to p (if c isn't EMPTY), refresh p's sz.
    void link(node* p, node* c, int dir){
        p->ch[dir] = c;
        if (c != EMPTY) c->p = p;
        pull(p);
    }

    // Recover the child slot index (0 or 1) that c occupies in its parent.
    int getDir(node* c){
        return c->p->ch[1] == c ? 1 : 0;
    }

    // Rotate child c above its parent. The direction is inferred from c's slot.
    void rotate(node* c){
        node* p = c->p;
        node* g = p->p;
        int dir = getDir(c);
        int other = dir ^ 1;
        // Save p's old direction under g BEFORE rewiring. After link(c, p, other)
        // fires below, p is no longer a child of g, so getDir(p) would report
        // p's new slot under c instead of its original slot under g — the bug.
        int pdir = (g == EMPTY ? -1 : getDir(p));
        // c's other-side child becomes p's dir-side child.
        link(p, c->ch[other], dir);
        // p becomes c's other-side child.
        link(c, p, other);
        // c takes p's old slot under g; or becomes root if p was root.
        if (g == EMPTY) { root = c; c->p = EMPTY; }
        else link(g, c, pdir);
    }

    // Insert one occurrence of v into the treap.
    void insert(int v){
        if (root == EMPTY) { root = make_node(v); return; }
        node* u = root;
        node* parent = EMPTY;
        int dir = 0;
        while (u != EMPTY){
            if (u->v == v){ u->frq++; for (node* w = u; w != EMPTY; w = w->p) pull(w); return; }
            parent = u; dir = v > u->v;
            u = u->ch[dir];
        }
        node* x = make_node(v);
        link(parent, x, dir);
        // rotate x up while x's priority beats its parent's priority.
        while (x->p != EMPTY && x->pri > x->p->pri) rotate(x);
        // after rotation, root may have changed; refresh sz on the spine above x.
        for (node* w = x->p; w != EMPTY; w = w->p) pull(w);
    }

    // Erase one occurrence of v from the treap (no-op if v is not present).
    void erase(int v){
        node* u = root;
        while (u != EMPTY && u->v != v) u = u->ch[v > u->v];
        if (u == EMPTY) return;
        if (u->frq > 1){ u->frq--; for (node* w = u; w != EMPTY; w = w->p) pull(w); return; }
        // rotate u down until it becomes a leaf, choosing the higher-priority child to rotate up.
        while (u->ch[0] != EMPTY || u->ch[1] != EMPTY){
            int dir = u->ch[0]->pri > u->ch[1]->pri ? 0 : 1;
            rotate(u->ch[dir]);
        }
        node* par = u->p;
        if (par == EMPTY) root = EMPTY;
        else link(par, EMPTY, getDir(u));
        for (node* w = par; w != EMPTY; w = w->p) pull(w);
        delete u;
    }

    // Number of occurrences of v.
    int count(int v){
        node* u = root;
        while (u != EMPTY && u->v != v) u = u->ch[v > u->v];
        return u == EMPTY ? 0 : u->frq;
    }

    // Number of multiset elements strictly less than v.
    int order_of_key(int v){
        node* u = root; int ans = 0;
        while (u != EMPTY){
            if (v <= u->v) u = u->ch[0];
            else { ans += u->ch[0]->sz + u->frq; u = u->ch[1]; }
        }
        return ans;
    }

    // k-th smallest occurrence, 1-indexed; UB if k is out of range.
    int kth(int k){
        node* u = root;
        while (true){
            int ls = u->ch[0]->sz;
            if (k <= ls) u = u->ch[0];
            else if (k <= ls + u->frq) return u->v;
            else { k -= ls + u->frq; u = u->ch[1]; }
        }
    }

    // Smallest value >= q, or INT_MAX if none.
    int first_ge(int q){
        node* u = root; int ans = INT_MAX;
        while (u != EMPTY){
            if (u->v >= q){ ans = u->v; u = u->ch[0]; }
            else u = u->ch[1];
        }
        return ans;
    }

    // Largest value <= q, or INT_MIN if none.
    int last_le(int q){
        node* u = root; int ans = INT_MIN;
        while (u != EMPTY){
            if (u->v <= q){ ans = u->v; u = u->ch[1]; }
            else u = u->ch[0];
        }
        return ans;
    }

    int size(){ return root->sz; }

    void inorder_rec(node* u, vector<int>& out){
        if (u == EMPTY) return;
        inorder_rec(u->ch[0], out);
        for (int i = 0; i < u->frq; ++i) out.push_back(u->v);
        inorder_rec(u->ch[1], out);
    }
    vector<int> inorder(){ vector<int> out; inorder_rec(root, out); return out; }
};

📝 12 · Line-by-Line Walkthrough

The sentinel block

The static EMPTY_STORAGE and the pointer EMPTY together implement the single sentinel. The function init_empty sets the six fields explicitly: value zero (never compared), priority zero (never beats any real node's priority), size zero (so addition on the size invariant gives the right answer for missing children), frequency zero (so the kth descent never selects this node), parent pointing at itself, and both children pointing at itself. Any read of EMPTY->ch[d] returns EMPTY again, which means a descent that walks past a leaf simply stays on the sentinel forever instead of crashing on a null dereference.

For the priority-zero sentinel to be safe, every real node must receive a priority of at least one. The erase routine rotates the doomed node downward by repeatedly choosing whichever child has the larger priority and rotating it up; if the only real child somehow ended up with a priority less than the sentinel's zero, the comparison would pick EMPTY as the larger one and the rotate-down loop would try to rotate the sentinel, which corrupts the treap or hangs. That is exactly why make_node generates priorities from [1, INT_MAX] with a uniform_int_distribution<int> instead of casting raw rng() output directly to int: the cast can produce negative numbers and would silently break the invariant real_pri > EMPTY->pri.

pull and link

The pull function recomputes the subtree size of one node from its children's subtree sizes plus its own frequency. The link function performs the only mutation of child pointers in the entire class. By routing every assignment through link, we guarantee three things: the parent pointer of the child gets updated, the child slot of the parent gets updated, and the subtree size of the parent gets refreshed. This eliminates a whole category of bugs where one of these three actions is forgotten.

The rotate function

The rotate function takes the child c that should rise above its parent. We read the parent p and the grandparent g. The direction dir is recovered from which slot c currently occupies in p. Before any rewiring happens, we also save pdir, which is the slot p currently occupies under g (or -1 if g is EMPTY). This save is essential, and the order matters: the second link call below relinks p to sit as a child of c, so by the time we want to reattach c under g, calling getDir(p) would no longer return p's original slot under g — it would return p's new slot under c. Reading the grandparent attachment direction after the rewiring is logically wrong and can attach the rotated subtree to the wrong side of the grandparent, breaking the BST invariant. Saving pdir up front avoids the issue entirely.

The three pointer updates are then written as three link calls in the correct order: first the middle subtree (c's other-side child) moves to become p's dir-side child, then p moves to become c's other-side child, then c takes p's old place under g using the saved pdir. The final step has two cases: if there was no grandparent, c becomes the new root; otherwise c slots into pdir under g, which is the direction p originally occupied.

The insert function

The insert descent walks down the tree comparing the incoming value v with the current node's value. If we find a match, we increment the frequency and pull up the spine; the insert is done. Otherwise we keep walking until we hit EMPTY, remembering the parent and the direction. We create a new node, link it in, and then rotate it up. The rotate-up loop is just while (x->p != EMPTY && x->pri > x->p->pri) rotate(x); which is two lines including the brace. After the loop terminates, the spine above x may have changed (because x climbed), so we pull sz on every ancestor of x just to be safe; in fact link already pulled most of them, but the cost is logarithmic and the correctness is worth the extra write.

The erase function

Erase walks the tree to find the node carrying v. If not found, we return. If found and the frequency is greater than one, we just decrement and pull the spine. Otherwise we rotate the doomed node downward: at each step we choose the higher-priority child and rotate it up. This pushes the doomed node down by one level, and the higher-priority child takes over as the parent. When the doomed node has neither child left, we detach it by linking EMPTY into the parent's slot. Memory is freed with delete. The pulls along the spine are again just to ensure size correctness.

Query functions

The count function is a plain BST descent that returns the frequency at the matching node or zero if the value is absent. The order_of_key function counts everything strictly less than v: it adds sz(left) + frq whenever the descent crosses to the right, because everything on the left and at the current node is less than v in that case. The kth function compares the running rank k against sz(left) to decide whether to descend left, stop at the current node's run of frequencies, or descend right after subtracting. The first_ge and last_le descents are symmetric: each remembers the most recent candidate that satisfies the inequality and refines it by descending on the side that might give a tighter candidate.

🧩 13 · Edge Cases (Centralized Audit)

  • Empty tree. Both root == EMPTY and any descent terminates immediately at EMPTY. count, order_of_key, first_ge, last_le return their natural empty results.
  • Insert into empty tree. Special-cased at the top of insert: the new node becomes the root and we return without rotating.
  • Insert a duplicate. The descent finds the matching node, increments frq, pulls the spine. No rotation is required because the tree shape is unchanged.
  • Erase a value that is absent. The descent reaches EMPTY and we return immediately. The tree is unchanged.
  • Erase a value with frq > 1. We decrement and pull. The tree shape is unchanged.
  • Erase the root. Handled uniformly by the rotate-down loop; once the root has no children, we set root = EMPTY.
  • kth(k) with k = 0 or k > size. Behavior is undefined; assert at the top in debug builds or document the precondition.
  • Reading sz on EMPTY. Always returns 0 by initialization, so it composes correctly in the size formula.
  • Rotation when c's parent is the root. Handled inside rotate by checking g == EMPTY and updating root directly.
  • Tied priorities. 32-bit random integers from mt19937 are practically distinct on contest sizes (≤ 10^6); on n > 10^9 use 64-bit priorities.

🪲 14 · Common Pitfalls and Bugs

Pitfall 1 — Forgetting that frq counts occurrences, not nodes
The size formula must be sz = sz(left) + frq + sz(right), with the frequency as the contribution of the current node, not the constant one. If you write the plain-BST formula sz = sz(left) + 1 + sz(right), then kth and order_of_key silently return the wrong answers on any input with duplicates.
Pitfall 2 — Mutating the EMPTY sentinel
If you accidentally write to EMPTY->p or to one of EMPTY->ch[i] during a rotation, every future descent will misbehave. The link helper guards against this by checking if (c != EMPTY) c->p = p;; the rotate function similarly avoids reading the grandparent slot when g == EMPTY.
Pitfall 3 — Wrong rotation direction in rotate-down for erase
The doomed node must rotate so that the higher-priority child rises. If you accidentally choose the lower-priority child, the heap invariant breaks at that step and never recovers. Read u->ch[0]->pri > u->ch[1]->pri ? 0 : 1 as "pick the side whose priority is larger; that child should be the new local root".
Pitfall 4 — Forgetting to pull sz after a rotation
The two affected nodes (the one that descended and the one that rose) must have their sz recomputed in the correct order. The link helper bakes this in: every link call ends with a pull, and rotate is just three link calls, so sizes are always correct as long as you go through link.
Pitfall 5 — Comparing pointers when you meant priorities
The expression x->p->pri < x->pri compares priorities; x->p < x compares raw pointer addresses, which is undefined and almost certainly wrong. Use a -Wshadow or careful pair programming pass to catch this.
Pitfall 6 — Returning a stale root after the rotate-up loop
When the new node climbs to the top, the root pointer must be updated. Our rotate function handles this inside the if (g == EMPTY) branch; if you write a different rotate, double-check that the root pointer is reassigned at the right moment.
Pitfall 7 — Drawing priorities with rand() instead of mt19937
The classic rand() has only 15 bits on Windows, which means tied priorities are extremely likely on n > 30000 and the treap degenerates. Always use mt19937 seeded by chrono::steady_clock.

🧭 15 · Decision Guide

When the key treap is the right tool

need k-th smallest or order_of_key ⇒ key treap or PBDS
need k-th occurrence with duplicates (frq) ⇒ key treap (PBDS is strictly distinct)
need to set-merge two ordered sets ⇒ key treap with split/merge
need persistence (historical versions) ⇒ persistent treap (tutorial 29)
need sequence operations (insert at position, range reverse) ⇒ implicit treap (tutorial 28)
no order-statistic queries needed ⇒ std::multiset is shorter

🎭 16 · Variants and Extensions

Variant A — split and merge primitives

Instead of rotate-up and rotate-down, the treap family admits an alternative formulation in terms of two primitives: split(root, x) separates a treap into two treaps, one holding values strictly less than x and one holding values greater-or-equal to x; merge(a, b) combines two treaps where every value in a is strictly less than every value in b. Insert becomes split, build leaf, merge, merge; erase becomes split, split, merge dropping the middle. The split/merge style does not need parent pointers and generalises directly to the implicit treap and to persistent treaps.

Variant B — distinct-only (no frq)

If duplicates are forbidden by the problem statement, drop the frq field and replace sz(left) + frq + sz(right) with sz(left) + 1 + sz(right). Insert refuses to add a duplicate; erase finds at most one matching node and rotates it down to a leaf.

Variant C — augment with subtree sum

Add an integer field sum to every node that stores the sum of values in the subtree (weighted by frq). The pull function becomes sum = sum(left) + frq * v + sum(right). With this augmentation, the treap can answer the sum of the k smallest values via a kth-style descent that accumulates sums. Any commutative-associative aggregate works the same way.

Variant D — finger search

Caching a pointer to the last accessed node enables amortised faster lookups when accesses are spatially close. The descent starts from the cached finger, walks up only as far as necessary, and then descends. Useful in offline workflows where consecutive queries are correlated.

Variant E — tagged augmentation (sum, min, count of a predicate)

Generalise sum to any aggregate. The pull function combines the left, current, and right aggregates with the operator. For weighted counts, store (weight, count) pairs and pull as componentwise addition. The order-statistic queries adapt to "find the smallest value such that the prefix aggregate satisfies a condition" by a kth-style descent on the aggregate.

🎯 18 · Final Summary

One sentence: a treap is a BST plus a random priority heap; the heap invariant breaks the user's ability to make the tree degenerate, giving expected O(log n) for the whole ordered-set menu in fewer than 250 lines of code.

Five things to remember

First, the treap satisfies two invariants at once: BST on values and max-heap on random priorities. Given distinct priorities, the tree shape is uniquely determined by the priorities alone, so random priorities mean random shape, which means expected O(log n) height. Second, insertion does a BST descent and then rotates the new node up while its priority exceeds its parent's. Third, deletion rotates the doomed node down to a leaf, always choosing the higher-priority child to rotate up, and then detaches the leaf. Fourth, the size field sz aggregates frequencies, not node counts, so kth and order_of_key handle multisets directly. Fifth, the rotate(child) style with parent pointers and a single link helper produces the shortest correct code; all mutations route through one function that maintains both pointers and sizes.

Mastery checklist

  • I can write the entire 220-line master template from memory in under 30 minutes.
  • I can hand-trace insert and erase on a small example, showing each rotation.
  • I know why the random priority breaks adversarial input order.
  • I can derive kth and order_of_key from the BST invariant plus the size formula.
  • I can extend the template with subtree sum and use it for sum-of-smallest queries.
