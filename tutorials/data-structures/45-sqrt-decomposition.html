SQRT Decomposition

√n Blocks · Range Queries · Sorted Blocks · Lazy Flags · Split/Build

🧩 §1 Problem Definition

SQRT decomposition is a competitive-programming block technique for answering range queries efficiently while supporting updates. The canonical entry problem is:

Given

An array a[0 … n−1] of integers (n ≤ 3 × 10⁵).

Support

update(idx, v) — set a[idx] = v
query(l, r) — return some aggregate of a[l … r]

The aggregate can be a sum, minimum, count satisfying a predicate, inversion delta, and many others — as long as it can be answered per-block quickly. The naive O(n) scan per query becomes O(nq), which is too slow for n, q = 10⁵.

When SQRT beats everything else

Segment trees and Fenwick trees solve sums/min beautifully, but some aggregates — count of values ≤ x, inversion count with updates, range-set then range-reverse then range-count — are hard or impossible with those structures. SQRT decomposition gives a universal, easy-to-implement O(√n) solution adaptable to complex aggregates with minimal extra thought.

O(√n) per query Simple to code Best for n, q ≤ 3 × 10⁵ Flexible aggregates

💡 §2 Intuition from Scratch

The shelf analogy

Imagine a bookshelf with 16 books. Each has a page count. You want the total pages for books 3–14. The naive approach pulls every book. The SQRT approach groups books into boxes of 4. Each box is labelled with its total pages.

Now "books 3–14" becomes: scan books 3–4 (edge of their box) + read labels on boxes 2 and 3 (full boxes) + scan books 13–14 (edge of their box).

Worked example

a = [5, 8, 6, 3,  2, 7, 2, 6,  7, 1, 7, 5,  6, 2, 3, 2]
     ─── Block 0 ──  ─── Block 1 ──  ─── Block 2 ──  ─── Block 3 ──
     blockSum = 22        17               20               13

Query: sum(2, 13)
  Left partial  (indices 2–3):   a[2]+a[3]       =  6+3  =  9
  Full block 1  (indices 4–7):   blockSum[1]              = 17
  Full block 2  (indices 8–11):  blockSum[2]              = 20
  Right partial (indices 12–13): a[12]+a[13]     =  6+2  =  8
                                                          ────
  Answer                                                    54

Why O(√n)?

Partial scan cost

At most B elements per partial block × 2 partial blocks.

Cost = O(B) = O(√n)

Full block cost

At most K = n/B full blocks, each answered O(1) from stored summary.

Cost = O(K) = O(√n)

Choosing B = √n makes both costs equal. The balance argument: minimise f(B) = 2B + n/B by setting df/dB = 0, which gives B = √(n/2) ≈ √n.

🧠 §3 Visual Mental Model

Four colour-coded blocks for n = 16, B = 4. The query range [2, 13] is bracketed in red. Full middle blocks are used with O(1) summary lookups; boundary blocks are scanned element by element.

🔵 Left Partial Tail

l to end of l's block. Scanned individually. Cost: O(B).

🟢 Full Middle Blocks

Blocks entirely inside [l, r]. Use stored blockSum. Cost: O(K).

🟠 Right Partial Tail

Start of r's block to r. Scanned individually. Cost: O(B).

Corner case: l and r in the same block

When both endpoints share a block, there are no full blocks and no separate partial zones. The same-block guard fires and scans l to r directly in O(B). Without this guard, the middle loop would have inverted bounds and the left partial loop would scan past r, over-counting elements.

📐 §4 Formal Abstraction

Fields

int n;           // array length
int blockSize;   // B ≈ √n
int blockCount;  // K = ⌈n / B⌉
vector<T> a;            // actual values
vector<T> summary;      // one entry per block

Key formulas

// Block containing index i
blockId = i / blockSize;

// Block boundaries
blockStart = b * blockSize;
blockEnd   = min(n, (b+1)*blockSize) - 1;

// Standard block size
blockSize  = (int)sqrt(n) + 1;
blockCount = (n + blockSize - 1) / blockSize;

Why sqrt(n) + 1: when n is not a perfect square, sqrt(n) truncates. Adding 1 guarantees a safe block size. The ceiling-division formula gives the exact number of blocks for any n.

🔍 §5 Key Observations

Observation 1 — Every index belongs to exactly one block

blockId = i / blockSize is a total function on [0, n). A point update at idx touches only block idx/B. This locality is what makes point update cheap.

Observation 2 — A range has at most two partial blocks

The left boundary can fall inside one block; the right boundary can fall inside another. Everything strictly between them is whole blocks. This structural property holds for any interval, regardless of its length.

Observation 3 — Full blocks are the source of speed

Scanning a block costs O(B). Using a full-block summary costs O(1) or O(log B). By making B ≈ √n, the number of full blocks is also ≈ √n, so both zones cost O(√n).

Observation 4 — B is a tunable parameter

B = √n is the theoretical balance point, but practical sweet spots are B = 300, 450, or 700. The right value depends on query/update asymmetry, memory constraints, and cache behaviour. Always benchmark on the actual constraints.

When to deviate from √n

Updates far outnumber queriesSmaller BSmaller block means cheaper per-block repair
Per-block Fenwick uses heavy memorySmaller BFewer blocks, less total memory

Query algorithm — pseudocode

query(l, r):
  lb = l / B,  rb = r / B
  if lb == rb:                          // same-block guard (correctness critical)
      scan a[l..r] directly
      return ans

  // Left partial block: l to end of block lb
  scan a[l .. (lb+1)*B − 1]

  // Full middle blocks — O(1) each from stored summary
  for b in [lb+1 .. rb−1]:  ans += blockSum[b]

  // Right partial block: start of block rb to r
  scan a[rb*B .. r]
  return ans

Point update algorithm

update(idx, newValue):
  b = idx / B
  blockSum[b] -= a[idx]    // remove old contribution first
  a[idx] = newValue
  blockSum[b] += a[idx]    // add new contribution

The subtract-then-assign-then-add idiom avoids recomputing the entire block sum from scratch. For range-sum it gives O(1) update. For range-min the block must be fully rescanned after the update — O(B).

🔒 §7 Invariants

State invariant

At all times, for every block b, the stored summary correctly represents the current values inside it. For range-sum:

blockSum[b] == sum(a[i])  for all i  with  i/B == b

Established by build(). Maintained by every update(). This contract is what makes O(1) full-block lookups correct.

Transition invariant (after update)

After update(idx, v), only block idx/B has changed. Its summary has been decremented by the old value and incremented by the new value. All other blocks remain valid without touching them.

Query invariant (termination)

At the end of query(l, r), every index i ∈ [l, r] has contributed to the answer exactly once. The three zones — left partial, full middle, right partial — are disjoint and their union is exactly [l, r]. No element is missed; none is counted twice.

✅ §8 Proof of Correctness

We prove that query(l, r) returns the exact sum of a[l … r], given that the state invariant holds at call time.

Case 1: leftBlock = rightBlock

The loop for i in [l..r] visits every index in [l, r] exactly once and no index outside it. No blockSum is used. The result is correct by direct inspection.

Case 2: leftBlock < rightBlock

Three zones:

  • Left partial: [l … (lb+1)·B − 1]
  • Full middle: blocks lb+1 … rb−1, using blockSum
  • Right partial: [rb·B … r]

Disjointness: The left partial ends at (lb+1)·B−1, strictly less than (lb+1)·B (start of block lb+1). The right partial starts at rb·B, strictly after rb·B−1 (end of block rb−1). The three zones are pairwise disjoint.

Completeness: Every i ∈ [l, r] falls in exactly one zone. Indices in l's block from l onward → left partial. Indices in r's block up to r → right partial. All indices in strictly interior blocks → full middle loop.

blockSum correctness: By the state invariant, blockSum[b] = sum(a[i]) for all i in block b. Since the full middle blocks are entirely contained in [l, r], using blockSum is equivalent to scanning them. ∎

⏱ §9 Complexity Analysis

Build — O(n)

One pass over all n elements. Each contributes to one blockSum entry.

Point update — O(1) for sum

One subtraction + one addition to blockSum. Becomes O(B) = O(√n) for structures that must rescan the block (min, sorted).

Range query — O(√n)

At most 2B elements scanned (two partial blocks) + at most K = n/B full blocks. With B = √n: 2√n + √n = O(√n).

Memory — O(n)

n raw values + K ≈ √n summary entries. Total O(n).

Balance derivation

Total query cost f(B) = 2B + n/B. Setting df/dB = 2 − n/B² = 0 gives B = √(n/2) ≈ √n. The minimum cost is f(√n) ≈ 2√n + √n = 3√n = O(√n).

📄 §11 Master Template — Point Update + Range Sum

The canonical starting template. Solves CSES 1648 — Dynamic Range Sum Queries without modification.

Build O(n) Update O(1) Query O(√n) Memory O(n)
Full AC-Ready C++ Template click to collapse 
#include <bits/stdc++.h>
using namespace std;

struct SqrtSum {
    int n, blockSize, blockCount;
    vector<long long> a;
    vector<long long> blockSum;

    SqrtSum() {}
    SqrtSum(const vector<long long>& initial) { build(initial); }

    void build(const vector<long long>& initial) {
        a = initial;
        n = (int)a.size();
        blockSize  = (int)sqrt(n) + 1;
        blockCount = (n + blockSize - 1) / blockSize;
        blockSum.assign(blockCount, 0);
        for (int i = 0; i < n; i++)
            blockSum[i / blockSize] += a[i];
    }

    // O(1) — only one block changes
    void update(int idx, long long newValue) {
        int b = idx / blockSize;
        blockSum[b] -= a[idx];
        a[idx] = newValue;
        blockSum[b] += a[idx];
    }

    // O(sqrt(n)) — left partial + full blocks + right partial
    long long query(int l, int r) {
        long long ans = 0;
        int leftBlock  = l / blockSize;
        int rightBlock = r / blockSize;

        // Same block — scan directly (guard against inverted middle loop)
        if (leftBlock == rightBlock) {
            for (int i = l; i <= r; i++) ans += a[i];
            return ans;
        }

        // Left partial block
        int leftEnd = (leftBlock + 1) * blockSize - 1;
        for (int i = l; i <= leftEnd; i++) ans += a[i];

        // Full middle blocks — O(1) each
        for (int b = leftBlock + 1; b <= rightBlock - 1; b++)
            ans += blockSum[b];

        // Right partial block
        int rightStart = rightBlock * blockSize;
        for (int i = rightStart; i <= r; i++) ans += a[i];

        return ans;
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, q;
    cin >> n >> q;

    vector<long long> a(n);
    for (int i = 0; i < n; i++) cin >> a[i];

    SqrtSum ds(a);

    while (q--) {
        int type; cin >> type;
        if (type == 1) {
            int idx; long long val;
            cin >> idx >> val;
            --idx;              // 1-based input → 0-based internal
            ds.update(idx, val);
        } else {
            int l, r; cin >> l >> r;
            --l; --r;
            cout << ds.query(l, r) << '\n';
        }
    }
    return 0;
}

📖 §12 Line-by-Line Code Explanation

Build — establishing the state invariant

blockSize  = (int)sqrt(n) + 1;
blockCount = (n + blockSize - 1) / blockSize;

Adding 1 to sqrt(n) prevents the truncation hazard for non-perfect-square n. The ceiling-division formula gives the exact number of blocks needed even when the last block is shorter than blockSize.

for (int i = 0; i < n; i++)
    blockSum[i / blockSize] += a[i];

One linear pass establishes the state invariant for all blocks simultaneously. O(n).

Update — maintaining the state invariant

int b = idx / blockSize;
blockSum[b] -= a[idx];   // subtract BEFORE changing a[idx]
a[idx] = newValue;
blockSum[b] += a[idx];   // add AFTER the assignment

The order matters. Subtracting after the assignment would subtract the new value and add it again — a no-op that corrupts blockSum. Always subtract old, assign, add new.

Query — same-block guard

if (leftBlock == rightBlock) {
    for (int i = l; i <= r; i++) ans += a[i];
    return ans;
}

Without this guard, the middle-block loop for b in [leftBlock+1 .. rightBlock-1] is vacuous, but the left partial loop would run from l to end of leftBlock, scanning past r into adjacent blocks — over-counting. The guard is a correctness requirement, not just an optimisation.

Query — left and right partial loops

int leftEnd = (leftBlock + 1) * blockSize - 1;
for (int i = l; i <= leftEnd; i++) ans += a[i];
...
int rightStart = rightBlock * blockSize;
for (int i = rightStart; i <= r; i++) ans += a[i];

Since the same-block guard fired for leftBlock = rightBlock, when we reach these loops we know leftBlock < rightBlock. This guarantees leftEnd < n and rightStart ≤ r, so neither loop overruns.

The "last block shorter than B" trap

Never write i < (b+1)*blockSize as a loop bound without a min(n, …) guard. The correct form for iterating a block:

int bEnd = min(n, (b + 1) * blockSize);
for (int i = b * blockSize; i < bEnd; i++) ...

In the master template this never arises because the partial loops end at r or leftEnd (both < n), not at block ends. But it becomes critical in build loops, rebuild loops, and variant update functions.

Use SQRT when you see:

  • n, q ≤ 3 × 10⁵
  • Range query with a non-trivial aggregate
  • "How many values in [l,r] satisfy P?"
  • Range lazy + range count
  • Inversion count with point updates

Avoid SQRT when you see:

  • n, q ≥ 10⁶ (O(√n) too slow)
  • Problem explicitly needs O(log n)
  • 2D range updates and queries
  • Persistent queries
  • Simple range sum with no exotic aggregate

⚠️ Common confusion: "SQRT = only range sum"

Range sum is only the entry example. SQRT decomposition applies to any aggregate where you can answer a full block in O(1) or O(log B) and repair a changed block in O(B). As soon as you ask "What should each block remember?", you are designing a SQRT decomposition.

🔀 §16 Variants and Extensions

Variant 1 — Range Minimum / Maximum expand

Problem: Point update + range minimum query.

What changes: Replace blockSum with blockMin. Full-block query returns blockMin[b] in O(1). Partial-block query scans and takes element-wise min.

Why update is O(√n): After changing one element, the minimum of its block may have changed — either a new smaller value arrived, or the old minimum was deleted. Either way, you must rescan all B elements to find the new minimum.

void update(int idx, int newValue) {
    a[idx] = newValue;
    int b = idx / blockSize;
    int bEnd = min(n, (b + 1) * blockSize);
    // Rebuild block minimum — O(B)
    blockMin[b] = *min_element(a.begin() + b*blockSize, a.begin() + bEnd);
}

int query(int l, int r) {
    int ans = INT_MAX;
    int lb = l/blockSize, rb = r/blockSize;
    if (lb == rb) {
        for (int i = l; i <= r; i++) ans = min(ans, a[i]);
        return ans;
    }
    int leftEnd = (lb+1)*blockSize - 1;
    for (int i = l; i <= leftEnd; i++)       ans = min(ans, a[i]);
    for (int b = lb+1; b < rb; b++)           ans = min(ans, blockMin[b]);
    for (int i = rb*blockSize; i <= r; i++)   ans = min(ans, a[i]);
    return ans;
}
Update O(√n)Query O(√n)
Variant 2 — Sorted Blocks (point update + count ≤ x) expand

Problem: Count elements ≤ x in a range, with point updates. Used in SPOJ RACETIME.

What changes: Each block keeps a sorted copy. Full-block query: binary search with upper_bound. Update: remove old value via lower_bound erase, insert new value via upper_bound insert.

Variant 3 — Lazy Range Add + Point Query expand

Problem: Add x to every element in [l, r]. Query the value at a single index i.

What changes: Store lazy[b] per block. Full-block range-add just increments lazy[b] in O(1). Partial-block range-add modifies individual elements directly. Point query returns a[i] + lazy[i/B].

int lazy[MAXBLK] = {};

void rangeAdd(int l, int r, int x) {
    int lb = l/B, rb = r/B;
    if (lb == rb) { for (int i = l; i <= r; i++) a[i] += x; return; }
    for (int i = l; i < (lb+1)*B; i++)    a[i] += x;   // left partial
    for (int b = lb+1; b < rb; b++)        lazy[b] += x; // full blocks — O(1)
    for (int i = rb*B; i <= r; i++)        a[i] += x;   // right partial
}

int pointQuery(int idx) {
    return a[idx] + lazy[idx / B];  // lazy applied at read time
}

If you also need range-sum queries on top of range-add, push the lazy tag into the elements of the partial blocks before scanning, then rebuild blockSum for those blocks.

Range add O(√n)Point query O(1)
Variant 4 — Per-Block Fenwick for Value Frequencies expand

Problem: Values lie in domain [0, V]. Count elements satisfying a value predicate in a range, with point updates.

What changes: Each block stores a Fenwick tree over value frequencies. Full-block query: bucket[b].countLT(x) in O(log V). Update: remove old value, insert new value into the block's Fenwick tree.

When to use vs sorted blocks: Fenwick trees support O(log V) insert/delete — faster than O(B) vector insert for large blocks. Use when V is small, B is large, or when you need both count < x and count > x efficiently.

This is the structure behind SPOJ SWAPS — the per-block Fenwick trees allow O(√n · log V) computation of the inversion contribution of any single element before and after an update.

Update O(log V)Query O(√n · log V)Memory O(K · V)
Variant 5 — Batch Processing (SQRT of Queries) expand

Idea: Sometimes the data structure is not an array but the operations themselves. Process operations in batches of size ≈ √q.

Pattern:

For each batch of ~sqrt(q) operations:
    Rebuild a strong global structure (e.g. merge sort tree, BIT) from current state.
    Inside the batch:
        For each query: answer = global_answer - correction_from_recent_updates
        Recent updates ≤ sqrt(q) → brute-force correction costs O(sqrt(q)) each.

Total cost: O(√q · rebuildCost + q · √q). For rebuildCost = O(n log n) and q = n = 3×10⁵, this is O(n√n log n) ≈ 5×10⁸ — borderline, but fast with small constants.

Clean offline patternWorks when rebuild is affordable per √q ops
Variant 6 — Light / Heavy Case Splitting expand

Idea: SQRT decomposition over object frequency, not array positions. Choose threshold T ≈ √total appearances S.

Core invariant: If total appearances = S, there are at most S/T heavy objects (each appearing > T times). With T = √S, there are at most √S heavy objects.

Processing:

  • Heavy objects (freq > T): Few of them — precompute answers for all of them.
  • Light objects (freq ≤ T): Each contributes at most T steps — brute-force per query.

Classic use: Offline distinct-values-in-range, or group queries where groups have skewed sizes. The pigeonhole principle guarantees at most √S heavy groups.

Heavy objects ≤ √SLight objects: O(T) cost each
Variant 7 — Split / Build Dynamic Blocks (range-set + range-reverse + count ≥ x) expand

Problem: Support range-set, range-reverse, and range count ≥ x on a dynamic sequence. No simple segment-tree lazy can handle reversal while also counting values above a threshold.

Core idea: Maintain a dynamic list of blocks. To operate on [l, r], call split(l) and split(r+1) to align block boundaries, making [l, r] correspond to whole blocks. Apply lazy flags to those blocks O(1) each. Periodic full rebuild prevents block-count explosion from repeated splits.

Why standard segment tree fails: A lazy reversal tag inverts the order of children at every node it reaches, which changes all path computations below — it cannot be composed simply with other lazy operations without storing the full sequence at each node.

Full C++ Template — DynamicSqrt (Split/Build) expand 
#include <bits/stdc++.h>
using namespace std;

struct DynamicSqrt {

    struct Block {
        vector<int> v;       // logical values
        vector<int> sv;      // sorted copy (for countGE)
        int sz = 0;
        bool hasAssign = false;
        int  assignVal = 0;
        bool reversed  = false;

        Block() {}
        Block(const vector<int>& vals) : v(vals), sz((int)vals.size()) { rebuild(); }

        int size() const { return sz; }

        void rebuild() {
            sz = (int)v.size();
            sv = v;
            sort(sv.begin(), sv.end());
        }

        // Materialise all lazy flags, then rebuild sorted copy
        void push() {
            if (hasAssign) { v.assign(sz, assignVal); hasAssign = false; }
            if (reversed)  { reverse(v.begin(), v.end()); reversed = false; }
            rebuild();
        }

        void applySet(int x) {
            hasAssign = true; assignVal = x; reversed = false;
            v.clear(); sv.clear();
        }
        void applyReverse() { reversed = !reversed; }

        int countGE(int x) const {
            if (hasAssign) return assignVal >= x ? sz : 0;
            return (int)(sv.end() - lower_bound(sv.begin(), sv.end(), x));
        }
    };

    int n = 0, blockSize = 330;
    vector<Block> blocks;

    void build(const vector<int>& a, int B = 330) {
        n = (int)a.size(); blockSize = max(1, B);
        blocks.clear();
        for (int l = 0; l < n; l += blockSize) {
            int r = min(n, l + blockSize);
            blocks.emplace_back(vector<int>(a.begin()+l, a.begin()+r));
        }
    }

    vector<int> materialize() {
        vector<int> out; out.reserve(n);
        for (auto& bl : blocks) { bl.push(); out.insert(out.end(), bl.v.begin(), bl.v.end()); }
        return out;
    }
    void rebuildAll() { auto tmp = materialize(); build(tmp, blockSize); }

    // Returns the block index where logical position `pos` starts.
    // Splits a block in the interior if necessary.
    int split(int pos) {
        if (pos <= 0)  return 0;
        if (pos >= n)  return (int)blocks.size();
        int cur = 0;
        for (int i = 0; i < (int)blocks.size(); i++) {
            int s = blocks[i].size();
            if (pos == cur) return i;
            if (pos < cur + s) {
                blocks[i].push();
                int lLen = pos - cur;
                vector<int> right(blocks[i].v.begin()+lLen, blocks[i].v.end());
                blocks[i].v.resize(lLen); blocks[i].rebuild();
                blocks.insert(blocks.begin()+i+1, Block(right));
                return i + 1;
            }
            cur += s;
        }
        return (int)blocks.size();
    }

    void rangeSet(int l, int r, int x) {
        if (l > r) return;
        int L = split(l), R = split(r+1);
        for (int i = L; i < R; i++) blocks[i].applySet(x);
    }

    void rangeReverse(int l, int r) {
        if (l > r) return;
        int L = split(l), R = split(r+1);
        for (int i = L; i < R; i++) blocks[i].applyReverse();
        reverse(blocks.begin()+L, blocks.begin()+R);
    }

    int rangeCountGE(int l, int r, int x) {
        if (l > r) return 0;
        int L = split(l), R = split(r+1);
        int ans = 0;
        for (int i = L; i < R; i++) ans += blocks[i].countGE(x);
        return ans;
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n; cin >> n;
    vector<int> a(n);
    for (int i = 0; i < n; i++) cin >> a[i];

    DynamicSqrt ds;
    ds.build(a, 330);

    int q; cin >> q;
    for (int op = 1; op <= q; op++) {
        if (op % 600 == 0) ds.rebuildAll(); // prevent block-count explosion

        string type; cin >> type;
        if (type == "get") {
            int l, r, x; cin >> l >> r >> x; --l; --r;
            cout << ds.rangeCountGE(l, r, x) << '\n';
        } else if (type == "set") {
            int l, r, x; cin >> l >> r >> x; --l; --r;
            ds.rangeSet(l, r, x);
        } else {   // "reverse"
            int l, r; cin >> l >> r; --l; --r;
            ds.rangeReverse(l, r);
        }
    }
    return 0;
}
Range set O(√n) Range reverse O(√n) Count GE O(√n) Rebuild every ~600 ops

17.1 — CSES 1648: Dynamic Range Sum Queries

vjudge.net/problem/CSES-1648

Story: An array of integers. Set one element. Query the sum of a range. Answer all queries.

Abstraction: Point update + range sum. Exactly the master template. CSES uses 1-based indices; subtract 1 immediately after reading.

Key pitfall: Use long long — with n = 2 × 10⁵ and values up to 10⁹, the sum can reach 2 × 10¹⁴.

Full AC Solution expand 
#include <bits/stdc++.h>
using namespace std;

struct SqrtSum {
    int n, blockSize, blockCount;
    vector<long long> a;
    vector<long long> blockSum;

    SqrtSum() {}
    SqrtSum(const vector<long long>& initial) { build(initial); }

    void build(const vector<long long>& initial) {
        a = initial;
        n = (int)a.size();
        blockSize  = (int)sqrt(n) + 1;
        blockCount = (n + blockSize - 1) / blockSize;
        blockSum.assign(blockCount, 0);
        for (int i = 0; i < n; i++)
            blockSum[i / blockSize] += a[i];
    }

    // O(1) — only one block changes
    void update(int idx, long long newValue) {
        int b = idx / blockSize;
        blockSum[b] -= a[idx];
        a[idx] = newValue;
        blockSum[b] += a[idx];
    }

    // O(sqrt(n)) — left partial + full blocks + right partial
    long long query(int l, int r) {
        long long ans = 0;
        int leftBlock  = l / blockSize;
        int rightBlock = r / blockSize;

        // Same block — scan directly (guard against inverted middle loop)
        if (leftBlock == rightBlock) {
            for (int i = l; i <= r; i++) ans += a[i];
            return ans;
        }

        // Left partial block
        int leftEnd = (leftBlock + 1) * blockSize - 1;
        for (int i = l; i <= leftEnd; i++) ans += a[i];

        // Full middle blocks — O(1) each
        for (int b = leftBlock + 1; b <= rightBlock - 1; b++)
            ans += blockSum[b];

        // Right partial block
        int rightStart = rightBlock * blockSize;
        for (int i = rightStart; i <= r; i++) ans += a[i];

        return ans;
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, q;
    cin >> n >> q;

    vector<long long> a(n);
    for (int i = 0; i < n; i++) cin >> a[i];

    SqrtSum ds(a);

    while (q--) {
        int type; cin >> type;
        if (type == 1) {
            int idx; long long val;
            cin >> idx >> val;
            --idx;              // 1-based input → 0-based internal
            ds.update(idx, val);
        } else {
            int l, r; cin >> l >> r;
            --l; --r;
            cout << ds.query(l, r) << '\n';
        }
    }
    return 0;
}

17.2 — SPOJ RACETIME: Count ≤ x in Range with Point Updates

vjudge.net/problem/SPOJ-RACETIME

Story: Race times in an array. Operation C l r x: count how many times in [l, r] are ≤ x. Operation M p x: set position p to x.

Why sorted blocks fit: For a full sorted block, upper_bound(sB, x) - sB.begin() counts elements ≤ x in O(log B). Partial blocks scanned directly. Update: erase old value (lower_bound), insert new (upper_bound), update raw array.

Full AC Solution expand 
#include <bits/stdc++.h>
using namespace std;

struct SqrtCountLE {
    int n, blockSize, blockCount;
    vector<int> a;
    vector<vector<int>> sortedBlocks;

    void build(const vector<int>& initial) {
        a = initial;
        n = (int)a.size();
        blockSize  = (int)sqrt(n) + 1;
        blockCount = (n + blockSize - 1) / blockSize;
        sortedBlocks.assign(blockCount, {});
        for (int i = 0; i < n; i++)
            sortedBlocks[i / blockSize].push_back(a[i]);
        for (int b = 0; b < blockCount; b++)
            sort(sortedBlocks[b].begin(), sortedBlocks[b].end());
    }

    int countDirect(int l, int r, int x) {
        int ans = 0;
        for (int i = l; i <= r; i++) if (a[i] <= x) ans++;
        return ans;
    }

    // O(sqrt(n) * log(sqrt(n)))
    int query(int l, int r, int x) {
        int ans = 0;
        int lb = l / blockSize, rb = r / blockSize;
        if (lb == rb) return countDirect(l, r, x);

        int leftEnd = (lb + 1) * blockSize - 1;
        ans += countDirect(l, leftEnd, x);

        for (int b = lb + 1; b < rb; b++)
            ans += (int)(upper_bound(sortedBlocks[b].begin(),
                                     sortedBlocks[b].end(), x)
                         - sortedBlocks[b].begin());

        int rightStart = rb * blockSize;
        ans += countDirect(rightStart, r, x);
        return ans;
    }

    // O(sqrt(n)) — erase old value, insert new value in sorted position
    void update(int idx, int value) {
        int b = idx / blockSize;
        auto& bl = sortedBlocks[b];
        // Remove exactly one occurrence of old value
        bl.erase(lower_bound(bl.begin(), bl.end(), a[idx]));
        // Insert new value maintaining sorted order
        bl.insert(upper_bound(bl.begin(), bl.end(), value), value);
        a[idx] = value;   // always update raw array too
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, q; cin >> n >> q;
    vector<int> a(n);
    for (int i = 0; i < n; i++) cin >> a[i];

    SqrtCountLE ds;
    ds.build(a);

    while (q--) {
        char type; cin >> type;
        if (type == 'C') {
            int l, r, x; cin >> l >> r >> x;
            --l; --r;
            cout << ds.query(l, r, x) << '\n';
        } else {        // 'M'
            int idx, val; cin >> idx >> val;
            --idx;
            ds.update(idx, val);
        }
    }
    return 0;
}

17.3 — SPOJ SWAPS: Inversion Count with Point Updates

vjudge.net/problem/SPOJ-SWAPS

Story: An array. After each point update, output the current total number of inversions (pairs i < j with a[i] > a[j]).

Key observation: Only inversion pairs involving the changed index idx are affected. All other pairs remain unchanged. So: subtract old contribution of idx, update the per-block Fenwick tree, add new contribution of idx.

Contribution formula: For index idx with current value v: count of earlier elements > v (blocks before idx: use Fenwick countGT; same block: direct scan) + count of later elements < v (blocks after idx: use Fenwick countLT; same block: direct scan).

Important assumption

Values must fit in [0, MAXV] = [0, 50000]. If values exceed this range or can be negative, coordinate-compress all initial and update values before passing them to the Fenwick trees.

Full AC Solution expand 
#include <bits/stdc++.h>
using namespace std;
typedef long long ll;

const int MAXN   = 300000 + 5;
const int MAXV   = 50000  + 5;   // values assumed in [0, 50000]
const int MAXBLK = 700;           // ceil(sqrt(300000)) + margin

struct Fenwick {
    int bit[MAXV + 2];
    void add(int idx, int delta) {
        for (++idx; idx <= MAXV + 1; idx += idx & -idx) bit[idx] += delta;
    }
    int prefix(int idx) const {
        if (idx < 0)    return 0;
        if (idx > MAXV) idx = MAXV;
        int s = 0;
        for (++idx; idx > 0; idx -= idx & -idx) s += bit[idx];
        return s;
    }
    int countLE(int x)      const { return prefix(x);     }
    int countLT(int x)      const { return prefix(x - 1); }
    int countGT(int x)      const { return prefix(MAXV) - prefix(x); }
};

int n, q, blockSize, blockCount;
int a[MAXN];
Fenwick bucket[MAXBLK];  // per-block value frequency
Fenwick all;             // global (used only for initial inversion build)

// Number of inversion pairs that involve index idx in the current array
ll contribution(int idx) {
    int b      = idx / blockSize;
    int bStart = b * blockSize;
    int bEnd   = min(n, (b + 1) * blockSize);
    int val    = a[idx];
    ll ans = 0;

    // Same-block: direct scan
    for (int i = bStart; i < idx;   i++) if (a[i] > val)  ans++;
    for (int i = idx + 1; i < bEnd; i++) if (a[i] < val) ans++;

    // Other blocks: use per-block Fenwick trees
    for (int bi = 0;     bi < b;          bi++) ans += bucket[bi].countGT(val);
    for (int bi = b + 1; bi < blockCount; bi++) ans += bucket[bi].countLT(val);
    return ans;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    cin >> n;
    if (!n) return 0;

    blockSize  = (int)sqrt(n) + 1;
    blockCount = (n + blockSize - 1) / blockSize;

    ll inv = 0;
    for (int i = 0; i < n; i++) {
        cin >> a[i];
        int b = i / blockSize;
        bucket[b].add(a[i], 1);
        inv += i - all.countLE(a[i]);  // elements already seen that are > a[i]
        all.add(a[i], 1);
    }

    cin >> q;
    while (q--) {
        int idx, nv; cin >> idx >> nv;
        --idx;                          // 1-based → 0-based

        inv -= contribution(idx);
        bucket[idx / blockSize].add(a[idx], -1);
        a[idx] = nv;
        bucket[idx / blockSize].add(a[idx], +1);
        inv += contribution(idx);

        cout << inv << '\n';
    }
    return 0;
}

17.4 — Codeforces 13E: Holes

vjudge.net/problem/CodeForces-13E

Story: There are n holes in a row, numbered 1..n. Each hole i has a jump length a[i]. From hole i, a ball jumps to hole i + a[i]. If i + a[i] > n, the ball exits the system. Query type 1 asks: starting at hole p, where does the ball exit and how many jumps does it make? Query type 0 updates a[p] to a new value x.

Abstraction: Decompose positions into blocks of size B ≈ √n. For each position i, precompute three values inside its block:

  • nxt[i] — the first position outside i's block reached by repeatedly jumping from i within the block, or n if the ball exits the array.
  • cnt[i] — the number of jumps needed to reach nxt[i].
  • last[i] — the last position inside the block visited before leaving it.

A query then travels block-by-block via nxt[], accumulating cnt[] in O(1) per block. An update rebuilds only the changed block in O(B) by processing from right to left so later positions are already correct when earlier positions reference them.

Why SQRT decomposition fits:

  • Direct simulation can be O(n) per query — too slow for n, q ≤ 10⁵.
  • Block compression replaces many internal jumps with a single O(1) lookup per block.
  • Each query crosses at most K = n/B = O(√n) blocks.
  • Each update rebuilds O(B) = O(√n) positions in exactly one block.

Invariant: block summary correctness

For every position i, nxt[i], cnt[i], and last[i] correctly describe the path from i to the first exit point of i's block, under the current values of a[]. This invariant is established by build() and restored by every call to rebuildBlock() in update().

rebuildBlock correctness — right-to-left order

When rebuilding block b from right (index r) to left (index l), the invariant for position i + a[i] is already established before we process i. This means the chain nxt[i] = nxt[to], cnt[i] = cnt[to] + 1, last[i] = last[to] is valid whenever to = i + a[i] is still inside the same block. If to exits the block or the array, the base case applies directly. ∎

Full AC Solution — SqrtHoles click to expand 
#include <bits/stdc++.h>
using namespace std;

struct SqrtHoles {
    int n;
    int blockSize;
    int blockCount;

    vector<int> a;
    vector<int> nxt;   // first position outside block b after jumping from i
    vector<int> cnt;   // number of jumps needed to reach nxt[i]
    vector<int> last;  // last position visited inside block b before leaving

    SqrtHoles() {}
    SqrtHoles(const vector<int>& initial) { build(initial); }

    int blockId(int i)    const { return i / blockSize; }
    int blockStart(int b) const { return b * blockSize; }
    int blockEnd(int b)   const { return min(n, (b + 1) * blockSize) - 1; }

    // Rebuild from right to left so later positions are already correct
    void rebuildBlock(int b) {
        int l = blockStart(b);
        int r = blockEnd(b);
        for (int i = r; i >= l; i--) {
            int to = i + a[i];
            if (to >= n) {
                // Jump exits the array entirely
                nxt[i]  = n;
                cnt[i]  = 1;
                last[i] = i;
            } else if (blockId(to) != b) {
                // Jump exits the current block
                nxt[i]  = to;
                cnt[i]  = 1;
                last[i] = i;
            } else {
                // Jump stays inside the block — chain through precomputed values
                nxt[i]  = nxt[to];
                cnt[i]  = cnt[to] + 1;
                last[i] = last[to];
            }
        }
    }

    void build(const vector<int>& initial) {
        a = initial;
        n = (int)a.size();
        blockSize  = (int)sqrt(n) + 1;
        blockCount = (n + blockSize - 1) / blockSize;
        nxt.assign(n, n);
        cnt.assign(n, 0);
        last.assign(n, 0);
        for (int b = 0; b < blockCount; b++)
            rebuildBlock(b);
    }

    // O(sqrt(n)) — rebuild only the one changed block
    void update(int idx, int value) {
        a[idx] = value;
        rebuildBlock(blockId(idx));
    }

    // O(sqrt(n)) — jump block by block using nxt[]
    pair<int,int> query(int idx) const {
        int pos = idx, jumps = 0, exitLast = idx;
        while (pos < n) {
            jumps   += cnt[pos];
            exitLast = last[pos];
            pos      = nxt[pos];
        }
        return {exitLast, jumps};
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, q;
    cin >> n >> q;

    vector<int> a(n);
    for (int i = 0; i < n; i++) cin >> a[i];

    SqrtHoles ds(a);

    while (q--) {
        int type; cin >> type;
        if (type == 0) {
            int p, x; cin >> p >> x;
            --p;                          // 1-based input → 0-based internal
            ds.update(p, x);
        } else {
            int p; cin >> p;
            --p;
            auto [exitPos, jumps] = ds.query(p);
            cout << exitPos + 1 << ' ' << jumps << '\n';
        }
    }
    return 0;
}
Build O(n) Update O(√n) Query O(√n) Memory O(n)

Key pitfall: Always rebuild from right to left within the block. A left-to-right rebuild would use stale nxt[to] values for positions that still land inside the block. The right-to-left order guarantees every dependency is resolved before it is needed.

📋 §18 Final Summary

The core formula

range answer  =  left partial scan
              +  full block summaries (O(1) each)
              +  right partial scan

This formula drives every variant. What changes is what "summary" means per block: a single number (sum/min), a sorted vector (count ≤ x), a Fenwick tree (count by value), or a lazy flag (set/reverse).

The core invariant

At all times, every block summary correctly represents the current values inside that block. Established by build(). Maintained by every update(). A query only produces a correct answer if this invariant holds at call time.
