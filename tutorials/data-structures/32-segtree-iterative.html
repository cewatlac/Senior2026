Lesson 3 — Iterative Segment Tree

Non-Recursive · Flat Heap · L/R Pointer Sweep · Compact and Fast

🧩 Problem Definition

We solve the same problem as Lesson 1 — point update and range query on an array of n elements — but using a non-recursive segment tree built on a single flat array. The reasons to learn this version:

Faster constant factor (no function call overhead) Smaller code (≈ 25 lines) Cache-friendly (array sweep) Same O(n), O(log n), O(log n) complexities

The iterative version shines when the operation is associative and simple (sum, min, max, gcd, xor, multiply). Once you need lazy propagation, persistence, dynamic coordinates, or composite nodes that depend on left/right order in a non-commutative way, the recursive version is usually clearer and safer.

💡 Intuition from Scratch

The recursive version processes one node by descending into both children. The iterative version flips the direction: it starts at the two leaves l and r, and walks upward, harvesting the right nodes as it climbs.

Think of the leaves as a row of array cells. The query interval [l, r] sits as a horizontal stripe across the bottom. As we climb, two pointers (one starting at l, one at r) move toward each other. At every level we check a simple parity rule:

✅ Left pointer rule

If L is a right child, its left sibling is outside our range, so we must take tree[L] on its own and then move past it (L++).

In 0-based heap indexing, right children have even indices, so the check is (L & 1) == 0.

✅ Right pointer rule

If R is a left child, its right sibling is outside our range, so we must take tree[R] on its own and then move past it (R--).

In 0-based heap indexing, left children have odd indices, so the check is (R & 1) == 1.

After harvesting where needed, both pointers move to their parents: L = (L−1)/2, R = (R−1)/2. They keep climbing until L > R.

🌳 Visual Mental Model

With n = 5, we round up to base = 8, so the tree has 2·base − 1 = 15 used nodes. Leaves start at index leaf0 = base − 1 = 7:

leaf0 = 7 a[0] → tree[7] a[1] → tree[8] a[2] → tree[9] a[3] → tree[10] a[4] → tree[11] tree[12], tree[13], tree[14] ← padded leaves (identity)

So array position p lives at tree index leaf0 + p. The internal nodes tree[0..6] hold the recursively merged values, exactly the same heap structure as the recursive version — only this time we never recurse into it; we just index into it.

📐 Formal Abstraction

The tree uses the same 0-based heap indexing as Lesson 1:

root = 0 left = 2i + 1 (always ODD) right = 2i + 2 (always EVEN) parent(i) = (i − 1) / 2 for i ≥ 1 leaf0 = base − 1 tree size = 2 · base

Parity is the engine of this version. Because left children are odd and right children are even, the parity of a node's index tells us whether its sibling is to its left or to its right — and that's what decides whether we have to harvest it before climbing.

✅ Parity rule (cheat sheet)

Index parity ↔ direction:

• i odd → i is a left child, sibling is on the right (i+1).

• i even → i is a right child, sibling is on the left (i−1).

🔬 Key Observations

Three observations make the iterative sweep work.

Observation 1. Every internal value can be recomputed from its two children. So after writing a single leaf, we only need to walk up the path and re-merge each parent — O(log n) work.

Observation 2. For a query [l, r], the leaves leaf0+l and leaf0+r bracket the answer. At every level we ask: "Are my pointer's siblings inside the range?" If yes, climbing the parent already covers everything. If not, we must harvest the pointer node first and shift past it.

Observation 3. The pointers meet after at most log₂(base) levels — the height of the tree — so the whole sweep is O(log n).

⚙️ Algorithm

Build (bottom-up)

Place all n values at leaves tree[leaf0..leaf0+n−1]. Pad the rest with identity. Then sweep i from leaf0−1 down to 0 and set tree[i] = merge(tree[2i+1], tree[2i+2]). O(n) total.

Point update

Set tree[leaf0 + pos] = value. Then walk up: i = (i−1)/2 while i ≥ 0, re-merging from children at each step. O(log n).

Range query

Start with L = leaf0 + l, R = leaf0 + r. Two accumulators: left_answer (grows by appending on the right) and right_answer (grows by appending on the left). While L ≤ R:

  1. If L & 1 == 0, then L is a right child whose left sibling is outside the range — absorb tree[L] into left_answer and do L++.
  2. If R & 1 == 1, then R is a left child whose right sibling is outside the range — absorb tree[R] into right_answer (prepended) and do R--.
  3. If L > R after these adjustments, break.
  4. Climb: L = (L−1)/2, R = (R−1)/2.

Final answer is merge(left_answer, right_answer). O(log n).

🛡️ Invariants

✅ Invariant — Leaf invariant

tree[leaf0 + p] always equals the current value of a[p] for every 0 ≤ p < n. Padded leaves tree[leaf0 + n .. 2·base − 2] always hold identity.

✅ Invariant — Parent invariant

For every internal node i (i < leaf0), tree[i] = merge(tree[2i+1], tree[2i+2]) at all times outside the middle of an update.

✅ Invariant — Pointer-sweep invariant

During a query, after processing level k, the accumulators left_answer and right_answer together contain exactly the merge of the original query interval up to but not including the subranges still covered by [L, R] at the next level.

🧮 Proof of Correctness

🧮 Proof — Why the L/R sweep returns the exact range merge

Define covered(L, R) as the leaves currently in the conceptual interval bounded by pointers L and R at the present level. Initially L = leaf0 + l, R = leaf0 + r, so covered(L, R) equals exactly the query leaves [l, r].

Claim. After each iteration of the loop, the merged contribution we have to still account for is exactly the union of covered(L, R) at the new level plus the leaves already accumulated in left_answer ∪ right_answer, with no double-counting.

Proof. If L is a right child, climbing its parent would include leaves to the left of L that are outside the original query — so we must harvest tree[L] first (these are the leaves under L, all inside the query) and shift L++ past its parent. Symmetric argument for R being a left child. After both adjustments, L and R are exactly the boundary children needed for the parent level. Climb is the recursive step. ∎

Termination. Each level halves the height; after log₂(base) climbs, the pointers meet or cross and the loop ends.

⏱ Complexity Analysis

Constant factor is roughly 2–4× faster than the recursive version on dense inputs, thanks to better cache locality and no function-call overhead. This matters in problems where n, q are both ≥ 10⁶ and the operation per node is trivial.

🧰 Core Template Philosophy

This template keeps the same 0-based heap convention as Lesson 1, the same base rule, and the same merge_values hook — only the body of build, point_set, query changes. You can swap sum ↔ min ↔ max ↔ gcd ↔ xor by changing only:

the ID constant the static merge_values function

For non-commutative merges (e.g. matrix multiplication, polynomial chains) you must preserve order: in the loop, left_answer grows on the right side and right_answer grows on the left side. The final return is merge(left_answer, right_answer) — never swap that order.

💻 Master Template (C++)

Verbatim template from the course notes:

#include <bits/stdc++.h>
using namespace std;

struct IterativeSumSegTree {
    using T = long long;

    static constexpr T ID = 0;

    int n = 0;
    int base = 1;
    int leaf0 = 0;
    vector<T> tree;

    IterativeSumSegTree() {}

    IterativeSumSegTree(const vector<T>& a) {
        init(a);
    }

    static T merge_values(T a, T b) {
        return a + b;
    }

    void init(const vector<T>& a) {
        n = (int)a.size();

        base = 1;
        while (base < max(1, n)) {
            base <<= 1;
        }

        leaf0 = base - 1;
        tree.assign(2 * base, ID);

        for (int i = 0; i < n; i++) {
            tree[leaf0 + i] = a[i];
        }

        for (int i = leaf0 - 1; i >= 0; i--) {
            tree[i] = merge_values(tree[2 * i + 1], tree[2 * i + 2]);
        }
    }

    void point_set(int pos, T value) {
        int i = leaf0 + pos;
        tree[i] = value;

        while (i > 0) {
            i = (i - 1) / 2;
            tree[i] = merge_values(tree[2 * i + 1], tree[2 * i + 2]);
        }
    }

    T query(int l, int r) const {
        int L = leaf0 + l;
        int R = leaf0 + r;

        T left_answer = ID;
        T right_answer = ID;

        while (L <= R) {
            if ((L & 1) == 0) {
                left_answer = merge_values(left_answer, tree[L]);
                L++;
            }

            if ((R & 1) == 1) {
                right_answer = merge_values(tree[R], right_answer);
                R--;
            }

            if (L > R) {
                break;
            }

            L = (L - 1) / 2;
            R = (R - 1) / 2;
        }

        return merge_values(left_answer, right_answer);
    }
};

🔎 Line-by-Line Explanation

init(a) — bottom-up build click to expand/collapse

base is rounded up to a power of two so every leaf has a real heap index. leaf0 = base − 1 is the index of a[0]. We fill the first n leaf positions with input values; the remaining padded leaves keep their identity (0). The downward loop for i = leaf0 − 1; i ≥ 0; i-- visits internal nodes in reverse-BFS order, so by the time we touch node i, both children 2i+1 and 2i+2 are already finalized. This is a key reason the build is O(n) with no recursion.

point_set(pos, value) — climb the path click to expand/collapse

We jump straight to the leaf at leaf0 + pos, overwrite it, and then move to its parent using the integer formula (i − 1) / 2. Each parent's value depends only on its two children, so we re-merge them and continue climbing. The loop terminates when i becomes 0 (we still re-merge the root) and then i would become -1; we use while (i > 0) before the climb to keep the parent index meaningful.

query(l, r) — the L/R sweep click to expand/collapse

We position L and R at the two end leaves. Two accumulators are needed because for non-commutative merges, contributions from L belong on the right side of left_answer and contributions from R belong on the left side of right_answer. After each level's harvest, we test L > R early to avoid a needless climb when the two pointers have already met. The final return merges the two accumulators in the unique correct order.

⚠️ Common Pitfalls

⚠️ Pitfall — Wrong identity for non-sum operations

For min, use LLONG_MAX or a sentinel. For max, use LLONG_MIN. For gcd, use 0 (since gcd(x, 0) = x). For xor, use 0. Using 0 as identity for min would make every query return 0.

⚠️ Pitfall — Off-by-one on parity

The rule for 0-based heap is: L & 1 == 0 means L is a right child, R & 1 == 1 means R is a left child. If you copy code from a tutorial that uses 1-based heap, the parity is inverted and the tree breaks silently.

⚠️ Pitfall — Wrong order in final merge

For commutative operations (sum, min, max, gcd, xor) the order doesn't matter. For non-commutative operations (matrix product, polynomial chain), the final return must be merge(left_answer, right_answer) — never the reverse.

⚠️ Pitfall — Climbing past the root

The update loop uses while (i > 0) and computes i = (i − 1) / 2 first. Don't replace it with while (i >= 0) — when i = 0, (0 − 1) / 2 = 0 in C++ (integer division truncates toward zero), which produces an infinite loop.

Quick reflex

If the problem says "no updates" → use a sparse table instead. If the problem says "point update + simple aggregate" → iterative. Anything else → start from the recursive base.

🎛️ Variants & Extensions

The iterative skeleton stays identical; only the constants and the merge function change.

Variant 1 — Range minimum

static constexpr T ID = LLONG_MAX; static T merge_values(T a, T b) { return min(a, b); }

Variant 2 — Range maximum

static constexpr T ID = LLONG_MIN; static T merge_values(T a, T b) { return max(a, b); }

Variant 3 — Range GCD

static constexpr T ID = 0; static T merge_values(T a, T b) { return std::gcd(a, b); }

Variant 4 — Range XOR

static constexpr T ID = 0; static T merge_values(T a, T b) { return a ^ b; }

Variant 5 — Range product modulo p

static constexpr T ID = 1; static T merge_values(T a, T b) { return (long long)a * b % MOD; }

🏁 Final Summary

The iterative segment tree is the same idea as Lesson 1, expressed as a tight loop over a flat array. The build runs bottom-up in a single sweep, the point update climbs one path, and the range query walks two pointers upward, using parity to decide what to harvest before each climb.

Smallest code Fastest constant factor Best for simple associative operations Awkward for lazy/persistent/dynamic

Once the parity rule clicks, this version becomes a 25-line reflex that's hard to write incorrectly. Use it as your default for contests that ask for point updates and a simple merge.
