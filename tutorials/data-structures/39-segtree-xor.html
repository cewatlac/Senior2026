Lesson 10 — XOR Segment Tree

Virtual b[i] = a[i ⊕ mask] · Sum Without Materializing · O(log n) per Query

🧩 Problem Definition

Given an array a[0..n−1] with n a power of two, and a non-negative integer mask m, define the virtual array b[i] = a[i ⊕ m] for 0 ≤ i < n. We want to support:

Range sum

For query (m, l, r), return b[l] + b[l+1] + ... + b[r] where b[i] = a[i ⊕ m].

Point update

Set a[p] = v, which simultaneously changes b[p ⊕ m] for every conceivable mask m.

Materializing b for each query would take O(n). The XOR segment tree answers each query in O(log n) by exploiting the bit-level structure of XOR: the mask permutes blocks of the array, not individual elements, in a way that respects the segment tree's binary partitioning.

💡 Intuition from Scratch

XOR is the most segment-tree-friendly permutation imaginable. At the top of the tree, the mask's top bit decides whether the left half and the right half are swapped. Move down one level — the second-from-top bit decides whether each sub-half's left and right are swapped, independently. And so on, all the way down to the leaves.

So at level k (counting from the top, level 0 being the root, level logn − 1 being the leaves), the relevant bit of the mask is (mask >> (logn − 1 − k)) & 1. If that bit is 1, every node at level k conceptually swaps its left and right children for the purpose of this query's view of the array. If 0, no swap.

That gives us the algorithm: descend the tree as usual, but at every level decide which physical child (left or right of a) corresponds to the “logical left” (small-index half of b) based on the mask bit for that level. Once we land on a fully-covered node, the sum we want is exactly the precomputed sum of that node — because XOR preserves multisets, the sum of a node's interval in b equals the sum of the corresponding swapped node in a.

🌳 Visual Mental Model

Suppose n = 8 and mask = 5 in binary 101. The three bits of the mask, read from most significant to least, determine the swap pattern level by level:

level 0 (top bit = 1): swap left half & right half level 1 (mid bit = 0): no swap within each half level 2 (low bit = 1): swap each pair of leaves Original a: [a0 a1 a2 a3 a4 a5 a6 a7] After level-0: [a4 a5 a6 a7 | a0 a1 a2 a3] After level-1: [a4 a5 a6 a7 | a0 a1 a2 a3] (no change) After level-2: [a5 a4 a7 a6 | a1 a0 a3 a2] ← this is b[]

Equivalently, b[i] = a[i ⊕ 5]: e.g. b[0] = a[0 ⊕ 5] = a[5], b[1] = a[1 ⊕ 5] = a[4], etc. The level-by-level swap pattern is just the bit-level unrolling of XOR.

📐 Formal Abstraction

Number the levels so level 0 is the root and level L = log₂(base) is the leaves. A node at level k covers a block of size base / 2^k. The mask bit relevant at level k (deciding the swap at that level's children) is bit (L − 1 − k) of the mask:

bit_at_level(k) = (mask >> (L - 1 - k)) & 1 // controls left/right swap when descending past a node at level k

Equivalently, the “high_mask” relevant when our recursion is inside a node of length len is mask & ~(len − 1). The bit deciding whether to swap at this node's children is mask & (len / 2).

The recursion processes a query (l, r) in logical b-space. At each node, it uses the level's mask bit to decide which physical child of a's segment tree corresponds to the lower (logical-left) half of b at this node's interval.

🔬 Key Observations

Observation 1 (XOR distributes over level). Because i ⊕ m has the same high bits as i XOR-ed with the high bits of m, and the same low bits as i XOR-ed with the low bits of m, every level of the segment tree gets a clean “swap or don't swap” decision based on a single bit of the mask.

Observation 2 (full-overlap sums are XOR-invariant). A segment tree node covers a contiguous block of a. Under XOR with mask m, the same node's interval in b is also a contiguous block (because the swap at this node's level either keeps the block in place or moves it to the sibling block). And since the sum is the same on either side (the XOR just permutes the elements within), sum_a(node) = sum_b(corresponding-node-in-b-tree) — we can read it off without recomputation.

Observation 3 (three-piece split). For a partial-overlap node at level k, the descent uses the mask bit at level k to decide which physical child to recurse into for the logical-left part of the query and which for the logical-right part. If the mask bit is 0, no swap: left of a is left of b. If the mask bit is 1, swap: left of a is right of b. The query then bifurcates as usual.

⚙️ Algorithm

The data structure is a normal recursive segment tree from Lesson 1 over a. The clever part is only in the query.

build / point update

Exactly as Lesson 1.

query(mask, ql, qr)

Recurse from the root. At each node with interval [lo, hi]:

  1. If [lo, hi] is fully inside the logical query [ql, qr] (in b-space), return tree[i] — by Observation 2, this is the b-sum we want.
  2. If [lo, hi] is disjoint from the logical query, return 0.
  3. Otherwise compute the level's mask bit, decide swap, and recurse into the children in the swapped order. The query range in b maps to the same range — we are descending in b-space, not a-space, so the children's [lo, mid] and [mid+1, hi] intervals are interpreted in b while the physical tree nodes we recurse into are in a.

In code, this is conveniently expressed by passing the mask down and at each recursion deciding which physical child to call.

🛡️ Invariants

✅ Invariant — Tree invariant

The segment tree over a obeys the standard invariant from Lesson 1: every node's stored sum equals the sum of its interval in a.

✅ Invariant — Permutation invariant

For every mask m and every segment-tree node u covering interval [lo, hi] in a, the same interval [lo, hi] in b = a ⊕ m is also a contiguous interval (because XOR with mask m permutes blocks of size hi − lo + 1 as wholes), and its sum equals tree[u]. So full-overlap nodes are sum-invariant under any XOR mask.

✅ Invariant — Level-bit invariant

At level k of the descent (block length len = base / 2^k), the relevant bit of the mask is mask & (len / 2). This bit decides whether the two children are swapped in b's view: bit = 0 keeps the order; bit = 1 swaps.

🧮 Proof of Correctness

🧮 Proof — Why full-overlap nodes are sum-invariant under XOR

Let [lo, hi] be the interval of segment-tree node u. Because the tree is built with power-of-two block sizes, [lo, hi] has the form [k · L, (k+1) · L − 1] where L is the block length at u's level and k is the block index.

Consider the map φ: i ↦ i ⊕ m restricted to [lo, hi]. The high bits of i (above the bits used to address within L) are fixed inside [lo, hi]; XOR-ing them with the corresponding high bits of m produces a new fixed prefix — which addresses a different block of length L in a, but still a length-L block. The low bits of i range over [0, L − 1]; XOR-ing them with the low bits of m permutes [0, L − 1] back onto itself.

So φ([lo, hi]) is another length-L block, and φ restricted to [lo, hi] is a bijection onto that block. Sums of bijectively-related multisets are equal, so the sum of b over [lo, hi] equals the sum of a over φ([lo, hi]) — but in the descent we read tree[u], which is the sum of a over [lo, hi]. These two sums are equal because they're both equal to the sum over the same block-length-L contiguous interval in a… as long as the descent reaches the correct physical node, namely the one whose interval is φ([lo, hi]).

That is precisely what the swap-or-no decision at each level does: it picks the child of a whose interval corresponds to the logical child of b, by reading the relevant bit of the mask. By induction over the depth of the descent, the node we land on at full overlap is the right one. ∎

🧰 Core Template Philosophy

The XOR segment tree is a tiny addition to Lesson 1. The only new code is in the query function: it receives a mask, and at each recursion it decides which physical child to descend into based on a single bit of the mask. Build, update, and the rest of the structure are unchanged.

The mental discipline: separate the logical coordinate (what b sees) from the physical coordinate (what a stores). The descent talks about the logical coordinate; at each level, it maps the logical child it wants to enter into the physical child by XOR-ing with the level's bit.

💻 Master Template (C++)

#include <bits/stdc++.h>
using namespace std;

struct XorSumSegTree {
    using T = long long;

    int n = 0;
    int base = 1;
    vector<T> tree;

    XorSumSegTree() {}
    XorSumSegTree(const vector<T>& a) { init(a); }

    void init(const vector<T>& a) {
        n = (int)a.size();
        base = 1;
        while (base < max(1, n)) base <<= 1;
        tree.assign(2 * base, 0);
        for (int i = 0; i < n; i++) tree[base - 1 + i] = a[i];
        for (int i = base - 2; i >= 0; i--)
            tree[i] = tree[2*i + 1] + tree[2*i + 2];
    }

    void point_set(int pos, T value) {
        int i = base - 1 + pos;
        tree[i] = value;
        while (i > 0) {
            i = (i - 1) / 2;
            tree[i] = tree[2*i + 1] + tree[2*i + 2];
        }
    }

    // Returns sum_{i in [ql, qr]} a[i ^ mask]
    T xor_range_sum(int mask, int ql, int qr) const {
        return xor_range_sum(mask, ql, qr, 0, 0, base - 1);
    }

    T xor_range_sum(int mask, int ql, int qr,
                    int i, int lo, int hi) const {
        if (qr < lo || hi < ql) return 0;
        if (ql <= lo && hi <= qr) return tree[i];

        int len = hi - lo + 1;
        int half = len >> 1;
        int mid = lo + half - 1;

        // Which physical child corresponds to b's left half?
        bool swap = (mask & half) != 0;

        int leftPhys  = swap ? 2*i + 2 : 2*i + 1;
        int rightPhys = swap ? 2*i + 1 : 2*i + 2;

        return xor_range_sum(mask, ql, qr, leftPhys,  lo,    mid)
             + xor_range_sum(mask, ql, qr, rightPhys, mid+1, hi);
    }
};

🔎 Line-by-Line Explanation

Build / point_set — unchanged click to expand/collapse

The segment tree is constructed and updated exactly as in Lesson 1, with a 0-based heap layout. The mask is invisible to these routines.

xor_range_sum — the swap-aware descent click to expand/collapse

The interesting work happens at every internal node. We compute the current block length len = hi − lo + 1 and the bit of the mask that controls the swap at this level: mask & (len >> 1). If that bit is set, the physical-left child of a is actually the logical-right child of b, so we route the descent accordingly. The two recursive calls are then identical to Lesson 1's structure — the only difference is which physical child each call addresses.

Why the no-overlap and full-overlap cases don't need the mask click to expand/collapse

Both early-exit cases compare the logical coordinate to the query range — and the recursion has been keeping the logical coordinate [lo, hi] consistent with the b-view. Full-overlap returns tree[i] directly because of the Permutation invariant: the sum over a block-aligned interval in a equals the sum over the same logical interval in b.

⚠️ Common Pitfalls

⚠️ Pitfall — Treating n = non-power-of-2 like power-of-2

The block-swap property of XOR only works when the universe is a power of 2. If n isn't, you must pad a with identity elements (0 for sum, +∞ for min, etc.) up to the next power of 2, then accept that queries on the padded portion return identity.

⚠️ Pitfall — Using the wrong mask bit

At block length len, the relevant mask bit is mask & (len >> 1), not mask & (len − 1) or mask & len. The bit you want is the one that decides between the two halves of the current block.

⚠️ Pitfall — Forgetting that point_set is in physical (a) coordinates

Point updates always target a[p], not b[p]. If the problem asks to update b[p] for some mask m, you must update a[p ⊕ m] instead.

⚠️ Pitfall — Computing mid as (lo + hi) >> 1 with sign issues

For coordinates that can be very large or negative, use lo + (hi − lo) / 2. For the XOR segment tree the coordinates are non-negative array indices, so this is rarely an issue — but watch out if you adapt the template to a 2D or huge-coordinate variant.

🎛️ Variants & Extensions

Variant 1 — Persistent XOR segment tree

Combine the path-copying machinery of Lesson 5 with the XOR descent of this lesson. Used for “sum of values ≤ x in b[l..r]” and similar order queries on the XOR-permuted array.

Variant 2 — Multiple chained XORs

If the problem applies multiple XORs in sequence, the result is a single XOR (XOR is associative and commutative). So the structure is unchanged; just precompute the cumulative mask.

Variant 3 — Min / Max / GCD instead of Sum

Any operation that respects multisets — min, max, gcd, xor, multiplication mod p — works with the same descent. The Permutation invariant still applies because the sum is replaced by the operation.

Variant 4 — XOR on a 2D grid

Apply the same idea to a 2D segment tree: at each level, two mask bits (one for x, one for y) determine which of the four physical quadrants corresponds to each logical quadrant. Used in some FFT-related contest problems.

🏁 Final Summary

The XOR segment tree answers range-sum queries on a virtual array b[i] = a[i ⊕ mask] in O(log n), without ever materializing b. The key insight is that XOR with a mask permutes blocks of a in a way that respects the segment tree's binary partitioning: at every level of the descent, a single bit of the mask decides whether to swap the two physical children.

The structure is just Lesson 1 with a swap-aware query function. No changes to build or point update. The technique generalizes: you can combine XOR with persistence (Variant 1), with composite nodes (Variant 3), or with 2D (Variant 4). The pattern of separating logical coordinate (what the query sees) from physical coordinate (what the tree stores) is broadly useful.
