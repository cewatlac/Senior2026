Introduction — What Dijkstra Really Does

Given a weighted graph with nonneg edges and a fixed source s, Dijkstra finds the minimum-cost path from s to every reachable vertex. It does this by greedily extracting the unfinalized vertex with the smallest tentative distance and permanently sealing its label.

All edge weights ≥ 0 required
Negative edge → use Bellman-Ford / SPFA
Sparse: O(m log n)
Dense: O(n²)
dist[v] = minimum total cost from s to v

Problem definition

Given G = (V, E) with w(u,v) ≥ 0 and a source s, compute for every v:

dist[v] = min cost over all paths from s to v.

Optional extras: the actual path, count of paths, a tie-broken path, or shortest distance over an expanded state space.

Core intuition

Because all weights are nonneg, extending any path can only increase its cost. So the unfinalized vertex with the current smallest label can never be improved by any future relaxation — it is safe to seal it permanently. This is the greedy invariant that makes Dijkstra correct.

Mental model — the expanding frontier

Imagine a flood spreading from s. Each wave front reaches vertices in order of non-decreasing cost. The first time water touches a vertex is the cheapest route. Dijkstra simply simulates this flood digitally — it always processes the cheapest not-yet-flooded vertex first, then floods all its outgoing edges.

Classical pseudocode

for each vertex v:
    dist[v] = INF, par[v] = -1
dist[s] = 0
push s into pq with key 0

while pq not empty:
    u = extract min from pq         // greedy step
    if extracted key is stale: continue
    for each edge (u, v, c):
        nd = dist[u] + c
        if nd < dist[v]:
            dist[v] = nd, par[v] = u
            push v into pq with key nd
Dense variant: replace pq with linear scan of unvisited set
0-1 BFS variant: replace pq with deque
Never use on negative-weight graphs

One Family — Two Templates — Many Controlled Edits

Every variant in this document is the same algorithm. To solve any variant you identify which of four dimensions changes from the base, then apply exactly that small edit. Nothing else changes.

① Relaxation formula
Sum: nd = dist[u] + c
Bottleneck max: nd = max(dist[u], c)
Bottleneck min: nd = min(dist[u], c)
Product: nd = dist[u] * p
② State definition
Standard: (node)
Layered: (node, extra)
Parity: (node, 0/1)
Bitmask: (node, mask)
Fuel: (node, fuel)
Time-mod: (node, t mod k)
③ Frontier structure
Heap: general nonneg weights
Linear scan: dense / tiny n
Deque: binary {0, 1} weights only
Bucket / Dial's: small integer C
④ Bookkeeping
Path: par[v] = u
Count: ways[v]
Edge-count DP: mn[], mx[]
Tie-break: secondary criterion array
DAG: post-filter by dist[u]+c==dist[v]
The key claim: Dijkstra is correct on any nonneg-weight graph where the path-cost function is monotone (adding an edge never decreases cost). That monotonicity is what the four edit types above must preserve. When you pick a new relaxation formula or objective, your first question is always: "Is the new cost function still monotone?" If yes, Dijkstra is correct. If no, you need a different algorithm.

When NOT to use Dijkstra

Use Bellman-Ford / SPFA instead when:
  • Any edge weight is negative
  • You need to detect negative cycles
  • Graph is extremely sparse and SPFA heuristic helps
Use plain BFS instead when:
  • All edge weights are exactly 1 (unweighted graph)
  • Use 0-1 BFS when weights are exactly {0, 1}

Correctness Proof

Claim + proof by contradiction The greedy invariant argument
Claim: When a node u is popped from the priority queue with distance d, dist[u] = d*(u) (the true shortest distance).

Proof by contradiction:

  1. Suppose u is the first node popped with an incorrect distance: dist[u] > d*(u).
  2. Let P be a true shortest path from s to u. Let x be the first not-yet-popped vertex on P, and y its predecessor on P.
  3. Since y was already popped and (by assumption) correctly, when (y,x) was relaxed we pushed x with key d*(y) + w(y,x) = d*(x).
  4. All weights nonneg, so d*(x) ≤ d*(u) < dist[u]. Therefore x sits in the queue with priority smaller than u.
  5. x should have been popped before u — contradiction.
  6. Therefore every popped distance is correct. ∎
The entire argument rests on nonneg weights guaranteeing that d*(x) ≤ d*(u) in step 4. A single negative edge can violate this, producing a "shorter" path discovered after u is already sealed — which is exactly why Dijkstra fails on negative-weight graphs.
Edge cases and gotchas When to be extra careful

Disconnected graph: If t is unreachable from s, dist[t] remains INF. Always check for this and return −1 or print "impossible."

Overflow: Edge weights up to 10⁹ with up to 10⁵ nodes → path costs up to 10¹⁴. Always use long long for the distance array.

Self-loops: A self-loop (u,u,c) with c>0 can never improve dist[u]. The stale check handles this automatically.

Parallel edges: Multiple edges between the same pair of vertices are fine — Dijkstra just processes each one independently during relaxation.

The stale-entry check is mandatory: if (d > dist[u]) continue; — without it a node can be processed multiple times with outdated distances, corrupting results and degrading complexity to O(m²).

Complexity Analysis

Sparse — heap + adjacency list

Each of m edges triggers at most one heap push: O(log n) per push.
Each of the ≤ m pops also costs O(log n).

Time: O((n + m) log n)
Written as O(m log n) when m ≥ n.

Space: O(n + m)

Dense — linear scan + adjacency matrix

n outer iterations, each scanning all remaining unvisited nodes: O(n).
Relaxation is fused into the same pass: no extra cost.

Time: O(n²) — independent of m.
Optimal when m ≈ n² (complete/dense graph).

Space: O(n²)

Master Template A — Sparse (Heap)

Your accepted CF 20C solution. All heap-based variants inherit from this. Conventions: adj adjacency list · par parent · dist distance · max-heap with negated keys · n nodes · s source · t target.

Full template with annotations GNU++17/20 style 
#include <bits/stdc++.h>
using namespace std;

long long dijkstra(vector<vector<pair<int,int>>> &adj, int s, int t, vector<int> &par) {
    int n = adj.size();
    vector<long long> dist(n, LLONG_MAX);   // long long avoids overflow
    par.assign(n, -1);
    dist[s] = 0;
    priority_queue<pair<long long,int>> q;  // max-heap; store (-dist, node)
    q.emplace(0LL, s);
    while (q.size()) {
        auto [d, u] = q.top(); q.pop();
        d = -d;                              // negate back to true distance
        if (d > dist[u]) continue;           // stale-entry check — NEVER remove
        if (u == t) return d;                // early-stop optimisation
        for (auto &[v, c] : adj[u]) {
            auto nd = dist[u] + c;           // base relaxation
            if (dist[v] > nd) {
                dist[v] = nd;
                q.emplace(-nd, v);
                par[v] = u;
            }
        }
    }
    return -1;  // t unreachable
}

void printPath(const vector<int> &par, int u) {
    if (par[u] != -1) { printPath(par, par[u]); printf(" "); }
    printf("%d", u + 1);
}
Line-by-line explanation Why every decision exists

priority_queue<pair<long long,int>> — C++ default is a max-heap. We store (-dist, node) so the smallest distance is at the top. This avoids the verbose greater<> comparator and is the accepted CP idiom.

d = -d — Immediately after popping, we negate to recover the true distance. The pair was (-dist, node) so d is currently negative.

if (d > dist[u]) continue — The stale-entry guard. Because we use lazy insertion (we push new entries rather than decreasing existing keys), the queue may hold multiple entries for the same node. When an older entry for a node whose distance has already been improved is popped, we discard it. Without this check, we process nodes repeatedly and correctness is lost.

if (u == t) return d — Early exit: the first time t is popped its distance is final (Dijkstra invariant). Saves processing the rest of the graph for single-target queries.

auto nd = dist[u] + c — The base relaxation. Every variant changes exactly this line (or adds to the relaxation block).

par[v] = u — Whenever dist[v] improves, the predecessor on the new best path is u. Once dist[v] is final, par gives the complete path by tracing backwards.

Master Template B — Dense (Linear Scan O(n²))

Your accepted Luogu P1529 solution. No heap — scan the unvisited set linearly. Perfect for tiny alphabets and dense graphs where heap overhead outweighs savings.

Full template with annotations O(n²) · no heap 
#include <bits/stdc++.h>
using namespace std;
const int OO = 0x3f3f3f3f;
const int MAXN = 135;
int dist[MAXN], adj[MAXN][MAXN], notVis[MAXN], n;

void dijkstra(int src) {
    int sz = n;
    memset(dist, 0x3f, n * sizeof(dist[0]));  // fills each int with 0x3f3f3f3f
    dist[src] = 0;
    int nxt = src;
    iota(notVis, notVis + n, 0);               // notVis = {0,1,...,n-1}

    while (nxt != -1) {
        int u = notVis[nxt];
        notVis[nxt] = notVis[--sz];            // O(1) swap-remove
        nxt = -1;
        int best = OO;
        for (int i = 0; i < sz; i++) {
            int v = notVis[i];
            if (dist[u] + adj[u][v] < dist[v]) dist[v] = dist[u] + adj[u][v];
            if (dist[v] < best) { best = dist[v]; nxt = i; }
        }
    }
}
Key design decisions Why notVis[] trick?

notVis[] compact set: Instead of a boolean visited[], we maintain a shrinking array of unvisited nodes. Removing node u is O(1): overwrite its slot with the last element and decrement size. This halves the work per outer iteration.

nxt is an index, not a node ID: It stores the position within notVis[] of the next-best node. Both relaxation and min-finding happen in one pass of the inner loop — no second scan needed.

memset(dist, 0x3f, ...): Sets each byte to 0x3f, making each 4-byte int become 0x3f3f3f3f ≈ 1.06×10⁹ — a convenient "infinity" for int-range problems. Adding one more edge weight won't overflow a 32-bit int (safe up to ~2.1×10⁹ total sum).

Variants 1–3: Basic Sparse, Dense, Path Reconstruction

V1 Standard Sparse Dijkstra Roads — print shortest route

Story: Roads between cities have lengths. Print the shortest route from city 1 to city n.

Abstraction: Single-source shortest path on an undirected nonneg-weight graph. Source and target are given.

What stays the same: This is the baseline — everything here defines "the same" for all variants.

What changes: Nothing. V1 is the master template from which all other variants are derived.

Why correctness still holds: Exactly the classical proof: all weights nonneg → greedy invariant holds → first pop of any node gives its true shortest distance.

Delta from base Dijkstra: Relaxation: same baseline · State: same baseline · Frontier: same baseline · Bookkeeping: same baseline

Code delta:

auto nd = dist[u] + c;
if (dist[v] > nd) { dist[v] = nd; par[v] = u; q.emplace(-nd, v); }

When to use: Any single-source shortest-path problem on a sparse graph (m ≪ n²) with nonneg edge weights. This is the CP default. Complexity: O((n+m) log n).

Common pitfall: Forgetting to handle the unreachable case — if dist[t] remains ∞ after the loop, the target is unreachable; print −1, not ∞.

Practice problem: CF 20C — Dijkstra? → exact base-template match requiring single-source shortest path and lexicographically smallest path printing.

V2 Dense Dijkstra O(n²) Tiny or complete graph — no heap needed

Story: The graph is small or dense; an adjacency matrix is the natural representation.

Abstraction: Same SSSP problem, but with adjacency matrix and a linear-scan frontier replacing the heap.

What stays the same: The greedy invariant, the relaxation condition, the correctness proof, and the termination argument are all identical to V1.

What changes: Only the frontier selection method: heap extraction is replaced by a linear scan of the shrinking notVis[] array. Graph representation changes from adjacency list to adjacency matrix.

Why correctness still holds: The unvisited vertex with minimum tentative distance is still extracted at each step — exactly the same guarantee as the heap version. The proof is unchanged; only the data structure differs.

Delta from base Dijkstra: Relaxation: same · State: same · Frontier: changed → linear scan over notVis[] · Bookkeeping: same

Code delta:

for(int i=0; i

When to use: When m ≈ n² (dense/complete graphs) or when n ≤ ~5 000. Also natural for alphabet-sized graphs (n ≤ 128) where an adjacency matrix is given directly. Complexity: O(n²) time, O(n²) space.

Common pitfall: Using the dense template on a sparse large graph — if n = 200 000 and m = 300 000, the heap version is roughly 100× faster. Always check density before choosing.

Practice problem: Luogu P1529 — Tram → dense alphabet graph (n ≤ 128) where the O(n²) linear-scan template is the natural and optimal choice.

V3 Path Reconstruction The judge wants the actual route, not just the cost

Story: The problem asks to print the actual shortest path, not just its total distance.

Abstraction: Same SSSP computation as V1, with a predecessor array maintained alongside the distance array.

What stays the same: The greedy invariant, the relaxation formula, the frontier, and the stale check are all unchanged. The distance computation is completely identical to V1.

What changes: Only bookkeeping: we add par[v] = u inside the relaxation block, and call a path-printing function after Dijkstra completes.

Why correctness still holds: Whenever dist[v] is improved, the best known path to v goes through u. Once dist[v] is finalized (first pop), par[v] gives the correct predecessor on an actual shortest path.

Delta from base Dijkstra: Relaxation: same · State: same · Frontier: same · Bookkeeping: changed → add par[v] = u on each improvement

Code delta:

par[v] = u;
printPath(par, t);

When to use: Any problem asking "print the path" or "output the route." The cost computation is unaffected; par[] only adds O(n) extra space.

Common pitfall: Printing the path in forward order from the par[] chain. The chain gives path in reverse (target → source). Use a stack, recursive call, or reverse() to flip it before printing.

Practice problem: CF 20C — Dijkstra? → requires printing the actual lexicographically smallest shortest path using par[] reconstruction.

Variants 4–6: Graph Transformations

V4 Multi-source Dijkstra Hospitals / fire stations — nearest source to every node

Story: Several special nodes (hospitals, fire stations) all start at distance zero. Find the distance from the nearest source to every other node.

Abstraction: Equivalent to adding a virtual super-source connected to every original source by a zero-weight edge, then running standard single-source Dijkstra from that super-source.

What stays the same: The greedy invariant, the relaxation formula, the frontier, and the stale check are all identical to V1. The core algorithm loop is completely unchanged.

What changes: Only initialization: instead of seeding one source at distance 0, we seed every source at distance 0 before the loop starts.

Why correctness still holds: Initializing all sources at distance 0 is identical to having a virtual super-source with zero-weight edges to each. Standard single-source Dijkstra on the augmented graph is correct by V1's proof.

Delta from base Dijkstra: Relaxation: same · State: same · Frontier: same · Bookkeeping: changed → multi-source init

Code delta:

for (int s : sources) {
    dist[s] = 0;
    q.emplace(0LL, s);
}

When to use: Any problem saying "find the nearest X for every node" where X is a set of special nodes: hospitals, zeros in a matrix, fire stations, infected nodes. Complexity same as V1.

Common pitfall: Adding a physical virtual super-source node instead. While correct, it shifts all node indices and complicates I/O. The initialization-only approach is simpler and preferred.

Practice problem: LC 542 — 01 Matrix → push all zero-valued cells at dist=0 simultaneously before the loop — exact multi-source Dijkstra pattern.

V5 Reverse-Graph Dijkstra One fixed exit, many starting nodes

Story: Many starting nodes want to reach the same single exit. Find which starting nodes can reach the exit within a given cost limit.

Abstraction: The distance from any node x to a fixed target t in the original graph equals the distance from t to x in the reversed graph. Run one Dijkstra from t on reversed edges.

What stays the same: The greedy invariant, the relaxation formula, the frontier, and the stale check are all identical to V1. We are running standard Dijkstra — just on a different graph.

What changes: Only graph construction: every directed edge u→v with weight c is stored as v→u instead. The Dijkstra call itself is untouched.

Why correctness still holds: There is a one-to-one correspondence between paths from x to t in the original and paths from t to x in the reversed graph, with identical total cost. V1's proof applies to the reversed graph.

Delta from base Dijkstra: Relaxation: same · State: same · Frontier: same · Bookkeeping: changed → reverse directed edges at input time

Code delta:

adj[v].push_back({u, c});

When to use: Whenever many sources must reach one target on a directed graph. Turns O(k · m log n) into O(m log n). Has no effect on undirected graphs (edge reversal is a no-op).

Common pitfall: Applying this technique to an undirected graph — there is nothing to reverse. Also: forgetting to run Dijkstra from the exit node (not from the sources) on the reversed graph.

Practice problem: SPOJ MICEMAZE — Mice and Maze → reverse all edges, run Dijkstra from the single exit node, then count mice whose reversed-distance is within the time limit.

V6 Early-Stop Dijkstra Single-target query — stop the moment the answer is known

Story: You only need the shortest distance from s to one specific target t. Once that distance is known, stop.

Abstraction: Standard Dijkstra with one extra termination condition: return immediately when t is popped from the queue.

What stays the same: The greedy invariant, the relaxation formula, the frontier, and the stale check are all identical to V1. The algorithm runs normally up until the early stop.

What changes: Only the termination condition: add if (u == t) return d; immediately after popping, before the relaxation loop.

Why correctness still holds: By the Dijkstra invariant, the first time a node is popped with a non-stale key, its distance is final. So the first time t is popped, dist[t] is optimal and we can safely return.

Delta from base Dijkstra: Relaxation: same · State: same · Frontier: same · Bookkeeping: changed → add early-return on target pop

Code delta:

if (u == t) return d;

When to use: Any single-source, single-target problem. Saves time in practice when t is close to s. In the worst case (t is furthest from s) it offers no saving.

Common pitfall: Adding early stop when the problem asks for distances to ALL nodes. In that case the early return must be removed.

Practice problem: CF 20C — Dijkstra? → single-target query where early stop avoids processing the full graph after t is finalized.

Variant 7: 0-1 BFS

V7 0-1 BFS — Deque Binary edge weights {0,1} · O(n+m)

Story: Every move costs either 0 (free) or 1 (one step). Find the minimum number of 1-cost moves to travel from source to target.

Abstraction: SSSP with weights in {0, 1}. Replace the heap with a deque; 0-cost edges push to the front (same distance level), 1-cost edges push to the back (next level).

What stays the same: We still propagate shortest distances through relaxations. The greedy invariant holds — nodes are finalized in nondecreasing distance order. The relaxation condition prevents re-pushing optimal nodes.

What changes: Only the frontier: priority_queue is replaced by deque<int>. There is no negation trick. 0-cost neighbors go to the front; 1-cost neighbors go to the back.

Why correctness still holds: 0-cost edges keep the distance the same, so they belong at the front of the deque (same layer). 1-cost edges advance the layer by one, so they go at the back. The deque therefore always maintains non-decreasing distance order, matching the Dijkstra invariant.

Delta from base Dijkstra: Relaxation: same · State: same · Frontier: changed → deque; push_front for w=0, push_back for w=1 · Bookkeeping: same

Code delta:

deque dq;
if (w == 0) dq.push_front(v);
else        dq.push_back(v);

When to use: Whenever all edge weights are exactly 0 or 1. Keywords: "free to stay in same type, costs 1 to change." Complexity: O(n + m) — no log factor.

Common pitfall: Using 0-1 BFS when weights include any value other than 0 and 1. Even {0, 2} breaks the deque monotonicity. Also: swapping push_front and push_back silently produces wrong answers.

Practice problem: SPOJ KATHTHI — Kaththi → grid where same-character moves are free and different-character moves cost 1 — exact 0-1 BFS pattern.

Variants 8–9: State-Space Dijkstra

V8 Generic State-Space Dijkstra Coupon, fuel, keys, parity — expand the state

Story: The optimal cost depends not just on which node you are at, but also on extra accumulated information: coupons used, fuel level, items collected, or parity of edges taken.

Abstraction: Replace each node u with a set of states (u, extra). Build an expanded graph whose vertices are (node, extra) pairs. Run standard Dijkstra on this expanded graph.

What stays the same: The greedy invariant, the relaxation formula, the frontier, and the stale check are all identical to V1 — they operate on the expanded graph exactly as on the original.

What changes: Only the state: a single node u becomes a pair (u, extra). The distance array becomes 2D: dist[u][state]. The queue stores (cost, node, state) triples.

Why correctness still holds: The expanded graph has nonneg-weight edges. Classical Dijkstra on any nonneg-weight graph is correct. We are simply running V1 on a larger graph.

Delta from base Dijkstra: Relaxation: same · State: changed → (node, extra); dist becomes 2D · Frontier: same · Bookkeeping: same

Code delta:

struct State { long long d; int u, st;
    bool operator<(const State &o) const { return d > o.d; } };
vector> dist(n, vector(K, LLONG_MAX));

When to use: Whenever "which node am I at" is not enough to determine the optimal next move. Keywords: "at most k free edges," "fuel-limited travel," "collect items," "even number of edges." The state space size is n × |extra|.

Common pitfall: Choosing an extra dimension that is too large. If extra has range K, the expanded graph has Kn nodes and Km edges. Typically K ≤ 2 for coupons, K ≤ 200 for fuel, K ≤ 2^k for bitmask with k ≤ 18.

Practice problem: CSES 1195 — Flight Discount → state = (node, coupon_used); classic 2-layer state-space Dijkstra with one optional 50% discount.

V9 Layered Dijkstra — Coupon (Flight Discount) One 50%-off edge allowed

Story: You may halve the cost of exactly one edge on the route from s to t. Find the minimum total cost.

Abstraction: Expand state to (node, used_coupon). Run Dijkstra on two layers: layer 0 = coupon not yet used, layer 1 = coupon already used. One extra transition from layer 0 to layer 1 uses the coupon.

What stays the same: The greedy invariant, heap frontier, and stale check are identical to V1. Within each layer, relaxation is standard dist[u][used] + c.

What changes: The state expands from (node) to (node, coupon_used). An extra cross-layer transition exists: from (u, 0) to (v, 1) with cost c/2.

Why correctness still holds: Each layer is a copy of the original graph. Cross-layer transitions are nonneg. Standard V8 correctness applies: Dijkstra on the 2-layer expanded graph is correct.

Delta from base Dijkstra: Relaxation: same · State: changed → (node, coupon_used); dist[n][2] · Frontier: same · Bookkeeping: same

Code delta:

vector> dist(n, vector(2, LLONG_MAX));
if (!used && dist[v][1] > d + c/2)
    { dist[v][1] = d+c/2; q.push({d+c/2, v, 1}); }

When to use: Any problem saying "you may use one special ability once" — one free edge, one half-price edge, one teleport. The ability creates exactly 2 layers. Complexity: O((n+m) log n) with 2× constant.

Common pitfall: Forgetting to take min(dist[n-1][0], dist[n-1][1]) at the end. The answer is the best of "never used the coupon" and "used it somewhere."

Practice problem: CSES 1195 — Flight Discount → textbook 2-layer layered Dijkstra; dist[u][coupon_used] with a layer-0→1 transition at cost c/2.

Variant 10: Counting Shortest Paths

V10 Counting + Min/Max Edge Count on Shortest Paths CSES Investigation style

Story: For each node, find: the shortest distance, the number of shortest routes (mod 10⁹+7), and the minimum and maximum number of edges used by any shortest route.

Abstraction: Augment the standard Dijkstra with three auxiliary arrays synchronized with dist[]: ways[], mn[], mx[]. Update them on both strict improvements and equal-distance relaxations.

What stays the same: The greedy invariant, the relaxation condition, the heap frontier, and the stale check are all unchanged. The core distance computation is identical to V1.

What changes: Only bookkeeping: ways[], mn[], mx[] arrays that mirror dist[]. On strictly shorter path, all counters reset. On equal-length path, counters accumulate.

Why correctness still holds: Nodes are processed in nondecreasing distance order. When a strictly shorter path replaces the old one, previous counts become irrelevant. Equal-length paths accumulate correctly because every predecessor on any shortest path to v is finalized before v itself is finalized.

Delta from base Dijkstra: Relaxation: same · State: same · Frontier: same · Bookkeeping: changed → add ways[], mn[], mx[] updated on strict and equal relaxations

Code delta:

vector ways(n,0), mn(n,LLONG_MAX), mx(n,LLONG_MIN);
if (dist[v] > nd) {
    dist[v]=nd; ways[v]=ways[u]; mn[v]=mn[u]+1; mx[v]=mx[u]+1; q.emplace(-nd,v);
} else if (dist[v]==nd) {
    ways[v]=(ways[v]+ways[u])%MOD;
    mn[v]=min(mn[v],mn[u]+1); mx[v]=max(mx[v],mx[u]+1);
}

When to use: Any problem asking "how many shortest paths" or "fewest/most edges among shortest paths." Keywords: "count routes," "minimum flights," "maximum stops."

Common pitfall: Forgetting the modulo on ways[] accumulation. Counts grow exponentially and overflow without % MOD on the equal-distance branch.

Practice problem: CSES 1202 — Investigation → requires all four simultaneously: dist[], ways[], mn[], mx[] bookkeeping in one Dijkstra pass.

Variants 11–13: Alternative Objectives

V11 Minimax Path — Minimize the Worst Edge Replace + with max in relaxation

Story: Each road has a danger level. Choose the path from s to t that minimizes the maximum danger edge encountered.

Abstraction: Path cost = max edge weight along path. Replace the sum relaxation with a max relaxation. Standard Dijkstra structure is otherwise unchanged.

What stays the same: The greedy invariant and the heap frontier are identical to V1. The stale check is unchanged.

What changes: Only the relaxation formula: nd = dist[u] + c becomes nd = max(dist[u], c).

Why correctness still holds: The max function is monotone: adding any edge to a path never decreases its bottleneck value. The greedy invariant holds under the new objective — the first time a node is popped with its bottleneck value, that value is optimal.

Delta from base Dijkstra: Relaxation: changed → max(dist[u], c) · State: same · Frontier: same · Bookkeeping: same

Code delta:

auto nd = max(dist[u], 1LL * c);

When to use: Any problem where path cost is the maximum edge weight: "bottleneck path," "danger level," "minimum effort." Note: dist[s] = 0 still works because max(0, c) = c for nonneg c.

Common pitfall: Initializing dist[s] = INF instead of 0. For minimax, dist[s] = 0 is correct — the cost of a zero-edge path is 0.

Practice problem: LC 1631 — Path With Minimum Effort → minimize the maximum absolute height difference along any path — exact minimax relaxation.

V12 Maximin / Widest Path — Maximize the Minimum Capacity Max-heap, reversed comparison

Story: Each road has a capacity. Find the path from s to t that maximizes the minimum capacity edge (the "widest" bottleneck path).

Abstraction: Path cost = min edge weight on path, and we want to maximize it. Use a max-heap with positive values; initialize dist[s] = ∞; update when nd > dist[v].

What stays the same: The greedy extraction principle and the stale check remain. We still finalize nodes in a monotone order, but now we extract the maximum-capacity node first.

What changes: Three changes: (1) relaxation becomes nd = min(dist[u], c); (2) update condition becomes nd > dist[v] (larger is better); (3) stale check reverses to if (d < dist[u]) continue.

Why correctness still holds: If u is popped with capacity dist[u], any alternative path to u with larger min-edge would have produced a predecessor popped earlier with at least that capacity, already relaxing u to a better value — contradiction. The greedy argument holds with the new direction.

Delta from base Dijkstra: Relaxation: changed → min(dist[u],c), maximize · State: same · Frontier: changed → max-heap with positive values · Bookkeeping: same

Code delta:

dist[s] = LLONG_MAX;
auto nd = min(dist[u], 1LL*c);
if (dist[v] < nd)
if (d < dist[u]) continue;

When to use: Widest path, maximum bandwidth route, maximum capacity transport. Keywords: "capacity," "bandwidth," "heaviest path."

Common pitfall: Using dist[s] = 0 as in standard Dijkstra. For maximin the source must start at ∞ (infinite capacity). Also reversing the stale check direction is required.

Practice problem: POJ 1797 — Heavy Transportation → maximize the minimum-weight edge along the transport route — textbook maximin relaxation with max-heap.

V13 Maximum Probability Path Multiply edge probabilities — maximize the product

Story: Each edge succeeds with probability p ∈ [0,1]. Find the path from s to t that maximizes the probability of successfully traversing all edges.

Abstraction: Path cost = product of edge probabilities; maximize over all s→t paths. Initialize dist[s] = 1.0, dist[v] = 0.0 for others. Use max-heap with positive values.

What stays the same: The greedy extraction principle and the stale check are preserved. We extract highest-probability nodes first; the monotone order argument still applies.

What changes: Three changes from V1: (1) relaxation becomes nd = dist[u] * c; (2) update when nd > dist[v]; (3) floating-point epsilon comparison replaces exact equality in stale check.

Why correctness still holds: Greedy order by highest probability is correct: if a better path to u existed, its prefix would also have ≥ that probability and would have been pushed earlier with a higher key, contradicting u being popped now.

Delta from base Dijkstra: Relaxation: changed → dist[u]*c, maximize · State: same · Frontier: changed → max-heap positive values · Bookkeeping: same

Code delta:

dist[s] = 1.0;
auto nd = dist[u] * c;
if (dist[v] + 1e-15 < nd)

When to use: Path probability or reliability problems. Keywords: "probability of success," "survival chance." Use double for dist[].

Common pitfall: Initializing dist[s] = 0. The source starts with probability 1.0. Also: exact equality in the stale check is unreliable for doubles — always use epsilon comparison.

Practice problem: LC 1514 — Path with Maximum Probability → maximize product of edge probabilities with multiply relaxation, max-heap, and dist[s]=1.0.

Variants 14–15: K-th Shortest & Second Shortest

V14 K Shortest Paths A node may be popped up to k times

Story: Find the first k shortest route costs from source to target (repetitions of nodes allowed).

Abstraction: Allow each node to be popped up to k times. The i-th time the target is popped from the heap gives the i-th shortest path cost.

What stays the same: The heap frontier, extraction order (smallest cost first), and relaxation formula are identical to V1.

What changes: The stale-entry check is replaced by a pop-count limit. We allow up to k pops per node via cnt[u]. No dist[] array — neighbors are always pushed unconditionally.

Why correctness still holds: The heap always extracts the globally smallest available path cost. The i-th extraction of target is the i-th shortest route cost over all possible paths (including revisits).

Delta from base Dijkstra: Relaxation: same · State: changed → allow k pops per node via cnt[] · Frontier: same · Bookkeeping: changed → cnt[] tracks pop count

Code delta:

vector cnt(n, 0);
if (cnt[u] == k) continue;
cnt[u]++;
for (auto &[v,c] : adj[u]) q.emplace(-(d+c), v);

When to use: Problems asking for the k-th cheapest route. Keywords: "k cheapest flights," "k-th path." Complexity: O(k·m log(k·n)).

Common pitfall: Forgetting cnt[u] == k guard — without it nodes are processed infinitely, causing TLE. Also: negative-weight cycles make k-shortest undefined; this variant requires nonneg weights.

Practice problem: CSES 1198 — Flight Routes → find the K cheapest route costs from node 1 to node n; allow up to k pops per node.

V15 Second Shortest Path Maintain d1[] and d2[] per node

Story: Find the strictly second-shortest route cost from s to t (must differ from the shortest).

Abstraction: Maintain two distance labels per node: d1[v] (shortest) and d2[v] (second shortest, strictly longer). Update both on each relaxation using a swap-cascade.

What stays the same: The heap frontier and relaxation formula (d + c) are identical to V1. We still extract in nondecreasing cost order.

What changes: Instead of one dist[v], we maintain d1[v] and d2[v]. The stale check becomes if (d > d2[u]) continue. The relaxation cascades: a new candidate first competes for d1[v], then for d2[v].

Why correctness still holds: We maintain the two smallest discovered costs for every node. Extracted in nondecreasing order, no unseen candidate can invalidate established labels. Answer is d2[t].

Delta from base Dijkstra: Relaxation: same · State: same · Frontier: same · Bookkeeping: changed → maintain d1[], d2[] per node with cascade

Code delta:

vector d1(n,LLONG_MAX), d2(n,LLONG_MAX);
if (nd < d1[v]) { swap(nd, d1[v]); q.emplace(-d1[v], v); }
if (nd > d1[v] && nd < d2[v]) { d2[v]=nd; q.emplace(-d2[v], v); }

When to use: Problems asking for the strictly second-shortest path. Strictly means d2[t] > d1[t]; if the problem allows equal-cost second path, adjust the condition.

Common pitfall: Allowing d2[v] == d1[v]. The condition must be nd > d1[v] (strictly), otherwise equal-cost paths contaminate the second-shortest slot.

Practice problem: CSES 1196 — Second Shortest Path → maintain d1[v] and d2[v] per node; answer is d2[n−1].

Variant 16: Shortest-Path DAG Extraction

V16 Extract the DAG of All Shortest Paths, then Run DP Dijkstra + O(m) filter + any DAG DP

Story: After finding all shortest distances, answer a DP question over the entire set of shortest paths simultaneously (e.g., count them, find minimum-edge-count one).

Abstraction: Phase 1: standard Dijkstra fills dist[]. Phase 2: filter edges by the shortest-path identity dist[u]+c == dist[v] to build the shortest-path DAG. Phase 3: any DP on the DAG.

What stays the same: Phase 1 is identical to V1 — all greedy invariants, the relaxation, frontier, and stale check are unchanged.

What changes: Only postprocessing: Phases 2 and 3 are added after Dijkstra completes. The Dijkstra code itself is untouched.

Why correctness still holds: An edge (u,v,c) lies on some shortest path from s iff dist[u]+c == dist[v]. Every path composed solely of such edges is a shortest path from s. Phase 3 DP on the DAG is correct because it is a DAG (no cycles).

Delta from base Dijkstra: Relaxation: same · State: same · Frontier: same · Bookkeeping: changed → post-filter edges by dist[u]+c==dist[v], build DAG, run DP

Code delta:

vector> dag(n);
for (auto &[u,v,c] : edges)
    if (dist[u] != LLONG_MAX && dist[u]+c == dist[v])
        dag[u].push_back(v);

When to use: Any problem asking about ALL shortest paths: "count them," "is edge X on some shortest path," "minimum/maximum edges among shortest paths." Complexity: O((n+m) log n + m).

Common pitfall: Checking dist[u]+c == dist[v] without verifying dist[u] != LLONG_MAX first. Overflow on LLONG_MAX + c wraps to negative, falsely matching the identity.

Practice problem: CSES 1202 — Investigation → build the shortest-path DAG via the dist[u]+c==dist[v] filter, then count paths on it.

Variants 17–18: Implicit & Grid Dijkstra

V17 Implicit-Graph Dijkstra Generate neighbors on demand — state space too large to store

Story: The graph is too large to build explicitly upfront. From each state, neighbors are generated by a transition function — only reachable states are ever visited.

Abstraction: Replace the adjacency list with a lazy neighbor-generation function gen(u). Dijkstra operates exactly as normal, calling gen(u) instead of iterating over adj[u].

What stays the same: The greedy invariant, the relaxation formula, the heap frontier, and the stale check are all identical to V1.

What changes: Only graph representation: instead of for (auto &[v,c] : adj[u]), we call for (auto &[v,c] : gen(u)).

Why correctness still holds: As long as gen(u) correctly enumerates all and only the legal outgoing transitions with nonneg costs, standard Dijkstra on the implicit graph is correct by V1's proof.

Delta from base Dijkstra: Relaxation: same · State: changed → implicitly defined via gen(u) · Frontier: same · Bookkeeping: same

Code delta:

for (auto &[v,c] : gen(u)) {
    auto nd = dist[u] + c;
    if (dist[v] > nd) { dist[v]=nd; q.emplace(-nd,v); }
}

When to use: Problems where the state space is too large to enumerate upfront but transitions from any given state are computable in O(1) or O(small). Keywords: "operations +1, ×2, −3," "minimum moves."

Common pitfall: Forgetting to bound the reachable state space. If gen(u) can produce states up to 10⁹, a vector for dist[] overflows memory — use an unordered_map instead.

Practice problem: LC 1654 — Minimum Jumps to Reach Home → implicit state transitions via +a and −b jumps with a forbidden zone; generate neighbors on demand.

V18 Grid Dijkstra 2D grid with weighted cells

Story: Move on a 2D grid where each cell has a movement cost. Find the cheapest path from the source cell to the target cell.

Abstraction: Each cell (r,c) is a node; 4-directional moves are edges with nonneg costs. dist is a 2D array. Priority queue entries are (cost, row, col) triples.

What stays the same: The greedy invariant, the relaxation formula, the heap frontier, and the stale check are all identical to V1.

What changes: Only state encoding: node IDs become (row, col) pairs, dist is 2D, and neighbor generation uses the 4-direction delta array.

Why correctness still holds: Each cell is a node in a graph with nonneg edge weights. V1's proof applies directly.

Delta from base Dijkstra: Relaxation: same · State: changed → (row, col) pairs; dist[r][c] · Frontier: same · Bookkeeping: same

Code delta:

if (dist[nx][ny] > dist[x][y] + c) {
    dist[nx][ny] = dist[x][y] + c;
    q.push({dist[nx][ny], nx, ny});
}

When to use: Grid-based shortest-path problems where cell traversal costs vary. If all costs are 1, use BFS. If all costs are {0,1}, use V7 (0-1 BFS). If costs vary, use this. Complexity: O(nm log(nm)).

Common pitfall: Forgetting bounds checking before accessing dist[nx][ny]. Out-of-bounds access causes undefined behavior and is the most common grid-Dijkstra bug.

Practice problem: SPOJ SHOP — Shopping → grid Dijkstra with numeric-cell traversal costs; state = (row, col) with 4-directional neighbors.

Variants 19–20: Tie-Breaking & A* Search

V19 Lexicographic / Secondary-Criterion Shortest Path Among all shortest paths, choose the best by secondary criterion

Story: Among all shortest-distance paths from s to t, choose the lexicographically smallest node sequence (or the one with fewest edges, or the one optimizing some secondary measure).

Abstraction: Add a secondary criterion array alongside dist[]. On equal-distance relaxations, apply a tie-break update. The distance computation is completely unchanged.

What stays the same: The greedy invariant, the distance computation, the heap frontier, and the stale check are all identical to V1.

What changes: Only bookkeeping: add secondary arrays (e.g., par[v], bestEdge[v]) and an additional else if (dist[v]==nd) branch that updates only the secondary criterion.

Why correctness still holds: The shortest-distance correctness is unchanged. Tie-breaking via equal-distance relaxations updates secondary arrays without ever changing dist[]. The final secondary values are optimal over all shortest paths by induction over the topological order of the shortest-path DAG.

Delta from base Dijkstra: Relaxation: same · State: same · Frontier: same · Bookkeeping: changed → add secondary arrays + equal-dist tie-break branch

Code delta:

vector bestEdge(n, INT_MAX);
} else if (dist[v]==nd && bestEdge[v] > c) {
    par[v]=u; bestEdge[v]=c;
}

When to use: Problems saying "among all shortest paths, choose the one with…" — fewest edges, smallest node IDs, minimum secondary cost. Adds O(1) work per equal-distance relaxation.

Common pitfall: Pushing a new heap entry on equal-distance updates. The distance is unchanged so no new push is needed; pushing creates duplicates that slow processing without changing the answer.

Practice problem: CF 20C — Dijkstra? → among all shortest paths print the lexicographically smallest sequence of node IDs — a secondary tie-break on equal distances.

V20 A* Search Admissible heuristic guides expansion order

Story: You know a lower-bound estimate h(u) from every node to the target. Use it to expand fewer nodes by prioritizing nodes whose f = g + h is smallest.

Abstraction: Replace the priority key in the heap from g (true cost from s) to f = g + h(v). Store both f and g in each PQ entry. dist[] tracks g-values, not f-values.

What stays the same: The heap frontier, the relaxation formula (nd = dist[u] + c), and the stale check on g-values are all identical to V1.

What changes: The priority key changes from g to f = g + h(v). PQ entries are (f, g, u) triples. The heuristic guides expansion order but does not change the final g-values stored in dist[].

Why correctness still holds: If h is admissible (h(v) ≤ d*(v,t)) and consistent (h(u) ≤ c(u,v) + h(v)), then f-values are monotone along the extraction sequence. The first time t is popped, dist[t] is optimal — same as V1.

Delta from base Dijkstra: Relaxation: changed → priority = g + h(v) · State: same · Frontier: same heap with f key · Bookkeeping: changed → store (f,g) in PQ entries

Code delta:

struct State { long long f, g; int u;
    bool operator<(const State &o) const { return f > o.f; } };
q.push({ng + h[v], ng, v});

When to use: Single-source single-target problems where a tight admissible heuristic exists (e.g., Euclidean distance on geometric graphs). A* expands fewer nodes than Dijkstra in practice. Worst-case complexity same as Dijkstra.

Common pitfall: Using an inadmissible heuristic (h(v) > d*(v,t) for some v). This can cause A* to miss the optimal path entirely. Always verify admissibility before using A*.

Practice problem: LC 1778 — Shortest Path in Hidden Grid → A* with admissible heuristic to guide exploration through an unknown grid with minimum queries.

Part IV — Four High-Frequency State Sub-variants

Each is an instance of V8 (state-space Dijkstra) with a specific extra dimension. All run standard Dijkstra on the expanded graph.

A. Parity Dijkstra Shortest path using an even number of edges

Story: Find the shortest path from s to t that uses an even number of edges.

Abstraction: State = (node, parity). Parity flips on every edge traversal. The answer is dist[t][0] (parity 0 = even).

What stays the same: The greedy invariant, the relaxation formula d+c, the heap frontier, and the stale check are identical to V1 on the 2-layer expanded graph.

What changes: State expands from (node) to (node, parity). On each relaxation, parity flips: dist[v][p^1].

Why correctness still holds: Dijkstra on the 2-layer expanded graph (2n nodes, 2m edges) with nonneg weights. Standard V1 proof applies.

Delta from base Dijkstra: Relaxation: same · State: changed → (node, parity); dist[u][0/1] · Frontier: same · Bookkeeping: same

Code delta:

auto nd = d + c;
if (dist[v][p^1] > nd) { dist[v][p^1]=nd; q.push({nd,v,p^1}); }

When to use: Problems with a modular constraint on edge count: "even number of edges," "odd number of edges," "edge count mod k."

Common pitfall: Returning dist[t][1] for even-edge queries. Parity 0 = even, parity 1 = odd — the labels are easy to swap.

Practice problem: CF 498C — Array and Operations → parity-layered graph; state = (node, parity) with parity flipping on each edge.

B. Bitmask Dijkstra Collect keys or items while traversing

Story: You must collect certain items while traversing the graph. Each node may contain an item. Find the cheapest path that collects all required items.

Abstraction: State = (node, mask) where mask is a bitmask of items collected so far. On entering node v carrying item i, mask updates to mask | (1<

What stays the same: The greedy invariant, the relaxation formula d+c, the heap frontier, and the stale check are identical to V1 on the n·2^k expanded graph.

What changes: State expands from (node) to (node, bitmask). The transition: ndMask = mask | gain[v].

Why correctness still holds: Expanded graph has n·2^k nodes and n·2^k·degree edges, all nonneg-weight. Standard V1 proof applies.

Delta from base Dijkstra: Relaxation: same · State: changed → (node, bitmask); dist[u][mask] · Frontier: same · Bookkeeping: same

Code delta:

int ndMask = mask | gain[v];
auto nd = d + c;
if (dist[v][ndMask] > nd) { dist[v][ndMask]=nd; q.push({nd,v,ndMask}); }

When to use: Problems with a small set of collectible items (k ≤ ~18). Keywords: "collect all keys," "visit all checkpoints." State space is n·2^k.

Common pitfall: Using k > 20 without checking memory. At k=20, n·2^k with n=100 exceeds 100 million states — verify feasibility before choosing.

Practice problem: LC 847 — Shortest Path Visiting All Nodes → state = (node, visited_mask); collect all nodes at minimum total cost.

C. Fuel Dijkstra Travel consumes fuel; refuelling costs money

Story: Travel between cities consumes fuel. At each city you may buy fuel at a local price. Find the cheapest journey from s to t.

Abstraction: State = (node, fuel_remaining). Two transition types: buy one unit of fuel at current node (cost += price[u], fuel += 1), or traverse an edge (fuel -= edge_fuel_cost, money unchanged).

What stays the same: The greedy invariant, the heap frontier, and the stale check are identical to V1 on the n·(cap+1) expanded graph.

What changes: State expands to (node, fuel). Both buying and traveling are nonneg-cost transitions.

Why correctness still holds: All transition costs nonneg. Dijkstra on the expanded graph is correct by V1's proof.

Delta from base Dijkstra: Relaxation: same · State: changed → (node, fuel); dist[u][fuel] · Frontier: same · Bookkeeping: same

Code delta:

if(fuel d+price[u])
    { dist[u][fuel+1]=d+price[u]; q.push({dist[u][fuel+1],u,fuel+1}); }
if(fuel>=c && dist[v][fuel-c] > d)
    { dist[v][fuel-c]=d; q.push({d,v,fuel-c}); }

When to use: Problems with a consumable resource (fuel, battery, energy) that can be replenished at nodes for a cost. Keywords: "limited range," "buy fuel," "recharge."

Common pitfall: Forgetting to take min over all fuel levels at the target: answer = min(dist[t][0], dist[t][1], ..., dist[t][cap]).

Practice problem: POJ 3635 — Full Tank? → fuel-constrained travel with variable city prices and tank capacity limit; state = (node, fuel_remaining), exact fuel Dijkstra pattern.

D. Time-modulo Dijkstra Edge costs depend on current time mod k

Story: Edge costs or availability change periodically: the cost of edge (u,v) at time t is cost_at(u,v,t%k). Find the minimum-cost journey from s to t.

Abstraction: State = (node, time_mod_k). Edge cost depends on the current time residue. Expanded graph has k copies of each node.

What stays the same: The greedy invariant, the heap frontier, and the stale check are identical to V1 on the k·n expanded graph.

What changes: State expands to (node, time_mod_k). The new time residue after traversal is (tm + c) % k.

Why correctness still holds: All transition costs nonneg. Dijkstra on the k·n expanded graph is correct by V1's proof.

Delta from base Dijkstra: Relaxation: same · State: changed → (node, time_mod_k); dist[u][tm] · Frontier: same · Bookkeeping: same

Code delta:

int ntm = (tm + c) % k;
if (dist[v][ntm] > d + c)
    { dist[v][ntm]=d+c; q.push({d+c,v,ntm}); }

When to use: Problems with periodic edge costs or availability: "toll prices cycle every k hours," "traffic lights cycle every k minutes." State space is k·n.

Common pitfall: Choosing k too large. If k = 10⁶, the expanded graph has k·n states which may exceed memory. Verify k is manageable (typically ≤ 1000).

Practice problem: CF 827F — Dirty Arkady's Kitchen → time-modulo path problem; state = (node, time mod k) with time-dependent transitions.

Part VI — Final Mental Model: Recognize the Edit, Apply the Template

Every Dijkstra-family problem in competitive programming is a small, controlled edit to one base algorithm. Identify which of the four dimensions changes, apply only that edit, and you are done.

① Change the relaxation
nd = dist[u] + c — sum (V1)
nd = max(dist[u], c) — minimax (V11)
nd = min(dist[u], c) — widest (V12)
nd = dist[u] * c — probability (V13)
② Change the state
(u) — standard
(u, used) — coupon / layered
(u, parity) — even/odd edges
(u, mask) — item collection
(u, fuel) — fuel constraint
(u, tm) — time-modulo
③ Change the frontier
Heap — general nonneg weights (V1)
Linear scan — dense / tiny n (V2)
Deque — binary {0,1} weights (V7)
Bucket — small integer C
④ Change the bookkeeping
par[v] = u — path reconstruction
ways[v] — count paths
mn[], mx[] — edge-count DP
DAG filter — DP over all shortest paths
Tie-break array — secondary criterion

Contest decision guide

When you see a shortest-path problem, ask: "What is the path cost?" If it is a sum → use ①-standard. If it is a max or min → ①-minimax or ①-widest. If it is a product → ①-probability. Then ask: "Is there extra state that affects transitions?" If yes → ②-state-space. Then ask: "Are weights exactly {0,1}?" If yes → ③-deque. Finally ask: "Do I need more than the distances?" Path → ④-par. Count → ④-ways. All paths → ④-DAG.

The master insight: Dijkstra is not 20 algorithms. It is one algorithm with four tunable knobs. Mastering the four knobs means you can solve the entire family from a single template you already have accepted.
