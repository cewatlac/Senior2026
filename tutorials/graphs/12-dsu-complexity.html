① Foundation
What is DSU?

DSU stands for Disjoint Set Union. It's also called Union-Find. Imagine you have a bunch of people, and you're tracking which people are in the same friend group. DSU answers two questions instantly:

find(x)Who is the representative (leader) of the group that x belongs to?
unite(a, b)Merge the groups of a and b into one group.

Real uses: connected components, Kruskal's MST, cycle detection, grouping identical classes.

The Central Challenge

We need both operations to be as fast as possible. The entire story of this guide is about how clever data structure design takes us from terrible O(n) all the way down to practically constant O(α(n)).

Key idea to keep in mind: The "representative" of a group is just whichever node we designate as the root. Two nodes are in the same group if and only if they have the same root.
② Foundation
The Parent Array — Negative Size Trick

We store everything in one integer array called parent[]. The rules are simple:

parent[x] < 0 → x is a ROOTThe group size is -parent[x]. So parent[x] = -5 means x leads a group of 5.
parent[x] ≥ 0 → x is a MEMBERparent[x] is the index of x's parent node. Follow parents upward to reach the root.

The find() Operation
int find(int x) {
    if (parent[x] < 0) return x;  // x is a root
    return find(parent[x]);        // climb to parent
}

Time = depth of node x in the tree. If trees are deep → slow. If trees are shallow → fast. Everything we do is about keeping trees shallow.

The unite() Operation
bool unite(int a, int b) {
    a = find(a); b = find(b);  // get roots
    if (a == b) return false;    // already same group
    parent[b] = a;               // attach b's tree under a
    return true;
}

This is the naïve version. It works, but the trees can become very tall chains. We'll fix that next.

③½ Formal Abstraction
Formal Cost Model — What Are We Bounding?
Operation Definitions
find(x): Returns the canonical representative (root) of the component containing x. May modify parent pointers as a side effect (path compression).
unite(a, b): Merges the component of a and the component of b. Returns false if already same component. Internally calls find() twice.
The Cost Model

We count the number of parent-pointer traversals — how many times we follow a parent[] link. This is the bottleneck operation in both find() and unite().

What Tarjan's theorem actually says:

Let n = number of elements. Let m = total number of find() and unite() operations (m ≥ n).

Then the total number of parent-pointer traversals over all m operations is at most O(m · α(n)).

This gives an amortized cost of O(α(n)) per operation.
Three Distinct Complexity Notions
TypeMeaningFor DSU
Worst-case per operation Upper bound on any single operation in isolation O(log n) — a single find() can still walk O(log n) steps
Amortized per operation Upper bound on average cost per operation over a sequence O(α(n)) — Tarjan's result
Practical / contest Observed behavior on real inputs Effectively O(1). α(n) ≤ 4 for all real inputs.
Do not confuse these: Writing "each find() is O(α(n))" is technically correct (amortized sense) but is dangerous if you then argue a worst-case bound depends on it. A single cold find() before any compression can still cost O(log n).
③¾ Key Observations
Four Key Observations That Drive Everything
1
Naive chains are the enemy.
Without any optimization, repeated unite() calls that always attach the current root under the last root create a linear chain. find() on the tail must traverse the entire chain: O(n) per call.
2
Union by size limits rank growth logarithmically.
By always attaching the smaller tree under the larger, every time a node's depth increases by 1, the group it belongs to at least doubles in size. Since the group can never exceed n, depth is bounded by log₂n. This gives O(log n) find() without any path compression at all.
3
Path compression makes previously expensive paths cheap forever.
After find(x) traverses a path of length k, every node on that path is rewired directly to the root. Future find() calls on these nodes cost O(1). The "expensive" call buys cheapness for all future calls — this is the amortized trade-off.
4
The combination gives Ackermann-level savings.
With union by size bounding how tall trees can get, and path compression aggressively flattening whatever height exists, the combined effect is so powerful that Tarjan could prove the total work is bounded using the inverse Ackermann function — growing so slowly it is constant for all practical purposes.
④ Optimization 1
Union by Size — Proof that depth ≤ log₂ n

The rule: always attach the smaller tree under the larger tree.

bool unite(int a, int b) {
    a = find(a); b = find(b);
    if (a == b) return false;

    if (-parent[a] < -parent[b]) swap(a, b); // a = larger root

    parent[a] += parent[b];  // sum the (negative) sizes
    parent[b] = a;            // attach smaller under larger
    return true;
}
Why parent[a] += parent[b] works

Both are roots, so both values are negative. If a has size 5 and b has size 3:

Before:
parent[a] = -5
parent[b] = -3
After:
parent[a] = -5 + (-3) = -8 ✓
parent[b] = a (now points to a)
The Key Insight (One Sentence)
Every time a node's depth increases by 1, the size of its group at least doubles.
Full Proof
1
Take any node x. Initially it's alone: size = 1, depth = 0.
2
x's depth only increases when x's root is attached under a bigger root. By union-by-size, that other tree has size ≥ current size. So new size ≥ size + size = 2 × old size.
3
Build the chain: depth 0 → size ≥ 1  |  depth 1 → size ≥ 2  |  depth 2 → size ≥ 4  |  depth d → size ≥ 2d
4
But no group can have more than n nodes. So: 2d ≤ n  →  d ≤ log₂ n. QED.

⑥ Optimization 2
Path Compression — Flattening on the Way Up

Even with union by size (giving O(log n)), we can do better. Path compression is a one-line change to find():

// Without path compression:
int find(int x) {
    if (parent[x] < 0) return x;
    return find(parent[x]);
}

// With path compression (one extra assignment):
int find(int x) {
    if (parent[x] < 0) return x;
    return parent[x] = find(parent[x]); // ← CHANGED
}

The difference: after finding the root, we directly set every visited node's parent to the root. The path is flattened.

What is Amortized Complexity?

Amortized does not mean every operation is fast. It means:

Over a long sequence of operations, the average cost per operation is small.

Example: 10 operations with costs [8, 1, 1, 1, 1, 1, 1, 1, 1, 1] → total=17, average=1.7. The one expensive operation "donated" speed to the future ones.

⑩ The Big Question
Why Does Path Compression Lead to O(α(n))?

This is the hardest conceptual part. We'll build the intuition step by step without going through Tarjan's full formal proof.

Step 1 — Union by size already limits heights to O(log n)

We proved this. Without path compression, trees are never deeper than log₂(n). So the raw structure is already pretty good.

Step 2 — Path compression is more than just "preventing height growth"
Path compression doesn't just stop trees from getting taller. It actively rewires nodes much closer to the root. So a node that was expensive once cannot keep being expensive forever.
Step 3 — The bucket grouping insight

Here is the core of Tarjan's argument in simple language:

A
Assign each node a "rank" (roughly = the maximum tree height it was ever a root of). Ranks grow slowly with n.
B
Group ranks into "buckets" — but not ordinary buckets. The boundaries between buckets grow using Ackermann-like speeds:

Bucket 1: ranks 0–1
Bucket 2: ranks 2–3
Bucket 3: ranks 4–... (up to A(3,4) = 125)
Bucket 4: ranks 126–... (up to A(4,4) = unimaginably large)
Bucket 5: ...never encountered in practice
C
Count the buckets: Because the bucket boundaries grow at Ackermann speed, the number of buckets up to rank n is only α(n).
D
When path compression moves a node upward, its parent jumps to a node of much higher rank. This means the node crosses into a higher bucket. Since there are only α(n) buckets total, this crossing can only happen α(n) times per node.
E
Therefore the total work done across all path compressions over m operations is bounded by O(m · α(n)).
Simplest Mental Picture
Think of it this way:

Without path compression: each find() cost is proportional to depth. Depth is O(log n). So every find = O(log n). This measures progress in "doubling steps" — log₂(n) of them.

With path compression: nodes jump upward in their tree so aggressively that Tarjan can measure progress using Ackermann-sized jumps instead. These are so enormous that the total number of "levels" to traverse is only α(n) — not log n.
Why Not Just Say O(1)?

Mathematically, α(n) is not a constant. As n → ∞, α(n) → ∞ as well (just incredibly slowly). The precise statement is:

α(n) is not O(1). It is a well-defined function that grows, just absurdly slowly. We write O(α(n)) to be mathematically correct.
In practice: α(n) ≤ 4 for any n you'll ever see. So it behaves like a constant, but we don't say it is a constant.
The Critical Correctness Point
This is AMORTIZED O(α(n)), not worst-case per operation.

A single find() call can still visit many nodes (if the path hasn't been compressed yet). What Tarjan proved is that over any sequence of m operations, the total time is O(m · α(n)), so the average time per operation is O(α(n)).
⑪ Code Deep-Dive
The Final DSU Template — Line by Line

Here is the full, production-grade DSU template. Below the code every non-trivial line is explained with the invariant it maintains and why it is written the way it is.

struct DSU {
    vector<int> parent;               // [1]

    DSU(int n) {                       // [2]
        parent.assign(n + 1, -1);    // [3]
    }

    int find(int x) {
        if (parent[x] < 0) return x;            // [4]
        return parent[x] = find(parent[x]);    // [5]
    }

    bool unite(int a, int b) {
        a = find(a); b = find(b);              // [6]
        if (a == b) return false;               // [7]
        if (-parent[a] < -parent[b]) swap(a, b); // [8]
        parent[a] += parent[b];                  // [9]
        parent[b] = a;                           // [10]
        return true;
    }

    int  size(int x) { return -parent[find(x)]; }  // [11]
    bool same(int a, int b) { return find(a) == find(b); } // [12]
};
Line-by-Line Explanations
[1] vector<int> parent — The entire data structure in one array

A single integer array encodes two things using the sign of each entry:

parent[x] < 0 → x is a root. −parent[x] is the group size.
parent[x] ≥ 0 → x is an internal node. parent[x] is x's parent index.

This "negative-size encoding" avoids a second array for sizes, halving memory usage and improving cache locality. The invariant is: every root's entry is negative; every non-root's entry is a valid index into the array.

[2] DSU(int n) — Constructor takes the number of elements

n is the count of elements. The array is sized n+1 to allow 1-indexed usage (nodes 1…n). If you use 0-indexed nodes (0…n-1), pass n and use parent.assign(n, -1) instead. This is a frequent indexing bug — see the pitfalls section.

[3] parent.assign(n + 1, -1) — Initialize all as singleton roots of size 1

Starting with −1 means every node is a root of size −(−1) = 1. This is the correct initial state: n disjoint singletons. Common mistake: initializing with 0 or with the node's own index — both break the negative-size invariant and will produce silent wrong answers.

[4] if (parent[x] < 0) return x — Base case: x is a root

The invariant says a root has a negative parent entry. This is the recursion's base case. Note: this also handles the case where x is its own singleton — after initialization, every node satisfies this check immediately, returning itself.

Invariant maintained: At any point in the algorithm, if parent[x] < 0, then x is exactly the canonical representative of its group.
[5] return parent[x] = find(parent[x]) — Path compression (the magic line)

This is the entire path-compression mechanism in one assignment. The recursion climbs up to the root, and on the way back down (due to how recursive return works in C++), each node's parent pointer is updated to point directly to the root that was just returned.

→
Without this assignment: return find(parent[x]) — just climbs up. No side effects.
→
With assignment: the recursive call returns the root r. Then parent[x] = r fires before returning r upward. Every node on the path gets rewired to r in this single call.
Why it's safe: Setting parent[x] = root doesn't change which group x belongs to — root is still the representative. It only changes how quickly we reach it next time.
Stack depth concern: On very deep trees before any compression, recursion depth can reach O(log n) (with union by size). For n ≤ 10⁶ this is roughly 20 levels — safe. For extremely large n or repeated calls on uncompressed paths in adversarial inputs, consider an iterative two-pass implementation.
[6] a = find(a); b = find(b) — Get the canonical representatives first

We never unite arbitrary nodes — we always unite their roots. This is crucial: if you forget these lines and pass a and b directly, you may attach a non-root as a parent of another non-root, corrupting the tree structure and breaking the invariant. After this line, both a and b are roots (parent entries are negative).

These two find() calls also trigger path compression on both paths — a welcome side effect.

[7] if (a == b) return false — Same group: nothing to do

If both roots are identical, a and b were already connected. We return false to signal "no merge happened." This allows callers to count the number of distinct merge operations, which equals the number of edges that reduce the component count — useful for Kruskal's algorithm. Without this check, you'd corrupt the data structure by self-merging a root.

[8] if (-parent[a] < -parent[b]) swap(a, b) — Union by size

Since parent[a] and parent[b] are both negative, -parent[a] is the size of group a. We want a to be the larger group. If a is currently smaller, we swap so that after the swap, a is the larger one.

Why size and not rank? Union by size and union by rank both give O(log n) depth. This template uses size because it is directly available in the negative-size encoding — no extra tracking needed. The complexity theorem holds for both variants.
Note on union by rank: "Rank" is an upper bound on tree height that never decreases, even after path compression flattens the tree. Ranks are thus more conservative than actual heights. Union by rank is slightly more elegant theoretically but uses a second encoding. In contests, union by size is standard.
[9] parent[a] += parent[b] — Update the new root's size

Both values are negative. Adding them gives the correct (negative) total size:

parent[a] = −size_a, parent[b] = −size_b
parent[a] + parent[b] = −(size_a + size_b) = −new_total_size ✓

After this line, parent[a] correctly encodes that a is a root of group size size_a + size_b. This is the only line that modifies the root's own entry — root entries are otherwise never touched except during this merge.

[10] parent[b] = a — Attach the smaller root under the larger

This makes b a non-root. Its parent entry is now a (a non-negative index), so the invariant is maintained. The smaller tree is now a subtree of the larger tree. Because of line [8], a is guaranteed to be the larger tree, so depth of existing members of a's tree is unchanged — only members of b's tree get their depth increased by 1.

This single-pointer change is why union is O(1) after the two find() calls.

[11] size(x) and [12] same(a, b) — Convenience helpers

size(x) calls find(x) to get the root, then negates the root's parent value. Note this triggers path compression on x's path — a useful side effect.

same(a,b) just compares roots. Two nodes are in the same component if and only if their root is the same canonical representative.

⑫ Edge Cases
Tricky Edge Cases You Must Know
0-indexing vs 1-indexing
The bug: Constructor called with n for 1-indexed nodes (1…n), but array sized to n. Node n accesses parent[n] which is out of bounds.

Fix: always pass n+1 to the constructor when nodes are 1-indexed, or explicitly document your convention. A safe idiom: always use parent.assign(n + 1, -1) and index from 1.

// 1-indexed (nodes 1..n): use assign(n+1, -1)
DSU(int n) { parent.assign(n + 1, -1); }

// 0-indexed (nodes 0..n-1): use assign(n, -1)
DSU(int n) { parent.assign(n, -1); }
find() called on an out-of-range node
If your graph reads edges as 1-indexed but you initialize DSU as 0-indexed, find(n) accesses undefined memory. This is not caught at compile time and produces silent wrong answers or crashes.

Always add an assertion in debug builds: assert(x >= 0 && x < (int)parent.size()); at the top of find().

unite(x, x) — Merging a node with itself

find(x) == find(x) is always true, so the check if (a == b) return false handles this correctly — nothing happens. This edge case comes up when reading self-loops in graphs (an edge u–u). The template handles it safely.

Negative-size encoding: reading size correctly
Wrong: int sz = parent[x]; — this gives a negative number if x is a root.
Wrong: int sz = parent[find(x)]; — still negative.
Correct: int sz = -parent[find(x)];

The helper size(x) exists precisely to avoid this mistake. Use it.

Path compression changes parent[] entries of non-roots

After path compression, parent[x] for a non-root x may jump past several levels directly to the root. This means parent[x] no longer necessarily encodes x's original structural parent. This is intentional and correct for find()/unite(), but it means you cannot use parent[] to traverse the original tree structure after operations begin.

If your algorithm needs to remember which nodes were merged in which order (e.g., for rollback), path compression is incompatible. Use rollback DSU (without path compression) instead.
Amortized vs worst-case: a single find() can still be O(log n)

On a freshly initialized DSU with union-by-size, the very first find() on the deepest node of a tall tree costs O(log n). The O(α(n)) guarantee is amortized over a sequence of operations — not a per-operation guarantee. In practice, after a warm-up phase of operations, nearly all find() calls become O(1). But for complexity proofs, write O(α(n)) amortized, not O(1).

Recursive find() and stack overflow for large n

With union by size, max tree depth before compression is O(log n). For n = 10⁶ that's ~20 recursive calls — safe. However, if you use path compression alone without union by size, a worst case input can create O(n) deep trees, causing stack overflow on the first find(). Using both optimizations together avoids this.

For ultra-safe code, use the iterative two-pass find():

int find(int x) {
    int root = x;
    while (parent[root] >= 0) root = parent[root];  // find root
    while (parent[x] >= 0) {                         // compress path
        int nxt = parent[x];
        parent[x] = root;
        x = nxt;
    }
    return root;
}
⑬ Common Pitfalls
Bugs, Misconceptions, and Contest Traps
🐛 Pitfall: Forgetting to call find() before merging
Wrong:
parent[b] = a; // directly attaching arbitrary nodes

Why it breaks: If a and b are not roots, you link non-roots to non-roots, creating cycles or multiple parents. The union-by-size logic also fails because -parent[a] is a positive index, not a negative size.
Always: call a = find(a); b = find(b); first.
🐛 Pitfall: Writing O(1) instead of O(α(n)) in a proof
α(n) is not a constant. Mathematically, α(n) → ∞ as n → ∞. Writing "O(1) per operation" is incorrect and will lose marks in formal analysis. The correct statement is: O(α(n)) amortized per operation, which means O(m · α(n)) for m operations on n elements.

In competitive programming commentary, "effectively O(1)" is acceptable as informal shorthand. In formal proofs or theoretical computer science, always write O(α(n)).

🐛 Pitfall: Assuming path compression alone gives O(α(n))

Path compression without union by size (or rank) gives amortized O(log n) per operation, not O(α(n)). Both optimizations are required together for the O(α(n)) bound. Tarjan's theorem applies to the combination.

What you haveAmortized bound
NeitherO(n)
Union by size onlyO(log n) worst-case
Path compression onlyO(log n) amortized
Both togetherO(α(n)) amortized
🐛 Pitfall: Using rollback + path compression together
Fatal combination: Path compression modifies many parent[] entries in a single find() call. If you need to roll back operations (e.g., for offline divide-and-conquer algorithms), you cannot efficiently undo these side effects. You would need to remember every single parent[] change, not just the two entries modified by unite().

The standard fix: rollback DSU uses union by rank (not size) and no path compression. This gives O(log n) per operation — worse than O(α(n)), but supports rollback. See the Variants section.

🐛 Pitfall: Forgetting that size() calls find()

size(x) calls find(x), which may compress x's path. This is a beneficial side effect, but be aware: if you rely on parent[x] being a specific value right after calling size(x), it may have changed due to path compression.

🐛 Pitfall: Initializing with i instead of -1
Some tutorials write parent[i] = i (node is its own parent) instead of parent[i] = -1. This requires a different find() implementation. If you mix these conventions — one for find and one for the other — you get silent wrong answers. Stick to one convention consistently throughout your code.
🐛 Pitfall: DSU on Trees (small-to-large) is NOT the same concept
"DSU on tree" or "small-to-large merging" is a completely different algorithmic technique for tree DP problems. It happens to use similar size-based reasoning but has nothing to do with the Union-Find data structure or the O(α(n)) bound. Do not confuse them in analysis or conversation.
⑭ DSU Family
DSU Variants & Extensions — How Complexity Changes

The basic DSU is a foundation. Real problems often require augmented versions. Here is how the complexity analysis applies to each variant.

Basic DSU (this tutorial)
Union by size + path compression → O(α(n)) amortized per operation. The full theorem applies. This is the standard CP template.
Rollback DSU (DSU with undo)

Use case: Dynamic connectivity with edge deletions offline; divide-and-conquer on time segments.

Key change: Path compression is disabled. Instead, union by rank is used, and a stack records every parent[] change. To roll back, pop the stack and restore old values.

Complexity: O(log n) per operation (no path compression means no α(n) magic). This is acceptable because rollback DSU is used in situations where each edge appears for a contiguous time interval, and the total number of roll-backs is bounded.
// Rollback DSU — no path compression!
struct RollbackDSU {
    vector<int> par, rnk;
    vector<pair<int*,int>> history; // pointer + old value

    RollbackDSU(int n) : par(n+1,-1), rnk(n+1,0) {}

    int find(int x) {                         // NO compression
        while (par[x] >= 0) x = par[x];
        return x;
    }

    bool unite(int a, int b) {
        a = find(a); b = find(b);
        if (a == b) return false;
        if (rnk[a] < rnk[b]) swap(a, b);    // union by rank
        history.push_back({&par[b], par[b]});
        history.push_back({&rnk[a], rnk[a]});
        par[b] = a;
        if (rnk[a] == rnk[b]) rnk[a]++;
        return true;
    }

    void rollback() {             // undo last unite
        *history.back().first = history.back().second; history.pop_back();
        *history.back().first = history.back().second; history.pop_back();
    }
};
Parity DSU (bipartite / XOR relation DSU)

Use case: Determining if a graph is bipartite, detecting odd cycles, XOR difference constraints between nodes.

Key change: Each node x carries a "parity" value (0 or 1) representing its relationship to its parent. Find() returns both the root and the XOR-parity of the path. Unite() combines parities.

Complexity: Same O(α(n)) amortized. The parity field is maintained alongside the parent pointer, adding O(1) per operation. Path compression compresses the parity information as well.
struct ParityDSU {
    vector<int> par, dist; // dist[x] = XOR parity to parent

    ParityDSU(int n) : par(n+1,-1), dist(n+1,0) {}

    pair<int,int> find(int x) {
        if (par[x] < 0) return {x, 0};
        auto [root, d] = find(par[x]);
        par[x] = root;
        dist[x] ^= d;          // compress parity too
        return {root, dist[x]};
    }

    // unite(a, b, w): declare parity(a) XOR parity(b) = w
    bool unite(int a, int b, int w) {
        auto [ra, da] = find(a);
        auto [rb, db] = find(b);
        if (ra == rb) return (da ^ db) == w; // consistency check
        if (-par[ra] < -par[rb]) swap(ra, rb), swap(da, db);
        par[ra] += par[rb]; par[rb] = ra;
        dist[rb] = da ^ db ^ w;
        return true;
    }
};
Weighted DSU (relation/potential DSU)

Use case: "The weight of node a minus the weight of node b is exactly d" — generalized weighted difference constraints; modular relations; union with fractional / integer offsets.

Key change: Each node carries a "potential" (weight offset relative to its parent). Find() accumulates the potential on the way up. Path compression sums potentials correctly.

Complexity: O(α(n)) amortized. The extra potential field adds O(1) per operation, so the asymptotic bound is unchanged.
Caveat: Path compression with potentials requires careful accumulation — you must propagate the potential sum up during the recursive unwinding. Missing this step produces wrong answers that can be very hard to debug.
Path Compression Variants (halving, splitting)

Besides full path compression (every node on path → root), there are milder variants:

Path halving: every other node on the path skips one level. Easier to implement iteratively, same O(α(n)) bound.
Path splitting: every node points to its grandparent. Also O(α(n)), easier to implement correctly than full compression in iterative style.

All three variants achieve the same O(α(n)) amortized complexity when combined with union by size or rank. Full path compression (as in the template) is standard in competitive programming.

⑮ CP Applications
Competitive Programming Applications

DSU appears in many problem categories. Here is a selection of representative applications, showing how the complexity analysis connects to each.

📌 Connectivity Queries (Static)

Story: Given n cities and m roads, answer q queries: "Are cities a and b connected?"

Abstraction: Each city is a node. Each road is a unite(). Each query is a same(). After building the DSU in O(m·α(n)), each query is O(α(n)).

Why DSU: If the graph is static (no edge deletions), DSU is optimal. Total complexity O((m+q)·α(n)).

Try it: POJ 1611 — The Suspects (VJudge)  |  HDU 1213 — How Many Tables

📌 Kruskal's MST Algorithm

Story: Find the minimum spanning tree of a weighted graph.

Abstraction: Sort edges by weight. Iterate: for each edge (u,v,w), if same(u,v) is false, call unite(u,v) and add the edge to the MST. Stop after n−1 edges are added.

Why DSU: The "is adding this edge a cycle?" check is exactly same(). Total time: O(m log m) for sorting + O(m·α(n)) for DSU operations = O(m log m).
Complexity relevance: The α(n) term is completely dominated by the O(m log m) sort. DSU is not the bottleneck — but correctness and speed of the unite/same operations matter for large m.

Try it: POJ 1287 — Networking (VJudge)  |  CSES — Road Reparation

📌 Dynamic Connectivity (Offline)

Story: Edges are added and removed over time. For each query at time t, are nodes a and b connected?

Abstraction: Offline divide-and-conquer on the time axis. Each edge [l,r] in time contributes to a segment tree of time. At each segment tree node, process edges using rollback DSU.

Complexity: O((m + q) log² n) using rollback DSU (O(log n) per operation) on O(log q) segment tree levels. Path compression must be disabled — this is why rollback DSU cannot use O(α(n)).

Try it: CF 566D — Restructuring Company  |  CF Gym — Dynamic Connectivity (classic)

📌 Bipartite Check / Parity Contradiction Detection

Story: Given an undirected graph, determine if it is bipartite. Equivalently: color nodes 2-colorably so that no edge connects same-colored nodes.

Abstraction: Use Parity DSU. For each edge (u,v), unite(u,v, 1) meaning "u and v must differ in color." If unite() returns false (same component), check that parity(u) XOR parity(v) == 1. If not, there is an odd cycle → not bipartite.

Complexity: O(m·α(n)) — same as standard DSU. The parity overhead is O(1) per operation.

Try it: POJ 1986 — Distance Queries (weighted DSU)  |  CF 277A — Learning Languages

📌 Cycle Detection in Undirected Graphs

Story: Given a graph, does it contain a cycle? (Classic pre-check before running tree algorithms.)

Abstraction: For each edge (u,v): if same(u,v) → cycle found. If not: unite(u,v). If we process all m edges without a same() returning true → no cycle (the graph is a forest).

This is the backbone of Kruskal's MST: each edge that creates a cycle is discarded, each that doesn't is added.

Try it: CSES — Building Roads  |  HDU 1233 — Still Another Warcraft (Kruskal + DSU)

📌 Weighted/Potential DSU — Difference Constraints

Story: "The salary of employee A is exactly d more than employee B." Given many such relations, answer queries: "What is the salary difference between X and Y?" or detect contradictions.

Abstraction: Weighted DSU where each node stores a potential (offset from its root). unite(a, b, d) means potential(a) - potential(b) = d. find() returns the root and the accumulated offset.

Complexity: O(α(n)) amortized — same as standard DSU.

Try it: POJ 1986 — Distance Queries  |  CF 1850H — The Third Generation (weighted DSU)

⑯ Advanced
Advanced Extensions & What Comes Next
When Does the O(α(n)) Theorem Stop Applying?

The theorem requires:

1. Static elements: The n elements are fixed throughout all operations.
2. Both optimizations together: Union by size (or rank) and path compression.
3. Amortized analysis context: You are counting total work over a sequence, not bounding individual operations.
4. No rollback: Path compression is not undone.

Any of the following invalidate the exact O(α(n)) bound:

• Path compression disabled (rollback DSU): O(log n) per operation.
• Online dynamic edge deletions: standard DSU doesn't support this at all. Need Link-Cut Trees or offline techniques.
• Persistent DSU (copying the structure): O(log n) per operation using persistent arrays or fat nodes.
Link-Cut Trees

The natural "what comes next" after DSU is the Link-Cut Tree (LCT), which supports dynamic connectivity with both edge insertions and deletions in O(log n) per operation. LCTs use a completely different representation (splay trees of access paths) and are significantly more complex to implement. DSU's elegance — a single array, ~10 lines — has no equivalent in LCTs.

Persistent DSU

Using a persistent array (functional tree of arrays), we can make DSU persistent: maintain multiple "versions" of the DSU simultaneously. Each unite() creates a new version in O(log n). Find() on any historical version costs O(log n). Path compression is typically disabled for clean persistence semantics. This is useful for offline queries that need to "branch" the state of connectivity.

Offline vs Online
DSU (offline-friendly): All queries known in advance. Process in any order. Ideal for sorting edges by weight (Kruskal), time-segment tricks (rollback), or sorting queries by some parameter.
Online connectivity: Queries arrive in real time, can't reorder. Need LCT or other data structures. O(log n) per operation, much harder to code.
Formal Proof Reference
Tarjan (1975): "Efficiency of a Good But Not Linear Set Union Algorithm." Journal of the ACM, 22(2), 215–225.

Tarjan & van Leeuwen (1984): Extended the result to path splitting and path halving variants.

Fredman & Saks (1989): Proved that Ω(α(n)) is also a lower bound — no union-find data structure can be asymptotically faster.

This means DSU with union by size + path compression is asymptotically optimal.

🧠 Big Picture
DSU is a masterclass in amortized analysis.

The reason O(α(n)) is achievable is that the two optimizations cooperate: union by size prevents the tree from getting taller (so there is a finite amount of "depth budget"), and path compression rapidly spends that budget by flattening. Once a path is flattened, it costs almost nothing forever after. Tarjan proved that the rate at which this flattening budget is exhausted is governed by the Ackermann function — the fastest-growing computable function — and therefore the inverse Ackermann governs how slowly the budget runs out.

Complete DSU Template
struct DSU {
    vector<int> parent;

    DSU(int n) { parent.assign(n + 1, -1); }

    int find(int x) {
        if (parent[x] < 0) return x;
        return parent[x] = find(parent[x]); // path compression
    }

    bool unite(int a, int b) {
        a = find(a); b = find(b);
        if (a == b) return false;
        if (-parent[a] < -parent[b]) swap(a, b); // union by size
        parent[a] += parent[b];
        parent[b] = a;
        return true;
    }

    int size(int x) { return -parent[find(x)]; }
    bool same(int a, int b) { return find(a) == find(b); }
};
