All-Pairs Shortest Paths — one problem, two worlds. Complete proofs, normalized applications, animated tracers, and AC-ready templates.

1 · One Problem, Two Worlds

Before studying Floyd–Warshall and Johnson as separate algorithms, you must understand the single unifying problem they both solve — and why they solve it in two completely different ways.

The APSP Problem: Given a weighted graph with n vertices, compute the shortest path distance between every pair of vertices (u, v). Output: an n×n matrix where dist[u][v] = minimum-weight path from u to v.
Single-Source (SSSP):
One chosen source → distances to all nodes.
→ Dijkstra: O(m log n), nonneg edges only.
→ Bellman-Ford: O(nm), handles negatives.
All-Pairs (APSP):
Every vertex as a source → full distance matrix.
Output: dist[u][v] for all pairs (u,v).
→ Floyd–Warshall or Johnson.

Johnson's Algorithm World — Graph Transformation

These are two entirely different algorithmic ideas solving the same global problem.

Floyd is a pure DP — no graph transformation, no single-source subroutine. Johnson is a two-stage transformation algorithm — it transforms the problem into something easier, then solves the easier problem repeatedly.

The choice between them is determined by: graph density (dense → Floyd, sparse → Johnson), whether negative edges exist, and whether you need matrix-based variants (closure, minimax, etc. → Floyd only).

O(m log n) per source → O(nm log n) total. Fast! But invalid when negative edges exist.

O(n³) DP. Works with negatives. Best when n is small/medium and graph is dense.
Johnson: Transform graph so Dijkstra becomes valid, then run repeated Dijkstra. O(nm log n). Best on sparse graphs with potential negative edges.

2 · Graph Fundamentals We Need

Directed graph:
Edge u → v is one-way. Going from v to u requires a separate edge. Both Floyd and Johnson work on directed graphs natively.
Undirected graph:
Edge {u,v} means travel both ways — equivalent to u→v and v→u. A negative undirected edge creates a negative 2-cycle, which makes Johnson inapplicable (Bellman-Ford detects and rejects it).

A path is a sequence of vertices linked by edges. The path cost is the sum of edge weights along it.

Negative edge: An edge with weight < 0. Example: u → v = −5 means using this edge reduces total path cost by 5.

Both Floyd–Warshall and Johnson handle negative edges correctly. They were specifically designed to do so. Dijkstra cannot — it assumes all edges are non-negative.
Negative cycle: A cycle whose total weight is negative.

Example: A→B = 2, B→C = −7, C→A = 1 → total = −4. Traversing this cycle again gives −8, then −12... The path cost goes to −∞. Shortest paths are undefined.
Floyd: Run normally. Afterward, dist[i][i] < 0 means vertex i is on a negative cycle. You can also propagate the −∞ influence to all reachable pairs.
Johnson: Bellman-Ford detects negative cycles during the potential-computation phase. If detected, johnson() returns false and no APSP matrix is produced.

3 · The Decision Split: Floyd or Johnson?

Before touching any code, you need to correctly identify which world your problem lives in. Here is the fast-decision view:

Choose Floyd–Warshall when:
• n ≤ 500–700 (O(n³) is feasible)
• Graph may be dense (m ≈ n²)
• You need matrix-based variants (closure, minimax, widest path, neg-cycle propagation)
• You want the simplest correct implementation
• Negative edges may exist — no problem
Choose Johnson when:
• Graph is sparse (m ≪ n²)
• n is large, O(n³) is too slow
• Negative edges may exist but no negative cycle
• You need full APSP and can afford more complex code
• Think: Bellman-Ford once + Dijkstra n times

4 · Why Brute Force Fails for APSP

The path-enumeration disaster start here
Naive approach: For each pair (i, j), list every simple path from i to j, compute its cost, take the minimum.
Problem: The number of simple paths can be exponential in n. Even for n = 20, this is completely impractical.
Floyd's insight: Don't enumerate paths directly. Instead, control which vertices may appear as internal intermediates. Process vertices 0, 1, 2, …, n−1 one at a time. After processing vertex k, maintain: the shortest path from every i to every j using only vertices {0, 1, …, k} as internal intermediates.

This creates a perfect DP structure — the state space is small, the transition is simple, and it is correct by induction.

5 · The Core DP Idea

Process vertices 0, 1, 2, …, n−1 one by one. When we process vertex k, we ask: for every pair (i, j), does routing through k improve the known distance?

Case 1 — path does NOT use k
The best path stays at whatever dist[i][j] already holds.
Case 2 — path DOES use k
Split at k: go i → … → k, then k → … → j.
Cost = dist[i][k] + dist[k][j]
Therefore, the Floyd recurrence is:

dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j])

Run this for every k from 0 to n−1, iterating over all pairs (i, j) for each k. That is the entirety of Floyd–Warshall. 
for (int k = 0; k < n; ++k)
    for (int i = 0; i < n; ++i)
        for (int j = 0; j < n; ++j)
            dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j]);

6 · Formal DP Definition & Correctness Proof

State definition, base case, transition the full DP
Define: dp[k][i][j] = shortest path from i to j whose internal intermediate vertices all belong to {0, 1, …, k−1}.
Base case k = 0
No intermediates allowed — only direct edges.
dp[0][i][i] = 0  (self-loop = 0)
dp[0][i][j] = weight(i→j) if edge exists
dp[0][i][j] = INF otherwise
Transition
Path from i to j using intermediates ⊆ {0…k} either:
→ doesn't use k: cost = dp[k][i][j]
→ uses k: cost = dp[k][i][k] + dp[k][k][j]

dp[k+1][i][j] = min(both cases)
Why can we compress to one matrix? Layer k+1 depends only on layer k. When we read dist[i][k] and dist[k][j] during iteration k, they already hold optimal values for intermediates ⊆ {0…k−1} — exactly what the recurrence needs. In-place update is safe.
Full correctness proof by induction expand to read
Claim: After finishing iteration k, dist[i][j] = shortest path from i to j with all internal vertices ⊆ {0,…,k}.

Base Before any iteration, no intermediates allowed → only direct edges and zero-cost self-loops. Matches initialization. ✓

Step Assume true after k−1. Consider the shortest path from i to j with intermediates ⊆ {0,…,k}:

Doesn't use k: Best cost = old dist[i][j]. By inductive hypothesis, this is already optimal for intermediates ⊆ {0…k−1}. ✓
Uses k: Split at k. Both halves use only intermediates ⊆ {0…k−1}. By hypothesis: cost = dist[i][k] + dist[k][j]. ✓

Taking the min of both cases = the Floyd transition. The claim holds. By induction over all k, the final matrix is correct. □

📐 Formal Invariants — Floyd–Warshall

These three invariants together constitute a complete correctness certificate for Floyd–Warshall. Every line of code in the template (§9) directly enforces one or more of them.

Invariant type Formal statement What enforces it in code
State invariant dist[i][j] = length of the shortest path from i to j whose internal vertices all belong to the set {0, 1, …, k−1}, where k is the current outer-loop iteration index. Diagonal init (dist[i][i]=0), edge init (dist[u][v]=w), INF for all other pairs. Maintained by the transition step.
Transition invariant dist[i][j] = min(dist[i][j], dist[i][k] + dist[k][j]). Safe because: (a) dist[i][k] and dist[k][j] are optimal for intermediates ⊆ {0…k−1} at the moment of the update (the in-place update is safe — see §6 proof); (b) the INF/2 guard prevents overflow before the addition. The if (dist[i][k] >= INF/2) continue; outer guard and if (dist[k][j] < INF/2) inner guard in the §9 template.
Termination invariant After the outer loop completes (k = n), dist[i][j] holds the true shortest-path distance from i to j in the original graph, for every pair. If a negative cycle is reachable from i on the way to j, then dist[i][j] may be negative or even dist[i][i] < 0, signalling the cycle. The outer for (int k=0; k<n; ++k) loop. Post-run check: hasNegativeCycle() reads the diagonal.

Cross-variant note: Every Floyd variant in §10 (path reconstruction, transitive closure, minimax, maximin, neg-cycle detection, neg-cycle propagation) preserves the state and termination invariants exactly. Only the transition operator changes (e.g., OR replaces min+, or max/min replaces min/+). This is the formal justification for why all seven variants share the same triple-loop skeleton.

8 · Common Pitfalls in Floyd–Warshall

1
Forgetting dist[i][i]=0 Without this, path computations through any node break. Always initialize the diagonal before running.
2
Multiple edges — keep min If input has several u→v edges, keep only the minimum: dist[u][v] = min(dist[u][v], w).
3
INF overflow Never compute INF+INF. Guard: if (dist[i][k] < INF/2 && dist[k][j] < INF/2). Use 4e18 with long long.
4
Wrong loop order Only valid order: k → i → j. Swapping to i→k→j or any other order breaks the DP invariant and produces wrong answers.
5
Floyd on huge n Floyd is O(n³). For n=1000 that is 10⁹ operations → TLE. Switch to Johnson for large sparse graphs.

9 · Floyd–Warshall Master Template

Full struct with addEdge, run, negative cycle check, path reconstruction AC-ready 
#include <bits/stdc++.h>
using namespace std;
using ll = long long;
const ll INF = (ll)4e18;

struct FloydWarshall {
    int n;
    vector<vector<ll>> dist;
    vector<vector<int>> nxt;

    FloydWarshall(int n) : n(n), dist(n, vector<ll>(n, INF)), nxt(n, vector<int>(n, -1)) {
        for (int i = 0; i < n; ++i) {
            dist[i][i] = 0;
            nxt[i][i] = i;
        }
    }

    void addEdge(int u, int v, ll w, bool directed = true) {
        if (w < dist[u][v]) { dist[u][v] = w; nxt[u][v] = v; }
        if (!directed && w < dist[v][u]) { dist[v][u] = w; nxt[v][u] = u; }
    }

    void run() {
        for (int k = 0; k < n; ++k) {
            for (int i = 0; i < n; ++i) {
                if (dist[i][k] >= INF / 2) continue;
                for (int j = 0; j < n; ++j) {
                    if (dist[k][j] >= INF / 2) continue;
                    ll nd = dist[i][k] + dist[k][j];
                    if (nd < dist[i][j]) {
                        dist[i][j] = nd;
                        nxt[i][j] = nxt[i][k];
                    }
                }
            }
        }
    }

    bool hasNegativeCycle() const {
        for (int i = 0; i < n; ++i)
            if (dist[i][i] < 0) return true;
        return false;
    }

    vector<int> getPath(int u, int v) const {
        if (nxt[u][v] == -1) return {};
        vector<int> path = {u};
        while (u != v) { u = nxt[u][v]; path.push_back(u); }
        return path;
    }
};
🚫 Floyd–Warshall: This Algorithm Fails When…

The conditions below cause Floyd–Warshall to either produce wrong answers, overflow silently, or time out. Each failure mode is distinct and requires a specific fix or algorithm switch.

n > 600 on a 1-second time limit

O(n³) = 2.16×10⁸ at n=600 (borderline), 10⁹ at n=1000 (always TLE). The failure is deterministic and cannot be optimised away — the algorithm must be replaced with Johnson or Dijkstra×n. Fix: Switch algorithm based on n and graph density.

INF guard omitted → silent overflow

Without if (dist[i][k] >= INF/2) continue, adding two large-but-finite sentinels overflows int64. The wrapped negative value then contaminates the shortest-path matrix, producing wrong answers on all pairs reachable through that sentinel. Fix: Always use the guarded template from §9.

Negative cycle present but not checked

If a negative cycle exists, dist[i][i] goes negative for cycle vertices. Without calling hasNegativeCycle(), the algorithm returns silently with a corrupt matrix — some distances appear finite but are actually −∞. Every pair whose path passes through the cycle is wrong. Fix: Always call hasNegativeCycle() and handle the return value at output time.

Wrong loop order (i or j as outermost)

The state invariant requires that when evaluating row i, column j, the values dist[i][k] and dist[k][j] already reflect all intermediates ⊆ {0…k−1}. This is only true when k is outermost. If i or j is outermost, the values read during the update may themselves be mid-update values for the current k, violating the inductive assumption. Result: systematically wrong distances with no visible error. Fix: Outermost loop must always be k.

32-bit int used instead of 64-bit long long

With 32-bit int and INF = 0x3f3f3f3f (≈10⁹), even a single edge with weight 10⁹ plus the sentinel ≈10⁹ overflows. With 64-bit ll and INF = (ll)4e18, there is headroom for any realistic path length. Fix: Always declare dist as ll (or long long) and set const ll INF = (ll)4e18.

Diagonal not initialised to 0

If dist[i][i] is not set to 0 before running (e.g., left at INF because no self-loop input was given), then paths of the form i → … → i → … → j will incorrectly cost INF + something. Every path that loops through its own source is wrong. Fix: Always set dist[i][i] = 0 in the constructor, before any edges are added.

Everything on this page is calibrated for a 1-second time limit, 256 MB memory, and a modern contest judge running at ~3×10⁸ simple operations per second. Numbers are practical, not theoretical.

Real Constraint Thresholds

The asymptotic O(n³) tells you the shape of the curve but not where the cliff is. These are the empirically calibrated thresholds used by ICPC competitors:

n Iterations Wall-clock estimate Verdict
n ≤ 300 2.7 × 10⁷ ~90 ms ✓ Comfortably safe
n ≤ 400 6.4 × 10⁷ ~215 ms ✓ Safe
n ≤ 500 1.25 × 10⁸ ~420 ms ⚠ Acceptable with fast I/O
n ≤ 600 2.16 × 10⁸ ~720 ms ⚠ Borderline — depends on constant factor
n ≤ 700 3.43 × 10⁸ ~1150 ms ✗ TLE risk — guard with INF skip
n > 700 > 3.43 × 10⁸ > 1150 ms ✗ TLE — switch algorithm

The INF-skip guard (if (dist[i][k] >= INF/2) continue) can cut inner iterations by 40–70% on sparse graphs, pushing the effective n limit up to ~650–700. On dense graphs it saves nothing.

Why Floyd's Constant Factor Is Smaller Than You Expect

The inner triple loop accesses memory in a pattern that modern CPUs handle very efficiently. Understanding why lets you predict performance more accurately than any big-O formula:

Row-major access in the j-loop

The innermost loop iterates over j = 0..n-1, reading dist[k][j] and dist[i][j] sequentially. Both are contiguous in memory (row-major C++ arrays). The CPU prefetcher predicts the access pattern and loads cache lines before they are needed — effectively hiding memory latency. This is the primary reason Floyd runs at ~2–3 operations per clock rather than the ~10–15 you'd expect from the raw memory bandwidth.

dist[i][k] is loop-invariant in the j-loop

For a fixed i and k, the value dist[i][k] does not change across the j-loop. A modern compiler will hoist it into a register, turning it into a zero-cost read. This means the innermost loop body effectively executes as: nd = REG + dist[k][j]; if (nd < dist[i][j]) dist[i][j] = nd; — just two memory reads and one branch per iteration. Johnson's Dijkstra, by contrast, has data-dependent branching through the priority queue that the CPU cannot predict or prefetch.

When cache-friendliness breaks down

For large n (≥700), the dist matrix exceeds L2 cache (typically 256–512 KB for a row at n=700: 700×8 = 5.6 KB per row, 700 rows = 3.9 MB total matrix). At this point the outer-k loop accesses dist[i][k] from a different row than dist[i][j], causing L2/L3 misses for the i-loop. This is why performance degrades super-linearly around n=600–700 on most judges — it is not just the operation count but the memory subsystem becoming the bottleneck.

Template-Level Optimizations for Tight TL

These are the concrete changes to apply when n is borderline (500–650) and you need every millisecond:

// ── OPTIMIZATION 1: hoist dist[i][k] out of the j-loop ──
for (int k = 0; k < n; ++k) {
  for (int i = 0; i < n; ++i) {
    const ll dik = dist[i][k];           // hoisted — one read, reused n times
    if (dik >= INF / 2) continue;       // INF-skip: skip entire i-row if unreachable
    for (int j = 0; j < n; ++j) {
      if (dist[k][j] < INF / 2) {
        const ll nd = dik + dist[k][j];
        if (nd < dist[i][j]) dist[i][j] = nd;
      }
    }
  }
}

// ── OPTIMIZATION 2: use a raw 2D array, not vector<vector<>> ──
// vector<vector<ll>> adds one pointer dereference per row access.
// A flat array avoids this. For n ≤ 700:
static ll dist[705][705];               // ~3.9 MB — fits in BSS, not stack

// ── OPTIMIZATION 3: for transitive closure, use bitset ──
// Each row is one bitset. OR-ing two bitsets processes 64 bits at once.
// Time drops from O(n³) to O(n³/64) — at n=500 that's ~2×10⁶ 64-bit ops.
static bitset<705> reach[705];
for (int k = 0; k < n; ++k)
  for (int i = 0; i < n; ++i)
    if (reach[i][k]) reach[i] |= reach[k]; // one bitset OR = 64 bools at once

// ── OPTIMIZATION 4: early-exit on negative cycle detection ──
// If you only need to DETECT a negative cycle (not compute all paths),
// stop the k-loop as soon as any diagonal goes negative:
bool neg_cycle = false;
for (int k = 0; k < n && !neg_cycle; ++k) {
  for (int i = 0; i < n; ++i) {
    const ll dik = dist[i][k];
    if (dik >= INF / 2) continue;
    for (int j = 0; j < n; ++j)
      if (dist[k][j] < INF / 2)
        dist[i][j] = min(dist[i][j], dik + dist[k][j]);
  }
  if (dist[k][k] < 0) neg_cycle = true;   // early exit
}

// ── OPTIMIZATION 5: fast I/O boilerplate ──
ios::sync_with_stdio(false); cin.tie(nullptr);
// For large output (full n×n matrix), prefer printf or putchar_unlocked
// over cout — on n=500 that's 250,000 numbers, and cout is ~3× slower.

Common Contest Mistakes — Floyd

Using int instead of long long — silent overflow

With int and INF = 0x3f3f3f3f ≈ 10⁹, an edge weight of 10⁶ × path length of 1000 already overflows. The failure is silent — the wrapped negative number looks like a very short path and corrupts subsequent relaxations. Rule: always declare dist as ll, even when edge weights are small, because Floyd sums them.

Omitting the INF/2 guard — same silent overflow

dist[i][k] + dist[k][j] when both are (ll)4e18 overflows 64-bit signed. The result is negative and "beats" the current dist[i][j], producing a wrong entry that then propagates to all pairs. The guard if (dik >= INF/2) continue costs one comparison per row and eliminates the overflow entirely.

Not checking for negative cycles before reporting distances

When a negative cycle exists, dist[i][i] < 0 for cycle vertices, and any reported distance through them is semantically −∞ but numerically finite. Reading these values without checking hasNegativeCycle() first gives plausible-looking wrong answers. On problems that guarantee no negative cycle, this is safe to skip — but on any problem with arbitrary weights, always check.

Using vector<vector<ll>> for borderline n

Each row access in a vector<vector<>> follows one extra pointer dereference through the heap. For n=500 that is 500 × 500 × 500 = 1.25 × 10⁸ extra pointer dereferences. A static ll dist[605][605] flat array eliminates this and typically saves 15–25% wall time. The tradeoff: stack allocation fails for large n — declare it static or global.

Applying Floyd when graph compression is possible

Some contest problems give n up to 10⁵ vertices but only ask about shortest paths among a small set of k ≤ 300 "important" vertices. Run BFS/Dijkstra from each important vertex to build a k×k distance sub-matrix, then optionally run Floyd on just those k vertices. This compresses an impossible O(n³) into a feasible O(km + k³).

A Standard All-Pairs Shortest Paths

① Problem Story: You have n cities connected by roads with integer weights (possibly negative). Answer Q queries: "what is the shortest travel time from city u to city v?"

② Abstraction: Compute dist[i][j] = minimum-weight path from i to j for all pairs (i, j), where path weight = sum of edge weights along the path.

③ Why Floyd fits: Floyd–Warshall is the canonical APSP algorithm for dense or moderate graphs (n ≤ 500). Its DP formulation handles positive and negative edge weights identically, requires zero graph transformation, and produces the full n×n distance matrix in a single O(n³) pass. The sum-over-edges cost algebra maps perfectly onto the recurrence — no algebraic substitution is needed. When n is small enough that O(n³) is feasible and you want the simplest correct implementation, Floyd is the unconditional default choice.

④ What changes from the base algorithm: Nothing — this is the base case itself. The standard Floyd template is the complete answer to this problem class.

⑤ What stays invariant: The DP state dist[i][j], the transition recurrence dist[i][j] = min(dist[i][j], dist[i][k]+dist[k][j]), the mandatory loop order (outermost k → then i → then j), the INF initialization for all non-edges, and the diagonal initialization dist[i][i] = 0 — all unchanged from the master template.

⑥ Correctness reasoning: By the inductive proof in §6: after finishing iteration k, dist[i][j] holds the shortest path using only intermediates ⊆ {0,…,k}. After all n iterations, every possible intermediate is allowed, so dist[i][j] is the true global shortest path. The INF/2 overflow guard is required to prevent dist[i][k] + dist[k][j] from wrapping around when either sub-path is unreachable.

#include <bits/stdc++.h>
using namespace std;
using ll = long long;
const ll INF = (ll)4e18;

struct FloydWarshall {
    int n;
    vector<vector<ll>> dist;
    vector<vector<int>> nxt;

    FloydWarshall(int n) : n(n), dist(n, vector<ll>(n, INF)), nxt(n, vector<int>(n, -1)) {
        for (int i = 0; i < n; ++i) { dist[i][i] = 0; nxt[i][i] = i; }
    }

    void addEdge(int u, int v, ll w, bool directed = true) {
        if (w < dist[u][v]) { dist[u][v] = w; nxt[u][v] = v; }
        if (!directed && w < dist[v][u]) { dist[v][u] = w; nxt[v][u] = u; }
    }

    void run() {
        for (int k = 0; k < n; ++k)
            for (int i = 0; i < n; ++i) {
                if (dist[i][k] >= INF / 2) continue;
                for (int j = 0; j < n; ++j) {
                    if (dist[k][j] >= INF / 2) continue;
                    ll nd = dist[i][k] + dist[k][j];
                    if (nd < dist[i][j]) { dist[i][j] = nd; nxt[i][j] = nxt[i][k]; }
                }
            }
    }

    bool hasNegativeCycle() const {
        for (int i = 0; i < n; ++i) if (dist[i][i] < 0) return true;
        return false;
    }

    vector<int> getPath(int u, int v) const {
        if (nxt[u][v] == -1) return {};
        vector<int> path = {u};
        while (u != v) { u = nxt[u][v]; path.push_back(u); }
        return path;
    }
};

B Path Reconstruction

① Problem Story: Not just the distance — you need to actually output the sequence of vertices on the shortest u → v path.

② Abstraction: Maintain a nxt[i][j] matrix alongside dist[i][j]. nxt[i][j] = the next vertex after i on the shortest i → j path. Reconstruct by following the nxt pointers until reaching the destination.

③ Why Floyd fits: Path reconstruction in APSP requires remembering, for every ordered pair (i, j), the first hop of the optimal route. Floyd's matrix structure is naturally suited for this: a second n×n matrix nxt[i][j] can be maintained in lockstep with dist[i][j] at zero asymptotic overhead. The update rule nxt[i][j] = nxt[i][k] emerges directly from the DP transition — when a better path i→k→j is found, the first step toward j is identical to the first step toward k. No other APSP algorithm supports global next-hop tables as cleanly.

④ What changes from the base algorithm: You add a second n×n integer matrix nxt. During edge insertion, initialize nxt[u][v] = v whenever an edge improves dist[u][v]. During the relaxation step, when a shorter path is found, also update nxt[i][j] = nxt[i][k]. Add a getPath(u, v) function that follows pointers.

⑤ What stays invariant: The full Floyd triple-nested loop, all three levels, INF initialization, diagonal initialization, and the main recurrence dist[i][j] = min(dist[i][j], dist[i][k]+dist[k][j]) — all identical to Variant A. The nxt update is a side-effect appended to the same conditional branch.

⑥ Correctness reasoning: When we find a better path i → k → j, the first step from i toward j is the same as the first step from i toward k. So nxt[i][j] = nxt[i][k] correctly records the next hop toward j via intermediate k. Following nxt pointers from u traces out the shortest path vertex by vertex until reaching v. If nxt[u][v] == -1, the pair is unreachable.

C Transitive Closure / Reachability

① Problem Story: You have a directed graph and want to know: for every pair (i, j), can we reach j from i at all? No edge weights — only reachability matters.

② Abstraction: Replace the distance matrix with a boolean reachability matrix reach[i][j]. Replace the min-sum recurrence with an OR-AND recurrence. After running, reach[i][j] = true means j is reachable from i.

③ Why Floyd fits: Floyd's DP argument is purely about intermediate vertex expansion — it says nothing about what "path cost" means. Reachability is a degenerate form of shortest-path where the cost is boolean (reachable / not). The same three-loop structure works because we still ask: "can we route from i to j via k?" The OR-AND operators are the boolean analogue of min-sum. Johnson's approach (graph transformation + Dijkstra) provides no benefit for pure reachability — there are no edge weights to reweight. Floyd is the natural, clean choice.

④ What changes from the base algorithm: Distance arithmetic → boolean logic. dist[i][j] (ll) → reach[i][j] (bool). The recurrence changes from min(a, b+c) to a || (b && c). No INF, no overflow concerns. Initialization uses true/false instead of 0/INF.

⑤ What stays invariant: The triple nested loop structure, the mandatory loop order k → i → j, the interpretation of "allowing k as an intermediate vertex," and the diagonal initialization (reach[i][i] = true for all i, representing zero-length reachability to self). The inductive correctness argument is structurally identical.

⑥ Correctness reasoning: The Floyd DP argument transfers directly. After processing k, reach[i][j] = true iff a path from i to j exists using intermediates ⊆ {0…k}. The boolean recurrence correctly captures the two cases: j is reachable from i without k (keep old value via OR), or j is reachable via k (both reach[i][k] and reach[k][j] must be true, connected via AND). The OR replaces min; the AND replaces +. The induction goes through unchanged.

#include <bits/stdc++.h>
using namespace std;

struct TransitiveClosure {
    int n;
    vector<vector<bool>> reach;

    TransitiveClosure(int n) : n(n), reach(n, vector<bool>(n, false)) {
        for (int i = 0; i < n; ++i) reach[i][i] = true;  // self-reachability
    }

    void addEdge(int u, int v, bool directed = true) {
        reach[u][v] = true;
        if (!directed) reach[v][u] = true;
    }

    void run() {
        for (int k = 0; k < n; ++k)
            for (int i = 0; i < n; ++i) {
                if (!reach[i][k]) continue;  // pruning: skip if i can't reach k
                for (int j = 0; j < n; ++j)
                    reach[i][j] = reach[i][j] || reach[k][j]; // OR-AND recurrence
            }
    }

    bool canReach(int u, int v) const { return reach[u][v]; }
};

int main() {
    ios::sync_with_stdio(false); cin.tie(nullptr);
    int n, m; cin >> n >> m;
    TransitiveClosure tc(n);
    while (m--) {
        int u, v; cin >> u >> v; --u; --v;
        tc.addEdge(u, v);
    }
    tc.run();
    int q; cin >> q;
    while (q--) {
        int u, v; cin >> u >> v; --u; --v;
        cout << (tc.canReach(u, v) ? "YES" : "NO") << "\n";
    }
}

// when reach[i][k] is true — reducing time to O(n³/64).

⑨ When to use / detection pattern: Any "is city j reachable from city i?" style problem on a directed graph. Also useful as a preprocessing step in SCC analysis, DAG reasoning, and tournament ranking problems. Detection keywords: "can X reach Y," "is there a path from," "reachability," "connectivity all pairs."

Common pitfall: Forgetting reach[i][i] = true. This matters because intermediate vertex reasoning requires that every vertex can "reach itself" — without it, paths that pass through a vertex and loop back will be incorrectly computed. On undirected graphs with a negative edge, note that the boolean version still works but a negative undirected edge would create a false positive; use the directed-only version.

D Minimax Path (Bottleneck Path)

① Problem Story: A frog wants to jump across a pond using stepping stones. The "cost" of a path is the length of the largest single jump. The frog wants to minimize this maximum jump. Given all pairwise stone positions, find the optimal route between every pair.

② Abstraction: Path cost = maximum edge weight on the path. Goal: minimize that maximum over all paths. best[i][j] = the minimum possible "worst single edge" on any i → j path.

③ Why Floyd fits: The minimax problem decomposes at every intermediate vertex in exactly the same way as shortest paths. For any path i → … → k → … → j, the worst edge on the full path is the max of the worst edge on the left half and the worst edge on the right half. This is precisely the Floyd DP decomposition — split at k, combine the two sub-problems. Floyd's triple loop iterates over all possible intermediate vertices k, which is exactly what we need to find the globally optimal split point. No single-source algorithm (Dijkstra, Bellman-Ford) can express this all-pairs minimax recurrence. Only Floyd's matrix DP structure works natively.

④ What changes from the base algorithm: The recurrence. Instead of sum → take the max of the two sub-path segments. Instead of outer min → still min. Result: best[i][j] = min(best[i][j], max(best[i][k], best[k][j])). Initialization: best[i][j] = edge weight for direct edges, INF for unreachable pairs, 0 for the diagonal.

⑤ What stays invariant: The triple nested loop, the mandatory k → i → j loop order, the interpretation of "allowing k as an intermediate," and the two-case DP decomposition (path uses k vs. path does not use k). The outer operator is still min — only the inner combination operator changes from addition to max.

⑥ Correctness reasoning: The same Floyd DP inductive argument applies. After processing k, best[i][j] = the best (minimum max-edge) path using intermediates ⊆ {0…k}. Splitting at k: the max-edge of the concatenated path i→k→j is the max of the two sub-path max-edges. Taking the outer min over all choices of k (the Floyd loop) is correct by the same induction. The base case holds because a single direct edge is its own worst edge.

    }

⑨ When to use / detection pattern: Whenever path cost = maximum single edge (bottleneck). Classic examples: frog jump problems, road width problems (narrowest road on the route), satellite communication relay strength. Detection keywords: "minimize the maximum," "bottleneck path," "minimize the largest edge," "optimal worst-case single hop."

Common pitfall: Initializing best[i][j] = 0 instead of INF for unreachable pairs. This makes every path appear to have cost 0 (since max(0, anything) = anything but min(anything, 0) = 0). Use INF for unreachable pairs and 0 only for best[i][i].

E Widest Path / Maximin (Maximum Minimum)

① Problem Story: Each edge has a "bandwidth" or "capacity." The effective capacity of a path is determined by its weakest link — the minimum-capacity edge. You want to route data from every source to every destination via the highest-capacity path.

② Abstraction: Path value = minimum edge weight on the path. Goal: maximize that minimum. best[i][j] = the maximum possible "weakest link" on any i → j path.

③ Why Floyd fits: The maximin problem has the same DP decomposition structure as minimax, just with dual operators. For any path i → … → k → … → j, the capacity of the full path is the minimum of the left half's capacity and the right half's capacity — the weakest link is the minimum of two sub-path minima. Maximizing this over all splits is exactly what Floyd's outer loop over k accomplishes. As with minimax, the all-pairs nature of the problem and the split-at-intermediate-vertex decomposition make Floyd the natural and only clean choice.

④ What changes from the base algorithm: The recurrence is the dual of minimax: instead of max-then-min, use min-then-max: best[i][j] = max(best[i][j], min(best[i][k], best[k][j])). Initialization: best[i][j] = edge capacity, best[i][j] = 0 for unreachable pairs (not INF — 0 capacity means no connection), best[i][i] = ∞ or a large number (infinite bandwidth to self).

⑤ What stays invariant: The triple nested loop, the k → i → j loop order, and the DP split argument — all identical. The outer operator changes from min to max, and the inner combination changes from max to min. The correctness argument follows the same inductive structure.

⑥ Correctness reasoning: Same Floyd DP argument, dually applied. After processing k, best[i][j] = the maximum possible min-edge path using intermediates ⊆ {0…k}. Splitting at k gives: the min of the left half and the right half's capacities. The outer max takes the best over all k. By induction, this is correct. Base case: a single direct edge has capacity equal to its own weight.

F Negative-Cycle Detection

① Problem Story: You suspect the graph might have a negative cycle. If it does, certain shortest paths are undefined (−∞). You need to detect this before reporting any distances.

② Abstraction: Run standard Floyd to completion. Then check the diagonal: if dist[i][i] < 0 for any i, vertex i is on a negative cycle.

③ Why Floyd fits: Floyd naturally visits every vertex as a potential intermediate, which means it naturally computes dist[i][i] — the length of the shortest path from i back to i via any route. A negative value precisely indicates a negative cycle through i. This detection is entirely free — it costs zero additional computation beyond the standard Floyd run. Bellman-Ford (in the Johnson approach) also detects negative cycles, but Floyd detects them after computing the full distance matrix, allowing you to also identify all affected pairs. For dense graphs or small n, Floyd is unambiguously the simpler and more informative choice.

④ What changes from the base algorithm: You add a single post-run check on the diagonal: one O(n) loop after run() that checks dist[i][i] < 0 for each i. No change to the Floyd algorithm itself.

⑤ What stays invariant: The entire Floyd algorithm — the triple loop, the recurrence, the INF guards, the diagonal initialization — runs completely unmodified. The negative cycle check is a read-only post-processing step.

⑥ Correctness reasoning: If vertex i is on a negative cycle, then there exists a path from i back to i (through the cycle) with negative total weight. Floyd will correctly compute this: after considering all possible intermediate vertices, dist[i][i] will equal the minimum-weight cycle through i, which is negative. The converse also holds: if no negative cycle exists, dist[i][i] remains 0 for all i (since the shortest path from i to i with no negative cycle is the empty path with cost 0).

using ll = long long;
const ll INF = (ll)4e18;

struct FloydWarshall {
    int n;
    vector<vector<ll>> dist;
    vector<vector<int>> nxt;
    FloydWarshall(int n):n(n),dist(n,vector<ll>(n,INF)),nxt(n,vector<int>(n,-1)){
        for(int i=0;i<n;++i){dist[i][i]=0;nxt[i][i]=i;}
    }
    void addEdge(int u,int v,ll w,bool dir=true){
        if(w<dist[u][v]){dist[u][v]=w;nxt[u][v]=v;}
        if(!dir&&w<dist[v][u]){dist[v][u]=w;nxt[v][u]=u;}
    }
    void run(){
        for(int k=0;k<n;++k)
          for(int i=0;i<n;++i){
            if(dist[i][k]>=INF/2)continue;
            for(int j=0;j<n;++j){
              if(dist[k][j]>=INF/2)continue;
              ll nd=dist[i][k]+dist[k][j];
              if(nd<dist[i][j]){dist[i][j]=nd;nxt[i][j]=nxt[i][k];}
            }
          }
    }
    // ← KEY ADDITION: check diagonal after run()
    bool hasNegativeCycle() const {
        for(int i=0;i<n;++i) if(dist[i][i]<0) return true;
        return false;
    }
    vector<int> getPath(int u,int v) const {
        if(nxt[u][v]==-1)return{};
        vector<int> p={u};
        while(u!=v){u=nxt[u][v];p.push_back(u);}
        return p;
    }
};

int main() {
    ios::sync_with_stdio(false); cin.tie(nullptr);
    int n, m; cin >> n >> m;
    FloydWarshall fw(n);
    while(m--){int u,v;ll w;cin>>u>>v>>w;--u;--v;fw.addEdge(u,v,w);}
    fw.run();
    if (fw.hasNegativeCycle()) {
        cout << "NEGATIVE CYCLE EXISTS\n";
        return 0;
    }
    // Safe to query: no negative cycle. dist[i][j] is well-defined.
    int q; cin >> q;
    while(q--){
        int u,v;cin>>u>>v;--u;--v;
        if(fw.dist[u][v]>=INF/2)cout<<"UNREACHABLE\n";
        else cout<<fw.dist[u][v]<<"\n";
    }
}

⑧ Complexity: Time O(n³) for Floyd plus O(n) for the diagonal check — the check is dominated by the main run. Space O(n²). The check must happen after run(); checking the diagonal before running Floyd is meaningless because the diagonal is initialized to 0 and only Floyd can update it to a negative value.

⑨ When to use / detection pattern: Any problem where the graph might have negative cycles and you must report "no solution," "IMPOSSIBLE," or identify the affected vertices. Detection keywords: "negative cycle," "arbitrage," "detect if a path can be made arbitrarily negative," "is the graph well-defined for shortest paths."

Common pitfall: Checking the diagonal before running Floyd. The DP must complete first; the diagonal is initially 0 and only becomes negative if Floyd finds a cheaper path from i to i through negative edges. The check must always follow run().

G Negative-Cycle Influence Propagation

① Problem Story: A graph has some negative cycles. Some pairs (i, j) have a well-defined finite shortest path. Other pairs have dist = −∞ because the path from i to j can pass through a negative cycle. You must distinguish finite answers from −∞ answers.

② Abstraction: After running standard Floyd, identify every negative-cycle vertex (dist[k][k] < 0). Then mark all pairs (i, j) where the i→j path can pass through such a vertex as having "−∞ distance." A pair is −∞ iff there exists a vertex k with a negative cycle where i can reach k and k can reach j.

③ Why Floyd fits: Floyd is the only APSP algorithm that both (a) runs on graphs with negative cycles and (b) produces a fully populated distance matrix from which the −∞ propagation can be read. Johnson stops entirely when a negative cycle is detected — it cannot produce partial APSP results. The Floyd approach gives you dist[i][k] and dist[k][j] for every k, making it possible to check the negative-cycle influence condition in a second O(n³) pass. This two-pass Floyd approach is unique to the DP formulation.

④ What changes from the base algorithm: You add a second pass after run(). This second pass iterates over every vertex k with a negative cycle (dist[k][k] < 0) and marks all pairs (i, j) where i can reach k and k can reach j as having dist = −∞. The marking is stored in a separate boolean matrix neg[i][j].

⑤ What stays invariant: Standard Floyd runs completely and unmodified first. The dist matrix is populated normally. The second pass only reads from the dist matrix — it does not modify it. The condition for marking (i,j) as −∞ uses the already-computed dist[i][k] and dist[k][j] values.

⑥ Correctness reasoning: If dist[k][k] < 0 and dist[i][k] < INF/2 and dist[k][j] < INF/2, then the path i → k → (negative cycle, traversed arbitrarily many times) → k → j has no lower bound — it is −∞. The two-pass approach captures exactly these pairs: first Floyd correctly computes dist[i][k] and dist[k][j]; then the second pass correctly identifies which pairs (i,j) can funnel through a negative-cycle vertex k.

11 · Why Brute Force Fails on Sparse Graphs

The motivating question — why not just run Dijkstra from every source? the key insight

Suppose the graph is sparse (m ≪ n²). Floyd does O(n³) regardless of m — deeply wasteful when m ≈ n log n.

Run Bellman-Ford from every source
Correct. But O(nm) per source → O(n²m) total. For n=10000, m=50000: 5×10¹³ operations. Way too slow.
Run Dijkstra from every source
O(m log n) per source → O(nm log n) total. Fast! But Dijkstra is invalid with negative edges. Produces wrong answers silently.
Johnson's insight
Transform the graph so all edges become non-negative, without changing which path is shortest. Then Dijkstra becomes valid. Run it from every source.

12 · The Core Idea — Edge Reweighting

Assign each vertex a potential h[v]. Redefine every edge weight as:

w′(u,v) = w(u,v) + h[u] − h[v]

Two goals must be simultaneously satisfied:

Goal 1 — All w′(u,v) ≥ 0
So Dijkstra can be applied without producing incorrect answers.
Goal 2 — Shortest path ordering preserved
The path minimizing original cost w also minimizes reweighted cost w′, between the same endpoints.
Recovery formula — after Dijkstra gives d′(s,t):

d(s,t) = d′(s,t) − h[s] + h[t]

In code, after Dijkstra from src: dst[i] += pot[i] - pot[src]

13 · Why Reweighting Preserves Shortest Paths

The telescoping argument — the mathematical heart of Johnson expand

Take any path P = v₀ → v₁ → v₂ → … → vₖ. Its reweighted total cost is:

w′(P) = Σ (w(vᵢ,vᵢ₊₁) + h[vᵢ] − h[vᵢ₊₁])
      = Σ w(vᵢ,vᵢ₊₁)  +  h[v₀]  −  h[vₖ]
             ↑ everything in the middle cancels (telescopes)

So: w′(P) = w(P) + h[s] − h[t]   for a path from s to t
Key insight: For fixed endpoints s and t, the term h[s] − h[t] is a constant — the same for every path from s to t. Therefore, the path minimizing w(P) also minimizes w′(P). Shortest paths are perfectly preserved. The recovery formula d(s,t) = d′(s,t) − h[s] + h[t] follows directly by rearranging.

14 · Computing Potentials — Theory and CP Trick

Textbook — explicit super-source the theory

Add super-source S. Connect S → v with weight 0 for every vertex v. Run Bellman-Ford from S. Set h[v] = dist(S, v).

Bellman-Ford guarantees: for every edge (u,v): h[v] ≤ h[u] + w(u,v) (triangle inequality).

Rearranging: w(u,v) + h[u] − h[v] ≥ 0. Every reweighted edge is non-negative. ✓

Goal 1 is achieved. Dijkstra is valid. ✓
CP trick — no explicit super-source your code
Simpler CP implementation (what your code does):
• Initialize pot[v] = 0 for all v
• Push all vertices into the SPFA queue initially
• Run SPFA-style relaxation for exactly n rounds

Mathematically equivalent to having S with 0-weight edges to all nodes. Negative cycle: if relaxations still improve after n rounds, a negative cycle exists.

15 · Correctness Proof of Johnson

Full 4-step proof expand

Step 1 Valid potentials. h[v] = dist(S,v). For every edge (u,v): h[v] ≤ h[u]+w(u,v) (triangle inequality). So w′(u,v) = w(u,v)+h[u]−h[v] ≥ 0. All reweighted edges non-negative. ✓

Step 2 Paths preserved. For any path P from s to t: w′(P) = w(P)+h[s]−h[t]. The additive term is constant for fixed endpoints. Same shortest path in both graphs. ✓

Step 3 Dijkstra valid. All w′ ≥ 0 → Dijkstra computes correct shortest paths. ✓

Step 4 Recover distances. d′(s,t) from Dijkstra → d(s,t) = d′(s,t)−h[s]+h[t]. All original APSP distances recovered. ✓

Johnson is correct whenever no negative cycle exists. If one exists, Bellman-Ford detects it during potential computation. □

📐 Formal Invariants — Johnson's Algorithm

Johnson is a 4-phase pipeline. Each phase has its own invariant, and each invariant is a precondition for the next phase. Violating any one of them silently corrupts all subsequent results.

It is the only property needed for the reweighting to be valid. SPFA iterates until no potential decreases further (i.e., until the triangle inequality holds globally). Path-preservation: follows from the telescoping identity (§13) — w′(P) = w(P) + pot[s] − pot[t], and the correction term is constant for fixed endpoints.

17 · Common Pitfalls in Johnson

1
Wrong reweight formula Correct: w′=w+h[u]−h[v]. The minus is on the destination vertex, not the source. Swapping them produces invalid weights.
2
Wrong restore formula Correct: d=d′−h[u]+h[v]. In code: dst[i] += pot[i] - pot[src]. Note the sign: subtract source potential, add destination potential.
3
Dijkstra on original edges Wrong. Dijkstra must run on reweighted edges with +pot[u]−pot[v] added inline during relaxation. Never run it on the original graph.
4
Missing negative-cycle check If Bellman-Ford finds a negative cycle, johnson() must return false immediately. Never proceed to Dijkstra on invalid potentials — it will silently give wrong answers.
5
Johnson on dense graphs For dense graphs (m≈n²), Johnson is O(n³ log n) — worse than Floyd's O(n³). Use Johnson only when the graph is genuinely sparse.
6
Undirected + negative edges Any negative undirected edge creates a negative 2-cycle (u→v→u with total negative weight). Johnson's Bellman-Ford will detect this and reject. Design your Johnson solution for directed graphs.

#include "bits/stdc++.h"
using namespace std;
using ll = long long;

const int N = 2005;
const int M = 200000 + 5;
const ll INF = (ll)4e18;

int n, m;
int head[N], to[M], nxt_e[M], cst[M], ne;

void init() {
    ne = 0;
    memset(head, -1, n * sizeof head[0]);
}

void addEdge(int u, int v, int w) {
    to[ne] = v;
    cst[ne] = w;
    nxt_e[ne] = head[u];
    head[u] = ne++;
}

#define edges(u, v, e, cs) \
    for (int e=head[u], v, cs; ~e && ((v=to[e]), (cs=cst[e]), 1); e=nxt_e[e])

int inq[N], inqid;
ll pot[N];

bool bellman() {
    ++inqid;
    queue<int> q;

    for (int src = 0; src < n; ++src) {
        pot[src] = 0;
        q.push(src);
        inq[src] = inqid;
    }

    int cnt = n;
    while (!q.empty() && cnt--) {
        int s = (int)q.size();
        while (s--) {
            int u = q.front();
            q.pop();
            inq[u] = 0;

            edges(u, v, e, cs) {
                ll dv = pot[u] + cs;
                if (pot[v] > dv) {
                    if (!cnt) return false; // negative cycle!
                    pot[v] = dv;
                    if (inq[v] != inqid) {
                        inq[v] = inqid;
                        q.push(v);
                    }
                }
            }
        }
    }
    return true;
}

void dijkstra(int src, ll *dst, int *par) {
    memset(dst, 0x3f, n * sizeof dst[0]);
    dst[src] = 0;
    par[src] = -1;

    priority_queue<pair<ll, int>> q;
    q.push({0, src});

    while (!q.empty()) {
        auto [d, u] = q.top(); q.pop();
        d = -d;
        if (dst[u] < d) continue;

        edges(u, v, e, cs) {
            ll nd = dst[u] + (ll)cs + pot[u] - pot[v]; // reweighted edge!
            if (dst[v] > nd) {
                dst[v] = nd;
                par[v] = e;
                q.push({-nd, v});
            }
        }
    }

    ll inf; memset(&inf, 0x3f, sizeof inf);
    for (int i = 0; i < n; ++i)
        if (dst[i] < inf)
            dst[i] += pot[i] - pot[src]; // recover original distance
}

ll dist[N][N];
int par[N][N];

bool johnson() {
    if (!bellman()) return false; // negative cycle detected
    for (int s = 0; s < n; ++s)
        dijkstra(s, dist[s], par[s]);
    return true;
}

int main() {
    ios::sync_with_stdio(false); cin.tie(nullptr);
    cin >> n >> m; init();
    while (m--) {
        int u, v, w; cin >> u >> v >> w; --u; --v;
        addEdge(u, v, w);
    }
    if (!johnson()) { cout << "Negative cycle exists\n"; return 0; }
    for (int i = 0; i < n; ++i) {
        for (int j = 0; j < n; ++j) {
            if (dist[i][j] > (ll)1e15) cout << "#";
            else cout << dist[i][j];
            cout << " \n"[j+1==n];
        }
    }
}
🚫 Johnson's Algorithm: This Algorithm Fails When…

Each failure below is a distinct failure mode with a different root cause and a different fix. Unlike Floyd, most Johnson failures are silent — the algorithm terminates with a plausible-looking but wrong answer.

Bellman-Ford returns false and Dijkstra runs anyway

If bellman() detects a negative cycle but the caller ignores the return value and calls Dijkstra, the potentials are invalid (they do not satisfy the triangle inequality). Dijkstra runs but produces wrong shortest paths — all distances in the region of the cycle are corrupted. The failure is silent: no crash, no exception, wrong numbers. Fix: if (!bellman()) return false; is mandatory in johnson(), and the caller must handle false before reading any distances.

Reweighting formula applied with wrong sign

The correct formula is w′(u,v) = w(u,v) + pot[u] − pot[v] (subtract the destination potential). Swapping to + pot[v] − pot[u] produces some negative reweighted edges, making Dijkstra invalid. Distances look plausible on small examples but fail on longer paths. Fix: Always double-check the sign: the node you are leaving (u) is added, the node you are arriving at (v) is subtracted.

Recovery formula omitted or wrong sign

Dijkstra produces distances in the reweighted graph. Without recovery dst[i] += pot[i] − pot[src], all reported distances are in the reweighted metric — systematically wrong for any pair where potentials differ. The failure is invisible unless you verify against a known answer. Fix: Recovery is mandatory. Correct sign: add destination potential, subtract source potential.

Using dist[i][j] == INF to detect unreachable pairs

After the recovery step, some entries that were initialised to the 0x3f memset sentinel (≈4.56×10¹⁸) may have been arithmetically shifted to a different large value. Comparing with the named INF = (ll)4e18 constant fails to match them. The threshold check > (ll)1e15 catches both the original sentinel and any shifted variant. Fix: Use threshold comparison everywhere — never equality to a sentinel constant.

Dense graph (m ≈ n²) — TLE from Dijkstra overhead

On dense graphs, Johnson's O(nm log n) degrades to O(n³ log n) — worse than Floyd's O(n³) by a log factor. The priority-queue operations add a constant factor of 5–10× on top of that. Fix: If m > n²/log n, switch to Floyd. The decision crossover is explicit in §19-D.

Bellman-Ford round count too small (cnt < n)

The SPFA version uses a level-by-level BFS structure with cnt = n rounds. If the queue is not properly layered (e.g., mixing levels), the n-round bound may terminate too early, leaving some potentials sub-optimal and some edges with negative reweighted weight. Dijkstra then produces wrong answers. Fix: Use the exact layered-SPFA structure in §18, where int s = (int)q.size() captures the current layer before processing it.

Johnson's strength is on sparse graphs with negative edges. On everything else it is slower than the alternatives. These benchmarks quantify exactly where it wins and where it loses.

Real Constraint Thresholds

Johnson's cost is O(nm log n) — it depends on both n and m. This table fixes the ratio m/n to compare different graph families:

Key insight: Even when Johnson "wins" in ops, its constant factor is 5–10× larger than Floyd's because of the priority-queue overhead. A good rule is: Johnson needs a 5–10× op-count advantage just to break even on wall time. The crossover point is therefore roughly m < n²/(5 log n), not n²/log n.

Why Johnson's Constant Factor Is Larger

Johnson is built around Dijkstra with a binary heap (priority_queue). Every step of Dijkstra involves memory access patterns that CPUs handle less efficiently than Floyd's sequential scan:

Priority queue — unpredictable memory

Each heap push/pop accesses a node whose memory address depends on the priority values — a data-dependent access the CPU prefetcher cannot predict. On a graph with m edges, this causes O(m) cache misses. At m = 50 000 edges and 64-byte cache lines, that is ~3 MB of random memory fetches per Dijkstra call, multiplied by n source calls.

Lazy deletion bloat

The template uses lazy deletion — when a shorter path is found to vertex v, we push a new entry rather than updating the existing one. This can grow the heap to O(m) entries instead of O(n). The if (dst[u] < d) continue guard skips stale entries but they still consume heap-pop time. On dense subgraphs, the heap may hold 5–10× more entries than vertices.

Adjacency list — sequential edge scan

The forward-star structure (head/to/nxt_e arrays) stores all edges contiguously. Iterating all edges of vertex u via the edges() macro walks a linked list of indices into contiguous arrays — cache-friendly for sparse graphs where each vertex has few edges. This is significantly faster than chasing pointers through vector<pair<>> adjacency lists.

Avoid rebuilding graph between calls

The reweighting in Johnson modifies edge weights: nd = dst[u] + (ll)cs + pot[u] - pot[v]. The formula applies the reweighting inline during Dijkstra, reading the original cs value and the precomputed pot[] array. No separate reweighted copy of the edge list is needed. Never rebuild the graph between source calls — pot[] is computed once and reused for all n Dijkstras.

Common Contest Mistakes — Johnson

Incorrect reweighting formula — wrong sign on destination potential

The formula is nd = dst[u] + (ll)cs + pot[u] - pot[v]. The source potential is added, the destination potential is subtracted. Reversing these produces some negative reweighted edges, making Dijkstra's greedy invariant invalid. The failure is invisible on small examples where potentials happen to be equal, but shows up on any path with non-uniform potential values.

Ignoring the Bellman-Ford return value

If bellman() returns false (negative cycle detected) and the code continues to call Dijkstra anyway, the potentials are invalid and all distances are wrong. The failure is silent. On a contest this produces a WA where the checker rejects all output. Always write: if (!johnson()) { /* handle neg-cycle */ return 0; } — never let the false case fall through.

Using equality (== INF) to detect unreachable pairs

After the recovery step dst[i] += pot[i] - pot[src], unreachable entries that were at the 0x3f memset sentinel (≈4.56×10¹⁸) may have been shifted to a different large value. Comparing to the named constant INF = 4e18 misses these. Use threshold: if (dist[i][j] > (ll)1e15). This reliably catches all unreachable values regardless of potential shifts.

Rebuilding the graph / reinitialising pot[] between Dijkstra calls

Some implementations mistakenly reinitialise pot[] or re-run Bellman-Ford before each Dijkstra call. This wastes O(nm) time per source and converts the O(nm + nm log n) algorithm into O(n²m). The potentials are valid for all n Dijkstras — compute them once and leave them unchanged. Only dst[] and par[] are reinitialised per Dijkstra call.

Choosing Johnson on a dense graph — guaranteed TLE

On a dense graph (m ≈ n²), Johnson's O(nm log n) = O(n³ log n) — a log factor worse than Floyd. Combined with Johnson's larger constant factor (5–10×), the practical gap is 10–20× worse than Floyd at the same n. If the problem gives m ≈ n² and negative edges, Floyd is still the correct choice — Johnson would TLE.

A Sparse APSP with Negative Edges — No Negative Cycle

① Problem Story: You have a directed graph with n = 5000 vertices and m = 20000 edges. Some edges have negative weights. You need the full APSP matrix. Floyd is O(n³) = 1.25 × 10¹¹ — way too slow. What do you do?

② Abstraction: This is the canonical Johnson use case. Run Bellman-Ford once to compute vertex potentials pot[v] (called h[v] in the §12–§15 mathematical derivation; the same value, just the name used in code), reweight all edges so they become non-negative, then run Dijkstra from each source and recover original distances via the inverse reweighting formula.

③ Why Johnson fits: Floyd does O(n³) regardless of m — on sparse graphs this is catastrophically wasteful. Dijkstra alone is invalid because negative edges exist. Johnson resolves this precisely: the Bellman-Ford phase costs O(nm), computes valid potentials that neutralize all negative edges, and enables Dijkstra. Each of n Dijkstras costs O(m log n). Total: O(nm log n) — which, for m ≪ n², is dramatically faster than O(n³). This is exactly the problem Johnson was designed to solve.

④ What changes from the base algorithm: Nothing — this is the base Johnson case itself. The §18 template is the complete answer.

⑤ What stays invariant: The full Johnson 4-phase pipeline as given in §18: bellman() → n × dijkstra() → recovery formula dst[i] += pot[i] - pot[src]. The reweighting formula w′(u,v) = w(u,v) + h[u] − h[v] and the recovery formula d(s,t) = d′(s,t) − h[s] + h[t] are both mandatory and unchanged.

⑥ Correctness reasoning: By the 4-step proof in §15. Bellman-Ford computes pot[v] satisfying pot[v] ≤ pot[u] + w(u,v) for every edge, making all reweighted edges non-negative. By the telescoping argument (§13), the path minimizing reweighted cost equals the path minimizing original cost. Dijkstra gives correct reweighted distances. The recovery formula inverts the constant h[s] − h[t] shift per (s,t) pair, yielding correct original distances.

#include "bits/stdc++.h"
using namespace std;
using ll = long long;

const int N = 2005;
const int M = 200005;
const ll INF = (ll)4e18;

int n, m;
int head[N], to[M], nxt_e[M], cst[M], ne;

void init() { ne = 0; memset(head, -1, n * sizeof head[0]); }

void addEdge(int u, int v, int w) {
    to[ne]=v; cst[ne]=w; nxt_e[ne]=head[u]; head[u]=ne++;
}

#define edges(u,v,e,cs) \
    for(int e=head[u],v,cs;~e&&((v=to[e]),(cs=cst[e]),1);e=nxt_e[e])

int inq[N], inqid;
ll pot[N];

bool bellman() {
    ++inqid;
    queue<int> q;
    for(int s=0;s<n;++s){pot[s]=0;q.push(s);inq[s]=inqid;}
    int cnt=n;
    while(!q.empty()&&cnt--){
        int s=(int)q.size();
        while(s--){
            int u=q.front();q.pop();inq[u]=0;
            edges(u,v,e,cs){
                ll dv=pot[u]+cs;
                if(pot[v]>dv){
                    if(!cnt)return false;
                    pot[v]=dv;
                    if(inq[v]!=inqid){inq[v]=inqid;q.push(v);}
                }
            }
        }
    }
    return true;
}

void dijkstra(int src, ll *dst, int *par) {
    memset(dst, 0x3f, n*sizeof dst[0]);
    dst[src]=0; par[src]=-1;
    priority_queue<pair<ll,int>> q;
    q.push({0,src});
    while(!q.empty()){
        auto[d,u]=q.top();q.pop();d=-d;
        if(dst[u]<d)continue;
        edges(u,v,e,cs){
            ll nd=dst[u]+(ll)cs+pot[u]-pot[v];
            if(dst[v]>nd){dst[v]=nd;par[v]=e;q.push({-nd,v});}
        }
    }
    ll inf;memset(&inf,0x3f,sizeof inf);
    for(int i=0;i<n;++i)if(dst[i]<inf)dst[i]+=pot[i]-pot[src];
}

ll dist[N][N]; int par[N][N];

bool johnson(){
    if(!bellman())return false;
    for(int s=0;s<n;++s)dijkstra(s,dist[s],par[s]);
    return true;
}

int main(){
    ios::sync_with_stdio(false);cin.tie(nullptr);
    cin>>n>>m;init();
    while(m--){int u,v,w;cin>>u>>v>>w;--u;--v;addEdge(u,v,w);}
    if(!johnson()){cout<<"Negative cycle exists\n";return 0;}
    for(int i=0;i<n;++i)
        for(int j=0;j<n;++j){
            if(dist[i][j]>(ll)1e15)cout<<"#";
            else cout<<dist[i][j];
            cout<<" \n"[j+1==n];
        }
}

Space O(n² + m): O(n²) for the full dist and par matrices, O(m) for the adjacency-list graph. On sparse graphs with m = O(n log n), total time reduces to O(n log²n).

Common pitfall: Assuming no negative cycle without checking. Always let bellman() run — if it returns false, report "negative cycle" and stop. Never skip this check on unknown graphs.

In practice, for the same raw operation count, Floyd is 2–5× faster. This shifts the practical crossover toward Johnson needing an even larger sparsity advantage.

⑨ When to use / detection pattern: Any time you read the problem constraints and must decide the right APSP algorithm before the contest clock runs out. Compute Floyd cost = n³ and Johnson cost = n × m × log₂(n) as integers, then pick the smaller one. Always do this computation — never guess.

Common pitfall: Automatically reaching for Floyd because it is "simpler." For large sparse graphs with n > 700, Floyd will TLE. Conversely, automatically reaching for Johnson because "it has negative edges" — on dense graphs with n = 400, m = 160000, Floyd is 5× faster than Johnson. Always compute the crossover.

E CP Implementation Style: SPFA Preprocessing + Dijkstra Reruns

① Problem Story: In competitive programming, the standard textbook Johnson description adds an explicit super-source vertex S and n extra edges to the graph. Your §18 template avoids this entirely. You need to understand precisely what your own code is doing and why it is equivalent — because understanding it is essential for debugging and adaptation.

② Abstraction: The CP version initializes pot[v] = 0 for all v and pushes all vertices into the SPFA queue simultaneously. This is mathematically equivalent to having a virtual super-source S with S→v edges of weight 0 for every vertex v. The SPFA finds shortest paths from S, setting h[v] = dist(S, v), which satisfies the triangle inequality h[v] ≤ h[u] + w(u,v) for all edges (the only property we need for reweighting to produce non-negative edges).

③ Why the CP-style SPFA fits: The textbook super-source approach requires modifying the graph (adding a node and n edges), then running Bellman-Ford from S, then cleaning up. This is error-prone in a contest. The CP version achieves the same result by exploiting the fact that initializing all potentials to 0 and starting all vertices in the SPFA queue is bit-for-bit equivalent to the super-source approach. No graph modification is needed, no extra memory for the super-source edges, and negative cycle detection is preserved identically.

④ What changes from the textbook algorithm: Implementation only. No explicit super-source node is added to the graph. No extra edges are allocated. The SPFA queue starts with all n vertices. The cnt-- round counter replaces the explicit "n rounds of Bellman-Ford" check. Negative cycle detection fires when cnt hits 0 while relaxations are still pending.

⑤ What stays invariant: The mathematical guarantee that after bellman(), for every edge (u,v): pot[v] ≤ pot[u] + w(u,v) (triangle inequality). This is the only property required for reweighting to make all edges non-negative. All theoretical guarantees from §12–§15 transfer exactly to the CP version because the starting state (all h=0, all vertices in queue) is provably equivalent to the super-source starting state.

⑥ Correctness reasoning: The super-source S has dist(S,v) = 0 for every vertex v (since S→v has weight 0). Running Bellman-Ford from S gives pot[v] = dist(S,v) = shortest path from S to v considering all original edges. Starting the SPFA with all pot[v]=0 and all vertices in the queue achieves exactly this: the first relaxation pass propagates from all vertices simultaneously, which is equivalent to relaxing from S with 0-weight edges to all vertices. After n rounds, the SPFA result equals the Bellman-Ford result from S. All subsequent reweighting and recovery arguments hold identically.

⑦ Full AC-ready code:

// The §18 bellman() IS the CP-style SPFA. Annotated here to show equivalence:

bool bellman() {
    ++inqid;
    queue<int> q;
    for(int src=0;src<n;++src){
        pot[src]=0;         // ↔ dist(S→src) = 0 via 0-weight edge from super-source S
        q.push(src);        // ↔ start BF from super-source S, all 0-weight edges enqueued
        inq[src]=inqid;
    }
    int cnt=n;              // ↔ at most n rounds of Bellman-Ford (n-1 relaxations + 1 check)
    while(!q.empty()&&cnt--){  // ↔ process one full "round" per iteration
        int s=(int)q.size();
        while(s--){
            int u=q.front();q.pop();inq[u]=0;
            edges(u,v,e,cs){
                ll dv=pot[u]+cs;
                if(pot[v]>dv){
                    if(!cnt)return false; // ↔ n+1th relaxation → negative cycle detected
                    pot[v]=dv;
                    if(inq[v]!=inqid){inq[v]=inqid;q.push(v);}
                }
            }
        }
    }
    return true; // ↔ all n rounds completed without further relaxation → potentials valid
}
// Textbook equivalence table:
// pot[v]=0 for all v  ←→  S→v edge with weight 0
// all v pushed to q   ←→  start BF from super-source S
// cnt-- rounds        ←→  n rounds of Bellman-Ford
// !cnt early exit     ←→  n+1th relaxation → negative cycle

⑧ Complexity: Time O(nm) for the SPFA — identical to standard Bellman-Ford from a super-source, since each of the n "rounds" processes each edge at most once when the queue is well-managed. Space O(n + m): O(n) for pot[], inq[], O(m) for the graph. This phase dominates only when Dijkstra is very fast (sparse, small m), otherwise the n×Dijkstra phase dominates at O(nm log n).

⑨ When to use / detection pattern: Any time you use the forward-star / adjacency-list Johnson template from §18 in a contest. Also when debugging: understanding this equivalence helps trace incorrect potential values. If pot[] contains unexpected values after bellman(), trace the round counter and queue initialization to verify the super-source equivalence holds.

Common pitfall: Not understanding the cnt-- early-exit condition. If the outer while loop runs all n rounds (cnt decrements from n to 0) and there are still pending relaxations when cnt reaches 0, a negative cycle is detected. If all relaxations complete within n rounds (queue drains while cnt > 0), potentials are valid. Confusing "queue drains" with "cnt hits 0" leads to incorrect negative cycle detection.

F Sparse APSP with Negative Edges + Path Reconstruction

① Problem Story: Same sparse graph with negative edges as Pattern A — but the problem now asks you to output the actual vertex sequence on the shortest path between queried pairs, not just the distance.

② Abstraction: Run full Johnson (Bellman-Ford + n Dijkstras) while maintaining a par[src][v] parent-edge array inside each Dijkstra call. After Dijkstra from source s, trace back from destination t via par[s][t] to reconstruct the path in the original graph.

③ Why Johnson fits: Path reconstruction in a large sparse graph with negative edges requires running a correct SSSP from each source. Johnson makes Dijkstra valid on such graphs via reweighting. The critical insight is that the telescoping property (§13) not only preserves which path is shortest — it preserves which edge is the correct predecessor at each vertex. The parent-edge array recorded in the reweighted Dijkstra is identical to the parent-edge array in the original-weight shortest-path tree, making reconstruction fully correct without any additional adjustment.

④ What changes from the base Johnson algorithm: Inside dijkstra(), when a node v is relaxed via edge e, record par[src][v] = e. After Dijkstra completes, add a path-trace helper getPath(s, t) that follows parent edges backwards from t to src. The par[][] array is already allocated in the §18 template as the second argument to dijkstra().

⑤ What stays invariant: The entire 4-phase Johnson pipeline: bellman() → n × dijkstra() → recovery. The reweighting formula w′(u,v) = w(u,v) + h[u] − h[v] and the recovery formula are unchanged. The parent edges stored in par[][] are already being recorded in the §18 template — no new data structure is needed.

⑥ Correctness reasoning: Reweighting does not change which edge is the correct predecessor on the shortest path — it only changes the numeric weight of each edge. The same edge that minimizes reweighted cost at vertex v minimizes original cost at v (by the telescoping argument: for any two paths from s to v, the reweighted costs differ by the same constant h[s] − h[v], so the ordering is preserved). Parent edges recorded in the reweighted Dijkstra are therefore identical to parent edges in the original-weight shortest-path tree.

⑦ Full AC-ready code:

// Uses §18 template with par[N][N] already allocated. Add getPath() helper:

vector<int> getPath(int s, int t) {

⑨ When to use / detection pattern: Any sparse Johnson problem that requires printing the actual path. The par[][] array is already built into the §18 template — the only overhead is adding the frm[] array for edge-source tracking and the getPath() helper. Detection keywords: "print the path," "output the vertices," "trace the route," "find a route."

Common pitfall: Tracing parent edges using only edge destinations (to[e]) and forgetting to store edge sources (frm[e]). Since par[s][v] stores the incoming edge index at v, you need frm[e] to find the vertex before v on the path. Always add int frm[M]; frm[ne]=u; to addEdge() when path reconstruction is needed. G Detecting and Reporting Unreachable Pairs After Johnson

① Problem Story: Your graph is not strongly connected — some pairs (u, v) have no path at all. After running Johnson, you need to distinguish between "finite shortest path," "unreachable," and correctly avoid printing garbage values for disconnected pairs.

② Abstraction: Inside Dijkstra, any vertex whose final distance remains at the sentinel value (0x3f3f3f3f3f3f3f3f ≈ 4.56×10¹⁸, set by memset(dst, 0x3f, …) — note this is slightly larger than the named constant INF = (ll)4e18, but both are far greater than any real path length, so the threshold check > (ll)1e15 correctly identifies both as unreachable) was never reached. The recovery step inside dijkstra() already guards against this with if (dst[i] < inf) before adding the potential correction. After johnson() returns, check dist[s][t] against a large concrete threshold (1e15) to detect unreachability.

③ Why Johnson with threshold check fits: Not all real graphs are strongly connected. After Johnson runs Dijkstra from each source, unreachable vertices retain their sentinel distance — the guard if (dst[i] < inf) in §18's dijkstra() already prevents the potential correction from being applied to unreachable pairs. The only remaining step is a threshold check at query time. This requires zero structural change to the Johnson pipeline and zero overhead to the main computation — it is a purely cosmetic addition to the output logic.

④ What changes from the base Johnson algorithm: At query time, add an explicit unreachability check before reporting any distance. The check compares dist[s][t] against a large concrete threshold (1e15 works reliably). No change to the pipeline itself.

⑤ What stays invariant: The full Johnson pipeline. The Dijkstra sentinel value and the recovery guard already in §18's dijkstra() function. The recovery formula pot[i] − pot[src] is only applied when dst[i] < inf, correctly leaving unreachable distances at the large sentinel value. The output loop structure is unchanged; only a conditional branch is added.

⑥ Correctness reasoning: An unreachable vertex v satisfies dst[v] = 0x3f3f… after Dijkstra (set by memset and never improved). The guard if (dst[i] < inf) in the recovery step leaves it unchanged — the potential correction is not applied. So dist[s][t] remains large and the threshold check dist[s][t] > (ll)1e15 correctly identifies it. No false finite distances are introduced: a reachable vertex always has dst[v] improved to below the sentinel before recovery adds at most pot[v] − pot[src], which is bounded by the actual distances.

H Subset APSP — Running Johnson Only from a Specific Set of Sources

① Problem Story: You have a large sparse graph (n = 10000, m = 50000) but only need shortest paths from a small set of k source vertices (k ≤ 50) to all other vertices — not the full n × n matrix. Running n Dijkstras would waste time on sources you don't care about.

② Abstraction: Run Bellman-Ford once as usual to compute valid potentials pot[v] for all vertices. Then run Dijkstra only from the k sources you actually need. Recover distances as normal. You get k rows of the full APSP matrix in O(nm + k·m·log n) instead of O(nm + nm log n) = O(nm log n) for full Johnson.

③ Why Johnson fits: Johnson's potential computation (the Bellman-Ford phase) is a global graph property — it does not depend on which sources you care about. Once valid potentials exist, each Dijkstra is an independent, self-contained SSSP computation from a single source. The k Dijkstras you run give exactly k rows of the APSP matrix, correct and efficiently. This selectivity is architecturally impossible with Floyd — Floyd's single cubic-time pass computes all n² distances inseparably. Johnson's two-phase structure is the key enabler: separate "prepare the graph" from "query per source."

④ What changes from the base Johnson algorithm: Instead of looping for (int s = 0; s < n; ++s) inside johnson(), you loop only over your source set: for (int s : sources). Allocate only k rows of the dist and par matrices if memory is tight. The bellman() call is unchanged — it must still process all vertices globally.

⑤ What stays invariant: The Bellman-Ford phase is completely unchanged — you still need globally valid potentials, which requires processing all n vertices from the virtual super-source. The dijkstra() function and recovery arithmetic are unchanged. Only the set of sources iterated over changes from {0, 1, …, n-1} to the k-element source set.

⑥ Correctness reasoning: Potentials are globally valid regardless of how many Dijkstras you run — they are a property of the edge set, not of a specific source. Each individual Dijkstra is a correct SSSP computation on the reweighted graph, giving the correct shortest distances from its source to all other vertices. Recovery is per-source and independent. Running fewer Dijkstras simply produces fewer rows of the APSP matrix — not incorrect rows.

⑦ Full AC-ready code:

⑨ When to use / detection pattern: Any problem where you need "shortest paths from every terminal / special vertex to all others," but the number of terminals k is far smaller than n. Bitset Transitive Closure — O(n³/64)

The boolean transitive closure is the single Floyd variant where a simple code change gives a 64× speedup. The key: replace each row with a bitset<N>, so each OR-AND operation processes 64 reachability bits in a single CPU instruction:

// Standard bool[][] — O(n³), ~2.5 seconds at n=1000
for (int k=0;k<n;++k) for (int i=0;i<n;++i) if(reach[i][k])
  for (int j=0;j<n;++j) reach[i][j]|=reach[k][j];

// Bitset version — O(n³/64), ~40 ms at n=1000 (64× faster)
const int MAXN = 1005;
static bitset<MAXN> reach[MAXN];
// Init: reach[i].set(i); for each edge u→v: reach[u].set(v);
for (int k=0;k<n;++k)
  for (int i=0;i<n;++i)
    if(reach[i][k]) reach[i] |= reach[k];  // one SIMD OR = 64 bool updates
// Query: reach[u].test(v) or reach[u][v]
// Count reachable: reach[u].count()
// This makes transitive closure feasible at n=2000 (where bool[][] would TLE)

Graph Compression for Large-n Variants

When the problem has large n (say 10⁵) but you only need distances among a small set of "special" vertices (say k ≤ 300), you can compress the graph without approximation:

// Graph compression pattern — APSP on k vertices from an n-vertex graph
// 1. Run Dijkstra from each of the k special vertices (indices in 'important[]')
static ll sub_dist[305][305];  // k×k distance matrix
for (int a = 0; a < k; ++a) {
  int src = important[a];
  dijkstra_full(src, tmp_dist);  // single-source Dijkstra on full n-vertex graph
  for (int b = 0; b < k; ++b)
    sub_dist[a][b] = tmp_dist[important[b]];  // project to k×k
}
// 2. Now run Floyd on the k×k matrix
for (int ki=0;ki<k;++ki) for (int i=0;i<k;++i) {
  const ll d = sub_dist[i][ki]; if(d>=INF/2) continue;
  for (int j=0;j<k;++j)
    sub_dist[i][j] = min(sub_dist[i][j], d + sub_dist[ki][j]);
}
// Total cost: O(k × m log n + k³)  — feasible even at n=10⁵, k=300
// vs. Floyd on full graph: O(n³) = 10¹⁵ — completely impossible

return false;
Johnson: skip Dijkstra if source is isolated

After Bellman-Ford, a vertex with no outgoing edges has pot[v] = 0 and produces a trivial Dijkstra (queue empties instantly). More usefully: if the problem only asks about pairs where source and destination are both in a given set, only run Dijkstra from sources in that set.

for (int s : query_sources) {
  if (head[s] == -1) continue; // isolated
  dijkstra(s, dist[s], par[s]);
}

Micro-Optimizations Checklist

These are the sub-5-minute changes that can push a borderline solution from TLE to AC. Apply in order of impact:

Fast I/O
Impact: ~10-20% on output-heavy problems
ios::sync_with_stdio(false); cin.tie(nullptr); at the start of main. For large output (full n×n matrix), prefer printf("%lld ", dist[i][j]); over cout — printf is ~3× faster for formatted integers.
Static arrays over vectors
Impact: ~15-25% for Floyd at n=500
static ll dist[605][605]; declared globally or with static keyword. Avoids heap allocation, pointer-chasing through vector-of-vectors, and stack overflow for large arrays. One flat 2D array = one pointer to contiguous memory.
Hoist loop-invariants
Impact: ~5-10% — compiler usually does this
const ll dik = dist[i][k]; before the j-loop ensures the compiler treats dik as a register value. While modern compilers (GCC -O2) hoist this automatically, making it explicit prevents regression if the array is aliased or volatile.
Use pointers, not copies
Impact: critical for Johnson's n-call loop
Always pass ll *dst and int *par to dijkstra(), not vector<ll>. Call as dijkstra(s, dist[s], par[s]) — passes pointers to existing rows. Never allocate a new vector per Dijkstra call (that would be O(n²) in allocation alone).
Avoid std::endl
Impact: ~2-5× slower than '\n'
std::endl flushes the output buffer on every call. For a 500×500 output matrix (250 000 lines), this is 250 000 flush syscalls. Use '\n' instead — the buffer flushes once at program exit. Combined with sync_with_stdio(false), this can be 10× faster for large output.
memset for bulk init
Impact: ~3× faster than explicit loop
memset(dst, 0x3f, n * sizeof dst[0]) initialises n long longs to the 0x3f sentinel in one call. For Floyd's initial matrix: memset(dist, 0x3f, sizeof dist) initialises the entire n×n array in one memset call. Only then set diagonals and edges explicitly.

After Floyd, dist[u][u] < 0 → hasNegativeCycle() returns true. Any path through u has undefined (−∞) distance. Negative edges (no cycle) Floyd handles negative edges natively — unlike Dijkstra, there is no greedy invariant that requires non-negative weights. The induction proof in §6 is valid for any real-valued weights. No special handling. Ensure ll (64-bit) is used — negative edges combined with large INF can cause sign issues with 32-bit int.
Negative cycle present After Floyd, dist[i][i] < 0 for every vertex i on the cycle. Any path that enters and exits the cycle would have cost −∞. The propagation variant (§10-G) is needed to mark all affected pairs. Call hasNegativeCycle() after run(). For full −∞ propagation, apply the second O(n³) pass from §10-G.
Undirected graph + negative edge Instant negative cycle. An undirected edge (u, v, w) with w < 0 adds u→v and v→u both with weight w. The 2-cycle u→v→u has total cost 2w < 0 — a negative cycle exists by definition. Floyd is still correct in detecting it, but the graph semantics are usually wrong. Warn if negative-weight undirected edges appear in input. In addEdge(bool directed), if !directed && w < 0: either reject the input or accept it and expect hasNegativeCycle() to trigger.
INF overflow in dist[i][k] + dist[k][j] Without a guard, adding two INF values (4e18 + 4e18) overflows a 64-bit signed integer. The result wraps to a large negative number which then appears to be a very short path, corrupting the matrix. if (dist[i][k] >= INF/2) continue; eliminates all unreachable-source rows before the inner j-loop. Any remaining dist[k][j] >= INF/2 is also guarded.

Dijkstra handles zero edges correctly (they are non-negative). Zero-weight cycles are not negative cycles in either the original or reweighted graph. No special handling. The if (dst[v] > nd) strict inequality in Dijkstra correctly skips equal-cost non-improvements.
Negative cycle present bellman() detects it: if any vertex's potential still improves after n rounds of relaxation, a negative cycle exists. The if (!cnt) return false guard fires. Johnson returns false. Dijkstra is never called — no corrupt distances are produced. The if (!bellman()) return false gate in johnson(). Always handle the false case at the call site: output "Negative cycle" and stop.
Undirected graph + negative edge Same issue as Floyd: adding u→v and v→u with weight w < 0 creates a 2-cycle of total cost 2w < 0. bellman() will detect it and return false. If the problem guarantees undirected graphs with non-negative weights only, this case cannot arise, but the detection is automatic. Handled automatically by bellman(). No special code needed — just always check the return value.
Reweight overflow (pot values very negative) If a vertex has a very negative potential (e.g., pot[v] = −10⁹) and an edge weight w = 10⁹, the reweighted value w + pot[u] − pot[v] may be near 2×10⁹. With ll and edge weights fitting in 32-bit int, this stays within 64-bit range. However, if edge weights themselves can be ±(ll)4e18, overflow in the reweighting arithmetic is possible. Use int for edge weights (as in the §18 template) and ll for distances. This keeps reweighting arithmetic safe: (ll)cs + pot[u] − pot[v] stays in range when cs fits in int and pot values are bounded by the sum of all edge weights.
Restoring distances from wrong source sentinel The recovery step dst[i] += pot[i] − pot[src] only fires when dst[i] < inf. For the source itself: dst[src] = 0 before recovery, and pot[src] − pot[src] = 0, so dist[src][src] stays 0. Correct. For unreachable vertices the guard keeps them at the sentinel — but the sentinel after memset(0x3f) is ≈4.56×10¹⁸, which differs from the named INF=(ll)4e18. Use the threshold > (ll)1e15 for all unreachability tests, not equality to either sentinel constant. Use if (dist[i][j] > (ll)1e15) in all output and query code. Never use dist[i][j] == INF.

🔍 Edge-Case Rules

⛔ Negative Cycles

Run Floyd first. After the triple loop, check every diagonal: if dist[i][i] < 0 then vertex i is on a negative cycle. Any path passing through such a vertex has undefined (−∞) distance — mark those pairs explicitly rather than returning a spurious number. With Johnson, the Bellman-Ford phase returns false as the early-abort signal; do not proceed to Dijkstra.

📊 Dense vs Sparse

The crossover lives at m ≈ n² / log n. Concretely: if n = 1 000 and m ≤ ~100 000, use Dijkstra × n (or Johnson). If m > ~100 000 for that n, Floyd wins on constant factors even though both are nominally O(n³). Always benchmark if you are right at the boundary — the log factor is not always small in practice.

🔁 Multi-Source Queries

When the problem gives you a fixed source set S and asks all S→V distances (not all-pairs), don't run the full APSP. Run Bellman-Ford once (or confirm no negatives), then run Dijkstra from each source in S. This costs O(|S| · m log n) instead of O(n · m log n). The saving is multiplicative when |S| ≪ n.

🔄 Repeated Queries

If the graph is static and you receive Q offline queries of the form "distance from u to v", always precompute the full dist[n][n] matrix upfront and answer each query in O(1). Do not run a fresh Dijkstra per query — even O(m log n) per query blows up when Q is large. The break-even is roughly Q > n / log n queries.

🚨 Contest Traps

These are the three mistakes that cost the most points in timed contests. Each one looks fine locally and only explodes under judge constraints.

⏱️
When Floyd TLEs

Floyd's constant factor is deceptively large: the innermost loop touches n³ cells with random-ish memory access. At n = 700 you get ~343 M iterations, which is borderline on a 1 s limit. At n = 1 000 it is 1 B iterations — always TLE. The fix is not to optimize Floyd; it is to switch to Johnson (if negatives exist) or repeated Dijkstra (if not). Never try to squeeze Floyd past n ≈ 600 in a contest.

💥
When Johnson Fails (silently)

Johnson silently produces garbage when a negative cycle exists and you forget to check the Bellman-Ford return value. Dijkstra, run after broken reweighting, will appear to terminate but distances are meaningless. Always gate Dijkstra on if (!bellman()) { /* report neg-cycle */ return; }. The second trap: Johnson requires the reweighting formula w'(u,v) = w(u,v) + pot[u] − pot[v] to be applied before Dijkstra and then undone after. Forgetting the undo step corrupts every distance you output.

❌
When Dijkstra Is Wrong

Dijkstra with a priority queue is incorrect the moment any edge weight is negative — even a single edge of −1 breaks the greedy invariant and produces wrong answers with no runtime error. If you see negative weights anywhere in the input, you must use Bellman-Ford, SPFA, or Johnson. The symptom is subtle: Dijkstra will terminate, it just returns distances that are too large for some pairs. This is one of the most common wrong-answer bugs in APSP problems.

Changes edge weights so shortest paths become well-behaved — i.e., all edges non-negative — without disturbing which path is shortest. Then delegates to Dijkstra, which it would otherwise be illegal to use.
🔑 The Unified View
Both solve APSP by transforming the problem into something easier to solve. Floyd transforms "all pairs" into a series of single-intermediate expansions. Johnson transforms a graph with negatives into one without. Transformation is the master idea.
One sentence each, to carry forever:

Floyd: "After allowing vertex k, update every pair through k." — 3 nested loops, one DP.
Johnson: "Reweight so Dijkstra is legal, run Dijkstra from everyone, undo the reweighting." — 4 clean phases.
Both: O(n³) vs O(nm log n) — density decides the winner; the math decides the possibility.

🧠 Final Mental Model — Never Forget This

This is not a summary. It is the single insight that makes both algorithms click into place — permanently.

Floyd–Warshall
Controls which vertices are allowed as shortcuts.

Before the algorithm runs, only direct edges exist. After processing vertex k, every pair (i, j) is allowed to route through k as an intermediate. After all n vertices are processed, every possible shortcut has been considered. The shortest path has nowhere left to hide.

Unlock k → update every pair through k → repeat for all k.
Johnson's Algorithm
Transforms the graph so the right tool becomes legal to use.

Dijkstra is the fastest single-source shortest-path algorithm — but it breaks on negative edges. Johnson's only job is to rewrite the edge weights so that all weights become non-negative, without changing which path is shortest. Once the graph is "safe", Dijkstra is run from every vertex.

Make Dijkstra legal → run Dijkstra for every source → undo the rewrite.
The Unified Abstraction
Both algorithms solve APSP by changing the frame of the problem — not by searching harder.

Floyd reframes "find shortest paths between all pairs simultaneously" into "extend the set of allowed intermediate vertices one step at a time." Johnson reframes "find shortest paths on a graph with negative edges" into "find shortest paths on a graph with no negative edges." Neither algorithm fights the problem directly. Both make the problem easier to solve by reframing it first — and then solve the easier version.

Floyd: reframe the search space
Both: reframe before solving
Johnson: reframe the graph itself

🏁 How to Think in Contests

Run this five-question mental script the moment you see "shortest path" anywhere in a problem statement. Each question eliminates one algorithm or narrows you to one variant. The whole script takes under three seconds once practiced.

1
Is n ≤ 500?
Yes → Floyd, immediately. Stop reading. The size constraint is the only signal you need. Floyd handles dense, sparse, negative, positive, everything within this range.
2
Do negative-weight edges exist?
No → Dijkstra × n (sparse) or Floyd (dense). You don't need Johnson. Yes → go to question 3.
3
Can a negative cycle exist?
Yes, and the problem asks to detect it → Floyd diagonal check (small n) or Johnson's Bellman-Ford phase (large n). No cycles guaranteed → go to question 4.
4
Is the graph sparse (m ≪ n²)?
Yes → Johnson (negative edges exist) or Dijkstra × n (no negatives). No (dense, m ≈ n²) → Floyd wins even over Johnson, because Johnson degrades to O(n³ log n) on dense graphs.
5
Is the objective non-additive? (minimax, maximin, closure, bottleneck)
Floyd variant, always. Johnson and Dijkstra are built on the additive distance semiring and cannot be modified for these objectives without losing their correctness proofs. Floyd's triple-loop naturally accepts any associative binary operator in the inner step.

The One Thing to Carry Forever
Every great algorithm is a reframing, not a brute force.

Floyd didn't try to find all n² paths at once. It asked: "what if I only added one new shortcut option?" Johnson didn't try to patch Dijkstra to handle negatives.
