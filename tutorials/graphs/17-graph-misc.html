How to Think About Graph Misc Problems

The word miscellaneous sounds like a junk drawer. It is not. Every topic in this file is a real graph problem; what makes them feel miscellaneous is that the hard part is not the final traversal algorithm, but discovering the right graph model in the first place.

💡

Core insight: Standard graph problems hand you a graph and ask you to traverse or measure it. Misc graph problems hide the graph inside a story, and your first job is always to construct the correct graph before any algorithm can run.

Why These Problems Are Hard to Recognize

When a problem says "find the shortest path from A to B," you immediately recognize the model. But when it says "find the minimum flips to turn string S into string T," or "determine whether every road can be painted in one pass," the graph is hidden inside the wording. Misc problems are hard because the objects that become nodes and the relationships that become edges are not explicitly stated — you have to infer them.

There are six main ways a graph gets hidden in a problem:

❶

Hidden node identity. The nodes are not "cities" but something else: edges, states, configurations, bit-masks, colors. Ask: "what object am I really reasoning about?"

❷

Hidden edge meaning. Edges are not explicit connections but transitions, interactions, or conflicts between objects.

❸

Existence reasoning. The question is not "find the shortest path" but "does any path of this type exist?" — Euler, Hamiltonian, valid colorings.

❹

Graph transformation. You cannot solve the problem in the original graph, but after transforming it the solution becomes standard.

❺

Implicit graph. The graph is never explicitly given — states are generated on-the-fly from a starting configuration by applying moves.

❻

Structural properties. The question asks about a graph-theoretic property (planarity, diameter, chain shape) that characterizes the structure itself.

The Recognition Checklist

🔎 Model identification

  • Are we traversing edges or vertices as the primary objects?
  • Is this really an Euler, Hamilton, or hidden-state problem?
  • Should an edge of the original graph become a node in a new graph?
  • Should each reachable configuration become a node?
  • Are we reasoning about existence conditions instead of shortest paths?
  • Is the graph implicit rather than explicitly given?

🔎 Algorithm selection

  • Can a degree condition (Euler) solve this instantly?
  • Is n small enough for bitmask DP (n ≤ 20)?
  • Is this a tree? (use two-BFS for diameter)
  • Does the graph fit in memory? (if not: IDDFS)
  • Are both source and target known? (consider bidirectional BFS)
  • Does planarity bound the edge count to O(V)?

1. Path Graph

🎯

Why this section exists: The path graph is the simplest graph structure, yet students regularly confuse it with "a path inside a larger graph." Recognizing when an entire graph is a chain is a classification skill that appears in reduction arguments and structural proofs throughout CP.

⚠

Common Wrong Model: Treating "a path in a graph" (a walk visiting some vertices) as if it were "a path graph" (a graph whose entire structure is a chain). Always ask: is the entire abstract graph shaped like a chain? Any branching vertex (degree ≥ 3) or cycle disqualifies it.

A path graph on n ≥ 2 vertices is a graph whose vertices can be arranged as v₁, v₂, …, vₙ such that the only edges are (v₁,v₂), (v₂,v₃), …, (vₙ₋₁,vₙ). The graph looks like a straight chain.

Formal Condition

✓

Key characterization: A graph is a path graph iff it is connected, exactly two vertices have degree 1 (the endpoints), and every other vertex has degree 2.

What It Is Not

⚠

Important distinction: A path graph = the whole graph is a chain. A path in a graph = a sequence of vertices inside a larger graph. Do not confuse them.

Mental Model

Picture a line of train stations: each station connects only to the one before it and the one after it. The two end-stations have just one connection each (degree 1); every middle station has exactly two connections (degree 2). No branching, no loops, no shortcuts — just a pure chain. That mental image of a railway line is the path graph. Any deviation — a branch, a cycle, a disconnected segment — immediately disqualifies the graph from being a path graph.

Where It Appears in Problems

Path graphs appear in structural classification tasks (is this graph a path?), in reduction arguments (transform an instance to path-graph form), and in problems that enumerate graphs with restricted degree sequences. They are also the simplest nontrivial connected graph, so they serve as base cases and sanity checks in proofs.

Algorithm / Method

  1. Verify connectivity — the graph must have exactly one connected component.
  2. Count degree-1 vertices — there must be exactly 2.
  3. Check all other degrees — every non-endpoint vertex must have degree exactly 2.
  4. No vertex may have degree ≥ 3. If any branching exists, it is not a path graph.
Time: O(V + E)  |  Space: O(V + E) for adjacency representation
⚠

Common Pitfall: Checking the degree sequence alone is insufficient — a disconnected graph can have the same degrees as a path graph. Always verify connectivity first, then degree conditions.

Practice Problem

🏆

Problem: CSES — Building Roads — Given a forest (disconnected tree), find the minimum number of edges to add to make it connected. Understanding path-graph structure is key to reasoning about which tree components form chains. Classify each component and then count edges needed.

2. Planar Graph

🎯

Why this section exists: In CP, planarity is almost never about testing whether a graph is planar — it is about using "the graph is planar" as a given condition to conclude E = O(V), justifying linear-time algorithms that would otherwise be too slow on dense graphs.

⚠

Common Wrong Model: Treating one bad drawing as proof of non-planarity. A graph is non-planar only if every possible drawing has crossings. One messy picture proves nothing — try a different layout before concluding.

A planar graph is a graph that can be drawn in the plane so that no two edges cross except at shared endpoints. The key phrase: can be drawn — planarity is a property of the abstract graph itself, not of any one picture.

What It Is Not

⚠

Common confusion: A graph with a crossing in one drawing is not automatically non-planar. It may just be a bad drawing. Non-planar means every possible drawing has crossings.

Planar Embedding Demo — Same graph, two drawings
This graph has a crossing in the default layout. Press "Redraw Planar" to see that it IS actually planar — the graph just needed a better drawing.

Formal Condition

For any connected planar graph with V vertices, E edges, and F faces:

V − E + F = 2

This is one of the most important facts in planar graph theory.

Famous Non-Planar Graphs

K₅ — Complete Graph on 5 Vertices
Truly non-planar — no redrawing removes all crossings
K₃₇₃ — Complete Bipartite 3×3
Truly non-planar — the other classic obstacle
Trees are always planar Any cycle is planar K₅ is non-planar K₃₇₃ is non-planar Bad drawing ≠ non-planar

Mental Model

Imagine drawing the graph on a rubber sheet — a planar graph can always be stretched and rearranged until no edges cross. The graph's abstract structure is what matters, not any particular picture. If any drawing avoids crossings, the graph is planar.

What Euler's Formula Can and Cannot Do

✓

Euler's formula can: Immediately refute planarity. If a connected graph has V vertices and E edges and V − E + F = 2 cannot be satisfied with positive integer F, it is non-planar. Also: for simple planar graphs with V ≥ 3, we always have E ≤ 3V − 6. Violating this bound proves non-planarity.

⚠

Euler's formula cannot: Confirm planarity by itself. Satisfying E ≤ 3V − 6 is necessary but not sufficient. K₃,₃ satisfies the edge bound yet is non-planar. Full planarity testing (Kuratowski or Boyer-Myrvold algorithm) is required for confirmation.

Algorithm / Method

The most important algorithmic operation in CP is applying the density bound: check if E > 3V − 6. If it is, the graph is immediately non-planar (no drawing test needed). For full confirmation, the Boyer-Myrvold or Hopcroft-Tarjan algorithm runs in O(V) time, but in contests planarity is almost always a given precondition.

Quick rejection: if E > 3V − 6 then non-planar  |  Full test: O(V) Boyer-Myrvold algorithm

Where It Appears in Problems

In competitive programming, planar graph problems almost never ask you to test planarity from scratch. Instead, the problem typically states "the graph is planar" and uses it to bound E ≤ 3V − 6, meaning the total number of edges is O(V). This lets O(V) algorithms handle inputs that would be too slow for dense general graphs. Recognizing this density bound and using it to justify linear-time solutions is the primary practical skill.

⚠

Common Pitfall: Using Euler's formula as both a necessary and sufficient condition for planarity. The formula is a necessary condition — satisfying it does not guarantee the graph is planar. Always distinguish "rules out planarity" from "confirms planarity."

Practice Problem

🏆

Problem: Codeforces 193A — Kirill the Gardener 1 — This problem involves reasoning about graph density. Planar graphs allow you to bound the number of edges tightly, which is a key technique. Practice applying E ≤ 3V − 6 to eliminate impossible cases in graph construction problems.

3. Line Graph

🎯

Why this section exists: Some problems are naturally about edges rather than vertices. When problem objects are edges and the constraint is "two objects conflict if they share an endpoint," the line graph transformation converts it into a standard vertex-based problem.

⚠

Common Wrong Model: Trying to solve edge-interaction problems directly on the original graph. If the problem objects are "roads" and two roads "conflict if they share an intersection," the correct model has roads as nodes in a new graph, not the original city graph.

The line graph L(G) is a graph built from another graph G. Construction rule: every edge of G becomes a vertex in L(G). Two vertices in L(G) are adjacent iff the corresponding edges in G share an endpoint.

L(G): edge adjacency in G → vertex adjacency in L(G)
💡

The line graph answers: Which edges in the original graph touch each other? Instead of studying adjacency between vertices, we study adjacency between edges.

Line Graph Construction — Hover an edge to see its vertex in L(G)
Hover over an edge in the original graph (left) to highlight the corresponding vertex in L(G) (right). Two edges sharing a vertex → connected in L(G).

What It Is Not

⚠

Name confusion alert: Do not confuse line graph (a transformation of a graph) with path graph (a graph shaped like a chain). They are completely different concepts.

Why line graphs matter click to expand

Line graphs appear in:

  • Edge-based problem formulations
  • Matching-related thinking (edge matchings become vertex matchings in L(G))
  • Graph transformation and reduction proofs
  • Problems where "objects are edges" rather than vertices
  • Theoretical characterizations (Whitney's theorem)

Story → Abstraction → Why This Model Works

1. Problem story

A city has several roads. Two roads "conflict" if they share an intersection. Find the maximum number of roads that can be active simultaneously.

2. Raw observation

The objects are roads (edges of the city graph). The relation is "share an endpoint." The question is maximum independent set — of roads, not intersections.

3. Abstraction

Build L(G): each road becomes a vertex. Two vertices are adjacent iff the roads share an intersection. Now the question is: find the maximum independent set in L(G).

4. Why this is correct

The conflict relation (sharing an endpoint) is exactly the edge-adjacency relation in L(G) by construction. Roads can be active simultaneously ⇔ they do not conflict ⇔ they form an independent set in L(G).

5. Common wrong interpretation

Solving "max non-conflicting roads" directly on the city graph, treating intersections as nodes — this mismodels the problem because the objects being selected are roads (edges), not intersections (vertices).

6. Final usable rule

When problem objects are edges and the constraint is "two cannot both be selected if they share an endpoint" — build L(G) and find an independent set. Applies to edge coloring, edge matching, conflict avoidance.

Mental Model

Suppose every road in a city is an object — you want to study which roads intersect each other. Build a new graph: each road becomes a dot, and draw a line between two dots if the corresponding roads share an intersection. That new graph is exactly the line graph L(G). The transformation shifts your perspective from "which vertices are adjacent?" to "which edges are adjacent?" — and that shift unlocks a family of edge-based problems that are hard to solve directly but straightforward after the transformation.

Formal Condition

✓

Formal definition: Given G = (V, E), the line graph L(G) has vertex set E (one vertex per edge of G). Two vertices e₁ and e₂ in L(G) are adjacent if and only if the corresponding edges in G share an endpoint. The number of vertices in L(G) equals the number of edges in G.

⚠

Not every graph is a line graph. The claw K₁,₃ (one center connected to three leaves) is the smallest graph that is NOT a line graph of any graph. Whitney's theorem gives the full characterization: a graph is a line graph if and only if it contains no induced K₁,₃ and satisfies additional conditions on odd cliques.

Where It Appears in Problems

The key insight is that edge-based problems can become vertex-based problems after the line graph transformation. Examples include: edge coloring (finding a proper coloring of edges of G corresponds to vertex coloring of L(G)), problems where "roads that share an intersection cannot both be used simultaneously" (model roads as vertices in the line graph, then find an independent set), and Eulerian path problems recast as Hamiltonian paths in the line graph.

Algorithm / Method

  1. Create one new vertex ve for each edge e in G.
  2. For every pair of edges (e₁, e₂) in G that share an endpoint in G, add edge (ve₁, ve₂) in L(G).
  3. The resulting graph L(G) has |E(G)| vertices and Σv C(deg(v), 2) edges, where the sum is over all vertices of G.
|V(L(G))| = |E(G)|  |  Build time: O(V + E)  |  Space: O(V + E)
⚠

Common Pitfall: Confusing the line graph transformation with the path graph concept because of similar naming. A line graph is a new graph built from edge-adjacency relationships; a path graph is a graph whose overall structure is a chain. They have nothing to do with each other.

Practice Problem

🏆

Problem: CSES — Monster Game I — This requires reasoning about which nodes (or edges) interact. Practicing the edge-to-vertex mental shift that the line graph encodes prepares you to recognize when a problem naturally lives in "edge space" and should be transformed before solving.

4. Hamiltonian Path and Circuit

🎯

Why this section exists: Hamiltonian problems look similar to Euler problems but require completely different tools. Recognizing which type you face — edge traversal vs vertex visitation — determines which algorithm you reach for immediately.

⚠

Common Wrong Model: Applying Euler-style degree conditions to Hamiltonian problems. Hamiltonian existence has no simple degree test. The correct approach for small n is bitmask DP; for general large n the problem is NP-hard.

A Hamiltonian path visits every vertex exactly once. A Hamiltonian circuit (cycle/tour) visits every vertex exactly once and returns to the start.

Euler vs Hamilton — The Most Important Distinction

These two concepts use different words (edges vs vertices) but look superficially similar. The diagrams below make the difference visually unmistakable.

✓ All vertices visited • Some edges unused

What It Is Not

🛠

Exact meaning: "every vertex exactly once" — not every edge, vertices are the focus. This is the main difference from Euler ideas.

🕒

Euler: use every edge exactly once. Simple degree conditions exist.

⚠

Hamiltonian: use every vertex exactly once. No simple universal degree test exists for general graphs.

Mental Model

Imagine a salesperson who must visit every city in a country exactly once before returning home. The question "does such a tour exist?" is the Hamiltonian circuit problem. There is no simple parity trick, no degree test, no polynomial-time shortcut for general graphs. You must either try all permutations (exponential), use bitmask DP for small n, or exploit special graph structure. This is fundamentally different from the Euler problem, where a single degree check gives an instant answer.

Formal Condition

Deciding whether a Hamiltonian path exists is NP-complete, meaning no polynomial-time algorithm is known for general graphs and finding one would imply P = NP. Unlike Euler circuits — which have a clean degree-based necessary and sufficient condition (all vertices even) — Hamiltonian paths have no simple structural certificate. You must either explore all possibilities or use problem-specific structure.

⚠

Dirac's and Ore's theorems give sufficient conditions for Hamiltonian cycles (e.g., every vertex has degree ≥ n/2), but they are not necessary. A graph can have a Hamiltonian cycle with low-degree vertices, and a high-degree graph may still lack one. These theorems give you a "shortcut" in some cases, but not a complete algorithm.

5. Tree Diameter

🎯

Why this section exists: Tree diameter is the canonical structural measurement on trees. The two-BFS trick is elegant and non-obvious — understanding it prepares you for tree center, radius, and subtree-DP patterns in harder problems.

⚠

Common Wrong Model: Applying the two-BFS trick to graphs that are not trees. The algorithm is proven correct for trees only. On cyclic or general weighted graphs, use all-pairs shortest paths instead.

The diameter of a tree is the length of the longest simple path — equivalently, the maximum distance between any two vertices. Distance = number of edges (unweighted) or total weight (weighted).

🌟

Beautiful 2-search property: Start from any vertex s, find farthest vertex a. Start from a, find farthest vertex b. The path a→b is a diameter. This works because trees have unique simple paths and no cycles.

What It Is Not

⚠

Trees only! The two-search trick is guaranteed correct only for trees. Be careful when applying to general weighted or cyclic graphs.

Mental Model

Picture a tree as a branching river system — water flows from many small tributaries into ever-larger channels. The diameter is the longest journey you can take through this river network from one source to another without ever backtracking. The two-BFS trick works perfectly here because a tree has no cycles: every path between two vertices is unique and well-defined, so "walk to the farthest reachable point" always lands you at one end of a longest path.

Where It Appears in Problems

Finding the longest path in a tree Tree center (midpoint of diameter path) Prerequisite for subtree-DP problems Routing and network design on trees

Algorithm / Method

  1. Step 1: Run BFS/DFS from any vertex s. Find the farthest vertex — call it a.
  2. Step 2: Run BFS/DFS from a. Find the farthest vertex from a — call it b.
  3. Result: The path a → b is a diameter of the tree. dist(a, b) is the diameter length.

Formal Condition

The correctness of the two-BFS method relies on a key tree property: between any two vertices in a tree there is exactly one simple path. This means distances are well-defined and there are no cycles to confuse BFS. The proof sketch: let (u*, v*) be a true diameter pair. Run BFS from any vertex s. It can be shown that the farthest vertex found from s is always one of the diameter endpoints (or equally far). Call it a. Running BFS from a then finds b — the other diameter endpoint — because a is guaranteed to be a diameter endpoint, and b is simply its farthest vertex.

Two BFS/DFS passes, each O(V + E)  |  Total: O(V) for trees (E = V − 1)
⚠

Trees only! The two-BFS diameter trick is proven correct only for trees (connected acyclic graphs). For general weighted graphs or graphs with cycles, the "farthest from any vertex" heuristic does NOT guarantee a diameter endpoint. Use a different algorithm (e.g., all-pairs shortest paths) for general graphs.

Practice Problem

🏆

Problem: CSES — Tree Diameter — The canonical practice problem. Given a tree, find its diameter. Implement the two-BFS method directly. After solving, also try the weighted variant to make sure you handle edge weights correctly in your BFS/DFS.

6. Iterative Deepening DFS (IDDFS)

🎯

Why this section exists: IDDFS solves a real dilemma: BFS guarantees the shallowest solution but requires exponential memory; DFS uses minimal memory but finds arbitrary-depth solutions first. IDDFS gets both benefits at once.

⚠

Common Wrong Model: Using IDDFS when BFS fits in memory and the graph is explicitly stored. IDDFS re-explores the top of the tree on every depth increase. For explicit graphs where all nodes fit in memory, standard BFS is simpler, faster, and has no repeated work.

IDDFS is a search strategy that repeatedly runs DFS with increasing depth limits: depth 0, 1, 2, 3, … until the goal is found. It combines the low memory style of DFS with the layer-by-layer depth guarantee of BFS.

Mental Model

Think of searching a filing cabinet whose drawers are labeled 0, 1, 2, 3, …. You first look only in drawer 0 (depth limit 0). If the target isn't there, you look in drawers 0 and 1 (depth limit 1). Then 0, 1, and 2 (depth limit 2) — and so on, re-opening shallower drawers each time. Yes, you re-examine the shallow drawers repeatedly, but because the last drawer is always the largest, the repeated work is dominated by the final search. You get the "find shallowest result first" guarantee of BFS while only remembering the current path in memory — just like DFS.

✅

DFS advantage: Low memory — only stores the current path on the stack.

✅

BFS advantage: Always finds the shallowest (minimum-depth) solution first.

✨

IDDFS gets both: BFS-like shallowest-solution guarantee + DFS-like O(d) memory, where d is the answer depth.

Complexity

Let b = branching factor, d = shallowest solution depth:

Time: O(b⁰ + b¹ + b² + … + bᵈ) = O(bᵈ)     Space: O(d)

What It Is Not

⚠

IDDFS is not a memory-efficient BFS replacement in all cases — it has higher constant factors than BFS because of repeated work near the root. It is also not the right tool when the graph is small, memory is plentiful, or the branching factor is very low. If BFS fits in memory, prefer BFS. IDDFS repeats work near the top of the tree many times. But in tree-like search spaces the last level dominates the total node count, so the repeated cost is often acceptable.

Algorithm / Method

A very common use of IDDFS in online judges: find the minimum number of edges from source s to target t using Iterative Deepening DFS.

#include <bits/stdc++.h>
using namespace std;

struct IDDFS {
    int n;
    vector<vector<int>> adj;
    vector<int> path;
    vector<int> answerPath;

    IDDFS(int n) : n(n) {
        adj.assign(n + 1, {});
    }

    void addEdge(int u, int v, bool undirected = true) {
        adj[u].push_back(v);
        if (undirected) adj[v].push_back(u);
    }

    bool dls(int u, int target, int depth, vector<int>& used) {
        path.push_back(u);

        if (u == target) {
            answerPath = path;
            return true;
        }

        if (depth == 0) {
            path.pop_back();
            return false;
        }

        used[u] = 1;

        for (int v : adj[u]) {
            if (!used[v]) {
                if (dls(v, target, depth - 1, used))
                    return true;
            }
        }

        used[u] = 0;
        path.pop_back();
        return false;
    }

    int solve(int s, int t, int maxDepth) {
        for (int limit = 0; limit <= maxDepth; limit++) {
            vector<int> used(n + 1, 0);
            path.clear();
            answerPath.clear();

            if (dls(s, t, limit, used)) {
                return limit;
            }
        }
        return -1;
    }
};

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, m;
    cin >> n >> m;

    IDDFS solver(n);

    for (int i = 0; i < m; i++) {
        int u, v;
        cin >> u >> v;
        solver.addEdge(u, v, true);
    }

    int s, t, maxDepth;
    cin >> s >> t >> maxDepth;

    int ans = solver.solve(s, t, maxDepth);

    if (ans == -1) {
        cout << "Target not reachable within depth " << maxDepth << '\n';
    } else {
        cout << "Minimum depth = " << ans << '\n';
        cout << "Path: ";
        for (int x : solver.answerPath) cout << x << ' ';
        cout << '\n';
    }

    return 0;
}
Code walkthrough — what each part teaches click to expand

dls(...) — Depth-Limited Search: Behaves like DFS but refuses to go below the remaining depth budget. If depth == 0, stop expanding.

Outer loop over limit: for (int limit = 0; limit ≤ maxDepth; limit++) — this is the "iterative deepening" part. Try depth 0, then 1, then 2, …

Why first found = optimal: Because we try limits in increasing order, the first successful limit is the minimum depth at which the target is reachable. Same shallowest-depth guarantee as BFS.

Note for normal graphs: For a normal explicit unweighted graph, BFS is usually better for shortest path. IDDFS shines when: the graph is not explicitly stored, states are generated dynamically, memory matters, or the search tree is huge.

Generic Implicit-State Template

For OJ problems where the graph is implicit (string transformations, move puzzles, board states):

#include <bits/stdc++.h>
using namespace std;

string target;
vector<string> curPath, answerPath;

// Replace this with your problem's move generator
vector<string> nextStates(const string& s) {
    vector<string> res;
    string t = s;

    // Example move 1: rotate left
    rotate(t.begin(), t.begin() + 1, t.end());
    res.push_back(t);

    // Example move 2: reverse whole string
    t = s;
    reverse(t.begin(), t.end());
    res.push_back(t);

    return res;
}

bool dls(const string& state, int depth, unordered_set<string>& onPath) {
    curPath.push_back(state);

    if (state == target) {
        answerPath = curPath;
        return true;
    }

    if (depth == 0) {
        curPath.pop_back();
        return false;
    }

    onPath.insert(state);

    for (const string& nxt : nextStates(state)) {
        if (!onPath.count(nxt)) {
            if (dls(nxt, depth - 1, onPath))
                return true;
        }
    }

    onPath.erase(state);
    curPath.pop_back();
    return false;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    string start;
    int maxDepth;
    cin >> start >> target >> maxDepth;

    for (int limit = 0; limit <= maxDepth; limit++) {
        unordered_set<string> onPath;
        curPath.clear();
        answerPath.clear();

        if (dls(start, limit, onPath)) {
            cout << "Found at depth " << limit << '\n';
            for (auto& s : answerPath) cout << s << '\n';
            return 0;
        }
    }

    cout << "No solution within depth limit\n";
    return 0;
}

Formal Condition

IDDFS seems wasteful because it re-explores the top of the tree on every iteration. But the math shows this cost is dominated by the final pass. In a tree with branching factor b and solution at depth d, each level i has bⁱ nodes. The last level has bd nodes. The sum of all previous levels is bd−1 + bd−2 + … + 1 = (bd − 1)/(b − 1), which is at most bd/(b−1). For b = 2 the total overhead from repeated work is at most equal to the final level — so total work is at most 2× compared to doing each level only once. For larger b the overhead is even smaller proportionally.

Total nodes explored ≤ bd · (b / (b−1))  ≈  O(bd) for b ≥ 2
✅

When IDDFS is the right choice: The graph is not stored explicitly (states are generated on the fly), the branching factor is large, the solution depth is not known in advance, and memory is the binding constraint. If memory allows BFS, prefer BFS — it has lower constant factors and no repeated work.

⚠

Common Pitfall: Using IDDFS when the problem fits normal BFS and memory is not constrained. IDDFS has higher constant factors than BFS. Also, forgetting to track "on-path" visited nodes (to avoid cycles) while allowing the same node to be revisited across different depth limits — you must distinguish between within-path cycle prevention and across-iteration revisiting.

7. Bidirectional Search

🎯

Why this section exists: When both source and target are known and the graph is large, one-sided BFS explores O(bd) nodes. Bidirectional BFS reduces this to O(bd/2) — an exponential improvement. Recognizing "both endpoints known, large branching factor, unweighted" is the trigger.

⚠

Common Wrong Model: Stopping the moment any node appears in both visited sets. The first meeting point is not necessarily on a shortest path. You must continue both frontiers until neither can produce a shorter candidate path.

Bidirectional BFS runs two simultaneous BFS sweeps — one forward from the source S, one backward from the target T — and stops the moment their frontiers meet in the middle.

Formal Condition

Search from S and T simultaneously → stop when frontiers overlap
💡

Why it is faster: Normal BFS from S explores O(bd) nodes (branching factor b, depth d). Bidirectional BFS explores only O(bd/2) nodes from each side — an exponential saving when d is large.

Mental Model

Imagine two search parties looking for each other in a large forest: one sets out from town S, the other from town T. Both walk toward each other and meet somewhere in the middle. Each party explores roughly half the forest depth, and since the number of reachable nodes grows exponentially with depth, halving the depth cuts the total nodes explored from bd to about 2 × bd/2 — an enormous saving for large d.

Algorithm / Method

#include <bits/stdc++.h>
using namespace std;

// Bidirectional BFS on an undirected unweighted graph.
// Returns shortest distance from src to dst, or -1 if unreachable.
// Sets pathOut to one shortest path.
int bidirBFS(int n,
             const vector<vector<int>>& adj,
             int src, int dst,
             vector<int>& pathOut)
{
    if (src == dst) { pathOut = {src}; return 0; }

    // dist[0] = dist from src,  dist[1] = dist from dst
    vector<int> dist[2];
    dist[0].assign(n, -1);
    dist[1].assign(n, -1);

    // parent arrays for path reconstruction
    vector<int> par[2];
    par[0].assign(n, -1);
    par[1].assign(n, -1);

    queue<int> q[2];
    dist[0][src] = 0; q[0].push(src);
    dist[1][dst] = 0; q[1].push(dst);

    int meeting = -1;
    int best    = INT_MAX;

    auto expand = [&](int side) {
        if (q[side].empty()) return;
        int u = q[side].front(); q[side].pop();
        int other = 1 - side;

        for (int v : adj[u]) {
            if (dist[side][v] != -1) continue;   // already visited this side
            dist[side][v] = dist[side][u] + 1;
            par[side][v]  = u;
            q[side].push(v);

            // check if the other side already reached v
            if (dist[other][v] != -1) {
                int cand = dist[side][v] + dist[other][v];
                if (cand < best) { best = cand; meeting = v; }
            }
        }
    };

    while (!q[0].empty() || !q[1].empty()) {
        expand(0);   // one step forward
        expand(1);   // one step backward

        // early exit: once both frontiers have advanced past best/2 we're done
        if (meeting != -1) {
            int fwdDepth = q[0].empty() ? INT_MAX : dist[0][q[0].front()];
            int bwdDepth = q[1].empty() ? INT_MAX : dist[1][q[1].front()];
            if (fwdDepth + bwdDepth >= best) break;
        }
    }

    if (meeting == -1) return -1;   // unreachable

    // reconstruct: src → ... → meeting → ... → dst
    vector<int> fwd, bwd;
    for (int v = meeting; v != -1; v = par[0][v]) fwd.push_back(v);
    for (int v = par[1][meeting]; v != -1; v = par[1][v]) bwd.push_back(v);
    reverse(fwd.begin(), fwd.end());
    pathOut = fwd;
    pathOut.insert(pathOut.end(), bwd.begin(), bwd.end());
    return best;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n, m; cin >> n >> m;
    vector<vector<int>> adj(n);
    for (int i = 0; i < m; i++) {
        int u, v; cin >> u >> v;
        adj[u].push_back(v);
        adj[v].push_back(u);
    }

    int src, dst; cin >> src >> dst;

    vector<int> path;
    int dist = bidirBFS(n, adj, src, dst, path);

    if (dist == -1) {
        cout << "No path\n";
    } else {
        cout << "Distance = " << dist << "\nPath: ";
        for (int v : path) cout << v << ' ';
        cout << '\n';
    }
}
Code walkthrough — key design decisions click to expand

Two dist arrays + two queues: Each side tracks its own distances and parent pointers independently. This makes path reconstruction clean.

Meeting detection: When side 0 visits node v and dist[1][v] is already known (not -1), we have a candidate path of length dist[0][v] + dist[1][v]. We keep the minimum.

Early exit condition: Once the sum of the two frontier depths exceeds best, no shorter path can exist. This is the key correctness guard — without it, you might miss shorter paths found slightly later.

Path reconstruction: Walk backwards from meeting using par[0] to get the forward half, and using par[1] to get the backward half. Reverse the forward half and concatenate.

Note on correctness: The simple "stop immediately when frontiers touch" approach is wrong in general — the first meeting point may not be on the shortest path. The early-exit guard above handles this correctly.

What It Is Not

Bidirectional BFS is not simply "run BFS twice and combine results." The two frontiers must be coordinated: you must keep tracking all meeting candidates and only stop when both frontiers are deeper than the current best path length. Stopping at the first frontier overlap gives wrong answers. Also, the speedup is not a constant 2× — it is exponential in d, so the deeper the solution, the greater the benefit; for very shallow solutions (d ≤ 2) the overhead may not be worth it.

⚠

Common Pitfall: Stopping the moment any node appears in both visited sets does not always give the shortest path. You must check all candidate meeting nodes until both frontiers are deeper than best/2. This is the most common bug in bidirectional BFS implementations.

What Graph Misc Really Means

After studying all seven topics, the right perspective is this: these are graph problems where the hard part is often not the final traversal algorithm, but discovering the right graph representation.

💡

The governing idea: In standard graph problems you are handed a graph and apply an algorithm. In misc graph problems you are handed a story and your first task is always to extract or construct the graph. The algorithm comes second — and once you have the right model, it becomes obvious.

📌 Graph Classification

Path graph and Planar graph: recognizing what kind of graph you have. Does it belong to a structural family that enables simpler reasoning or faster algorithms?

  • Path graph → chain, exactly 2 degree-1 vertices
  • Planar graph → embeddable without crossings, E = O(V)

📌 Graph Transformation

Line graph: converting one graph problem into another. When objects are edges, make each edge a node in a new graph.

  • L(G): edge adjacency = vertex adjacency
  • Rule: if problem objects are edges, consider L(G)

📌 Existence Problems

Hamiltonian: deciding whether a path of a specific type exists. NP-complete — requires exponential search or special structure.

  • Hamiltonian path → every vertex once, no simple test
  • Bitmask DP (n ≤ 20) is the standard CP approach

📌 Structural Measurement

Tree diameter: measuring a specific property of a structure in linear time via the two-BFS argument.

  • Two BFS passes from any starting vertex
  • Only correct for trees — not cyclic graphs

📌 Search on Implicit Graphs

IDDFS and Bidirectional BFS: searching when the graph is huge or implicit.

  • IDDFS → BFS depth guarantee + DFS memory
  • Bidirectional BFS → bd → bd/2 exponential saving
