Union-Find data structure — Essential for graph connectivity, MSTs, offline queries, and more.

What is DSU?

DSU (Disjoint Set Union), also called Union-Find, is a data structure that tracks a partition of a set into disjoint (non-overlapping) subsets. It supports two core operations in near-constant amortized time.

find(u)

Returns the representative (root) of the component containing node u. With path compression, all nodes on the path are directly attached to the root — future calls are O(1).

join(u, v)

Merges the components of u and v. With union by size/rank, the smaller tree always goes under the larger, keeping tree height at O(log n).

When to use DSU

Graph connectivity MST (Kruskal's) Cycle detection (undirected) Component counting Offline LCA Bipartite checking Relative weight queries Dynamic graph (use Rollback DSU)

Key idea: one array, two meanings — par[u] < 0 means root

We store both parent pointers and component sizes in a single par[] array. The sign is the switch:

par[u] < 0 → u is a ROOT
Size of u's component = −par[u]
e.g. par[u] = −3 → root of 3-node component
par[u] ≥ 0 → u has a parent
Parent of u = par[u]
e.g. par[u] = 2 → u's parent is node 2

Init with memset(par, -1, n * sizeof par[0]) sets every entry to −1, meaning every node is its own root with size 1. Perfect.

// par[u] < 0  →  root. Component size = -par[u]
// par[u] ≥ 0  →  non-root. Parent = par[u]

int find(int u){
    return par[u] < 0 ? u : par[u] = find(par[u]);
    //     ^^^^^^^^^^
    //     negative par means u IS the root
}

The par[] Trick — One Array, Two Jobs

This is the key idea that makes the DSU template compact. Instead of two arrays (par[] for parents, sz[] for sizes), we use one array with a sign convention. The sign tells you which job the number is doing right now.

What happens during join(u, v)?

Step 1: find roots
Walk up from u and v until you hit a root (par < 0). These are the two "captains".
int find(int u){
  return par[u]<0 ? u : par[u]=find(par[u]);
}
Step 2: merge
Bigger team absorbs smaller. Two par[] entries change: the new root's size grows (more negative), the old root points to the new root (becomes non-negative).
par[big] += par[small]; // −3 + −2 = −5
par[small] = big;       // was −2, now 0,1,2...

Simple DSU — Minimal Struct

The bare-bones version: just par[], ncmp, and the three core operations. No linked lists, no root-set tracking — use this when you only need connectivity queries and don't need to iterate over components.

Simple Template Minimal DSU — no extras 
struct DSU {
    // par[u] < 0  →  root, component size = -par[u]
    // par[u] ≥ 0  →  non-root, parent = par[u]
    int par[N], ncmp;

    void init(int n) {
        memset(par, -1, n * sizeof par[0]); // -1 = root of size 1
        ncmp = n;
    }

    int find(int u) {
        return par[u] < 0 ? u : par[u] = find(par[u]); // path compression
    }

    // s = small, b = big (by convention — swap if inconsistent).
    // Returns false if s and b are already in the same component.
    bool join(int s, int b) {
        s = find(s); b = find(b);
        if (s == b) return false;
        if (par[b] > par[s]) swap(s, b); // par more negative = bigger; ensure b is bigger
        par[b] += par[s];                  // merge sizes (both negative → sum more negative)
        par[s] = b;                        // s becomes child of b
        ncmp--;
        return true;
    }
};
When to use: Kruskal's MST, cycle detection, offline connectivity — any situation where you only call find() and join() and never need to walk a component's members. Upgrade to the Basic DSU below when you need component iteration.

Basic DSU — Union by Size + Path Compression

The standard template combines two optimizations: Union by size keeps the tree shallow by always attaching the smaller set under the larger, and path compression flattens the tree during find. Together they give O(α(n)) amortized per operation.

Full Template Coach Academy standard DSU 
struct DSU {
    // par[u] < 0  →  root, component size = -par[u]
    // par[u] ≥ 0  →  non-root, parent = par[u]
    int par[N], ncmp, sets[N], pos[N];
    int tail[N], nxt[N];

    void init(int n) {
        memset(par, -1, n * sizeof par[0]); // -1 = root of size 1 (-(-1)=1)
        iota(sets, sets + n, 0);  // active root list: sets[0..ncmp-1]
        iota(pos, pos + n, 0);    // pos[u] = index of u in sets[]
        iota(tail, tail + n, 0);  // tail[u] = last node in u's linked list
        ncmp = n;
        memset(nxt, -1, n * sizeof nxt[0]);
    }

    int find(int u) {
        return par[u] < 0 ? u : par[u] = find(par[u]); // path compression
    }

    // s = small, b = big (by convention — swap if inconsistent).
    // Returns false if s and b are already in the same component.
    bool join(int s, int b) {
        s = find(s); b = find(b);
        if (s == b) return false;
        if (par[b] > par[s]) swap(s, b); // par more negative = bigger; ensure b is bigger
        par[b] += par[s];                  // merge sizes: both negative, sum = larger size
        par[s] = b;                        // s becomes child of b
        int &t = tail[b];
        nxt[t] = s;  t = tail[s];
        int p = pos[s];
        sets[p] = sets[--ncmp];
        pos[sets[p]] = p;
        return true;
    }

    // Returns all nodes in the component containing u.
    vector<int> getCmp(int u) {
        u = find(u);
        vector<int> ret;
        ret.reserve(-par[u]); // preallocate with component size
        for (int v = u; v != -1; v = nxt[v])
            ret.push_back(v);
        return ret;
    }

    // Returns all components as a list of node lists.
    vector<vector<int>> getCmps() {
        vector<vector<int>> ret;
        ret.reserve(ncmp); // preallocate with number of components
        for (int i = 0; i < ncmp; i++)
            ret.push_back(getCmp(sets[i]));
        return ret;
    }
};
par[b] += par[s]: since both are negative (both roots), adding them gives a more-negative value — i.e. a larger size. No separate sz[] array needed. sets[] and pos[]: iterate over all roots in O(ncmp). tail[] and nxt[]: iterate over all nodes in a component in O(sz).

Rollback DSU

This template supports path compression AND rollback simultaneously — normally considered impossible. The key insight: operator[] (find) pushes every par[] write — including compression rewrites — onto the same ops stack. Rollback then undoes compressions too, not just union operations. curTtoSs[] maps timestamps to stack checkpoints so rollUntil(t) can snap back to any point in O(log n) per rolled-back op.

struct DSU
{
    // par[u] < 0  →  root, size = -par[u]
    // par[u] ≥ 0  →  non-root, parent = par[u]
    int par[N], curT, curTtoSs[M];

    void init(int n)
    {
        curT = 0;
        memset(par, -1, n * sizeof par[0]); // -1 = root of size 1
    }

    // Each op saves {node, its old par, other node or -1, its old par}
    struct operation { int a, parA, b, parB; };
    stack<operation> ops;

    int operator[](int u)
    {
        if(par[u] < 0) return u;            // root check
        ops.push({u, par[u], -1, 0});        // compression: only par[u] changes
        return par[u] = (*this)[par[u]];
    }

    bool operator()(int a, int b)
    {
        curTtoSs[curT++] = ops.size();
        a = (*this)[a];
        b = (*this)[b];
        if(a == b) return 0;
        if(par[a] > par[b]) swap(a, b);     // a has bigger size (more negative)
        ops.push({b, par[b], a, par[a]});    // save BOTH roots (both par values change)
        par[a] += par[b];                    // merge sizes (both negative → sum more negative)
        par[b] = a;                          // b becomes child of a
        return 1;
    }

    void rollBack()
    {
        auto &op = ops.top();
        par[op.a] = op.parA;
        if(~op.b) par[op.b] = op.parB;      // restore other root's par (size) if union op
        ops.pop();
    }

    void rollUntil(int t)
    {
        while(ops.size() > curTtoSs[t + 1])
            rollBack();
        curT = t + 1;
    }
} dsu;

How path compression + rollback coexist

Each operator[](u) compression pushes {u, old_par[u], -1, 0}. The b==-1 sentinel tells rollBack() this was a compression — only restore par[op.a]. Union ops push {b, old_par[b], a, old_par[a]} saving both roots' par values (since both change: par[a] += par[b] and par[b] = a).

curT and curTtoSs[]

curTtoSs[t] = stack size just before the t-th operator() call. rollUntil(t) pops until the stack returns to that size, then sets curT = t+1. This lets you snap to any timestamp in the history.

Common Use: D&C on time

Build a segment tree over time. For each tree node, join edges active during that interval, recurse into children, then rollUntil back. O(m log² n) for m offline updates.

Path compression ✓ Rollback ✓ O(α(n)) amortized find O(1) rollback per op rollUntil(t) in O(ops popped)

DSU with Parity

Parity DSU maintains a dist[] array where dist[u] is the XOR-parity of the path from u to its root. This lets you answer: "are u and v in the same group / different groups?" and is the standard tool for bipartite graph checking.

struct ParityDSU {
    int par[N], dist[N]; // dist[u] = XOR parity to root

    void init(int n){
        fill(par, par+n, -1);
        fill(dist, dist+n, 0);
    }

    // Returns {root, distance_to_root}
    pair<int,int> find(int u){
        if(par[u] < 0) return {u, 0};  // par[u]<0 means u is root
        auto [root, d] = find(par[u]);
        dist[u] ^= dist[par[u]]; // path compression + parity
        par[u] = root;
        return {root, dist[u]};
    }

    // p=0: u and v same group, p=1: different group
    // Returns: -1=conflict, 0=new edge ok, 1=redundant ok
    int addEdge(int u, int v, int p){
        auto [ru, du] = find(u);
        auto [rv, dv] = find(v);
        if(ru == rv){
            return (du ^ dv) == p ? 1 : -1;
        }
        if(par[ru] > par[rv]){
            swap(ru, rv);
            swap(du, dv);
        }
        par[ru] += par[rv];
        par[rv] = ru;
        dist[rv] = du ^ dv ^ p;
        return 0;
    }
};

How dist[] works

dist[u] stores the XOR-parity of edges on the path from u to its root. When two nodes in the same component are compared:

// u and v are in same component:
// effective parity between them = dist[u] ^ dist[v]
// If == 0: same group
// If == 1: different group

// Adding edge (u,v,p=1) means "u and v different group"
// Check: dist[u] ^ dist[v] should equal 1

DSU with Weights

A generalization of parity DSU. Each node stores a weight relative to its root. When merging components you can enforce constraints like "w[v] − w[u] = d" and then query the relative weight between any two nodes in the same component.

struct WeightedDSU {
    int par[N];
    long long w[N]; // w[u] = value[u] - value[root]

    void init(int n){
        fill(par, par+n, -1);
        fill(w, w+n, 0LL);
    }

    pair<int, long long> find(int u){
        if(par[u] < 0) return {u, 0};  // par[u]<0 means u is root
        auto [root, wr] = find(par[u]);
        par[u] = root;
        w[u] += wr; // path compression: accumulate weight
        return {root, w[u]};
    }

    // Constraint: value[v] - value[u] = d
    // Returns false if contradiction
    bool join(int u, int v, long long d){
        auto [ru, wu] = find(u); // wu = value[u] - value[ru]
        auto [rv, wv] = find(v); // wv = value[v] - value[rv]
        if(ru == rv){
            return (wv - wu) == d;
        }
        if(par[ru] > par[rv]){
            swap(ru, rv);
            swap(wu, wv);
            d = -d;
        }
        par[ru] += par[rv];
        par[rv] = ru;
        w[rv] = wu - wv + d;
        return true;
    }

    // Query: value[v] - value[u] (must be in same component)
    long long query(int u, int v){
        auto [ru, wu] = find(u);
        auto [rv, wv] = find(v);
        if(ru != rv) return LLONG_MIN; // not connected
        return wv - wu;
    }
};

How it works

w[u] stores value[u] − value[root]. During path compression, accumulated weights along the path are summed. After compression, w[u] directly gives the offset from root.

Example

// Answer: 3 + 5 = 8

Common Mistakes

Forgetting to call find() before comparing
Never compare par[u]==par[v]. Always find(u)==find(v).
Rollback DSU with path compression
Path compression and rollback are both supported — every par[] write (including compression rewrites) is pushed to the ops stack and can be undone. This gives O(α(n)) amortized find while retaining full rollback.
Resetting only par[], not sz[]/dist[]/w[]
Always reset ALL arrays in init(). Partial resets cause subtle bugs.
Applying DSU to directed graphs
Basic DSU handles undirected connectivity only. Directed graphs need SCCs.

DSU Complexity — A Complete Analysis

This section goes deep. We cover everything from the naive worst case to the inverse Ackermann bound, tie every result to the actual code, list pitfalls, connect to the DSU variant family, and give competitive-programming applications with links. Beginner-friendly presentation, but nothing important is omitted.

1 · Problem Definition What are we analysing?

We have n elements, initially each in its own singleton set. Two operations are performed repeatedly: find(u), which returns the canonical representative (root) of the set containing u, and join(s,b) (also called union), which merges the sets of s and b into one. We want to answer the question: if we perform a total of m operations (a mix of finds and joins) on n elements, what is the total cost?

Without any optimisation, a single find can cost Θ(n) in the worst case, making a sequence of m operations cost Θ(mn). For n and m up to 106, that is 1012 — hopelessly slow in competitive programming. The whole point of DSU optimisations is to bring this down to nearly Θ(m), making each operation essentially constant.

Why does this matter in CP? DSU is the backbone of Kruskal's algorithm, offline connectivity, component tracking, bipartite checking, and many more patterns. If your DSU is slow, your entire solution is slow. Understanding the complexity tells you why the optimisations work and when they apply.

2 · Intuition from Scratch Parent trees, what goes wrong, what fixes it

Parent Trees

DSU represents each set as a rooted tree. Every node stores a single value: its parent. The root points to itself (or, in the negative-size encoding, stores a negative number). To find the representative of a node, you walk up parent pointers until you hit the root.

What Goes Wrong: Naive DSU

With no optimisation, join simply makes one root the child of the other — say, we always attach the second argument under the first. The resulting tree can degenerate into a chain (a linked list). If you join elements 0→1→2→…→(n−1) in sequence, you get a path of depth n−1. Now find(0) must walk n−1 edges. Do that m times and you pay Θ(mn).

Why Union by Size (or Rank) Helps

The key insight: tall trees happen when you attach a big tree under a small tree (or two comparably sized trees in an unlucky order). Union by size always attaches the smaller tree under the root of the larger. This ensures that every time a node's depth increases by 1, the component it belongs to at least doubles in size. So the maximum depth is at most log2(n). That alone reduces each find from O(n) to O(log n), bringing m operations to O(m log n).

Why Path Compression Helps

During find(u), we walk a path from u up to the root r. Path compression rewrites every node along that path to point directly to r. Future finds for those nodes will be O(1). The more often you call find, the flatter the tree gets. Path compression alone (without union by size) gives O(m log n) amortised — the same asymptotic bound as union by size alone, though achieved through a very different mechanism.

Why the Combination Is Extraordinary

When you use both union by size and path compression, the amortised cost per operation drops to O(α(n)), where α is the inverse Ackermann function. This is not O(log n) or O(log log n) — it is so slow-growing that for any n that could ever fit in a computer's memory, α(n) ≤ 4. Effectively constant, but provably not literally O(1).

4 · Formal Abstraction Operations, cost model, what we bound

Operations

Make-Set(x): create a new singleton set {x}. In our code this is init(n) — O(n) total for n elements. Find(x): return the representative of the set containing x. Union(x,y): merge the sets containing x and y (calls Find internally). Our code calls this join(s,b).

Cost Model

We measure cost as the number of parent pointer traversals (edges walked) plus the number of parent pointer writes (for path compression). Each edge traversal and each pointer write counts as O(1) work. The total cost T(m,n) is the sum over all operations.

What We Are Bounding

We bound T(m,n) for a sequence of m operations on n elements. The central result is: T(m,n) ≤ O(m · α(n)) when using union by size/rank with path compression. This is an amortised bound — it says the average cost per operation, spread over the whole sequence, is O(α(n)). Individual operations may occasionally cost O(log n), but they "pay for" future cheap operations.

Worst-Case vs Amortised vs Practical

Worst-case single operation: O(log n) with union by size + path compression. A single find can still traverse a path of logarithmic depth; compression only helps future operations. Amortised per operation: O(α(n)). This is the theorem. In practice (CP context): treat it as O(1). With n ≤ 106, α(n) ≤ 4. The constant factor in DSU is tiny — typically faster than a single hash-map lookup.

5 · Key Observations The facts that make the proof work

Observation 1 — Naive chain is the worst case. Without optimisation, adversarial join sequences produce a linked list of depth n−1. This makes the worst-case find O(n) and a sequence of m finds O(mn).

Observation 2 — Union by size limits rank growth. If we always attach the smaller tree under the larger, a tree of rank (depth) k must contain at least 2k nodes. Therefore rank ≤ log2(n). This alone gives O(m log n) for m operations.

Observation 3 — Path compression flattens structure. Every find from node u to root r rewrites every intermediate node to point directly to r. This destroys the tree depth along that path. Future finds become shorter. Path compression alone (without union by size) also gives O(m log n) amortised, but via a different argument.

Observation 4 — The combination is super-logarithmic. When both optimisations act together, union by size keeps the tree from ever getting deeper than O(log n), and path compression continuously flattens even those shallow trees. The result is an amortised cost of O(m · α(n)), where α grows slower than any fixed number of iterated logarithms. This is Tarjan's 1975 result, later shown to be tight by Fredman and Saks (1989): you cannot do better than Ω(m · α(n)) in the pointer-machine model.

Practical meaning of α(n): α(n) ≤ 4 for all n ≤ 265536. Since 265536 is a number with about 20,000 digits — far beyond any conceivable input size — you will never encounter α(n) ≥ 5 in practice. In competitive programming, treating it as O(1) is entirely justified for runtime analysis, though not for formal proofs.

7 · The Ackermann and Inverse Ackermann Functions Gentle but complete

Intuition First

Think of a ladder of increasingly violent operations: addition, multiplication, exponentiation, tetration (iterated exponentiation), pentation (iterated tetration), and so on. Each step produces numbers that are mind-bogglingly larger than the previous level. The Ackermann function A(k, n) climbs this ladder: roughly, A(0, n) ≈ n+1, A(1, n) ≈ 2n, A(2, n) ≈ 2n, A(3, n) ≈ 2↑↑n (tower of 2s of height n), A(4, n) ≈ 2↑↑↑n, and so on.

The inverse Ackermann function α(n) asks: "what is the smallest level k such that A(k, k) ≥ n?" Since A grows unimaginably fast, α grows almost imperceptibly slowly. Even α(2↑↑↑↑16) is just 5.

More Formal Definition

There are several slightly different definitions in the literature. A commonly used one for DSU analysis defines a sequence of functions: A0(n) = n + 1 for the base case, and Ak+1(n) = Ak applied n times starting from n (i.e., Ak(n)(n)). Then α(n) = min{k ≥ 0 : Ak(1) ≥ n}.

Some concrete values: A0(1) = 2, A1(1) = 3, A2(1) = 7, A3(1) = 2047, A4(1) is a tower of 2s of height 2047 — a number with roughly 10616 digits. So α(n) = 0 for n ≤ 2, α(n) = 1 for n = 3, α(n) = 2 for n ≤ 7, α(n) = 3 for n ≤ 2047, α(n) = 4 for all n up to that unimaginably large tower.

What the Formal Theorem Says

Tarjan (1975), refined by Tarjan and van Leeuwen (1984), proved: a sequence of m Make-Set, Union, and Find operations, including at most n Make-Set operations, takes O(m · α(n)) time when using union by rank (or size) with path compression. Fredman and Saks (1989) proved a matching lower bound Ω(m · α(n)) in the cell-probe model, showing that no pointer-based union-find implementation can do asymptotically better.

This section presents an intuition-focused explanation. A full formal proof uses a rank-decomposition argument and a potential function based on iterated-logarithm blocks. The details fill several pages of Tarjan's original paper. What matters for CP is: the bound is real, it's tight, and it's effectively constant.

    int tail[N], nxt[N];

    void init(int n) {
        // O(n) total.  Every element is a root of size 1.
        // par[u] = -1 means: root, size = -(-1) = 1.
        memset(par, -1, n * sizeof par[0]);
        iota(sets, sets + n, 0);   // sets[0..ncmp-1] = active roots
        iota(pos, pos + n, 0);     // pos[u] = index of u in sets[]
        iota(tail, tail + n, 0);   // tail[u] = last node in u's list
        ncmp = n;
        memset(nxt, -1, n * sizeof nxt[0]);
    }

    int find(int u) {
        // PATH COMPRESSION — this is where α(n) comes from.
        // Base case: par[u] < 0 means u is a root → return u.
        // Recursive case: find the root, then set par[u] = root.
        // After this call, u and every ancestor up to the root
        // all point directly to the root → future finds are O(1).
        //
        // Worst-case single call: O(log n) — the tree depth with
        // union by size.  Amortised over m calls: O(α(n)) each.
        return par[u] < 0 ? u : par[u] = find(par[u]);
    }

    bool join(int s, int b) {
        s = find(s); b = find(b);       // O(α(n)) amortised each
        if (s == b) return false;        // already same component

        // UNION BY SIZE:
        // par[s] and par[b] are both negative (both roots).
        // More negative = larger component.
        // "par[b] > par[s]" means |par[b]| < |par[s]|, i.e. b is smaller.
        // We want b to be the bigger root, so swap if needed.
        if (par[b] > par[s]) swap(s, b);

        // Merge sizes: par[b] += par[s].
        // Both negative → sum is more negative → larger size.
        // E.g. par[b]=-5, par[s]=-3 → par[b]=-8 (size 8).
        par[b] += par[s];

        // Attach smaller root s under bigger root b.
        // This is the union-by-size step that keeps depth ≤ log₂(n).
        par[s] = b;

        // Maintain linked-list and root tracking (O(1) bookkeeping):
        int &t = tail[b];
        nxt[t] = s;  t = tail[s];
        int p = pos[s];
        sets[p] = sets[--ncmp];
        pos[sets[p]] = p;
        return true;
    }
};

Why Each Line Matters for Complexity

The find function is a single line of code, but it does two things: it walks up to the root (paying for the current path length), and it rewrites every pointer along the way (paying for writes, but earning future savings). The assignment par[u] = find(par[u]) is the path compression — without it, find would still be O(log n) worst-case with union by size, but the amortised bound would not improve to α(n).

The comparison par[b] > par[s] in join is the union-by-size decision. Because sizes are stored as negatives, "greater" means "smaller component." This single comparison ensures that tree depth never exceeds log2(n). Without it, path compression alone would give O(m log n) total — still good, but not the near-constant α(n) bound.

The line par[b] += par[s] is the negative-size merge: adding two negative numbers gives a more-negative number, which encodes the combined size. This is the "one array, two meanings" trick that avoids a separate sz[] array.

10–11 · Edge Cases & Common Pitfalls Bugs that waste hours
0-indexed vs 1-indexed
If your problem uses 1-based nodes but you call init(n) instead of init(n+1), node n is uninitialised. Subtle: par[n] may be 0, which looks like "parent is node 0" instead of "root". Always match init(n) to your actual node range.
Negative-size encoding misunderstandings
par[u] = -1 means root of size 1, not "no parent." A common mistake is checking par[u] == -1 to detect roots; the correct check is par[u] < 0, because a component of size 5 has par[root] = -5. If you accidentally check == -1, you break as soon as any component grows beyond size 1.
Recursion depth in find
With path compression, the recursive find can hit the system stack limit on very deep initial trees (before compression kicks in). With union by size the depth is at most log₂(n) ≈ 20 for n = 106, so this is safe. But if you use path compression without union by size, the depth could be O(n) on the first call, causing stack overflow. In that case use an iterative find.
Rollback + path compression interaction
Classic wisdom says rollback DSU cannot use path compression, because compression changes par[] entries that were not part of the union, making undo impossible. However, the implementation in this tutorial pushes every par[] write — including compression writes — onto the undo stack. This supports both O(α(n)) amortised find and full rollback, at the cost of a larger undo stack (O(m log n) instead of O(m)).
Amortised ≠ worst-case
O(α(n)) is amortised. A single find can cost O(log n). If you need worst-case O(log n) per operation (e.g., for persistent DSU or in real-time systems), use union by rank without path compression. For CP, amortised is almost always fine — the total over all operations is guaranteed.
α(n) is not literally constant
In formal proofs and complexity arguments, do not write "DSU is O(1) per operation." It is O(α(n)) amortised. α(n) grows — just immeasurably slowly. The distinction matters in theory (Fredman–Saks lower bound), even though it never matters in practice.
Forgetting to initialise correctly
Always call init(n) before using the DSU, and reset all arrays: par[], nxt[], tail[], sets[], pos[], and ncmp. If you use the DSU in multiple test cases, forgetting to re-init is a classic bug. Also: for Parity and Weighted DSU, you must reset dist[]/w[] as well.
12 · Proof Sketch — Why O(α(n)) Correctness-level reasoning

Union by Size: Depth ≤ log₂(n)

Claim: if we always merge the smaller tree into the larger, then every tree of rank (maximum depth) k has at least 2k nodes.

Proof by induction: base case k = 0, tree has 1 = 20 nodes. For the inductive step: a tree of rank k is formed by merging two trees, at least one of which had rank k−1. Since we only increase the rank of the bigger tree, both subtrees have at least 2k−1 nodes, giving a total of at least 2k. So rank ≤ log₂(n).

Path Compression: Amortised via Potential Function

The standard proof assigns a potential to each node based on which "block" of the rank decomposition it belongs to. The blocks are defined using iterated logarithms (log*, log**, etc.), eventually reaching the inverse Ackermann function. Each find operation may pay O(log n) actual cost, but it decreases the potential by enough to compensate. Summed over m operations, the total potential decrease bounds the total cost at O(m · α(n)).

The full proof is in Tarjan (1975) and the textbook by Cormen, Leiserson, Rivest, and Stein (CLRS), Chapter 21. The key insight is: path compression causes nodes to "jump" to higher-rank ancestors, and each jump is paid for by a decrease in potential. Since the potential cannot decrease below zero, the total cost is bounded.

Lower Bound (Fredman–Saks)

The O(m · α(n)) bound is tight. Fredman and Saks proved in 1989 that any union-find data structure in the cell-probe model requires Ω(m · α(n)) time for some sequence of operations. You cannot beat α(n) amortised per operation — this is the best any pointer-based structure can do.

Key Caveats by Variant

Rollback DSU (classic): most implementations intentionally avoid path compression because undoing compression is complex. They rely solely on union by size, giving O(log n) per operation. This is still fast enough for most segment-tree-on-DSU patterns. However, the template in this tutorial demonstrates that compression + rollback can coexist by logging every par[] write.

Parity / Weighted DSU: the α(n) complexity still holds, but you must be careful to update dist[]/w[] correctly during path compression. If the compression step forgets to accumulate the parity or weight along the compressed path, the data becomes inconsistent even though the structural complexity is correct.

DSU on Tree: this is an entirely different technique — it refers to the small-to-large merging of data structures during DFS, not to the Union-Find data structure at all. The name is confusing. DSU on tree has O(n log n) complexity and has nothing to do with the α(n) bound.

15 · Advanced Extensions What comes next

Rollback DSU + Segment Tree of Time: for problems where edges appear and disappear, process queries offline. Build a segment tree over the timeline, assign each edge to the tree nodes covering its alive interval, then DFS the segment tree, using rollback DSU to add/remove edges at each node. Cost: O(n log² n) with union-by-size rollback DSU. With the compression-supporting rollback DSU from this tutorial, the per-find cost drops to amortised O(α(n)), though the undo stack is larger.

DSU with Potentials / Weighted Relations: generalise parity to an arbitrary group (e.g., integers mod k, or real-valued differences). Each edge carries a weight, and path compression accumulates weights along the path. The α(n) complexity still applies to the structural part; the extra cost is the weight computation per step, typically O(1).

Link-Cut Trees: if you need worst-case O(log n) per operation (not just amortised) or need to support cut operations, link-cut trees are the next step. They are significantly more complex to implement but give O(log n) worst-case for link, cut, and find-root.

"DSU on Tree" (Small-to-Large Merging): despite the name, this technique is not a union-find structure. It is a DFS-based strategy for merging child data into parent data, always merging the smaller set into the larger. Total cost O(n log n). The name creates confusion — if someone says "DSU on tree," check whether they mean the merging trick or an actual union-find on a tree structure.

When the Complexity Theorem Doesn't Apply: the O(m · α(n)) bound assumes you use standard find with path compression and standard union with size/rank. If you modify find to not compress (e.g., for persistence), or if you use path splitting/halving instead of full compression, the bound changes. Path splitting and path halving also achieve O(m · α(n)), but partial compression does not. If you use a persistent DSU (functional, immutable pointers), you lose compression entirely and get O(log n) per operation.

16 · Key Takeaways & Memory Checklist What you should now know

What You Should Now Understand

The Complexity Ladder
Naive = O(n) per find. Union by size OR path compression alone = O(log n). Both together = O(α(n)) amortised ≈ effectively O(1).
Why α(n) ≤ 4 in Practice
The inverse Ackermann function grows slower than any iterated logarithm. For n ≤ 265536, α(n) ≤ 4. You will never see α(n) = 5.
Amortised vs Worst-Case
O(α(n)) is amortised — the average over m operations. A single find can still cost O(log n). This almost never matters in CP.
The Code–Theory Connection
Path compression is the single line par[u] = find(par[u]). Union by size is the if (par[b] > par[s]) swap(s,b) comparison. Both are needed for α(n).

Memory Checklist for Contests

✓ DSU with union by size + path compression = O(α(n)) amortised per operation.
✓ α(n) ≤ 4 for all practical n. Treat as O(1) in runtime estimates.
✓ Single find can cost O(log n) in worst case — but total is guaranteed.
✓ Rollback DSU (classic, no compression) = O(log n) per find — still fast enough.
✓ This tutorial's rollback DSU supports compression + rollback = O(α(n)) amortised.
✓ Parity/Weighted DSU = same O(α(n)) — just maintain extra data during compression.
✓ "DSU on tree" ≠ union-find. It is small-to-large merging, O(n log n).
✓ Always init ALL arrays. Always match init(n) to your indexing (0-based vs 1-based).
✓ Check par[u] < 0 for root, not par[u] == -1.

Common Mistakes Summary

✗ Forgetting union by size → O(n) per find, TLE on large inputs.
✗ Forgetting path compression → O(log n) per find instead of O(α(n)), usually still AC but slower.
✗ Using par[u] == -1 instead of par[u] < 0 for root check → wrong for components of size > 1.
✗ Not re-initialising DSU between test cases → garbage from previous case.
✗ Using recursive find without union by size → stack overflow on chains.
✗ Claiming O(1) per operation in formal proofs → technically incorrect; it is O(α(n)).
✗ Using path compression with classic rollback → breaks unless you log ALL par[] writes.
