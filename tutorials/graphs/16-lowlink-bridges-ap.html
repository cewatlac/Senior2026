What You'll Learn

This tutorial builds the entire Low-Link DFS framework from first principles. No formula is memorized — every condition is derived from a single core idea. We go from the naive O(N·(N+M)) approach to a single O(N+M) DFS that answers all connectivity questions simultaneously.

Core Promise: One DFS. Every bridge, every articulation point, every BCC — all found in linear time using a single low-link invariant.

1. Problem Definition

The family of questions

Given an undirected connected graph, we want to identify its structural weak points — vertices or edges whose removal would break connectivity:

Articulation Point
A vertex u whose removal disconnects the graph. The graph loses connectivity because u was the only link between two parts.
Bridge
An edge (u,v) whose removal disconnects the graph. It is the sole path between the two sides it connects.
BCC / EBCC
Maximal subgraphs that are robust: vertex-biconnected (no articulation inside) or edge-biconnected (no bridge inside).

Why the naïve approach fails

❌ Naïve: O(N·(N+M))

For each of the N vertices: remove it, run DFS to check connectivity. With N = 200,000 and M = 400,000, that is 120 billion operations. Far too slow.

✅ Low-Link DFS: O(N+M)

During a single DFS pass, we accumulate enough information about escape routes to answer all questions simultaneously. No vertex or edge needs to actually be removed.

2. DFS Tree & Edge Types

When DFS runs on an undirected graph, it builds a DFS tree. Every edge of the original graph becomes one of exactly two types:

Tree Edge
The edge we follow when we discover a new, unvisited node. It becomes part of the DFS tree. Points from parent to child.
u → v   (v is unvisited)
Back Edge
An edge to an already-visited ancestor. In undirected DFS there are no forward or cross edges — only back edges.
u → ancestor   (ancestor is visited)
DFS Tree — Blue = tree edges, Orange dashes = back edges
DFS Root Inner nodes Leaf nodes Back edge
🔑

Key fact for undirected DFS: There are NO forward edges or cross edges. Only tree edges and back edges. This makes the math clean and powerful — back edges are the only way to "climb back up" in the tree.

Back edges are the structural "ropes" that prevent parts of the tree from being isolated. A subtree with a back edge can survive the removal of its parent-connecting tree edge; one without cannot.

3. The Escape Question — The Heart of Everything

🧠

This section is the most important in the entire tutorial. Once you deeply understand the following single question, every formula that follows will feel obvious rather than memorized.

Imagine you are executing DFS and you follow the tree edge from node u down to node v. You are now inside the subtree rooted at v. Ask yourself:

❓ Can the subtree of v reach an ancestor of u
without using the edge (u → v)?

The two outcomes of this question directly give us our algorithm:

✅ YES — Can escape

Some node in v's subtree has a back edge that jumps to an ancestor of u (or to u's parent or higher).

  • Edge (u,v) is NOT a bridge — there's an alternative route.
  • u is NOT an articulation point due to this child — v's subtree can survive without u.
❌ NO — Cannot escape

Every path from v's subtree back to the rest of the graph passes through the edge (u,v).

  • Edge (u,v) IS a bridge — removing it disconnects v's subtree.
  • u IS an articulation point — removing u cuts v's subtree off.
🔥

Everything else is just computing this efficiently. The variable low[v] quantifies exactly "how far up the tree can v's subtree reach." Once we can compute low[v] for every node, the entire algorithm follows in 2 lines.

Why back edges are the only "escape routes"

In an undirected DFS, the only edges that go "upward" in the tree are back edges. Tree edges always go downward (from parent to child in the DFS order). So "escaping above u" literally means having at least one back edge in v's subtree that reaches a node with smaller discovery time than u.

4. ord[] and low[] — Built from Scratch

We need a way to quantify the escape question for every edge in one DFS pass. We define two values per node:

ord[u] — Discovery Time

A global timer increments each time we visit a new node. ord[u] records when u was first visited.

ord[u] = ++timer     (assigned on first entry to u)

If ord[a] < ord[b], then a was visited before b. In the DFS tree, this means a is an ancestor of b (or they're in completely separate branches). The root has ord[root] = 1, the smallest possible.

low[u] — The Escape Reach

📐

Semantic definition: low[u] = the minimum ord[x] over all nodes x reachable from the subtree of u using any number of tree edges and at most one back edge.

In English: low[u] is the discovery time of the highest ancestor reachable from u's entire subtree. Smaller = higher = further up the tree = better escape.

low[u] < ord[parent(u)]

The subtree of u can escape above its parent. The connecting tree edge is not a bridge. The parent is not necessarily an articulation point because of this child.
low[u] ≥ ord[parent(u)]

The subtree of u is trapped at or below its parent. The connecting tree edge might be a bridge. The parent is an articulation point due to this child.

The initialization is: low[u] = ord[u] — at minimum, every node can "reach itself."

Why "at most one back edge" in the definition?

In an undirected DFS, once you use a back edge to jump upward, you've left the subtree. The definition of "reachable from subtree of u" allows you to walk down tree edges inside the subtree freely, then take one jump back up via a back edge. You can't then walk down tree edges again and use another back edge — that would require counting paths outside the subtree.

5. Computing low[] — The Two-Case Recurrence

During DFS from node u, we look at each neighbor v. We update low[u] depending on whether we're traversing a tree edge or a back edge:

Case 1: Tree Edge (v is unvisited)

We recurse into v and compute DFS(v). After DFS(v) returns, we know low[v]. Since u's subtree contains v's subtree, anything v can escape to, u can also escape to:

low[u] = min(low[u], low[v])

Case 2: Back Edge (v is already visited — ancestor)

The edge u→v is a back edge. We can jump directly to v from u, reaching an ancestor with discovery time ord[v]:

low[u] = min(low[u], ord[v])
⚠️

Why ord[v] and NOT low[v] for back edges? Using low[v] would mean "I can also escape wherever v can escape via its own back edges" — but that would require two back edges (one from u to v, then one from v's subtree upward). Our semantic allows at most one back edge. Using ord[v] (not low[v]) is correct and essential.

Case 3: The Parent Edge — Skip It!

In undirected graphs, every edge (u,v) is stored as two directed edges. When we process neighbors of v, we'll see the edge back to u. But u is v's parent — treating this as a "back edge" would be wrong! We skip it.

if (e == pe) continue;   // pe = parent edge index (not vertex!)
⚠️

Use edge index, NOT parent vertex. If there are two parallel edges between u and v, skipping by vertex would incorrectly skip both — making neither one count as a back edge. Skipping by the specific edge index pe only skips the one we came from. The reverse edge is pe ^ 1 in the forward-star layout.

Full recurrence summary

Init: low[u] = ord[u]
Tree: low[u] = min(low[u], low[v])
Back: low[u] = min(low[u], ord[v])

6. Bridge & Articulation Conditions — Derived

We now translate the escape question into precise mathematical conditions using low[v] and ord[u].

Bridge Condition

Tree edge (u,v) is a bridge if and only if v's subtree has no way to reach u or any ancestor of u. The best the subtree can reach is represented by low[v]. For the subtree to NOT escape above u, it needs low[v] to be higher than u's own discovery time:

low[v] > ord[u]  ⟹  edge (u,v) is a BRIDGE
Why strictly > and not ≥ for bridges? click to expand

If low[v] = ord[u], then v's subtree can reach exactly u via a back edge. Now remove the tree edge (u,v). The back edge still connects v's subtree to u! The graph remains connected. So low[v] = ord[u] is NOT a bridge.

Only when low[v] > ord[u] does v's subtree have absolutely no route to u or above, making (u,v) the only connection — a true bridge.

Summary: ≥ means "reaches u or above" (not a bridge). > means "can't even reach u" (IS a bridge).

Articulation Point Condition — Non-Root

For a non-root node u with child v in the DFS tree: if v's subtree cannot escape above u, then removing u would disconnect v's subtree from u's ancestors. The condition is:

low[v] ≥ ord[u]  ⟹  u is an ARTICULATION POINT
Why ≥ for articulation but > for bridges? click to expand

Bridge (edge removal): If low[v] = ord[u], there's a back edge from v's subtree to u. Remove the tree edge — the back edge still connects v to u. NOT a bridge. Need >.

Articulation (vertex removal): If low[v] = ord[u], v's subtree can only reach u itself — but u is being removed! After removal, v's subtree has no way to connect to u's ancestors. So u IS an articulation point. Need ≥.

The difference: a back edge to u helps when only the tree edge is removed (bridge check), but not when u itself is removed (articulation check).

Articulation Condition — Root (Special Case)

The root is special: it has no parent. The condition low[v] ≥ ord[root] would always be true (ord[root] = 1), so we can't use the same rule.

🧠

Intuition: If the root has only one DFS-tree child, removing it just "un-roots" the tree — everything stays connected. If the root has ≥ 2 DFS-tree children, those children form independent subtrees that were only connected through the root. Removing the root disconnects them.

Root is ARTICULATION POINT  ⟺  root has ≥ 2 DFS-tree children

7. Proof of Correctness

Proof 1: low[u] invariant is correctly maintained click to collapse

Claim: After DFS(u) completes, low[u] = min{ord[x] : x reachable from subtree of u via tree edges and ≤ 1 back edge}.

Base case (leaf node): A leaf has no children and no back edges (or its back edge goes to some ancestor). If it has a back edge to ancestor a, low[u] = min(ord[u], ord[a]). If no back edge, low[u] = ord[u]. The leaf's subtree is just itself plus any single back edge. ✓

Inductive step: Assume the invariant holds for all children c₁, c₂, …, cₖ of u. For each child cᵢ:

  • low[cᵢ] correctly holds the best escape from cᵢ's subtree via tree edges + ≤ 1 back edge.
  • Since u can reach cᵢ's subtree via tree edges, anything cᵢ can reach, u can also reach (by going into cᵢ's subtree first, then taking the back edge). So we take min with low[cᵢ].

For each direct back edge (u → ancestor a): we can reach a with one back edge. So we take min with ord[a].

After all updates, low[u] = min over all children and back edges = correct invariant value. ∎

Proof 2: Bridge condition is correct click to expand

Theorem: Tree edge (u,v) is a bridge if and only if low[v] > ord[u].

(⟹) If (u,v) is a bridge:

Assume (u,v) is a bridge. Then there is no simple path from v to u's ancestors that doesn't use (u,v). In particular, no back edge from v's subtree goes to u or above u. So low[v] ≥ ord[u]. If low[v] = ord[u], there'd be a back edge from v's subtree to u — but then removing (u,v) still leaves v connected to u via that back edge, contradicting that (u,v) is a bridge. Hence low[v] > ord[u]. ✓

(⟸) If low[v] > ord[u]:

No node in v's subtree can reach u or any ancestor of u. Every path from any node in v's subtree to any node outside v's subtree must pass through the tree edge (u,v) going upward. Removing (u,v) disconnects v's subtree. So (u,v) is a bridge. ✓ ∎

Proof 3: Articulation condition is correct click to expand

Theorem (non-root): Non-root u is an articulation point iff it has some child v with low[v] ≥ ord[u].

(⟹) If u is an articulation point:

Removing u disconnects the graph. Some component must be a subtree of u in the DFS tree (since all non-tree edges from inside the subtree go to ancestors, not to other disconnected components). Let v be the root of that subtree. Its subtree has no path to u's ancestors without using u, hence no back edge above u. So low[v] ≥ ord[u]. ✓

(⟸) If low[v] ≥ ord[u] for some child v:

Every path from v's subtree to u's ancestors passes through u. When u is removed, v's subtree loses all access to u's ancestors. u IS an articulation point. ✓ ∎

Root case: If root has ≥ 2 DFS-tree children c₁, c₂: after DFS, every edge from c₁'s subtree that goes "upward" can only reach root at best (since root has no ancestors). When root is removed, c₁'s subtree and c₂'s subtree become disconnected. Conversely if root has 1 child, removing root leaves 1 component still connected. ∎

Stack depth and iterative DFS note click to expand

Recursive DFS can stack overflow on chains with N = 200,000+ nodes. The recursion depth is O(N) in the worst case (a path graph). In contest settings you can increase the stack size with pragmas or convert to an iterative DFS using an explicit stack. For most competitive programming judges, recursive DFS with N ≤ 200,000 is accepted.

For problems with N up to 10^6 or on systems with limited stack, prefer iterative DFS with an explicit stack storing (node, edge_iterator, parent_edge) tuples.

10. Master Template — Full AC-Ready C++

✅

This template handles: articulation points, bridges, multiple test cases (versioned reset), multigraphs (edge-index parent tracking), disconnected graphs (DFS loop over all nodes).

#include <bits/stdc++.h>
using namespace std;

// ── Constants ────────────────────────────────────────────
const int MAXN = 200005;
const int MAXM = 400005;

// ── Forward-Star Adjacency List ──────────────────────────
int head[MAXN], to[MAXM], nxt[MAXM], ne;

// ── DFS State ────────────────────────────────────────────
int ord[MAXN], low[MAXN];    // discovery time, escape reach
int vis[MAXN], vid;           // versioned visitation
int timer_val;                // DFS clock

// ── Results ──────────────────────────────────────────────
bool isArt[MAXN];             // articulation point flag
bool isBridge[MAXM];          // bridge flag (per edge index)

int n, m;

// ── Graph Construction ───────────────────────────────────
void init(int _n, int _m) {
    n = _n; m = _m;
    fill(head,     head     + n + 1, -1);
    fill(isArt,    isArt    + n + 1, false);
    fill(isBridge, isBridge + 2*m + 5, false);
    ne        = 0;
    vid++;            // version bump: no need to clear vis[]
    timer_val = 0;
}

void addEdge(int u, int v) {
    to[ne] = v; nxt[ne] = head[u]; head[u] = ne++;
}

void addBiEdge(int u, int v) {
    addEdge(u, v);    // edge index:  2k     (even)
    addEdge(v, u);    // edge index:  2k+1   (odd = e^1)
}

// ── Core DFS ─────────────────────────────────────────────
void dfs(int u, int pe = -1) {
    vis[u] = vid;
    ord[u] = low[u] = ++timer_val;
    int children = 0;

    for (int e = head[u]; ~e; e = nxt[e]) {
        int v = to[e];

        if (e == pe) continue;          // skip the edge we came from

        if (vis[v] != vid) {
            // ── Tree Edge ─────────────────────────────
            children++;
            dfs(v, e ^ 1);              // pass reverse edge as parent edge

            low[u] = min(low[u], low[v]);

            // Bridge check
            if (low[v] > ord[u])
                isBridge[e] = isBridge[e ^ 1] = true;

            // Articulation check (non-root only)
            if (pe != -1 && low[v] >= ord[u])
                isArt[u] = true;

        } else {
            // ── Back Edge ─────────────────────────────
            low[u] = min(low[u], ord[v]);
        }
    }

    // Root articulation check
    if (pe == -1 && children > 1)
        isArt[u] = true;
}

// ── Entry Point ──────────────────────────────────────────
int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    cin >> n >> m;
    init(n, m);

    for (int i = 0; i < m; i++) {
        int u, v; cin >> u >> v;
        addBiEdge(u, v);
    }

    // Run DFS from every unvisited node (handles disconnected graphs)
    for (int i = 1; i <= n; i++)
        if (vis[i] != vid)
            dfs(i);

    // ── Output: articulation count ────────────────────────
    int artCnt = 0;
    for (int i = 1; i <= n; i++)
        if (isArt[i]) artCnt++;
    cout << artCnt << "\n";

    // ── Output: bridges ───────────────────────────────────
    for (int e = 0; e < ne; e += 2)
        if (isBridge[e])
            cout << to[e ^ 1] << " -- " << to[e] << "\n";

    return 0;
}

11. Line-by-Line Explanation

Forward-star adjacency: head / to / nxt / ne click to collapse

head[u] = index of the first edge starting at u (−1 if none). to[e] = destination of edge e. nxt[e] = index of the next edge from the same source (a linked list). ne = next available edge slot.

This layout is faster than vector<vector> because all data is in contiguous arrays. Cache-friendly for DFS traversal.

The e ^ 1 reverse-edge trick click to expand

When addBiEdge(u,v) is called, it adds two directed edges: edge at index 2k (u→v) and edge at index 2k+1 (v→u). Since 2k XOR 1 = 2k+1 and (2k+1) XOR 1 = 2k, the expression e ^ 1 always gives the reverse edge regardless of which direction e is.

Consequence: we start with ne = 0 and edges MUST be added in pairs via addBiEdge. Never add directed edges individually when using this trick.

vid versioning: why it beats clearing arrays click to expand

If we have T test cases, clearing vis[] costs O(N) per test case. With vid, we just increment an integer: vid++. A node u is "unvisited" if vis[u] != vid. This costs O(1) reset per test case. Over T test cases, this saves O(T·N) work.

if (e == pe) continue — the parent skip click to expand

When we enter v from u via edge e, we pass e ^ 1 as the parent edge to DFS(v). Inside DFS(v), when we iterate over v's neighbors and encounter the edge e ^ 1 (which goes back to u), we skip it with if (edge == pe) continue.

Using vertex check if (v == parent_vertex) continue is wrong for multigraphs: if there are two edges between u and v, both would be skipped, missing a valid back edge.

The two low[] updates and when each fires click to expand

Tree edge update (after DFS(v) returns):

low[u] = min(low[u], low[v]) — v's subtree escape propagates up to u's subtree escape.

Back edge update (v is already visited):

low[u] = min(low[u], ord[v]) — NOT low[v]. See section 5 for why.

Bridge and articulation checks: placement matters click to expand

Bridge check: if (low[v] > ord[u]) — mark both e and e^1 as bridges.

Non-root articulation check: if (pe != -1 && low[v] >= ord[u]) isArt[u] = true — the pe != -1 condition is the non-root guard. If pe = −1, u is the root, so we skip this.

Root check: if (pe == -1 && children > 1) isArt[u] = true — happens after the for loop, so children has the final count of DFS-tree children.

12. Edge Cases

Parallel Edges (Multigraph)

Two edges between same pair (u,v). Using parent by vertex would skip both, incorrectly making low[v] not update via the second edge. Solution: track by edge index. Works automatically with the e == pe check.

Self-Loops

Edge from u to u. Stored as e (u→u) and e^1 (u→u). When DFS(u) processes e, it sees v = u which is already visited, so it does low[u] = min(low[u], ord[u]) — no change. Self-loops never affect articulation or bridge detection. Handled naturally.

Disconnected Graph

Always loop over all nodes and start DFS from any unvisited one. Each connected component is handled independently. Each component root uses the children ≥ 2 rule for its own root independently.

Single Node or Single Edge

Single node: no edges, no articulation, no bridges. Single edge (u,v): it is a bridge (low[v] = ord[v] > ord[u] always since v is the only child). Neither endpoint is an articulation point (removing either still leaves one node). Our template handles both correctly.

1-indexed vs 0-indexed

Our template uses 1-indexed nodes. Ensure arrays are sized N+1. Some problems give 0-indexed input — add or subtract 1 accordingly. Be consistent throughout.

Multiple Test Cases

Call init(n, m) before each test. This does: fill head with −1, fill isArt/isBridge with false, reset ne = 0, increment vid, reset timer_val = 0. All O(N+M) but amortized fine.

Reflex Recognition Rules

removing one edge disconnects graph  →  that edge is a bridge
removing one vertex disconnects graph  →  that vertex is an articulation point
compress graph by collapsing bridges  →  bridge tree

⚠

Common confusion: Bridges and articulation points are outputs of one DFS pass — they are individual edges or vertices you detect. EBCC, VBCC, bridge tree, and block-cut tree are structures you build on top of those results by grouping nodes or contracting the graph. If the problem asks you to detect weak points, stop at bridges/APs. If it asks you to query connectivity or navigate the compressed structure, proceed to the tree.

14. Variants & Extensions

Vertex-Biconnected Components (VBCC)

A VBCC is a maximal subgraph with no articulation point inside. Articulation points sit on the boundary between BCCs — they belong to multiple BCCs simultaneously.

What changes: We add a node stack. When we detect low[v] ≥ ord[u], we pop nodes from the stack — those form one BCC together with u.

// ── VBCC Extraction — add to base template ───────────────
stack<int> stk;
vector<vector<int>> bccs;

void dfs_bcc(int u, int pe = -1) {
    vis[u] = vid;
    ord[u] = low[u] = ++timer_val;
    stk.push(u);
    int children = 0;

    for (int e = head[u]; ~e; e = nxt[e]) {
        int v = to[e];
        if (e == pe) continue;

        if (vis[v] != vid) {
            children++;
            dfs_bcc(v, e ^ 1);
            low[u] = min(low[u], low[v]);

            if (low[v] >= ord[u]) {
                // Extract one VBCC
                vector<int> comp = {u};
                while (stk.top() != v) {
                    comp.push_back(stk.top());
                    stk.pop();
                }
                comp.push_back(v);
                stk.pop();
                bccs.push_back(comp);

                if (pe != -1) isArt[u] = true;
            }
        } else {
            low[u] = min(low[u], ord[v]);
        }
    }
    if (pe == -1 && children > 1) isArt[u] = true;
}
💡

Key insight: a single articulation point belongs to multiple BCCs. After extraction, each BCC is a list of nodes. You can then check for each BCC how many articulation points it contains.

Use for: UVA 1108 (Mining Your Own Business), block-cut tree construction, any problem needing explicit BCC enumeration.

Edge-Biconnected Components (EBCC)

An EBCC is a maximal subgraph with no bridges. Unlike VBCCs, the partition is clean — every node belongs to exactly one EBCC.

Algorithm: Run the base template to mark all bridges. Then run a second DFS (or BFS) on non-bridge edges to assign component IDs.

// ── EBCC Construction ─────────────────────────────────────
int comp[MAXN];

void dfs_ebcc(int u, int id) {
    comp[u] = id;
    for (int e = head[u]; ~e; e = nxt[e]) {
        int v = to[e];
        if (comp[v] == -1 && !isBridge[e]) {
            dfs_ebcc(v, id);
        }
    }
}

int buildEBCC() {
    fill(comp, comp + n + 1, -1);
    int numComp = 0;
    for (int i = 1; i <= n; i++) {
        if (comp[i] == -1) {
            dfs_ebcc(i, numComp++);
        }
    }
    return numComp; // number of EBCCs
}
✅

Each EBCC is 2-edge-connected (any edge can be removed without disconnecting it). Bridges connect EBCCs — the bridge tree uses EBCCs as nodes.

Use for: Bridge tree construction (below), 2-edge-connectivity queries, "minimum edges to add to make graph bridgeless" problems.

Bridge Tree

Contract each EBCC into a single super-node. Connect super-nodes with the bridges between them. The result is always a tree. This tree encodes all bridge-level structure of the graph.

// ── Bridge Tree Construction ─────────────────────────────
// Prerequisite: run base template + buildEBCC()

vector<int> tree[MAXN];  // bridge tree adjacency

void buildBridgeTree(int numComp) {
    // For each bridge edge, connect its two EBCC endpoints
    for (int e = 0; e < ne; e += 2) {
        if (isBridge[e]) {
            int cu = comp[to[e ^ 1]];  // EBCC of edge's source
            int cv = comp[to[e]];      // EBCC of edge's destination
            tree[cu].push_back(cv);
            tree[cv].push_back(cu);
        }
    }
    // tree[0..numComp-1] is now the bridge tree
}

// ── Bridge Tree Diameter ─────────────────────────────────
// Standard tree diameter via two BFS/DFS passes
int bfs_far(int src, int n, int& farthest) { /* standard BFS */ }
int bridgeTreeDiameter(int numComp) {
    int far1; bfs_far(0, numComp, far1);
    int far2; return bfs_far(far1, numComp, far2);
}

Use for: CF 1000E — We Need More Bosses ↗ (bridge tree diameter), "minimum bridges to cross" path queries, any bridge-level path reasoning.

💡

On the bridge tree, the path between any two nodes in the original graph corresponds to the unique path in the bridge tree — crossing exactly those bridges that must be traversed.

Block-Cut Tree

The block-cut tree is the "dual" of the bridge tree, but for vertex-connectivity instead of edge-connectivity. It's a bipartite tree where:

  • Block nodes = one node per VBCC
  • Cut nodes = one node per articulation point
  • A block node connects to a cut node if that articulation point belongs to that BCC
// ── Block-Cut Tree Construction ──────────────────────────
// After VBCC extraction (bccs[] and isArt[])

// Node IDs: [1..artCount] = cut nodes (original IDs mapped)
//           [artCount+1..artCount+bccCount] = block nodes

int artID[MAXN], cutCnt = 0;
vector<int> bct[MAXN * 2];  // block-cut tree

void buildBlockCutTree() {
    // Assign IDs to articulation points
    for (int i = 1; i <= n; i++)
        if (isArt[i]) artID[i] = ++cutCnt;

    // For each BCC, create a block node and link to its art points
    for (int b = 0; b < (int)bccs.size(); b++) {
        int blockNode = cutCnt + b + 1;
        for (int u : bccs[b]) {
            if (isArt[u]) {
                bct[blockNode].push_back(artID[u]);
                bct[artID[u]].push_back(blockNode);
            }
        }
    }
}

Use for: "Does path from u to v necessarily pass through articulation point x?" — query on block-cut tree via LCA. Path queries in problems like Gym 102900K.

⚠️

The block-cut tree has at most 2N nodes (N original + N BCCs). Size your arrays accordingly.

Bridge-Aware Edge Orientation (UVA 610 style)

Problem: Orient all edges of an undirected graph such that the result is strongly connected. Only possible if the graph has no bridges. Bridges must be reported separately.

Key insight: In a graph with no bridges, DFS naturally gives a valid orientation: tree edges go parent→child, back edges go child→ancestor. This creates a cycle through every edge, ensuring strong connectivity.

// ── Orientation DFS ───────────────────────────────────────
vector<pair<int,int>> directed;  // oriented edges

void dfs_orient(int u, int pe = -1) {
    vis[u] = vid;
    ord[u] = low[u] = ++timer_val;

    for (int e = head[u]; ~e; e = nxt[e]) {
        int v = to[e];
        if (e == pe) continue;

        if (vis[v] != vid) {
            // Tree edge: orient parent → child
            directed.push_back({u, v});
            dfs_orient(v, e ^ 1);
            low[u] = min(low[u], low[v]);

            if (low[v] > ord[u])
                isBridge[e] = isBridge[e ^ 1] = true;
        } else if (ord[v] < ord[u]) {
            // Back edge: only orient once (when going to ancestor)
            directed.push_back({u, v});
            low[u] = min(low[u], ord[v]);
        }
    }
}

// Bridges: must be output in BOTH directions
// Non-bridges: use directed[] list

Problems: VJudge — UVA 610 Street Directions ↗, CSES 2177 — Strongly Connected Edges ↗

MST Edge Classification (CF 160D)

Classify each edge as: in no MST, in some MST, or in every MST.

Key insight: Process Kruskal's algorithm. For each group of equal-weight edges, build a temporary graph on DSU-compressed nodes and run bridge detection. Bridges in that group = edges that must be in every MST (if they connect two components). Non-bridge same-weight edges = optional.

// ── MST Edge Classification ───────────────────────────────
// Sort edges by weight. For each equal-weight group:
//   1. Only consider edges connecting different DSU components.
//   2. Map those DSU-component IDs to 1..K (compress).
//   3. Build a temporary graph on K nodes.
//   4. Run bridge detection on that graph.
//   5. Bridges → in EVERY MST
//      Non-bridges connecting different components → in SOME MST
//      Edges connecting same component → in NO MST
//   6. Merge all nodes touched by this group in DSU.

// Each edge[i].type = 0 (none), 1 (some), 2 (every)
void classifyMSTEdges(vector<Edge>& edges) {
    sort(edges.begin(), edges.end(),
         [](auto& a, auto& b){ return a.w < b.w; });

    DSU dsu(n);
    for (int i = 0; i < (int)edges.size(); ) {
        int j = i;
        while (j < (int)edges.size() && edges[j].w == edges[i].w) j++;

        // Build temp graph for edges[i..j-1]
        // ... (compress DSU IDs, rebuild adjacency, run dfs(), classify)

        // Merge in DSU
        for (int k = i; k < j; k++) dsu.merge(edges[k].u, edges[k].v);
        i = j;
    }
}

🧩 UVA 1108 — Mining Your Own Business (BCC + Counting) VJudge ↗

Story: A mining company has tunnels (a graph). Each tunnel junction can be an escape exit. Miners need safety: placing exits so any group can always evacuate. Minimize exits needed and count placements.

Abstraction: Extract all VBCCs. A BCC with exactly one articulation point on its boundary needs exactly one exit (any non-articulation vertex inside it). Count BCCs with ≤ 1 boundary articulation. Answer = count, ways = 2^count.

Why BCC? Within a VBCC, any vertex remains reachable from any other even after removing any single vertex. So one exit anywhere inside is sufficient. But if a BCC touches two articulation points, the miners can always choose which side to escape to — still needs one exit.

Key insight and approach click to expand

For each VBCC, count how many of its nodes are articulation points. Let that count be c.

  • If c = 0: all nodes are non-articulation. This BCC needs one exit (can be any node). count++, ways × 2.
  • If c = 1: one boundary articulation point. The BCC needs one exit inside it (any non-articulation node). count++, ways × (size - 1) or just note there are choices.
  • If c ≥ 2: the BCC has multiple articulation points; miners can escape to multiple BCCs. No exit needed here.

Special case: if the whole graph has no articulation points (single BCC), answer is 2 exits, 1 way (any 2 vertices).

Complexity: O(N + M). VJudge ↗

🧩 UVA 610 — Street Directions (Edge Orientation) VJudge ↗

Story: A city wants to convert all streets to one-way while keeping the city strongly connected from any starting point. Find a valid orientation or report where it's impossible.

Key insight: Strong orientation is possible iff the graph has no bridges. Non-bridge edges can always be oriented using DFS: tree edges go downward, back edges go upward (toward ancestor). This creates a strong orientation. Bridges must be output in both directions (they can't be made one-way in a strongly connected orientation).

Complexity: O(N + M). VJudge ↗

🧩 Codeforces 1000E — We Need More Bosses (Bridge Tree Diameter) CF ↗

Story: In a directed graph (after condensation to SCCs), find the largest SCC, then find the diameter of the bridge tree within it.

Abstraction: SCC condensation → build bridge-free graph on condensed nodes → find EBCCs → build bridge tree → compute tree diameter (two-BFS).

Key observations:

  • After SCC condensation, treat the condensed graph as undirected for bridge detection.
  • Build bridge tree: EBCCs become nodes, bridges become edges.
  • Diameter of bridge tree = max edges to cross = answer.

🧩 Codeforces 160D — Edges in MST (MST Classification) CF ↗

Story: Given a weighted undirected graph, classify each edge: does it appear in no MST, some MSTs, or every MST?

Abstraction: Process edges grouped by weight (Kruskal style). For each weight group, temporarily contract the graph via DSU from lighter edges, build the group graph, run bridge detection. Bridges in group = must-have edges. Non-bridge edges between different components = sometimes-present. Edges within same component = never used.

16. Advanced Extensions

Block-Cut Tree LCA Queries

After building the block-cut tree, binary lifting enables O(log N) queries like: "Does removing vertex x disconnect nodes u and v?" — check if x lies on the path from u to v in the block-cut tree.

Bridge Tree DP

Tree DP on the bridge tree solves problems like "minimum edges to add so the graph has no bridges" (answer = ⌈leaves/2⌉ of the bridge tree), diameter under edge additions, etc.

Offline Removal Queries

If queries ask "are u and v connected after removing vertex x?", precompute BCC structure and check component membership. Low-link DFS makes O(1) per query possible after O(N+M) preprocessing.

Directed Analogue: Tarjan SCC

Tarjan's SCC algorithm uses a very similar low-link invariant on directed graphs. The "escape from subtree" idea becomes "can you reach back to your ancestor in a directed sense?" — strongly connected components play the role of BCCs.

2-SAT via SCC

2-SAT builds an implication graph and uses SCC to determine satisfiability. The connection: SCCs are the "atomic units" of the implication structure, analogous to how BCCs are atomic units of connectivity.

Pair Impact Counting

Problems asking "how many pairs (u,v) become disconnected if vertex x is removed?" can be answered in O(N+M) using BCC sizes and articulation tree structure (CodeChef SADPAIRS style).
