Class 01 — Distance & Comparison Tasks

The foundation class. We establish the type system, the cmp gatekeeper, the safe math wrappers, and the dot/cross derivations that every later class depends on.

🎯 1 — Problem Definition

Competitive programming geometry is not really about memorising formulas. It is about converting a visual statement into reliable predicates and measurements — answers to questions like:

How far is this point from that one? Are these two values "equal enough"? Is this number negative, zero, or positive? Is one distance bigger than another?

This first class focuses on the two cheapest, most-used operations: computing a distance and comparing two numeric values reliably. Every other geometry class — segment intersection, polygon area, circle tangents, convex hull — leans on these two operations being correct.

The four classic CP geometry bugs. Almost every wrong-answer geometry submission traces back to one of these:
  1. Using the wrong model — e.g. y = m·x + c for a vertical line.
  2. Comparing floating values with raw ==.
  3. Forgetting degenerate cases — zero-length vectors, coincident points.
  4. Mixing line, ray, and segment behavior under the same predicate.
Bugs 2 and 3 are entirely the responsibility of Class 01. Class 01 cures them at the root by introducing one comparison policy that every later predicate funnels through.

🧠 2 — Intuition from Scratch

2.1 — A point is a location

In 2D Cartesian coordinates a point is a place: an x tells you how far right or left, a y tells you how far up or down. We write it P = (x, y). A point is not an arrow; it only says "where is the object?"

Diagram · A point on the Cartesian plane

2.2 — A vector is a displacement

A vector is a movement. If we move from point A to point B, the vector is AB = B − A. Concretely, if A = (x₁, y₁) and B = (x₂, y₂) then AB = (x₂ − x₁, y₂ − y₁).

Important distinction. A point is a location; a vector is a movement. But both are stored as two numbers in code, so CP geometry uses one C++ type for both — typically complex<long double> — and lets the operations distinguish them.

2.3 — Distance is "length of the displacement"

The distance from A to B is the length of the vector B − A. So once we know how to measure the length of a vector, we know how to measure distance.

Diagram · Distance as the length of the displacement vector

👁 3 — Visual Mental Model

3.1 — The right triangle and Pythagoras

If a vector is v = (x, y), then its length is the hypotenuse of a right triangle with legs |x| and |y|. By Pythagoras:

|v|² = x² + y²
|v|  = √(x² + y²)
Diagram · Length of (x, y) as a hypotenuse

3.2 — Use hypot, not sqrt(x*x + y*y)

In code, prefer hypot(x, y) over sqrt(x*x + y*y). The naive form overflows for huge values (when x² already exceeds the floating type's range, even though |v| would still fit) and underflows for tiny values. The standard library's hypot rescales internally to avoid both ends.

3.3 — Comparing two distances without taking a square root

Square roots are slow and lossy. When we only need to compare two distances — "is A closer to me than B?" — we don't need the actual distance. We compare squared distances:

// "is dist(p, a) <= dist(p, b)?"
// equivalent to "is |p-a|² <= |p-b|²?"
dot(p - a, p - a)  vs  dot(p - b, p - b)

Squared distance is monotone with distance for non-negative reals, so any ordering question reduces to an integer/rational comparison and avoids sqrt entirely. This is the single most common optimisation in geometry contests.

🧮 4 — Formal Abstraction

4.1 — How floating numbers are stored

Mathematical reals may have infinitely many digits (0.1, 1/3, √2, π). A variable has finite memory, so floating types store an approximation in binary scientific notation:

number ≈ sign × mantissa × 2^exponent

The exponent decides scale; the mantissa stores significant digits. The crucial consequence:

Floating-point keeps a fixed number of significant digits, not a fixed number of decimal places. Near zero, representable numbers are densely packed. Around 10¹⁸, the gaps between neighbours are huge.
Diagram · Density of representable doubles

4.2 — Type guide

4.3 — The cmp gatekeeper and EPS

The single most important policy in this template family is: never compare floating values with ==, <, or > directly. Every comparison goes through cmp. The function returns a tri-state sign:

cmp(a, b) = -1   if a is "definitely less"    than b
cmp(a, b) =  0   if a is "indistinguishable"  from b
cmp(a, b) = +1   if a is "definitely greater" than b

For integer types the comparison is exact. For floating types, any difference inside an interval [−EPS, +EPS] is treated as zero. We use EPS = 1e-12L with long double arithmetic as a practical default — see §8.2 for a discussion of when this is appropriate and when a different choice is needed.

Diagram · The EPS band

4.4 — Safe wrappers for sqrt, acos, asin

Geometry formulas often produce values that are mathematically inside a valid domain, but floating noise pushes them just outside. Classic failures:

sqrt(-0.00000000000000001)        // NaN
acos(1.00000000000000002)         // NaN
asin(-1.00000000000000002)        // NaN

The fix is a thin clamping wrapper:

  • safeSqrt uses cmp to treat values inside the EPS band around zero as exactly zero (since the input should have been non-negative, and a tiny negative value is interpreted as numerical drift around zero).
  • safeAcos / safeAsin clamp the argument directly to [−1, +1] using raw > / < comparisons — not cmp. The reason is precise: even an EPS-small overshoot like 1 + EPS/2 falls inside the cmp-equality band, so a cmp-based clamp would let it through unchanged, and acosl(1 + EPS/2) returns NaN. A raw-domain clamp catches every value outside the mathematical domain, including sub-EPS drift.

These wrappers are not meant to hide truly invalid geometry — they protect against the tiny numerical drift that happens in every real formula.

💡 5 — Key Observations

  • O1. Equality on doubles is undefined behaviour in spirit. A mathematically zero quantity, computed by a chain of float operations, may be stored as 0.00000000000000003. So if (x == 0) is a bug, not a check.
  • O2. Floating-point addition is non-associative. (a + b) + c can differ from a + (b + c) by many ULPs. So computing the same geometric quantity by two different code paths can give two slightly different floats — and you cannot rely on == to detect that they "should be" equal.
  • O3. Around very large magnitudes, x + 1 == x may be true. Demo: with double x = 1e18; the comparison x == x + 1 evaluates to true. The +1 is below the resolution at that scale.
  • O4. hypot(x, y) is not a luxury — it prevents catastrophic overflow for large |x|, |y|, and catastrophic underflow for tiny ones. Even when both are well-behaved, it costs one extra rescale.
  • O5. Squared distance is better than distance for ordering. Don't sqrt just to compare.
  • O6. The zero vector has no direction. So dot(0, b) = 0 and cross(0, b) = 0 do not mean "right angle" or "collinear" in any useful sense — they mean nothing at all. Every angle/orientation predicate must document that its inputs are non-zero.

⚙ 6 — Algorithm / Method Explanation

6.1 — The cmp family

Three functions, one public face:

  1. cmpInt(a, b). For integral types, returns exactly (a > b) − (a < b). This is the standard branchless sign-of-difference idiom.
  2. cmpFloat(a, b). For floating types, computes d = (long double)(a − b) and returns (d > EPS) − (d < −EPS). Anything in the EPS band returns 0.
  3. cmp(a, b). The single user-facing wrapper. Uses if constexpr (is_floating_point_v<U>) at compile time to dispatch to the correct sibling.

Inside the geometry library, every sign question — "is this discriminant negative?", "is this orientation collinear?", "is this distance smaller?" — flows through cmp. Outside the library, the user never calls cmpInt or cmpFloat directly.

6.2 — The safe* family

Three wrappers that absorb tiny floating-point drift just before it crashes a domain check:

  • safeSqrt(x). If cmp(x) == 0, return 0. Otherwise return sqrtl(x). This rescues formulas like h² = r² − x² when the right-hand side computes to −1e-18.
  • safeAcos(x). Clamp x raw to [−1, +1] using direct > / < comparisons, then call acosl. We deliberately do not route this clamp through cmp: even tiny drift like 1 + EPS/2 falls inside the cmp-equality band and would otherwise reach acosl unchanged, producing NaN.
  • safeAsin(x). Same raw-domain clamp, then asinl.

6.3 — Length and distance

Length of a vector is just hypotl(x, y). Distance between two points is the length of their difference vector.

6.4 — samePoint

Two points are equal when both coordinates compare equal under cmp. This is why we cannot rely on the default complex<T>::operator== for floating geometry — that operator uses raw == on each component.

🔒 7 — Invariants

State invariants

  • I1 — Tri-state output. cmp never returns anything other than −1, 0, +1. Downstream code can rely on switching on the sign.
  • I2 — Reflexivity. cmp(a, a) == 0 for every value a, integer or floating. Required for samePoint(p, p) to be true.
  • I3 — Antisymmetry. cmp(a, b) == −cmp(b, a). Required because predicates like orient(a, b, p) get reversed by reversing the line direction.
  • I4 — Type-safe dispatch. An integer comparison can never trigger the EPS branch. A floating comparison can never run the raw integer-difference branch. if constexpr enforces this at compile time.

Transition invariants

  • I5 — safeSqrt maps tiny negative noise to zero. Any caller can safely compute safeSqrt(r² − x²) when mathematics guarantees the result is real, even if floats have produced −1e-18.
  • I6 — safeAcos / safeAsin output is in [0, π] / [−π/2, π/2]. The raw-domain clamp guarantees this for every input: anything > 1 becomes 1, anything < −1 becomes −1, and acosl/asinl on those endpoints returns a real number, never NaN.

Termination invariants

Every function in this class returns in O(1) with no loops, no recursion, no I/O. Each operation is a constant number of arithmetic instructions.

Failure mode — applicability. cmp's EPS branch is appropriate when the inputs are differences of geometric quantities of comparable magnitude. It is the wrong tool when the inputs span 10+ orders of magnitude — in that regime a relative tolerance is needed, not an absolute one. See §16 Variants for cmpRel.

📜 8 — Proof of Correctness

8.1 — Why (a > b) − (a < b) is the right sign

For integers, exactly one of three things is true: a < b, a = b, a > b. The expression evaluates as:

a < b   :  (0) - (1) = -1
a == b  :  (0) - (0) =  0
a > b   :  (1) - (0) = +1

Branchless, correct, and inlined by every compiler.

8.2 — Why EPS = 1e-12L is a reasonable practical default

This is a contest heuristic, not a formal bound. The C++ standard does not pin down the precision of long double; in practice, on g++ for x86-64 Linux it is typically an 80-bit extended type with around 18–19 decimal digits of mantissa, but on MSVC or on ARM it may be the same as double. So we cannot prove anything about EPS in a portable way.

As a working heuristic: each elementary floating operation typically introduces a relative error around 10⁻¹⁸ to 10⁻¹⁹ on a long-double system. A geometry formula chaining roughly 10 operations on operands bounded by 10⁶ tends to accumulate absolute error around 10⁻¹². So EPS = 1e-12L is a workable starting point for problems with coordinates up to 10⁶ and short formula chains. For larger coordinates, longer formula chains, or when sub-EPS distinctions actually decide the answer, switch to relative tolerance (§16) or rescale coordinates.

Calibration rule. Pick EPS by trying a tighter and a looser value on a few stress tests. If the verdict changes with EPS, the algorithm is on the edge — re-derive it on more stable predicates rather than nudging EPS.

8.3 — Derivation of the dot product

We give three derivations because each gives a different intuition about what dot product means.

Coordinate form

Define dot(a, b) := x₁·x₂ + y₁·y₂. This is purely algebraic; no geometry yet.

Complex form

Let a = x₁ + y₁i and b = x₂ + y₂i. Then:

conj(a) · b
= (x₁ - y₁i)(x₂ + y₂i)
= (x₁x₂ + y₁y₂) + (x₁y₂ - y₁x₂)i

So Re(conj(a) · b) = x₁x₂ + y₁y₂ = dot(a, b). This is why our code can compute the dot product with one complex multiplication: real(conj(a) * b).

Polar form

Write a = |a|(cos α, sin α) and b = |b|(cos β, sin β). Then:

dot(a, b)
= |a||b|(cos α cos β + sin α sin β)
= |a||b| cos(α - β)
= |a||b| cos θ

So the dot product is |a||b|·cos θ where θ is the angle between a and b. This is why the sign of dot tells you whether the angle is acute / right / obtuse:

dot > 0 ⇒ acute angle dot = 0 ⇒ right angle dot < 0 ⇒ obtuse angle

8.4 — Derivation of the cross product (2D)

Coordinate form

cross(a, b) := x₁·y₂ − y₁·x₂.

Complex form

From the same complex expansion as above, Im(conj(a) · b) = x₁y₂ − y₁x₂ = cross(a, b). Our code reads imag(conj(a) * b).

Polar form

cross(a, b)
= |a||b|(cos α sin β - sin α cos β)
= |a||b| sin(β - α)
= |a||b| sin θ

So the sign of cross tells you which side b is on relative to a:

cross > 0 ⇒ b is left/CCW from a cross = 0 ⇒ a, b are collinear cross < 0 ⇒ b is right/CW from a

3D lift (sanity check)

If we lift a = (x₁, y₁) and b = (x₂, y₂) into 3D as (x, y, 0), their 3D cross product is (0, 0, x₁y₂ − y₁x₂). The z component is exactly our 2D cross product. This confirms the geometric interpretation (signed area, turn direction).

Diagram · dot vs cross signs (angle θ between a and b)

📐 10 — Core Template Philosophy

  • Compact. No abstract base classes, no exception machinery, no struct Point { double x, y; … }; with hand-rolled operators. We use complex<long double> and rely on the standard library to provide +, −, *, /, conj, polar, and norm.
  • Central cmp. Every sign-of-quantity check funnels through cmp. There is exactly one place in the codebase where EPS appears.
  • Const-reference inputs. Geometry primitives are small but not free to copy. const Point& in every signature.
  • Documented preconditions. Every helper that requires a non-zero vector or two distinct points says so in a comment. We trust callers; we do not add defensive checks that hide bugs.
  • No macro named rotate. It collides with std::rotate. We use a small helper function named rot instead — a typed function rather than a macro, which also avoids std::polar mixed-type deduction issues.
  • Unified naming. dist, dot, cross, cmp — short, memorable, consistent across every class in the family.

💻 11 — Master / Base Implementation

This is the geometry header that every later class in the family extends. It compiles standalone with g++ -std=c++17.

#include <bits/stdc++.h>
using namespace std;

using T     = long double;
using Point = complex<T>;

#define X real()
#define Y imag()

const long double EPS = 1e-12L;
const T PI = acosl(-1.0L);

// ---- rotation helper (typed function, not a macro) ----------------
// A real function fixes the std::polar mixed-type deduction problem:
// rot(Point(1,0), 0.5) coerces the double 0.5 to T before the call.
Point rot(const Point& v, T t) {
    return v * polar((T)1, t);
}

// ---- comparison gatekeeper ----------------------------------------
template<class U>
int cmpInt(const U& a, const U& b = U(0)) {
    return (a > b) - (a < b);
}

template<class U>
int cmpFloat(const U& a, const U& b = U(0)) {
    long double d = (long double)a - (long double)b;
    return (d > EPS) - (d < -EPS);
}

template<class U>
int cmp(const U& a, const U& b = U(0)) {
    if constexpr (is_floating_point_v<U>) return cmpFloat(a, b);
    else return cmpInt(a, b);
}

// ---- safe math wrappers -------------------------------------------
T safeSqrt(T x) {
    if (cmp(x) == 0) return 0;
    return (T)sqrtl((long double)x);
}

// Raw-domain clamp: cmp's EPS band is too wide here.
// 1 + EPS/2 is still mathematically out of the acos/asin domain
// and would yield NaN if it reached the libm call unchanged.
T safeAcos(T x) {
    if (x > (T) 1) x =  1;
    if (x < (T)-1) x = -1;
    return (T)acosl((long double)x);
}

T safeAsin(T x) {
    if (x > (T) 1) x =  1;
    if (x < (T)-1) x = -1;
    return (T)asinl((long double)x);
}

// ---- dot and cross via complex multiplication ---------------------
T dot  (const Point& a, const Point& b) { return real(conj(a) * b); }
T cross(const Point& a, const Point& b) { return imag(conj(a) * b); }

// ---- length and distance ------------------------------------------
T len (const Point& a)                    { return (T)hypotl((long double)a.X, (long double)a.Y); }
T dist(const Point& a, const Point& b)    { return len(a - b); }

// ---- equality on points (uses cmp on each coordinate) -------------
bool samePoint(const Point& a, const Point& b) {
    return cmp(a.X, b.X) == 0 && cmp(a.Y, b.Y) == 0;
}
Two macros that look identical to fields but are not. #define X real() means p.X expands to p.real(). They are method calls dressed up as field access — pure CP convenience, never used in production C++.

⚠ 14 — Common Pitfalls / Bugs

  • P1 — Comparing doubles with raw == / <. The single most common geometry bug. Funnel everything through cmp.
  • P2 — Calling acos / asin on a raw formula result. Even when the formula is mathematically inside [-1, 1], floats may push it out. Use safeAcos / safeAsin.
  • P3 — Routing the acos/asin clamp through cmp. The EPS-equality band swallows sub-EPS overshoots like 1 + EPS/2, which then crash acosl. Use raw > / < against the literal endpoints.
  • P4 — Calling sqrt on a "should be non-negative" expression. Use safeSqrt.
  • P5 — Using sqrt(double) instead of sqrtl(long double). Loses the entire benefit of long double.
  • P6 — One EPS for all magnitudes. 1e-12 is fine for coordinates up to 10⁶. For coordinates near 10⁹ consider 1e-9 or use cmpRel.
  • P7 — Computing a distance just to compare two distances. Compare dot(v, v) instead. Faster and exact when inputs are integer.
  • P8 — Treating dot((0,0), b) = 0 as "right angle". The zero vector has no direction. Any angle predicate must guard against zero-length inputs.
  • P9 — Macro named rotate instead of rot. Collides with std::rotate from <algorithm>. Class 02 enforces the rot name.
  • P10 — Using complex<T>::operator== for point equality. That operator uses raw == on each component. Use samePoint.
  • P11 — Forgetting long double promotion in cmpFloat. If a and b are double, computing a - b in double can lose precision before the EPS test even runs. We explicitly cast to long double first.

I'm about to call sqrt on a non-negative formula.safeSqrt
I'm about to call acos / asin.safeAcos / safeAsin
I need to check two points are the same.samePoint(a, b)
Coordinates are integers ≤ 10⁹ and I need exact orientation.Use long long; cmp dispatches to cmpInt.
Coordinates are integers near 10⁹ and I'm forming a cross product.Promote to __int128 for the temporary.
Coordinates span 10+ orders of magnitude.Switch to relative tolerance — see §16 cmpRel.

Reflex rules (contest speed)

Every comparison goes through cmp. Every sqrt becomes safeSqrt. Every acos/asin becomes safeAcos/safeAsin. No == on doubles, ever. No sqrt just to compare. No macro named "rotate".

Common confusion

"Why isn't my EPS catching this?" Two usual causes: (a) you compared a quantity built by squaring (so the magnitudes are 10⁶ instead of 10³), and your absolute EPS is now too tight; (b) you called == directly on a double somewhere before cmp got a chance to run. Search the file for raw == on floating types — there should be none.

🌿 16 — Variants & Extensions

16.1 — Squared distance for ordering

The single most-used optimisation. Replace every dist(a, b) by dot(a − b, a − b) when the goal is comparison only. What changes: avoids sqrt. Why: on integer coordinates, the comparison becomes exact. When to use: any closest-pair / farthest-pair / nearest-neighbour question.

// Compare distances p-a and p-b without sqrt
bool closerToA(const Point& p, const Point& a, const Point& b) {
    return cmp( dot(p-a, p-a), dot(p-b, p-b) ) < 0;
}

16.2 — Relative tolerance cmpRel

When operands range over many orders of magnitude, an absolute EPS = 1e-12 may be far too tight for the large end and far too loose for the small end. The fix is to scale the tolerance with the magnitudes:

template<class U>
int cmpRel(const U& a, const U& b) {
    long double d   = (long double)a - (long double)b;
    long double scl = max((long double)1,
                          max(fabsl((long double)a), fabsl((long double)b)));
    long double eps = EPS * scl;
    return (d > eps) - (d < -eps);
}

What changes: the EPS adapts. Why: robust across many magnitudes. When to use: coordinates with huge dynamic range, or repeated multiplications that grow magnitudes.

16.3 — Manhattan and Chebyshev distance

Two common alternative metrics:

T distManhattan (const Point& a, const Point& b) { return fabsl(a.X-b.X) + fabsl(a.Y-b.Y); }
T distChebyshev (const Point& a, const Point& b) { return max(fabsl(a.X-b.X), fabsl(a.Y-b.Y)); }

The 45° rotation (x, y) ↦ (x + y, x − y) converts Manhattan distance to Chebyshev distance — a classic trick for "rook moves on a tilted grid" problems.

16.4 — __int128 for large integer cross products

When integer coordinates are bounded by 10⁹, the product x₁·y₂ can reach 10¹⁸, just inside long long; the difference x₁y₂ − y₁x₂ can overflow. Use __int128 for the intermediate, then read off the sign with cmpInt.

🚀 17 — Problem-Based Applications

UVA 10242 — Fourth Point classic vector-arithmetic warm-up

Story. You are given four points on input, three of which are the distinct vertices of a parallelogram (in some order) and the fourth being one of the three repeated — the repeated point names the shared vertex of the two edges that the input describes. Output the missing fourth vertex.

Abstraction. A parallelogram has two pairs of parallel, equal-length edges. If shared, a, b are the three distinct vertices with shared being the vertex where the two given edges meet, the missing point is a + b − shared — the vertex opposite to shared.

Why this geometry model fits. "Parallelogram" is the smallest possible setting in which point-as-location and vector-as-displacement need to coexist. The opposite-vertex identity is just vector addition in disguise.

Algorithm. The four input points contain one repeat — that's the shared vertex. The other two are a and b. Apply a + b − shared. Output uses 3 decimal places.

#include <bits/stdc++.h>
using namespace std;

struct Point {
    double x, y;
};

bool samePoint(const Point& a, const Point& b) {
    return fabs(a.x - b.x) < 1e-9 && fabs(a.y - b.y) < 1e-9;
}

Point missingPoint(Point p1, Point p2, Point p3, Point p4) {
    Point shared, a, b;

    if (samePoint(p1, p3)) {
        shared = p1;
        a = p2;
        b = p4;
    } else if (samePoint(p1, p4)) {
        shared = p1;
        a = p2;
        b = p3;
    } else if (samePoint(p2, p3)) {
        shared = p2;
        a = p1;
        b = p4;
    } else {
        // p2 == p4
        shared = p2;
        a = p1;
        b = p3;
    }

    return {a.x + b.x - shared.x, a.y + b.y - shared.y};
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    Point p1, p2, p3, p4;

    while (cin >> p1.x >> p1.y >> p2.x >> p2.y
               >> p3.x >> p3.y >> p4.x >> p4.y) {
        Point ans = missingPoint(p1, p2, p3, p4);
        cout << fixed << setprecision(3) << ans.x << ' ' << ans.y << '\n';
    }

    return 0;
}

Quick verification.

  • Input 0 0 1 0 0 0 0 1 → shared = (0,0), a = (1,0), b = (0,1), answer = (1,1).
  • Input 1 1 2 3 4 5 2 3 → shared = (2,3), a = (1,1), b = (4,5), answer = (3,3).

Complexity. O(1) per test case.

Pitfalls. The four-point input format with one repeat is easy to misread. The same-point test must use a tolerance; using == on double would miss legitimate repeats that round differently. Output precision (3 decimals) is judge-strict.

Closest-pair preview exercise — squared-distance comparison pattern snippet, not a full worked problem

This is not a full contest solution in this class. It is a preview of the squared-distance comparison primitive that will be used later in closest-pair and nearest-neighbour problems. The intent is to show the pattern in isolation so it is recognisable when it reappears in a larger algorithm.

Source URL. URL not provided in source.

Pattern. Given many integer-coordinate points, find the smallest pairwise distance. The brute force is O(n²) and is the right warm-up before Class 10 (closest pair in O(n log n)). The inner loop only compares distances — so we compute squared distances on integers and avoid sqrt entirely.

// Partial pattern snippet — not AC-ready code.
// Assumes p is a vector<pair<long long, long long>> of n points.

long long bestSq = LLONG_MAX;
for (int i = 0; i < n; ++i)
    for (int j = i + 1; j < n; ++j) {
        long long dx = p[i].first  - p[j].first;
        long long dy = p[i].second - p[j].second;
        long long sq = dx*dx + dy*dy;
        if (sq < bestSq) bestSq = sq;
    }
// Only call sqrt at the very end if a distance is required as output.

Pitfalls of the pattern. dx*dx can overflow int — make sure all arithmetic happens in long long. Use __int128 if coordinates reach 10⁹.

📝 18 — Final Summary

The mental model

  • Point = location.
  • Vector = displacement.
  • Distance = length of displacement.
  • dot = projection / angle test (|a||b|cos θ).
  • cross = orientation / area test (|a||b|sin θ).
  • cmp = the single comparison policy.

The four hard rules

  • Every comparison goes through cmp.
  • Every sqrt becomes safeSqrt.
  • Every acos / asin becomes safeAcos / safeAsin with a raw-domain clamp.
  • No macro named rotate — use rot.

Class 01 is small but load-bearing. Everything from segment intersection through Welzl's minimum enclosing circle depends on the cmp gate behaving exactly as described in §7 (Invariants) and §8 (Proof). If a later class produces a numerical bug, the very first place to audit is the EPS choice and the cmp dispatch.

Numeric substrate

Spot checks (run mentally before submitting)

  • cmp(0.0) returns 0.
  • cmp(EPS / 2) returns 0.
