Class 04 — Segment Intersection

When do two segments meet? We build the four-orientation straddle test, the bounding-box pre-filter, the collinear-overlap handling, and the line-line intersection point — all on top of the orient and pointOnSegment tools from Classes 02–03.

🎯 1 — Problem Definition

Two line segments are scattered on the plane. Class 04 answers the questions that every higher geometry algorithm eventually needs:

Do segments [a, b] and [c, d] intersect at all? Do they cross cleanly, or just touch at an endpoint? Are they collinear and overlapping — or collinear and disjoint? If two infinite lines cross, where is the point?

The headline tool is segmentsIntersect(a, b, c, d), built from four orient evaluations plus a handful of pointOnSegment checks for the degenerate cases. Separately, the safe bool lineLineIntersection(a, b, c, d, ret) returns true and fills the unique crossing point into ret when the two infinite lines cross, and returns false for parallel or coincident lines — never divides by zero. When the caller needs to tell parallel apart from coincident, classify with lineRelation first; for hot loops that have already classified, the legacy lineLineIntersectionPreconditioned skips the guard and divides directly.

Recall the four classic geometry bugs (Class 01). Segment intersection is the textbook home of bug 3 ("forgetting degenerate cases") and bug 4 ("mixing line / ray / segment"). Endpoint touches, collinear overlaps, and parallel lines are exactly where naive code silently returns the wrong answer. We treat each explicitly.

🧠 2 — Intuition from Scratch

2.1 — "Straddling" is the whole idea

Picture segment [a, b] as an infinite fence line. If segment [c, d] is going to cross it, then c and d must sit on opposite sides of that fence — one to the left, one to the right. That "opposite sides" condition is precisely a sign change in orient (Class 02): orient(a, b, c) and orient(a, b, d) have different signs.

But straddling one fence is not enough. [c, d] could cross the infinite line through a, b while missing the actual segment. So we require the condition to hold both ways: a and b must also straddle the line through c, d. When each segment straddles the other's line, they genuinely cross.

Diagram · Mutual straddling — each segment crosses the other's line

2.2 — When a sign is zero, something is collinear

If orient(a, b, c) = 0, then c is exactly on the line through a, b — no left, no right. The clean "opposite signs" test does not apply, and we fall into the collinear / touching world, which we handle with pointOnSegment. Every boundary case in this class is announced by one of the four orientations being zero.

2.3 — A cheap pre-filter: bounding boxes

Before any orientation work, ask a trivial question: do the two segments' axis-aligned bounding boxes even overlap? If the boxes are disjoint, the segments cannot touch, and we skip the rest. This is most valuable for the collinear case, where orientation alone cannot separate "overlapping" from "far apart on the same line".

👁 3 — Visual Mental Model

3.1 — The five shapes of "intersection"

"Do they intersect?" hides five visually distinct situations. A correct test must give the right yes/no for every one of them.

Diagram · Five cases — proper cross, endpoint touch, T-junction, collinear overlap, collinear disjoint

3.2 — The bounding box of a segment

A segment's bounding box is the smallest axis-aligned rectangle containing it: from min(x) to max(x) horizontally and min(y) to max(y) vertically. Two boxes overlap exactly when their x-intervals overlap and their y-intervals overlap.

Diagram · Bounding boxes overlap iff both coordinate intervals overlap

🧮 4 — Formal Abstraction

4.1 — The four orientations

Given segments [a, b] and [c, d], compute four side tests. The first pair asks how c and d sit relative to line ab; the second pair asks how a and b sit relative to line cd:

o1 = orient(a, b, c)
o2 = orient(a, b, d)
o3 = orient(c, d, a)
o4 = orient(c, d, b)
ConditionMeaning
o1 * o2 < 0 and o3 * o4 < 0Proper interior crossing — each segment has its endpoints strictly on opposite sides of the other segment's supporting line. (This automatically rules out any zero orientation.)
some oi = 0A point is collinear with the other segment; resolve with pointOnSegment.
all four signs consistent, none zeroNo intersection.

4.2 — The general (proper) case

The strict proper-crossing test is o1 * o2 < 0 and o3 * o4 < 0. This automatically implies none of the four orientations is zero, since a zero would make one product 0, which is not strictly negative. In that situation the crossing is a single interior point — a clean "X". The orientation signs alone encode everything; the bounding box only ever serves as a cheap pre-filter (§4.4).

Strict vs inclusive. The strict/proper crossing branch must use o1 * o2 < 0 && o3 * o4 < 0. The looser condition o1 != o2 && o3 != o4 also fires when one orientation is zero (because 0 != 1), so it wrongly counts endpoint touches as proper crossings. Endpoint touches must instead be left to the guarded pointOnSegment cases.

4.3 — The collinear / touching cases

When at least one orientation is zero, the proper-crossing rule no longer applies. We test, for each endpoint that is collinear with the other segment, whether it actually lies on that segment using pointOnSegment (Class 03 — collinear plus "between"). If any such endpoint lies on the other segment, they intersect. This single mechanism covers endpoint touches, T-junctions, and collinear overlap, and correctly rejects collinear-but-disjoint pairs (no endpoint of either lies on the other).

4.4 — Bounding box as a fast reject

Two segments can only meet if their bounding boxes overlap. Box overlap reduces to two 1-D interval-overlap tests:

// intervals [lo1, hi1] and [lo2, hi2] overlap iff
max(lo1, lo2) <= min(hi1, hi2)

This is a necessary but not sufficient condition: overlapping boxes do not guarantee the segments meet (think of two diagonals of a square's opposite corners that miss). It is a cheap pre-filter, and it is the natural disambiguator for collinear segments, where the orientation signs are all zero and only the box (interval) tells overlap from disjointness.

4.5 — Segment test vs infinite-line intersection

These are two different problems and it pays to keep them apart:

QuestionToolOutput
Do these two segments meet?segmentsIntersectboolean (with all boundary cases handled)
Where do these two infinite lines cross?lineLineIntersection (bool API)true + a point, or false on parallel / coincident lines
How are two infinite lines related?lineRelationcrossing / parallel / coincident

💡 5 — Key Observations

  • O1 — Proper crossing = double straddle. Each segment must have its two endpoints on opposite sides of the other's line. Two sign-change tests, no more.
  • O2 — A zero orientation is a flag, not an answer. It says "collinear here"; the actual yes/no then comes from pointOnSegment.
  • O3 — pointOnSegment unifies every degenerate case. Endpoint touch, T-junction, and collinear overlap are all "an endpoint of one segment lies on the other". One predicate handles all three.
  • O4 — Collinear-disjoint is the trap. All four orientations are zero, yet the segments do not meet. The "between" check inside pointOnSegment (or, equivalently, the bounding-box interval test) is what rejects it.
  • O5 — Bounding box is necessary, never sufficient. Disjoint boxes ⇒ no intersection. Overlapping boxes ⇒ maybe; still need orientations.
  • O6 — Line intersection needs non-parallel lines. The safe bool lineLineIntersection(...) returns false when the denominator cross(r, s) is zero, so a copy-pasted snippet never divides by zero; the legacy lineLineIntersectionPreconditioned assumes the caller has already classified with lineRelation.

⚙ 6 — Algorithm / Method Explanation

6.1 — The intersection test, step by step

  1. Bounding-box reject: if bboxOverlap(a, b, c, d) is false, return false immediately — disjoint boxes cannot intersect.
  2. Compute the four orientations o1, o2, o3, o4.
  3. Strict proper case: if o1 * o2 < 0 and o3 * o4 < 0, return true.
  4. Collinear / touching cases: for each orientation that is zero, test whether the corresponding endpoint lies on the other segment with pointOnSegment; if any does, return true.
  5. Otherwise return false.

The four pointOnSegment checks are guarded by their matching zero orientation, so they only run when the point is actually collinear — keeping the common (proper) case fast.

Diagram · Decision flow of segmentsIntersect

6.2 — The intersection point of two infinite lines

Write line 1 as a + t·r with r = b − a, and line 2 as c + u·s with s = d − c. Setting them equal and solving with cross products gives

t = cross(c - a, s) / cross(r, s)
intersection = a + r * t

The denominator cross(r, s) is zero exactly when the lines are parallel or coincident. The safe bool lineLineIntersection(a, b, c, d, ret) checks this and returns false in that case (no unique intersection); on success it returns true and writes the crossing point into ret. If you also need to tell parallel from coincident, classify with lineRelation. (Symmetrically, u = cross(c − a, r) / cross(r, s) would locate the same point on line 2 — the legacy lineLineIntersectionPreconditioned assumes you have already classified and divides without the guard.)

6.3 — Classifying two infinite lines

lineRelation looks at two cross products. If cross(r, s) ≠ 0 the directions differ and the lines cross at one point. If they are parallel (cross(r, s) = 0), check whether c lies on line 1 via cross(r, c − a): zero means the lines coincide, non-zero means they are parallel and distinct.

🔒 7 — Invariants

State invariants

  • I1 — Symmetry. segmentsIntersect(a, b, c, d) equals segmentsIntersect(c, d, a, b), and is unaffected by swapping a segment's own endpoints. The four-orientation set is the same up to relabelling.
  • I2 — Boundary inclusion. The test treats touching as intersecting: shared endpoints, T-junctions, and collinear overlaps all return true. If a problem wants strict (interior-only) crossings, that is the o1 * o2 < 0 && o3 * o4 < 0 branch alone.
  • I3 — Containment with bounding box. If bboxOverlap is false, segmentsIntersect is necessarily false. The orientation logic never contradicts a disjoint-box verdict.

Transition invariants

  • I4 — Proper crossings ignore the box. When all four orientations are non-zero, the sign tests alone decide the answer; the bounding box is redundant there.
  • I5 — Collinear resolution is exhaustive. When orientations are zero, the four pointOnSegment checks cover every way two collinear segments can touch: any shared point is one segment's endpoint lying on the other.
  • I6 — Line intersection lands on both lines. The returned point satisfies pointOnLine(a, b, ·) and pointOnLine(c, d, ·) up to floating drift.

Termination invariants

Every routine is straight-line code: a fixed number of orientations, dot/cross products, and at most four membership checks. No loops, no recursion. Each returns in O(1).

Failure mode — applicability. The safe bool lineLineIntersection(...) reports false on parallel or coincident lines, so a careless caller never divides by zero. Distinguish parallel from coincident with lineRelation when the surrounding algorithm needs to know which case fired. The legacy lineLineIntersectionPreconditioned assumes the lines are non-parallel and divides directly — keep using it only inside hot loops that have already classified. segmentsIntersect has no such restriction — it is total over all inputs, including degenerate (zero-length) segments, which it treats as their single point.

📜 8 — Proof of Correctness

8.1 — Why mutual straddling is exactly a proper crossing

Suppose all four orientations are non-zero. Parametrize segment [c, d] as c + u(d − c) for u ∈ [0, 1]. The signed distance of this moving point from line ab is a linear function of u. If orient(a, b, c) and orient(a, b, d) have strictly opposite signs (so o1 * o2 < 0), that linear function changes sign on [0, 1], so by the intermediate value theorem it is zero at some interior u* — i.e. the segment [c, d] meets the line ab at one interior point. Symmetrically, o3 * o4 < 0 means segment [a, b] meets line cd at one interior point. Two segments that each cross the other's supporting line at an interior point must cross each other at that common point. Conversely, if either product is non-negative, that segment either lies entirely on one side of the other's line or has an endpoint exactly on it, so the interiors cannot cross. Hence a strict (non-zero sign-change) interior crossing holds iff o1 * o2 < 0 and o3 * o4 < 0. Any case where an orientation is zero is deliberately excluded here and handled by the collinear / touching branch instead.

Diagram · Signed distance to line ab is linear in u, so opposite signs force a zero crossing

8.2 — Why pointOnSegment resolves every collinear case

Suppose some orientation is zero, say orient(a, b, c) = 0, so c lies on line ab. Two collinear segments overlap iff they share at least one point; on a line, a shared point set is a sub-interval, and a non-empty sub-interval must contain an endpoint of one of the two segments. Therefore overlap occurs iff some endpoint of one segment lies on the other — which is exactly what the four pointOnSegment checks test. If none of the four endpoints lies on the opposite segment, the (collinear) segments are disjoint. This also dispatches endpoint touches and T-junctions, where the shared point is itself an endpoint.

8.3 — Why collinear-disjoint correctly returns false

For two collinear but separated segments, all four orientations are zero, so the proper-crossing branch is skipped. Each of the four pointOnSegment checks then fails its "between" sub-test (Class 03, §8.2): no endpoint of either segment sits inside the other's [min, max] extent. Every branch returns false, so the function returns false — the correct answer.

8.4 — The line-intersection formula

Solve a + t·r = c + u·s for t. Take the cross product of both sides with s. Since cross(s, s) = 0, the u·s term vanishes:

cross(a + t*r - c, s) = 0
cross(a - c, s) + t * cross(r, s) = 0
t = cross(c - a, s) / cross(r, s)

The denominator cross(r, s) is the signed area of the parallelogram spanned by the two directions; it is zero precisely when r and s are parallel. Under the non-parallel precondition the division is safe and yields the unique crossing a + t·r.

8.5 — Why the bounding box is necessary but not sufficient

If two segments share a point q, then q lies in both segments, hence in both bounding boxes, so the boxes overlap — proving necessity (disjoint boxes ⇒ no shared point). It is not sufficient: two opposite, non-touching diagonals can have overlapping boxes while missing each other, so box overlap alone cannot confirm intersection. That is why the box is only a pre-filter and the orientation logic remains the deciding test.

📐 10 — Core Template Philosophy

  • Orientation-first. Decide the common (proper) case with four orient signs before touching any special-case machinery. The fast path stays fast.
  • Reuse, do not re-derive. orient comes from Class 02 and pointOnSegment from Class 03. Class 04 is mostly the composition of those two predicates.
  • Boolean test and point computation are separate functions. "Do they meet?" and "where do the lines cross?" have different preconditions and different failure modes; keep them apart so neither contaminates the other.
  • Total where possible. segmentsIntersect is defined for every input, including degenerate segments. The safe bool lineLineIntersection(...) is total too; only the legacy lineLineIntersectionPreconditioned carries a non-parallel precondition, and it is documented loudly.
  • Const-reference inputs, documented preconditions, typed rot. Same family conventions as Classes 01–03; no rotate macro.

💻 11 — Master / Base Implementation

Standalone, compiles with g++ -std=c++17. The first block repeats just enough of the Class 01–03 substrate for the file to build on its own; the Class 04 additions follow the divider.

#include <bits/stdc++.h>
using namespace std;

using T     = long double;
using Point = complex<T>;

#define X real()
#define Y imag()

const long double EPS = 1e-12L;
const T PI = acosl(-1.0L);

// ---- rotation helper (typed function, not a macro) ----------------
Point rot(const Point& v, T t) { return v * polar((T)1, t); }

// ---- Class 01-03 substrate ----------------------------------------
template<class U> int cmpInt(const U& a, const U& b = U(0)) { return (a > b) - (a < b); }
template<class U> int cmpFloat(const U& a, const U& b = U(0)) {
    long double d = (long double)a - (long double)b;
    return (d > EPS) - (d < -EPS);
}
template<class U> int cmp(const U& a, const U& b = U(0)) {
    if constexpr (is_floating_point_v<U>) return cmpFloat(a, b);
    else return cmpInt(a, b);
}
T dot  (const Point& a, const Point& b) { return real(conj(a) * b); }
T cross(const Point& a, const Point& b) { return imag(conj(a) * b); }
T len (const Point& a) { return (T)hypotl((long double)a.X, (long double)a.Y); }
T dist(const Point& a, const Point& b) { return len(a - b); }
bool samePoint(const Point& a, const Point& b) {
    return cmp(a.X, b.X) == 0 && cmp(a.Y, b.Y) == 0;
}
// orient: +1 left/CCW, -1 right/CW, 0 collinear   (Class 02)
int orient(const Point& a, const Point& b, const Point& p) {
    return cmp(cross(b - a, p - a));
}
// pointOnSegment: collinear AND between            (Class 03)
bool pointOnSegment(const Point& a, const Point& b, const Point& p) {
    if (cmp(cross(b - a, p - a)) != 0) return false;
    return cmp(dot(p - a, p - b)) <= 0;
}

// ===================================================================
// CLASS 04 — segment intersection
// ===================================================================

// 1-D overlap: do the intervals [lo1,hi1] and [lo2,hi2] overlap?
bool overlap1D(T lo1, T hi1, T lo2, T hi2) {
    if (lo1 > hi1) swap(lo1, hi1);
    if (lo2 > hi2) swap(lo2, hi2);
    return cmp(max(lo1, lo2), min(hi1, hi2)) <= 0;
}

// Bounding-box overlap for two segments. Necessary, not sufficient.
bool bboxOverlap(const Point& a, const Point& b,
                 const Point& c, const Point& d) {
    return overlap1D(a.X, b.X, c.X, d.X)
        && overlap1D(a.Y, b.Y, c.Y, d.Y);
}

// Do segments [a,b] and [c,d] intersect (touching or crossing counts)?
// Handles proper crossings, endpoint touches, T-junctions, collinear overlap.
bool segmentsIntersect(const Point& a, const Point& b,
                       const Point& c, const Point& d) {
    // fast reject: disjoint bounding boxes cannot intersect
    if (!bboxOverlap(a, b, c, d)) return false;

    int o1 = orient(a, b, c);
    int o2 = orient(a, b, d);
    int o3 = orient(c, d, a);
    int o4 = orient(c, d, b);

    // strict proper crossing: each segment's endpoints are on strictly
    // opposite (non-zero) sides of the other segment's line
    if (o1 * o2 < 0 && o3 * o4 < 0) return true;

    // collinear / touching: an endpoint lies on the other segment
    if (o1 == 0 && pointOnSegment(a, b, c)) return true;
    if (o2 == 0 && pointOnSegment(a, b, d)) return true;
    if (o3 == 0 && pointOnSegment(c, d, a)) return true;
    if (o4 == 0 && pointOnSegment(c, d, b)) return true;

    return false;
}

// Intersection point of two INFINITE lines (a->b) and (c->d).
//
// SAFE API (preferred): returns false on parallel/coincident lines instead of
// dividing by zero. The caller still has to decide what "no unique intersection"
// means for the surrounding algorithm — distinguish parallel from coincident
// with `lineRelation` below if it matters.
bool lineLineIntersection(const Point& a, const Point& b,
                          const Point& c, const Point& d,
                          Point& ret) {
    Point r = b - a;            // direction of line 1
    Point s = d - c;            // direction of line 2
    T denom = cross(r, s);      // 0 -> parallel or coincident
    if (cmp(denom) == 0) return false;
    T t = cross(c - a, s) / denom;
    ret = a + r * t;
    return true;
}

// Preconditioned variant: assumes cross(b-a, d-c) != 0 and divides directly.
// Kept for hot-loop code paths that have already classified with lineRelation.
// In new code prefer the safe bool API above.
Point lineLineIntersectionPreconditioned(const Point& a, const Point& b,
                                         const Point& c, const Point& d) {
    Point r = b - a;
    Point s = d - c;
    T t = cross(c - a, s) / cross(r, s);    // PRECONDITION: cross(r, s) != 0
    return a + r * t;
}

// Classify two INFINITE lines (a->b) and (c->d):
//   0 = cross at one point, 1 = parallel & distinct, 2 = coincident.
int lineRelation(const Point& a, const Point& b,
                 const Point& c, const Point& d) {
    Point r = b - a, s = d - c;
    if (cmp(cross(r, s)) != 0) return 0;        // directions differ -> one crossing
    if (cmp(cross(r, c - a)) == 0) return 2;    // c on line 1 -> same line
    return 1;                                    // parallel, distinct
}
Two macros that look identical to fields but are not. #define X real() means v.X expands to v.real(); #define Y imag() likewise. rot is a typed helper function, never a macro — and never named rotate (which collides with std::rotate).

⚠ 14 — Common Pitfalls / Bugs

  • P1 — Handling only the proper-crossing case. Code that returns o1 * o2 < 0 && o3 * o4 < 0 and stops will miss every endpoint touch, T-junction, and collinear overlap. Add the four pointOnSegment guards. (And do not use the looser o1 != o2 && o3 != o4 as the "strict" test — it fires on zero orientations too.)
  • P2 — Forgetting the collinear-disjoint trap. All four orientations zero does not mean "intersect". Without the "between" check you would wrongly return true for two far-apart collinear segments.
  • P3 — Calling the preconditioned line intersection without checking parallelism. The denominator cross(r, s) is zero for parallel lines, so the division is undefined. Prefer the safe bool lineLineIntersection(...); if you need the legacy lineLineIntersectionPreconditioned, classify with lineRelation first.
  • P4 — Confusing the segment test with the line point. "Do segments meet?" is a boolean with boundary cases; "where do the lines cross?" is a formula with a parallelism precondition. They are different functions — do not fuse them.
  • P5 — Using the bounding box as the decision. Overlapping boxes do not imply intersection (it is only necessary). The orientations decide; the box only rejects early.
  • P6 — Raw == on orientation results from floats. Compare the cmp-derived −1/0/+1 values, which already absorb EPS; never compare raw cross products with ==.
  • P7 — Integer overflow in orient. With coordinates near 10⁹, the cross product overflows long long. Promote to __int128 (§16).
  • P8 — Assuming a unique intersection point on collinear overlap. Overlapping collinear segments meet in a whole sub-segment, not a single point. The safe lineLineIntersection returns false there (the lines are coincident); compute the overlap interval explicitly.

🧭 15 — Decision / Recognition Guide

Reflex rules (contest speed)

Proper cross = two sign changes. Zero orientation → pointOnSegment. Collinear disjoint = no endpoint on the other. Box overlap is necessary, not sufficient. Check parallel before line-line point. Never fuse the boolean test with the point formula.

Common confusion

"My test says two collinear segments far apart intersect." You implemented only the orientation signs and treated "all zero" as a hit. Collinear segments need the "between" check — add the four pointOnSegment guards, which reject the disjoint case.

🌿 16 — Variants & Extensions

16.1 — Exact integer segment intersection

For lattice inputs, replace the floating orient and the "between" dot with exact integer versions using __int128 intermediates. What changes: no EPS, fully exact verdicts. Why: intersection counting and polygon construction on lattices must be exact. When to use: any integer-coordinate problem.

// Exact integer orientation: +1 / -1 / 0   (Class 02 variant)
int orientInt(long long ax, long long ay,
              long long bx, long long by,
              long long px, long long py) {
    __int128 c = (__int128)(bx - ax) * (py - ay)
               - (__int128)(by - ay) * (px - ax);
    return (c > 0) - (c < 0);
}
// "between" on a collinear point, exact:
//   dot(p - a, p - b) <= 0
bool betweenInt(long long ax, long long ay,
                long long bx, long long by,
                long long px, long long py) {
    __int128 d = (__int128)(px - ax) * (px - bx)
               + (__int128)(py - ay) * (py - by);
    return d <= 0;
}
Overflow watch. With |coord| ≤ 10⁹, both the cross and the dot reach about 10¹⁸; the subtraction can exceed long long. The __int128 intermediates above are why the integer version is exact.

16.2 — Returning the intersection geometry, not just a boolean

Sometimes a problem wants the actual shared geometry: a single point for a proper crossing, or a sub-segment for collinear overlap. Compute the boolean first; if true, call the safe bool lineLineIntersection — it returns true with the unique point when the lines cross, and false when they are collinear (the shared geometry is then the overlap interval, found by intersecting the two collinear extents).

16.3 — Segment–ray and segment–line tests

The same orientation framework adapts to a segment against a ray or an infinite line by relaxing the "between" constraint on the unbounded object: a ray keeps the t ≥ 0 half (Class 03's pointOnRay), and an infinite line drops the bound entirely (pointOnLine). The straddle logic is unchanged; only the membership predicate on the unbounded side changes.

🚀 17 — Problem-Based Applications & Training Drills

Training drill — do two segments intersect? the canonical four-orientation pattern, not a contest problem

Scope. This is a training drill for the segment-intersection predicate, not a full contest problem. It isolates the four-orientation test plus collinear handling before later classes use it inside polygon clipping and sweep-line routines.

Story. Given two segments [a, b] and [c, d] per query, print whether they intersect (touching counts as intersecting).

Abstraction. This is exactly segmentsIntersect(a, b, c, d): four orientations decide the proper case, and four guarded pointOnSegment checks decide the collinear / touching cases.

Why this geometry model fits. "Cross" is mutual straddling (§8.1); "touch / overlap" is an endpoint of one lying on the other (§8.2). The two mechanisms together are complete and division-free.

Algorithm. Read both segments; reject early if the bounding boxes are disjoint; evaluate the four orientations exactly with __int128; apply the strict proper-crossing rule o1 * o2 < 0 && o3 * o4 < 0, then the four collinear guards.

#include <bits/stdc++.h>
using namespace std;

// exact integer orientation: +1 left, -1 right, 0 collinear
int orientInt(long long ax, long long ay,
              long long bx, long long by,
              long long px, long long py) {
    __int128 c = (__int128)(bx - ax) * (py - ay)
               - (__int128)(by - ay) * (px - ax);
    return (c > 0) - (c < 0);
}

// is collinear point p on the closed segment [a, b]?
bool onSeg(long long ax, long long ay,
           long long bx, long long by,
           long long px, long long py) {
    // assumes collinearity already checked by the caller
    __int128 d = (__int128)(px - ax) * (px - bx)
               + (__int128)(py - ay) * (py - by);
    return d <= 0;   // "between": dot(p - a, p - b) <= 0
}

// 1-D interval overlap (necessary box pre-filter)
bool overlap1D(long long lo1, long long hi1, long long lo2, long long hi2) {
    if (lo1 > hi1) swap(lo1, hi1);
    if (lo2 > hi2) swap(lo2, hi2);
    return max(lo1, lo2) <= min(hi1, hi2);
}

bool segmentsIntersect(long long ax, long long ay, long long bx, long long by,
                       long long cx, long long cy, long long dx, long long dy) {
    // fast reject: disjoint bounding boxes cannot intersect
    if (!overlap1D(ax, bx, cx, dx) || !overlap1D(ay, by, cy, dy)) return false;

    int o1 = orientInt(ax, ay, bx, by, cx, cy);
    int o2 = orientInt(ax, ay, bx, by, dx, dy);
    int o3 = orientInt(cx, cy, dx, dy, ax, ay);
    int o4 = orientInt(cx, cy, dx, dy, bx, by);

    if (o1 * o2 < 0 && o3 * o4 < 0) return true;          // strict proper crossing

    if (o1 == 0 && onSeg(ax, ay, bx, by, cx, cy)) return true;
    if (o2 == 0 && onSeg(ax, ay, bx, by, dx, dy)) return true;
    if (o3 == 0 && onSeg(cx, cy, dx, dy, ax, ay)) return true;
    if (o4 == 0 && onSeg(cx, cy, dx, dy, bx, by)) return true;

    return false;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    long long ax, ay, bx, by, cx, cy, dx, dy;
    while (cin >> ax >> ay >> bx >> by >> cx >> cy >> dx >> dy) {
        cout << (segmentsIntersect(ax, ay, bx, by, cx, cy, dx, dy)
                 ? "YES" : "NO") << '\n';
    }
    return 0;
}

Complexity. O(1) per query.

Pitfalls. Use __int128 for the cross and the dot so coordinates near 10⁹ do not overflow. Do not forget the four collinear guards — without them, endpoint touches and collinear overlaps are missed and collinear-disjoint pairs are mis-handled.

Training drill — intersection point of two crossing lines non-parallel line-line point, not a contest problem

Scope. A pattern drill for the line-line intersection formula, not a standalone contest problem. It shows the cross-product solve and the mandatory parallelism guard that recur in polygon clipping (Class 09) and circle-line work (Class 06).

Story. Given two lines, each by two points, report their unique intersection point, or report that they are parallel or coincident.

Abstraction. Classify with lineRelation; if crossing, compute the point with the safe bool lineLineIntersection(a, b, c, d, ret) — a single call that returns false on parallel/coincident input instead of dividing by zero.

Why this geometry model fits. Solving a + t·r = c + u·s by crossing with s isolates t (§8.4); the denominator cross(r, s) is the parallelism detector, so classification and computation share one quantity.

Algorithm. Compute cross(r, s); if zero, decide coincident vs parallel; otherwise solve for t and emit the point.

#include <bits/stdc++.h>
using namespace std;

using T     = long double;
using Point = complex<T>;
#define X real()
#define Y imag()

const long double EPS = 1e-12L;
template<class U> int cmpFloat(const U& a, const U& b = U(0)) {
    long double d = (long double)a - (long double)b;
    return (d > EPS) - (d < -EPS);
}
template<class U> int cmp(const U& a, const U& b = U(0)) {
    if constexpr (is_floating_point_v<U>) return cmpFloat(a, b);
    else return (a > b) - (a < b);
}
T cross(const Point& a, const Point& b) { return imag(conj(a) * b); }

int lineRelation(const Point& a, const Point& b, const Point& c, const Point& d) {
    Point r = b - a, s = d - c;
    if (cmp(cross(r, s)) != 0) return 0;        // one crossing
    if (cmp(cross(r, c - a)) == 0) return 2;    // coincident
    return 1;                                    // parallel, distinct
}
// SAFE bool-return API: false on parallel/coincident lines.
bool lineLineIntersection(const Point& a, const Point& b,
                          const Point& c, const Point& d, Point& ret) {
    Point r = b - a, s = d - c;
    T denom = cross(r, s);
    if (cmp(denom) == 0) return false;          // parallel or coincident -> no unique point
    ret = a + r * (cross(c - a, s) / denom);
    return true;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    long double ax, ay, bx, by, cx, cy, dx, dy;
    while (cin >> ax >> ay >> bx >> by >> cx >> cy >> dx >> dy) {
        Point a(ax, ay), b(bx, by), c(cx, cy), d(dx, dy);
        int rel = lineRelation(a, b, c, d);
        if (rel == 1) { cout << "PARALLEL\n"; }
        else if (rel == 2) { cout << "COINCIDENT\n"; }
        else {
            Point p;
            // After lineRelation said 0 (one crossing) the safe call cannot fail,
            // but we still go through the bool API so copy-pasted snippets stay safe.
            if (lineLineIntersection(a, b, c, d, p))
                cout << fixed << setprecision(6) << p.X << ' ' << p.Y << '\n';
            else
                cout << "PARALLEL\n";       // defensive: should not be reached here
        }
    }
    return 0;
}

Complexity. O(1) per query.

Pitfalls. The safe bool lineLineIntersection returns false on parallel/coincident lines, so a copy-pasted snippet never divides by zero; even so, classify with lineRelation when the surrounding algorithm needs to distinguish parallel from coincident. As a worked check: lines through (0,0)–(4,4) and (0,4)–(4,0) cross at (2, 2).

📝 18 — Final Summary

The mental model

  • Proper cross = each segment straddles the other's line.
  • Four orientations o1..o4 decide that case.
  • Zero orientation = collinear flag → pointOnSegment.
  • Collinear-disjoint = no endpoint on the other.
  • Bounding box = cheap necessary pre-filter.
  • Line point = cross-product solve, needs non-parallel.

The hard rules

  • Handle proper, touching, and collinear — not just proper.
  • Collinear all-zero needs the "between" check.
  • Box overlap is necessary, never sufficient.
  • Classify parallelism before the line-line point.
  • Keep the boolean test and the point formula separate.

Class 04 composes the orientation side test (Class 02) and the point-on-segment predicate (Class 03) into a complete intersection toolkit. segmentsIntersect is the workhorse of polygon clipping and sweep-line algorithms; the safe bool lineLineIntersection (with the preconditioned variant lineLineIntersectionPreconditioned for hot loops) is reused for circle-line work (Class 06) and halfplane / polygon cutting (Class 09).

What's next. Class 05 builds triangle construction — locating a third point from given constraints — which becomes the geometric spine for the circle constructions (circumcircle, tangents) developed in Class 06.

Runtime Validation Checklist

Before using the intersection toolkit in a contest solution, verify each item below. The Class 01 numeric checklist, the Class 02 orientation checklist, and the Class 03 segment-predicate checklist all apply underneath.

Segment intersection

  • The bounding boxes are checked first; disjoint boxes reject immediately.
  • All four orientations o1..o4 are computed before any decision.
  • The strict proper-crossing branch o1 * o2 < 0 && o3 * o4 < 0 is checked first.
  • Each collinear branch is guarded by its matching oi == 0 and uses pointOnSegment.
  • Collinear-disjoint correctly returns false (the "between" test rejects it).
  • If only strict interior crossings are wanted, use only the strict branch o1 * o2 < 0 && o3 * o4 < 0.

Line intersection & classification

  • lineLineIntersection (safe bool API) reports false on parallel/coincident lines; the legacy preconditioned variant is only called after confirming the lines are not parallel.
  • lineRelation distinguishes crossing / parallel / coincident before any point is computed.
  • The bounding box is used only as a necessary pre-filter, never as the final verdict.
  • Integer inputs use the __int128 exact orient to avoid overflow.
