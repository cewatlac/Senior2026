Class 02 — Point Location & Orientation

The orientation toolkit. We build orient, the perpendicular helpers, rotation and reflection via complex numbers, and the heavy local-frame classifier ccwQuadrant — all on top of the dot/cross/cmp substrate from Class 01.

🎯 1 — Problem Definition

Class 01 measured things: distances, lengths, the sign of a value. Class 02 answers a different family of questions — where is a point relative to a direction? These are the predicates that drive nearly every higher-level algorithm in geometry:

Is point p to the left or right of the directed line a → b? Are three points collinear? Which quadrant of a local frame does p fall in? How do I rotate a point by an angle around any centre? How do I reflect a point across an arbitrary mirror line?

The single most important tool in this class is orient(a, b, p) — the signed side test. Convex hull, segment intersection, polygon cutting, and point-in-polygon are all built on it. Alongside it we develop the rotation and reflection isometries, because so many problems become trivial once you move the configuration into a convenient frame, solve, and move back.

Recall the four classic geometry bugs (Class 01). This class is where bug 1 ("wrong model") and bug 3 ("forgetting degeneracies") most often bite: using atan(y/x) instead of atan2(y, x), or feeding a zero-length vector into an orientation predicate. We confront both head-on.

🧠 2 — Intuition from Scratch

2.1 — The ASTC sign chart

Before any code, recall how the signs of sine, cosine, and tangent behave by quadrant. The mnemonic ASTC ("All Students Take Calculus") names which functions are positive in each quadrant, counter-clockwise from the first:

Diagram · ASTC — which trig functions are positive in each quadrant

Why does this matter for code? Because the sign of a dot product follows cos θ and the sign of a cross product follows sin θ (we proved both in Class 01). So the dot/cross signs are exactly the ASTC chart in disguise — and that is what lets us classify "which side" and "which quadrant" with two cheap multiplications instead of a slow, fragile atan.

2.2 — Slope is fragile; sign-of-cross is not

A beginner's instinct for "which side of a line" is to compute the slope. But slope is undefined for a vertical line (dx = 0 gives division by zero) and numerically unstable when nearly vertical. The cross product has neither problem: it is a single subtraction of two products, defined and stable for every direction. This is the same lesson as Class 01's "don't use y = mx + c for a vertical line", now applied to side tests.

2.3 — The degenerate zero vector has no direction

Every predicate in this class assumes its input vectors are non-zero. A vector of length 0 — for example when a == b so the "line" a → b collapses to a point — does not point anywhere. Asking "which side of a point is p on?" is meaningless. This is Class 01's observation O6 made concrete, and it is the most common silent bug in orientation code.

Diagram · The zero vector has no direction

👁 3 — Visual Mental Model

3.1 — Polar coordinates

A point can be named two ways. Cartesian uses (x, y); polar uses a radius r (distance from the origin) and an angle θ (measured CCW from the positive x-axis). The conversions are x = r·cos θ, y = r·sin θ, with r = |v| and θ = atan2(y, x).

Diagram · Polar coordinates (r, θ)

3.2 — The complex plane is the geometry plane

The single most powerful idea in this whole class: a 2D point (x, y) is a complex number x + yi. Addition is translation, multiplication by polar(1, θ) is rotation, and conjugation is reflection across the x-axis. By storing points as complex<long double> (the Class 01 type), we get rotation and reflection almost for free.

Diagram · A point as a complex number z = x + yi

🧮 4 — Formal Abstraction

4.1 — atan2(y, x), never atan(y / x)

The direction of a vector is its angle. The naive formula atan(y / x) fails twice over: it divides by zero when x = 0 (a vertical vector), and it collapses two genuinely different directions into one because atan only returns values in (−π/2, π/2) — it cannot tell (1, 1) from (−1, −1). The standard library function atan2(y, x) takes the two components separately, inspects their signs, and returns the correct angle in the full range (−π, π].

Diagram · atan2 covers all four quadrants; atan(y/x) does not

4.2 — The trig sum/difference bridge

In Class 01 we showed that dot(a, b) = |a||b| cos θ and cross(a, b) = |a||b| sin θ, where θ is the angle from a to b. Those came from the angle-difference identities:

cos(β - α) = cos α cos β + sin α sin β     ← dot
sin(β - α) = cos α sin β - sin α cos β     ← cross

The consequence we use constantly in this class: to recover the signed angle from a to b in the full range (−π, π], combine both into one atan2:

signedAngle(a, b) = atan2( cross(a, b), dot(a, b) )

This is exactly atan2(sin θ, cos θ) = θ, scaled by the (positive) product of magnitudes which cancels inside atan2.

4.3 — Complex numbers as geometric operators

We re-use the Class 01 type aliases. A point and a vector are both complex<long double>, with the convenience macros #define X real() and #define Y imag(). The complex operations carry geometric meaning:

Complex operationGeometric meaning
a + bTranslation of point a by vector b.
a * polar(1, θ)Rotation of a by angle θ (CCW) about the origin.
conj(a)Reflection of a across the x-axis.
a * i (i.e. a * Point(0,1))Rotation by exactly +90° — the basis of perpCCW.

4.4 — Euler's formula and rotation

Euler's formula e^{iθ} = cos θ + i sin θ says that the unit complex number at angle θ is polar(1, θ). Multiplying any complex number by it adds θ to its argument while leaving its magnitude unchanged — that is precisely a rotation. The Class 01 / Class 02 helper captures this:

Point rot(const Point& v, T t) { return v * polar((T)1, t); }
Why a typed helper and not a macro. As fixed in Class 01, rot is a real function, not a #define. A macro would text-substitute and hand std::polar a mixed long double / double pair (e.g. from rot(v, 0.5)), tripping template deduction. The typed parameter T t converts the argument at the call boundary. And the name is rot, never rotate — the latter collides with std::rotate.
Diagram · Euler rotation — multiplying z by e^{iθ} adds θ to its angle

💡 5 — Key Observations

  • O1 — The sign of the cross product is a side test. cross(b − a, p − a) > 0 means p is to the left of the directed line a → b; < 0 means right; = 0 means collinear. This single fact powers orient.
  • O2 — A rotation by 90° is multiplication by i. So (x, y) · i = (−y, x), which is exactly perpCCW. No trig, no polar call needed for the right-angle case.
  • O3 — Conjugation reflects across the x-axis, full stop. To reflect across any other line, first rotate the world so that line becomes the x-axis, conjugate, then rotate back. This "rotate–conjugate–rotate-back" pattern reflects across any line through the origin.
  • O4 — Reflection across an arbitrary line = translate + reflect-through-origin-line + translate-back. Move one point of the line to the origin, do the through-origin reflection, move back.
  • O5 — A local frame turns "which quadrant" into two sign tests. Project onto the local x-axis with dot and onto the local y-axis with cross; the two signs name the region. This is the engine of ccwQuadrant.
  • O6 — Reversing the line direction flips orient. orient(a, b, p) = −orient(b, a, p), because swapping a and b negates the cross product. This is Class 01's antisymmetry invariant I3 in action.

⚙ 6 — Algorithm / Method Explanation

6.1 — Perpendicular vectors

Rotating (x, y) by +90° gives (−y, x) — that is perpCCW. Rotating by −90° gives (y, −x) — that is perpCW. These are constant-time, trig-free, and exact on integer coordinates.

Diagram · perpCCW (+90°) and perpCW (−90°)

6.2 — orient(a, b, p)

One line: return the sign (via cmp) of cross(b − a, p − a). The result is +1 (left / CCW), −1 (right / CW), or 0 (collinear). Precondition: a ≠ b.

Diagram · orient(a, b, p) — left / right / collinear

6.3 — Rotation around an arbitrary centre

rotateAround(p, o, t) translates p so the centre o sits at the origin, applies rot, then translates back: o + rot(p − o, t).

6.4 — Reflection, three layers

We build reflection from the inside out:

  1. reflectXAxis(p) = conj(p) — reflect across the x-axis.
  2. reflectAroundLineThroughOrigin(p, ang) — rotate by −ang so the mirror aligns with the x-axis, conjugate, rotate back by +ang.
  3. reflectAroundLine(p, a, b) — translate by −a, reflect through the origin line in direction b − a (whose angle is atan2), translate back by +a.
Diagram · Conjugation reflects z = x + yi to conj(z) = x − yi across the x-axis
Diagram · Rotate–conjugate–rotate-back reflection across a tilted line

6.5 — The local-frame classifier ccwQuadrant

Given an anchor a and a direction point b, build a local frame whose +x axis runs along u = b − a and whose +y axis is 90° CCW from it. For the query point p let w = p − a. Then the local x-coordinate has the sign of dot(w, u) and the local y-coordinate has the sign of cross(u, w). The pair of signs names one of nine regions: the origin, the four axis rays, or the four open quadrants.

Diagram · ccwQuadrant — nine regions in the local frame at a along (b − a)

🔒 7 — Invariants

State invariants

  • I1 — orient is tri-state. It returns only −1, 0, +1 (it forwards a cmp result). Callers may switch on the sign.
  • I2 — Antisymmetry of orient. orient(a, b, p) = −orient(b, a, p), and likewise swapping the roles of the query reverses the side. This is inherited from cross's sign behaviour.
  • I3 — ccwQuadrant returns exactly one of nine values. The cascade is exhaustive: the two signs (each in {−1, 0, +1}) give nine combinations, each mapped to exactly one Pos2D value.

Transition invariants (isometries preserve structure)

  • I4 — Rotation preserves distances and angles. rot multiplies by a unit complex number, so |rot(v, t)| = |v| for every v and t. rotateAround inherits this about any centre.
  • I5 — Reflection is an involution. Reflecting twice across the same line returns the original point: reflectAroundLine(reflectAroundLine(p, a, b), a, b) = p (up to floating drift). Conjugation makes this exact for the x-axis, and the rotate-conjugate-rotate-back wrapper preserves it.
  • I6 — Points on the mirror are fixed. If p lies on the reflection line, the reflection returns p (again up to drift). For the x-axis, conj(x + 0i) = x + 0i exactly.

Termination invariants

Every function in this class is straight-line code: a constant number of arithmetic operations, one possible atan2, and one possible polar. No loops, no recursion. Each returns in O(1).

Failure mode — applicability. orient, ccwQuadrant, and reflectAroundLine all assume a ≠ b so that the direction vector is non-zero (Class 01 O6). They do not defensively check this — by family policy we trust callers and document the precondition. Feeding a = b yields a meaningless zero direction.

📜 8 — Proof of Correctness

8.1 — Why perpCCW(v) = (−y, x) rotates by +90°

Complex proof

Multiplying by i rotates by +90° (Euler: i = polar(1, π/2)). Compute it directly: (x + yi)·i = xi + yi² = −y + xi, whose components are (−y, x). That is exactly perpCCW.

Dot-product check

A +90° rotation must be perpendicular to the original and the same length. Perpendicularity: dot((x, y), (−y, x)) = x(−y) + y(x) = 0. Length: |(−y, x)| = √(y² + x²) = |(x, y)|. Both hold, and the cross product cross((x, y), (−y, x)) = x·x − y·(−y) = x² + y², which is > 0 for non-zero v, confirms the turn is CCW rather than CW.

8.2 — Why cross(b − a, p − a) is a signed side test

From Class 01, cross(u, w) = |u||w| sin φ, where φ is the angle from u to w, measured CCW. Here u = b − a is the line direction and w = p − a points from the line's base to the query. The magnitudes are positive, so the sign of the cross product is the sign of sin φ: positive when p is on the CCW (left) side, negative on the CW (right) side, zero when p is collinear (φ = 0 or π). Geometrically, |cross(u, w)| is twice the area of triangle a, b, p; a zero area means the three points are collinear.

8.3 — Why atan2(y, x) is quadrant-correct

atan2 receives the numerator y and denominator x separately, so it can read their individual signs. The sign of x tells it the half-plane (right vs left), the sign of y tells it up vs down, and together they place the angle in the correct quadrant of (−π, π]. By contrast atan(y/x) first divides — destroying the individual signs (since (−)/(−) = (+)/(+)) and dividing by zero on the y-axis — and then can only return a value in (−π/2, π/2), which covers only the right half-plane.

8.4 — Why rotate–conjugate–rotate-back reflects across a tilted line

Let the mirror pass through the origin at angle ang. The map z ↦ rot(z, −ang) rotates the whole plane so the mirror lands on the x-axis. In that frame, reflection across the x-axis is exactly conjugation: z ↦ conj(z). Finally z ↦ rot(z, +ang) rotates the plane back to its original orientation. Composing the three steps gives

reflectAroundLineThroughOrigin(p, ang) = rot( conj( rot(p, -ang) ), ang )

Each step is an isometry, so the composition is an isometry; the middle step is the only orientation-reversing one, which is what makes the whole map a reflection rather than a rotation. To reflect across a line that does not pass through the origin, conjugate the configuration by a translation: subtract one anchor point a, reflect through the origin line, add a back. This is reflectAroundLine, with the mirror's angle computed as atan2((b − a).Y, (b − a).X).

8.5 — Why the local-frame signs classify the region

The local frame at a uses u = b − a as its +x basis vector and perpCCW(u) as its +y basis vector. For the query offset w = p − a, the local x-coordinate is the projection onto u, whose sign is sign(dot(w, u)); the local y-coordinate is the projection onto perpCCW(u), and since dot(w, perpCCW(u)) = cross(u, w), its sign is sign(cross(u, w)). Two independent signs in {−1, 0, +1} give exactly nine cases, mapped one-to-one to the nine Pos2D values. The mapping is exhaustive and disjoint, which is precisely invariant I3.

⏱ 9 — Complexity Analysis

OperationTimeNotes
perpCCW / perpCWO(1)Two negations and a swap; no trig.

Everything in Class 02 is constant-time. When these helpers appear inside a loop — for example orient inside a convex-hull scan — the asymptotic cost belongs to the loop, not to these primitives.

📐 10 — Core Template Philosophy

  • Build on Class 01, do not duplicate it. This class assumes cmp, dot, cross, the type aliases, and the X/Y macros are already in scope. The master block below shows them for a standalone compile, but in practice you paste only the Class 02 additions on top of the Class 01 header.
  • Typed rot, never a rotate macro. rot is a helper function (Point rot(const Point& v, T t)). It avoids the std::polar mixed-type deduction trap, and the name dodges the collision with std::rotate.
  • Sign tests over angles. Prefer orient (a cross-product sign) to anything involving atan2 when you only need "which side". Reserve atan2 for when you genuinely need the angle's value.
  • Const-reference inputs, documented preconditions. Every helper takes const Point& and documents its a ≠ b requirement where relevant.
  • Unified naming. perpCCW, perpCW, orient, rot — consistent across the whole family so later classes read fluently.

💻 11 — Master / Base Implementation

Standalone, compiles with g++ -std=c++17. The first block repeats just enough of the Class 01 substrate for the file to build on its own; the Class 02 additions follow the divider.

#include <bits/stdc++.h>
using namespace std;

using T     = long double;
using Point = complex<T>;

#define X real()
#define Y imag()

const long double EPS = 1e-12L;
const T PI = acosl(-1.0L);

// ---- rotation helper (typed function, not a macro) ----------------
Point rot(const Point& v, T t) {
    return v * polar((T)1, t);
}

// ---- Class 01 substrate (cmp / dot / cross) -----------------------
template<class U>
int cmpInt(const U& a, const U& b = U(0)) { return (a > b) - (a < b); }
template<class U>
int cmpFloat(const U& a, const U& b = U(0)) {
    long double d = (long double)a - (long double)b;
    return (d > EPS) - (d < -EPS);
}
template<class U>
int cmp(const U& a, const U& b = U(0)) {
    if constexpr (is_floating_point_v<U>) return cmpFloat(a, b);
    else return cmpInt(a, b);
}
T dot  (const Point& a, const Point& b) { return real(conj(a) * b); }
T cross(const Point& a, const Point& b) { return imag(conj(a) * b); }

// ===================================================================
// CLASS 02 — orientation toolkit
// ===================================================================

// Perpendicular vectors. perpCCW rotates v by +90 degrees, perpCW by -90.
Point perpCCW(const Point& v) { return Point(-v.Y, v.X); }
Point perpCW (const Point& v) { return Point( v.Y, -v.X); }

// Orientation of point p relative to the directed line a -> b.
// +1 = p is to the LEFT (CCW), -1 = RIGHT (CW), 0 = collinear.
// Precondition: a != b (otherwise the "line" has no direction).
int orient(const Point& a, const Point& b, const Point& p) {
    return cmp(cross(b - a, p - a));
}

// Rotate point p by angle t (radians, CCW) around centre o.
Point rotateAround(const Point& p, const Point& o, T t) {
    return o + rot(p - o, t);
}

// Reflect a point across the x-axis: conjugation.
Point reflectXAxis(const Point& p) { return conj(p); }

// Reflect p across the line through the ORIGIN at angle ang.
// Rotate the world by -ang so the mirror lands on the x-axis,
// conjugate, then rotate back by +ang.
Point reflectAroundLineThroughOrigin(const Point& p, T ang) {
    return rot(conj(rot(p, -ang)), ang);
}

// Reflect p across the infinite line through points a and b.
// Translate so a is at the origin, reflect across the line through
// the origin in direction (b - a), then translate back.
// Precondition: a != b.
Point reflectAroundLine(const Point& p, const Point& a, const Point& b) {
    Point d = b - a;
    T ang = atan2l((long double)d.Y, (long double)d.X);
    return a + reflectAroundLineThroughOrigin(p - a, ang);
}

// ---- local-frame nine-region classifier ---------------------------
enum Pos2D {
    ORIGIN, POS_X_RAY, POS_Y_RAY, NEG_X_RAY, NEG_Y_RAY,
    QUAD_1, QUAD_2, QUAD_3, QUAD_4
};

// Classify where p lies in the local frame anchored at a with the
// local +x axis pointing along (b - a) and local +y axis 90 deg CCW.
// Precondition: a != b.
Pos2D ccwQuadrant(const Point& a, const Point& b, const Point& p) {
    Point u = b - a;            // local +x direction
    Point w = p - a;            // vector to classify
    int sx = cmp(dot(w, u));    // sign of local x-coordinate
    int sy = cmp(cross(u, w));  // sign of local y-coordinate (CCW positive)

    if (sx == 0 && sy == 0) return ORIGIN;
    if (sy == 0) return (sx > 0) ? POS_X_RAY : NEG_X_RAY;
    if (sx == 0) return (sy > 0) ? POS_Y_RAY : NEG_Y_RAY;
    if (sx > 0 && sy > 0) return QUAD_1;
    if (sx < 0 && sy > 0) return QUAD_2;
    if (sx < 0 && sy < 0) return QUAD_3;
    return QUAD_4;
}
Two macros that look identical to fields but are not. #define X real() means v.X expands to v.real(); #define Y imag() likewise. They are method calls dressed up as field access — pure CP convenience. Note rot is not in this list: it is a typed helper function, not a macro.

⚠ 14 — Common Pitfalls / Bugs

  • P1 — Using atan(y / x) for a direction. Divides by zero on the y-axis and collapses opposite directions. Always atan2(y, x) (or atan2l for long double).
  • P2 — Naming the rotation helper rotate. It collides with std::rotate from <algorithm>. Use rot — and as a typed function, not a macro (Class 01 fix).
  • P3 — Treating dot = 0 as "perpendicular" without checking for zero-length inputs. The zero vector has no direction (O6 / §2.3), so a zero dot product against it means nothing.
  • P4 — Feeding a = b into orient / ccwQuadrant / reflectAroundLine. The direction collapses; results are meaningless. Guard the caller.
  • P5 — Confusing math convention with screen convention. All proofs here assume the math convention: y points up and CCW is positive. On a screen canvas y points down, which flips the sign of every cross product and turns CCW into CW. Decide one convention and stick to it; if you must mix, negate consistently.
  • P6 — Using orient's integer result in a boolean context. if (orient(a, b, p)) is true for both left and right (any non-zero) and false only for collinear — rarely what you want. Compare against the explicit sign you mean.
  • P7 — Building ccwQuadrant-style logic with hand-rolled if/else and missing a case. There are nine regions. Use the two-sign cascade exactly as written, ending with an unconditional return for the last quadrant.
  • P8 — Comparing rotated coordinates with raw ==. A rotation by π leaves ~1e-16 noise. Route every comparison through cmp (Class 01).

🧭 15 — Decision / Recognition Guide

SituationReflex tool
I only need left / right / collinear of a point vs a directed line.orient(a, b, p)
I need quadrant or axis-ray classification in a local frame.ccwQuadrant(a, b, p)
I need a vector turned 90°.perpCCW (+90°) or perpCW (−90°)
I need to rotate a point around an arbitrary centre.rotateAround(p, o, t)

I need to reflect across an arbitrary line.reflectAroundLine(p, a, b)
I genuinely need the angle of a vector.atan2(v.Y, v.X) — never atan
I need the signed angle from a to b.atan2(cross(a, b), dot(a, b))

Reflex rules (contest speed)

Side test = sign of cross. +90° = perpCCW = (−y, x). Reflect = rotate · conj · rotate-back. Angle = atan2, never atan. Check a ≠ b before any direction test. No macro named "rotate".

Common confusion

"My orientations are all backwards." Almost always the y-down screen convention (P5). On a screen, increasing y goes downward, so a cross product that is "left" in math space reads as "right" on screen. Either render in a y-up frame, or accept the flip and negate every orient result consistently — never mix the two.

🌿 16 — Variants & Extensions

16.1 — Exact integer orient

For lattice problems with integer coordinates, the floating cross is unnecessary and lossy. Compute the cross product directly in integers and read off its sign. What changes: exact arithmetic, no EPS. Why: orientation must be exact for hulls and intersection on lattice points. When to use: any problem with integer input coordinates.

// Exact orientation on integer coordinates.
// Returns +1 (left/CCW), -1 (right/CW), 0 (collinear).
__int128 cross128(long long ax, long long ay,
                  long long bx, long long by) {
    return (__int128)ax * by - (__int128)ay * bx;
}

int orientInt(long long ax, long long ay,
              long long bx, long long by,
              long long px, long long py) {
    __int128 c = cross128(bx - ax, by - ay, px - ax, py - ay);
    return (c > 0) - (c < 0);
}
Overflow watch. With |coord| ≤ 10⁹, products can reach around 10¹⁸, and the subtraction can overflow long long. That is why the template above promotes the products to __int128 before taking the sign.

16.2 — Signed angle between two vectors

The trig bridge of §4.2 gives the signed angle in (−π, π] directly. What changes: one atan2 over the cross and dot. Why: it carries both magnitude and direction, unlike orient which is only a sign. When to use: total-turning / winding computations, angular sweeps (previewed in Class 07).

Precondition. Both input vectors must be non-zero. If either vector is zero, both cross and dot may be zero, and atan2(0,0) is not a meaningful angle.

// Precondition: a and b are non-zero vectors.
T signedAngle(const Point& a, const Point& b) {
    return (T)atan2l((long double)cross(a, b), (long double)dot(a, b));
}

16.3 — Fast integer ccwQuadrant for grids

On an integer grid the same two-sign cascade works with integer dot and cross, avoiding floating point entirely. The structure is identical; only the sign computation changes to exact integer arithmetic (with __int128 if coordinates are large). This is useful for grid-walk and rook/queen-direction problems.

🚀 17 — Problem-Based Applications & Training Drills

CSES — Point Location Test the canonical orient problem

Story. You are given a line that goes through the points p1 and p2 (in that direction), and a third point p3. For each test case, determine whether p3 is located on the left side of the line, on the right side of the line, or touches the line.

Abstraction. Output is one of three labels per test, driven entirely by orient(p1, p2, p3): +1 → "LEFT", −1 → "RIGHT", 0 → "TOUCH" (collinear).

Why this geometry model fits. "Which side of a directed line" is the literal meaning of the cross-product sign (§8.2). The slope approach would choke on vertical lines and lose precision; the cross product is one exact subtraction of two products and handles every direction uniformly.

Algorithm. Read the test count t. For each test read the six integers x1 y1 x2 y2 x3 y3, compute the orientation of (x3, y3) against the directed line (x1,y1) → (x2,y2) using a __int128 cross product (coordinates reach 10⁹, so the products reach 10¹⁸ and the difference would overflow long long), and print the matching label.

#include <bits/stdc++.h>
using namespace std;

// Exact cross product on integer coordinates using __int128 so that
// products of coordinates up to ~1e9 (reaching ~1e18) and their
// difference cannot overflow a 64-bit type.
__int128 cross128(long long ax, long long ay,
                  long long bx, long long by) {
    return (__int128)ax * by - (__int128)ay * bx;
}

// Orientation of p3 relative to the directed line p1 -> p2.
// +1 = LEFT (CCW), -1 = RIGHT (CW), 0 = TOUCH (collinear).
int orientInt(long long x1, long long y1,
              long long x2, long long y2,
              long long x3, long long y3) {
    __int128 c = cross128(x2 - x1, y2 - y1, x3 - x1, y3 - y1);
    return (c > 0) - (c < 0);
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int t;
    if (!(cin >> t)) return 0;
    while (t--) {
        long long x1, y1, x2, y2, x3, y3;
        cin >> x1 >> y1 >> x2 >> y2 >> x3 >> y3;
        int o = orientInt(x1, y1, x2, y2, x3, y3);
        if      (o > 0) cout << "LEFT\n";
        else if (o < 0) cout << "RIGHT\n";
        else            cout << "TOUCH\n";
    }
    return 0;
}

Complexity. O(1) per test case, O(t) overall.

Pitfalls. The coordinates can be as large as 10⁹ in magnitude, so the cross product must be computed in __int128 — a plain long long product overflows. Keep the convention straight: left of the directed line is CCW (positive cross).

Teaching snippet — local-frame classification with ccwQuadrant template-using, not standalone AC code

Scope. This is a template-using teaching snippet, not AC-ready standalone code. It assumes the Class 02 master implementation (the Point type, cmp, dot, cross, the Pos2D enum, and ccwQuadrant) is already pasted above it. It is included to show the nine-region classifier in a realistic usage shape before later classes lean on it.

Story. A robot sits at a facing toward b. For each landmark p, report which sector of the robot's own frame it falls in — straight ahead, behind, hard left, hard right, or one of the four diagonal quadrants.

Abstraction. This is exactly ccwQuadrant(a, b, p): the robot's facing direction is the local +x axis, "left" is local +y, and the nine Pos2D values are the sectors.

Why this geometry model fits. A heading plus a sideways axis is a local frame. The two sign tests (dot for ahead/behind, cross for left/right) classify the sector with no trig and no angle wraparound.

Algorithm. Build the local frame once per (a, b), then evaluate the two signs per landmark and map to a label.

// Usage snippet, not standalone AC-ready code.
// Assumes the Class 02 master implementation is already pasted above:
// Pos2D, Point, ccwQuadrant, cmp, dot, cross, and related helpers.
const char* label(Pos2D q) {
    switch (q) {
        case ORIGIN:    return "AT ROBOT";
        case POS_X_RAY: return "STRAIGHT AHEAD";
        case NEG_X_RAY: return "DIRECTLY BEHIND";
        case POS_Y_RAY: return "HARD LEFT";
        case NEG_Y_RAY: return "HARD RIGHT";
        case QUAD_1:    return "AHEAD-LEFT";
        case QUAD_2:    return "BEHIND-LEFT";
        case QUAD_3:    return "BEHIND-RIGHT";
        case QUAD_4:    return "AHEAD-RIGHT";
    }
    return "?";
}

int main() {
    long double ax, ay, bx, by;
    cin >> ax >> ay >> bx >> by;
    Point a(ax, ay), b(bx, by);   // precondition: a != b

    long double px, py;
    while (cin >> px >> py) {
        Point p(px, py);
        cout << label(ccwQuadrant(a, b, p)) << '\n';
    }
    return 0;
}

Complexity. O(1) per landmark.

Pitfalls. The robot must actually be facing somewhere: require a ≠ b, or the frame collapses and everything classifies as ORIGIN. For integer landmarks, switch to the exact integer variant (§16.3) to avoid EPS questions on the axes.

📝 18 — Final Summary

The mental model

  • orient = sign of cross(b−a, p−a) = which side.
  • perpCCW / perpCW = ±90° turns, trig-free.
  • rot = multiply by polar(1, θ) = rotate.
  • conj = reflect across the x-axis.
  • reflect = rotate · conj · rotate-back, then translate.
  • ccwQuadrant = two local-frame signs → one of nine regions.

The hard rules

  • Angle from a vector: atan2, never atan.
  • Side test: prefer orient over any angle.
  • Every direction test needs a ≠ b.
  • Watch the y-down screen convention.
  • No macro named rotate — typed rot.

Class 02 is the workhorse of the whole family. orient reappears in segment intersection (Class 04), polygon cutting and convex hull (Class 09); the rotation and reflection helpers feed ellipse local/world transforms (Class 07) and mirror-unfolding (Class 10). Master the cross-product side test here and the later classes read like applications of one idea.

What's next. Class 03 introduces line representations — slope-intercept, general implicit, and parametric forms — and the point-on-line / on-ray / on-segment predicates plus projection and point-to-line distance, all built on the orientation and dot/cross machinery established here.

📮 19 — Footer / Contact

Runtime Validation Checklist

Before using this toolkit in a contest solution, verify each item below. These are the failure modes specific to orientation and isometry code; the Class 01 numeric checklist still applies underneath.

Orientation & frame

  • orient uses cmp(cross(...)) — no slopes, no division.
  • Every direction test (orient, ccwQuadrant, reflectAroundLine) is called with a ≠ b.
  • ccwQuadrant's cascade ends with an unconditional return so all nine regions are covered.
  • Math convention (y up, CCW positive) is consistent with the problem's coordinate system; the y-down screen flip is handled deliberately if present.

Angles & isometries

  • Every vector angle uses atan2 / atan2l — never atan(y/x).
  • rot is the typed helper, not a rotate macro.
  • Comparisons on rotated/reflected coordinates go through cmp (rotation by π leaves ~1e-16 drift).

Spot checks (run mentally before submitting)

  • perpCCW(Point(3,1)) returns (−1, 3); perpCW(Point(3,1)) returns (1, −3).
  • orient((0,0),(4,0),(2,1)) = +1; orient((0,0),(4,0),(2,−1)) = −1; orient((0,0),(4,0),(2,0)) = 0.
  • rotateAround((1,0),(0,0),π/2) ≈ (0, 1).
  • reflectXAxis((3,2)) = (3, −2).
  • reflectAroundLine((3,0),(0,0),(1,1)) ≈ (0, 3) (mirror is the line y = x).
  • ccwQuadrant((0,0),(4,0),(2,3)) = QUAD_1; (−2,3) → QUAD_2; (0,3) → POS_Y_RAY; (0,0) → ORIGIN.
