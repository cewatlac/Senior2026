Class 10 — Closest Pair & Advanced Mixed Geometry

The series finale. The closest-pair problem asks for the two nearest points among many, and its elegant solution shows how sorting plus a sliding active window turns an O(n²) search into O(n log n). We compare squared distances to stay exact, then close with a decision map and recap that tie together every technique from Classes 1–10.

🎯 1 — Problem Definition

Given n points in the plane, the closest pair problem asks for the two points whose distance is smallest. The naive answer compares all pairs in O(n²); the classic result is that sorting and a localized search bring it down to O(n log n). Because only the comparison of distances matters, we work with squared distance and avoid sqrt entirely — keeping integer inputs exact.

Two nearest points among many? Minimum squared distance? The actual pair, with ties resolved? O(n²) brute vs O(n log n) sweep?

The key realization is that closeness is local: once we process points in sorted order and remember the best distance so far, a new point can only improve the answer by pairing with a handful of nearby points — not with the whole set. That locality is what the optimized algorithm exploits.

Exactness carry-over. For integer points, squared distance dx² + dy² is an exact integer; comparing squared distances needs no sqrt and no EPS. With coordinates up to 10⁹, a difference reaches 2·10⁹ and its square 4·10¹⁸, so accumulate in __int128 to stay safe. Only take a real sqrt at the very end if the problem demands the Euclidean distance itself.

🧠 2 — Intuition from Scratch

2.1 — Why all-pairs is wasteful

Comparing every pair tests n(n−1)/2 distances, but the vast majority involve points far apart that cannot possibly be the closest. If we already know some pair is distance d apart, any point more than d away in x alone is irrelevant — its full distance is at least its x-gap. So most comparisons are wasted, and a smarter method ignores them.

2.2 — Sort, then keep a moving window

Sort the points by x. Sweep left to right keeping a running best distance d. Maintain an active set of recently seen points whose x-coordinate is within d of the current point; anything farther back in x can be dropped because it can never beat d. For the new point, only the active points within a vertical band of height 2d can improve the answer — and there are only a constant number of those.

2.3 — Compare squared, not rooted

Distances are only ever compared, never added, so we compare their squares. The squared distance of integer points is an exact integer, so the entire closest-pair computation can run without a single floating-point operation — converting to a real distance is a final, optional step.

🧮 4 — Formal Abstraction

4.1 — Squared distance

The squared Euclidean distance between a and b is

dist2(a, b) = (a.x − b.x)² + (a.y − b.y)²

Since d(a,b) ≤ d(c,e) ⟺ dist2(a,b) ≤ dist2(c,e) for non-negative distances, every comparison the algorithm makes can use the squared form. The real distance is √dist2, computed only if the output requires it.

4.2 — The active set and its window

Let d² be the best squared distance found so far and d = ⌊√d²⌋. Sweeping by x, the active set holds every earlier point whose x-gap to the current point is below d. For the current point p, only active points with y ∈ [p.y − d, p.y + d] can improve the best — a window of width d in x and height 2d in y.

Diagram · The sweep-line active window

4.3 — Why the y-range is bounded

Within the active window, all earlier points are mutually at least d apart (otherwise we would have updated the best). Packing points that are pairwise d-separated into a d × 2d rectangle admits only a constant number — at most a handful — so each new point triggers only O(1) distance checks.

Diagram · y-range candidate filtering inside the window

💡 5 — Key Observations

  • O1 — Compare squares, never roots. Distance ordering equals squared-distance ordering, so the whole algorithm is integer-exact for integer points.
  • O2 — Closeness is local. A point can only improve the best by pairing within a small window; far points are provably irrelevant.
  • O3 — Only O(1) candidates per point. Mutually d-separated points fit only a constant number into a d × 2d box, so each step is constant amortized work.
  • O4 — Duplicates make the answer 0. Two identical points have squared distance 0, the minimum possible, and the algorithm finds it immediately.
  • O5 — Overflow, not precision, is the risk. Squared distances of large coordinates can exceed 64 bits; accumulate in __int128 rather than rounding.
  • O6 — The active set is a sorted-by-y structure. Maintaining it as a balanced set lets the y-window be queried and pruned in logarithmic time.

⚙ 6 — Algorithm / Method Explanation

6.1 — Brute force O(n²) baseline

Try every pair, track the minimum squared distance. It is trivially correct and the right choice for small n (up to a few thousand), and it serves as the oracle for testing the optimized version.

6.2 — Sweep-line with an active set

Sort by x. Keep an active set ordered by y. For each point in x-order: drop active points whose x-gap squared already exceeds the best; query the active set for the y-window [y−d, y+d] and test those candidates; then insert the current point. Each point is inserted and removed once, and triggers O(1) distance checks, for O(n log n) overall (dominated by the sort and the set operations).

Diagram · Sweep-line: drop expired, query window, insert

6.3 — The divide-and-conquer alternative

An equivalent O(n log n) method splits the sorted points at a middle line, solves each half recursively, then checks a strip of width 2d around the line. Within the strip, sorting by y means each point need only compare with a constant number of following points — the same packing bound as the sweep. Both methods rest on the identical locality fact.

Diagram · Divide-and-conquer strip around the middle line

6.4 — Duplicate points and zero distance

If two points coincide, their squared distance is 0. Both the brute force and the sweep detect this through the ordinary distance test. Once the optimized sweep sees best = 0, it returns immediately because no smaller distance exists; this preserves the O(n log n) guarantee even when many points are identical.

Diagram · Duplicate points give zero distance

6.5 — Integer squared-distance overflow

With coordinates up to ±10⁹, a coordinate difference reaches 2·10⁹, whose square is 4·10¹⁸ — already near the long long ceiling of about 9.2·10¹⁸, and the sum dx² + dy² can reach 8·10¹⁸. Use __int128 to keep the squared distance exact.

Diagram · Integer squared-distance overflow thresholds

🔒 7 — Invariants

State invariants

  • I1 — Best is the true minimum over processed points. After handling the first i points, best equals the smallest squared distance among them.
  • I2 — The active set holds exactly the x-relevant points. Every point with x-gap below d to the current point is present; everything farther back has been removed.
  • I3 — All comparisons are squared and exact. No sqrt enters the comparison path; squared distances are integer-exact in __int128.

Transition invariants

  • I4 — Each point enters and leaves the active set once. Insertion happens when reached in x-order; removal happens when the sweep passes beyond its x-window — never twice.
  • I5 — Only window candidates are tested. A new point compares only against active points in its y-band, a constant number, preserving the amortized bound.

Termination invariants

The sweep visits each point once; the active set sees one insert and one erase per point. With a balanced set the total work is O(n log n). The brute force is two finite nested loops. Both always terminate.

Failure mode — preconditions. Closest pair needs at least two points; with fewer, there is no pair and the routine returns a sentinel (here a negative squared distance). Duplicate points are valid input and yield distance 0.

📜 8 — Proof of Correctness

8.1 — Squared distance preserves ordering

For non-negative reals, u ≤ v ⟺ u² ≤ v². Distances are non-negative, so comparing squared distances yields the identical ordering and the same minimizing pair. Hence replacing distance by squared distance never changes which pair is closest, while keeping integer inputs exact.

8.2 — Far points cannot be the closest

If the current best squared distance is d², any point whose x-gap to p exceeds d has full squared distance at least (x-gap)² > d², so it cannot improve the best. Dropping such points from the active set therefore loses no candidate closer than the current best.

8.3 — Only a constant number of candidates per point

All earlier points in the active window are pairwise at least d apart, because any closer pair would already have lowered d. A region of width d and height 2d can contain only a bounded number of points that are mutually d-separated — partitioning the rectangle into cells of side d/2, each cell holds at most one such point, giving a small constant bound. Therefore each new point performs O(1) distance checks, and the sweep is O(n log n) dominated by sorting and set operations.

8.4 — Completeness of the sweep

Consider the true closest pair (a, b) with a earlier in x-order. When the sweep reaches b, point a is still active because its x-gap to b is at most their Euclidean distance, which is no larger than the current best distance threshold. Also, a lies in b's y-window because its y-gap is at most their Euclidean distance. So a is among the candidates tested against b, and the pair's distance is recorded. No closer pair is ever missed.

⏱ 9 — Complexity Analysis

The leap from O(n²) to O(n log n) is the locality argument: sorting exposes x-order, and the active window confines each point to a constant number of comparisons. For small n, brute force is simpler and just as fast in practice.

📐 10 — Core Template Philosophy

  • Compare squared distances. Ordering is preserved, sqrt is avoided, and integer inputs stay exact.
  • Sort once, sweep once. x-order plus a y-ordered active set turns a quadratic search into a near-linear one.
  • Prune by the x-gap, query by the y-band. Drop points too far back in x; test only the constant y-window.
  • Let the distance test detect duplicates, then stop. Identical points give squared distance 0; once found, return immediately because the answer cannot improve.
  • Guard the squared-distance width. Use __int128 once coordinates exceed a few times 10⁴.
  • Root only at the end. Convert to a real distance solely if the output demands it.

💻 11 — Master / Base Implementation

Library/template block; compiles as a translation unit with g++ -std=c++17 -c, or after adding a main. Everything is integer-exact: squared distances are carried in __int128, and no sqrt appears in the comparison path. The brute force serves as an oracle for the sweep.

#include <bits/stdc++.h>
using namespace std;

// ===================================================================
// CLASS 10 — closest pair of points (exact integer).
// Compare SQUARED distances so no sqrt is needed; the squared distance
// of integer points is exact. Carry it in __int128 because a coordinate
// difference up to ~2e9 squares to ~4e18, and dx*dx + dy*dy can reach
// ~8e18 — close to the long long ceiling.
// ===================================================================
using ll  = long long;
using lll = __int128_t;

struct Pt { ll x, y; };

// exact squared distance, safe in __int128.
lll dist2(const Pt& a, const Pt& b){
    lll dx = (lll)a.x - b.x;
    lll dy = (lll)a.y - b.y;
    return dx*dx + dy*dy;
}

// ---------- brute force O(n^2) baseline (oracle) -------------------
// Returns the minimum squared distance, or -1 for fewer than 2 points.
lll closestPairBrute(const vector<Pt>& p){
    int n = (int)p.size();
    lll best = -1;
    for (int i = 0; i < n; i++)
        for (int j = i + 1; j < n; j++){
            lll d = dist2(p[i], p[j]);
            if (best < 0 || d < best) best = d;
        }
    return best;
}

// floor of the integer square root of a __int128 value (for the window width).
static ll floorSqrtLL(lll v){
    if (v <= 0) return 0;
    long double r = sqrtl((long double)v);
    ll s = (ll)r;
    while (s > 0 && s * (lll)s > v) s--;
    while ((s + 1) * (lll)(s + 1) <= v) s++;
    return s;                       // floor(sqrt(v))
}

// ---------- optimized O(n log n) sweep line ------------------------
// Sort by x. Keep an active multiset ordered by (y, x). For each point,
// drop active points whose x-gap squared already exceeds best, then test
// only the active points within the y-window [y - d, y + d], where
// d = floor(sqrt(best)). Each point is inserted and erased once, and only
// a constant number of candidates are tested (proof 8.3).
lll closestPairSweep(vector<Pt> p){
    int n = (int)p.size();
    if (n < 2) return -1;
    sort(p.begin(), p.end(), [](const Pt& a, const Pt& b){
        return a.x != b.x ? a.x < b.x : a.y < b.y;
    });
    auto cmpY = [](const Pt& a, const Pt& b){
        return a.y != b.y ? a.y < b.y : a.x < b.x;
    };
    multiset<Pt, decltype(cmpY)> active(cmpY);
    lll best = dist2(p[0], p[1]);
    if (best == 0) return 0;          // duplicate points: minimum possible distance
    active.insert(p[0]);
    int left = 0;
    for (int i = 1; i < n; i++){
        ll d = floorSqrtLL(best);
        // drop points whose x-gap alone already exceeds the best distance.
        while (left < i &&
               (lll)(p[i].x - p[left].x) * (p[i].x - p[left].x) > best){
            active.erase(active.find(p[left]));
            left++;
        }
        // test only the y-window [p[i].y - d, p[i].y + d].
        Pt lo{LLONG_MIN / 2, p[i].y - d};
        for (auto it = active.lower_bound(lo);
             it != active.end() && it->y <= p[i].y + d; ++it){
            lll dd = dist2(*it, p[i]);
            if (dd < best){
                best = dd;
                if (best == 0) return 0;      // cannot improve beyond zero
            }
        }
        active.insert(p[i]);
    }
    return best;                    // minimum squared distance
}

// ---------- closest pair returning the pair too --------------------
// Fills (ra, rb) with the closest pair. Returns the squared distance, or -1
// if fewer than two points (in which case ra and rb are left unchanged).
// Tie behavior: the first pair that achieves the minimum during the sweep is
// kept (ties are not broken further).
lll closestPairWithPoints(vector<Pt> p, Pt& ra, Pt& rb){
    int n = (int)p.size();
    if (n < 2) return -1;             // no pair; ra, rb unchanged
    sort(p.begin(), p.end(), [](const Pt& a, const Pt& b){
        return a.x != b.x ? a.x < b.x : a.y < b.y;
    });
    auto cmpY = [](const Pt& a, const Pt& b){
        return a.y != b.y ? a.y < b.y : a.x < b.x;
    };
    multiset<Pt, decltype(cmpY)> active(cmpY);
    lll best = dist2(p[0], p[1]);
    ra = p[0]; rb = p[1];
    if (best == 0) return 0;          // duplicate points: pair already recorded
    active.insert(p[0]);
    int left = 0;
    for (int i = 1; i < n; i++){
        ll d = floorSqrtLL(best);
        while (left < i &&
               (lll)(p[i].x - p[left].x) * (p[i].x - p[left].x) > best){
            active.erase(active.find(p[left]));
            left++;
        }
        Pt lo{LLONG_MIN / 2, p[i].y - d};
        for (auto it = active.lower_bound(lo);
             it != active.end() && it->y <= p[i].y + d; ++it){
            lll dd = dist2(*it, p[i]);
            if (dd < best){
                best = dd; ra = *it; rb = p[i];
                if (best == 0) return 0;      // cannot improve beyond zero
            }
        }
        active.insert(p[i]);
    }
    return best;
}
Why squared distance, not distance. The algorithm only ever compares distances, and ordering is preserved by squaring, so sqrt is never needed for correctness — only for a final human-readable output. Staying squared keeps integer inputs exact and removes the one place floating point could corrupt the answer.
One rotation helper, never a macro. This integer class does not rotate, but the family convention still holds: any rotation is the typed helper Point rot(const Point& v, T t), never a rotate macro (which would shadow std::rotate).

⚠ 14 — Common Pitfalls / Bugs

  • P1 — Taking sqrt inside the comparison loop. It is slow and introduces rounding; compare squared distances and root only at the very end if needed.
  • P2 — 64-bit overflow on the squared distance. Large coordinates square past the long long range; promote to __int128 before multiplying.
  • P3 — Forgetting to prune the active set by x. Without dropping far-back points, the active set grows and the y-window scan is no longer constant — the method degrades to quadratic.
  • P4 — Scanning the whole active set instead of the y-window. Query only [y−d, y+d]; iterating all active points loses the complexity guarantee.
  • P5 — A floating window width that is too small. Use ⌊√best⌋ corrected to an exact integer; a rounded-down estimate that is too small could skip a valid candidate.
  • P6 — Continuing after finding duplicate points. Equal points give squared distance 0 through the ordinary test, but the optimized sweep should return immediately once best == 0; otherwise many identical points can cause unnecessary quadratic scanning.
  • P7 — Mixing exact and EPS comparisons. Closest pair on integer points is fully exact; do not introduce an EPS band that belongs to floating-point geometry like the polygon cut of Class 09.

The whole series reduces to recognizing which tool a problem calls for. This map routes a problem from its objects to its technique.

Reflex rules (contest speed)

Nearest two points ⇒ closest pair (sweep). Small n ⇒ brute O(n²) is fine. Compare squared distances, root last. Tightest wrap ⇒ convex hull (Class 09). Lattice counts ⇒ Pick (Class 08). Big coords ⇒ __int128 squared distance.

Common confusion

"Closest pair needs the actual distance, so I must use sqrt and floating point." No — the pair that minimizes distance is the pair that minimizes squared distance, so the whole search is integer-exact. Take a single sqrt at the end only if the problem prints the Euclidean distance.

🌿 16 — Variants & Extensions

16.1 — Divide-and-conquer closest pair

Sort by x, split at the median, recurse on each half for their best distances d, then merge by checking the strip of width 2d around the dividing line, where points sorted by y compare only with a constant number of successors. When to use: a classic alternative to the sweep with the same O(n log n) bound; the strip bound is the same packing argument.

// Strip check sketch (after recursive halves give best d):
// keep points within |x - midX| < d, sort the strip by y, and for each
// point compare only with the next few (a bounded constant) points whose
// y-gap is below d. Update best from these cross-pairs.

16.2 — Returning the actual Euclidean distance

When the output is the real distance rather than its square, take a single square root of the final answer. When to use: problems that print the distance to a fixed precision.

// real distance from the exact squared distance, computed once at the end.
long double closestPairDistance(const vector<Pt>& p){
    lll d2 = closestPairSweep(p);          // exact squared distance
    if (d2 < 0) return -1;                  // fewer than 2 points
    return sqrtl((long double)d2);
}

16.3 — Mixed-geometry composition

Hard problems chain the series' tools: take the convex hull of a point set (Class 09), compute its area by the shoelace formula (Class 08), test whether query points fall inside it (orientation, Classes 02–04), or find the closest pair among the hull vertices (this class). When to use: multi-step geometry where each stage feeds the next; recognizing the composition is the skill the decision map trains.

16.4 — Mirror / reflection unfolding

Many "shortest path bouncing off a wall" problems collapse to a single straight line once you reflect the target across the wall instead of bending the ray. The reflected target T' together with the source S forms a straight segment whose crossing point with the mirror is the reflection point — and its length is the path length you were after.

The picture.
Before unfolding: S → mirror → T (a bent path).
After unfolding: reflect T across the mirror to get T'; the segment S → T' is straight, and its intersection with the mirror is the bounce point.

The complex-number form of "reflect p across the line through a with direction b − a" is a one-liner: divide by the direction (rotates the line onto the real axis), conjugate (flip across the real axis), multiply back. It is much cleaner than the dot/projection version when Point is std::complex.

#include <bits/stdc++.h>
using namespace std;

using T     = long double;
using Point = complex<T>;

// Reflect p across the infinite line through a in direction (b - a).
// Precondition: a != b. The result is exact for floating arithmetic up
// to the usual rounding of one division and one multiplication.
Point reflectPointLineFast(const Point& p, const Point& a, const Point& b) {
    Point m = b - a;                         // line direction (must be non-zero)
    return a + conj((p - a) / m) * m;        // unfold around the line
}

Preconditions and caveats.

  • Direction must be non-zero. If a == b the line is undefined; the division by m blows up. Validate upstream.
  • Unfolding does not validate the bounce point. After producing the straight segment S → T', you still need to verify that its intersection with the actual mirror lies within the mirror segment, not just the supporting line. Use the segment routines from Class 04 / 58.
  • Repeated reflections magnify error. Each reflection is a complex multiplication and division — a few are fine, but unfolding across dozens of mirrors compounds rounding. Use long double and the EPS-aware comparators from Class 01 / 56.
  • The mirror must be a line, not an arbitrary curve. This trick is specific to straight-edged reflectors (polygon edges, walls).

Problem-use sketch. For UVA 1286 (and most "laser bouncing off a polygon edge" tasks), iterate over candidate reflection sequences. For each candidate, unfold the target across the chosen edges one by one (so the final reflected target T''… sees a straight ray from S), then walk back through the unfoldings checking that each segment really crosses the corresponding mirror segment. The candidate that survives every intersection test is the valid reflected path; its length is dist(S, T_unfolded).

16.5 — Canonical integer line and segment splitting

When coordinates are integers and you need to compare infinite lines exactly (group collinear segments, build a hash map keyed by line, decide whether two edges support the same line), the float form ax + by + c = 0 with floating-point comparison is brittle. The canonical form normalises the triple (A, B, C) so that each line has exactly one representative: divide by gcd(|A|, |B|, |C|), then fix the sign of the leading non-zero coefficient. After that, two lines are equal iff their canonical triples are equal — perfect for std::map, std::set, or sort-and-unique passes.

#include <bits/stdc++.h>
using namespace std;

// Canonical integer line: A*x + B*y + C = 0 in fully reduced form.
struct IntLine {
    long long A, B, C;
    bool operator<(const IntLine& other) const {
        if (A != other.A) return A < other.A;
        if (B != other.B) return B < other.B;
        return C < other.C;
    }
    bool operator==(const IntLine& other) const {
        return A == other.A && B == other.B && C == other.C;
    }
};

// Build the unique canonical line through two distinct integer points.
// Precondition: (x1, y1) != (x2, y2). Identical points do NOT define a line.
IntLine canonicalLine(long long x1, long long y1, long long x2, long long y2) {
    long long dx = x1 - x2;
    long long dy = y1 - y2;
    long long A  = dy;
    long long B  = -dx;
    long long C  = -A * x1 - B * y1;          // plug (x1, y1) into A*x + B*y + C = 0

    // Reduce by gcd so each line has one canonical triple.
    long long g = __gcd(llabs(A), __gcd(llabs(B), llabs(C)));
    if (g) { A /= g; B /= g; C /= g; }

    // Fix the sign of the leading non-zero coefficient to remove the +/- ambiguity.
    if (A < 0 || (A == 0 && B < 0) || (A == 0 && B == 0 && C < 0)) {
        A = -A; B = -B; C = -C;
    }
    return { A, B, C };
}

Use it when:

  • Coordinates are integers and you want exact equality of infinite lines (no EPS).
  • You need to group collinear segments — every segment's supporting line gets the same canonical key, so a single map<IntLine, vector<Segment>> bucket-sorts them.
  • You need to deduplicate edges in a polygon arrangement before counting boundary uses.

Do not use it when:

  • Coordinates are floating-point — there is no canonical bit pattern; use EPS-aware comparators instead.
  • You need tolerant equality (two near-parallel lines that you want to treat as equal): canonicalisation only collapses exactly identical lines.

Segment splitting on shared canonical lines

An arrangement that uses canonical lines as the grouping key lets you split overlapping collinear segments into elementary pieces exactly. The pattern (used in problems like Kattis triangularclouds) is:

  1. Compute the canonical line for every segment.
  2. Bucket all endpoints by canonical line.
  3. Within a bucket, sort endpoints along the line (e.g. by lexicographic (x, y) if the line is non-vertical, otherwise by y).
  4. For each original segment, split it into the elementary sub-segments between consecutive bucket points that fall inside it.
  5. Count how many times each elementary piece is used. Pieces used once are boundary; pieces used twice (or more) are internal.
Picture.
Original segments on one canonical line:
A──────────────────D
     B──────C
Collected sorted points: A — B — C — D. Elementary pieces: AB, BC, CD.

Edge cases to remember.

  • Identical endpoints. A zero-length "segment" does not define a line. Filter or skip such inputs before calling canonicalLine.
  • Overflow. A*x1 + B*y1 can overflow long long when coordinates approach 10⁹ and slopes are large. Promote to __int128 for the multiplication step if your inputs need it.
  • Mixed precision. Never canonicalise floating coordinates by rounding them to integers first; the rounding itself destroys collinearity.
  • Direction is lost. Two opposite directed segments that share the same supporting line produce the same canonical triple — by design. Track direction separately if it matters.

Training drill — closest pair squared distance not a contest problem

Scope. A drill for the integer closest-pair sweep, not a standalone contest problem.

Story. Read n integer points and output the minimum squared distance between any two.

Abstraction. Run closestPairSweep and print its exact squared-distance result.

Why this geometry model fits. Only comparisons matter, so squared distance is exact and sufficient (§8.1), and the sweep is O(n log n).

Algorithm. Read points; sweep; print the squared distance (or NO_PAIR for fewer than two).

#include <bits/stdc++.h>
using namespace std;
using ll  = long long;
using lll = __int128_t;

struct Pt { ll x, y; };

lll dist2(const Pt& a, const Pt& b){
    lll dx = (lll)a.x - b.x, dy = (lll)a.y - b.y;
    return dx*dx + dy*dy;
}
static ll floorSqrtLL(lll v){
    if (v <= 0) return 0;
    long double r = sqrtl((long double)v);
    ll s = (ll)r;
    while (s > 0 && s * (lll)s > v) s--;
    while ((s + 1) * (lll)(s + 1) <= v) s++;
    return s;
}
lll closestPairSweep(vector<Pt> p){
    int n = (int)p.size();
    if (n < 2) return -1;
    sort(p.begin(), p.end(), [](const Pt& a, const Pt& b){
        return a.x != b.x ? a.x < b.x : a.y < b.y;
    });
    auto cmpY = [](const Pt& a, const Pt& b){
        return a.y != b.y ? a.y < b.y : a.x < b.x;
    };
    multiset<Pt, decltype(cmpY)> active(cmpY);
    lll best = dist2(p[0], p[1]);
    if (best == 0) return 0;          // duplicate points: minimum possible distance
    active.insert(p[0]);
    int left = 0;
    for (int i = 1; i < n; i++){
        ll d = floorSqrtLL(best);
        while (left < i &&
               (lll)(p[i].x - p[left].x) * (p[i].x - p[left].x) > best){
            active.erase(active.find(p[left]));
            left++;
        }
        Pt lo{LLONG_MIN / 2, p[i].y - d};
        for (auto it = active.lower_bound(lo);
             it != active.end() && it->y <= p[i].y + d; ++it){
            lll dd = dist2(*it, p[i]);
            if (dd < best){
                best = dd;
                if (best == 0) return 0;      // cannot improve beyond zero
            }
        }
        active.insert(p[i]);
    }
    return best;
}
string i128(lll v){
    if (v == 0) return "0";
    bool n = v < 0; if (n) v = -v;
    string s;
    while (v){ s += char('0' + (int)(v % 10)); v /= 10; }
    if (n) s += '-';
    reverse(s.begin(), s.end());
    return s;
}
int main(){
    ios::sync_with_stdio(false);
    cin.tie(nullptr);
    int n;
    if (!(cin >> n)) return 0;
    vector<Pt> p(n);
    for (auto& q : p) cin >> q.x >> q.y;
    if (n < 2){ cout << "NO_PAIR\n"; return 0; }
    cout << i128(closestPairSweep(p)) << "\n";
    return 0;
}

Complexity. O(n log n), integer-exact.

Output meaning. NO_PAIR for fewer than two points; otherwise the exact minimum squared distance (0 when points coincide).

Training drill — closest pair with actual distance output not a contest problem

Scope. A drill that also prints the real Euclidean distance and the achieving pair, not a standalone contest problem.

Story. Read n integer points; output the squared distance, the real distance, and the closest pair.

Abstraction. Run the pair-returning sweep, then take a single sqrt of the exact squared distance for the printed value.

Why this geometry model fits. The search stays integer-exact; only the final output needs a root (§16.2).

Algorithm. Sweep keeping the best pair; print squared distance, its root, and the two points. Ties keep the first pair found.

#include <bits/stdc++.h>
using namespace std;
using ll  = long long;
using lll = __int128_t;

struct Pt { ll x, y; };

lll dist2(const Pt& a, const Pt& b){
    lll dx = (lll)a.x - b.x, dy = (lll)a.y - b.y;
    return dx*dx + dy*dy;
}

    ll s = (ll)r;
    while (s > 0 && s * (lll)s > v) s--;
    while ((s + 1) * (lll)(s + 1) <= v) s++;
    return s;
}
// closest pair returning the achieving points; ties keep the first found.
// Returns -1 (ra, rb unchanged) if fewer than two points.
lll closestPair(vector<Pt> p, Pt& ra, Pt& rb){
    int n = (int)p.size();
    if (n < 2) return -1;
    sort(p.begin(), p.end(), [](const Pt& a, const Pt& b){
        return a.x != b.x ? a.x < b.x : a.y < b.y;
    });
    auto cmpY = [](const Pt& a, const Pt& b){
        return a.y != b.y ? a.y < b.y : a.x < b.x;
    };
    multiset<Pt, decltype(cmpY)> active(cmpY);
    lll best = dist2(p[0], p[1]);
    ra = p[0]; rb = p[1];
    if (best == 0) return 0;          // duplicate points: pair already recorded
    active.insert(p[0]);
    int left = 0;
    for (int i = 1; i < n; i++){
        ll d = floorSqrtLL(best);
        while (left < i &&
               (lll)(p[i].x - p[left].x) * (p[i].x - p[left].x) > best){
            active.erase(active.find(p[left]));
            left++;
        }
        Pt lo{LLONG_MIN / 2, p[i].y - d};
        for (auto it = active.lower_bound(lo);
             it != active.end() && it->y <= p[i].y + d; ++it){
            lll dd = dist2(*it, p[i]);
            if (dd < best){
                best = dd; ra = *it; rb = p[i];
                if (best == 0) return 0;      // cannot improve beyond zero
            }
        }
        active.insert(p[i]);
    }
    return best;
}
string i128(lll v){
    if (v == 0) return "0";

    string s;
    while (v){ s += char('0' + (int)(v % 10)); v /= 10; }
    if (n) s += '-';
    reverse(s.begin(), s.end());
    return s;
}
int main(){
    ios::sync_with_stdio(false);
    cin.tie(nullptr);
    int n;
    if (!(cin >> n)) return 0;
    vector<Pt> p(n);
    for (auto& q : p) cin >> q.x >> q.y;
    if (n < 2){ cout << "NO_PAIR\n"; return 0; }
    Pt a, b;
    lll d2 = closestPair(p, a, b);
    cout << fixed << setprecision(6);
    cout << i128(d2) << " " << sqrtl((long double)d2) << " "
         << a.x << " " << a.y << " " << b.x << " " << b.y << "\n";
    return 0;
}

Complexity. O(n log n); one sqrt at the end.

Output meaning. Squared distance, then real distance, then the two closest points; NO_PAIR for fewer than two points.

📝 18 — Final Summary & Series Recap

Closest pair, in one screen

  • Compare squared distances; root only at output.
  • Brute O(n²) for small n; sweep O(n log n) otherwise.
  • Sort by x, keep a y-ordered active set.
  • Prune by x-gap, query the y-window.
  • Only O(1) candidates per point (packing bound).
  • Duplicates give distance 0 naturally; use __int128 for big coordinates.

The hard rules

  • No sqrt in the comparison path.
  • Promote to __int128 before squaring.
  • Prune the active set or the bound is lost.
  • Query only the y-band, not the whole set.
  • Keep integer-exact; no stray EPS.

Runtime Validation Checklist

Before using these closest-pair routines in a contest solution, verify each item below. The Class 01–09 numeric, geometric, signed-area, and exactness checklists still apply underneath.

Closest pair

  • Distances are compared as squared distances; no sqrt appears in the comparison path.
  • Squared distance is accumulated in __int128, with operands promoted before multiplying.
  • The points are sorted by x, and the active set is ordered by y.
  • Far-back points are pruned by the x-gap, and only the [y−d, y+d] window is queried.
  • The window width uses an exact ⌊√best⌋, corrected for floating rounding.

Outputs & preconditions

  • Fewer than two points returns the negative sentinel (no pair).
  • Duplicate or coincident points yield squared distance 0; the optimized sweep returns immediately once zero is found.
  • The real Euclidean distance, if needed, is one sqrt of the final squared value.
  • The pair-returning routine keeps the first pair achieving the minimum (documented tie behavior).

Spot checks (run mentally before submitting)

  • Points (0,0),(3,4),(10,10),(11,11),(100,100): minimum squared distance 2 (the pair (10,10),(11,11)).
  • Points with a duplicate, e.g. (5,5),(1,1),(5,5),(9,9): squared distance 0.
  • All points equal: squared distance 0.
  • Two points (0,0),(6,8): squared distance 100, real distance 10.
  • The sweep agrees with the brute force on every randomized small input.
  • 100000 identical points (5,5): the answer is 0, returned immediately once zero is detected — no quadratic scan of the active set.
