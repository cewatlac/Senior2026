Class 03 — Line Conversion & Point-on-Segment

Three ways to name a line, how to convert between them, and the predicates that ask "is this point on the line / ray / segment?" — plus projection and point-to-line distance, all built on the orient and dot/cross machinery from Classes 01–02.

🎯 1 — Problem Definition

A line is one of the most basic objects in geometry, yet "how do I store a line in code?" has several answers, each with its own strengths and traps. Class 03 settles that question and then answers the predicates that depend on it:

How do I represent a line — slope, equation, or two points? Is point p on this line? On this ray? On this segment? What is the closest point of a segment to p? How far is p from a line or a segment?

The thread tying this together is the parametric form p(t) = a + t·(b − a), which lets one representation describe a line, a ray, or a segment just by restricting the range of t. Combined with the cross-product collinearity test (Class 02) and the dot-product "between" test, it gives clean, division-free predicates.

Recall the four classic geometry bugs (Class 01). This class is exactly where bug 1 ("wrong model") strikes hardest: reaching for y = mx + c and then crashing on a vertical line, or mixing up line / ray / segment behaviour (bug 4) by forgetting to constrain t. We pin down the right model so those bugs never start.

🧠 2 — Intuition from Scratch

2.1 — Three ways to write the same line

The same physical line can be described in three different algebraic costumes. They are interchangeable, but each shines in a different situation.

Diagram · One line, three representations

2.2 — Slope-intercept y = m·x + c, and why it breaks

The schoolbook form is intuitive: m is the slope, c is where the line meets the y-axis. But a vertical line has infinite slope — m = Δy / Δx with Δx = 0 is a division by zero. This form simply cannot represent a vertical line, which is fatal in contests where vertical inputs are routine.

Diagram · The vertical line breaks slope-intercept

2.3 — General form A·x + B·y + C = 0

The implicit form fixes the vertical-line problem: a vertical line through x = k is just 1·x + 0·y − k = 0. The coefficients (A, B) form a normal vector — perpendicular to the line. Given a direction d = (dx, dy), a normal is (A, B) = (dy, −dx) (that is perpCW(d) from Class 02), and C is fixed by plugging in any point on the line.

2.4 — Parametric form p(t) = a + t·(b − a)

The workhorse of this class. Start at a, walk in the direction b − a, scaled by a real number t. The single most useful fact is that the range of t changes what object you have described:

Range of tObjectMeaning
t = 0point athe start.
t = 1point bthe end.
0 ≤ t ≤ 1segment [a, b]between the two endpoints.
t ≥ 0ray from a through bforward only.
any real tinfinite lineboth directions forever.

👁 3 — Visual Mental Model

3.1 — The parameter t as a ruler

Think of t as a position on a ruler laid along the line, where 0 sits at a and 1 sits at b. Numbers in between land inside the segment; numbers outside [0, 1] overshoot one of the ends.

Diagram · The parametric ruler — t = 0 at a, t = 1 at b

3.2 — Projection: dropping a perpendicular

To find the closest point of a line to an external point p, drop a perpendicular from p to the line. The foot of that perpendicular, call it q, is the projection. For a segment, if the foot would land outside [a, b], the true closest point is whichever endpoint the foot overshot.

Diagram · Projection of p onto a line — the foot q and the right angle

🧮 4 — Formal Abstraction

4.1 — The three forms precisely

FormEquationStrengthWeakness
Slope-intercepty = m·x + cIntuitive; easy to plot.No vertical lines (m infinite).
General / implicitA·x + B·y + C = 0Handles every line; (A,B) is a normal; good for exact lattice keys.Not unique unless normalized.
Parametricp(t) = a + t·(b − a)Unifies line / ray / segment via the range of t; division-free predicates.Needs two distinct points a ≠ b.

4.2 — Converting between forms

From two points a = (x₁, y₁) and b = (x₂, y₂) with direction d = (dx, dy) = b − a:

// two points  →  general form A·x + B·y + C = 0
A = dy                       // = (b - a).y
B = -dx                      // = -(b - a).x   so (A, B) is a normal
C = -(A * x1 + B * y1)       // force a (and hence b) to satisfy it
                             //   = dx*y1 - dy*x1

// general form  →  slope-intercept (only when B != 0, i.e. not vertical)
m = -A / B
c = -C / B

// two points  →  slope-intercept (only when dx != 0)
m = dy / dx
c = y1 - m * x1

The conversions that involve m or division by B are exactly the ones that fail on vertical lines. The general and parametric forms never divide by a coordinate, which is why we prefer them.

4.3 — Vertical and horizontal cases

  • Vertical line through x = k: slope-intercept fails; general form is 1·x + 0·y − k = 0; parametric uses any two points with the same x.
  • Horizontal line through y = k: slope m = 0, so slope-intercept is fine (y = k); general form is 0·x + 1·y − k = 0.

4.4 — Normalizing the general form (canonical line)

The triple (A, B, C) is not unique: scaling all three by any non-zero constant describes the same line. To compare two lines for equality — or to use a line as a hash key — we normalize. For integer coefficients: divide (A, B, C) by gcd(|A|, |B|, |C|), then flip the sign of all three so that the first non-zero coefficient is positive. After this, two triples are equal exactly when the lines coincide.

Where this goes deeper. The full exact-integer canonical line — used as a map/set key for grouping segments by their supporting line — is developed in Class 10. Here we only need the idea: a line's coefficient triple must be normalized before it can be compared or hashed.

💡 5 — Key Observations

  • O1 — "On the line" is a collinearity test. p lies on the line through a, b exactly when cross(b − a, p − a) = 0 — the same zero-cross test as orient = 0 from Class 02.
  • O2 — "Between" is a dot-product sign test. If p is collinear with a, b, then p lies between them exactly when dot(p − a, p − b) ≤ 0: the two vectors from p to the endpoints point in opposite directions.
  • O3 — "In front of a" (on the ray) is also a dot-product sign. A collinear p is on the ray from a through b when dot(p − a, b − a) ≥ 0: the offset from a agrees with the forward direction.
  • O4 — Projection is one dot-product ratio. The parameter of the foot is t = dot(p − a, d) / dot(d, d) with d = b − a. No trig, no sqrt.
  • O5 — Segment closest point = projection, then clamp. Compute the line projection's t, then clamp it into [0, 1]. Clamping to 0 gives a; to 1 gives b.
  • O6 — Point-to-line distance is the cross product over the base length. |cross(b − a, p − a)| is twice the triangle area; dividing by |b − a| gives the height, which is the distance. No projection needed.

⚙ 6 — Algorithm / Method Explanation

6.1 — The three "on" predicates

All three start with the same collinearity gate, then add a constraint:

  • pointOnLine: collinear only. cross(b − a, p − a) = 0.
  • pointOnRay: collinear and dot(p − a, b − a) ≥ 0 (not behind a).
  • pointOnSegment: collinear and dot(p − a, p − b) ≤ 0 (between the endpoints).
Diagram · on-line vs on-ray vs on-segment — the same collinear point, three verdicts

6.2 — Projection onto a line

With d = b − a, the foot of the perpendicular is at parameter t = dot(p − a, d) / dot(d, d), so the projected point is q = a + d·t. Note dot(d, d) = |d|² is the squared length — never zero when a ≠ b.

6.3 — Projection onto a segment (clamping)

Compute the same t, then clamp: if t < 0 set t = 0 (closest point is a); if t > 1 set t = 1 (closest point is b); otherwise the foot is genuinely inside the segment.

Diagram · Segment projection clamps t into [0, 1]

6.4 — Point-to-line and point-to-segment distance

For a line, use the area shortcut directly: distance = |cross(b − a, p − a)| / |b − a|. For a segment, the answer is the distance from p to its clamped projection, because the closest point of a segment is exactly that clamped foot.

Diagram · Point-to-line distance is the parallelogram area divided by the base

🔒 7 — Invariants

State invariants

  • I1 — The collinearity gate is shared. All three "on" predicates first require cross(b − a, p − a) = 0 (via cmp). If a point is not on the infinite line, it is on neither the ray nor the segment.
  • I2 — Nesting. pointOnSegment ⟹ pointOnRay ⟹ pointOnLine. The segment is a subset of the ray, which is a subset of the line. The predicates respect this containment.
  • I3 — Endpoints are included. pointOnSegment returns true at p = a and p = b, because the "between" dot product is 0 there (one factor is the zero vector).

Transition invariants

  • I4 — Projection lands on the line. projectPointLine(a, b, p) always returns a point that satisfies pointOnLine (up to floating drift): the foot is on the line by construction.
  • I5 — Clamped projection lands on the segment. projectPointSegment always returns a point with 0 ≤ t ≤ 1, hence satisfying pointOnSegment.
  • I6 — Distance agreement. When the unclamped foot lies inside [a, b], distPointSegment = distPointLine. Outside, the segment distance equals the distance to the nearer endpoint.

Termination invariants

Every predicate and projection is straight-line code — a constant number of arithmetic operations, no loops or recursion. Each returns in O(1).

Failure mode — applicability. pointOnLine, pointOnRay, the projections, and the distances all require a ≠ b so the direction is non-zero (Class 01 O6); otherwise dot(d, d) = 0 and the projection divides by zero. Only pointOnSegment is safe with a = b — it then degenerates to "is p equal to that single point?".

📜 8 — Proof of Correctness

8.1 — The general form's normal (A, B) = (dy, −dx)

A point (x, y) is on the line through a = (x₁, y₁) with direction d = (dx, dy) iff the offset (x − x₁, y − y₁) is parallel to d, i.e. their cross product vanishes:

cross(d, (x - x1, y - y1)) = dx*(y - y1) - dy*(x - x1) = 0
⇒  dy*x - dx*y + (dx*y1 - dy*x1) = 0
⇒  A*x + B*y + C = 0   with  A = dy,  B = -dx,  C = dx*y1 - dy*x1

Because dot((A, B), d) = dy·dx + (−dx)·dy = 0, the vector (A, B) is perpendicular to the line — it is the normal, as claimed.

8.2 — Why dot(p − a, p − b) ≤ 0 means "between"

Assume p is collinear with a and b, so all three lie on one line. Look at the two vectors from p: one points to a, the other to b. If p is strictly between the endpoints, these vectors point in opposite directions along the line, so the angle between them is π and their dot product is negative. If p is outside the segment, both vectors point the same way, giving a positive dot product. At an endpoint one vector is zero, giving exactly 0. Hence dot(p − a, p − b) ≤ 0 ⟺ p ∈ [a, b].

Diagram · Between vs outside — direction of (p−a) and (p−b)

8.3 — The projection formula from perpendicularity

The foot q = a + t·d is defined by the perpendicular condition: the residual p − q must be orthogonal to the line direction d. Set the dot product to zero and solve for t:

dot(p - q, d) = 0
dot(p - a - t*d, d) = 0
dot(p - a, d) - t * dot(d, d) = 0
t = dot(p - a, d) / dot(d, d)

Since a ≠ b, dot(d, d) = |d|² > 0, so the division is safe and t is well-defined.

8.4 — Why segment distance reduces to clamp-then-measure

The squared distance from p to a point a + t·d on the segment is a quadratic in t that opens upward, so it has a single minimum at the unclamped foot t* = dot(p − a, d)/dot(d, d). Restricting t to [0, 1], the minimum of an upward parabola over an interval is attained either at the vertex (if it lies inside) or at the nearer endpoint (if the vertex lies outside). That is precisely "clamp t* into [0, 1], then measure" — which is what projectPointSegment followed by dist computes.

8.5 — Point-to-line distance from the parallelogram area

The vectors b − a and p − a span a parallelogram whose area is |cross(b − a, p − a)|. The same area equals base times height, where the base is |b − a| and the height is the perpendicular distance from p to the line. Solving, height = |cross(b − a, p − a)| / |b − a|, which is exactly distPointLine.

⏱ 9 — Complexity Analysis

📐 10 — Core Template Philosophy

  • Parametric is the default working form. Store a line/segment as two points a, b and reason with t. It never divides by a coordinate, unifies line / ray / segment, and reuses the Class 02 cross/dot machinery.
  • General form for storage and equality. When you need to compare lines or use one as a key, convert to the normalized general form (Class 10 develops the exact-integer version).
  • Slope-intercept almost never. It cannot represent vertical lines; reach for it only when a problem explicitly speaks in slopes and guarantees no vertical case.
  • One collinearity gate, reused. The "on" predicates share the same cross = 0 test, then each adds one dot-product constraint — readable and consistent.
  • Const-reference inputs, documented preconditions. Every helper takes const Point& and states its a ≠ b requirement where it matters.

💻 11 — Master / Base Implementation

Standalone, compiles with g++ -std=c++17. The first block repeats just enough of the Class 01–02 substrate for the file to build on its own; the Class 03 additions follow the divider.

#include <bits/stdc++.h>
using namespace std;

using T     = long double;
using Point = complex<T>;

#define X real()
#define Y imag()

const long double EPS = 1e-12L;
const T PI = acosl(-1.0L);

// ---- rotation helper (typed function, not a macro) ----------------
Point rot(const Point& v, T t) { return v * polar((T)1, t); }

// ---- Class 01-02 substrate (cmp / safeSqrt / dot / cross / len / dist) ----
template<class U> int cmpInt(const U& a, const U& b = U(0)) { return (a > b) - (a < b); }
template<class U> int cmpFloat(const U& a, const U& b = U(0)) {
    long double d = (long double)a - (long double)b;
    return (d > EPS) - (d < -EPS);
}
template<class U> int cmp(const U& a, const U& b = U(0)) {
    if constexpr (is_floating_point_v<U>) return cmpFloat(a, b);
    else return cmpInt(a, b);
}
T safeSqrt(T x) { if (cmp(x) == 0) return 0; return (T)sqrtl((long double)x); }
T dot  (const Point& a, const Point& b) { return real(conj(a) * b); }
T cross(const Point& a, const Point& b) { return imag(conj(a) * b); }
T len (const Point& a) { return (T)hypotl((long double)a.X, (long double)a.Y); }
T dist(const Point& a, const Point& b) { return len(a - b); }
bool samePoint(const Point& a, const Point& b) {
    return cmp(a.X, b.X) == 0 && cmp(a.Y, b.Y) == 0;
}

// ===================================================================
// CLASS 03 — line forms & point-on-segment
// ===================================================================

// Is p on the INFINITE line through a and b?  (Precondition: a != b.)
// p is on the line iff (b - a) and (p - a) are parallel, i.e. cross == 0.
bool pointOnLine(const Point& a, const Point& b, const Point& p) {
    return cmp(cross(b - a, p - a)) == 0;
}

// Is p on the RAY starting at a, through b?  (Precondition: a != b.)
// Collinear AND in front of a along (b - a).
bool pointOnRay(const Point& a, const Point& b, const Point& p) {
    if (cmp(cross(b - a, p - a)) != 0) return false; // not collinear
    return cmp(dot(p - a, b - a)) >= 0;              // not behind a
}

// Is p on the SEGMENT [a, b]?  (Safe even if a == b: then tests p == a.)
// Collinear AND p lies "between" a and b: the vectors p-a and p-b point
// in opposite directions, so their dot product is <= 0.
bool pointOnSegment(const Point& a, const Point& b, const Point& p) {
    if (cmp(cross(b - a, p - a)) != 0) return false; // not collinear
    return cmp(dot(p - a, p - b)) <= 0;              // between endpoints
}

// Foot of perpendicular from p onto the INFINITE line through a, b.
// (Precondition: a != b.)  q = a + t*(b-a), t = dot(p-a, d)/dot(d, d).
Point projectPointLine(const Point& a, const Point& b, const Point& p) {
    Point d = b - a;
    T t = dot(p - a, d) / dot(d, d);
    return a + d * t;
}

// Closest point on the SEGMENT [a, b] to p: project, then clamp t to [0,1].
// (Precondition: a != b.)
Point projectPointSegment(const Point& a, const Point& b, const Point& p) {
    Point d = b - a;
    T t = dot(p - a, d) / dot(d, d);
    if (cmp(t) < 0)        t = 0;   // before a
    if (cmp(t, (T)1) > 0)  t = 1;   // after b
    return a + d * t;
}

// Distance from p to the INFINITE line through a, b.  (Precondition: a != b.)
// |cross(b-a, p-a)| is twice the triangle area = base * height,
// so height = |cross| / |base|.
T distPointLine(const Point& a, const Point& b, const Point& p) {
    return fabsl((long double)cross(b - a, p - a)) / len(b - a);
}

// Distance from p to the SEGMENT [a, b].  (Precondition: a != b.)
T distPointSegment(const Point& a, const Point& b, const Point& p) {
    return dist(p, projectPointSegment(a, b, p));
}
Two macros that look identical to fields but are not. #define X real() means v.X expands to v.real(); #define Y imag() likewise. rot is a typed helper function, never a macro — and never named rotate (which collides with std::rotate).

⚠ 14 — Common Pitfalls / Bugs

  • P1 — Using y = mx + c for a vertical line. The slope is infinite. Use the parametric or general form, which handle every direction.
  • P2 — Storing the general form without normalizing. Two coefficient triples that differ only by a scale factor describe the same line but compare as different. Normalize (divide by gcd, fix the sign) before comparing or hashing — see Class 10 for the exact-integer version.
  • P3 — Computing t for a segment and forgetting to clamp. Without the clamp you get the line's foot, which may lie outside the segment, giving a wrong "closest point" and a too-small distance.
  • P4 — Claiming "on the ray" from the dot test alone. dot(p − a, b − a) ≥ 0 only means "not behind a"; you must first confirm collinearity, or an off-line point in front of a would falsely pass.
  • P5 — Building point-to-line distance via the projection foot. It works, but |cross| / |base| is shorter and avoids constructing the foot. Reserve the projection for when you actually need the closest point.
  • P6 — Dividing by dot(d, d) when a == b. That is a division by zero. Guarantee a ≠ b before projecting or measuring line distance.
  • P7 — Comparing t against 0 / 1 with raw < on floats. Use cmp(t) and cmp(t, 1) so the boundary is handled consistently with the rest of the family.

Reflex rules (contest speed)

On-line = cross is zero. Between = dot(p−a, p−b) ≤ 0. In front = dot(p−a, b−a) ≥ 0. Project: t = dot(p−a, d) / dot(d, d). Segment closest = project, then clamp t to [0,1]. Never y = mx + c for vertical lines.

Common confusion

"Two equal lines are comparing as different." You stored them in general form without normalizing. The triples (2, 4, 6) and (1, 2, 3) are the same line. Divide by the gcd and fix the leading sign before any equality check.

🌿 16 — Variants & Extensions

16.1 — Halfplane representation

A directed line splits the plane into two sides. A halfplane stores a line plus a chosen "positive" side — naturally (anchor, direction) with the convention that the left side (positive orient) is inside. What changes: you carry an inside/outside notion. Why: halfplane intersection and polygon clipping (Class 09) are built from these. When to use: linear-constraint and clipping problems.

// A halfplane: points x with orient(a, b, x) >= 0 are "inside".
struct Halfplane { Point a, b; };   // directed line a -> b; left side is inside
bool inside(const Halfplane& h, const Point& x) {
    return cmp(cross(h.b - h.a, x - h.a)) >= 0;   // reuses the Class 02 side test
}

16.2 — Parametric ray with t ∈ [0, ∞)

For "shooting" problems (a particle leaves a heading toward b and travels forever), keep the parametric form but accept any t ≥ 0. This is the ray; intersection routines (Class 04) clamp t accordingly.

16.3 — Signed point-to-line distance

Dropping the absolute value keeps the side information: signedDist = cross(b − a, p − a) / |b − a| is positive on the left of a → b and negative on the right. When to use: "how far, and which side" questions, and as the kernel of halfplane tests.

T signedDistPointLine(const Point& a, const Point& b, const Point& p) {
    return cross(b - a, p - a) / len(b - a);   // sign encodes the side
}

Training drill — closest point and distance from a point to a segment the canonical projection pattern, not a contest problem

Scope. This is a training drill for the projection-and-clamp pattern, not a full contest problem. It isolates the segment-distance computation before later classes use it inside segment intersection, polygon distance, and circle-segment tests.

Story. Given a segment [a, b] and a query point p, report the closest point of the segment to p and the distance between them.

Abstraction. This is projectPointSegment(a, b, p) for the closest point and distPointSegment(a, b, p) for the distance — projection onto the line followed by a clamp of t into [0, 1].

Why this geometry model fits. The squared distance along the segment is an upward parabola in t (§8.4), so its minimum over [0, 1] is the clamped vertex. The parametric form makes the clamp a one-line operation.

Algorithm. Read a, b, p; project; clamp t; output the point and its distance to p.

#include <bits/stdc++.h>
using namespace std;

using T     = long double;
using Point = complex<T>;
#define X real()
#define Y imag()

const long double EPS = 1e-12L;

template<class U> int cmpFloat(const U& a, const U& b = U(0)) {
    long double d = (long double)a - (long double)b;
    return (d > EPS) - (d < -EPS);
}
template<class U> int cmp(const U& a, const U& b = U(0)) {
    if constexpr (is_floating_point_v<U>) return cmpFloat(a, b);
    else return (a > b) - (a < b);
}
T dot (const Point& a, const Point& b) { return real(conj(a) * b); }
T len (const Point& a) { return (T)hypotl((long double)a.X, (long double)a.Y); }
T dist(const Point& a, const Point& b) { return len(a - b); }

// closest point of segment [a,b] to p: project then clamp t to [0,1]
Point projectPointSegment(const Point& a, const Point& b, const Point& p) {
    Point d = b - a;                          // precondition: a != b
    T t = dot(p - a, d) / dot(d, d);
    if (cmp(t) < 0)       t = 0;
    if (cmp(t, (T)1) > 0) t = 1;
    return a + d * t;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    long double ax, ay, bx, by, px, py;
    while (cin >> ax >> ay >> bx >> by >> px >> py) {
        Point a(ax, ay), b(bx, by), p(px, py);
        Point q = projectPointSegment(a, b, p);
        cout << fixed << setprecision(6)
             << q.X << ' ' << q.Y << ' ' << dist(p, q) << '\n';
    }
    return 0;
}

Complexity. O(1) per query.

Pitfalls. Require a ≠ b so dot(d, d) ≠ 0. Without the clamp you would report the line's foot, which may lie off the segment and underestimate the distance. As a worked check: for a = (0,0), b = (4,0), p = (7,3) the foot's t = 1.75 clamps to 1, so the closest point is b = (4,0) and the distance is √(3² + 3²) ≈ 4.2426.

Training drill — point-on-segment membership collinear + between, not a contest problem

Scope. A pattern drill for the membership predicate, not a standalone contest problem. It shows the "collinear gate, then between test" structure that recurs in segment intersection (Class 04) and polygon point tests (Class 09).

Story. Given a segment [a, b] and a point p, decide whether p lies on the closed segment (endpoints count as on).

Abstraction. This is exactly pointOnSegment(a, b, p): a zero cross product (collinear) combined with a non-positive dot(p − a, p − b) (between).

Why this geometry model fits. Collinearity is the cross-product test from Class 02; "between" is the dot-sign argument of §8.2. Together they decide membership with no division and no special-casing of vertical segments.

Algorithm. For integer coordinates the test is exact: compute the cross product in __int128 for the collinearity gate, then the between-dot, also in __int128.

#include <bits/stdc++.h>
using namespace std;

// Exact integer point-on-segment for the closed segment [a, b].
// Endpoints count as on. Works even when a == b (tests p == a).
bool pointOnSegmentInt(long long ax, long long ay,
                       long long bx, long long by,
                       long long px, long long py) {
    // collinear: cross(b - a, p - a) == 0   (use __int128 to avoid overflow)
    __int128 crossv = (__int128)(bx - ax) * (py - ay)
                    - (__int128)(by - ay) * (px - ax);
    if (crossv != 0) return false;
    // between: dot(p - a, p - b) <= 0
    __int128 dotv = (__int128)(px - ax) * (px - bx)
                  + (__int128)(py - ay) * (py - by);
    return dotv <= 0;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    long long ax, ay, bx, by, px, py;
    while (cin >> ax >> ay >> bx >> by >> px >> py) {
        cout << (pointOnSegmentInt(ax, ay, bx, by, px, py) ? "ON\n" : "OFF\n");
    }
    return 0;
}

Complexity. O(1) per query.

Pitfalls. Use __int128 for both the cross and the dot when coordinates can reach 10⁹, or the products overflow long long. Do not skip the collinearity gate — the between-dot alone can be non-positive for points off the line.

📝 18 — Final Summary

The mental model

  • Three forms — slope-intercept, general, parametric.
  • Parametric a + t·(b − a) unifies line / ray / segment.
  • On-line = cross is zero (collinear).
  • On-segment = collinear + dot(p−a, p−b) ≤ 0.
  • Projection = t = dot(p−a, d)/dot(d, d).
  • Line distance = |cross| / |base|.

The hard rules

  • No y = mx + c for vertical lines.
  • Normalize the general form before comparing.
  • Segment closest point = project, then clamp.
  • Confirm collinearity before any "between"/"in front" test.
  • Require a ≠ b before projecting or measuring line distance.

Class 03 turns the orientation toolkit into line-and-segment predicates. pointOnSegment reappears inside segment intersection (Class 04) and point-in-convex-polygon (Class 09); projectPointSegment feeds circle-segment distance (Class 07); the parametric t semantics recur throughout Classes 04, 06, and 09.

What's next. Class 04 uses the parametric form and the orient side test to answer when two segments intersect — proper crossings, endpoint touches, and collinear overlap — and computes the intersection point of two infinite lines.

Runtime Validation Checklist

Before using these line and segment predicates in a contest solution, verify each item below. The Class 01 numeric checklist and the Class 02 orientation checklist still apply underneath.

Representation & predicates

  • Lines/segments stored parametrically (two points) or in normalized general form — never slope-intercept for inputs that may be vertical.
  • General-form triples are normalized (divide by gcd, fix the leading sign) before any equality or hash use.
  • Each "on" predicate applies the collinearity gate first, then its dot-sign constraint.
  • Every projection / line-distance call has a ≠ b; only pointOnSegment is used with possibly-equal endpoints.

Projection & distance

  • Segment projection clamps t into [0, 1] via cmp before constructing the point.
  • Point-to-line distance uses |cross| / |base|; point-to-segment distance uses the clamped projection.
  • Comparisons of t against 0 and 1 go through cmp, not raw < / >.
