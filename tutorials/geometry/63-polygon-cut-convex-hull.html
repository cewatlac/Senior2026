Class 09 — Polygon Cut & Convex Hull

Two operations that reshape a set of points. Polygon cut slices a polygon by a directed line and keeps one side, splicing in new vertices where edges cross. The convex hull wraps a point cloud in the tightest convex polygon that contains it. Both turn on the same primitive from Class 08 — the sign of a cross product — to decide which side of a line a point falls on.

🎯 1 — Problem Definition

This class joins two classic polygon operations. The polygon cut takes a polygon and a directed line, and returns the part of the polygon lying on one chosen side of that line — the half-plane clip. The convex hull takes a set of points and returns the smallest convex polygon enclosing all of them.

Cut a polygon by a directed line, keep one side Splice in intersection points where edges cross Wrap a point cloud in its tightest convex polygon Keep or drop collinear boundary points

Both reduce to one question asked repeatedly: which side of a directed line is this point on? The signed cross product answers it — positive on the left, negative on the right, zero on the line. The cut keeps left-side vertices and adds crossings; the hull keeps only the points that make left turns around the boundary.

Two convex-hull APIs, on purpose. The primary convexHull follows the Markdown tutorial's pivot/angle-sort recipe on the floating Point = complex<T> type and returns the strict-corner hull (no intermediate collinear vertices). The secondary convexHullAndrew stays on the integer Pt type with __int128 cross products — use it when coordinates are bit-exact long longs, or pass inclusive = true for CSES-style hulls that must keep collinear boundary points. The polygon cut creates intersection points that are generally not integers, so it always runs in long double with an EPS-based side test.

🧠 2 — Intuition from Scratch

2.1 — Cutting is walking the boundary and deciding per edge

Imagine tracing the polygon's boundary edge by edge while a directed line divides the plane into a "keep" side and a "discard" side. For each edge you ask two questions: is the current vertex on the keep side, and does this edge cross the line? Keep every vertex on the chosen side; whenever an edge passes from one side to the other, the crossing point becomes a new vertex of the result. Walking all the way around stitches together exactly the kept region.

2.2 — The hull is a rubber band around nails

Picture a nail at each point and a rubber band stretched around all of them. When released, the band snaps to the outermost points and forms a convex polygon — the convex hull. Points strictly inside never touch the band; intermediate points lying on a straight edge of the band are removed by default (the hull keeps only true corners). To build the band algorithmically, we anchor at a pivot — the leftmost (and, on ties, bottommost) point — sort every other point by polar angle around the pivot, and walk through them once while popping the stack on any non-left turn.

2.3 — Left turns only

Once the points are processed in pivot-angular order, building the hull is a single stack walk: add the next point, then look at the turn made by the last three vertices. A left turn (counterclockwise) is fine; a right turn means the middle point caved inward, so you pop it. The Markdown tutorial's default policy also pops on a straight (collinear) turn, so the result is the strict-corner hull — no intermediate collinear vertices, just the true corners. (A clearly named alternative below — convexHullAndrew — keeps collinear boundary points when you need the inclusive form.)

🧮 4 — Formal Abstraction

4.1 — Side classification by signed cross product

For a directed line through a → b and a query point p, the sign of cross(b − a, p − a) classifies p: positive means left, negative means right, zero means on the line. With floating coordinates the zero band is widened to ±EPS so points within tolerance are treated as on-line.

4.2 — Polygon cut (Sutherland–Hodgman against one half-plane)

Walk the polygon's edges (cur, nxt). Let sc, sn be the side classifications of the two endpoints. The output is built by two rules:

if (sc >= 0)        keep cur               // vertex on the kept side or on the line
if (sc * sn < 0)    add intersection(cur,nxt, line)   // edge strictly crosses the line

Keeping vertices with sc ≥ 0 retains on-line vertices; an intersection is added only when the two endpoints are strictly on opposite sides (sc·sn < 0), so an edge that merely touches the line at a vertex adds nothing spurious.

4.3 — Segment–line intersection for the crossing point

When edge p → q crosses the line a → b, parameterize the edge as p + t(q − p) and solve for the t that lands on the line. With c₁ = cross(a,b,p) and c₂ = cross(a,b,q),

t = c₁ / (c₁ − c₂),     intersection = p + t·(q − p)

Because the endpoints are on opposite sides, c₁ and c₂ have opposite signs, so the denominator is nonzero and t ∈ (0,1).

Diagram · Edge crossing the cut line creates an intersection point

4.4 — Convex hull (pivot + angle sort, default policy)

The primary hull this class teaches follows the Markdown tutorial's pivot/angle-sort recipe:

  1. Pivot. Choose the leftmost point (ties broken by smallest y) as the anchor.
  2. Angle sort. Sort every other point by polar angle around the pivot using cross. For collinear ties (same direction from the pivot), order by (x, y) so the chain enters from the nearest point first.
  3. Single stack walk. Iterate through the sorted points and pop the top of the stack while the last three vertices make a non-left turn — i.e. while cross(top − below, p − top) ≤ 0. The default policy drops intermediate collinear vertices, returning only true corners.
  4. All-collinear fallback. If every input point is collinear, the stack collapses to a single survivor; return the two extremes from the lex-sorted list instead. (The Markdown returns {front, back}.)

The Markdown tutorial labels this "Graham-style with a pivot". Andrew's monotonic chain is a valid alternative — it builds a lower hull left-to-right and an upper hull right-to-left and concatenates them — but it is kept here as the secondary convexHullAndrew for inputs that need the inclusive (keep-collinear) boundary.

// primary, pivot/angle-sort policy (Markdown default): drop collinear corners.
pop while  cross(h[k-2], h[k-1], p[i])  <= 0

// secondary, convexHullAndrew(..., /*inclusive=*/true): keep collinear.
pop while  cross(h[k-2], h[k-1], p[i])  <  0

💡 5 — Key Observations

  • O1 — One primitive drives both algorithms. The signed cross product classifies a point against a directed line; the cut and the hull are both built entirely on its sign.
  • O2 — The cut keeps sc ≥ 0 and adds a crossing only on sc·sn < 0. This pair of rules retains on-line vertices and never adds a phantom intersection at a grazing vertex.
  • O3 — Hull orientation is exact in integers. Cross products of integer points are integers, so the hull never needs floating point — only the cut does.
  • O4 — Strict-corner is the default; inclusive is a labelled variant. Popping on ≤ 0 drops collinear boundary points (Markdown default, this class's primary convexHull). Popping on < 0 keeps them (use convexHullAndrew(..., /*inclusive=*/true)). Nothing else changes.
  • O5 — Pivot + angle sort + one stack walk is the whole hull. Sorting by polar angle around the leftmost-bottommost pivot visits every candidate corner in circular order, so a single pass with popping suffices.
  • O6 — Degenerate inputs collapse cleanly. Duplicates are removed up front; if every input is collinear, the primary hull returns the two extremes from the lex-sorted list, while the inclusive Andrew variant returns every unique boundary point once.

⚙ 6 — Algorithm / Method Explanation

6.1 — The cut, vertex on the line

A vertex lying exactly on the cut line has sc = 0. The rule sc ≥ 0 keeps it, and because sc·sn = 0 for any neighbor, no intersection is spuriously added at that vertex. This is what makes on-line vertices behave gracefully rather than duplicating points.

Diagram · A vertex exactly on the cut line

6.2 — Before and after the clip

The result of one half-plane clip is itself a polygon: the kept vertices in order, with crossings inserted where the boundary entered or left the half-plane.

Diagram · Before and after polygon clipping

6.3 — Pivot anchor and polar-angle sweep

The Markdown tutorial's primary hull anchors at the pivot — the leftmost point in the cloud, ties broken by smallest y — and then visits every other point in CCW order of polar angle around the pivot. Equal-angle ties (points collinear with the pivot) are resolved by lex order on (x, y), so the nearer point comes first. After sorting, a single stack walk pops the previous top while the last three vertices fail to make a left turn (or sit collinear) — the survivors form the strict-corner hull in CCW order.

Diagram · Pivot and polar-angle sweep around it

6.4 — The orientation test and popping

When a new point would make the last three turn the wrong way, the middle point is popped. The diagram shows a right turn being undone: the caved-in vertex is removed so the chain stays convex.

Diagram · Orientation test and popping in the stack walk (cross ≤ 0 ⇒ pop)

6.5 — Collinear boundary points and the degenerate case

On a straight edge of the hull, intermediate points are collinear with the endpoints. The default (Markdown) policy pops them — the comparison cross ≤ 0 triggers a pop — so only true corners survive. The secondary inclusive convexHullAndrew(..., /*inclusive=*/true) keeps them. When all points are collinear there is no enclosed area: the primary hull returns the two extremes from the lex-sorted list, while the inclusive Andrew variant returns every unique collinear boundary point once.

🔒 7 — Invariants

State invariants

  • I1 — Each hull chain stays convex. After every step of a chain, the kept points turn consistently (left for the counterclockwise result), because any offending middle point was popped.
  • I2 — The cut output traces the kept region in order. Vertices are appended in boundary order, so the result is a valid polygon, not a scrambled point set.
  • I3 — Hull arithmetic is integer-exact. Orientation tests use integer cross products; only the cut forms floating intersection points.

Transition invariants

  • I4 — A crossing is added exactly when endpoints differ in strict side. The test sc·sn < 0 fires once per genuine crossing and never at a grazing vertex.
  • I5 — Popping only removes non-convex or (strict) collinear vertices. A pop never discards a point that belongs on the final hull boundary under the chosen strictness.

Termination invariants

Each hull chain appends every point once and pops each at most once, so the two sweeps are linear after the sort. The cut visits each edge once. Both terminate in a single pass over their input.

Failure mode — preconditions. The polygon cut assumes vertices are given in boundary order and the cut line is directed; a general non-convex polygon may be split into several pieces by one line, which a single-output clip cannot represent. The convex hull assumes a finite point set and may receive duplicates, which it removes; an empty or single-point input returns itself.

📜 8 — Proof of Correctness

8.1 — The clip keeps exactly the chosen half-plane

Consider the kept half-plane H (side ≥ 0 of the directed line). Walking the boundary, each maximal run of vertices inside H is emitted in order; when the boundary leaves H across an edge, the exit crossing is inserted, and when it re-enters, the entry crossing is inserted. Thus the emitted sequence alternates between the polygon's boundary inside H and chords along the cut line, which is precisely the boundary of polygon ∩ H. On-line vertices satisfy sc = 0 ≥ 0 and are kept once, with no spurious crossing because sc·sn = 0 there.

8.2 — The intersection point is correct

For a crossed edge p → q, the signed sides c₁ = cross(a,b,p) and c₂ = cross(a,b,q) are an affine function of position along the edge. The point where that function is zero is at parameter t = c₁/(c₁−c₂), and since c₁, c₂ have opposite signs the denominator is nonzero and t ∈ (0,1), placing the intersection strictly inside the edge and exactly on the line.

8.3 — Pivot + angle sort + stack walk produces the convex hull

After choosing the pivot (leftmost, ties by smallest y) every other point lies strictly above or at the pivot's y, and the angular order around the pivot is a well-defined CCW sweep from "east" through "north" to "west". Processing points in that order with the stack-walk rule maintains the invariant:

After step i, the stack is the convex boundary of the points processed so far, in CCW order, with no intermediate collinear vertex.

The pop condition cross(top − below, p − top) ≤ 0 fires exactly when adding p would either bend inward (right turn) or leave the previous top sitting on a straight edge (collinear) — both cases mean top is not a true corner and must be removed. When all points have been processed, the surviving stack is the convex boundary of the whole input. No point lies outside it (each was either a corner or popped on its way in), and every consecutive triple makes a strict left turn, so the result is convex.

8.4 — Default (strict-corner) and inclusive variants are both correct hulls

The convex hull as a set is unique, but its vertex list is not: collinear points on an edge lie on the hull boundary yet are not corners. The primary convexHull pops on cross ≤ 0, giving the minimal corner list (Markdown default). The secondary convexHullAndrew(..., /*inclusive=*/true) pops only on cross < 0, retaining every boundary point. Both describe the same convex region; they differ only in whether non-corner boundary points appear in the returned vertex list.

📐 10 — Core Template Philosophy

  • Primary hull = pivot + angle sort (Markdown style). The default convexHull uses the same pivot/cross-sort recipe as the Markdown tutorial: leftmost (then bottommost) anchor, polar-angle sort, single stack walk, drop collinear corners. The integer Andrew monotonic chain is kept as convexHullAndrew for inclusive (CSES) outputs and for cases that need exact integer cross products.
  • Integer hull (variant), floating cut. When you have exact integer coordinates, prefer convexHullAndrew on Pt so the orientation stays bit-exact; the polygon cut still uses long double + EPS because intersection points are not generally integers.
  • One side test, reused. The signed cross product is the single primitive; write it once and let both algorithms call it.
  • Make the boundary policy an explicit switch. Expose inclusive on convexHullAndrew rather than hard-coding whether collinear boundary points are kept.
  • Remove duplicates before building. Deduplicate sorted points (under samePoint / EPS for the primary, exact for the integer variant) so degenerate inputs do not corrupt the chain.
  • Guard the cross product's width. For integer coordinates up to 10⁹, the integer Andrew variant computes cross products in __int128.
  • Keep the rotation helper typed. If a rotation is ever needed it is Point rot(const Point& v, T t), never a rotate macro.

💻 11 — Master / Base Implementation

Library/template block; compiles as a translation unit with g++ -std=c++17 -c, or after adding a main. Two convex-hull APIs ship side by side:

  • Primary convexHull(vector<Point>) — Markdown-tutorial style. Pivot at the leftmost-bottommost point, polar-angle sort around it, single stack walk that drops collinear corners. Works on the floating Point = complex<T> type, so it slots into the rest of the geometry library.
  • Secondary convexHullAndrew(vector<Pt>, bool inclusive = false) — integer-exact Andrew monotonic chain. Use it for inclusive (keep-collinear, CSES) hulls and for problems whose coordinates fit in long long and need bit-exact orientation. Cross products run in __int128.

The cut half is floating because intersection points are generally not integers.

#include <bits/stdc++.h>
using namespace std;

// ===================================================================
// CLASS 09 — polygon cut & convex hull.
//  * Primary convex hull (convexHull) uses the Markdown pivot/angle-sort
//    recipe on the floating Point type (the rest of the geometry library).
//  * Secondary convex hull (convexHullAndrew) uses EXACT INTEGER cross
//    products (orientation only) for inclusive / CSES-style outputs.
//  * Polygon cut produces intersection points that are generally NOT
//    integers, so the cut works in long double with EPS classification.
// ===================================================================

// ---------- INTEGER substrate (convex hull) -----------------------
using ll  = long long;
using lll = __int128_t;

struct Pt { ll x, y; };
bool operator<(const Pt& a, const Pt& b) { return a.x != b.x ? a.x < b.x : a.y < b.y; }
bool operator==(const Pt& a, const Pt& b) { return a.x == b.x && a.y == b.y; }

// cross product (b-a) x (c-a), exact in __int128.
lll crossI(const Pt& a, const Pt& b, const Pt& c) {
    return (lll)(b.x - a.x) * (c.y - a.y) - (lll)(b.y - a.y) * (c.x - a.x);
}

// ----- SECONDARY: Andrew's monotonic chain on EXACT INTEGER points --
// Use this when you need the inclusive (keep-collinear-boundary) hull, or
// when coordinates must stay in exact integers (e.g. CSES "Convex Hull").
// inclusive=false drops collinear boundary points; inclusive=true keeps them.
// Returns the hull in counterclockwise order.
vector<Pt> convexHullAndrew(vector<Pt> p, bool inclusive = false) {
    sort(p.begin(), p.end());
    p.erase(unique(p.begin(), p.end()), p.end());   // remove duplicate points
    int n = (int)p.size();
    if (n <= 2) return p;                            // 0,1,2 distinct points: hull is themselves

    // all-collinear: the monotone chain would otherwise duplicate middle points
    // in the inclusive sweep, so handle it explicitly.
    bool allCollinear = true;
    for (int i = 2; i < n; i++) {
        if (crossI(p[0], p[1], p[i]) != 0) { allCollinear = false; break; }
    }
    if (allCollinear) {
        if (!inclusive) return {p.front(), p.back()};    // strict-corner: two extremes
        return p;                                        // inclusive: every boundary point once, sorted
    }

    vector<Pt> h(2 * n);
    int k = 0;
    // lower hull, left to right
    for (int i = 0; i < n; i++) {
        while (k >= 2 && (inclusive ? crossI(h[k-2], h[k-1], p[i]) <  0
                                    : crossI(h[k-2], h[k-1], p[i]) <= 0)) k--;
        h[k++] = p[i];
    }
    // upper hull, right to left
    int lower = k + 1;
    for (int i = n - 2; i >= 0; i--) {
        while (k >= lower && (inclusive ? crossI(h[k-2], h[k-1], p[i]) <  0
                                        : crossI(h[k-2], h[k-1], p[i]) <= 0)) k--;
        h[k++] = p[i];
    }
    h.resize(k - 1);                                 // drop the duplicated start point
    return h;
}

// doubled area magnitude of an integer polygon, |2A| (Class 08 philosophy).
lll doubledArea(const vector<Pt>& poly) {
    int n = (int)poly.size();
    lll s = 0;
    for (int i = 0; i < n; i++) {
        const Pt& a = poly[i]; const Pt& b = poly[(i + 1) % n];
        s += (lll)a.x * b.y - (lll)b.x * a.y;
    }
    return s < 0 ? -s : s;
}

// ---------- FLOATING substrate (polygon cut) ----------------------
using T     = long double;
using Point = complex<T>;
#define X real()
#define Y imag()
const long double EPS = 1e-12L;

// typed rotation helper (never a macro, never named rotate)
Point rot(const Point& v, T t) { return v * polar((T)1, t); }

T crossF(const Point& O, const Point& A, const Point& B) {
    return (A.X - O.X) * (B.Y - O.Y) - (A.Y - O.Y) * (B.X - O.X);
}

// sign of cross(b-a, p-a): +1 left, 0 on the line, -1 right of directed a->b.
int sideOf(const Point& a, const Point& b, const Point& p) {
    T c = crossF(a, b, p);
    if (c >  EPS) return +1;
    if (c < -EPS) return -1;
    return 0;
}

// ---- Helpers used by the primary (Markdown-style) convex hull ----
template<class U> int cmpFloat(const U& a, const U& b = U(0)) {
    long double d = (long double)a - (long double)b;
    return (d > EPS) - (d < -EPS);
}
template<class U> int cmpInt(const U& a, const U& b = U(0)) {
    return (a > b) - (a < b);
}
template<class U> int cmp(const U& a, const U& b = U(0)) {
    if constexpr (is_floating_point_v<U>) return cmpFloat(a, b);
    else return cmpInt(a, b);
}
// 2-arg cross/dot in the standard Markdown form.
T cross(const Point& a, const Point& b) { return imag(conj(a) * b); }
T dot  (const Point& a, const Point& b) { return real(conj(a) * b); }
bool samePoint(const Point& a, const Point& b) {
    return cmp(a.X, b.X) == 0 && cmp(a.Y, b.Y) == 0;
}
// Lex order on (X, Y) using EPS-aware cmp.
bool cmpPoint(const Point& a, const Point& b) {
    int cx = cmp(a.X, b.X);
    if (cx != 0) return cx < 0;
    return cmp(a.Y, b.Y) < 0;
}

// ----- PRIMARY: pivot + angle-sort convex hull (Markdown style) ---
// Default policy:
//   * remove duplicate points;
//   * return only true corners (no intermediate collinear vertices);
//   * if all points are collinear, return the two extreme endpoints.
// The pivot is the leftmost point (ties broken by smallest y), and every
// other point is sorted by polar angle around it using cross. A single
// stack walk then pops on every non-left turn (cross <= 0). The result is
// in CCW order, with no duplicate first/last vertex.
//
// For exact integer coordinates, prefer convexHullAndrew (above) on Pt;
// when you need to keep collinear boundary points (CSES-style hulls),
// call convexHullAndrew(..., /*inclusive=*/true).
vector<Point> convexHull(vector<Point> p) {
    sort(p.begin(), p.end(), cmpPoint);
    p.erase(unique(p.begin(), p.end(), samePoint), p.end());
    int n = (int)p.size();
    if (n <= 2) return p;

    Point piv = p[0];
    sort(p.begin() + 1, p.end(),
         [&](const Point& a, const Point& b) {
             int cr = cmp(cross(a - piv, b - piv));
             if (cr != 0) return cr > 0;       // CCW first
             return cmpPoint(a, b);             // collinear with pivot: nearer first
         });

    vector<Point> h;
    for (const Point& pt : p) {
        while (h.size() >= 2 &&
               cmp(cross(h.back() - h[h.size()-2], pt - h.back())) <= 0)
            h.pop_back();
        h.push_back(pt);
    }
    // All-collinear fallback: stack collapses to a single survivor; return the extremes.
    if (h.size() == 1 && n > 1) return { p.front(), p.back() };
    return h;
}

// intersection of edge p->q with the infinite line a->b (endpoints on opposite sides).
Point lineSegInt(const Point& a, const Point& b, const Point& p, const Point& q) {
    T c1 = crossF(a, b, p);
    T c2 = crossF(a, b, q);
    T t  = c1 / (c1 - c2);          // p + t(q-p) lands on the line a->b
    return p + (q - p) * t;
}

// Cut a polygon by the directed line a->b, KEEPING the LEFT side (cross >= 0).
// Sutherland-Hodgman against one half-plane. Vertices on the line are kept.
vector<Point> cutPolygonLeft(const vector<Point>& poly, const Point& a, const Point& b) {
    vector<Point> out;
    int n = (int)poly.size();
    for (int i = 0; i < n; i++) {
        Point cur = poly[i];
        Point nxt = poly[(i + 1) % n];
        int sc = sideOf(a, b, cur);
        int sn = sideOf(a, b, nxt);
        if (sc >= 0) out.push_back(cur);             // keep inside / on-line vertex
        if (sc * sn < 0) {                           // edge strictly crosses the line
            out.push_back(lineSegInt(a, b, cur, nxt));
        }
    }
    return out;
}

// doubled area magnitude of a floating polygon (for area-after-cut).
T doubledAreaF(const vector<Point>& poly) {
    int n = (int)poly.size();
    T s = 0;
    for (int i = 0; i < n; i++) {
        const Point& u = poly[i]; const Point& v = poly[(i + 1) % n];
        s += u.X * v.Y - v.X * u.Y;
    }
    return s < 0 ? -s : s;
}
Why two number types in one file. The hull only ever asks "left, right, or on?", which integer cross products answer exactly — no rounding, just a sign. A polygon cut, by contrast, computes where an edge meets a line, and that point usually has fractional coordinates, so it must be floating. Mixing the two deliberately keeps each operation as exact as its geometry allows.
Cut precondition. One directed line can split a non-convex polygon into multiple disconnected pieces, which a single returned polygon cannot represent. cutPolygonLeft is exact for a convex polygon (the kept side is always one piece); for a general simple polygon it is the standard single-half-plane Sutherland–Hodgman step, correct when the kept side stays connected.

⚠ 14 — Common Pitfalls / Bugs

  • P1 — Confusing ≤ and < in the hull pop. Popping on ≤ 0 gives the strict-corner hull (the Markdown / primary default); on < 0 the inclusive hull (convexHullAndrew(..., /*inclusive=*/true)). Choosing the wrong one silently keeps or drops collinear boundary points.
  • P2 — 64-bit overflow in the hull cross product. With coordinates near 10⁹, the product reaches 10¹⁸; compute the cross in __int128 (or guarantee small coordinates).
  • P3 — Forgetting to remove duplicate points. Repeated points can stall the popping logic or produce zero-length edges; deduplicate after sorting.
  • P4 — Using floating point for the hull. Orientation is exact in integers; a double cross can mis-sign a near-collinear triple and corrupt the hull. Keep the hull integer.
  • P5 — Adding an intersection at an on-line vertex. Testing sc·sn <= 0 instead of < 0 inserts a duplicate point when an endpoint lies on the line. Use the strict product.
  • P6 — Cutting a non-convex polygon with a single-output clip. If the kept side is disconnected, one polygon cannot hold it; restrict to convex input or split explicitly.
  • P7 — Wrong winding assumption. The cut keeps the left of the directed line; reversing the line direction flips which side is kept. Be deliberate about the line's orientation.

Side = sign of cross(b−a, p−a). Cut: keep sc ≥ 0, add crossing on sc·sn < 0. Hull: sort, dedup, two sweeps, pop non-left turns. Strict ≤ 0 pops collinear; inclusive < 0 keeps them. Hull integer + __int128; cut floating + EPS. Single-line cut assumes a connected kept side.

Common confusion

"A polygon cut by one line always gives one polygon." Only for a convex polygon. A non-convex polygon can be split into two or more disconnected pieces by a single line, and a single returned polygon cannot represent that — cut convex shapes, or handle the split explicitly.

16.4 — Point in convex polygon, O(log n)

Once you have a convex hull, the dual question "is this query point inside it?" admits a far faster answer than a generic point-in-polygon scan. Fan the polygon from vertex p[0]: every other vertex pairs with the next one to form a triangle, and these triangles tile the interior with no overlap. The angle that a query q makes with the rays p[0]→p[i] grows monotonically with i, so a binary search finds the unique fan sector containing q, and a single triangle test finishes the job.

Preconditions. The polygon must be strictly convex (no three vertices collinear), given in CCW order, with no duplicate first/last vertex. If you only have a generic simple polygon, use the ray-casting test from earlier (O(n)); this fast version is only sound after a real convex hull.

How the search works.

  1. Two early rejects: if q is to the right of the first fan ray p[0]→p[1], or to the left of the last fan ray p[0]→p[n-1], it lies outside the polygon's angular sweep entirely.
  2. Binary search on indices l = 1, r = n-1. While more than one slot apart, take m = (l+r)/2. If q is on the CCW side of p[0]→p[m], the answer fan lies further around (l = m); otherwise it lies before m (r = m).
  3. When the loop exits with r − l == 1, the candidate triangle is (p[0], p[l], p[l+1]). One triangle containment test decides the answer.
// Inclusive triangle containment: returns true when p lies on or inside
// triangle a-b-c. Works regardless of vertex order.
bool pointInTriangle(const Point& a, const Point& b, const Point& c, const Point& p) {
    int s1 = orient(a, b, p);
    int s2 = orient(b, c, p);
    int s3 = orient(c, a, p);
    return (s1 >= 0 && s2 >= 0 && s3 >= 0)
        || (s1 <= 0 && s2 <= 0 && s3 <= 0);
}

// O(log n) point-in-strictly-convex-polygon test.
// Precondition: poly is CCW, has no duplicate first/last vertex,
//               and contains no three collinear vertices.
bool pointInConvex(const vector<Point>& poly, const Point& q) {
    int n = (int)poly.size();
    if (n == 0) return false;
    if (n == 1) return samePoint(poly[0], q);
    if (n == 2) return pointOnSegment(poly[0], poly[1], q);

    const Point& p0 = poly[0];
    // Outside the angular fan from p0.
    if (orient(p0, poly[1],   q) < 0) return false;
    if (orient(p0, poly[n-1], q) > 0) return false;

    // Binary search for the fan sector [l, l+1].
    int l = 1, r = n - 1;
    while (r - l > 1) {
        int m = (l + r) / 2;
        if (orient(p0, poly[m], q) >= 0) l = m;
        else                              r = m;
    }
    return pointInTriangle(p0, poly[l], poly[l+1], q);
}

Edge cases worth testing.

  • Outside the fan. A query strictly to the right of p[0]→p[1] or strictly to the left of p[0]→p[n-1] short-circuits with false.
  • On an edge. pointInTriangle is inclusive, so points on hull edges and at vertices return true. If you need strict interior, replace >= 0 / <= 0 with > 0 / < 0 there.
  • Tiny polygons. n < 3 is handled explicitly: a single point compares for equality, a "two-point polygon" degenerates to a segment containment.
  • Clockwise input. The orientation tests assume CCW. If the source data is CW, reverse the vertex list before calling; otherwise the early reject branches will fire for valid interior points.
  • Three collinear hull vertices. The primary convexHull (pivot + angle sort) already strips intermediate collinear corners; convexHullAndrew with the default inclusive = false does too. If you stored a hull built with inclusive = true (collinear boundary points kept, as required by CSES), run pointInConvex against the strict-corner version of that same hull instead.

Checklist before shipping.

  • ✅ Centroid of the polygon → true.
  • ✅ Each hull vertex itself → true.
  • ✅ Midpoint of each hull edge → true.
  • ✅ Point reflected just outside each edge → false.
  • ✅ A query collinear with p[0]→p[k] but well outside the polygon → false.

CSES — Convex Hull inclusive hull: all boundary points

Problem URL. https://cses.fi/problemset/task/2195

Story. Given n points, output the convex hull. CSES expects every point on the hull boundary, including collinear ones, so the inclusive variant is used.

Abstraction. Sort, deduplicate, and run the inclusive Andrew monotonic chain — popping only on a strict right turn (cross < 0) so collinear boundary points are retained. In the master library this is convexHullAndrew(..., /*inclusive=*/true). The drill below inlines a CSES-sized version for clarity; the primary convexHull (pivot + angle sort, Markdown style) returns the strict-corner hull and is NOT directly the CSES answer.

Why this geometry model fits. The hull is exactly the boundary of the smallest convex region (§8.3); keeping collinear points gives the full boundary listing CSES asks for (§8.4).

Algorithm. Lexicographic sort; dedup; lower then upper chain with the inclusive pop; print the hull size and vertices.

#include <bits/stdc++.h>
using namespace std;
using ll  = long long;
using lll = __int128_t;

struct Pt { ll x, y; };
bool operator<(const Pt& a, const Pt& b) { return a.x != b.x ? a.x < b.x : a.y < b.y; }
bool operator==(const Pt& a, const Pt& b) { return a.x == b.x && a.y == b.y; }

lll cross(const Pt& a, const Pt& b, const Pt& c) {
    return (lll)(b.x - a.x) * (c.y - a.y) - (lll)(b.y - a.y) * (c.x - a.x);
}

// inclusive hull: keep collinear boundary points (pop only on strict right turn).
vector<Pt> hull(vector<Pt> p) {
    sort(p.begin(), p.end());
    p.erase(unique(p.begin(), p.end()), p.end());
    int n = (int)p.size();
    if (n <= 2) return p;
    bool allCollinear = true;
    for (int i = 2; i < n; i++) {
        if (cross(p[0], p[1], p[i]) != 0) { allCollinear = false; break; }
    }
    if (allCollinear) return p;      // inclusive hull: every boundary point once
    vector<Pt> h(2 * n);
    int k = 0;
    for (int i = 0; i < n; i++) {
        while (k >= 2 && cross(h[k-2], h[k-1], p[i]) < 0) k--;
        h[k++] = p[i];
    }
    int lower = k + 1;
    for (int i = n - 2; i >= 0; i--) {
        while (k >= lower && cross(h[k-2], h[k-1], p[i]) < 0) k--;
        h[k++] = p[i];
    }
    h.resize(k - 1);
    return h;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    if (!(cin >> n)) return 0;
    vector<Pt> p(n);
    for (auto& q : p) cin >> q.x >> q.y;

    vector<Pt> h = hull(p);
    cout << h.size() << "\n";
    for (auto& q : h) cout << q.x << " " << q.y << "\n";
    return 0;
}

Complexity. O(n log n), integer-exact.

Pitfalls. Pop on cross < 0 (not ≤ 0) so collinear boundary points stay; use __int128 for the cross product on large coordinates.

Training drill — cut a convex polygon by a directed line keep the left half-plane

Scope. A focused drill on the half-plane clip, not a contest problem.

Story. Given a convex polygon in order and a directed line a → b, output the polygon kept on the left side, with its area.

Abstraction. One Sutherland–Hodgman pass keeping sc ≥ 0 and inserting a crossing on sc·sn < 0.

Why this geometry model fits. A convex polygon's kept side is always one connected piece (§8.1), so a single-output clip is exact.

Algorithm. Read the polygon and the line; clip; if the result has at least three vertices, print the count, the area, and the vertices, else print EMPTY.

#include <bits/stdc++.h>
using namespace std;
using T     = long double;
using Point = complex<T>;
#define X real()
#define Y imag()
const long double EPS = 1e-12L;

Point rot(const Point& v, T t) { return v * polar((T)1, t); }   // typed helper, never a macro

T crossF(const Point& O, const Point& A, const Point& B) {
    return (A.X - O.X) * (B.Y - O.Y) - (A.Y - O.Y) * (B.X - O.X);
}
int sideOf(const Point& a, const Point& b, const Point& p) {
    T c = crossF(a, b, p);
    if (c >  EPS) return +1;
    if (c < -EPS) return -1;
    return 0;
}
Point lineSegInt(const Point& a, const Point& b, const Point& p, const Point& q) {
    T c1 = crossF(a, b, p);
    T c2 = crossF(a, b, q);
    T t  = c1 / (c1 - c2);
    return p + (q - p) * t;
}
vector<Point> cutLeft(const vector<Point>& poly, const Point& a, const Point& b) {
    vector<Point> out;
    int n = (int)poly.size();
    for (int i = 0; i < n; i++) {
        Point cur = poly[i];
        Point nxt = poly[(i + 1) % n];
        int sc = sideOf(a, b, cur);
        int sn = sideOf(a, b, nxt);
        if (sc >= 0) out.push_back(cur);
        if (sc * sn < 0) out.push_back(lineSegInt(a, b, cur, nxt));
    }
    return out;
}
T doubledAreaF(const vector<Point>& poly) {
    int n = (int)poly.size();
    T s = 0;
    for (int i = 0; i < n; i++) {
        const Point& u = poly[i]; const Point& v = poly[(i + 1) % n];
        s += u.X * v.Y - v.X * u.Y;
    }
    return s < 0 ? -s : s;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    if (!(cin >> n)) return 0;
    vector<Point> poly(n);
    for (int i = 0; i < n; i++) {
        long double x, y; cin >> x >> y; poly[i] = Point(x, y);
    }
    long double ax, ay, bx, by;
    cin >> ax >> ay >> bx >> by;

    vector<Point> res = cutLeft(poly, Point(ax, ay), Point(bx, by));
    if ((int)res.size() < 3) { cout << "EMPTY\n"; return 0; }

    cout << fixed << setprecision(4);
    cout << res.size() << " " << (long double)doubledAreaF(res) / 2.0L << "\n";
    for (auto& q : res) cout << (long double)q.X << " " << (long double)q.Y << "\n";
    return 0;
}

Complexity. O(n) per cut.

Output meaning. EMPTY is printed when the kept side has fewer than three vertices; otherwise the kept polygon's vertex count, area, and vertices in order. Floating output carries EPS-level rounding, so compare areas with a tolerance.

Training drill — convex hull area exact doubled area of the hull

Scope. A drill combining the hull with the Class 08 integer area, not a contest problem.

Story. Given n integer points, output the area of their convex hull.

Abstraction. Build the strict hull (corners suffice for area), then take its doubled area and report it exactly as an integer plus a .0 or .5 half.

Why this geometry model fits. Hull vertices are integers, so the doubled area is exact (Class 08); halving it yields an exact multiple of one-half.

Algorithm. Hull the points; shoelace the result in __int128; print 2A/2 with the correct half.

#include <bits/stdc++.h>
using namespace std;
using ll  = long long;
using lll = __int128_t;

struct Pt { ll x, y; };
bool operator<(const Pt& a, const Pt& b) { return a.x != b.x ? a.x < b.x : a.y < b.y; }
bool operator==(const Pt& a, const Pt& b) { return a.x == b.x && a.y == b.y; }

lll cross(const Pt& a, const Pt& b, const Pt& c) {
    return (lll)(b.x - a.x) * (c.y - a.y) - (lll)(b.y - a.y) * (c.x - a.x);
}
vector<Pt> hull(vector<Pt> p) {
    sort(p.begin(), p.end());
    p.erase(unique(p.begin(), p.end()), p.end());
    int n = (int)p.size();
    if (n <= 2) return p;
    bool allCollinear = true;
    for (int i = 2; i < n; i++) {
        if (cross(p[0], p[1], p[i]) != 0) { allCollinear = false; break; }
    }
    if (allCollinear) return {p.front(), p.back()};
    vector<Pt> h(2 * n);
    int k = 0;
    for (int i = 0; i < n; i++) {
        while (k >= 2 && cross(h[k-2], h[k-1], p[i]) <= 0) k--;
        h[k++] = p[i];
    }
    int lower = k + 1;
    for (int i = n - 2; i >= 0; i--) {
        while (k >= lower && cross(h[k-2], h[k-1], p[i]) <= 0) k--;
        h[k++] = p[i];
    }
    h.resize(k - 1);
    return h;
}
lll doubledArea(const vector<Pt>& poly) {
    int n = (int)poly.size();
    lll s = 0;
    for (int i = 0; i < n; i++) {
        const Pt& a = poly[i]; const Pt& b = poly[(i + 1) % n];
        s += (lll)a.x * b.y - (lll)b.x * a.y;
    }
    return s < 0 ? -s : s;
}
string i128(lll v) {
    if (v == 0) return "0";
    string s;
    while (v) { s += char('0' + (int)(v % 10)); v /= 10; }
    reverse(s.begin(), s.end());
    return s;
}

int main() {
    ios::sync_with_stdio(false);
    cin.tie(nullptr);

    int n;
    if (!(cin >> n)) return 0;
    vector<Pt> p(n);
    for (auto& q : p) cin >> q.x >> q.y;

    vector<Pt> h = hull(p);
    lll d2 = doubledArea(h);
    cout << i128(d2 / 2) << (d2 % 2 ? ".5" : ".0") << "\n";
    return 0;
}

Complexity. O(n log n), integer-exact.

Output meaning. The exact hull area, printed as an integer with a .0 or .5 fractional part since a lattice polygon's area is always a multiple of one-half.

📝 18 — Final Summary

The mental model

  • Side test: sign of cross(b−a, p−a).
  • Cut: keep sc ≥ 0, add a crossing on sc·sn < 0.
  • Intersection: t = c₁/(c₁−c₂), point p + t(q−p).
  • Hull: sort, dedup, lower then upper chain.
  • Pop: on non-left turns; ≤ 0 strict, < 0 inclusive.
  • Area: integer shoelace of the hull, exact.

The hard rules

  • Hull is integer-exact; cut is floating with EPS.
  • Use __int128 for hull cross products on big coordinates.
  • Add a crossing only on a strict sign change.
  • Remove duplicates before building the hull.
  • A single-line cut assumes a connected (convex) kept side.

Class 09 reshapes point sets two ways with one primitive: the directed-line side test drives both the half-plane cut and the convex hull. The hull's exact integer area links straight back to Class 08, and its counterclockwise vertex order sets up the rotating-calipers techniques for farthest-pair and bounding-box problems.

What's next. Class 10 gathers the series into closest-pair and advanced mixed geometry, where the convex hull, signed area, and intersection tools combine on harder, multi-step problems.

Convex hull

  • Points are sorted lexicographically and de-duplicated before the chains are built.
